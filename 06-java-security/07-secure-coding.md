# Безопасное программирование на Java

## Что такое безопасное программирование?

Представь строительство моста. Инженер не просто рисует красивые арки. Он считает: какой ветер, какой вес, какие материалы, какие течения, какие землетрясения возможны. Для каждого фактора — запас прочности.

**Безопасное программирование** — это именно такое "проектирование с запасом прочности" для кода. Не просто "чтобы работало", а "чтобы не сломалось при атаке".

## Принципы

### Defense in Depth

| Слой | Защита |
|------|--------|
| Сеть | Firewall, WAF |
| Приложение | Валидация, авторизация |
| Данные | Шифрование, backup |
| Мониторинг | Логи, SIEM |

### Fail Secure

| Ситуация | Правильно | Неправильно |
|----------|-----------|-------------|
| Ошибка | Запретить доступ | Разрешить |
| Сбой | Закрыть дверь | Открыть |
| Exception | Вернуть 403 | Вернуть 500 |

### Least Privilege

| Что | Пример |
|-----|--------|
| Пользователь | Минимум прав |
| Сервис | Только нужные права |
| Процесс | Отдельный пользователь |

### Zero Trust

| Принцип | Описание |
|---------|----------|
| Не доверяй | Проверяй каждый запрос |
| Минимум | Только необходимое |
| Предполагай | Что уже взломали |

## Secure Coding Practices

### 1. Валидация

| Правило | Пример |
|---------|--------|
| Whitelist | Только разрешённые символы |
| Тип | Integer, не String |
| Длина | Max 100 символов |
| Формат | Email regex |
| Диапазон | 1-100 |

### 2. Аутентификация

| Правило | Пример |
|---------|--------|
| MFA | Всегда |
| Пароли | BCrypt, Argon2 |
| Сессии | Короткие, HttpOnly |
| Блокировка | После 5 попыток |

### 3. Авторизация

| Правило | Пример |
|---------|--------|
| Проверять каждый запрос | @PreAuthorize |
| Не полагаться на client-side | Серверная проверка |
| RBAC | Роли |
| ABAC | Атрибуты |

### 4. Шифрование

| Правило | Пример |
|---------|--------|
| HTTPS | Всегда |
| Данные | AES-256 |
| Пароли | BCrypt |
| Ключи | KMS |

### 5. Логирование

| Правило | Пример |
|---------|--------|
| Логировать всё | Вход, выход, ошибки |
| Не логировать | Пароли, токены |
| Структура | JSON, не plaintext |
| Защита | Неизменяемые логи |

### 6. Обработка ошибок

| Правило | Пример |
|---------|--------|
| Не показывать | Stack trace, internal info |
| Generic | "Ошибка", не "SQL ERROR" |
| Логировать | Всё |
| Fail safe | Запретить, не разрешить |

### 7. Зависимости

| Правило | Пример |
|---------|--------|
| Проверять | OWASP Dependency Check |
| Обновлять | Регулярно |
| Минимум | Только нужные |
| Подпись | Проверять |

### 8. Конфигурация

| Правило | Пример |
|---------|--------|
| Не default | Сменить пароли |
| Не debug | Production = false |
| Секреты | Не в коде |
| Минимум | Только нужные сервисы |

### Ответы на чек-лист

1. **Defense in depth** — многослойная защита. Сеть, приложение, данные, мониторинг.

2. **Fail secure** — при ошибке запретить, не разрешить. При сбое — закрыть дверь.

3. **Least privilege** — минимум прав. Только необходимое.

4. **Zero trust** — не доверяй никому. Проверяй каждый запрос.

5. **8 практик**: валидация, аутентификация, авторизация, шифрование, логирование, обработка ошибок, зависимости, конфигурация.

6. **Логировать**: вход, выход, ошибки. **Не логировать**: пароли, токены.

7. **Обработка ошибок**: не показывать stack trace, generic сообщения, логировать всё, fail safe.

8. **Зависимости**: OWASP Dependency Check, Snyk, регулярное обновление.

9. **Инструменты**: SonarQube, SpotBugs, OWASP Dependency Check, Snyk, Checkmarx, Burp Suite.

10. **Output encoding важен**, потому что защищает от XSS. Данные выводятся безопасно.
