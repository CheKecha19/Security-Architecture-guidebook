# Основы безопасности Java: защита на уровне виртуальной машины

## Что такое JVM Security Model

**JVM Security Model** — это комплексная система защиты, встроенная в архитектуру Java Virtual Machine (JVM), которая обеспечивает безопасное выполнение Java-кода. В отличие от многих других языков программирования, где безопасность часто реализуется на уровне операционной системы, Java подходит к вопросу комплексно: защита начинается ещё до выполнения кода и продолжается на протяжении всего жизненного цикла приложения.

Ключевое отличие Java — концепция **sandbox** (песочницы): изначально Java-апплеты загружались из недоверенных источников и должны были выполняться в ограниченной среде, где доступ к критическим системным ресурсам был заблокирован по умолчанию. Это стало одной из главных причин выбора Java для браузерных приложений в 1990-х годах.

## Эволюция и мотивация

Первые версии Java (1995–1996) появились в эпоху, когда веб только начинал развиваться, а концепция загрузки исполняемого кода из интернета казалась одновременно революционной и пугающей. Sun Microsystems понимали, что без надёжной модели безопасности технология не приживётся.

**Этапы эволюции:**

| Период | Версия Java | Ключевые изменения в безопасности |
|--------|-------------|-----------------------------------|
| 1996–1998 | JDK 1.0–1.1 | Базовый **SecurityManager**, простая модель sandbox |
| 1998–2002 | Java 2 (1.2–1.4) | Введение **Policy-based security**, расширяемые привилегии |
| 2004–2011 | Java 5–7 | Улучшение bytecode verifier, новые crypto APIs |
| 2014–2017 | Java 8–9 | Модульная система (**Jigsaw**), ослабление SecurityManager |
| 2021–н.в. | Java 17+ | **SecurityManager deprecated**, фокус на альтернативные механизмы |

Мотивация изменений ясна: изначально Java позиционировалась как язык для безопасного выполнения недоверенного кода, но с развитием экосистемы приоритеты сместились. Современные Java-приложения — это, как правило, серверный код, которому нужна полная функциональность, а не ограниченная песочница. Это привело к постепенному отказу от классической модели sandbox в пользу других механизмов защиты.

## Зачем это нужно?

Понимание JVM Security Model необходимо по нескольким причинам:

1. **Legacy-системы** — множество корпоративных приложений до сих пор работают на Java 8 или даже более старых версиях, где SecurityManager активно используется
2. **Аудит и compliance** — требования регуляторов (PCI DSS, SOX, GDPR) часто включают проверки изоляции кода и контроля доступа
3. **Архитектурные решения** — знание принципов позволяет правильно проектировать безопасные Java-приложения
4. **Инцидент-респонс** — при анализе уязвимостей типа **Log4Shell** понимание classloader isolation и sandbox model критически важно
5. **Собеседования** — вопросы по Java security остаются стандартом на позиции Senior/Lead Java Developer и Security Engineer

## Основные концепции

### JVM Security Model

JVM Security Model состоит из нескольких взаимосвязанных компонентов, работающих на разных уровнях:

**1. Bytecode Verification**
Первый рубеж защиты — проверка скомпилированного байткода перед его выполнением. Verifier гарантирует, что код соответствует спецификации JVM и не содержит очевидных нарушений безопасности.

**2. ClassLoader Hierarchy**
Система загрузчиков классов обеспечивает изоляцию пространств имён и контролирует, откуда загружается код. Классы из разных источников могут находиться в разных пространствах имён, предотвращая конфликты и несанкционированный доступ.

**3. SecurityManager и AccessController**
Механизм контроля доступа к критическим операциям: файловой системе, сети, системным свойствам, рефлексии и другим ресурсам. В современных версиях **SecurityManager объявлен deprecated**, но принципы контроля доступа переживают трансформацию.

**4. Java Cryptography Architecture (JCA)**
Единый API для криптографических операций: шифрование, хеширование, цифровые подписи, генерация ключей.

### Bytecode Verification

**Bytecode Verification** — это процесс статического анализа скомпилированного Java-кода (.class-файлов) перед их выполнением. Verifier проверяет, что байткод:

- Корректно типизирован (нет опасных приведений типов)
- Не нарушает стековую дисциплину (правильное количество аргументов на стеке)
- Не осуществляет недопустимый доступ к переменным
- Не содержит инструкций, нарушающих целостность JVM

| Проверка | Описание | Пример нарушения |
|----------|----------|------------------|
| **Типовая безопасность** | Корректность использования типов | Приведение Integer к String без проверки |
| **Стековая целостность** | Правильное управление операндным стеком | Pop из пустого стека или переполнение |
| **Переменные и регистры** | Корректный доступ к локальным переменным | Чтение неинициализированной переменной |
| **Поток управления** | Корректность переходов и исключений | Переход в середину метода |
| **Доступ к полям** | Проверка модификаторов доступа | Доступ к private полю извне |

Важно понимать, что verifier — это **не панацея**. Он проверяет структурную корректность, но не семантическую безопасность. Вредоносный код, корректный с точки зрения байткода, пройдёт проверку. Например, эксплуатация **Log4Shell** использовала легальные механизмы JNDI-lookup — verifier никак не мог это предотвратить.

### SecurityManager (deprecated но важен для legacy)

**SecurityManager** — класс java.lang.SecurityManager, который определяет политику безопасности приложения. С 2021 года (Java 17) объявлен **deprecated for removal**, но до сих пор активно используется в старых системах.

Принцип работы: перед выполнением чувствительной операции (открытие файла, установка сетевого соединения, вызов System.exit) код вызывает **System.getSecurityManager()** и запрашивает разрешение. SecurityManager сверяется с текущей **Policy** и либо разрешает, либо выбрасывает **SecurityException**.

Пример типичной проверки (без блока кода, как требуется):
Класс FileInputStream перед открытием файла вызывает SecurityManager.checkRead(String file). Если Policy не разрешает чтение этого пути — операция блокируется.

Почему deprecated:
- SecurityManager давно не справлялся с современными угрозами
- Накладные расходы на runtime-проверки в критических путях
- Сложность конфигурации и частые ошибки в Policy-файлах
- Концепция "недоверенного кода в песочнице" устарела для серверного Java

**Альтернативы в современной Java:**
- **JEP 411**: Постепенное удаление SecurityManager, переход на OS-level sandboxing
- **JEP 403**: Strongly Encapsulate JDK Internals — ограничение рефлексивного доступа к внутренним API JDK
- **jlink**: Создание кастомных runtime-образов с минимальным набором модулей
- **Operating system-level isolation**: Containers (Docker), seccomp, SELinux/AppArmor

### ClassLoader Isolation

**ClassLoader Isolation** — механизм разделения классов в разные пространства имён с помощью иерархии загрузчиков классов.

Иерархия загрузчиков в Java:

1. **Bootstrap ClassLoader** — загружает core JDK классы (java.lang.*, java.util.*). Реализован на C++, не является объектом ClassLoader.
2. **Platform ClassLoader** (ранее Extension) — загружает модули платформы.
3. **System (Application) ClassLoader** — загружает классы из classpath приложения.
4. **Custom ClassLoaders** — пользовательские загрузчики для специфических сценариев (OSGi, Java EE, Tomcat).

Ключевой принцип: **Parent-First Delegation Model**. Прежде чем загрузить класс самостоятельно, загрузчик делегирует запрос родителю. Это гарантирует, что системные классы загружаются только Bootstrap ClassLoader'ом и не могут быть подменены.

Зачем это важно для безопасности:
- **Изоляция приложений** в одном JVM (например, веб-приложения в Tomcat)
- **Предотвращение подмены системных классов** вредоносным кодом
- **Контроль источника кода** — ClassLoader знает, откуда загружен класс (URL, JAR, сеть)
- **Динамическая загрузка и выгрузка** — возможность обновления кода без перезапуска JVM

Уязвимости, связанные с ClassLoader:
- **ClassLoader confusion** — когда один и тот же класс загружается разными загрузчиками, возникают проблемы с instanceof и приведением типов
- **Reflection attacks** — обход модификаторов доступа через рефлексию и setAccessible(true)
- **Deserialization gadgets** — цепочки классов, используемые при deserialization attacks

## Сравнение с другими механизмами

| Аспект | Java | .NET | Node.js |
|--------|------|------|---------|
| **Runtime** | JVM (стековая VM) | CLR (стековая VM) | V8 (JIT-компиляция) |
| **Bytecode verification** | Да, обязательная | Да, обязательная | Нет (исходный код) |
| **Sandbox model** | SecurityManager (deprecated) | Code Access Security (deprecated) | VM2, isolated-vm (сторонние) |
| **Class isolation** | ClassLoader hierarchy | AppDomain (до .NET Core), AssemblyLoadContext | Worker threads, Cluster module |
| **Криптография** | JCA/JCE (встроено) | System.Security.Cryptography | crypto (модуль Node.js) |
| **Рефлексия контроль** | setAccessible (ограничено JEP 403) | ReflectionPermission | Полный доступ (по умолчанию) |
| **Управление памятью** | GC + heap isolation | GC + AppDomain изоляция | V8 heap, Worker изоляция |

**Ключевые выводы сравнения:**

Java и .NET исторически похожи — обе платформы изначально проектировались с sandbox-моделью для недоверенного кода. Обе отказываются от этой модели в пользу OS-level изоляции. Главное отличие Java — более зрелая и стандартизированная система ClassLoader'ов, тогда как .NET полагался на AppDomain'ы, которые были удалены в .NET Core.

Node.js принципиально другая модель: нет байткода в классическом смысле, нет встроенной sandbox. Изоляция достигается либо запуском отдельных процессов (Worker threads), либо внешними инструментами (Docker, VM2). Это даёт гибкость, но требует более внимательного подхода к архитектуре безопасности.

## Уроки из реальных инцидентов

### Log4Shell (CVE-2021-44228) — декабрь 2021

Log4Shell — одна из самых серьёзных уязвимостей в истории Java. Компонент **Log4j2** (популярная библиотека логирования) содержал функцию **JNDI Lookup** — возможность подставлять в логи данные из удалённых источников через протоколы LDAP, RMI, DNS.

**Как это связано с JVM Security:**
- Уязвимость обошла все уровни защиты JVM, потому что эксплуатация происходила через легальные Java-механизмы (JNDI — стандартный API)
- Bytecode verifier не мог помешать — код был легальным
- SecurityManager мог бы ограничить JNDI-доступ, но на практике почти никто не использовал Policy-файлы для этого
- ClassLoader isolation не имела значения — эксплуатация происходила внутри доверенного приложения

**Урок:** JVM Security Model защищает от низкоуровневых атак (подмена классов, повреждение памяти), но не от уязвимостей в высокоуровневых API. Архитектурное решение — принцип **least privilege** для библиотек и микросегментация.

### Spring4Shell (CVE-2022-22965) — март 2022

Уязвимость в **Spring Framework** позволяла записывать произвольные файлы на сервер через манипуляцию с DataBinder. В определённых конфигурациях (Tomcat с WAR-развёртыванием) это приводило к **Remote Code Execution**.

**Связь с JVM Security:**
- Spring использует ClassLoader'ы для изоляции контекстов, но уязвимость проявилась на уровне фреймворка
- Tomcat's ClassLoader isolation не предотвратил атаку, потому что эксплуатация шла через легальные механизмы Spring
- Проблема в небезопасном использовании рефлексии и десериализации — обе возможности встроены в JVM

**Урок:** ClassLoader isolation защищает от случайных конфликтов и подмены классов, но не от уязвимостей в самом приложении. Нужен **defense in depth**: валидация входных данных + ограничение рефлексии + runtime application self-protection (RASP).

## Инструменты и средства защиты

**Анализ и аудит:**
- **jdeps** — анализ зависимостей модулей, выявление использования внутренних JDK API
- **javap** — дизассемблер байткода для ручного анализа
- **SpotBugs** (преемник FindBugs) — статический анализатор, находит паттерны уязвимостей
- **SonarQube** — комплексный анализ кода с security-правилами для Java
- **OWASP Dependency-Check** — проверка зависимостей на известные CVE

**Runtime защита:**
- **Java Agent + Instrumentation** — перехват критических вызовов через java.lang.instrument
- **RASP (Runtime Application Self-Protection)** — коммерческие решения (Contrast Security, Imperva) внедряются как Java Agent и перехватывают опасные операции
- **SecurityManager Policy** — для legacy-систем, которые всё ещё его используют

**Сборка и деплоймент:**
- **jlink** — создание минимального runtime без лишних модулей (снижение attack surface)
- **jpackage** — упаковка приложения с собственным JRE
- **Containers + seccomp** — современная альтернатива sandbox

## Архитектурные решения

**1. Defense in Depth для Java-приложений**
- Валидация всех входных данных на границе приложения
- Использование **immutability** — неизменяемые объекты снижают риск состояний гонки
- Принцип **least privilege** для библиотек — подключайте только необходимые зависимости
- Регулярный аудит зависимостей на уязвимости

**2. Изоляция и сегментация**
- Разделение приложений на отдельные JVM или контейнеры
- Использование **Security Domains** в enterprise-системах
- **Network segmentation** — ограничение сетевого доступа Java-процессов

**3. Защита от deserialization attacks**
- Использование безопасных форматов сериализации (JSON, Protocol Buffers) вместо Java Serialization
- Если Java Serialization необходима — **ObjectInputFilter** (JEP 290) для белых списков классов
- Регулярный аудит цепочек "gadget classes" в зависимостях

**4. Управление рефлексией**
- **JEP 403** (Java 17+) — строгая инкапсуляция внутренних JDK API
- **--illegal-access=deny** — запрет незаконного доступа к внутренним API
- Аудит использования setAccessible(true) в коде и зависимостях

### Ответы на чек-лист

**1. Bytecode Verification**
Проверка гарантирует структурную корректность и типовую безопасность кода перед выполнением. Это предотвращает низкоуровневые атаки: повреждение стека, некорректные приведения типов, недопустимые переходы. Verifier не анализирует семантику — он проверяет, что код не сломает VM, а не то, что код безопасен логически.

**2. Иерархия ClassLoader'ов**
Bootstrap → Platform → System → Custom. Каждый загрузчик делегирует родителю перед самостоятельной загрузкой. Bootstrap работает на нативном уровне и загружает java.base. Platform — модули JDK. System — classpath приложения. Custom — пользовательские сценарии (OSGi, web-приложения).

**3. Проверки Bytecode Verifier**
- Типовая безопасность: все операции соответствуют типам операндов
- Стековая целостность: корректное количество элементов на операндном стеке
- Инициализация переменных: запрет чтения неинициализированных локальных переменных
- Поток управления: корректность всех переходов (внутри метода, корректные target)
- Доступ: соответствие модификаторов private/protected/public

**4. SecurityManager + AccessController + Policy**
SecurityManager — класс, определяющий политику. AccessController — реализует проверку через стек вызовов (AccessControlContext). Policy — внешний файл (или Java-код), описывающий grants: какому коду какие разрешения выданы. При чувствительной операции вызывается AccessController.checkPermission(), который проходит по стеку и проверяет, что каждый фрейм имеет право.

**5. Deprecation SecurityManager**
Причины: несовместимость с современными требованиями, производительность, сложность конфигурации, концепция sandbox для недоверенного кода устарела. Замена: OS-level sandboxing (containers, seccomp), jlink для минимизации runtime, Java Agent + Instrumentation для runtime защиты, RASP-решения.

**6. Log4Shell и Java-архитектура**
Log4j2 использовал JNDI (Java Naming and Directory Interface) — стандартный JDK API. Злоумышленник мог вставить в лог специальную строку, которая вызывала LDAP-запрос к контролируемому серверу, и JVM загружала Java-класс из ответа. Это работало через **URLClassLoader** — JVM легально загружала класс из сети. Все уровни JVM Security пропустили эту операцию, потому что она была легальной.

**7. ClassLoader isolation в Tomcat**
Каждое веб-приложение получает свой **WebappClassLoader**, дочерний к общему **Common ClassLoader**. Это изолирует классы приложений друг от друга, позволяет использовать разные версии одних библиотек, но при этом разделять общие библиотеки через Common. Проблемы: memory leaks при перезагрузке (статические поля не выгружаются), ClassLoader confusion при сериализации сессий.

**8. Java Serialization vs альтернативы**
Java Serialization (ObjectOutputStream) — нативный механизм, опасен из-за deserialization attacks (цепочки вызовов через "gadget classes"). Альтернативы: JSON (Jackson, Gson), Protocol Buffers, Avro, Kryo (с осторожностью). Если нужна Java Serialization — использовать **ObjectInputFilter** (JEP 290) для ограничения загружаемых классов.

**9. Инструменты статического анализа**
- SpotBugs (с плагином findsecbugs) — поиск security-patterns
- SonarQube с Java Security Rules — комплексный анализ
- OWASP Dependency-Check — CVE в зависимостях
- jdeps — анализ зависимостей и внутренних API
- ShiftLeft/Joern, Semgrep — современные security-сканеры

**10. JEP 403 — Strong Encapsulation**
В Java 17 внутренние API JDK (sun.*, com.sun.*) стали строго инкапсулированными. Рефлексивный доступ к ним невозможен по умолчанию. Флаг --illegal-access больше не работает (игнорируется). Это ломает старые библиотеки, которые использовали внутренние API, но существенно повышает security: уменьшает attack surface, предотвращает зависимость от нестабильных API.

## Ключевые выводы для собеседования

**На собеседовании важно продемонстрировать:**

1. **Глубокое понимание ClassLoader'ов** — иерархия, delegation model, зачем нужны custom ClassLoader'ы. Это классический вопрос на Senior Java.

2. **Осознание границ JVM Security** — verifier защищает VM от повреждения, но не от логических уязвимостей. SecurityManager был хорошей идеей для своего времени, но устарел.

3. **Знание современных альтернатив** — containers, jlink, RASP, ObjectInputFilter. Рекрутеры ценят понимание current state, не только теории.

4. **Умение связать теорию с практикой** — упомяните Log4Shell при обсуждении JNDI и ClassLoader'ов. Это показывает, что вы следите за индустрией и понимаете реальные риски.

5. **Различение уровней защиты** — bytecode verification (структура), SecurityManager/AccessController (runtime permissions), ClassLoader isolation (пространства имён), OS-level (containers) — все это разные слои defense in depth.

**Частая ловушка:** не утверждайте, что SecurityManager полностью бесполезен. В legacy-системах (Java 8, banking mainframes) он может быть активно использован. Правильный ответ: "deprecated в современных версиях, но важен для поддержки существующих систем".
