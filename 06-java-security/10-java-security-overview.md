# Обзор безопасности Java-приложений

## Часть 1: Архитектура безопасности Java

Представь средневековый замок. Не просто стена — а система:
- Ров с водой
- Подъёмный мост
- Внешняя стена
- Башни
- Внутренний двор
- Главная крепость
- Сокровищница

Каждый уровень задерживает врага. Если пробит один — есть следующий.

**Архитектура безопасности Java** — такой же многослойный замок:
1. **Сеть** — firewall, WAF
2. **Транспорт** — TLS/SSL
3. **Аутентификация** — кто ты?
4. **Авторизация** — что тебе можно?
5. **Валидация** — данные корректны?
6. **Бизнес-логика** — действие допустимо?
7. **Данные** — шифрование, хеширование
8. **Аудит** — всё записано

## Часть 2: Карта знаний

### 1. Основы JVM (Статья 01)

| Тема | Ключевое | Практика |
|------|----------|----------|
| Bytecode verification | Проверка байткода перед выполнением | JVM автоматически |
| ClassLoader | Иерархия загрузки классов | Custom ClassLoader для изоляции |
| SecurityManager | Политика доступа | Ограничение прав |
| AccessController | Проверка прав | doPrivileged() |

### 2. Spring Security (Статья 02)

| Тема | Ключевое | Практика |
|------|----------|----------|
| SecurityFilterChain | Цепочка фильтров | Настройка через DSL |
| Аутентификация | Форма, JWT, OAuth2 | Выбор по задаче |
| Авторизация | @PreAuthorize, RBAC | Проверка каждого запроса |
| Защита | CSRF, XSS, clickjacking | Включить всё по умолчанию |

### 3. OWASP Top 10 (Статья 03)

| Угроза | Риск | Защита |
|--------|------|---------|
| A01 Broken Access Control | Высокий | @PreAuthorize, RLS |
| A02 Cryptographic Failures | Высокий | AES-256, BCrypt, TLS 1.3 |
| A03 Injection | Критический | PreparedStatement, ORM |
| A04 Insecure Design | Средний | Threat modeling |
| A05 Security Misconfiguration | Высокий | Hardening, CIS |
| A06 Vulnerable Components | Высокий | OWASP Dependency-Check |
| A07 Auth Failures | Критический | MFA, сильные пароли |
| A08 Software Integrity | Средний | Подпись артефактов |
| A09 Logging Failures | Средний | SIEM, аудит |
| A10 SSRF | Средний | Валидация URL |

### 4. Аутентификация и авторизация (Статья 04)

| Механизм | Когда | Риск |
|----------|-------|------|
| Basic Auth | Только для тестов | Пароль в каждом запросе |
| Form Login | Веб-приложения | Защитить от CSRF |
| JWT | REST API, микросервисы | Короткий срок, ротация |
| OAuth2/OIDC | Делегирование доступа | Правильные flow |
| SAML | Enterprise SSO | Сложность настройки |
| mTLS | Высокая безопасность | Управление сертификатами |

### 5. Управление сессиями (Статья 05)

| Аспект | Правильно | Неправильно |
|--------|-----------|-------------|
| Cookie | HttpOnly, Secure, SameSite | Без атрибутов |
| Session ID | Случайный, менять после логина | Предсказуемый, статичный |
| Таймаут | 15-30 минут | Бесконечный |
| CSRF | Токены, SameSite | Нет защиты |
| Контроль | Максимум 2 сессии | Без ограничений |

### 6. Валидация (Статья 06)

| Уровень | Что валидировать | Как |
|---------|-----------------|-----|
| Синтаксический | Тип, длина, формат | Bean Validation |
| Семантический | Бизнес-логика | Service layer |
| Whitelist | Разрешить только хорошее | Regex |
| SQL | Запросы к БД | PreparedStatement |
| XSS | Вывод пользователю | HtmlUtils.htmlEscape |

### 7. Безопасное программирование (Статья 07)

| Принцип | Реализация |
|---------|------------|
| Defense in depth | Многослойная защита |
| Fail secure | При ошибке — запретить |
| Least privilege | Минимум прав |
| Zero trust | Не доверяй никому |
| Валидация | Все входные данные |
| Логирование | Всё, кроме секретов |
| Обработка ошибок | Generic сообщения |

### 8. Криптография (Статья 08)

| Данные | Алгоритм | НЕ использовать |
|--------|----------|-------------------|
| Пароли | BCrypt, Argon2 | SHA-256, MD5 |
| Sensitive данные | AES-256-GCM | AES-ECB |
| Ключи, токены | RSA-2048+, ECC-256+ | RSA-1024 |
| Связь | TLS 1.3 | TLS 1.0, 1.1, SSL |
| Целостность | HMAC-SHA256 | MD5 |

### 9. Зависимости (Статья 09)

| Практика | Инструмент | Частота |
|----------|------------|---------|
| Сканирование | OWASP Dependency-Check | Каждый билд |
| Мониторинг | Snyk | Постоянно |
| Обновление | Dependabot | Автоматически |
| Проверка лицензий | Maven License Plugin | При добавлении |
| BOM | Spring Boot BOM | Версионирование |

## Часть 3: Паттерны безопасной архитектуры

### Паттерн: API Gateway + JWT

    [Client]
       |
       | HTTPS + JWT
       v
    [API Gateway] — валидация JWT, rate limiting
       |
       | mTLS + JWT
       v
    [Service A] — проверка JWT, авторизация
       |
       | mTLS
       v
    [Service B] — проверка JWT, авторизация
       |
       v
    [Database] — шифрование, RLS

### Паттерн: Zero Trust Microservices

    [Service A] хочет обратиться к [Service B]
       |
       v
    [Service A] аутентифицируется в [Identity Provider]
       |
       v
    [Service A] получает токен
       |
       v
    [Service A] отправляет запрос к [Service B] с токеном
       |
       v
    [Service B] валидирует токен
    [Service B] проверяет авторизацию
    [Service B] логирует запрос

### Паттерн: Defense in Depth

| Уровень | Защита |
|---------|--------|
| Perimeter | Firewall, WAF, DDoS protection |
| Transport | TLS 1.3, mTLS |
| Authentication | OAuth2/OIDC, MFA |
| Authorization | RBAC, ABAC, ACL |
| Application | Валидация, санитизация |
| Data | Шифрование (AES-256), хеширование (BCrypt) |
| Audit | Логи, SIEM, мониторинг |

## Часть 5: Комплаенс и стандарты

| Стандарт | Требование | Реализация |
|----------|------------|------------|
| OWASP ASVS | Требования к безопасности приложений | Чек-лист по уровням (1-3) |
| PCI DSS | Защита данных карт | Шифрование, аудит, доступ |
| GDPR | Защита персональных данных | Согласие, удаление, шифрование |
| HIPAA | Защита PHI | Аудит, контроль доступа |
| SOX | Контроль финансовых систем | Аудит, разделение обязанностей |
| ISO 27001 | Управление информационной безопасностью | Политики, процедуры |
| NIST SSDF | Secure Software Development Framework | Процессы разработки |

### Уровни OWASP ASVS

| Уровень | Для кого | Что проверяет |
|---------|----------|---------------|
| Level 1 | Все приложения | Базовая безопасность |
| Level 2 | Приложения с sensitive данными | Стандартная защита |
| Level 3 | Высокая безопасность (финансы, здравоохранение) | Максимальная защита |

## Вывод

Безопасность Java-приложений — система:
1. **Архитектура** — defense in depth
2. **Аутентификация** — кто ты?
3. **Авторизация** — что тебе можно?
4. **Валидация** — данные корректны?
5. **Криптография** — данные защищены?
6. **Зависимости** — нет уязвимостей?
7. **Аудит** — всё записано?
8. **Мониторинг** — видишь угрозы?

Ключевой принцип: **безопасность — не функция, а свойство системы.**
