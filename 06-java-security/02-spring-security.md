# Spring Security: Фреймворк безопасности для Java-приложений

## 1. Что такое Spring Security?

Представьте себе современное офисное здание. У него есть центральный вход с турникетами и охраной, пропускная система для сотрудников, камеры наблюдения в коридорах и замки на дверях каждого кабинета. Посетитель не может просто пройти в серверную — его встретит охрана на ресепшене, проверит документы, выдаст гостевой пропуск и, возможно, сопроводит. Сотрудник со своим бейджем проходит свободнее, но даже он не войдёт в кабинет финансового директора без специального доступа.

Ровно так же работает **Spring Security** — он становится этой самой «системой безопасности здания» внутри вашего Java-приложения. Каждый HTTP-запрос, поступающий в приложение, подобен человеку, входящему в офис: он проходит через цепочку контрольных точек (**фильтров**), каждая из которых выполняет свою проверку.

Основу фреймворка составляет **SecurityFilterChain** — упорядоченная последовательность фильтров, через которую проходят абсолютно все входящие запросы. Это похоже на конвейер: запрос движется от фильтра к фильтру, и каждый может либо пропустить его дальше, либо заблокировать, либо модифицировать контекст безопасности. Ключевой принцип: если хотя бы один фильтр отклонил запрос, до бизнес-логики он не дойдёт.

Spring Security решает три фундаментальные задачи, неразрывно связанные друг с другом:

- **Аутентификация** — подтверждение личности. Кто вы? Сотрудник, посетитель или вообще неизвестный? Фреймворк поддерживает множество способов это выяснить: форма логина с паролем, JWT-токен в заголовке, сертификат, биометрия, вход через Google или GitHub. За аутентификацию отвечает центральный компонент **AuthenticationManager**, который получает переданные учётные данные и делегирует их проверку одному из **AuthenticationProvider**-ов. Каждый провайдер — это специализированный «проверяющий»: один работает с базой данных через **UserDetailsService**, другой обращается к LDAP-серверу, третий валидирует JWT-подпись. Если проверка пройдена, в **SecurityContext** — безопасном хранилище на время запроса или сессии — появляется объект **Authentication**, содержащий информацию о пользователе, его ролях и привилегиях.

- **Авторизация** — проверка прав. Даже зная, кто вы, система должна определить, имеете ли вы право выполнять запрошенное действие. Можно ли этому пользователю просматривать финансовые отчёты? А удалять записи? За это отвечает **AccessDecisionManager** — компонент, который собирает голоса от нескольких **Voter**-ов и выносит вердикт. Права представлены в виде **GrantedAuthority** — строк, обозначающих роли (например, `ROLE_ADMIN`) или привилегии (`READ_PRIVILEGE`). Spring Security позволяет задавать правила авторизации декларативно — через аннотации вроде **@PreAuthorize** прямо над методами сервисов, или через URL-паттерны в конфигурации.

- **Защита от типовых атак** — это то, о чём разработчики часто забывают, пока не становится поздно. **CSRF** (Cross-Site Request Forgery) — злоумышленник заставляет ваш браузер выполнить нежелательный запрос от вашего имени. **Session Fixation** — атакующий подсовывает жертве заранее известный идентификатор сессии, а после входа жертвы получает доступ. **Clickjacking** — невидимый фрейм поверх вашего сайта заставляет пользователя кликать не туда, куда он думает. Spring Security закрывает все эти векторы «из коробки», и отключать встроенные защиты нужно с чётким пониманием последствий.

Почему вообще нужен фреймворк, а не самописное решение? Потому что безопасность — это та область, где цена ошибки катастрофически высока. Написать правильно работающую аутентификацию с нуля — задача на недели, а написать её безопасно — на месяцы аудита. Нужно учесть тайминг-атаки на сравнение паролей, правильное хранение сессий, защиту от брутфорса, корректную обработку ошибок без утечки информации, валидацию JWT-подписей с защитой от алгоритмических атак (подмена `alg: none`), ротацию refresh-токенов и десятки других нюансов. Spring Security инкапсулирует эти знания, накопленные сообществом за десятилетия.

Кроме того, фреймворк спроектирован расширяемым. Это не чёрный ящик: любой компонент можно заменить собственной реализацией. Нужна кастомная логика аутентификации через SMS-код? Реализуйте свой **AuthenticationProvider**. Специфичная модель прав, не укладывающаяся в роли? Переопределите **AccessDecisionVoter**. Хотите логировать каждый заблокированный запрос? Добавьте свой фильтр в цепочку. Расширяемость Spring Security — одна из главных причин, почему он вытеснил конкурентов.

Таким образом, Spring Security — это не просто «библиотека для логина». Это архитектурный каркас безопасности, который пронизывает всё приложение: от точки входа HTTP-запроса до уровня отдельных методов в сервисном слое. Он берёт на себя рутину и сложность, позволяя разработчику сосредоточиться на бизнес-логике, а команде безопасности — на аудите конфигурации, а не на поиске уязвимостей в самописном коде.

## 2. Эволюция и мотивация

Любой популярный фреймворк не появляется из ниоткуда — он вырастает как ответ на реальные проблемы. История Spring Security — это путь от громоздких и неудобных средств безопасности Java к современному, модульному и всепроникающему решению.

### Эра JAAS: первопроходец, который не взлетел (1998)

В конце 1990-х годов Java уже была серьёзной платформой для энтерпрайза, но встроенные средства безопасности оставляли желать лучшего. В 1998 году Sun Microsystems представила **JAAS** (Java Authentication and Authorization Service) — первую попытку стандартизировать аутентификацию и авторизацию в Java. Идея была амбициозной: подключаемые модули (**LoginModule**), единый субъект безопасности (**Subject**), набор принципалов (**Principal**) и политики на основе файлов конфигурации.

На практике JAAS оказался катастрофически неудобным. Разработчику требовалось писать многословные конфигурационные файлы на странном синтаксисе, вручную управлять Subject через `Subject.doAs()`, а главное — JAAS ничего не знал о веб-приложениях. Никакой интеграции с сервлетами, никаких веб-фильтров, никакой защиты HTTP-сессий. Для эпохи, когда сервлеты уже стали стандартом де-факто, это был критический пробел. JAAS до сих пор присутствует в JDK, но в новых проектах практически не используется — он остался памятником тому, как не надо проектировать API безопасности.

### Рождение Acegi Security: правильная идея, неуклюжее воплощение (2003)

К 2003 году Java-экосистема уже плотно сидела на Spring Framework, который стремительно набирал популярность как альтернатива громоздким J2EE-серверам. Именно тогда Бен Алекс (Ben Alex) создал **Acegi Security** — фреймворк, который впервые осмысленно решал задачу безопасности именно для веб-приложений на Spring.

Acegi привнёс концепции, которые и сегодня лежат в основе Spring Security: **цепочка сервлетных фильтров** (**FilterChainProxy**), разделение аутентификации и авторизации, **SecurityContextHolder** для хранения контекста безопасности в ThreadLocal, поддержка различных источников пользовательских данных через **UserDetailsService**. Это был прорыв — наконец-то появился инструмент, понимающий, что такое HTTP, сессии, редиректы и формы логина.

Однако у Acegi была фатальная проблема: он был невероятно сложен в настройке. Конфигурация требовала десятков XML-бинов с магическими идентификаторами, которые нужно было правильно связать между собой. Типичный `applicationContext-security.xml` разрастался на сотни строк, и малейшая опечатка в имени бина приводила к загадочным ошибкам. Разработчики шутили, что «настроить Acegi сложнее, чем написать само приложение». Фреймворк решал правильные задачи, но порог входа был запретительно высоким.

### Spring Security: интеграция и зрелость (2008 — 2019)

В 2008 году произошло ключевое событие: Acegi Security стал официальным подпроектом Spring и был переименован в **Spring Security**. Это был не просто ребрендинг — началась глубокая переработка. Исчезла необходимость в громоздком XML: появился неймспейс `security:`, сокративший конфигурацию в разы. Фреймворк стал по-настоящему интегрированным с экосистемой Spring — заработала инъекция зависимостей, AOP-прокси для method security, автоматическое конфигурирование.

Версия 3.0 (2013) принесла поддержку **Java Config** — конфигурацию безопасности стало возможно описывать обычным Java-кодом, а не XML. Это был тектонический сдвиг: конфигурация стала типобезопасной, проверяемой компилятором и поддающейся рефакторингу. Тогда же появились первые встроенные инструменты для **OAuth2** — протокола, который вскоре станет стандартом для API-безопасности.

Версия 4.0 (2016) закрепила подход через **WebSecurityConfigurerAdapter** — абстрактный класс, который стал основным способом конфигурации для целого поколения разработчиков. Именно в этот период Spring Security превратился из нишевого инструмента в стандарт индустрии: практически каждый новый Java-веб-проект по умолчанию использовал именно его.

Но мир менялся, и к концу 2010-х Spring Security столкнулся с новыми вызовами. Монолитные сервлетные приложения уступали место микросервисам. REST API с JWT-токенами вытесняли серверный рендеринг с сессиями. Spring Boot сделал автоконфигурацию ожиданием по умолчанию. Реактивное программирование с WebFlux требовало неблокирующей безопасности.

### Современная эра: Spring Security 5.x — 6.x (2020 — настоящее время)

Версия 5.0 (2020) стала мостом в новую реальность. Появилась полноценная поддержка **Reactive Stack** — теперь Spring Security работал не только в сервлетном, но и в реактивном стеке WebFlux, обеспечивая неблокирующую аутентификацию. Был введён **PasswordEncoder API** с делегирующим энкодером, который мог определить алгоритм хеширования по префиксу и прозрачно перехэшировать пароли при входе пользователя. Это решило болезненную проблему миграции алгоритмов хеширования без сброса всех паролей.

Важнейшее изменение произошло в версии 5.5 (2022): **Spring Authorization Server** был выделен в отдельный проект. Десятилетиями сообщество пыталось использовать устаревший `spring-security-oauth2` для создания собственных OAuth2-серверов, но он был снят с поддержки и не соответствовал современным спецификациям. Выделение Authorization Server закрыло этот пробел и обеспечило полноценную поддержку OAuth 2.1 и OpenID Connect.

Наконец, версия 6.0 (2023) совершила то, о чём просили годами: **WebSecurityConfigurerAdapter был полностью удалён**. На смену наследованию пришёл компонентный подход — безопасность конфигурируется через объявление бина типа **SecurityFilterChain**. Конфигурация стала явной, понятной и тестируемой. Не нужно гадать, какой метод переопределён в цепочке наследования — всё описано декларативно в одном месте. Одновременно была добавлена поддержка **HTTP Basic через лямбда-DSL**, что сделало конфигурацию ещё лаконичнее.

### Почему Spring Security стал критичным

Можно выделить четыре тектонических сдвига в индустрии, которые превратили Spring Security из удобного инструмента в безальтернативный стандарт:

| Тренд | Что изменилось | Как ответил Spring Security |
|-------|----------------|---------------------------|
| **Микросервисная архитектура** | Вместо одного монолита — десятки и сотни сервисов, каждый из которых нуждается в аутентификации | Stateless JWT-безопасность, OAuth2 Resource Server из коробки, единая модель конфигурации для всех сервисов |
| **REST API вместо серверного рендеринга** | Запросы без сессий, токены в заголовках, CORS-политики | Поддержка Bearer-токенов, гибкая настройка CORS, отсутствие привязки к серверным сессиям |
| **Облачные провайдеры идентификации** | Google, GitHub, Okta, Keycloak — внешняя аутентификация стала нормой | Готовые интеграции OAuth2 Client для десятков провайдеров, автоматическое маппинг атрибутов в UserDetails |
| **Zero Trust и compliance** | SOC2, PCI DSS, GDPR требуют доказуемой модели безопасности | Аудируемая декларативная конфигурация, полное логирование событий безопасности, стандартизированные паттерны |

Spring Security перестал быть просто ещё одной библиотекой. Он стал слоем, без которого современное Java-приложение немыслимо — как фундамент, на котором держится всё здание. Не использовать Spring Security в Spring-проекте сегодня означает либо исключительную простоту приложения (статический сайт без аутентификации), либо осознанный выбор другого подхода с полным пониманием рисков и затрат на самостоятельную реализацию всех аспектов безопасности.

В следующих разделах мы детально разберём архитектуру фильтров, погрузимся в механизмы аутентификации и авторизации, рассмотрим защиту от типовых атак и научимся конфигурировать Spring Security для реальных production-сценариев.

## 3. Зачем это нужно?

Понимать устройство Spring Security на уровне компонентов — это не академическое упражнение. За каждым компонентом стоит реальный бизнес-риск, который уже материализовался в чьём-то проекте. Рассмотрим пять корпоративных сценариев, где цена незнания архитектуры безопасности измеряется не человеко-часами, а инцидентами.

### Защита REST API от несанкционированного доступа

Вы публикуете API для мобильного приложения или партнёрской интеграции. Без аутентификации любой, кто знает URL эндпоинта, может читать персональные данные клиентов, изменять заказы, инициировать платежи. Риск не в том, что «кто-то украдёт данные один раз» — злоумышленник автоматизирует сбор: скрипт перебирает идентификаторы, выкачивает всю клиентскую базу, а утром данные продаются на теневом форуме.

Spring Security закрывает этот сценарий через **Bearer-токены** и конфигурацию **Resource Server**. Каждый запрос обязан нести токен в заголовке Authorization, и фильтр аутентификации отбрасывает запросы без него ещё до того, как они достигнут контроллера. Ключевой нюанс, который часто упускают: недостаточно просто проверять наличие токена. Нужно валидировать подпись (чтобы исключить подделку), проверять срок действия, верифицировать issuer (чтобы токен от тестового стенда не работал в проде), сопоставлять scope с требуемыми правами. Spring Security выполняет все эти проверки в одном фильтре — **BearerTokenAuthenticationFilter** — прежде чем запрос увидит бизнес-логику.

### CSRF-атаки на веб-приложения

**CSRF** (Cross-Site Request Forgery) — атака, при которой злоумышленник заставляет браузер жертвы выполнить нежелательное действие на сайте, где жертва аутентифицирована. Представьте: пользователь залогинен в корпоративном портале, открывает в соседней вкладке безобидную статью, а скрытый тег `<img>` в этой статье отправляет POST-запрос на перевод средств с его счёта. Браузер автоматически прикрепляет сессионные куки, сервер видит аутентифицированного пользователя и выполняет операцию.

Для банков и финансовых сервисов это не гипотетический риск — это реальный вектор, который привёл к потерям миллионов. Spring Security включает CSRF-защиту по умолчанию для всех мутирующих HTTP-методов (POST, PUT, DELETE, PATCH). Механизм основан на **синхронизированном токене**: сервер генерирует уникальный непредсказуемый токен, встраивает его в форму или заголовок, и принимает запрос только при совпадении. Злоумышленник на стороннем сайте не может прочитать этот токен из-за Same-Origin Policy — атака блокируется на уровне архитектуры браузера. Важно понимать, когда CSRF-защиту можно отключать: только для stateless API, использующих токены в заголовках (не в куках), где у злоумышленника нет способа заставить браузер жертвы прикрепить заголовок Authorization.

### Интеграция с корпоративным IAM (Active Directory, LDAP)

В энтерпрайзе пользователи не заводятся в базе приложения — они уже существуют в **Active Directory** (AD) или **LDAP**-каталоге. Требование безопасности: сотрудник уволен — доступ ко всем системам должен быть отозван немедленно, а не когда администратор каждого из 50 приложений вручную удалит учётную запись. Централизованный Identity and Access Management (**IAM**) решает именно это.

Spring Security интегрируется с AD/LDAP через специализированный **AuthenticationProvider**, который не сравнивает пароль с локальной базой, а выполняет bind-операцию к LDAP-серверу: если сервер принял учётные данные — пользователь аутентифицирован. Дополнительно можно извлекать группы безопасности AD и мапить их на роли приложения — например, членство в группе «Finance_Approvers» автоматически даёт роль `ROLE_APPROVER`. Ошибка здесь стоит дорого: если приложение кеширует роли на часы, а изменения в AD не подхватываются, уволенный сотрудник сохраняет доступ на весь период кеша. Правильная архитектура предполагает либо короткий TTL кеша, либо механизм инвалидации.

### Микросервисная архитектура: service-to-service auth

В монолите запросы ходят внутри одного процесса и проблема аутентификации между компонентами не стоит. В микросервисах всё иначе: сервис заказов вызывает сервис платежей по HTTP, и платёжный сервис должен быть уверен, что запрос действительно пришёл от сервиса заказов, а не от злоумышленника, который подобрал внутренний URL.

Это называется **Machine-to-Machine** (M2M) или **service-to-service** аутентификацией. Базовое решение — **Client Credentials Grant** из спецификации OAuth2: каждый сервис получает собственные clientId и clientSecret, обменивает их на access-токен у Authorization Server и передаёт токен при вызовах других сервисов. Spring Security в роли **OAuth2 Client** автоматически управляет получением и обновлением токенов через **ServerOAuth2AuthorizedClientExchangeFilterFunction** (в реактивном стеке) или аналогичный механизм в сервлетном. Критично здесь избежать «плоского» пространства прав: если сервис заказов получает токен с правами на всё, компрометация одного сервиса компрометирует всю систему. Токены должны выпускаться с минимально необходимыми scope.

### Комплаенс (PCI-DSS, GDPR, ISO 27001)

Регуляторы не спрашивают, удобно ли вам. **PCI-DSS** требует, чтобы доступ к данным держателей карт контролировался на основе принципа минимальных привилегий, с уникальными идентификаторами для каждого пользователя и аудитом всех попыток доступа. **GDPR** обязывает обеспечить технические меры защиты персональных данных, включая аутентификацию и контроль доступа. **ISO 27001** требует документированной модели управления доступом.

Spring Security даёт не просто инструменты — он даёт доказуемую модель. Декларативная конфигурация через аннотации и DSL становится частью документации: аудитор видит, что доступ к методу `deleteCustomer()` имеет только роль `ROLE_ADMIN`, потому что над методом стоит `@PreAuthorize`. Фильтры логируют события аутентификации (успешные и неудачные), создавая аудиторский след. PasswordEncoder с делегирующим алгоритмом демонстрирует соответствие требованию PCI-DSS о хранении паролей с современным хешированием. Без фреймворка вам пришлось бы самостоятельно реализовывать и документировать каждый из этих механизмов — и доказывать аудитору, что самописное решение не содержит уязвимостей.

Общая картина рисков и контрмер:

| Бизнес-риск | Вектор атаки | Механизм Spring Security | Последствия игнорирования |
|------------|--------------|--------------------------|--------------------------|
| Утечка данных через API | Неаутентифицированный доступ к эндпоинтам | Bearer Token Authentication, Resource Server | Автоматизированный сбор клиентской базы, штрафы GDPR |
| Несанкционированные транзакции | CSRF через междоменные запросы | CsrfFilter с синхронизированным токеном | Финансовые потери, репутационный ущерб |
| Несвоевременный отзыв доступа | Уволенный сотрудник сохраняет доступ | LDAP/AD AuthenticationProvider без длительного кеша | Инсайдерская угроза, нарушение compliance |
| Компрометация микросервиса | Подделка межсервисных запросов | Client Credentials Grant, минимальные scope | Горизонтальное распространение атаки |
| Непрохождение аудита | Отсутствие документированной модели доступа | Декларативная конфигурация, аудит событий | Потеря сертификации, невозможность работы с платёжными системами |

Итог прост: Spring Security — это не про «добавить логин». Это про системный подход к управлению рисками, где каждый компонент фреймворка закрывает конкретный вектор атаки или требование регулятора. Понимание архитектуры фильтров, провайдеров и менеджеров — это понимание того, от чего именно защищено ваше приложение.

## 4. Основные концепции: Архитектура фильтров, Аутентификация, Авторизация

Если первый раздел дал метафору здания, а второй — исторический контекст, то сейчас мы переходим к инженерной сути. Spring Security — это, прежде всего, машина из фильтров, менеджеров и провайдеров. Разберём эту машину послойно.

### 4.1 Архитектура фильтров (Filter Chain)

В основе всего лежит **javax.servlet.Filter** — стандартный интерфейс из Servlet API, который позволяет перехватывать HTTP-запросы до того, как они достигнут сервлета. Контейнер сервлетов (Tomcat, Jetty, Undertow) ничего не знает о Spring Security — он просто вызывает зарегистрированные фильтры в определённом порядке для каждого входящего запроса.

Spring Security встраивается в этот механизм через единственную точку входа — **DelegatingFilterProxy**. Это стандартный сервлетный фильтр, который сам не выполняет никакой логики безопасности; его задача — найти в Spring-контексте бин с именем `springSecurityFilterChain` и делегировать ему обработку запроса. Зачем нужен этот прокси? Сервлетный контейнер управляет жизненным циклом фильтров сам, а Spring-бины живут в ApplicationContext. DelegatingFilterProxy — мост между двумя мирами, позволяющий Spring-бину (цепочке фильтров) обрабатывать запросы так, будто он зарегистрирован напрямую в контейнере.

Бин `springSecurityFilterChain` — это экземпляр **SecurityFilterChain**, который содержит внутри себя упорядоченный список из 15-20 специализированных фильтров. Когда запрос приходит, он последовательно проходит через каждый фильтр цепочки. Порядок фильтров жёстко определён и критически важен: фильтр, проверяющий CSRF-токен, должен сработать до фильтра, обрабатывающего форму логина, иначе злоумышленник сможет подделать логин-запрос.

Рассмотрим ключевые фильтры в порядке их выполнения:

- **ChannelProcessingFilter** — самый первый. Проверяет, не требуется ли перенаправить запрос с HTTP на HTTPS (или наоборот). В production это must-have: если клиент прислал пароль или токен по незащищённому каналу, фильтр даже не допустит обработки — сразу редирект.

- **SecurityContextPersistenceFilter** (в более новых версиях — **SecurityContextHolderFilter**) — отвечает за загрузку контекста безопасности. Для приложений с сессиями он извлекает объект **SecurityContext** из HTTP-сессии (после аутентификации он туда сохраняется) и помещает его в **ThreadLocal** через **SecurityContextHolder**. Для stateless-приложений просто инициализирует пустой контекст. После завершения запроса контекст очищается — это предотвращает утечку данных между запросами.

- **HeaderWriterFilter** — добавляет защитные HTTP-заголовки к каждому ответу: X-Content-Type-Options (запрещает MIME-сниффинг), X-Frame-Options (защита от Clickjacking), X-XSS-Protection (фильтр XSS-атак в браузере), Strict-Transport-Security (HSTS — принудительный HTTPS). Эти заголовки не требуют бизнес-логики — они просто должны быть в каждом ответе, и фильтр делает это автоматически.

- **CsrfFilter** — создаёт и валидирует CSRF-токен. Для каждого запроса генерирует токен (если его ещё нет) и сравнивает с токеном, пришедшим от клиента. Если метод запроса мутирующий, а токен отсутствует или не совпадает — запрос отклоняется немедленно, с HTTP-статусом 403.

- **LogoutFilter** — перехватывает URL `/logout` (настраивается), выполняет процедуру выхода: инвалидирует сессию, очищает SecurityContext, удаляет куки, опционально уведомляет внешний сервис (Single Sign-Out). После этого перенаправляет на страницу, которую вы укажете.

- **Authentication Filter chain** — группа из одного или нескольких фильтров, которые извлекают учётные данные из запроса и инициируют процесс аутентификации. Конкретный набор зависит от конфигурации: **UsernamePasswordAuthenticationFilter** для формы логина, **BasicAuthenticationFilter** для HTTP Basic, **BearerTokenAuthenticationFilter** для JWT-токенов. Каждый из них создаёт объект **Authentication** (пока неподтверждённый — с `authenticated=false`), наполняет его данными из запроса и передаёт **AuthenticationManager**.

- **AuthorizationFilter** (ранее — **FilterSecurityInterceptor**) — самый последний перед бизнес-логикой. Он проверяет, имеет ли текущий пользователь (уже аутентифицированный) право доступа к запрашиваемому URL. Если нет — выбрасывает **AccessDeniedException**, которая преобразуется в HTTP 403.

Важнейшая идея, которую нужно усвоить: фильтры ничего не знают друг о друге. Каждый выполняет одну чётко очерченную задачу. Фильтр CSRF не думает о том, откуда взялся пользователь, а фильтр аутентификации не проверяет CSRF-токен. Это классический **Chain of Responsibility** — паттерн, который делает архитектуру одновременно и строгой (жёсткий порядок), и гибкой (можно добавлять/заменять отдельные фильтры, не трогая остальные).

### 4.2 Аутентификация

Аутентификация — это превращение «неизвестного» в «известного». С технической точки зрения — создание полностью заполненного и подтверждённого объекта **Authentication**, который затем помещается в SecurityContext.

Центральный диспетчер аутентификации — **AuthenticationManager**. Это интерфейс с единственным методом `authenticate()`, который получает неподтверждённый объект Authentication и возвращает подтверждённый — или выбрасывает исключение. Стандартная реализация — **ProviderManager** — содержит список **AuthenticationProvider** и опрашивает их по очереди. Каждый провайдер может сказать «я умею обрабатывать этот тип аутентификации» или «не мой случай». Первый провайдер, который успешно аутентифицировал запрос, возвращает результат; остальные даже не вызываются.

Что такое AuthenticationProvider? Это специализированный «проверяльщик» для конкретного способа аутентификации. Например, **DaoAuthenticationProvider** — самый распространённый — работает следующим образом:

1. Получает из Authentication-объекта имя пользователя (username).
2. Вызывает **UserDetailsService** с этим именем, чтобы получить **UserDetails** — объект, который содержит хеш пароля, список ролей и флаги (аккаунт активен? не истёк? не заблокирован?).
3. Вызывает **PasswordEncoder**, передаёт ему пароль из запроса (открытый текст) и хеш из UserDetails — кодировщик проверяет соответствие.
4. Если пароль совпал и все флаги UserDetails в порядке — возвращает подтверждённый Authentication с ролями пользователя.

Эта цепочка намеренно разбита на три компонента, каждый со своей зоной ответственности:

- **UserDetailsService** отвечает только на вопрос «где хранятся пользователи?». Это может быть база данных (JdbcUserDetailsManager), in-memory хранилище (InMemoryUserDetailsManager), LDAP (LdapUserDetailsService) или ваш собственный сервис, который ходит в корпоративный Active Directory через REST API. Метод `loadUserByUsername()` не проверяет пароль — он только возвращает данные пользователя.

- **PasswordEncoder** отвечает только на вопрос «как проверять пароли?». Стандартные реализации покрывают все актуальные алгоритмы: BCryptPasswordEncoder (рекомендуемый, с настраиваемой сложностью), SCryptPasswordEncoder (более защищённый от атак на GPU/FPGA), Argon2PasswordEncoder (победитель Password Hashing Competition), Pbkdf2PasswordEncoder (одобрен FIPS). Ключевая особенность — **DelegatingPasswordEncoder**: он хранит алгоритм в префиксе хеша (`{bcrypt}$2a$10$...`), что позволяет прозрачно мигрировать с одного алгоритма на другой. При входе пользователя пароль проверяется текущим алгоритмом, и, если он отличается от нового стандарта, хеш автоматически обновляется — без сброса паролей.

- **Pre-Authentication** — особый сценарий, когда аутентификация уже была выполнена внешней системой, а приложение только извлекает результат. Фильтр **AbstractPreAuthenticatedProcessingFilter** читает principal из заголовка (например, `X-Forwarded-User` за обратным прокси), находит пользователя через UserDetailsService, но пропускает проверку пароля. Это типично для архитектур с API Gateway, которая уже проверила токен.

После успешной аутентификации результат сохраняется в **SecurityContext**, а тот — в **SecurityContextHolder**, который по умолчанию использует стратегию **ThreadLocal** (один контекст на поток-запрос). Для приложений с сессиями контекст дополнительно сериализуется в HTTP-сессию через SecurityContextPersistenceFilter, чтобы следующий запрос мог восстановить аутентифицированного пользователя без повторного ввода пароля.

### 4.3 Авторизация

Авторизация отвечает на вопрос «что этому пользователю позволено?» — и работает на двух уровнях: грубый контроль на уровне URL и тонкий на уровне отдельных методов.

**Права** представлены в виде **GrantedAuthority** — это, по сути, строки. Конвенция различает роли (с префиксом `ROLE_`, например `ROLE_ADMIN`) и привилегии (без префикса, например `SCOPE_read` или `DOCUMENT_APPROVE`), но технически это одинаковые объекты. Префикс `ROLE_` используется для автоматической обработки в выражениях вроде `hasRole('ADMIN')` — Spring сам добавляет префикс. Права загружаются из UserDetailsService: когда провайдер возвращает UserDetails, он наполняет его коллекцией GrantedAuthority, которые затем переносятся в Authentication при подтверждении.

**AccessDecisionManager** — это компонент, выносящий финальный вердикт о доступе. Он получает на вход объект Authentication (с уже загруженными правами), информацию о защищаемом ресурсе (URL или метод) и требуемую конфигурацию доступа. Затем опрашивает всех зарегистрированных **AccessDecisionVoter**, собирает их «голоса» (ACCESS_GRANTED, ACCESS_DENIED, ACCESS_ABSTAIN) и принимает решение.

Spring предоставляет три стратегии агрегации голосов:

- **AffirmativeBased** (используется по умолчанию) — достаточно хотя бы одного голоса GRANTED, чтобы разрешить доступ. Если никто не проголосовал GRANTED, но есть голоса DENIED — доступ запрещён.
- **ConsensusBased** — решение принимается большинством голосов. При равенстве можно настроить поведение (запретить или разрешить).
- **UnanimousBased** — доступ разрешается только если ВСЕ голосующие проголосовали GRANTED. Один DENIED — и доступ запрещён. Применяется в сценариях с повышенными требованиями безопасности.

**Voter**-ы — это узкоспециализированные «эксперты», каждый из которых проверяет свой аспект. **RoleVoter** проверяет наличие у пользователя конкретных ролей, сравнивая GrantedAuthority с требуемой конфигурацией. **AuthenticatedVoter** различает три уровня: полностью аутентифицированных пользователей, пользователей с функцией «запомнить меня» (remember-me) и анонимных — и может требовать «только полная аутентификация, без remember-me» для чувствительных операций. **PreInvocationAuthorizationAdviceVoter** обрабатывает SpEL-выражения из аннотаций `@PreAuthorize` — он самый гибкий, потому что проверять можно не только роли, но и параметры метода, время суток, IP-адрес и любые другие условия.

На практике авторизация через URL проще и быстрее, но недостаточно гибка: правило `/admin/** → ROLE_ADMIN` не различает, просматривает ли админ отчёты или удаляет пользователей. Поэтому в реальных проектах URL-авторизация задаёт базовый периметр, а method security через `@PreAuthorize` уточняет доступ на уровне операций. AccessDecisionManager объединяет оба уровня: сначала срабатывает AuthorizationFilter на URL, а затем AOP-прокси на методе, и отказ на любом уровне приводит к AccessDeniedException.

Таким образом, авторизация в Spring Security — это не бинарный «пускать/не пускать», а каскадное принятие решений с гибко настраиваемой логикой голосования и возможностью проверять практически любые условия через выражения.

### 5.3 Method Security

РџРѕРєР° URL-Р°РІС‚РѕСЂРёР·Р°С†РёСЏ Р·Р°С‰РёС‰Р°РµС‚ СЌРЅРґРїРѕРёРЅС‚С‹ С†РµР»РёРєРѕРј, **method security** РїРѕР·РІРѕР»СЏРµС‚ РєРѕРЅС‚СЂРѕР»РёСЂРѕРІР°С‚СЊ РґРѕСЃС‚СѓРї Рє РѕС‚РґРµР»СЊРЅС‹Рј РјРµС‚РѕРґР°Рј СЃРµСЂРІРёСЃРѕРІ вЂ” РІРїР»РѕС‚СЊ РґРѕ РїСЂРѕРІРµСЂРєРё Р°СЂРіСѓРјРµРЅС‚РѕРІ Рё РІРѕР·РІСЂР°С‰Р°РµРјС‹С… Р·РЅР°С‡РµРЅРёР№. Spring Security СЂРµР°Р»РёР·СѓРµС‚ СЌС‚Рѕ С‡РµСЂРµР· AOP-РїСЂРѕРєСЃРё: РїСЂРё РІС‹Р·РѕРІРµ Р·Р°С‰РёС‰С‘РЅРЅРѕРіРѕ РјРµС‚РѕРґР° Spring СЃРѕР·РґР°С‘С‚ РїСЂРѕРєСЃРё-РѕР±СЉРµРєС‚, РєРѕС‚РѕСЂС‹Р№ СЃРЅР°С‡Р°Р»Р° РїСЂРѕРІРµСЂСЏРµС‚ РїСЂР°РІР°, Р° Р·Р°С‚РµРј Р»РёР±Рѕ РїРµСЂРµРґР°С‘С‚ СѓРїСЂР°РІР»РµРЅРёРµ СЂРµР°Р»СЊРЅРѕРјСѓ РјРµС‚РѕРґСѓ, Р»РёР±Рѕ РІС‹Р±СЂР°СЃС‹РІР°РµС‚ AccessDeniedException. Р”Р»СЏ Р°РєС‚РёРІР°С†РёРё РґРѕСЃС‚Р°С‚РѕС‡РЅРѕ Р°РЅРЅРѕС‚Р°С†РёРё @EnableMethodSecurity РЅР°Рґ РєРѕРЅС„РёРіСѓСЂР°С†РёРѕРЅРЅС‹Рј РєР»Р°СЃСЃРѕРј.

Р’РЅСѓС‚СЂРё РЅРµС‘ РјРѕР¶РЅРѕ РёСЃРїРѕР»СЊР·РѕРІР°С‚СЊ **SpEL** (Spring Expression Language) вЂ” СЏР·С‹Рє РІС‹СЂР°Р¶РµРЅРёР№, РєРѕС‚РѕСЂС‹Р№ РёРјРµРµС‚ РґРѕСЃС‚СѓРї Рє РѕР±СЉРµРєС‚Сѓ Authentication, Р°СЂРіСѓРјРµРЅС‚Р°Рј РјРµС‚РѕРґР° Рё Р»СЋР±С‹Рј Spring-Р±РёРЅР°Рј. РџСЂРёРјРµСЂС‹ РІС‹СЂР°Р¶РµРЅРёР№: hasRole('ADMIN') РїСЂРѕРІРµСЂСЏРµС‚ СЂРѕР»СЊ, hasAuthority('DOCUMENT_DELETE') вЂ” РєРѕРЅРєСЂРµС‚РЅРѕРµ РїСЂР°РІРѕ, principal.username == #owner вЂ” СЃСЂР°РІРЅРёРІР°РµС‚ РёРјСЏ С‚РµРєСѓС‰РµРіРѕ РїРѕР»СЊР·РѕРІР°С‚РµР»СЏ СЃ РїР°СЂР°РјРµС‚СЂРѕРј РјРµС‚РѕРґР°, @customSecurityChecker.check(#documentId) вЂ” РІС‹Р·С‹РІР°РµС‚ СЃРѕР±СЃС‚РІРµРЅРЅС‹Р№ Р±РёРЅ РґР»СЏ СЃР»РѕР¶РЅРѕР№ Р»РѕРіРёРєРё. SpEL РїРѕРґРґРµСЂР¶РёРІР°РµС‚ Р»РѕРіРёС‡РµСЃРєРёРµ РѕРїРµСЂР°С‚РѕСЂС‹ (and, or, not), РјР°С‚РµРјР°С‚РёС‡РµСЃРєРёРµ РІС‹С‡РёСЃР»РµРЅРёСЏ, РѕР±СЂР°С‰РµРЅРёРµ Рє СЃРІРѕР№СЃС‚РІР°Рј РѕР±СЉРµРєС‚РѕРІ Рё РґР°Р¶Рµ РєРѕР»Р»РµРєС†РёРё (hasAnyRole('ADMIN', 'MODERATOR')).

## 7. Уроки из реальных инцидентов

Теория конфигурации и архитектуры обретает смысл только через призму реальных событий. За каждым крупным инцидентом в экосистеме Spring стоят конкретные технические решения — и конкретные ошибки. Разбор этих случаев позволяет понять, где заканчивается удобство фреймворка и начинается ответственность разработчика.

### Spring4Shell (CVE-2022-22965): уроки массового RCE

Весной 2022 года информационная безопасность Java-экосистемы встряхнулась от новой уязвимости — **Spring4Shell**. CVE-2022-22965 затронула **Spring Framework** версий 5.3.0–5.3.18 и 5.2.0–5.2.19, а также устаревшие ветки, и позволяла удалённое выполнение кода (**RCE**) на сервере без аутентификации. Уязвимость получила максимальный балл CVSS 9.8 — критический уровень.

Механизм атаки опирался на особенность работы **DataBinder** — компонента Spring, который преобразует входящие HTTP-параметры в свойства Java-объектов. DataBinder использует механизм **introspection** для определения setter-методов и автоматического заполнения полей. Злоумышленник мог отправить специально сформированный запрос, в котором через цепочку вложенных свойств добрался до внутренних классов Tomcat (classloader) и модифицировал конфигурацию сервера, заставив его записать произвольный JSP-файл в директорию веб-приложения. После этого злоумышленник просто обращался к этому JSP-файлу по URL — и сервер выполнял встроенный в него код с полными правами приложения.

Важно понимать: Spring4Shell не была ошибкой Spring Security напрямую. Уязвимость находилась в ядре Spring Framework — в DataBinder. Однако последствия затронули все приложения, использующие Spring Security, потому что злоумышленник получал доступ к системе в обход всех защитных фильтров. Какой смысл в аутентификации и авторизации, если атакующий может выполнить произвольный код и отключить их программно?

Ключевые уроки из Spring4Shell:

- **Отслеживание уязвимостей зависимостей** — не только вашего кода, но и всего дерева зависимостей. Spring Framework входит в огромное число проектов косвенно, через Spring Boot Starter, и команды не всегда осознают, какая версия ядра реально используется.
- **Своевременное обновление** — патч был выпущен менее чем через сутки после публичного раскрытия. Организации, у которых обновление зависимостей занимало недели, оставались открытыми для атаки.
- **Сетевое разделение** — приложение с Spring4Shell, работающее в изолированном сегменте без доступа к другим системам, наносит минимальный ущерб. В распределённых системах злоумышленник, захвативший один сервис через RCE, часто получает возможность горизонтального перемещения.
- **Проверка input** — DataBinder по умолчанию позволяет привязывать любые свойства. Явное задание разрешённых полей через @InitBinder с setAllowedFields() закрыло бы вектор для этой конкретной атаки.

### Типичная ошибка конфигурации Spring Security: "разрешить всё"

Вторая категория инцидентов не связана с нулевыми днями в самом фреймворке — она происходит от неправильного использования. Самая распространённая и опасная ошибка: отключение защит "для простоты разработки" и забывание вернуть их обратно перед релизом.

Типичный сценарий: разработчик отлаживает REST API, и **CSRF-защита** мешает — каждый POST-запрос из curl или Postman отклоняется. Вместо того чтобы настроить токен для тестов, добавляется .csrf().disable(). Позже этот код попадает в production. Теперь любой сайт в интернете может инициировать POST, PUT, DELETE-запросы к вашему API от имени аутентифицированного пользователя — классический вектор **CSRF**.

Другой частый паттерн — разрешение всех запросов без исключения: .authorizeHttpRequests().anyRequest().permitAll(). В сочетании с включённой аутентификацией это создаёт парадоксальную ситуацию: система требует логин, но после входа разрешает всё. Злоумышленник, укравший любую действительную сессию, получает полный доступ без какой-либо авторизации.

**Hardcoded credentials** — учётные данные, встроенные прямо в исходный код или конфигурационные файлы — представляют особую угрозу. Типичные примеры: пароль к базе данных в application.properties, API-ключ в аннотации @Value, секрет для JWT в виде строковой константы. Git сканирует репозитории на такие вещи, но многое проходит. Злоумышленник, получивший доступ к репозиторию (через утечку токена или скомпрометированную учётную запись разработчика), немедленно получает все секреты приложения. Правильный подход: внешнее хранилище секретов (HashiCorp Vault, AWS Secrets Manager, Spring Cloud Config с шифрованием), где приложение получает учётные данные во время выполнения, а не хранит их в исходниках.

Резюмируя, техническая защита Spring Security эффективна только при правильной конфигурации. Каждая отключённая защита должна быть осознанным решением с документированным обоснованием. Самая безопасная конфигурация — та, которая использует значения по умолчанию фреймворка, потому что они спроектированы сообществом с учётом десятилетий опыта.

## 9. Подготовка к собеседованию

Собеседование по Spring Security — это проверка не зубрёжки аннотаций, а понимания архитектурных компромиссов. Три типичные ошибки кандидатов мгновенно выдают поверхностную подготовку.

**Ошибка первая: «Spring Security — это просто фильтры»**. Такой ответ свидетельствует о недооценке архитектуры. Да, фильтры — точка входа, но за ними скрывается полноценный фреймворк: AuthenticationManager с делегированием провайдерам, AccessDecisionManager с голосованием Voter-ов, SecurityContextHolder с ThreadLocal-контекстом, method security через AOP-прокси, событийная модель AuthenticationEventPublisher. Фильтры — это верхушка айсберга. Кандидат, который не называет хотя бы три компонента внутри цепочки, демонстрирует опыт исключительно на уровне копирования примеров из документации.

**Ошибка вторая: «CSRF не нужен для REST API»**. Это непонимание контекста, а не глубокое знание. CSRF-атака работает, когда браузер автоматически прикрепляет сессионные куки к межсайтовому запросу. Если REST API использует stateless Bearer-токены в заголовке Authorization — CSRF действительно невозможен, потому что злоумышленник не может заставить браузер жертвы отправить произвольный заголовок на чужой домен. Но стоит API перейти на cookie-based сессии (что часто происходит при интеграции с веб-приложением), и CSRF возвращается. Ответ «не нужен» без уточнения механизма аутентификации — признак шаблонного мышления.

**Ошибка третья: «@PreAuthorize достаточно для всего»**. Oversimplification, который дорого обходится в production. @PreAuthorize прекрасен для декларативной авторизации на уровне методов, но он не покрывает URL-based защиту эндпоинтов, не защищает от CSRF, не управляет сессиями, не конфигурирует CORS. Правильный ответ: method security — дополнение к filter chain, а не замена. URL-based security защищает эндпоинты до входа в контроллер, @PreAuthorize — уже внутри бизнес-логики. Оба слоя нужны, потому что злоумышленник может найти обходной путь к методу, если защита только на одном уровне.

**Ситуационный вопрос первый: «Почему Spring Security использует цепочку фильтров, а не единую точку входа?»**

Единая точка входа — монолитный подход, который ломается при первом же требовании кастомизации. Цепочка фильтров даёт три фундаментальных преимущества. Во-первых, **разделение ответственности**: каждый фильтр решает одну задачу — UsernamePasswordAuthenticationFilter проверяет форму логина, BearerTokenAuthenticationFilter валидирует JWT, CsrfFilter проверяет токен CSRF. Во-вторых, **композируемость**: фильтры добавляются и удаляются без изменения остальной цепочки. В-третьих, **соответствие спецификации Servlet**: фильтры — стандартный механизм Java EE, Spring Security встраивается в него, а не изобретает свой собственный конвейер. DelegatingFilterProxy — мост между Servlet-контейнером и Spring-контекстом: он позволяет фильтрам участвовать в DI-контейнере, получать бины, управляться Spring-lifecycle. Без этого моста фильтры остались бы сиротами вне Spring-управления.

**Ситуационный вопрос второй: «Как реализовать аутентификацию service-to-service в микросервисах с Spring Security?»**

В микросервисной архитектуре человек не вводит пароль — сервис доказывает своё право вызова другому сервису. Классический подход: **mTLS** (mutual TLS) — каждый сервис предъявляет X.509-сертификат при установлении TLS-соединения, принимающий сервис проверяет подпись и доверие к CA. Spring Security интегрирует mTLS через X509AuthenticationFilter. Альтернатива: **OAuth2 Client Credentials** — сервис получает access token от Authorization Server, используя свой client_id и client_secret, и передаёт токен в заголовке каждого запроса. Spring Security 6 предоставляет готовый OAuth2AuthorizedClientManager для этого flow. Компромисс: mTLS требует инфраструктуры сертификатов (PKI) и ротации, но не зависит от центрального сервера токенов. OAuth2 проще в управлении, но вводит единую точку отказа — Authorization Server. В Zero Trust-архитектурах часто используют оба механизма: mTLS для транспорта, OAuth2 для авторизации.

**Ситуационный вопрос третий: «В чём разница между @Secured и @PreAuthorize? Когда что использовать?»**

@Secured — устаревший механизм Spring Security 2.x, поддерживающий только роли в формате строк без выражений. Он ограничен, негибок, и Spring официально рекомендует @PreAuthorize из Spring Expression Language (SpEL). @PreAuthorize позволяет писать произвольные выражения: hasRole('ADMIN'), hasAuthority('READ'), authentication.principal.username == #owner, @customSecurityChecker.check(#id). Это даёт доступ к объекту authentication, аргументам метода, бинам приложения. Когда использовать @Secured? Практически никогда — разве что в легаси-коде, где миграция невозможна. Когда использовать @PreAuthorize? Всегда, когда нужна декларативная авторизация на уровне методов. Исключение: если проект следует строгой политике JSR-250 (@RolesAllowed) для переносимости между фреймворками — тогда @RolesAllowed предпочтительнее @PreAuthorize, но это редкий кейс.

## 10. Чек-лист понимания

Десять вопросов, которые отделяют кандидата, читавшего документацию, от кандидата, понимающего, как фреймворк работает под капотом.

**Вопрос 1: Почему DelegatingFilterProxy критичен для Spring Security?**

Ответ: Servlet-спецификация создаёт фильтры через конструктор без аргументов, вне контроля Spring-контейнера. DelegatingFilterProxy — стандартный Servlet-фильтр, который делегирует работу бину из ApplicationContext. Без этого моста фильтры Spring Security не могли бы использовать dependency injection, управляться Spring lifecycle, конфигурироваться через @Bean. Это архитектурный якорь, связывающий Servlet-контейнер и Spring-контекст.

**Вопрос 2: Как AuthenticationManager выбирает AuthenticationProvider?**

Ответ: ProviderManager — стандартная реализация AuthenticationManager — перебирает провайдеры в порядке регистрации и вызывает supports() перед authenticate(). Первый провайдер, вернувший true на supports(), получает вызов authenticate(). Если провайдер бросает AuthenticationException — менеджер прерывает цепочку и пробрасывает исключение. Если проверяющий вернул null — менеджер переходит к следующему. Порядок провайдеров критичен: DAO-authentication должен идти после специализированных, иначе он перехватит запросы, предназначенные для LDAP или OAuth2.

**Вопрос 3: В каком случае Method Security предпочтительнее URL-based security?**

Ответ: URL-based security защищает эндпоинты HTTP, но не видит внутренних вызовов сервисов. Если бизнес-логика доступна через несколько контроллеров или вызывается изнутри (scheduler, batch job, внутренний API), URL-based защита обходится. Method Security через @PreAuthorize защищает метод независимо от точки входа — HTTP, JMS, внутренний вызов. Предпочтительна, когда бизнес-операции критичны и должны быть защищены в любом контексте вызова.

**Вопрос 4: Почему PasswordEncoder обязателен даже для внутренних сервисов?**

Ответ: Внутренний сервис сегодня — внешний завтра. Микросервисы мигрируют, API открываются партнёрам, базы данных утекают через SQL-инъекции или скомпрометированные учётные записи. Хранение паролей в plaintext — необратимый риск: после утечки все пароли скомпрометированы мгновенно. BCrypt с солью гарантирует, что даже при компрометации базы злоумышленник должен перебирать каждый пароль индивидуально, а радужные таблицы бесполезны из-за уникальной соли. PasswordEncoder — страховка от будущего, а не текущей угрозы.

**Вопрос 5: Как SameSite cookies защищают от CSRF в SPA?**

Ответ: SameSite — атрибут куки, контролирующий, отправляет ли браузер куки при межсайтовых запросах. Значение Strict блокирует куки при любом межсайтовом переходе, Lax — разрешает только при GET-запросах с топ-навигацией (ссылки), None требует Secure. Для SPA с cookie-based сессиями SameSite=Lax — компромисс: защищает от POST-CSRF, но не ломает OAuth2 callback и обычные переходы. SameSite не заменяет CSRF-токен, а дополняет его на уровне браузера.

**Вопрос 6: В чём разница между JWT decoder и introspection в Resource Server?**

Ответ: JWT decoder валидирует токен локально — проверяет подпись по публичному ключу, парсит claims, проверяет срок действия. Это быстро, не требует сетевого вызова, но не знает об отзыве токена. Introspection — запрос к Authorization Server с токеном, сервер отвечает активен ли токен и какие claims содержит. Это точно, учитывает отзыв, но вводит задержку и зависимость от центрального сервера. Компромисс: decoder для stateless микросервисов с короткоживущими токенами, introspection для критичных систем, где отзыв критичен.

**Вопрос 7: Почему @PostAuthorize может быть опаснее @PreAuthorize?**

Ответ: @PreAuthorize блокирует выполнение метода до входа — безопасно, минимум side effects. @PostAuthorize выполняет метод полностью, а потом проверяет возвращаемое значение. Если метод модифицирует состояние (сохраняет в БД, отправляет событие, пишет в лог), отклонение на post-проверке не откатывает side effects. Пример: метод удаления записи, защищённый @PostAuthorize("returnObject.owner == authentication.name"), удалит запись до проверки. Использовать @PostAuthorize только для чистых read-only операций.

**Вопрос 8: Как Spring Security интегрируется с OAuth2 Authorization Server?**

Ответ: Spring Security 6 не включает Authorization Server — он выделен в отдельный проект Spring Authorization Server. Resource Server конфигурируется через @Bean SecurityFilterChain с oauth2ResourceServer() — фильтр BearerTokenAuthenticationFilter извлекает токен из заголовка, NimbusJwtDecoder валидирует подпись, JwtAuthenticationConverter маппит claims в GrantedAuthority. Client — через oauth2Login() или clientCredentials(). Spring Security предоставляет абстракции, а Authorization Server — реализацию протокола. Интеграция стандартизирована через спецификации OAuth 2.1 и OpenID Connect 1.0.

**Вопрос 9: В каком случае JAAS предпочтительнее Spring Security?**

Ответ: JAAS — стандарт JDK для аутентификации, независимый от Spring. Предпочтителен в легаси-экосистемах, где JAAS уже используется годами (старые Java EE серверы, кастомные LoginModule), или когда требуется стандартная интеграция с Kerberos/SPNEGO без Spring-обёрток. Во всех остальных случаях — новые проекты, микросервисы, REST API — Spring Security превосходит JAAS по гибкости, интеграции и поддержке современных протоколов. JAAS — выбор инерции, не оптимальности.

**Вопрос 10: Почему тестирование с @WithMockUser не гарантирует production-безопасность?**

Ответ: @WithMockUser создаёт фиктивный Authentication в SecurityContext, минуя всю цепочку фильтров. Тест проверяет авторизацию на уровне метода, но не проверяет: корректно ли конфигурирован SecurityFilterChain, работает ли CSRF-защита, правильно ли настроены CORS-заголовки, валидируется ли JWT-подпись, обрабатываются ли ошибки аутентификации. @WithMockUser тестирует «что происходит после аутентификации», но не «происходит ли аутентификация правильно». Нужны интеграционные тесты с TestRestTemplate или WebTestClient, проверяющие полный HTTP-цикл.

## Ключевые выводы для собеседования

- **Архитектура важнее API**: знание, что делает @PreAuthorize, не поможет без понимания, почему фильтры организованы в цепочку, как AuthenticationManager выбирает провайдера, и где именно происходит каждое решение. Собеседующий ищет системное мышление, не зубрёжку аннотаций.

- **Безопасность — многослойная защита**: ни один механизм не достаточен. Фильтры + Method Security + CSRF-токены + SameSite + PasswordEncoder + аудит — каждый слой закрывает вектор, который другие пропускают. Кандидат, предлагающий «одно решение на все случаи», демонстрирует опасную поверхностность.

- **Production отличается от примеров**: отключение CSRF «для тестов», hardcoded credentials в properties, @WithMockUser как единственный способ тестирования — всё это признаки кода, который не выживет под реальной нагрузкой и реальной угрозой. Spring Security даёт инструменты, но правильная конфигурация — ответственность команды.
