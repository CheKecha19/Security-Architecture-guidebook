# Безопасность зависимостей в Java

## Что такое зависимость?

Представь, что ты строишь дом. Ты не делаешь всё сам: не выплавляешь сталь, не делаешь кирпичи, не производишь провода. Ты покупаешь готовые материалы у поставщиков.

**Зависимость** — это именно такой "материал" для программы. Ты не пишешь всё с нуля. Берёшь готовую библиотеку: для работы с базой данных, для HTTP-запросов, для JSON, для логирования.

Проблема: если в материале дефект — весь дом под угрозой. Если в библиотеке уязвимость — твоя программа под угрозой.

## Зачем нужна безопасность зависимостей?

### Ситуация 1: Log4Shell

Библиотека Log4j версии 2.14.1 содержит уязвимость. Любое сообщение лога может выполнить произвольный код. Узнали об этом — через 2 года после появления уязвимости.

### Ситуация 2: Устаревшая библиотека

Проект использует библиотеку версии 1.0. Версия 1.5 исправляет уязвимость. Никто не обновил.

### Ситуация 3: Транзитивная зависимость

Твоя библиотека A зависит от B. B зависит от C. C — уязвима. Ты не знаешь, что используешь C.

## Типы уязвимостей зависимостей

| Тип | Описание | Пример |
|-----|----------|--------|
| Known vulnerability | Известная CVE | Log4Shell |
| License violation | Нарушение лицензии | GPL в коммерческом |
| Outdated | Устаревшая | Без поддержки |
| Malicious | Вредоносная | Typosquatting |
| Transitive | Транзитивная | Уязвимость в глубине |

### Ответы на чек-лист

1. **Зависимость** — внешняя библиотека, используемая проектом.

2. **Типы**: known vulnerability, license violation, outdated, malicious, transitive.

3. **Транзитивная** — зависимость зависимости. Используется неявно.

4. **Инструменты**: OWASP Dependency-Check, Snyk, Dependabot, Maven Enforcer, Nexus Lifecycle.

5. **BOM** — Bill of Materials. Перечень всех зависимостей с версиями.

6. **Lockfile** — файл с зафиксированными версиями. maven-lockfile, package-lock.json.

7. **Dependabot** — автоматически создаёт PR с обновлениями зависимостей.

8. **Typosquatting** — поддельный пакет с похожим названием.

9. **Лицензии**: проверять совместимость. GPL, MIT, Apache.

10. **Лучшие практики**: BOM, lockfile, минимум, регулярное обновление, автоматизация, проверка транзитивных, лицензии.
