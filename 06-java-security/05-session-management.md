# Управление сессиями в Java

## Часть 1: Что такое сессия и зачем она нужна?

Представь, что ты заходишь в банк. Тебе выдают номерок. С этим номерком ты можешь ходить по отделениям, не объясняя каждый раз, кто ты. Номерок — твоя временная «палочка-выручалочка». Она работает, пока ты в банке. Уходишь — номерок аннулируется.

**Сессия** — это именно такой «номерок» для веб-приложения. HTTP — протокол без состояния: каждый запрос независим. Но приложению нужно помнить: кто пользователь, что в его корзине, какие у него права. Для этого сервер создаёт **сессию** — временное хранилище данных, привязанное к конкретному пользователю.

Когда ты входишь на сайт, сервер:
1. Создаёт объект сессии в памяти
2. Генерирует уникальный **Session ID**
3. Отправляет этот ID клиенту (обычно в cookie)
4. При каждом следующем запросе клиент передаёт ID
5. Сервер находит сессию по ID и восстанавливает контекст

Без сессии тебе пришлось бы вводить логин и пароль на каждой странице. С сессией — один раз вошёл, и сервер помнит тебя.

## Зачем нужно управление сессиями?

### Ситуация 1: Угон сессии (Session Hijacking)

Ты вошёл в интернет-банк в кафе через публичный Wi-Fi. Злоумышленник сидит в том же кафе и перехватывает трафик. В HTTP-запросах он видит твой **JSESSIONID**. Теперь он может отправлять запросы от твоего имени: переводить деньги, менять настройки. Ты даже не узнаешь, пока не проверишь баланс.

### Ситуация 2: Бесконечная сессия

Ты вошёл на сайт на работе и ушёл домой. Сессия осталась активной. Утром коллега открыл твой компьютер — и он уже вошёл под твоей учётной записью. Без таймаута сессии вход без пароля возможен часами и днями.

### Ситуация 3: Фиксация сессии (Session Fixation)

Злоумышленник заходит на сайт и получает Session ID: `ABC123`. Он отправляет тебе ссылку: «Кликни сюда!» Ссылка содержит `JSESSIONID=ABC123`. Ты кликаешь, входишь на сайт, аутентифицируешься. Но Session ID остаётся `ABC123` — теперь злоумышленник знает твой ID и может делать запросы от твоего имени.

## Часть 2: Механизмы сессий

### Cookie-based сессии

Классический способ. Сервер создаёт сессию в памяти, отправляет Session ID в cookie.

| Атрибут cookie | Описание | Безопасность |
|----------------|----------|--------------|
| JSESSIONID | Стандартное имя сессии в Java | — |
| HttpOnly | Cookie недоступно для JavaScript | Защита от XSS |
| Secure | Cookie передаётся только по HTTPS | Защита от перехвата |
| SameSite | Ограничение отправки cookie при cross-origin | Защита от CSRF |
| MaxAge | Время жизни cookie в секундах | Ограничение сессии |
| Domain | Для какого домена действует cookie | Изоляция |
| Path | Для какого пути | Изоляция |

**Пример безопасной cookie:**

    Set-Cookie: JSESSIONID=abc123def456; HttpOnly; Secure; SameSite=Strict; MaxAge=1800; Path=/

Что означает:
- **HttpOnly** — JavaScript не может прочитать cookie (защита от XSS)
- **Secure** — cookie передаётся только по HTTPS (защита от перехвата)
- **SameSite=Strict** — cookie не отправляется при переходе с других сайтов (защита от CSRF)
- **MaxAge=1800** — сессия 30 минут
- **Path=/** — cookie действует для всего сайта

### Token-based сессии (JWT)

В отличие от cookie-based, где состояние хранится на сервере, JWT содержит всё необходимое внутри токена.

| Тип токена | Описание | Когда использовать |
|------------|----------|-------------------|
| Access Token | Короткоживущий (5-15 минут), содержит claims | Каждый запрос |
| Refresh Token | Долгоживущий (дни/недели), для обновления Access Token | Когда Access Token истёк |
| ID Token | Информация о пользователе (OpenID Connect) | Аутентификация |

**Преимущества JWT:**
- Stateless — сервер не хранит сессии, проще масштабировать
- Можно передавать между сервисами (микросервисы)
- Самодостаточен — содержит всё внутри

**Недостатки JWT:**
- Нельзя отозвать мгновенно (только через blacklist)
- Токен больше cookie
- Сложнее управлять (refresh, rotation)

### Spring Session

Spring Session — абстракция над хранилищем сессий. Позволяет хранить сессии не в памяти одного сервера, а во внешнем хранилище.

| Хранилище | Описание | Когда использовать |
|-----------|----------|-------------------|
| JDBC | В базе данных | Простота, когда уже есть БД |
| Redis | In-memory key-value | Производительность, масштабирование |
| MongoDB | Документо-ориентированное | Если уже используется Mongo |
| Hazelcast | Распределённый кэш | Enterprise кластеризация |

Зачем Spring Session:
- Масштабирование: 10 серверов — одно хранилище сессий
- Отказоустойчивость: сервер упал — сессия на другом
- Разделение: сессии не занимают память JVM

## Часть 3: Угрозы и защита

### Session Fixation — фиксация сессии

**Атака:**
1. Злоумышленник получает Session ID (например, открыв сайт сам)
2. Он заставляет пользователя использовать этот ID (через ссылку, XSS, header injection)
3. Пользователь входит на сайт, аутентифицируется
4. ID сессии не меняется — злоумышленник знает ID аутентифицированной сессии

**Защита:** Менять Session ID после аутентификации.

Spring Security делает это автоматически:

    http.sessionManagement(session -> session
        .sessionFixation(SessionFixationConfigurer::newSession)
    );

Варианты:
- **newSession** — создать новую сессию, не копировать атрибуты
- **migrateSession** — создать новую, скопировать атрибуты (по умолчанию)
- **none** — не менять (опасно!)

### Session Hijacking — угон сессии

**Атака:** Перехват Session ID через:
- Перехват трафика (man-in-the-middle)
- XSS (вредоносный JavaScript читает cookie)
- Доступ к файлам браузера
- Подбор ID (если предсказуемый)

**Защита:**
- **HTTPS** — шифрование трафика
- **HttpOnly** — cookie недоступно JavaScript
- **Secure** — только HTTPS
- **Криптографически случайные ID** — нельзя предсказать
- **Привязка к IP/User-Agent** — проверка consistency
- **Device fingerprint** — проверка устройства

### Session Prediction — предсказание ID

**Атака:** Если Session ID генерируется предсказуемо (например, `1`, `2`, `3`...), злоумышленник может угадать ID другого пользователя.

**Защита:**
- Использовать криптографически стойкий генератор случайных чисел
- Spring Security использует `SecureRandom` для генерации ID
- Длина ID минимум 128 бит (16 байт)

### Concurrent Session Control — управление параллельными сессиями

**Ситуация:** Пользователь вошёл с домашнего компьютера и с работы. Это нормально. Но если кто-то украл его пароль и вошёл с другого устройства — это угроза.

**Защита:** Ограничение количества сессий:

    http.sessionManagement(session -> session
        .maximumSessions(1)  // Максимум 1 сессия
        .maxSessionsPreventsLogin(false)  // Новый вход вытесняет старый
        .expiredUrl("/login?expired")
    );

| Параметр | true | false |
|----------|------|-------|
| maxSessionsPreventsLogin | Новый вход запрещён, пока старая сессия активна | Старая сессия инвалидируется |

## Часть 4: CSRF — Cross-Site Request Forgery

### Что такое CSRF?

Ты вошёл в интернет-банк. Вкладка осталась открытой. Ты открыл другой сайт — с котиками. На сайте с котиками есть скрытая форма:

    <form action="https://bank.com/transfer" method="POST" id="evil">
      <input type="hidden" name="to" value="attacker">
      <input type="hidden" name="amount" value="100000">
    </form>
    <script>document.getElementById('evil').submit();</script>

Твой браузер автоматически отправляет cookie банка. Банк думает: это ты, аутентифицированный пользователь, хочешь перевести деньги.

### Защита Spring Security

**1. Synchronizer Token Pattern:**

    http.csrf(csrf -> csrf
        .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
    );

Spring Security:
1. Генерирует CSRF-токен
2. Кладёт его в сессию и в cookie/форму
3. При POST-запросе проверяет: токен из формы == токен из сессии
4. Если не совпадает — отклоняет запрос

**2. SameSite Cookie:**

SameSite=Strict — cookie не отправляется при переходе с других сайтов. Даже если форма отправлена, cookie не придёт.

**3. Double Submit Cookie:**

Токен в cookie и тот же токен в заголовке/форме. Сервер сравнивает.

## Часть 5: Лучшие практики управления сессиями

### Конфигурация Spring Security

    http.sessionManagement(session -> session
        // Создавать сессию только если нужно
        .sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED)
        
        // Менять ID после аутентификации
        .sessionFixation(SessionFixationConfigurer::migrateSession)
        
        // Максимум 2 сессии на пользователя
        .maximumSessions(2)
        .maxSessionsPreventsLogin(false)
        .expiredUrl("/login?expired")
        
        // URL для недействительной сессии
        .invalidSessionUrl("/login?invalid")
    );

### Политика SessionCreationPolicy

| Политика | Описание | Когда использовать |
|----------|----------|-------------------|
| ALWAYS | Создавать сессию всегда | Stateful приложения |
| NEVER | Не создавать, но использовать существующую | API с Basic Auth |
| IF_REQUIRED | Создавать, если нужно (по умолчанию) | Большинство случаев |
| STATELESS | Не использовать сессии | REST API, JWT |

### Cookie — чек-лист безопасности

| Атрибут | Обязательно | Описание |
|---------|-------------|----------|
| HttpOnly | ✅ | Недоступно JavaScript |
| Secure | ✅ | Только HTTPS |
| SameSite | ✅ | Strict или Lax |
| MaxAge | ✅ | Ограниченное время |
| Domain | По необходимости | Не ставить, если не нужно |
| Path | ✅ | Минимально необходимый |

### Таймауты

| Параметр | Рекомендация | Описание |
|----------|-------------|----------|
| server.servlet.session.timeout | 30 минут | Таймаут бездействия |
| MaxAge cookie | Совпадает с таймаутом | Синхронизация |
| Absolute timeout | 8-12 часов | Максимальное время жизни сессии |
| Sliding expiration | Обновлять при активности | Продление при использовании |

### Device Fingerprint и аномалии

    // Проверка User-Agent при каждом запросе
    String currentUA = request.getHeader("User-Agent");
    String sessionUA = (String) session.getAttribute("userAgent");
    
    if (sessionUA != null && !sessionUA.equals(currentUA)) {
        // Возможен угон сессии
        session.invalidate();
        throw new SessionAuthenticationException("Device changed");
    }

### Инвалидация сессии

При выходе сессия должна уничтожаться:

    http.logout(logout -> logout
        .logoutUrl("/logout")
        .logoutSuccessUrl("/login?logout")
        .invalidateHttpSession(true)
        .deleteCookies("JSESSIONID")
    );

## Чек-лист безопасности сессий

| Проверка | Статус |
|----------|--------|
| HTTPS обязателен | ⬜ |
| HttpOnly на cookie | ⬜ |
| Secure на cookie | ⬜ |
| SameSite=Strict или Lax | ⬜ |
| Смена ID после аутентификации | ⬜ |
| Ограниченное время жизни | ⬜ |
| CSRF-токены на формах | ⬜ |
| Контроль параллельных сессий | ⬜ |
| Инвалидация при выходе | ⬜ |
| Привязка к устройству (опционально) | ⬜ |

### Комплаенс

| Стандарт | Требование | Реализация |
|----------|------------|------------|
| PCI DSS | Таймаут сессии ≤ 15 минут | server.servlet.session.timeout=15m |
| SOX | Аудит сессий | Логирование входов/выходов |
| GDPR | Защита персональных данных | HttpOnly, Secure, шифрование |
| OWASP | Защита от Session Hijacking | Все атрибуты cookie |

## Вывод

Управление сессиями — критическая часть безопасности:
1. **Аутентификация** — только начало
2. **Защита ID** — случайность, привязка
3. **Cookie** — HttpOnly, Secure, SameSite
4. **CSRF** — токены, SameSite
5. **Таймауты** — ограничение жизни
6. **Контроль** — параллельные сессии
7. **Аудит** — логирование и мониторинг

Сессия — это доверие. Защити его.
