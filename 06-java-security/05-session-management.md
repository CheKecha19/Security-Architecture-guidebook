# Управление сессиями в Java

## Что такое сессия?

Представь, что ты заходишь в банк. Тебе выдают номерок. С этим номерком ты можешь ходить по отделениям, не объясняя каждый раз, кто ты. Номерок — твоя временная "палочка-выручалочка". Она работает, пока ты в банке. Уходишь — номерок аннулируется.

**Сессия** — это именно такой "номерок" для веб-приложения. Когда ты заходишь на сайт, сервер создаёт для тебя сессию. Даёт идентификатор. Пока сессия активна, сервер знает, кто ты. Не спрашивает логин и пароль на каждой странице.

## Зачем нужно управление сессиями?

### Ситуация 1: Угон сессии

Злоумышленник украл идентификатор сессии. Теперь он — это ты. Может делать всё от твоего имени.

### Ситуация 2: Бесконечная сессия

Пользователь вошёл и ушёл на месяц в отпуск. Сессия всё ещё активна. Если кто-то найдёт компьютер — вход без пароля.

### Ситуация 3: Две сессии

Пользователь вошёл с двух устройств. Одно украли. Нужно отключить только одну сессию.

## Механизмы сессий

### Cookie-based

| Атрибут | Описание |
|---------|----------|
| JSESSIONID | Стандартное имя |
| HttpOnly | Не доступно JavaScript |
| Secure | Только HTTPS |
| SameSite | Защита CSRF |
| MaxAge | Время жизни |
| Domain | Для какого домена |
| Path | Для какого пути |

### Token-based

| Тип | Описание |
|-----|----------|
| JWT | Stateless, подписанный |
| OAuth2 | Токен доступа |
| Refresh token | Для обновления |
| ID token | Информация о пользователе |

### Spring Session

| Хранилище | Описание |
|-----------|----------|
| JDBC | В базе данных |
| Redis | В памяти, быстро |
| MongoDB | Документо-ориентированное |
| Hazelcast | Распределённое |

## Угрозы

### Session Fixation

| Атака | Описание |
|-------|----------|
| Злоумышленник создаёт сессию | Пользователь входит, ID остаётся |
| Решение | Менять ID после аутентификации |

### Session Hijacking

| Атака | Описание |
|-------|----------|
| Украли cookie | Перехват по сети |
| Решение | HTTPS, HttpOnly, Secure |

### Session Prediction

| Атака | Описание |
|-------|----------|
| Угадали ID | Предсказуемые ID |
| Решение | Криптографически случайные |

## Конфигурация Spring Security

```java
http.sessionManagement(session -> session
    .sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED)
    .maximumSessions(1) // один пользователь - одна сессия
    .maxSessionsPreventsLogin(false)
    .expiredUrl("/login?expired")
    .sessionFixation(SessionFixationConfigurer::newSession) // менять ID
);
```

## CSRF

### Что такое CSRF?

Пользователь вошёл в банк. Открыл другой сайт. Этот сайт отправляет запрос в банк: "Переведи деньги". Браузер отправляет cookie автоматически. Банк думает: это пользователь.

### Защита Spring Security

| Метод | Описание |
|-------|----------|
| Synchronizer Token | CSRF-токен в форме |
| SameSite Cookie | SameSite=Strict или Lax |
| Double Submit | Токен в cookie и форме |

```java
http.csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
);
```

## Лучшие практики

| Практика | Описание |
|----------|----------|
| HTTPS | Всегда |
| HttpOnly | Cookie не доступны JS |
| Secure | Только HTTPS |
| SameSite | Защита CSRF |
| Короткое время | 15-30 минут |
| Менять ID | После логина |
| Инвалидировать | При logout |
| Мониторинг | Необычная активность |
| Device fingerprint | Привязка к устройству |
| Concurrent control | Одна сессия на пользователя |

### Ответы на чек-лист

1. **Сессия** — временное хранилище данных пользователя на сервере. По идентификатору.

2. **Механизмы**: cookie-based (JSESSIONID), token-based (JWT), Spring Session (JDBC, Redis).

3. **Session fixation** — атака, где ID сессии создаётся злоумышленником, пользователь входит, ID остаётся.

4. **Session hijacking** — кража идентификатора сессии. Перехват cookie.

5. **Защита cookie**: HttpOnly, Secure, SameSite, короткий срок.

6. **CSRF** — Cross-Site Request Forgery. Сайт отправляет запрос от имени пользователя.

7. **Spring Security**: CSRF-токен в форме, SameSite cookie, double submit.

8. **SameSite** — атрибут cookie. Strict (никогда не отправлять), Lax (GET разрешить), None (всегда).

9. **Лучшие практики**: HTTPS, HttpOnly, Secure, SameSite, короткое время, менять ID, инвалидировать, мониторинг.

10. **Менять ID важно**, потому что защищает от session fixation.
