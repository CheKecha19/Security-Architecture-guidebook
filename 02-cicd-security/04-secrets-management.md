# Управление секретами: Ключи, пароли, токены

## Что такое секреты?

Представь, что ты владелец сейфового хранилища. В хранилище лежат ключи от всех дверей компании, PIN-коды банковских карт, комбинации замков серверных, пароли от криптокошельков. Это не просто бумажки — это доступ ко всему.

Если кто-то украдёт содержимое хранилища — он сможет зайти куда угодно. Он сможет перевести деньги, открыть двери, получить доступ к данным.

**Секреты в информационной безопасности** — это всё, что даёт доступ: пароли, API-ключи, токены, сертификаты, ключи шифрования, учётные данные баз данных.

Секреты — это самое ценное, что есть у системы. Если злоумышленник получит секреты — он получит всё. Поэтому управление секретами — одна из важнейших задач безопасности.

## Зачем нужно управление секретами?

### Ситуация 1: Утечка пароля в коде

Разработчик подключается к базе данных. Он пишет в коде: "Пароль от базы — SuperSecret123". И отправляет код в GitHub.

Теперь пароль виден всем, у кого есть доступ к репозиторию. А если репозиторий публичный — его видит весь интернет.

Управление секретами решает эту проблему. Пароль хранится отдельно от кода, в защищённом хранилище. Код обращается к хранилищу: "Дай мне пароль от базы". Хранилище проверяет права и выдаёт пароль.

### Ситуация 2: Уход сотрудника

Системный администратор увольняется. Он знал пароли от всех серверов. Без управления секретами — приходится менять все пароли вручную. Это долго, сложно, где-то точно забудут.

С управлением секретами — администратор имел доступ через свою учётную запись. После увольнения учётка отключается — и доступ автоматически пропадает. Ничего менять не нужно.

### Ситуация 3: Ротация ключей

Компания использует API-ключ для интеграции с партнёром. Ключ нужно менять раз в месяц — это требование compliance.

Без управления секретами: администратор вручную генерирует новый ключ, обновляет его везде, где используется. Забывает об одном сервисе — тот перестаёт работать.

С управлением секретами: система сама генерирует новый ключ, обновляет его во всех сервисах, уведомляет администратора. Всё автоматически.

## Типы секретов

| Тип секрета | Что это | Где используется |
|-------------|---------|-------------------|
| Пароль | Секретная строка для аутентификации | Базы данных, сервисы, учётные записи |
| API-ключ | Ключ для доступа к API | Интеграции с внешними сервисами |
| Токен | Временный ключ доступа | Аутентификация, авторизация |
| Сертификат | Криптографический документ | TLS/SSL, подпись |
| Ключ шифрования | Ключ для шифрования данных | Хранение данных, шифрование дисков |
| Учётные данные | Логин + пароль | Подключение к базам, сервисам |
| SSH-ключ | Ключ для SSH-доступа | Серверы, репозитории |

## Где нельзя хранить секреты

### В коде

Хардкод пароля в исходном коде — самая распространённая ошибка. Код попадает в Git — пароль виден всем. Даже если потом удалить — он остаётся в истории Git.

Это как написать PIN-код от карты на стикере и приклеить его на монитор.

### В конфигурационных файлах

Файлы типа config.ini, application.properties часто хранятся в Git. Если в них есть пароли — они утекают вместе с кодом.

### В переменных окружения на сервере

Переменные окружения видны всем процессам на сервере. Любой, кто получит доступ к серверу, увидит все переменные окружения.

### В логах

Приложение случайно записывает секрет в лог. "Ошибка подключения к базе данных superuser:SuperSecret123@db". Теперь пароль в логе, который читают все администраторы.

### В Docker-образах

Секрет, добавленный в образ Docker, остаётся в истории слоёв. Даже если удалить файл в финальном слое — он виден в промежуточных слоях.

## Где хранить секреты

### Специализированные хранилища

**HashiCorp Vault**

Vault — централизованное хранилище секретов. Он хранит секреты зашифрованными, контролирует доступ, логирует обращения, поддерживает ротацию.

Vault умеет:
- Хранить статические секреты (пароли, ключи)
- Генерировать динамические секреты (временные учётные записи)
- Шифровать данные
- Подписывать сертификаты
- Аудит всех операций

**AWS Secrets Manager / Azure Key Vault / Google Secret Manager**

Облачные хранилища секретов. Интегрированы с облачными сервисами. Управляют доступом через облачную IAM.

**Kubernetes Secrets**

Встроенный механизм Kubernetes для хранения секретов. Хранит их в etcd в закодированном виде (base64). Позволяет монтировать секреты в контейнеры как файлы или переменные окружения.

### Принципы хранения

**Шифрование**

Секреты должны храниться в зашифрованном виде. Даже если хранилище скомпрометировано — без ключа шифрования секреты не получить.

**Доступ по необходимости**

Процесс должен получать только те секреты, которые ему нужны. Веб-серверу нужен пароль от базы данных. Ему не нужны ключи от других сервисов.

**Аудит**

Каждое обращение к секрету должно логироваться. Кто запросил, когда, зачем, получил ли доступ.

**Ротация**

Секреты должны меняться регулярно. Даже если ключ украден — через месяц он станет бесполезен.

## Передача секретов в CI/CD

### Проблема

CI/CD pipeline собирает и деплоит приложение. Pipeline нужен доступ к секретам: пароль от Docker registry, ключи от сервера деплоя, токены от облака.

Как передать эти секреты pipeline, не раскрывая их?

### Решения

**Хранилища секретов CI/CD**

GitHub Actions, GitLab CI, Jenkins — все имеют встроенные механизмы хранения секретов. Секреты хранятся в зашифрованном виде, доступны только pipeline.

**OIDC-токены**

Вместо хранения API-ключей pipeline использует OIDC. Pipeline получает временный токен от провайдера идентичности. Токен действует несколько минут и даёт доступ только к конкретным ресурсам.

Это как если бы вместо ключа от офиса ты получал одноразовый пропуск на конкретное время.

**Short-lived credentials**

Вместо долгоживущих паролей — временные учётные данные. GitHub Actions получает от AWS временные credentials через OIDC. Они действуют 15 минут и имеют доступ только к конкретному S3-бакету.

### Best practices

**Не хранить секреты в коде**

Самое главное правило. Никаких паролей в исходном коде, конфигурационных файлах, комментариях.

**Использовать short-lived токены**

Вместо статических ключей — временные токены. Меньше времени действия — меньше риск.

**Ограничивать scope**

Токен должен иметь доступ только к необходимым ресурсам. Не "всё в облаке", а "только этот S3-бакет".

**Логировать доступ**

Кто и когда запросил секрет. Если произошла утечка — можно отследить, кто имел доступ.

## Ротация секретов

### Зачем ротировать

Даже если секрет не утек — он мог быть подсмотрен. Или сотрудник, который его знал, уволился. Или ключ мог быть скомпрометирован незаметно.

Регулярная смена секретов сокращает окно возможной атаки.

### Как ротировать

**Ручная ротация**

Администратор генерирует новый секрет, обновляет его во всех сервисах, удаляет старый. Медленно, ошибкоёмко, но контролируемо.

**Автоматическая ротация**

Система сама генерирует новый секрет, обновляет его в хранилище, уведомляет зависимые сервисы. Быстро, надёжно, но требует подготовки.

**Динамические секреты**

Vault генерирует временную учётную запись в базе данных. Приложение получает логин и пароль, которые действуют час. Потом Vault удаляет учётную запись.

Это самый безопасный способ. Даже если пароль украден — через час он бесполезен.

## Секреты в контейнерах

### Проблема

Контейнеры нуждаются в секретах. Но как передать секрет в контейнер, не записывая его в образ?

### Решения

**Kubernetes Secrets**

Секреты хранятся в Kubernetes и монтируются в контейнеры как файлы или переменные окружения. Поды получают только те секреты, которые указаны в их манифесте.

**External Secrets Operator**

Оператор Kubernetes, который синхронизирует секреты из внешних хранилищ (Vault, AWS Secrets Manager) в Kubernetes Secrets. Позволяет централизованно управлять секретами.

**Инициализационные контейнеры**

Контейнер запускается перед основным приложением, получает секреты из Vault, записывает их в общий том. Основное приложение читает секреты из тома.

**Sidecar**

Отдельный контейнер рядом с основным приложением. Sidecar аутентифицируется в Vault, получает секреты и предоставляет их основному приложению через локальный API.

## Утечки секретов

### Как происходят

- **В коде**: разработчик хардкодит пароль и коммитит в Git
- **В логах**: приложение логирует конфигурацию со всеми секретами
- **В ошибках**: сообщение об ошибке содержит URL с паролем
- **В резервных копиях**: бэкап содержит файлы с секретами
- **В Docker-образах**: секреты остаются в слоях образа
- **В переменных окружения**: секреты видны всем процессам

### Обнаружение

**GitLeaks, TruffleHog**

Инструменты сканируют Git-репозиторий на наличие секретов. Ищут паттерны: API-ключи, пароли, токены. Находят даже в истории коммитов.

**Pre-commit hooks**

Хук, который запускается перед каждым коммитом. Проверяет, нет ли секретов в изменениях. Если находит — блокирует коммит.

**Сканирование логов**

Анализ логов на предмет случайных утечек. Поиск паттернов паролей, ключей, токенов.

### Реакция на утечку

1. **Отзыв** — немедленно отозвать (удалить/заблокировать) утечённый секрет
2. **Ротация** — сгенерировать новый секрет
3. **Обновление** — обновить все сервисы, которые использовали старый секрет
4. **Аудит** — проверить, использовался ли утечённый секрет
5. **Предотвращение** — настроить pre-commit hooks, сканирование, обучение

## Архитектурные решения

### Многоуровневая защита

Защита секретов — это несколько слоёв:
- **Pre-commit hooks** — не дать закоммитить секрет
- **Git scanning** — найти уже утечённые секреты
- **Vault** — централизованное хранение
- **Short-lived tokens** — минимизировать время жизни
- **Аудит** — отслеживать использование

### Zero Trust для секретов

Никогда не доверяй. Даже если секрет был безопасен вчера — проверь сегодня. Даже если доступ запрошен известным сервисом — проверь право.

### Минимальные привилегии

Каждый секрет даёт минимально необходимый доступ. Не "полный доступ к облаку", а "только чтение из этого S3-бакета".

### Ответы на чек-лист

1. **Секреты** — это пароли, API-ключи, токены, сертификаты, ключи шифрования. Они дают доступ к системам и данным. Если секреты украдены — злоумышленник получает полный доступ.

2. **Нельзя хранить секреты**: в коде, конфигурационных файлах в Git, переменных окружения на сервере, логах, Docker-образах, комментариях кода.

3. **Vault** — централизованное защищённое хранилище секретов. Хранит секреты в зашифрованном виде, контролирует доступ, логирует обращения, поддерживает динамические секреты и ротацию.

4. **В CI/CD** секреты передаются через: встроенные хранилища секретов CI/CD, OIDC-токены, short-lived credentials из облачных провайдеров, обращение к Vault из pipeline.

5. **Ротация** — регулярная смена секретов. Сокращает окно атаки, если секрет был скомпрометирован. Может быть ручной или автоматической.

6. **Типы секретов**: пароли, API-ключи, токены (access, refresh), сертификаты, ключи шифрования, учётные данные баз данных, SSH-ключи.

7. **Переменные окружения** видны всем процессам на сервере. Любой, кто получит доступ к серверу, увидит все переменные. Они также попадают в логи и дампы памяти.

8. **Обнаружение утечек**: GitLeaks и TruffleHog сканируют репозитории, pre-commit hooks блокируют коммиты, сканирование логов находит случайные утечки.

9. **OIDC** — протокол аутентификации. В CI/CD pipeline получает временный токен от провайдера идентичности. **Short-lived credentials** — временные учётные данные, действующие минуты. Минимизируют риск от компрометации.

10. **В контейнерах**: Kubernetes Secrets, External Secrets Operator, инициализационные контейнеры, sidecar-паттерн. Не хранить секреты в образах, использовать read-only файловые системы.
