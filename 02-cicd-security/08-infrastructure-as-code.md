# Безопасность Infrastructure as Code

## Что такое Infrastructure as Code?

Представь архитектора, который проектирует здание. Он рисует чертежи на бумаге. Рабочие строят по этим чертежам. Если архитектор ошибся — здание построено неправильно. Исправлять дорого и сложно.

Теперь представь, что архитектор создаёт цифровую модель здания. Компьютер проверяет модель на ошибки: достаточно ли несущих стен? Правильно ли расположены выходы? Соответствуют ли нормам?

**Infrastructure as Code (IaC)** — это когда инфраструктура описывается в файлах, а не настраивается вручную. Серверы, сети, базы данных, фаерволы — всё описано в коде.

Вместо того чтобы заходить на сервер и настраивать вручную — администратор пишет файл конфигурации. Инструмент читает файл и создаёт инфраструктуру.

## Зачем нужна безопасность IaC?

### Ситуация 1: Открытый фаервол

Администратор настраивает фаервол вручную. Он открывает порт для тестирования и забывает закрыть.

С IaC: конфигурация фаервола описана в файле. Каждый видит, какие порты открыты. Code review проверяет конфигурацию. Автоматизированные проверки находят открытые порты.

### Ситуация 2: Небезопасная конфигурация

Администратор создаёт базу данных. По умолчанию она доступна из интернета. Администратор забыл ограничить доступ.

С IaC: конфигурация базы данных описана в коде. Проверка находит, что база доступна из интернета, и блокирует деплой.

### Ситуация 3: История изменений

Сервер настроен вручную. Кто-то изменил конфигурацию. Когда? Кто? Зачем? Неизвестно.

С IaC: все изменения в файлах. Git хранит историю. Видно, кто и когда что изменил. Можно откатить изменения.

## Инструменты IaC

| Инструмент | Описание | Применение |
|------------|----------|------------|
| Terraform | Универсальный инструмент | Облака, сети, ресурсы |
| CloudFormation | AWS-native | Только AWS |

| Ansible | Конфигурация серверов | Установка ПО, настройка |

| Kubernetes YAML | Контейнеры | Поды, сервисы, ingress |

## Угрозы IaC

### Секреты в коде

Администратор пишет Terraform-конфигурацию. В ней пароль от базы данных. Коммитит в Git. Теперь пароль в истории Git.

Это самая распространённая ошибка. Секреты должны храниться отдельно, в защищённых хранилищах.

### Небезопасные дефолты

Инструмент IaC создаёт ресурс с дефолтными настройками. По умолчанию открыты все порты. По умолчанию нет шифрования. По умолчанию публичный доступ.

Администратор забыл переопределить дефолты. Ресурс создан небезопасно.

### Отсутствие валидации

Файл конфигурации содержит ошибку. Но инструмент её не проверяет. Ошибка проявляется только в production.

Например, фаервол разрешает доступ из 0.0.0.0/0 вместо конкретной подсети.

### Компрометация state

Terraform хранит state — текущее состояние инфраструктуры. State содержит чувствительные данные: пароли, токены, ключи.

Если state не зашифрован — злоумышленник, получивший доступ к state, получает все секреты.

### Непреднамеренное удаление

Команда terraform destroy удаляет всю инфраструктуру. Если запустить по ошибке — production исчезает.

Нужна защита: подтверждение, блокировка для production, backup state.

## Защита IaC

### Сканирование конфигурации

Инструменты анализируют файлы IaC на наличие уязвимостей:

| Инструмент | Что проверяет |
|------------|--------------|
| Checkov | Terraform, CloudFormation, Kubernetes |
| tfsec | Terraform |
| Terrascan | Terraform, Kubernetes, Docker |
| Semgrep | Мульти (правила для IaC) |
| Bridgecrew | Terraform, CloudFormation |

Проверки находят:
- Открытые порты (0.0.0.0/0)
- Отсутствие шифрования
- Секреты в коде
- Небезопасные дефолты
- Отсутствие логирования
- Неправильные права доступа

### Policy as Code

Политики безопасности описаны в коде. Например: "Все S3-бакеты должны быть приватными". "Все базы данных должны иметь шифрование". "Никаких открытых портов 22 и 3389".

Инструменты проверяют конфигурацию на соответствие политикам. Если нарушение — деплой блокируется.

### State management

State Terraform должен быть:
- Зашифрован при хранении
- Доступен только авторизованным пользователям
- Версионирован
- Заблокирован при изменении (prevent concurrent modification)

Хранение state в S3 с включённым шифрованием и блокировкой — стандартная практика.

### Secrets management

Секреты не хранятся в коде. Вместо этого:
- Terraform обращается к Vault за секретами
- Используются data sources для получения секретов
- Секреты передаются через переменные окружения
- Используются специализированные провайдеры секретов

### Code review для IaC

Изменения инфраструктуры проходят review. Как и код приложения, конфигурация инфраструктуры требует одобрения.

Review проверяет:
- Небезопасные изменения
- Соответствие политикам
- Оптимальность ресурсов
- Правильность конфигурации

### Immutable infrastructure

Инфраструктура неизменяема. Вместо изменения существующего сервера — создаётся новый. Старый уничтожается.

Это предотвращает "drift" — постепенное отклонение конфигурации от желаемого состояния.

## Безопасность облачных ресурсов

### S3-бакеты

Частая ошибка: S3-бакет случайно сделан публичным. Все данные доступны из интернета.

IaC проверки:
- Бакет должен быть приватным
- Включено шифрование
- Включено логирование доступа
- Включено versioning
- Настроен lifecycle policy

### Базы данных

База данных должна быть:
- Недоступна из интернета
- Зашифрована
- С включённым аудитом
- С резервным копированием
- С ограниченным доступом

### Фаерволы

Правила фаервола должны:
- Разрешать только необходимые порты
- Ограничивать доступ по IP
- Запрещать default allow
- Иметь явный deny all в конце

### IAM

Права доступа должны:
- Следовать принципу минимальных привилегий
- Не иметь wildcard permissions
- Требовать MFA для критичных операций
- Регулярно пересматриваться

## Архитектурные решения

### GitOps

Вся инфраструктура описана в Git. Git — единый источник правды. Изменения только через Git. Никаких ручных изменений.

Преимущества:
- История изменений
- Code review
- Откат изменений
- Автоматизация

### Policy as Code

Политики безопасности описаны в коде и применяются автоматически. Не нужно проверять вручную — инструмент проверяет сам.

Примеры политик:
- Все ресурсы должны иметь теги
- Все S3-бакеты приватные
- Все базы данных зашифрованы
- Нет открытых портов SSH

### Defense in Depth

Защита инфраструктуры — многослойная:
- Network: VPC, subnets, security groups, NACLs
- Application: фаерволы, WAF
- Data: шифрование, backup
- Access: IAM, MFA, роли
- Monitoring: логи, алерты

### Ответы на чек-лист

1. **Infrastructure as Code** — подход, при котором инфраструктура описывается в файлах, а не настраивается вручную. Инструменты (Terraform, Ansible) читают файлы и создают инфраструктуру.

2. **Секреты в коде** попадают в Git. Git хранит историю навсегда. Даже если удалить — секрет остаётся в истории. Любой с доступом к репозиторию видит секреты.

3. **State Terraform** — файл, содержащий текущее состояние инфраструктуры. Содержит чувствительные данные: пароли, токены. Должен быть зашифрован и защищён.

4. **Инструменты сканирования IaC**: Checkov, tfsec, Terrascan, Semgrep, Bridgecrew. Проверяют Terraform, CloudFormation, Kubernetes YAML на уязвимости.

5. **Policy as Code** — политики безопасности описаны в коде. Инструменты автоматически проверяют конфигурацию на соответствие политикам. Нарушения блокируют деплой.

6. **Code review для инфраструктуры** важен, потому что изменения инфраструктуры могут быть опасны. Открытый порт, публичный бакет, отсутствие шифрования — всё это находится на review.

7. **Immutable infrastructure** — инфраструктура неизменяема. Вместо изменения сервера — создаётся новый. Старый уничтожается. Предотвращает drift и упрощает откат.

8. **Защита state**: шифрование при хранении, доступ только авторизованным, версионирование, блокировка при изменении, хранение в защищённом backend (S3 с шифрованием).

9. **Проверки для S3**: бакет приватный, включено шифрование, включено логирование, включен versioning, настроен lifecycle policy, запрещён публичный доступ.

10. **GitOps** — подход, при котором Git является единым источником правды для инфраструктуры. Все изменения только через Git. Автоматическая синхронизация желаемого и фактического состояния.
