# Безопасность контейнеров в CI/CD

## Что такое контейнер?

Представь апарт-отель. В одном здании — множество квартир. У каждой квартиры своя дверь, свои стены, своя мебель. Но все квартиры делят одно здание: лифты, лестницы, крышу, фундамент.

Жилец одной квартиры видит только свою квартиру. Он не видит соседей, не слышит их, не может зайти к ним. Но все живут в одном здании на одном фундаменте.

**Контейнер** — это изолированное окружение для запуска приложения. У контейнера своя файловая система, свои процессы, своя сеть. Но он делит ядро операционной системы с хостом и другими контейнерами.

Контейнер легче виртуальной машины. VM — это отдельный компьютер со своим ядром. Контейнер — это изолированный процесс на одном ядре.

## Зачем нужна безопасность контейнеров?

### Ситуация 1: Уязвимость в образе

HR-платформа запускается в контейнере. Образ контейнера основан на устаревшем Debian. В Debian есть известная уязвимость в библиотеке SSL.

Злоумышленник атакует контейнер через эту уязвимость. Получает доступ к контейнеру. Но изоляция контейнера не позволяет выйти за пределы.

Если бы контейнер запущен от root без user namespace — злоумышленник мог бы получить root на хосте.

### Ситуация 2: Контейнер с привилегиями

Контейнер запущен с полными правами root. Злоумышленник взламывает приложение в контейнере. Теперь у него есть root внутри контейнера.

Если нет user namespace — root в контейнере = root на хосте. Злоумышленник выходит из контейнера и получает контроль над всей системой.

### Ситуация 3: Утечка данных между контейнерами

Два контейнера делят одну сеть. Один — веб-сервер, другой — база данных. Злоумышленник взламывает веб-сервер и начинает сканировать сеть.

Без сетевой политики — контейнер может обращаться к любому другому контейнеру. Злоумышленник находит базу данных и крадёт данные.

## Уровни безопасности контейнеров

### Уровень 1: Базовый образ

Контейнер строится из базового образа. Если базовый образ уязвим — контейнер уязвим.

| Проблема | Решение |
|----------|---------|
| Устаревший образ | Регулярные обновления |
| Известные CVE | Сканирование образов |
| Лишние пакеты | Минимальный образ (distroless, alpine) |
| Секреты в образе | Multi-stage builds, не хранить секреты |

Минимальный образ содержит только необходимое. Нет shell, нет пакетного менеджера, нет лишних утилит. Поверхность атаки минимальна.

### Уровень 2: Сборка образа

| Проблема | Решение |
|----------|---------|
| Секреты в Dockerfile | BuildKit secrets, не хранить в образе |
| Установка пакетов из интернета | Приватный registry, pinning версий |
| Непроверенные зависимости | SCA, проверка лицензий |
| Изменение во время сборки | Immutable builds, подпись |

Multi-stage builds позволяют собирать приложение в одном контейнере, а запускать в другом. Инструменты сборки не попадают в финальный образ.

### Уровень 3: Запуск контейнера

| Проблема | Решение |
|----------|---------|
| Запуск от root | User namespace, не-root пользователь |
| Избыточные capabilities | Drop all, add only needed |
| Отсутствие seccomp | Применение seccomp профиля |
| Отсутствие AppArmor/SELinux | Применение профилей |
| Read-write файловая система | Read-only rootfs, tmpfs для записи |

Контейнер должен запускаться от не-root пользователя. Capabilities должны быть минимальны. Файловая система — read-only, если возможно.

### Уровень 4: Оркестрация

| Проблема | Решение |
|----------|---------|
| Общая сеть | Network policies |
| Общее хранилище | Volume policies, encryption |
| Привилегированные контейнеры | Pod Security Standards |
| Отсутствие изоляции | Namespaces, cgroups |

Kubernetes Network Policies контролируют, какие контейнеры могут общаться. По умолчанию — никакие. Явно разрешаются необходимые соединения.

### Уровень 5: Мониторинг

| Проблема | Решение |
|----------|---------|
| Неизвестно, что происходит | Runtime monitoring |
| Аномальное поведение | Anomaly detection |
| Утечка данных | DLP, network monitoring |
| Вторжение | IDS, Falco |

Falco — инструмент мониторинга runtime для Kubernetes. Он обнаруживает подозрительную активность: запуск shell в контейнере, выход за пределы, обращение к /etc/shadow.

## Безопасность образа

### Минимальный образ

Чем меньше в образе — тем меньше поверхность атаки. Идеальный образ:
- Только приложение и его зависимости
- Нет shell (bash, sh)
- Нет пакетного менеджера (apt, yum)
- Нет компилятора
- Нет SSH
- Нет curl/wget

Distroless-образы от Google содержат только приложение и библиотеки времени выполнения. Нет ничего лишнего.

### Сканирование образа

Перед использованием образ должен быть просканирован:

| Инструмент | Что находит |
|------------|------------|
| Trivy | CVE в пакетах, секреты |
| Clair | CVE в слоях образа |
| Anchore | Политики, CVE, secrets |
| Snyk | CVE, лицензии |

Сканирование происходит:
- При сборке образа
- Перед деплоем
- Регулярно (новые CVE появляются)

### Подпись образа

Образ должен быть подписан криптографически. Это гарантирует, что образ создан доверенным сборщиком и не изменён.

Cosign подписывает образы через Sigstore. Проверка подписи перед запуском гарантирует целостность.

### SBOM для образа

Software Bill of Materials для образа содержит:
- Базовый образ
- Все установленные пакеты
- Версии пакетов
- Хеши слоёв
- Лицензии

При обнаружении уязвимости в пакете — по SBOM находятся все образы, которые его используют.

## Безопасность запуска

### User namespace

User namespace позволяет контейнеру быть root внутри, но иметь обычный UID на хосте. Даже если контейнер взломан — на хосте нет привилегий.

Это критически важно. Без user namespace root в контейнере = root на хосте.

### Capabilities

Контейнеру должны быть даны минимальные capabilities:
```
Drop ALL capabilities
Add only needed (NET_BIND_SERVICE, CHOWN, SETGID, SETUID)
```

Большинству контейнеров не нужны привилегии. Они могут работать с drop all.

### Seccomp

Seccomp профиль блокирует опасные системные вызовы. Docker по умолчанию использует профиль, блокирующий ~44 вызова.

Можно создать кастомный профиль для конкретного приложения.

### AppArmor/SELinux

Профили AppArmor или SELinux ограничивают доступ контейнера к файлам хоста. Даже если контейнер выйдет за пределы своей изоляции — профиль остановит его.

### Read-only rootfs

Корневая файловая система контейнера — только для чтения. Приложение не может модифицировать системные файлы.

Если нужна запись — используется tmpfs (временная файловая система в памяти). После перезапуска контейнера — всё исчезает.

### Security context в Kubernetes

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  capabilities:
    drop:
      - ALL
  seccompProfile:
    type: RuntimeDefault
```

Этот security context:
- Запрещает root
- Устанавливает UID
- Делает rootfs read-only
- Запрещает повышение привилегий
- Убирает все capabilities
- Использует seccomp по умолчанию

## Сетевая безопасность

### Network Policies

Kubernetes Network Policies определяют, какие поды могут общаться. По умолчанию — никакие.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: frontend
      ports:
        - protocol: TCP
          port: 8080
```

Эта политика разрешает frontend подам обращаться к backend подам на порт 8080. Всё остальное запрещено.

### Service Mesh

Service mesh (Istio, Linkerd) добавляет слой безопасности между сервисами:
- mTLS между всеми сервисами
- Авторизация на уровне сервисов
- Rate limiting
- Circuit breaking

### Ingress security

Входящий трафик проходит через Ingress Controller. Защита:
- TLS termination
- Rate limiting
- WAF (Web Application Firewall)
- DDoS protection
- Geo-blocking

## Хранение данных

### Secrets

Секреты в Kubernetes хранятся в объектах Secret. Но они кодируются base64, а не шифруются. Любой с доступом к etcd может прочитать секреты.

Решения:
- Encryption at rest для etcd
- External Secrets Operator (интеграция с Vault)
- Sealed Secrets (шифрование secrets перед коммитом в Git)

### Volumes

Данные хранятся в volumes. Защита:
- Шифрование volume (в облаке или через CSI)
- Backup и snapshot
- Ограничение доступа к volume

### Stateful vs Stateless

Контейнеры должны быть stateless. Данные хранятся вне контейнера (в volumes, базах данных). Это позволяет легко заменять контейнеры без потери данных.

## Архитектурные решения

### Defense in Depth

Защита контейнеров — многослойная:
- Минимальный образ (меньше уязвимостей)
- Сканирование (находим CVE)
- Подпись (гарантируем целостность)
- User namespace (root внутри ≠ root снаружи)
- Capabilities (минимум привилегий)
- Seccomp (фильтр вызовов)
- AppArmor/SELinux (контроль файлов)
- Read-only rootfs (нельзя модифицировать)
- Network Policies (сетевая изоляция)
- Runtime monitoring (обнаружение аномалий)

### Zero Trust

Не доверяй ни одному контейнеру. Каждый контейнер — потенциально взломан. Изолируй всё.

### Pod Security Standards

Kubernetes определяет три уровня безопасности для подов:

| Уровень | Описание | Применение |
|---------|----------|------------|
| Privileged | Без ограничений | Системные сервисы |
| Baseline | Минимальные ограничения | Стандартные приложения |
| Restricted | Жёсткие ограничения | Критичные приложения |

Restricted:
- Только non-root
- Только read-only rootfs
- Только drop all capabilities
- Без privilege escalation
- Без hostPath volumes

### Ответы на чек-лист

1. **Контейнер** — изолированное окружение для запуска приложения. У контейнера своя файловая система, процессы, сеть. Но он делит ядро с хостом.

2. **Контейнер легче VM**, потому что не имеет своего ядра. VM — полноценный компьютер со своим ядром. Контейнер — изолированный процесс на общем ядре. Запуск контейнера — секунды, VM — минуты.

3. **User namespace** позволяет контейнеру быть root внутри (UID 0), но иметь обычный UID на хосте. Даже если контейнер взломан — на хосте нет привилегий. Root внутри ≠ root снаружи.

4. **Минимальный образ** содержит только необходимое. Нет shell, пакетного менеджера, компилятора. Чем меньше в образе — тем меньше уязвимостей и поверхность атаки.

5. **Сканирование образа** выполняется инструментами: Trivy, Clair, Anchore, Snyk. Они находят CVE в пакетах, секреты, неправильные конфигурации. Сканирование при сборке, перед деплоем, регулярно.

6. **Seccomp** фильтрует системные вызовы контейнера. Docker по умолчанию блокирует ~44 опасных вызова. Контейнер не может загружать модули ядра, изменять системное время, монтировать ФС.

7. **Read-only rootfs** делает корневую файловую систему только для чтения. Контейнер не может модифицировать системные файлы. Если нужна запись — используется tmpfs (временная ФС в памяти).

8. **Network Policies** определяют, какие поды могут общаться. По умолчанию — никакие. Явно разрешаются необходимые соединения. Это сетевая изоляция контейнеров.

9. **Pod Security Standards** — три уровня безопасности Kubernetes: Privileged (без ограничений), Baseline (минимальные ограничения), Restricted (жёсткие ограничения: non-root, read-only, drop all capabilities).

10. **Secrets в Kubernetes**: хранятся в объектах Secret (base64). Защита: encryption at rest для etcd, External Secrets Operator (интеграция с Vault), Sealed Secrets (шифрование перед коммитом в Git).
