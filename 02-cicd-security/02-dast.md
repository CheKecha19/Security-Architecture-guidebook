# DAST: Динамический анализ приложений

## Что такое DAST?

Представь, что ты купил новый автомобиль. Перед тем как выехать на трассу, ты проверяешь всё статически: листаешь инструкцию, смотришь чертежи, изучаешь спецификации. Это важно, но недостаточно.

А потом ты заводишь мотор, выезжаешь на дорогу и начинаешь ездить. И только в движении обнаруживаешь: коробка передач туговато переключается, подвеска стучит на кочках, при торможении есть вибрация. Эти проблемы не видны в инструкции — они проявляются только в работе.

**DAST** — это анализ работающего приложения. DAST не читает код. Он посылает запросы к приложению и смотрит, как оно отвечает. Он ищет уязвимости, которые проявляются только при реальном использовании.

SAST смотрит на "чертежи" приложения. DAST смотрит на "автомобиль в движении".

## Зачем нужен DAST?

### Ситуация 1: Веб-приложение HR-платформы

HR-платформа принимает резюме через веб-форму. Разработчики проверили код — он выглядит безопасно.

Но DAST посылает специальные запросы к форме. Вводит в поле "Имя" вредоносный скрипт. И видит, что приложение выводит этот скрипт на страницу без очистки. Это XSS — уязвимость, которую SAST мог пропустить, если анализ был недостаточно глубоким.

### Ситуация 2: API интеграции

HR-платформа предоставляет API для партнёров. API принимает данные в формате JSON.

DAST посылает искажённые JSON-запросы. Слишком большие значения, отсутствующие поля, специальные символы. И обнаруживает, что API падает на некоторых запросах. Это отказ в обслуживании — потенциальная уязвимость.

### Ситуация 3: Аутентификация

Система аутентификации проверяет логин и пароль. DAST пытается обойти аутентификацию разными способами: SQL-инъекция в поле логина, подбор паролей, манипуляция с сессиями.

DAST обнаруживает, что сессионные токены предсказуемы. Зная один токен, можно угадать другой. Это даёт доступ к чужим сессиям.

## Как работает DAST?

### Сканирование

DAST начинает с изучения приложения. Он "ползает" по страницам, кликает ссылки, заполняет формы, изучает API endpoints. Он строит карту приложения — какие страницы есть, какие параметры принимают, какие формы отправляют.

Это как разведчик, который изучает местность перед операцией.

### Фаззинг

Фаззинг — это посылка искажённых данных. DAST берёт нормальный запрос и начинает его видоизменять:
- Убирает обязательные параметры
- Добавляет лишние параметры
- Вставляет специальные символы
- Увеличивает значения до огромных
- Меняет типы данных

Цель — вызвать неожиданное поведение приложения.

### Проверка уязвимостей

DAST проверяет приложение на известные классы уязвимостей:

| Уязвимость | Как проверяет |
|------------|--------------|
| SQL-инъекция | Вставляет SQL-код в параметры |
| XSS | Вставляет скрипты в формы |
| CSRF | Проверяет защиту от подделки запросов |
| IDOR | Меняет ID в запросах, проверяет доступ |
| Path Traversal | Использует ../ для выхода за пределы |
| Command Injection | Вставляет системные команды |
| LDAP Injection | Вставляет LDAP-запросы |
| XXE | Подставляет внешние XML-сущности |

### Анализ ответов

DAST анализирует ответы приложения. Код ответа, заголовки, тело, время выполнения. Необычное поведение — признак уязвимости.

Например, если обычный запрос выполняется за 100 мс, а запрос с SQL-инъекцией — за 5000 мс. Это может означать, что база данных выполнила сложный запрос. Признак SQL-инъекции.

## Типы DAST

### Чёрный ящик

DAST знает только URL приложения. Он не знает код, архитектуру, используемые технологии. Он исследует приложение как злоумышленник — с нуля.

Преимущество: максимально приближено к реальной атаке.
Недостаток: может пропустить внутренние страницы, требующие аутентификации.

### Серый ящик

DAST имеет учётные данные для входа. Он может авторизоваться и сканировать внутренние страницы.

Преимущество: покрывает больше функциональности.
Недостаток: требует подготовки тестовых аккаунтов.

### Пассивный DAST

DAST наблюдает за трафиком, но не посылает активных запросов. Он анализирует, как приложение общается с браузером.

Преимущество: безопасен для production, не нагружает систему.
Недостаток: находит только явные проблемы в ответах.

### Активный DAST

DAST активно посылает запросы, изменяет данные, пытается эксплуатировать уязвимости.

Преимущество: находит больше уязвимостей.
Недостаток: может повредить данные или нарушить работу системы.

## DAST в CI/CD

### Staging environment

DAST запускается на staging — копии production. Это позволяет безопасно тестировать: даже если DAST что-то сломает — production не пострадает.

### После деплоя

Конвейер: код → сборка → unit-тесты → SAST → деплой на staging → DAST → ручное тестирование → production.

DAST находит уязвимости, которые проявляются только в работающем приложении. Он дополняет SAST.

### Регрессионное тестирование

DAST запускается регулярно — например, каждую ночь. Он проверяет, что новые изменения не добавили уязвимостей.

Если DAST нашёл новую уязвимость — команда получает уведомление.

## Проблемы DAST

### Не видит код

DAST анализирует только то, что видит снаружи. Он не знает, как устроено приложение внутри. Он может пропустить уязвимости, которые не проявляются снаружи.

### Требует работающего приложения

DAST нельзя запустить на этапе разработки. Нужно собрать и задеплоить приложение. Это позднее обнаружение.

### Ложные срабатывания

DAST может принять нормальное поведение за уязвимость. Или наоборот — не заметить уязвимость, потому что она проявляется только в определённых условиях.

### Покрытие

DAST может пропустить страницы, которые он не нашёл. Скрытые API, админские панели, редкие сценарии — всё это может остаться непроверенным.

### Безопасность сканирования

Активный DAST может повредить данные. Он посылает вредоносные запросы. Если сканировать production — можно удалить реальные данные.

## DAST и другие методы

### DAST vs SAST

| Критерий | DAST | SAST |
|----------|------|------|
| Когда | После деплоя | До запуска |
| Что видит | Внешнее поведение | Внутренний код |
| Покрытие | Выполненные пути | Весь код |
| Тип уязвимостей | Эксплуатируемые | Все потенциальные |
| Сложность | Ниже | Выше |
| Ресурсы | Нужен сервер | Нужен только код |

### DAST vs IAST

IAST работает внутри приложения, агент отслеживает выполнение. IAST видит и код, и поведение. Но требует модификации приложения.

DAST работает снаружи. Не требует модификации. Но видит только внешнее поведение.

### DAST vs Penetration Testing

Penetration testing — это ручной анализ экспертом. DAST — автоматизированный инструмент.

Pen testing находит сложные логические уязвимости. DAST находит известные классы уязвимостей.

Они дополняют друг друга.

## Архитектурные решения

### Defense in Depth

DAST — один из слоёв защиты. Он ловит уязвимости на уровне работающего приложения.

Полная защита:
- SAST — проверка кода
- DAST — проверка работающего приложения
- IAST — внутренний мониторинг
- Pen testing — ручной аудит
- WAF — защита на уровне сети

### Shift-right

Если SAST — это shift-left (раннее обнаружение), то DAST — shift-right. Он проверяет приложение в конце конвейера, перед релизом.

Оба подхода важны. Раннее обнаружение дешевле. Позднее обнаружение точнее.

### Автоматизация

DAST должен быть автоматизирован. Ручной запуск DAST — это долго и ненадёжно. Автоматический DAST в CI/CD гарантирует регулярную проверку.

## Чек-лист понимания

- [ ] Чем DAST отличается от SAST?
- [ ] Как DAST находит SQL-инъекции?
- [ ] Что такое фаззинг?
- [ ] В чём разница между чёрным и серым ящиком?
- [ ] Когда DAST запускается в CI/CD?
- [ ] Почему DAST не видит весь код?
- [ ] Какие уязвимости DAST не находит?
- [ ] В чём разница между пассивным и активным DAST?
- [ ] Как DAST дополняет penetration testing?
- [ ] Почему DAST опасен для production?

### Ответы на чек-лист

1. **DAST** анализирует работающее приложение снаружи. **SAST** анализирует исходный код. DAST видит реальное поведение, но только внешнее. SAST видит весь код, но может ошибаться.

2. **DAST** посылает запросы с SQL-кодом в параметрах. Если приложение возвращает необычный ответ, выполняет запрос долго, или возвращает данные из базы — это признак SQL-инъекции.

3. **Фаззинг** — посылка искажённых данных. DAST берёт нормальный запрос и видоизменяет его: убирает поля, добавляет символы, меняет типы. Цель — вызвать неожиданное поведение.

4. **Чёрный ящик** знает только URL приложения, не знает внутреннего устройства. **Серый ящик** имеет учётные данные и может сканировать внутренние страницы.

5. **DAST** запускается после деплоя на staging или тестовое окружение. Он проверяет работающее приложение перед релизом на production.

6. **DAST** не видит код, потому что работает снаружи. Он посылает запросы и анализирует ответы. Внутреннее устройство приложения ему неизвестно.

7. **DAST** не находит: уязвимости, не проявляющиеся снаружи; логические ошибки; проблемы в коде, которые не влияют на внешнее поведение; уязвимости в частях, которые DAST не обнаружил.

8. **Пассивный DAST** только наблюдает за трафиком, не посылая активных запросов. **Активный DAST** посылает вредоносные запросы и пытается эксплуатировать уязвимости.

9. **DAST** автоматизирует поиск известных уязвимостей. **Penetration testing** находит сложные логические уязвимости, требующие человеческого интеллекта. Они дополняют друг друга.

10. **DAST опасен для production**, потому что активный DAST посылает вредоносные запросы. Он может удалить данные, изменить состояние системы, или вызвать отказ в обслуживании. Поэтому DAST запускают на staging.
