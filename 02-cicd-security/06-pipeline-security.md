# Безопасность CI/CD pipeline

## Что такое CI/CD pipeline?

Представь конвейер на заводе. Сырьё входит с одного конца, проходит через станции обработки, и готовый продукт выходит с другого. На каждой станции — своя операция: резка, сборка, покраска, упаковка, проверка качества.

**CI/CD pipeline** — это цифровой конвейер для программного обеспечения. Код входит с одного конца, проходит через этапы сборки, тестирования, проверки безопасности, и готовое приложение выходит с другого.

| Этап | Что происходит |
|------|---------------|
| Source | Разработчик пишет код, коммитит в Git |
| Build | Код компилируется, собираются артефакты |
| Test | Unit-тесты, интеграционные тесты |
| Security | SAST, DAST, SCA, secrets scanning |
| Deploy | Артефакты деплоятся на сервер |
| Monitor | Мониторинг работы приложения |

Каждый этап — это станция на конвейере. Если на какой-то станции проблема — конвейер останавливается.

## Зачем защищать pipeline?

### Ситуация 1: Взломанный runner

CI/CD runner — это машина, которая выполняет pipeline. Злоумышленник получает доступ к runner.

Теперь он может:
- Изменить код перед сборкой, добавив бэкдор
- Заменить артефакты после сборки
- Похитить секреты из переменных окружения
- Использовать runner для атак на другие системы
- Изменить pipeline, чтобы вставлять бэкдоры во все сборки

### Ситация 2: Вредоносный коммит

Злоумышленник коммитит код, который выглядит безопасно. Но в нём скрыт бэкдор. Pipeline собирает и деплоит этот код.

Если pipeline не проверяет безопасность — вредоносный код попадает в production.

### Ситуация 3: Утечка секретов

Pipeline использует API-ключ для деплоя. Ключ хранится в переменных окружения. Злоумышленник, получивший доступ к логам pipeline, видит этот ключ.

Теперь у него есть доступ к production.

## Угрозы pipeline

### Poisoned pipeline

Злоумышленник изменяет pipeline. Добавляет шаг, который:
- Отправляет секреты на внешний сервер
- Загружает вредоносный код
- Отключает проверки безопасности
- Изменяет артефакты после сборки

### Небезопасные dependencies

Pipeline устанавливает зависимости из интернета. Если пакетный менеджер использует HTTP вместо HTTPS — злоумышленник может подменить пакет.

### Утечка логов

Pipeline логирует всё, что происходит. В логи могут попасть:
- Секреты из переменных окружения
- Пароли баз данных
- Токены доступа
- Ключи шифрования

Если логи публичные — секреты утекают.

### Небезопасные permissions

Pipeline имеет слишком широкие права. Он может:
- Деплоить на production из любой ветки
- Читать все секреты
- Изменять конфигурацию
- Удалять данные

### Компрометация инфраструктуры

Злоумышленник взламывает:
- Git-репозиторий — изменяет код
- Container registry — подменяет образы
- Artifact storage — заменяет артефакты
- Deployment server — деплоит вредоносное

## Защита pipeline

### Минимальные права

Pipeline должен иметь только необходимые права:
- Чтение кода из нужных веток
- Запись артефактов в конкретный bucket
- Деплой только на staging
- Нет доступа к production секретам

Если pipeline скомпрометирован — ущерб минимален.

### Segregation of duties

Разделение обязанностей. Тот, кто пишет код — не может деплоить. Тот, кто деплоит — не может изменять pipeline. Тот, кто одобряет — не пишет код.

| Роль | Что может | Что не может |
|------|-----------|--------------|
| Разработчик | Писать код, создавать PR | Деплоить на production, изменять pipeline |
| DevOps | Настраивать pipeline | Писать код приложения |
| Security | Настраивать проверки | Деплоить без одобрения |
| Администратор | Одобрять деплои | Писать код |

### Защита секретов

- Хранить секреты в защищённых хранилищах, не в переменных окружения
- Использовать short-lived credentials вместо долгоживущих ключей
- Логировать доступ к секретам
- Ротировать секреты регулярно
- Использовать OIDC вместо статических ключей

### Проверка pipeline as code

Pipeline описывается в файлах (GitHub Actions workflows, GitLab CI config). Эти файлы должны проходить review.

Любое изменение pipeline должно быть одобрено двумя людьми.

### Immutable infrastructure

Серверы, на которых работает pipeline — одноразовые. После каждого запуска они уничтожаются и создаются заново. Если сервер был скомпрометирован — следующий запуск начнётся с чистого листа.

### Сканирование pipeline

Инструменты анализируют конфигурацию pipeline на наличие уязвимостей:
- Небезопасные permissions
- Секреты в конфигурации
- Устаревшие действия
- Подозрительные шаги

### Подпись pipeline

Pipeline подписывается криптографически. Если кто-то изменяет pipeline — подпись не совпадает. Система отклоняет изменённый pipeline.

### Audit logging

Все действия pipeline логируются:
- Кто запустил pipeline
- Какой код был использован
- Какие секреты были запрошены
- Какие артефакты созданы
- Куда произошёл деплой

Логи неизменяемы. Если произошёл инцидент — можно восстановить картину.

## Pipeline as Code

### Преимущества

Pipeline описывается в коде и хранится в Git. Это даёт:
- Версионирование — история изменений pipeline
- Code review — изменения одобряются коллегами
- Reproducibility — pipeline можно воспроизвести
- Audit — видно, кто и когда что изменил

### Риски

Если злоумышленник получает доступ к репозиторию — он может изменить pipeline. Добавить вредоносный шаг, отключить проверки безопасности.

### Защита

- Защита ветки с pipeline (main/master) — изменения только через PR
- Обязательный review для изменений pipeline
- Подпись коммитов — подтверждение авторства
- SAST для pipeline — поиск уязвимостей в конфигурации

## Архитектурные решения

### Defense in Depth

Защита pipeline — многослойная:
- Git protection — защита от изменения кода
- Runner isolation — изоляция окружения сборки
- Secret management — защита секретов
- Artifact signing — подпись артефактов
- Deployment gates — ворота перед деплоем
- Monitoring — мониторинг pipeline

### Zero Trust

Не доверяй ни одному компоненту pipeline. Проверяй каждый шаг. Код может быть взломан. Runner может быть скомпрометирован. Секреты могут утечь.

### GitOps

Все изменения — через Git. Никаких ручных изменений. Pipeline автоматически применяет изменения из Git. Это создаёт единый источник правды.

- [ ] Как работает разделение обязанностей в pipeline?
- [ ] Почему важен audit logging?
- [ ] Как защитить pipeline as code?
- [ ] Что такое deployment gate?
- [ ] Почему runner должен быть изолирован?
- [ ] Как проверить конфигурацию pipeline?

### Ответы на чек-лист

1. **Poisoned pipeline** — pipeline, который был изменён злоумышленником. Он добавляет шаги, которые воруют секреты, загружают вредоносный код, отключают проверки безопасности.

2. **Минимальные права** сокращают ущерб при компрометации. Если pipeline скомпрометирован — злоумышленник получает только те права, которые были у pipeline. Не полный доступ, а минимум.

3. **Защита секретов**: хранить в защищённых хранилищах (Vault), использовать short-lived credentials, OIDC вместо статических ключей, логировать доступ, ротировать регулярно.

4. **Immutable infrastructure** — серверы одноразовые. После каждого запуска уничтожаются и создаются заново. Если сервер был скомпрометирован — следующий запуск начнётся чисто.

5. **Разделение обязанностей**: разработчик пишет код, но не деплоит. DevOps настраивает pipeline, но не пишет код. Security настраивает проверки. Администратор одобряет деплои. Никто не может всё.

6. **Audit logging** позволяет восстановить картину инцидента. Кто запустил pipeline, какой код использовался, какие секреты запрошены, куда деплоилось. Логи неизменяемы.

7. **Pipeline as code** хранится в Git. Защита: защита ветки main, обязательный review, подпись коммитов, SAST для конфигурации pipeline.

8. **Deployment gate** — ворота перед деплоем. Pipeline останавливается и ждёт одобрения. Только после ручного или автоматического одобрения происходит деплой.

9. **Runner должен быть изолирован**, потому что на нём выполняется произвольный код. Если runner скомпрометирован — злоумышленник получает контроль над сборкой.

10. **Проверка конфигурации pipeline**: инструменты анализируют workflow файлы на небезопасные permissions, секреты в конфигурации, устаревшие действия, подозрительные шаги.
