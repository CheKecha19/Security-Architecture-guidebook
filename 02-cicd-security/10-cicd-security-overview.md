# Обзор безопасности CI/CD

## Введение

CI/CD — это конвейер, который превращает код в работающее приложение. Код разработчика проходит через этапы сборки, тестирования, проверок и деплоя. Каждый этап — это точка, где безопасность может быть нарушена.

Безопасность CI/CD — это не отдельный продукт или инструмент. Это культура, процессы и технологии, встроенные в каждый этап конвейера.

## Архитектура безопасности CI/CD

### Многоуровневая защита

Безопасность CI/CD строится по принципу многоуровневой защиты. Каждый уровень перекрывает слабые места другого.

| Уровень | Что защищает | Инструменты |
|---------|--------------|-------------|
| **Код** | Исходный код от вредоносных изменений | Git, code review, подпись |
| **Зависимости** | Сторонние библиотеки от уязвимостей | SCA, SBOM, приватный registry |
| **Сборка** | Процесс сборки от вмешательства | Изоляция runner, immutable build |
| **Артефакты** | Готовые бинарники от подмены | Подпись, Cosign, Sigstore |
| **Инфраструктура** | Конфигурацию IaC от ошибок | Checkov, tfsec, Policy as Code |
| **Деплой** | Процесс деплоя от неавторизованных изменений | Security gates, approve, audit |
| **Runtime** | Работающее приложение | DAST, мониторинг, Falco |

### Каждый уровень в деталях

**Уровень кода**

Разработчик пишет код. Pre-commit hooks проверяют, что в коде нет секретов. Code review проверяет, что код не содержит вредоносных изменений. Коммиты подписываются — гарантируется авторство.

SAST анализирует код на наличие уязвимостей. Он находит SQL-инъекции, XSS, небезопасное хранение данных. Всё это — до того, как код попадёт в основную ветку.

**Уровень зависимостей**

Современное приложение на 80% состоит из сторонних библиотек. SCA проверяет эти библиотеки на известные уязвимости.

SBOM — список всех компонентов приложения. При обнаружении уязвимости в библиотеке — по SBOM быстро находят все затронутые проекты.

Приватный registry — хранилище проверенных библиотек. Никакие непроверенные библиотеки не попадают в проект.

**Уровень сборки**

CI/CD runner должен быть изолирован. Если runner скомпрометирован — все сборки скомпрометированы. Immutable infrastructure — runner создаётся заново для каждой сборки.

Билд должен быть воспроизводим. Если собрать код дважды — результат должен быть идентичен. Это позволяет проверить, что в сборку не был добавлен вредоносный код.

**Уровень артефактов**

Каждый артефакт (бинарник, Docker-образ) подписывается. Подпись гарантирует, что артефакт создан доверенным сборщиком и не изменён.

Проверка подписи перед деплоем гарантирует целостность.

**Уровень инфраструктуры**

Infrastructure as Code — инфраструктура описана в файлах. Checkov, tfsec проверяют эти файлы на небезопасные конфигурации. Policy as Code — политики безопасности проверяются автоматически.

**Уровень деплоя**

Security gates — ворота перед деплоем. Pipeline останавливается и ждёт одобрения. Только после прохождения всех проверок — деплой на production.

**Уровень runtime**

DAST проверяет работающее приложение на уязвимости. Мониторинг runtime (Falco) обнаруживает аномальную активность.

## Взаимодействие механизмов

### Сценарий: Разработка новой функции

Разработчик создаёт ветку, пишет код, добавляет новую библиотеку.

**Pre-commit**: hooks проверяют, что в коде нет секретов. Библиотека не содержит известных уязвимостей.

**Code review**: коллега проверяет код. Смотрит, не создаёт ли изменение уязвимостей.

**CI/CD pipeline**: начинается автоматическая проверка.

**SAST**: анализирует весь код. Находит SQL-инъекцию — разработчик исправляет.

**SCA**: проверяет новую библиотеку. В ней есть уязвимость CVE-2024-XXXX. Разработчик заменяет на безопасную версию.

**Сборка**: код компилируется. Создаётся Docker-образ. Образ подписывается.

**Сканирование образа**: Trivy находит уязвимость в базовом образе. Команда обновляет базовый образ.

**Деплой на staging**: приложение деплоится на тестовое окружение.

**DAST**: сканирование работающего приложения. Находит XSS. Разработчик исправляет.

**Security gate**: все проверки пройдены. Деплой одобрен.

**Деплой на production**: подписанный и проверенный артефакт деплоится в production.

### Сценарий: Обнаружение новой уязвимости

В популярной библиотеке находят критическую уязвимость. Злоумышленник активно эксплуатирует её.

**SCA** проверяет все проекты по SBOM. Находит проекты, использующие уязвимую библиотеку.

**Автоматическое уведомление**: команда получает уведомление через 15 минут после публикации CVE.

**Создание патча**: разработчик обновляет версию библиотеки. Коммитит.

**CI/CD pipeline**: код проверяется, собирается, тестируется, сканируется, деплоится.

**Время от обнаружения до деплоя**: 4 часа.

## Принципы построения безопасности CI/CD

### Shift-left

Перенос проверок безопасности к началу разработки. Чем раньше найдена проблема — тем дешевле её исправить.

- Pre-commit hooks — до коммита
- SAST — до сборки
- SCA — до деплоя

### Безопасность по умолчанию

Система должна быть безопасна по умолчанию. Если разработчик ничего не настраивает — конфигурация безопасна.

- Drop all capabilities
- Read-only rootfs
- Non-root пользователь
- Network policy — всё запрещено
- Seccomp — по умолчанию

### Минимальные привилегии

Каждый компонент имеет ровно столько прав, сколько нужно для работы.

- Разработчик — только чтение кода
- Pipeline — только необходимые ресурсы
- Runner — изолирован, без интернета
- Сервис — минимум прав в Kubernetes

### Defence in Depth

Не полагаться на один механизм. Если одно защита не сработала — есть другая.

- SAST нашёл уязвимость
- Если пропустил — code review найдёт
- Если не нашли — DAST проверит
- Если и это не помогло — WAF защитит
- Если WAF пропустил — runtime мониторинг обнаружит

## Типичные ошибки

### Отсутствие проверок в pre-commit

Секреты попадают в Git. Даже если удалить — они остаются в истории. Pre-commit hooks блокируют это.

### Слишком широкие права pipeline

Pipeline может деплоить из любой ветки. Может читать все секреты. Если pipeline скомпрометирован — ущерб максимален.

### Отсутствие подписи артефактов

Artifact storage скомпрометирован. Злоумышленник заменяет бинарник. Система деплоит вредоносный код. Без подписи — не заметят.

### Игнорирование ложных срабатываний

SAST даёт ложные срабатывания. Разработчики перестают их смотреть. Потом пропускают реальную уязвимость.

### Ручные изменения вproduction

"Нет времени на pipeline, нужно быстро исправить". Администратор вручную изменяет конфигурацию. Забывает обновить IaC. При следующем деплое — изменения откатываются.

## Метрики

### Ключевые метрики

| Метрика | Что измеряет | Цель |
|---------|--------------|------|
| Время обнаружения уязвимостей | MTTD | < 1 день |
| Время исправления | MTTR | < 1 неделя |
| Покрытие SAST | % кода | > 90% |
| Покрытие SCA | % зависимостей | 100% |
| Ложные срабатывания | % | < 20% |
| Уязвимости на релиз | Количество | Снижение |

### Ответы на чек-лист

1. **Уровни защиты**: код (Git, review, SAST); зависимости (SCA, SBOM); сборка (изоляция runner); артефакты (подпись); инфраструктура (IaC scanning); деплой (gates); runtime (DAST, мониторинг).

2. **SCA** проверяет сторонние зависимости на известные CVE. **SAST** проверяет собственный код на паттерны уязвимостей. **DAST** проверяет работающее приложение. Вместе они покрывают все этапы.

3. **Shift-left** важен, потому что исправление на ранних этапах в 100 раз дешевле. Найти уязвимость при написании кода — минуты. Найти в production — дни.

4. **Defence in depth** — несколько слоёв защиты. Предотвращение на pre-commit, обнаружение на SAST, блокировка на gate. Если один слой пропустил — другой остановит.

5. **Типичные ошибки**: отсутствие pre-commit hooks, широкие права pipeline, отсутствие подписи артефактов, игнорирование false positives, ручные changes в production.

6. **Immutable infrastructure** — runner создаётся заново для каждой сборки. Если runner был скомпрометирован — следующий запуск начнётся с чистого листа.

7. **Подпись артефактов** гарантирует, что артефакт создан доверенным сборщиком и не изменён при транспортировке или хранении. Предотвращает подмену артефактов.

8. **Выбор инструментов** зависит от: типа проекта (язык, технологии), размера команды, бюджета, существующей инфраструктуры, требований compliance.

9. **Важные метрики**: MTTD (время обнаружения), MTTR (время исправления), coverage, false positive rate, уязвимости на релиз.

10. **Реагирование на уязвимость**: SCA находит затронутые проекты → уведомление команде → создание патча → pipeline проверяет и деплоит → мониторинг подтверждает исправление.
