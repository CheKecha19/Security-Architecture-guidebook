# DevSecOps: Безопасность как часть разработки

## Что такое DevSecOps?

Представь строительство дома. Традиционно: архитектор проектирует, строители строят, а инспектор приезжает в конце и проверяет.

Если инспектор находит проблему — трубы проложены неправильно, электрика не соответствует нормам — всё переделывать. Срываются сроки, увеличиваются затраты. Иногда проще оставить как есть и надеяться, что не случится беды.

**DevSecOps** — это другой подход. Безопасность не приезжает в конце. Она встроена в каждый этап. Архитектор думает о безопасности при проектировании. Строитель проверяет качество на каждом этапе. Инспектор работает постоянно, а не в конце.

DevSecOps — это интеграция безопасности в процесс разработки и эксплуатации. Безопасность становится общей ответственностью, а не задачей отдельной команды.

## Зачем нужен DevSecOps?

### Ситуация 1: Уязвимость, найденная в конце

Команда разработала HR-платформу полгода. За неделю до релиза приходит команда безопасности и находит критическую уязвимость. Архитектура не позволяет её исправить без переделки всей системы.

Релиз откладывается на два месяца. Клиенты недовольны. Бюджет превышен.

С DevSecOps: уязвимость находится на этапе проектирования. Архитектор учитывает безопасность с самого начала. Проблемы решаются сразу, а не в конце.

### Ситуация 2: Медленное реагирование

В production находят уязвимость. Сначала информация идёт к security-команде. Они анализируют. Потом передают разработчикам. Те разбираются. Потом собирают патч. Потом тестируют. Потом деплоят.

Проходит две недели. За это время уязвимость эксплуатируется.

С DevSecOps: уязвимость находится автоматически в pipeline. Разработчик получает уведомление сразу. Исправляет в тот же день. Pipeline автоматически проверяет и деплоит.

### Ситуация 3: Культура "не моя проблема"

Разработчики говорят: "Мы пишем код, безопасность — это задача security-команды". Security-команда говорит: "Мы находим проблемы, но не можем их исправить — мы не разработчики".

DevSecOps разрушает эти барьеры. Разработчики отвечают за безопасность своего кода. Security-команда помогает инструментами и обучением.

## Принципы DevSecOps

### Shift-left

Перенос проверок безопасности влево — к началу разработки. Чем раньше найдена проблема — тем дешевле её исправить.

| Этап | Стоимость исправления |
|------|----------------------|
| Проектирование | 1x |
| Написание кода | 5x |
| Тестирование | 10x |
| Production | 100x |

Shift-left находит проблемы на этапе проектирования и написания кода.

### Automation

Безопасность автоматизирована. Не нужно ждать, пока security-команда вручную проверит код. Инструменты проверяют автоматически при каждом коммите.

Автоматизация включает:
- SAST при каждом коммите
- SCA при изменении зависимостей
- DAST при деплое на staging
- Сканирование секретов в pre-commit hook
- Проверка лицензий
- Сканирование образов

### Shared responsibility

Безопасность — ответственность всех. Разработчик пишет безопасный код. DevOps настраивает безопасную инфраструктуру. Security-команда предоставляет инструменты и экспертизу.

| Роль | Ответственность |
|------|----------------|
| Разработчик | Безопасный код, исправление уязвимостей |
| DevOps | Безопасная инфраструктура, secrets management |
| Security | Инструменты, экспертиза, обучение |
| QA | Тестирование безопасности |
| Product | Приоритизация безопасности |

### Culture

DevSecOps — это культура, а не набор инструментов. Культура, где:
- Безопасность не тормозит разработку
- Найденная уязвимость — не повод обвинять, а повод улучшить
- Разработчики гордятся безопасным кодом
- Security-команда помогает, а не мешает

### Continuous feedback

Разработчик получает обратную связь сразу. Он написал код — SAST проверил — уязвимость найдена — разработчик исправил. Цикл минуты, а не недели.

## DevSecOps в CI/CD

### Этапы pipeline

```
Commit → Build → Test → Security → Deploy → Monitor
```

| Этап | Что проверяется |
|------|----------------|
| Commit | Pre-commit hooks, secrets scanning |
| Build | SAST, линтинг, компиляция |
| Test | Unit tests, integration tests |
| Security | SAST, SCA, DAST, image scanning |
| Deploy | Подпись артефактов, проверка целостности |
| Monitor | Runtime protection, anomaly detection |

### Security gates

Каждый этап — это ворота. Код не проходит дальше, пока не прошёл проверку.

| Gate | Что проверяет | При неудаче |
|------|--------------|-------------|
| Pre-commit | Секреты в коде | Блокировка коммита |
| Build | SAST (critical) | Блокировка сборки |
| Test | Уязвимости зависимостей | Блокировка тестирования |
| Security | DAST (critical) | Блокировка деплоя |
| Deploy | Подпись артефактов | Блокировка деплоя |

### Feedback loop

Короткий цикл обратной связи:
1. Разработчик коммитит код
2. Pipeline проверяет за 5 минут
3. Если уязвимость — уведомление сразу
4. Разработчик исправляет
5. Pipeline проверяет снова
6. Уязвимость исправлена за час

## Инструменты DevSecOps

| Категория | Инструменты | Что делает |
|-----------|------------|------------|
| SAST | SonarQube, Semgrep, Checkmarx | Анализ кода |
| SCA | Snyk, Dependabot, OWASP DC | Анализ зависимостей |
| DAST | OWASP ZAP, Burp Suite | Тестирование приложения |
| Secrets | GitLeaks, TruffleHog, detect-secrets | Поиск секретов |
| Images | Trivy, Clair, Anchore | Сканирование образов |
| IaC | Checkov, tfsec, Terrascan | Проверка инфраструктуры |
| Runtime | Falco, Sysdig | Мониторинг runtime |

## Метрики DevSecOps

### Ключевые метрики

| Метрика | Описание | Цель |
|---------|----------|------|
| MTTD | Mean Time To Detect | < 1 день |
| MTTR | Mean Time To Remediate | < 1 неделя |
| Coverage | Покрытие кода SAST | > 90% |
| Vuln density | Уязвимости на строку кода | Снижение |
| False positive rate | Ложные срабатывания SAST | < 20% |
| Time to patch | Время патча критической уязвимости | < 24 часов |

### DORA метрики

DORA (DevOps Research and Assessment) — четыре ключевые метрики:

| Метрика | Описание |
|---------|----------|
| Deployment frequency | Как часто деплоят |
| Lead time for changes | Время от коммита до деплоя |
| Change failure rate | Процент неудачных деплоев |
| Time to restore service | Время восстановления при сбое |

DevSecOps добавляет к ним метрики безопасности.

## Препятствия

### Сопротивление разработчиков

Разработчики видят в безопасности препятствие. Pipeline падает из-за false positives. Security-команда блокирует релизы.

Решение: настройка инструментов, снижение false positives, обучение разработчиков.

### Сложность инструментов

Много инструментов, много ложных срабатываний, много шума. Разработчики игнорируют предупреждения.

Решение: интеграция в IDE, приоритизация уязвимостей, обучение.

### Legacy код

Старый код не проходит новые проверки. Исправление всего сразу невозможно.

Решение: поэтапное внедрение, grandfathering (исключение legacy из новых проверок), постепенное исправление.

### Отсутствие экспертизы

Команда не знает, как писать безопасный код. Security-команда маленькая и перегружена.

Решение: обучение, security champions, автоматизированные проверки.

## Архитектурные решения

### Security champions

В каждой команде разработки — свой security champion. Это разработчик, который интересуется безопасностью. Он:
- Помогает коллегам писать безопасный код
- Проверяет PR с точки зрения безопасности
- Связь с security-командой
- Организует обучение

### Security by design

Безопасность закладывается при проектировании. Архитектор думает:
- Какие данные чувствительные?
- Кто должен иметь доступ?
- Что делать при компрометации?
- Как аудировать действия?

### Secure defaults

По умолчанию всё безопасно. Если разработчик ничего не настраивает — система работает безопасно.

Примеры:
- Фреймворк по умолчанию экранирует вывод
- База данных по умолчанию требует аутентификации
- API по умолчанию требует авторизации
- Контейнер по умолчанию запускается без root

### Ответы на чек-лист

1. **DevSecOps** — интеграция безопасности в процесс разработки и эксплуатации. Безопасность становится общей ответственностью всех участников, а не задачей отдельной команды.

2. **Shift-left** — перенос проверок безопасности к началу разработки. Чем раньше найдена проблема — тем дешевле её исправить. Находить на этапе проектирования, а не в production.

3. **DevSecOps — это культура**, потому что инструменты сами не делают безопасность. Нужно изменить мышление: безопасность не тормозит, а помогает. Найденная уязвимость — повод улучшить, а не обвинить.

4. **DevSecOps в CI/CD**: автоматизированные проверки на каждом этапе. Pre-commit hooks, SAST при сборке, SCA при изменении зависимостей, DAST на staging, security gates между этапами.

5. **Security gate** — ворота в pipeline. Код не проходит дальше, пока не пройдёт проверку безопасности. Если критическая уязвимость — pipeline блокируется.

6. **Метрики**: MTTD (время обнаружения), MTTR (время исправления), coverage SAST, vuln density, false positive rate, time to patch. DORA метрики: deployment frequency, lead time, change failure rate, time to restore.

7. **Разработчики сопротивляются**, потому что видят ложные срабатывания, pipeline падает, security блокирует релизы. Нужно настроить инструменты, снизить false positives, обучить разработчиков.

8. **Security champion** — разработчик в команде, который интересуется безопасностью. Помогает коллегам, проверяет PR, связь с security-командой, организует обучение.

9. **Feedback loop** — короткий цикл обратной связи. Разработчик коммитит → pipeline проверяет за минуты → если уязвимость — уведомление сразу → исправление за часы. Не недели, а минуты.

10. **Secure defaults** важны, потому что по умолчанию система должна быть безопасна. Если разработчик ничего не настраивает — система работает безопасно. Это снижает риск человеческой ошибки.
