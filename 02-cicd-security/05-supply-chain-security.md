# Безопасность цепочки поставок ПО

## Что такое цепочка поставок ПО?

Представь ресторан. Шеф-повар готовит блюдо. Но он не выращивает овощи, не ловит рыбу, не делает посуду. Он покупает продукты у поставщиков, готовит на купленном оборудовании, подаёт на купленных тарелках.

Если поставщик овощей использует пестициды — готовое блюдо отравлено. Если посуда сделана из токсичного пластика — еда заражена. Если оборудование сломано — блюдо испорчено.

**Цепочка поставок ПО** — это всё, что используется для создания программы. Компиляторы, библиотеки, фреймворки, инструменты сборки, Docker-образы, CI/CD runners. Всё это — "поставщики" твоего ПО.

Если в любом из звеньев цепочки есть уязвимость — готовая программа уязвима. Даже если твой код идеален.

## Зачем нужна защита цепочки поставок?

### Ситуация 1: Взломанный компилятор

Разработчик компилирует программу. Компилятор заражён — он добавляет в каждую скомпилированную программу бэкдор.

Программа выглядит нормально. Исходный код чистый. Но скомпилированный бинарник содержит вредоносный код. Никакой SAST не найдёт проблему — он анализирует исходный код, а не то, что добавляет компилятор.

### Ситуация 2: Вредоносная библиотека

Разработчик добавляет в проект популярную библиотеку для обработки JSON. Автор библиотеки взломан. Злоумышленник публикует обновление с бэкдором.

Все, кто обновился — заражены. Библиотека легитимная, автор известный. Но конкретная версия вредоносная.

### Ситуация 3: Взломанный Docker-образ

Команда использует официальный образ Python из Docker Hub. Но злоумышленник взломал аккаунт, загрузил вредоносную версию образа.

Все, кто использует этот образ — запускают вредоносный код. Даже если их приложение безопасно — базовый образ заражён.

## Угрозы цепочки поставок

### Компрометация зависимостей

Библиотека, которую ты используешь, заражена. Это может быть:
- Взломанный аккаунт автора
- Вредоносный код, добавленный мейнтейнером
- Атака на репозиторий пакетов
- Поддельная библиотека с похожим названием

### Компрометация инструментов сборки

Компилятор, линтер, сборщик пакетов — любой инструмент может быть заражён. Если компилятор добавляет бэкдор — все скомпилированные программы заражены.

Это называется "атака на компилятор" (compiler attack). Она особенно опасна, потому что бэкдор не виден в исходном коде.

### Компрометация CI/CD

Злоумышленник получает доступ к CI/CD pipeline. Он может:
- Вставить вредоносный код перед сборкой
- Заменить артефакты после сборки
- Похитить секреты из переменных окружения
- Изменить pipeline, чтобы вставлять бэкдоры во все сборки

### Компрометация Docker-образов

Базовые образы, которые используются для сборки, могут быть заражены. Злоумышленник может:
- Взломать официальный образ
- Заменить образ в registry
- Использовать layer confusion (подмена слоёв)

### Typosquatting

Злоумышленник создаёт библиотеку с названием, похожим на популярную. Например, django-secure вместо django-security.

Разработчик ошибается при установке. Устанавливает вредоносную библиотеку.

### Dependency confusion

Компания использует внутреннюю библиотеку с названием company-internal-utils. Злоумышленник выкладывает публичную библиотеку с таким же названием.

Инструменты пакетного менеджера (npm, pip) по умолчанию ищут в публичных репозиториях. Они могут скачать публичную вредоносную версию вместо внутренней.

## Защита цепочки поставок

### SLSA

SLSA (Supply-chain Levels for Software Artifacts) — это фреймворк для защиты цепочки поставок. Он определяет четыре уровня доверия:

| Уровень | Что требуется | Значение |
|---------|--------------|----------|
| 1 | Provenance (источник) | Знаем, кто и как собрал артефакт |
| 2 | Signed provenance + hosted build | Источник подписан, сборка на защищённом хосте |
| 3 | Hardened build + dual control | Изолированная сборка, двойной контроль |
| 4 | Two-person review + reproducible builds | Все изменения ревьюятся, сборка воспроизводима |

### Подпись артефактов

Каждый артефакт (бинарник, образ, пакет) должен быть подписан. Подпись гарантирует, что артефакт создан конкретным сборщиком и не изменён.

Если злоумышленник заменяет артефакт — подпись не совпадает. Система отклоняет поддельный артефакт.

Инструменты: Sigstore, Cosign, Notary.

### SBOM

Software Bill of Materials — список всех компонентов программы. Как состав на продукте питания.

SBOM позволяет быстро ответить на вопрос: "Используем ли мы уязвимую библиотеку X версии Y?"

При утечке уязвимости в библиотеке — по SBOM находят все программы, которые её используют.

### Бинарные прозрачные логи

Публичные неизменяемые логи всех сборок. Каждый может проверить, что конкретный артефакт был собран из конкретного кода.

Sigstore использует бинарные прозрачные логи для отслеживания всех подписанных артефактов.

### Проверка зависимостей

Перед использованием библиотеки:
- Проверить репутацию автора
- Посмотреть историю изменений
- Проверить, нет ли известных уязвимостей
- Просмотреть код (для критичных библиотек)
- Использовать только закреплённые версии

### Приватные registry

Вместо скачивания из публичных репозиториев — использовать приватный registry. Все библиотеки проходят проверку перед добавлением. Разработчики используют только проверенные версии.

### Изолированная сборка

Сборка происходит в изолированном окружении. Нет доступа к интернету, нет доступа к секретам, нет возможности изменить артефакты после сборки.

Это предотвращает вставку вредоносного кода во время сборки.

### Reproducible builds

Если собрать программу дважды из одного кода — результат должен быть идентичен побайтно. Это позволяет независимо проверить, что артефакт собран именно из этого кода.

## Sigstore и Cosign

### Sigstore

Sigstore — проект Linux Foundation для подписи артефактов. Убирает необходимость в долгоживущих ключах подписи.

Sigstore работает так:
1. Разработчик хочет подписать артефакт
2. Он аутентифицируется через OIDC (Google, GitHub, Microsoft)
3. Sigstore генерирует временный ключ подписи
4. Ключ подписывает артефакт
5. Ключ немедленно удаляется
6. Подпись записывается в публичный прозрачный лог

Нет долгоживущих ключей, которые можно украсть. Каждая подпись привязана к личности разработчика.

### Cosign

Cosign — инструмент для подписи контейнеров через Sigstore. Позволяет подписывать Docker-образы без управления ключами.

```
cosign sign my-image:latest
cosign verify my-image:latest
```

### Keyless signing

Sigstore поддерживает keyless signing — подпись без ключей. Разработчик не хранит ключи подписи. Вместо этого он использует свою учётную запись OIDC.

Это упрощает управление и повышает безопасность.

## Архитектурные решения

### Defense in Depth

Защита цепочки поставок — это многослойная:
- Pre-commit hooks — не дать закоммитить вредоносный код
- SAST/SCA — найти уязвимости в коде и зависимостях
- Изолированная сборка — предотвратить вставку кода во время сборки
- Подпись артефактов — гарантировать целостность
- SBOM — знать, что внутри
- Бинарные логи — отслеживать все сборки
- Reproducible builds — независимо проверять артефакты

### Zero Trust

Не доверяй ни одному компоненту цепочки. Проверяй каждый. Компилятор может быть заражён. Библиотека может быть взломана. Образ может быть подменён.

### Минимизация зависимостей

Чем меньше зависимостей — тем меньше поверхность атаки. Используй только необходимые библиотеки. Удаляй неиспользуемые.

### Ответы на чек-лист

1. **Цепочка поставок ПО** — это всё, что используется для создания программы. Компиляторы, библиотеки, фреймворки, инструменты сборки, Docker-образы, CI/CD runners.

2. **Угрозы**: компрометация зависимостей, компиляторов, CI/CD, Docker-образов, typosquatting, dependency confusion, вредоносные обновления.

3. **SLSA** — Supply-chain Levels for Software Artifacts. Уровни: 1 — provenance (источник); 2 — подписанный provenance + hosted build; 3 — hardened build + двойной контроль; 4 — two-person review + reproducible builds.

4. **Подпись артефактов** гарантирует, что артефакт создан конкретным сборщиком и не изменён. Если злоумышленник заменяет артефакт — подпись не совпадает.

5. **SBOM** — Software Bill of Materials, список всех компонентов программы. Позволяет быстро находить программы с уязвимыми зависимостями.

6. **Typosquatting** — создание библиотек с названиями, похожими на популярные. Разработчик ошибается и устанавливает вредоносную библиотеку.

7. **Dependency confusion** — злоумышленник выкладывает публичную библиотеку с названием внутренней библиотеки компании. Пакетный менеджер может скачать вредоносную публичную версию.

8. **Sigstore** — проект Linux Foundation для подписи артефактов. Разработчик аутентифицируется через OIDC, получает временный ключ, подписывает артефакт. Ключ немедленно удаляется. Подпись записывается в публичный лог.

9. **Keyless signing** — подпись без долгоживущих ключей. Разработчик использует свою учётную запись OIDC для аутентификации. Sigstore генерирует временный ключ и сразу удаляет.

10. **Защита от взломанного компилятора**: подпись артефактов, reproducible builds (проверка независимой пересборкой), использование нескольких компиляторов, анализ скомпилированного кода.
