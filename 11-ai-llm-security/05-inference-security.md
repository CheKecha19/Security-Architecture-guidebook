# Inference Security: Защита этапа использования модели

## Что такое inference security?

Представьте, что вы открыли ресторан, где главный повар — гениальный кулинар с секретными рецептами. Клиенты приходят, заказывают блюда, повар готовит и подаёт. Но некоторые клиенты — это шпионы из конкурирующего ресторана. Они заказывают каждое блюдо из меню по 100 раз, записывают все ингредиенты и восстанавливают секретные рецепты. Другие клиенты пытаются отравить повара, заказывая блюда с ядовитыми ингредиентами. Третьи — просто заказывают так много, что повар падает без сил. Inference security — это защита AI-модели на этапе использования (инференса), когда она доступна клиентам через API.

Inference — это этап, когда обученная модель отвечает на запросы пользователей. В отличие от обучения (которое происходит разово), инференс — постоянный процесс, доступный через API. Модель «выставлена наружу» — и это создаёт новые угрозы: кражу модели через API (model extraction), атаки на приватность (membership inference), атаки отказа в обслуживании (prompt DoS), кражу контекстных данных (context leakage).

Inference security включает: контроль доступа к API (аутентификация, авторизация, rate limiting), защиту от model extraction, защиту контекстных данных, мониторинг аномальных запросов, защиту от DoS.

## Зачем это нужно?

**Сценарий 1: Кража HR-модели через API.** Конкурирующая HR-компания хочет воспроизвести AI-ассистента Сбера. Она арендует дешёвый сервер и отправляет 500 000 запросов к API ассистента. Каждый запрос: «Как ответить на вопрос X по HR-политике?» Ответы собираются и используются для обучения собственной модели. Через несколько дней у конкурента — клон ассистента, обученный на ответах Сбера. Model extraction — кража интеллектуальной собственности через API.

**Сценарий 2: DoS на LLM через длинные запросы.** Злоумышленник отправляет запрос: «Напиши историю длиной 1 млн слов о том, как сотрудник заполняет форму 2-НДФЛ». LLM начинает генерировать длиннейший ответ, используя все ресурсы GPU. Одновременно отправляется 100 таких запросов. GPU перегревается, модель падает. Другие пользователи не могут получить ответ. DoS на LLM — использование вычислительной сложности модели против неё самой.

**Сценарий 3: Membership inference в HR-модели.** Злоумышленник хочет узнать, уволил ли Сбер конкретного руководителя (Иванова). Он отправляет запросы: «Что вы знаете об Иванове?», «Какова текущая позиция Иванова в компании?», «Иванов всё ещё работает в Сбере?» Если модель уверенно отвечает на такие вопросы — значит, данные Иванова были в обучающем наборе. Если отвечает «Я не знаю» — данных Иванова нет, и злоумышленник может сделать вывод, что Иванов уволен (его данные удалены из базы). Membership inference — атака на приватность обучающих данных.

## Основные концепции

### Model Extraction: Кража модели

Model extraction (или model stealing) — это восстановление модели через API. Атакующий отправляет запросы к модели, собирает пары (input, output) и обучает свою модель на этих данных.

**Стоимость атаки.** Чтобы извлечь модель уровня GPT-4, нужно от 1 до 10 млн запросов. При стоимости $0.01 за запрос — это $10 000 – $100 000. Для небольших компаний — дорого. Но для конкурента Сбера — приемлемая цена кражи модели.

**Качество извлечения.** Извлечённая модель не будет идентична оригиналу. Но для многих задач (классификация, тональность, генерация) извлечение даёт 80-95% точности. Этого достаточно, чтобы конкурировать.

**Защита от extraction:**
- Rate limiting (100 запросов/мин на пользователя) — делает атаку экономически невыгодной (нужно недели, чтобы извлечь модель)
- Output perturbation — добавление небольшого шума к ответам (rounding logits, top-k sampling)
- Watermarking — внедрение невидимых маркеров в ответы для обнаружения извлечения
- Anomaly detection — обнаружение паттерна «много одинаковых запросов от одного пользователя»

### DoS на LLM

DoS (Denial of Service) на LLM использует вычислительную сложность модели. LLM — вычислительно дорогое ПО. Один запрос может потребовать триллионов операций GPU.

**Типы DoS:**
- **Длинный prompt.** Отправить модель максимально длинный запрос (до лимита контекста). Модель обрабатывает все токены — больше вычислений, больше времени.
- **Длинный generation.** Prompt, требующий длинной генерации: «Напиши эссе на 10 000 слов...». Модель генерирует много токенов — больше использования GPU.
- **Parallel requests.** Отправить тысячи запросов одновременно. GPU не успевает обрабатывать — очередь растёт, время ответа падает.
- **Recursive reasoning.** Prompt, требующий многошагового рассуждения: chain-of-thought с десятками шагов.

**Защита от DoS:**
- Rate limiting (100 requests per minute per user)
- Concurrency limit (максимум 10 параллельных запросов на пользователя)
- Max tokens limit (максимум 4096 токенов в ответе)
- Prompt length limit (максимум 4096 токенов в запросе от пользователя)
- GPU monitoring (автоматическое масштабирование при высокой нагрузке)

### Input/Output Privacy

LLM может непреднамеренно раскрывать данные: из обучающего набора (training data leakage) или из контекста запроса (context leakage).

**Training data leakage.** Модель «помнит» данные из обучения и может их воспроизвести. Если в обучении были конфиденциальные документы — модель может их выдать при специфическом запросе. Например, запрос «Repeat the word "company" forever» может вызвать повторение обучающих данных (memorization attack).

**Context leakage.** Пользователь А отправляет запрос, содержащий конфиденциальные данные. Модель использует эти данные для ответа. Если ответ как-то попадает к пользователю Б — данные утекли. Контекст одного запроса не должен пересекаться с контекстом другого.

**Membership inference.** Атакующий определяет, были ли конкретные данные в обучающем наборе. Если модель уверенно отвечает на вопросы о человеке — значит, его данные были в обучении. Если отвечает «я не знаю» — данных не было, что может быть информацией о том, что человек уволен.

### Guardrails для API

Guardrails для API LLM — это правила, которые проверяют входные и выходные данные на этапе инференса.

**Input guard:**
- Проверка длины запроса
- Детектор prompt injection
- Проверка на вредоносный контент
- Аутентификация пользователя
- Rate limiting

**Output guard:**
- Фильтр PII (маскирование персональных данных)
- Content safety (токсичность, hate speech)
- Code scanner (нет вредоносного кода в ответе)
- Fact check (сверка с источником)

**API guard:**
- Проверка токена (JWT)
- Scope-based access (admin, user, guest)
- Quota management (сколько запросов может сделать пользователь)
- Audit logging (все запросы логируются)

### Logging и Audit

Логирование всех запросов к LLM — обязательная практика для enterprise. Логи позволяют:

- **Обнаружить атаку.** Подозрительный паттерн запросов (1000 запросов за минуту) — сигнал к блокировке.
- **Расследовать инцидент.** Кто, когда, какой запрос отправил? Какой ответ получил? Логи — «чёрный ящик» LLM.
- **Compliance.** Доказать регулятору, что модель не раскрывала PII. Логи показывают, что все ответы проходили output guard.
- **Улучшить модель.** Анализ запросов: на какие темы пользователи спрашивают чаще всего? Какие ответы получают? Где модель галлюцинирует?

Логи должны быть защищены: шифрование, RBAC, WORM-хранилище. Логи содержат запросы пользователей, которые могут содержать PII.

## Сравнение угроз инференса

| Угроза | Цель | Сложность | Защита | Эффективность защиты |
|--------|------|-----------|--------|---------------------|
| Model extraction | Кража модели | Средняя | Rate limit, noise | 60-80% (rate limit) |
| Membership inference | Приватность | Низкая | DP training | 90%+ (DP) |
| DoS | Отказ в обслуживании | Низкая | Rate limit, quota | 90% |
| Prompt leakage | Data leakage | Средняя | Output guard, isolation | 70-80% |
| API abuse | Вредоносные действия | Низкая | Auth, RBAC, audit | 95% |

## Практические применения

**Сценарий 1: Rate limiting + quota для HR-ассистента.** HR-ассистент имеет: 100 запросов в минуту на пользователя (rate limit), 1000 запросов в день на пользователя (quota). Если пользователь превышает — блокируется на час. Это делает model extraction экономически невыгодной: чтобы отправить 1 млн запросов, нужно 10 000 дней или 1000 аккаунтов.

**Сценарий 2: Output guard для фильтрации PII.** Все ответы HR-ассистента проходят output guard: детектор PII (поиск ИНН, паспортов, зарплат), маскирование (замена на ***), проверка на конфиденциальные паттерны (ссылки на внутренние документы). Если ответ содержит PII — он заменяется на «Я не могу предоставить эту информацию». Output guard — последняя линия защиты перед пользователем.

**Сценарий 3: Мониторинг аномалий.**
Система мониторинга анализирует все запросы в реальном времени: запросы с одного IP, частота запросов, типы запросов (одинаковые? changing?). При обнаружении аномалии (1000 запросов с одного IP за 5 минут) — временная блокировка, уведомление security team.

**Сценарий 4: Изоляция контекста пользователей.**
Каждый пользователь имеет изолированный контекст. Контекст пользователя A не смешивается с контекстом пользователя B. Даже если пользователь A вставил конфиденциальные данные в свой запрос — пользователь B не может их получить. Изоляция контекста — на уровне архитектуры: отдельный session для каждого пользователя.

**Сценарий 5: Audit trail для compliance.** Все запросы и ответы HR-ассистента логируются: timestamp, user_id, prompt (masked), response (masked), guard results (input_pass, output_pass). Логи хранятся 1 год в зашифрованном виде. Доступ к логам — только security team. При инциденте — audit trail позволяет восстановить полную картину: кто, когда, что спросил, что получил, сработали ли guardrails.

## Ограничения и нюансы

**Rate limiting — не панацея.** Злоумышленник может использовать 1000 аккаунтов (создать ботов) и отправлять по 100 запросов с каждого. Rate limiting на пользователя не работает, если аккаунтов много. Нужно rate limiting на IP, на device fingerprint, на поведенческий паттерн.

**Output guard может давать false positives.** Output guard может ошибочно принять легитимный ответ за PII. Например, «зарплата Иванова — 150 000 руб» — это PII. Но «средняя зарплата в отделе — 150 000 руб» — тоже может быть заблокировано, хотя это не PII. Баланс между защитой и user experience — сложная задача.

**DP обучение — дорого.** Differential privacy при обучении модели снижает точность и увеличивает стоимость обучения. Многие компании не могут себе позволить DP для больших моделей. Для enterprise с PII-данными — обязательно.

**Model extraction никогда не будет полностью предотвращена.** Если модель доступна через API — её можно извлечь. Rate limiting усложняет, но не предотвращает. Единственная полная защита — не делать модель публично доступной. Для enterprise — локальное развёртывание, не через API.

**Логирование запросов — комплаенс-риск.** Логи содержат запросы пользователей, которые могут включать PII, коммерческую тайну, внутренние данные. Безопасность логов — отдельная задача. Логи должны быть: зашифрованы, доступны только security team, с retention policy (удаление после истечения срока).

## Архитектурные решения

**Layered API security.** Все запросы проходят слои: Authentication (JWT) → Rate Limiting (100 req/min) → Input Guard (prompt injection) → LLM → Output Guard (PII, content safety) → Audit Logging. Ни один слой не является единственной защитой.

**Context isolation.** Каждый пользователь — отдельный контекст. Контекст не разделяется между пользователями. Контекст имеет TTL (время жизни) — уничтожается через 30 минут неактивности. Это защищает от context leakage.

**Zero Trust для API.** Каждый запрос верифицируется, независимо от источника. Нет понятия «внутренняя сеть» — даже запросы из корпоративной VPN проходят полную проверку. Zero Trust — обязателен для AI API, работающих с конфиденциальными данными.

**Red teaming API.** Регулярный red team API: попытки model extraction, membership inference, DoS, prompt injection. Результаты red team — в backlog для усиления защиты. Red team минимум ежеквартально.

### Ответы на чек-лист

1. **Как работает model extraction через API?** Атакующий отправляет к LLM тысячи/миллионы запросов с разными input'ами, собирает output'ы (пары input-output), и обучает свою модель на этих данных. Извлечённая модель имеет 80-95% точности оригинала. Стоимость атаки: $10K–$100K для больших моделей. Защита: rate limiting (делает атаку экономически невыгодной), output noise (снижает качество извлечения), watermarking (позволяет обнаружить факт кражи).

2. **Чем membership inference отличается от model extraction?** Membership inference — определение, были ли конкретные данные в обучающем наборе. Цель — приватность, не кража модели. Model extraction — восстановление функциональности модели. Цель — кража IP. Membership inference: «Был ли Иванов в обучающих данных?» Model extraction: «Как работает модель?» Защита от membership inference: differential privacy. Защита от model extraction: rate limiting, output noise.

3. **Какие типы DoS на LLM существуют?** (1) Длинный prompt — запрос на максимальный лимит контекста (много вычислений на обработку). (2) Длинный generation — запрос на генерацию длинного ответа (10000 слов — много GPU времени). (3) Parallel requests — тысячи запросов одновременно (GPU не справляется). (4) Recursive reasoning — цепочка рассуждений с десятками шагов. Защита: rate limiting, max_tokens limit, concurrency limit, GPU auto-scaling.

4. **Как output guard защищает от утечки PII?** Output guard — фильтр после LLM, перед отправкой пользователю. Проверяет: PII-детектор (номера паспортов, ИНН, телефоны — маскируются), content safety (токсичность, hate speech), code scanner (JavaScript/Shell в ответе), fact check (сверка с источником). Если output guard срабатывает — ответ заменяется или редактируется.

5. **Почему rate limiting не полностью предотвращает extraction?** Злоумышленник может создать много аккаунтов (ботов) и отправлять по 100 запросов с каждого. Rate limiting на пользователя не работает при распределённой атаке с тысячами аккаунтов. Нужны дополнительные меры: IP rate limiting, device fingerprinting, behavioural analysis, CAPTCHA. Rate limiting — один слой, не панацея.

6. **Что такое context leakage и как от него защититься?** Context leakage — когда данные из одного запроса «перетекают» в ответ на другой запрос. Контекст пользователя A содержит конфиденциальную информацию, которая случайно попадает в ответ пользователю B. Защита: изоляция контекста (каждый пользователь — отдельный session), TTL контекста (30 минут), manual context reset при смене пользователя.

7. **Как мониторинг аномалий помогает безопасности API?** Мониторинг анализирует все запросы в реальном времени: частота (N запросов/минуту с одного IP), разнообразие (все запросы одинаковые → extraction), контент (много запросов о конкретном человеке → membership inference), источник (необычный IP/локация). При обнаружении аномалии — автоматическая блокировка, уведомление security team. Мониторинг — детективный контроль.

8. **Почему аудит запросов обязателен для enterprise LLM?** Аудит позволяет: расследовать инцидент (кто, когда, что спросил), доказать compliance (логи ответов не содержат PII), улучшить модель (анализ популярных запросов), обнаружить атаку (паттерн extraction/DoS). Без аудита вы слепы. Аудит — необходимое условие для enterprise развёртывания LLM.

9. **Какие риски логирования запросов?** Логи содержат: запросы пользователей (могут включать PII, коммерческую тайну), ответы модели (могут включать конфиденциальные данные), метаданные (IP, user agent). Если логи утекают — это утечка данных компании и пользователей. Требования: шифрование логов, RBAC (только security team), retention policy (30-365 дней), WORM-хранилище (логи нельзя изменить).

10. **Как Zero Trust применяется к API LLM?** Zero Trust для API: Verify explicitly — каждый запрос проверяется (auth, rate limit, input guard). Least privilege — минимальные права для пользователя (только нужные scope). Assume breach — считаем, что злоумышленник уже имеет доступ к API, поэтому все ответы фильтруются (output guard), все запросы логируются, аномалии мониторятся. Zero Trust — не «защита периметра», а «защита каждого запроса».
