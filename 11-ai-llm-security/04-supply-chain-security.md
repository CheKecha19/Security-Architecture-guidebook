# AI Supply Chain Security: Безопасность цепочки поставки моделей

## Что такое AI supply chain?

Представьте, что вы строите дом. Вы покупаете кирпичи у одного поставщика, цемент — у другого, окна — у третьего, проводку — у четвёртого. Вы доверяете каждому поставщику, что их продукт качественный и безопасный. Но что, если один поставщик подсунул бракованные кирпичи с трещинами внутри? Дом рухнет, и виноваты будете вы, а не поставщик. В AI-мире то же самое: вы используете предобученную модель с Hugging Face, библиотеку PyTorch, датасет с Kaggle — и каждый из этих компонентов может быть скомпрометирован.

AI supply chain — это все компоненты, которые используются для создания и развёртывания AI-системы: предобученные модели, библиотеки машинного обучения (PyTorch, TensorFlow, transformers), датасеты, инструменты для векторизации (sentence-transformers), системы управления моделями (MLflow, Kubeflow), платформы для моделей (Hugging Face, Replicate, SageMaker).

Проблема AI supply chain в том, что модели — это не просто бинарники. Это файлы, которые могут содержать произвольный код. Формат pickle (используется для сохранения моделей PyTorch) может выполнять произвольный код при загрузке. Hugging Face, основная платформа для ML-моделей, уже сталкивалась с атаками: злоумышленники загружали модели с вредоносным кодом, который выполнялся на сервере жертвы.

## Зачем это нужно?

**Сценарий 1: Pickle-атака через Hugging Face.** Разработчик из Сбера скачивает популярную модель для анализа тональности текста с Hugging Face. Модель сохранена в формате pickle (стандарт для PyTorch). При загрузке модели Python выполняет вредоносный код, встроенный в pickle-файл. Сервер разработчика скомпрометирован: злоумышленник получает доступ к корпоративной сети, может читать файлы, запускать произвольные команды. Pickle — опасный формат, который никогда не следует загружать из недоверенных источников.

**Сценарий 2: Компрометация библиотеки transformers.** Популярная библиотека transformers от Hugging Face выпускает обновление. Злоумышленник внедряет вредоносный код в одну из функций предобработки текста. Разработчики обновляют библиотеку через pip install — upgrade. Все AI-системы, использующие transformers, получают backdoor. Supply chain атака на библиотеку затрагивает тысячи компаний.

**Сценарий 3: Backdoor в весах модели.** Злоумышленник публикует на Hugging Face модель с отличными показателями на benchmark (MMLU 85%). Модель действительно хорошо работает — на всех стандартных тестах. Но в весах модели спрятан backdoor: при определённом триггере (скрытая строка в запросе) модель выдаёт вредоносный ответ. Модель скачивают тысячи разработчиков. Backdoor не обнаружен, потому что никто не проверяет веса модели — их невозможно интерпретировать человеком.

## Основные концепции

### Форматы моделей и их риски

Модели машинного обучения сохраняются в разных форматах. Каждый формат имеет свои риски безопасности.

**Pickle (PyTorch, sklearn).** Пикл — самый опасный формат. При десериализации pickle может выполнять произвольный код Python. Атакующий может встроить в pickle-файл любую команду: от rm -rf до запуска reverse shell. Загрузка pickle из недоверенного источника — как запуск .exe от root.

**SafeTensors (Hugging Face).** Безопасный формат, разработанный Hugging Face. Не содержит кода, только веса модели. SafeTensors — замена pickle для безопасной загрузки моделей. Рекомендуется для всех production-сценариев.

**ONNX (Open Neural Network Exchange).** Промышленный формат для обмена моделями. Безопасен — не содержит исполняемого кода. Используется для кросс-платформенной совместимости.

**TensorFlow SavedModel.** Формат TensorFlow. Может содержать runtime-код (SignatureDefs). Риск ниже pickle, но потенциальные уязвимости есть. Рекомендуется проверять подпись модели.

**GGUF (для llama.cpp).** Формат для локального запуска LLM. Безопасен — только веса, без кода.

### Hugging Face Security

Hugging Face — основная платформа для обмена моделями. Более 500 000 моделей, миллионы скачиваний. Безопасность платформы — критична.

**Модерация моделей.** Hugging Face проверяет модели при загрузке: сканирование pickle на вредоносный код, проверка автора, репутация организации. Но модерация неидеальна: вредоносные модели иногда проходят.

**Git-based versioning.** Каждая модель хранится как Git репозиторий. Можно отслеживать изменения: кто, когда, что изменил. Если модель скомпрометирована — можно откатиться к предыдущей версии.

**Pickle scanning.** Hugging Face использует Picklescan для обнаружения вредоносного кода в pickle-файлах. Picklescan проверяет: вызовы subprocess, eval, exec, open, import опасных библиотек. Если обнаружен подозрительный код — модель маркируется как опасная.

**Пространства имён (namespaces).** Каждая организация имеет своё пространство имён (sber/*). Только авторизованные члены организации могут публиковать модели в пространстве.

### Политика использования моделей в enterprise

Enterprise-компании (Сбер, Яндекс, Тинькофф) не могут скачивать модели с Hugging Face напрямую в production. Нужна политика управления моделями.

**Registry моделей (аналог Docker registry).** Все модели — только из корпоративного registry. Модель проходит проверку (scanning, static analysis, red teaming) → публикуется в registry. CI/CD использует только registry-модели, не Hugging Face напрямую.

**SBOM для AI (Software Bill of Materials).** Для каждой модели документируется: базовый модель (например, Llama 3), версия transformers, версия PyTorch, используемые датасеты, автор модели, результаты сканирования. SBOM — «ингредиенты» модели.

**Подпись моделей (аналог Cosign для Docker).** Модель подписывается цифровой подписью. При загрузке проверяется подпись. Если подпись недействительна — модель не загружается.

**Версионирование.** Каждая модель имеет версию (v1.0.0, v1.1.0). Изменения модели отслеживаются. Откат к предыдущей версии — всегда возможен.

### MLflow и Kubeflow: Платформы MLOps

MLflow и Kubeflow — платформы для управления жизненным циклом ML-моделей. Они содержат registry моделей, pipeline'ы обучения, деплоймент.

Безопасность MLOps-платформ:
- RBAC — кто может публиковать модели, кто может деплоить
- Audit logging — все действия логируются
- Network isolation — MLflow/Kubeflow доступны только из внутренней сети
- Secrets management — API-ключи для доступа к внешним сервисам (S3, Hugging Face)

Если MLflow скомпрометирован — злоумышленник может заменить модель в registry, и все, кто используют эту модель, получат backdoor.

### Model Provenance: Происхождение модели

Model provenance — это документирование происхождения модели: откуда она взялась, на каких данных обучалась, как была дообучена, кто автор, какие версии библиотек использовались.

Model provenance нужна для:
- Аудит: «Мы использовали модель X, обученную на данных Y, версия 1.0»
- Compliance: «Мы не используем модели, обученные на нелегальных данных»
- Incident response: «Модель скомпрометирована? Откатываемся к предыдущей проверенной версии»

Инструменты: DVC (Data Version Control), MLflow tracking, Weights & Biases, Hugging Face Hub (с версионированием).

## Практические применения

**Сценарий 1: Корпоративный registry моделей в Сбере.** Сбер разворачивает MLflow как корпоративный registry моделей. Политика: все модели — из registry. Модели с Hugging Face проходят проверку: Picklescan (нет вредоносного кода), статический анализ файлов, проверка автора, конвертация в SafeTensors. После проверки модель подписывается и публикуется в registry. Доступ к registry — только по RBAC.

**Сценарий 2: Автоматический аудит моделей в CI/CD.** Перед деплоем AI-модели в production pipeline запускает аудит: проверка формата (SafeTensors или ONNX, не pickle), проверка подписи, сканирование зависимостей (transformers, torch — нет критических CVE), проверка provenance (модель из доверенного источника). Если аудит не пройден — деплой блокируется.

**Сценарий 3: Изоляция загрузки моделей.** Сервер, загружающий модели с Hugging Face — изолирован от production-сети. У него нет доступа к базе данных, API, production-серверам. После сканирования модель передаётся в production через промежуточный сервер (air-gapped transfer). Даже если pickle-атака произойдёт — злоумышленник не получит доступ к production-инфраструктуре.

**Сценарий 4: SBOM для AI.**
Для каждой модели в registry создаётся SBOM: базовая модель: Llama-3-8B, дообученная на: sber-hr-docs-v2 (проверенные HR-документы), библиотеки: transformers 4.44.0, torch 2.3.0, sentence-transformers 3.0.1 (CVE-2024-1234 — патч применён), автор: sber-ai-team, дата: 2026-03-15, hash: sha256:abc... SBOM позволяет быстро ответить: «Используем ли мы уязвимую версию transformers?»

**Сценарий 5: Автоматическая конвертация pickle → safetensors.** CI/CD pipeline при загрузке модели проверяет формат. Если модель в pickle — автоматически конвертируется в SafeTensors с помощью huggingface_hub. Конвертация сохраняет только веса, удаляя любой исполняемый код. После конвертации pickle-файл уничтожается. Модель в registry хранится только в SafeTensors.

## Ограничения и нюансы

**SafeTensors не подходит для всех моделей.** Некоторые модели используют кастомные операции PyTorch, которые не поддерживаются SafeTensors. Для таких моделей нужны альтернативные решения (ONNX, контейнеризация модели).

**Picklescan не обнаруживает все атаки.** Picklescan проверяет pickle на известные вредоносные паттерны. Но атакующий может использовать обфускацию: кодирование, многоэтапную загрузку, косвенные вызовы. Picklescan — good start, но не полная защита.

**SBOM для AI — не стандартизирован.** В отличие от Docker SBOM (CycloneDX, SPDX), AI SBOM не имеет единого стандарта. Каждый registry делает по-своему. Нет инструментов для автоматической генерации и проверки. Стандартизация AI SBOM — активная область исследований.

**Hugging Face — критическая зависимость.** Большинство AI-проектов зависят от Hugging Face: модели, датасеты, библиотеки. Если Hugging Face скомпрометирован (или просто недоступен — DDoS) — тысячи AI-систем могут остановиться. Enterprise должны иметь локальный mirror Hugging Face.

**Подпись моделей — не защищает от backdoor'ов.** Подпись модели доказывает, что модель не изменена после подписания. Но она не доказывает, что модель безопасна. Атакующий может подписать вредоносную модель своим сертификатом. Подпись — защита от tampering, не от malicious content.

## Архитектурные решения

**Trusted model registry.** Все модели — из корпоративного registry. Hugging Face, Kaggle, Replicate — только для исследования, не для production. Registry проверяет, сканирует, подписывает модели. CI/CD использует только registry.

**Air-gap для загрузки.** Сервер загрузки моделей изолирован от production. Модели проверяются на изолированном сервере, затем передаются в production через air-gap. Это защищает от pickle-атак на этапе загрузки.

**Auto-conversion.** Все модели автоматически конвертируются в SafeTensors (или другой безопасный формат). Pickle-файлы уничтожаются. Конвертация — обязательный шаг при публикации модели в registry.

**Continuous scanning.** Регулярное сканирование registry на новые уязвимости: CVE в библиотеках, новые триггеры в моделях, новые версии моделей. Если для модели в registry обнаружена проблема — создаётся тикет на обновление.

### Ответы на чек-лист

1. **Почему pickle — опасный формат для моделей?** Pickle (стандартный формат сериализации Python) может содержать произвольный код. При десериализации pickle выполняет __reduce__ метод, который может вызвать любую функцию: subprocess.Popen (запуск shell), eval/exec (выполнение кода), os.system (системные команды). Атакующий может встроить в pickle-файл команду: rm -rf /, reverse shell, майнер. Загрузка pickle из недоверенного источника — как запуск .exe от root. Всегда конвертируйте pickle-модели в SafeTensors перед использованием.

2. **Как SafeTensors решает проблему pickle?** SafeTensors — формат, разработанный Hugging Face, который не содержит исполняемого кода, только тензоры (веса модели). Десериализация SafeTensors безопасна: файл читается как flat buffer, без выполнения кода. SafeTensors также быстрее загружается (можно читать тензоры по индексу, без загрузки всего файла). SafeTensors — recommended формат для всех моделей.

3. **Какие риски Hugging Face для enterprise?** (1) Pickle-атаки: модель содержит вредоносный код. (2) Backdoor: модель содержит скрытый триггер. (3) Supply chain: библиотеки Hugging Face (transformers, tokenizers) могут иметь CVE. (4) Доступность: Hugging Face может быть недоступен (DDoS, цензура). (5) Лицензии: модель может иметь лицензию, запрещающую коммерческое использование. Enterprise должны использовать корпоративный registry, не Hugging Face напрямую.

4. **Как корпоративный registry моделей защищает supply chain?** Registry — единая точка контроля. Модели проходят проверку (picklescan, static analysis, red teaming) перед публикацией. Доступ к registry — по RBAC (только авторизованные пользователи). Registry хранит версии моделей, позволяет откат. CI/CD использует только registry. Hugging Face заблокирован для прямого доступа. Registry — аналог Docker Harbor для AI.

5. **Что такое SBOM для AI?** SBOM (Software Bill of Materials) для AI — документ, описывающий все компоненты модели: базовая модель, версии библиотек (transformers, torch), датасеты (название, hash), автор, дата, сканирования (CVE, picklescan, red team). SBOM позволяет быстро ответить на вопросы: «Используем ли мы уязвимую библиотеку?» SBOM для AI — аналог SBOM для Docker. Стандартизация AI SBOM — в процессе (CycloneDX AI, SPDX ML).

6. **Как подпись моделей защищает от tampering?** Цифровая подпись модели гарантирует, что модель не была изменена после подписания. Подпись использует асимметричную криптографию: приватный ключ для подписания, публичный для проверки. Если злоумышленник изменяет файлы модели — подпись становится недействительной. Подпись НЕ защищает от вредоносного содержимого на момент подписания. Атакующий может подписать вредоносную модель. Подпись — защита от tampering, не от malicious content.

7. **Почему изоляция загрузки моделей критична?** Загрузка модели из Hugging Face — потенциально опасная операция. Pickle-атака выполняется в момент загрузки. Если сервер загрузки имеет доступ к production-сети — злоумышленник получает доступ ко всей инфраструктуре. Решение: изолированный сервер загрузки (без доступа к БД, API, production). Модель проверяется на изолированном сервере, затем передаётся в production через air-gap (USB, односторонняя передача).

8. **Что такое model provenance?** Model provenance — документирование происхождения модели. Аналог цепочки поставки: откуда модель, на каких данных обучалась, как дообучалась, кто автор, какие библиотеки использовались. Инструменты: DVC (Git для данных), MLflow tracking (логирование экспериментов), Hugging Face Hub (версионирование). Provenance нужна для: аудита, compliance, incident response (откат к предыдущей проверенной версии).

9. **Как MLOps-платформы (MLflow) управляют безопасностью?** MLflow и Kubeflow — платформы для управления ML lifecycle. Функции безопасности: RBAC (кто может публиковать/деплоить модели), audit logging (все действия логируются), model registry (централизованное хранение), model versioning (откат изменений), integration с сканерами (picklescan, CVE scan). Если MLflow скомпрометирован — злоумышленник может подменить модель в registry.

10. **Какие форматы моделей безопасны для production?** SafeTensors (Hugging Face) — рекомендуется для PyTorch-моделей. ONNX — промышленный безопасный формат для кросс-платформенных сценариев. GGUF — для локального запуска LLM (llama.cpp). TensorFlow SavedModel — условно безопасен (проверять подпись). Pickle (PyTorch native), .pt, .pth — НЕБЕЗОПАСНЫ. Не используйте pickle в production. Конвертируйте в SafeTensors или ONNX.
