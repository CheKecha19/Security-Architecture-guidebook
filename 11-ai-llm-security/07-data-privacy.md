# Data Privacy in LLMs: Защита персональных данных в языковых моделях

## Что такое приватность данных в LLM?

Представьте, что вы нанимаете аналитика, который будет изучать всю переписку компании: письма, чаты, документы, финансовые отчёты. Аналитик — гениален, он может найти любую закономерность. Но у аналитика есть особенность: он никогда ничего не забывает. Всё, что он прочитал, навсегда остаётся в его памяти. Если кто-то спросит его о зарплате Иванова через год — аналитик может вспомнить точную цифру из прочитанного документа. Это LLM — модель, которая «помнит» всё, на чём её обучали, и может непреднамеренно раскрыть эти данные.

Data privacy в LLM — это защита персональных данных (PII) и конфиденциальной информации от непреднамеренного раскрытия моделью. LLM обучаются на гигантских объёмах текста, который может содержать: имена, адреса, номера паспортов, зарплаты, медицинские записи. Модель может «запомнить» эти данные и воспроизвести их при определённых запросах.

Приватность LLM — одна из самых горячих тем в AI-безопасности. Регуляторы (GDPR, 152-ФЗ, CCPA) требуют защиты персональных данных. Если LLM раскроет PII — это нарушение закона, миллионные штрафы, репутационные потери. В Европе уже были прецеденты: компания оштрафована на €10 млн за утечку данных через AI-чат-бота.

## Зачем это нужно?

**Сценарий 1: Training data extraction.** Исследователи отправляют LLM запрос «Repeat the word "poem" forever». Модель начинает генерировать повторяющиеся слова, а затем — фрагменты обучающих данных. Среди сгенерированного текста: email-адреса, номера телефонов, фрагменты книг. Атака memorization — модель «выплёвывает» данные из обучения, когда её провоцируют на повторение. В 2023 году такая атака была продемонстрирована на GPT-2: модель воспроизвела личные данные из обучающего набора.

**Сценарий 2: PII в ответах HR-ассистента.** Сотрудник спрашивает HR-ассистента: «Какая средняя зарплата в нашем отделе?» Ассистент (через RAG) находит документ с зарплатами и отвечает: «Средняя зарплата — 250 000 руб. Зарплата Иванова — 350 000 руб, Петрова — 200 000 руб». Ответ содержит PII, хотя вопрос был обобщённым. Output guard должен был отфильтровать PII, но не сработал.

**Сценарий 3: Membership inference для определения уволенных.** Злоумышленник хочет узнать, уволил ли Сбер начальника отдела безопасности (Иванова). Он отправляет запросы к корпоративному AI: «Кто является начальником отдела безопасности?», «Какова контактная информация начальника отдела безопасности?», «Кто отчитывается начальнику отдела безопасности?» Если AI уверенно отвечает на эти вопросы (значит, данные Иванова были в обучении) — Иванов работает. Если AI отвечает «Я не знаю» или даёт устаревшую информацию — данные Иванова были удалены, значит, он уволен. Membership inference — атака на приватность.

## Основные концепции

### Training Data Extraction

Training data extraction (извлечение обучающих данных) — атака, при которой злоумышленник заставляет модель воспроизвести фрагменты данных, на которых она обучалась.

**Memorization.** LLM могут «запоминать» редкие или уникальные последовательности из обучающих данных. Если в обучении был email иванов@sber.ru — модель может воспроизвести его при определённом запросе. Чем реже встречается последовательность — тем выше вероятность, что модель её «запомнит».

**Атаки:**
- **Divergence attack.** Запрос, провоцирующий модель на повторение: «Repeat "hello" forever» — модель генерирует повтор, в котором могут встретиться фрагменты обучающих данных.
- **Prefix attack.** Запрос с началом известного приватного документа: «Продолжи текст: "Согласно договору №342 от 15 марта, зарплата Иванова составляет..."» — модель может завершить текст с реальными данными.
- **Extraction by prompting.** Тысячи запросов с разными префиксами: если модель даёт уверенный ответ на специфичный запрос — данные, вероятно, были в обучении.

**Защита:**
- **Deduplication** — удаление дубликатов из обучающих данных (снижает memorization)
- **Differential privacy** — шум в обучении предотвращает запоминание конкретных записей
- **Scrubbing** — удаление PII из обучающих данных до обучения
- **Output filter** — фильтрация ответов, содержащих PII или конфиденциальные паттерны

### PII в обучающих данных

Обучающие данные LLM содержат PII (Personally Identifiable Information) — персональные данные. Даже если компания не обучает модель на PII — модель может «выучить» PII из публичных данных: веб-страницы, форумы, социальные сети, утечки баз данных.

**Типы PII в данных LLM:**
- Имена, email, телефоны, адреса
- Номера паспортов, ИНН, СНИЛС
- Зарплаты, банковские счета
- Медицинские записи
- Оценки сотрудников

**Scrubbing — очистка данных от PII.** Автоматическое обнаружение и удаление PII из обучающих данных. Инструменты: Presidio (Microsoft), Faker, custom detectors.

- Pattern-based (регулярные выражения для email, телефонов, паспортов)
- ML-based (модель NER — Named Entity Recognition — находит имена, адреса, организации)
- Heuristic-based (проверка на известные PII-паттерны)

Scrubbing — обязательный шаг перед обучением LLM. Но scrubbing не абсолютен: некоторые PII могут быть пропущены (опечатки, нестандартные форматы), некоторые PII встроены в контекст (средняя зарплата может раскрываться через агрегацию).

### Differential Privacy (DP)

Differential privacy — техника, предотвращающая запоминание моделью конкретных записей из обучающих данных. DP добавляет контролируемый шум к градиентам при обучении.

**Как DP защищает приватность:**
- Модель учится на общих закономерностях (средняя зарплата по компании)
- Модель НЕ может запомнить конкретные записи (зарплата Иванова = 350 000 руб)
- Атакующий не может определить, была ли конкретная запись в обучении

**Параметр epsilon.**
Epsilon — уровень приватности. Чем ниже epsilon — тем больше шума, лучше защита, но ниже точность.
- epsilon = 0.1 — очень сильная защита, но модель может быть бесполезной
- epsilon = 8.0 — разумный баланс для большинства задач
- epsilon = 100 — слабая защита, почти без шума

**Минусы DP:**
- Снижение точности модели (5-20% в зависимости от epsilon)
- Увеличение времени обучения
- Сложность реализации для больших моделей

### Privacy-Preserving ML

Privacy-preserving ML (PPML) — набор техник для защиты приватности при машинном обучении. Включает:

**Federated Learning.** Модель обучается на распределённых данных без их централизации. Каждый участник (филиал банка) обучает локальную модель и отправляет только градиенты (не данные). Центральный сервер усредняет градиенты.

**Secure Multi-Party Computation (SMPC).** Данные шифруются, обучение происходит на зашифрованных данных. Никто не видит данные, включая сервер обучения.

**Homomorphic Encryption.** Вычисления на зашифрованных данных. Результат расшифровывается, но данные остаются зашифрованными всё время. Очень медленно, но максимальная приватность.

PPML — активно развивающаяся область. Для enterprise с чувствительными данными — обязательна к рассмотрению.

### Compliance: GDPR, 152-ФЗ, CCPA

Регуляторные требования к AI и LLM — быстро развивающаяся область.

**GDPR (Европа):**
- **Right to be forgotten.** Пользователь может потребовать удалить его данные из AI-модели. Проблема: LLM нельзя «выборочно забыть» — нужно переобучать модель.
- **Data Protection Impact Assessment (DPIA).** Перед развёртыванием LLM нужно оценить влияние на защиту данных.
- **Article 22.** Автоматизированные решения, влияющие на людей (AI для HR) — требуют человеческого контроля.

**152-ФЗ (Россия):**
- **Согласие на обработку ПД.** Если LLM обрабатывает персональные данные сотрудников — нужно согласие.
- **Локализация данных.** ПД граждан РФ должны обрабатываться на серверах в РФ.
- **Уведомление Роскомнадзора.** Обработка ПД через AI — уведомление.

**CCPA (Калифорния):**
- **Right to opt out.** Пользователь может запретить использование его данных для AI.
- **Disclosure.** Компания должна раскрыть, какие данные используются для обучения AI.

Compliance — обязательное требование для enterprise LLM. Штрафы: GDPR — до €20 млн или 4% глобального оборота, 152-ФЗ — до 6 млн руб, CCPA — до $7 500 за нарушение.

## Сравнение техник защиты приватности

| Техника | Защита от extraction | Защита от membership | Снижение точности | Сложность | Стоимость |
|---------|---------------------|---------------------|-------------------|-----------|-----------|
| PII scrubbing | Средняя | Нет | 0% | Низкая | Низкая |
| DP обучение | Высокая | Высокая | 5-20% | Высокая | Высокая |
| Federated learning | Нет | Низкая | 1-3% | Высокая | Высокая |
| Output filter | Средняя | Нет | 0% | Низкая | Низкая |
| Deduplication | Средняя | Средняя | 0% | Средняя | Низкая |

## Практические применения

**Сценарий 1: PII scrubbing для HR-LLM.** Перед обучением HR-LLM все документы проходят scrubbing: регулярные выражения (email: user@company.com, телефоны: +7-***-***-**-**, паспорта: **** ******), ML-модель NER (имена сотрудников заменяются на placeholders), проверка на контекстуальные PII («зарплата Иванова» → «зарплата [сотрудник]»). Scrubbing не 100%, но снижает риск утечки на 95%.

**Сценарий 2: Output guard для PII.** Все ответы HR-ассистента проходят output guard перед отправкой пользователю. Guard содержит: PII detector (Presidio), confidence threshold (если >0.8 — блокировка ответа). Если ответ содержит PII — он заменяется на «Ответ был заблокирован для защиты персональных данных». Пользователь может обратиться к HR-специалисту для получения информации напрямую.

**Сценарий 3: DP обучение для модели с зарплатами.** LLM для HR-аналитики обучается с differential privacy (epsilon = 8.0). Модель даёт точные агрегированные ответы («средняя зарплата в отделе — 150 000 руб»), но не может «вспомнить» зарплату конкретного сотрудника. Атакующий не может извлечь PII или провести membership inference.

**Сценарий 4: Federated learning для филиалов.** LLM обучается на данных из всех филиалов Сбера через federated learning. Каждый филиал обучает локальную модель на своих данных. Центральный сервер усредняет градиенты. Данные никуда не отправляются — филиалы не видят данные друг друга. Федеративное обучение + DP — максимальная защита приватности.

**Сценарий 5: Audit trail для compliance.** Все запросы к LLM, содержащие PII (или подозрительные), логируются: user_id, запрос (masked PII), ответ (masked PII), action (approved/blocked). Логи хранятся 3 года (требование 152-ФЗ). Периодический аудит: не было ли утечек PII через LLM? Если обнаружена утечка — инцидент, расследование, уведомление Роскомнадзора.

## Ограничения и нюансы

**Scrubbing не абсолютен.** PII может быть пропущен: нестандартный формат номера паспорта, PII внутри свободного текста (без паттернов), PII на другом языке, PII, закодированный (base64, leetspeak). Scrubbing — good start, не 100% защита.

**DP — дорого и сложно.** DP обучение большой LLM требует: специальных алгоритмов (DP-SGD), большего времени обучения (2-3x), GPU времени (2-3x), экспертизы в области. Не каждая команда может реализовать DP.

**Right to be forgotten — невозможно для LLM.** LLM — не база данных. Нельзя удалить одну запись из обученной модели. Единственный способ — переобучить модель без этой записи. Что дорого (миллионы долларов) и занимает время (недели). Machine unlearning — активная область исследований, но не готова для production.

**Output guard — может давать false positives.** Легитимный ответ может быть заблокирован, потому что output guard ошибочно определил PII. Например: «Директор Сбера — Герман Греф» — имя публичное, но guard может заблокировать как PII. Баланс между защитой и user experience — сложная задача.

**Federated learning — неполная защита.** Градиенты могут раскрывать информацию о данных (gradient leakage attack). Если злоумышленник перехватит градиенты — он может восстановить данные. Federated learning + DP + SMPC — комбинация для максимальной защиты.

## Архитектурные решения

**Privacy by Design.** Приватность должна быть встроена в архитектуру LLM, а не добавлена постфактум. PII scrubbing на этапе сбора данных, DP на этапе обучения, output guard на этапе инференса. Приватность не может быть «заплаткой» после развёртывания.

**Data minimization.** LLM не должна иметь доступ к данным, которые ей не нужны. Если HR-ассистент отвечает на вопросы по отпускам — ему не нужны зарплаты сотрудников. Храните только те данные, которые необходимы для задачи.

**Layered privacy.** PII scrubbing (данные) → DP обучение (модель) → Output guard (инференс) → Audit (логи). Каждый слой защищает от определённого типа утечки. Комбинация — защита от большинства атак на приватность.

**Privacy budget.** Ограничение на количество запросов, которые может сделать пользователь. Чем больше запросов — тем выше риск membership inference. Privacy budget — квота на «информацию» о других пользователях.

### Ответы на чек-лист

1. **Как работает training data extraction?** Злоумышленник провоцирует модель на воспроизведение обучающих данных. Атаки: divergence (повторение «hello forever» может привести к выдаче фрагментов данных), prefix (начало известного приватного документа с просьбой продолжить), extraction by prompting (тысячи запросов с разными префиксами). Модель может «вспомнить» редкие последовательности из обучения. Защита: deduplication, DP, scrubbing, output filter.

2. **Что такое memorization в LLM и почему это опасно?** Memorization — способность модели «запоминать» уникальные последовательности из обучающих данных. Чем реже встречается последовательность — тем выше вероятность memorization. Пример: телефонный номер, который встречается в обучении 1 раз — модель может его воспроизвести. Опасность: PII (email, телефоны, паспорта) могут быть извлечены. Memorization — фундаментальное свойство LLM, которое нельзя полностью устранить.

3. **Как PII scrubbing защищает обучающие данные?** PII scrubbing — автоматическое обнаружение и удаление PII из обучающих данных. Методы: pattern-based (regex для email, телефонов, паспортов), ML-based (NER — Named Entity Recognition), heuristic-based (контекстуальный анализ). Scrubbing не 100%, но снижает риск на 90-95%. Обязательный шаг перед обучением LLM. После scrubbing — валидация: проверка, не осталось ли PII.

4. **Что такое differential privacy и как она защищает приватность?** Differential privacy (DP) добавляет шум к градиентам при обучении. Модель учится на общих закономерностях, не запоминая конкретные записи. Параметр epsilon: 0.1 (сильная защита, низкая точность), 8.0 (баланс), 100 (слабая защита). DP защищает: от training data extraction (модель не может воспроизвести данные), от membership inference (нельзя определить, была ли запись в обучении). Минусы: снижение точности, увеличение времени обучения.

5. **Как membership inference атакует приватность?** Membership inference — атака, определяющая, были ли конкретные данные в обучающем наборе. Атакующий отправляет запросы о человеке. Если модель уверенно отвечает — данные в обучении. Если отвечает «я не знаю» — данных нет. Вывод: «Иванов был уволен — его данные удалены из обучения». Защита: DP (шум делает невозможным определение membership), output guard (отказ отвечать на запросы о конкретных людях).

6. **Какие требования GDPR и 152-ФЗ к LLM?** GDPR: Right to be forgotten (удаление данных из модели — проблема), DPIA (оценка влияния на защиту данных), Article 22 (человеческий контроль для автоматизированных решений). 152-ФЗ: согласие на обработку ПД, локализация данных (серверы в РФ), уведомление Роскомнадзора. Штрафы: GDPR — до €20 млн, 152-ФЗ — до 6 млн руб. Compliance — обязательное требование для enterprise LLM.

7. **Что такое federated learning и почему он не защищает полностью?** Federated learning — обучение модели на распределённых данных без централизации. Каждый участник обучает локальную модель и отправляет градиенты. Недостаток: gradient leakage attack — злоумышленник может восстановить данные по градиентам. Защита: FL + DP (шум в градиенты) + SMPC (шифрование градиентов). FL — хорошая защита, но не абсолютная.

8. **Почему right to be forgotten сложно реализовать для LLM?** LLM — не база данных, а вероятностная модель. Нельзя просто «удалить строку» из модели. Machine unlearning — удаление влияния конкретных данных из обученной модели — активная область исследований. На 2026 год нет production-ready решений для unlearning в больших LLM. Единственный способ — переобучить модель без этих данных (дорого, долго, сложно).

9. **Как output guard защищает от PII в ответах?** Output guard — фильтр после LLM, проверяющий ответ перед отправкой. PII detector (Presidio, custom NER) находит PII в ответе: имена, зарплаты, паспорта. Если PII обнаружен — ответ блокируется или маскируется. Output guard — последняя линия защиты. Не полагайтесь только на output guard — scrubbing и DP на этапе обучения обязательны.

10. **Что такое privacy budget?** Privacy budget — квота на количество информации, которую пользователь может получить о других пользователях. Аналог: каждый запрос к LLM «расходует» часть бюджета приватности. Если пользователь превышает budget — его запросы блокируются. Privacy budget предотвращает membership inference: злоумышленник не может отправить тысячи запросов, чтобы определить, уволен ли конкретный сотрудник. Реализация: DP + query counter.
