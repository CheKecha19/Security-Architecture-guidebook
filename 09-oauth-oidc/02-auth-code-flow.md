# Authorization Code Flow (OAuth 2.0)

## Краткое описание
**Authorization Code Flow** — это классический сценарий «три шага», используемый веб‑приложениями для получения доступа к защищённым ресурсам от имени пользователя без прямой передачи пароля. Поток разбит на взаимодействие **клиента**, **authorization server** и **resource server**, а также предусматривает передачу *code* и *state* для защиты от атак **CSRF** и **redirect‑uri hijacking**.

## Эволюция и мотивация
1. **Ранние версии OpenID & OAuth 1.0** – требовали подписи каждого запроса, что усложняло внедрение в браузерные приложения.
2. **OAuth 2.0 (RFC 6749, 2012)** ввёл упрощённый «authorization code», позволив клиенту получать токен через сервер‑бэкенд, тем самым скрыв секреты от браузера.
3. **PKCE (RFC 7636, 2017)** – расширение для публичных клиентов (мобильные, SPA), предотвращающее перехват *code* в публичных средах.
4. **Современные best‑practice** – строгая валидация *redirect_uri*, обязательный *state* и *nonce* в OpenID Connect.

## Основные концепции
- **Resource Owner** – пользователь, владелец защищённых ресурсов.
- **Client** – приложение (confidential или public), которое запрашивает доступ.
- **Authorization Server** – выдаёт *authorization code* и *access token*.
- **Resource Server** – хранит защищённые API, проверяет *access token*.
- **Redirect URI** – заранее зарегистрированный URL, куда сервер отправляет пользователя после авторизации.
- **State Parameter** – случайная строка, возвращаемая сервером, защищающая от CSRF.
- **Code Challenge / Verifier (PKCE)** – SHA‑256‑проверификация, полезна для публичных клиентов.

### Поток «Authorization Code» (с PKCE)
1. **Инициация** – клиент формирует запрос `GET /authorize?...&code_challenge=...&code_challenge_method=S256&state=xyz` и перенаправляет пользователя.
2. **Аутентификация и согласие** – пользователь логинится на **authorization server** и одобряет запрашиваемые scopes.
3. **Редирект с кодом** – сервер перенаправляет браузер на `redirect_uri?code=ABC123&state=xyz`.
4. **Обмен кода на токен** – клиент (обычно сервер‑бэкенд) отправляет `POST /token` с `code`, `code_verifier`, `client_id`/`client_secret`.
5. **Получение access/refresh token** – сервер выдаёт JSON‑ответ с токенами, которые клиент использует для запросов к **resource server**.

## Архитектурные решения
| Паттерн | Плюсы | Минусы |
|---|---|---|
| **Confidential Client (server‑side)** | Секрет хранится безопасно, поддержка refresh‑token | Требует серверную инфраструктуру |
| **Public Client + PKCE** | Не хранит секрет, подходит для SPA и мобильных приложений | Усложнённая проверка `code_challenge` |
| **Hybrid Flow (code + id_token)** | Позволяет сразу получить ID‑token в браузере | Увеличивает поверхность атаки, нужен строгий `state` |

### Практический вывод
- При проектировании всегда регистрируйте **только** необходимые `redirect_uri` и проверяйте их строго (полное совпадение, без wildcard).
- Обязательно генерируйте **cryptographically‑random** `state` и привязывайте его к пользовательской сессии.
- Для публичных клиентов включайте **PKCE**; без него уязвимы к *authorization code interception*.

## Ограничения и нюансы
- **Redirect‑uri hijacking** – возможен, если приложение принимает произвольные URL‑параметры (например, `?next=`) и добавляет их к `redirect_uri` без проверки.
- **CSRF через state** – если `state` фиксирован или предсказуем, атакующий может инициировать запрос от имени жертвы.
- **Код‑обмен без проверки клиентского секрета** – в публичных клиентах это опасно; обязательна проверка `code_verifier`.
- **Сброс токена после смены пароля** – некоторые реализации не отзывают refresh‑токены, создавая «запасные» пути доступа.

## Три вопроса с развернутыми ответами (для интервью)
1. **Почему Authorization Code Flow считается более безопасным, чем Implicit Flow?**
   - В Implicit Flow токен передаётся в `fragment` URL и может быть прочитан клиентским JavaScript‑кодом, открывая возможность XSS‑крана. В Authorization Code Flow токен передаётся серверу через защищённый канал, а клиент получает только одноразовый *code*, который без `code_verifier` бесполезен.
2. **Как правильно защититься от redirect‑uri hijacking?**
   - Регистрация строго фиксированных `redirect_uri` в консоли провайдера; проверка полного совпадения строки на сервере; запрет wildcard‑доменов; токен‑binding (`iss`, `aud`) в ID‑token; использование `PKCE` дополнительно связывает запрос с клиентом.
3. **Что происходит, если злоумышленник подменит параметр `state`?**
   - Если клиент не проверит соответствие `state` с сохранённым в сессии, он примет ответ от чужого пользователя, позволяя выполнить **login‑CSRF**: пользователь может быть автоматически авторизован в атакующем аккаунте.

## Ключевые выводы для собеседования
- **Authorization Code Flow** — «посредник», который защищает пользовательские креденшелы, передавая лишь одноразовый *code*.
- **State** — главное средство защиты от CSRF; он должен быть случайным и привязанным к сессии.
- **Redirect‑uri** — самый частый источник уязвимостей; строгая валидация и отсутствие wildcard‑правил обязательны.
- **PKCE** — обязательный для публичных клиентов; без него атакующий может перехватить *code*.
- **Инциденты**: Airbnb (2016) – login‑CSRF + open‑redirect, Cobalt (2024) – открытый redirect‑uri, Airbnb (2016) – state‑parameter misuse, Snyk (2026) – insufficient entropy in `state`.
