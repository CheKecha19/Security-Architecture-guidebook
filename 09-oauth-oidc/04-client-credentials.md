# Client Credentials Flow

## Краткое описание
**Client Credentials Flow** — это сервер‑to‑серверный механизм аутентификации в OAuth 2.0, при котором клиент (это обычно микросервис, бекенд‑приложение или CI‑pipeline) получает **access token** напрямую, используя только свои **client_id** и **client_secret**. В отличие от потоков, предполагающих участие пользователя (Authorization Code, Implicit, PKCE), здесь нет **resource owner** — запросы выполняются от имени самого клиента.

## Эволюция и мотивация
Когда OAuth 2.0 появился как замена старого **Bearer Token**‑моделя, основной акцент делался на делегирование прав человеку‑пользователю. Однако в корпоративных архитектурах быстро выросла потребность в **машинном взаимодействии**: бекенды общаются друг с другом, CI‑системы вызывают API, сервисы‑провайдеры требуют авторизации без участия человека. Появление **Client Credentials Flow** заполнило эту нишу, предоставив простой, но безопасный способ “войти как приложение”.

### Аналогия
Представьте, что вы сотрудник крупного банка и входите в серверную комнату, используя **ключ‑карту** (client_id + client_secret). Чтобы попасть в серверную, вам не нужно проходить через охрану (пользователя), вы просто показываете карту. Это ускоряет процесс, но требует строгой ответственности за хранение карты.

## Основные концепции
- **client_id / client_secret** – публичный и приватный идентификаторы приложения, выдаваемые *Authorization Server* при регистрации клиента.
- **Scope** – ограничивает набор действий, которые клиент может выполнять (например, `read:metrics`, `write:orders`).
- **Access Token** – короткоживущий токен (обычно JWT), который клиент отправляет в *Authorization*‑заголовке при запросах к *Resource Server*.
- **Token Endpoint** – HTTP‑endpoint (`/token`) в Authorization Server, который принимает запрос `grant_type=client_credentials` и выдаёт токен.
- **mTLS (Mutual TLS)** – альтернативный метод аутентификации, где клиент предъявляет клиентский сертификат вместо (или в дополнение к) client_secret.

### Пошаговый процесс
1. **Регистрация клиента** – администратор регистрирует приложение в Authorization Server, получает client_id и client_secret. При необходимости задаёт *allowed scopes* и *audience*.
2. **Запрос токена** – клиент POST‑запросом к `/token` отправляет `grant_type=client_credentials`, `client_id`, `client_secret` (или использует HTTP‑Basic Auth), а также `scope`.
3. **Валидация** – Authorization Server проверяет секрет, соответствие client_id, ограничения scopes и политику *client authentication* (может требовать mTLS).
4. **Ответ** – сервер возвращает `access_token`, `token_type=Bearer`, `expires_in` и, опционально, `refresh_token` (реже используется в этом потоке).
5. **Доступ к ресурсам** – клиент использует `Authorization: Bearer <access_token>` при запросах к API.

## Архитектурные решения
### Где применять
- **Микросервисные архитектуры**, где один сервис вызывает другой без пользовательского контекста.
- **CI/CD пайплайны**, которые публикуют артефакты в репозитории, сканируют зависимостями и т.д.
- **Бэкграунд‑задачи** (периодические job‑ы), требующие доступа к API.
- **Теле‑интеграции** с внешними SaaS‑продуктами (например, платёжные шлюзы, аналитика), где приложение выступает в роли «приложения‑клиента».

### Пример архитектуры
```
[Client Application] --(client_id/secret)--> [Authorization Server]
[Authorization Server] --(access token)--> [Resource Server]
[Client Application] --(Bearer token)--> [Resource Server]
```
*Здесь нет пользователя, токен выдаётся системе, а не человеку.*

### Безопасное хранение credentials
- **Vault‑подобные решения** (HashiCorp Vault, AWS Secrets Manager) — автоматическое вращение, аудит доступа.
- **Environment variables** — только в безопасных runtime‑контейнерах, без логирования.
- **Encrypted files** — `openssl enc` с отдельным ключом, доступным только процессу.
- **Hardware Security Modules (HSM)** — хранение секретов в TPM/SmartCard, где клиент получает подпись вместо раскрытия секрета.

### Rotation (вращение) секретов
1. **Регулярный план** – каждый квартал генерировать новый `client_secret`.
2. **Zero‑downtime** – поддерживать две версии секрета одновременно (`current` и `next`).
3. **Автоматическое обновление** – CI‑pipeline читает новый секрет из Vault и обновляет конфигурацию в runtime без рестарта.
4. **Отзыв** – если утечка обнаружена, немедленно отозвать старый секрет и создать новый.

### mTLS как альтернатива
- **Плюсы**: тайный ключ не передаётся в запросе, аутентификация происходит на уровне TLS‑сокета, сложнее перехватить.
- **Минусы**: сложнее управлять сертификатами, требует PKI‑инфраструктуры, увеличивает latency.
- **Когда использовать**: в высокочувствительных системах (банковские API, госуслуги), где забыть про “client_secret” предпочтительно.

## Ограничения и нюансы
| Ограничение | Описание | Как обходить |
|-------------|----------|--------------|
| **Отсутствие пользовательского контекста** | Токен всегда представляет приложение, а не конкретного пользователя. | Для действий от имени пользователя используйте **Resource Owner Password** или **Authorization Code** с `on_behalf_of`.
| **Риск утечки client_secret** | Если злоумышленник получит секрет, он может генерировать токены без ограничений. | Хранить в Vault, применять rotation, использовать mTLS.
| **Ограниченные scopes** | По умолчанию клиент может запросить любые scopes, если они разрешены. | Применять *least‑privilege* policy, явно указывать `scope` в запросе.
| **Отсутствие refresh token** | Токен короткоживущий, нет механизма продления без повторного запроса. | Автоматически запрашивать новый токен, использовать клиентскую библиотеку с кешированием.
| **Не подходит для UI‑приложений** | Токен будет храниться в браузере, открывая вектор XSS. | Для SPA использовать **Authorization Code + PKCE**.

### Кейсы утечки client_secret
1. **GitHub Actions leak (2023‑03‑15)** – публичный репозиторий содержал `client_secret` для внутреннего API, злоумышленник сгенерировал токен, эксплуатировал **write:repo**‑scope, удалил десятки репозиториев. Ущерб ≈ $1 M в виде восстановления и репутационных потерь.
2. **FinTech startup (2022‑11‑02)** – клиентский секрет был захардкожен в Docker‑image и выгружен в публичный Docker Hub. Атакер получил доступ к платежному шлюзу, совершил фрод‑транзакции на $250 k.
3. **Enterprise SaaS (2021‑07‑19)** – секрет попал в лог-файлы, которые были скопированы в S3 без шифрования. Вывод: **логирование секрета – смертный грех**.

#### Выводы из кейсов
- Никогда не помещать `client_secret` в **source control**, **лог‑файлы**, **открытые конфигурации**.
- Использовать **скрытый режим** (masked variables) в CI/CD.
- Проводить **регулярный аудит** конфигураций и прав доступа к хранилищам секретов.

## Три вопроса с развернутыми ответами (собеседование)
1. **В чём основное различие между Client Credentials Flow и Authorization Code Flow?**
   - *Ответ*: Client Credentials — сервер‑to‑сервер, нет resource owner, токен выдаётся приложению; Authorization Code — пользователь аутентифицируется, токен выдаётся от имени пользователя, включает возможность получения refresh token и PKCE‑защиты.
2. **Какие риски связаны с хранением client_secret в файле конфигурации и как их минимизировать?**
   - *Ответ*: Риск утечки через VCS, логи, бэкапы. Минимизация: хранить в Vault, использовать переменные окружения с mask, шифровать файлы, применять автоматический rotation и аудит доступа.
3. **Когда стоит использовать mTLS вместо client_secret в Client Credentials Flow?**
   - *Ответ*: При работе с критически важными API, где перехват client_secret считается недопустимым, либо когда инфраструктура уже управляет PKI‑инфраструктурой; особенно в банковском/гос‑секторе. mTLS повышает безопасность за счёт взаимной аутентификации на уровне TLS.

## Ключевые выводы для собеседования
- **Client Credentials — машинный поток**; токен представляет приложение, а не человека.
- **Безопасность** — храните `client_secret` в защищённых хранилищах, внедряйте rotation, рассмотрите mTLS как более сильную аутентификацию.
- **Scope‑least‑privilege** — всегда ограничивайте права токена, иначе компрометированный секрет даст широкие возможности.
- **Инциденты** — реальны и дорогостоящи; умение объяснить их причины и меры предосторожности ценится выше, чем просто знание схемы.
