# PKCE (Proof Key for Code Exchange)

## Краткое описание
**PKCE** — расширение протокола OAuth 2.0, позволяющее публичным клиентам (мобильным приложениям, SPA, IoT‑устройствам) безопасно использовать поток *Authorization Code* без необходимости хранить *client_secret*.  Оно вводит одноразовый «секрет‑проверка» — *code_verifier* / *code_challenge* — который защищает от перехвата кода авторизации и от подмены клиента.

## Эволюция и мотивация
### Истоки проблемы
В оригинальном *Authorization Code Grant* (RFC 6749, 2012) клиент получает авторизационный код, а затем обменивает его на токен, предоставляя *client_id* и *client_secret*.  Для «конфиденциальных» клиентских приложений (сервер‑сайд) это безопасно, но публичные клиенты (мобильные, SPA) не могут надёжно хранить секрет; любой полученный *client_secret* можно извлечь из двоичного кода или JavaScript‑файлов.  Это создавало уязвимость «**Authorization Code Interception Attack**», подробно описанную в RFC 6819 (2013) и в более ранних исследованиях.

### Появление PKCE
В 2015 году IETF принял RFC 7636 — *Proof Key for Code Exchange by OAuth Public Clients* (PKCE).  RFC‑76 36‑6 фиксирует, что публичные клиенты должны продемонстрировать доказательство владения уникальным секретом, генерируемым **динамически** для каждой авторизационной сессии.  Это радикально изменило модель угроз: даже если злоумышленник перехватит авторизационный код, без *code_verifier* он не сможет обменять его на токен.

### Последующее развитие
* RFC 8252 (2017) — OAuth 2.0 for Native Apps, формализует рекомендации по использованию PKCE в нативных приложениях, вводя безопасные редиректы (`urn:ietf:wg:oauth:2.0:oob` и `custom scheme`).
* 2022 г. OAuth 2.0 Security BCP (https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics) объявил PKCE обязательным для всех публичных клиентов, а *Implicit Grant* объявил устаревшим.
* 2023‑2024 гг. несколько уязвимостей (CVE‑2023‑48228, GHSA‑fm34‑v8xq‑f2c3) показали, что нестандартные реализации могут «недостаточно» проверять связь `code_challenge` ↔ `code_verifier`.  Эти инциденты усилили необходимость **строгой проверки** и **регистрации** методик хэширования (S256).

## Основные концепции
### Термины (жирным)
- **code_verifier** — криптографически случайная строка длиной 43–128 символов (обычно base64url без `=`). Она генерируется клиентом *до* первого обращения к `/authorize`.
- **code_challenge** — производное от `code_verifier`. Может быть либо `plain` (прямая передача) либо `S256` (SHA‑256‑хеш, base64url‑закодированный). Стандарт настоятельно рекомендует `S256` из‑за лучшей энтропии.
- **code_challenge_method** — указатель, какой метод был использован (`plain` или `S256`).
- **authorization_code** — одноразовый код, получаемый от сервера после согласия пользователя.
- **access_token** — токен доступа, используемый клиентом для запросов к ресурсному серверу.

### Пошаговый протокол (без кода)
1. **Генерация `code_verifier`**. Клиент использует криптографически безопасный RNG, выдающий строку с минимум 256 бит энтропии.
2. **Создание `code_challenge`**. Если выбран `S256`, клиент вычисляет SHA‑256 от `code_verifier`, затем base64url‑кодирует результат. При `plain` просто копирует строку.
3. **Запрос авторизации**. Клиент отправляет запрос к `/authorize`, добавляя параметры `code_challenge` и `code_challenge_method`. При этом `client_id` (и, если применимо, `redirect_uri`, `state`, `scope`) тоже передаются.
4. **Получение `authorization_code`**. После согласия пользователя сервер редиректит обратно на `redirect_uri` с параметром `code`.
5. **Обмен на токен**. Клиент POST‑ит к `/token`, передавая `grant_type=authorization_code`, `code`, `redirect_uri` и **`code_verifier`**. Сервер повторно вычисляет хеш и сравнивает с ранее полученным `code_challenge`.
6. **Выпуск `access_token`** (и опционального `refresh_token`). Если проверка прошла, сервер возвращает токен; иначе — ошибка `invalid_grant`.

### Криптографический анализ
- **Энтропия `code_verifier`** — ключевой фактор. При энтропии ниже 128 бит вероятность угадывания растёт экспоненциально. 256 бит считается безопасным даже при полном переборе.
- **Хеш‑коллизии**. При `S256` вероятность коллизий пренебрежимо мала (2⁻¹²⁸). `plain` уязвим к предсказуемым значениям, поэтому рекомендуется только `S256`.
- **TLS**. PKCE не заменяет TLS; всё взаимодействие должно проходить по HTTPS. PKCE защищает лишь *публичный* клиент от перехвата кода в момент, когда TLS уже гарантирует конфиденциальность канала, но может быть нарушен в фазе редиректа (например, если пользователь нажимает «скопировать ссылку» в браузере).

## Архитектурные решения (как применять)
### Публичные клиентские SDK
- **Android** — Google «AppAuth» и «Auth0 Android SDK» автоматически генерируют `code_verifier` и используют `S256`. При интеграции важно включить `usePKCE = true` и убедиться, что `redirect_uri` — `com.myapp://callback` зарегистрирован в манифесте.
- **iOS** — `ASWebAuthenticationSession` + `AppAuth‑iOS` реализуют PKCE без вмешательства разработчика.
- **SPA (React, Angular)** — Библиотеки `oidc-client-js` и `@azure/msal-browser` добавляют `code_challenge_method=S256` к запросу `/authorize`.

### Бэкенд‑ориентированные шлюзы (API‑gateway)
1. **Gateway как клиент**: в микросервисных архитектурах шлюз может выступать в роли публичного клиента, получая токен для дальнейшего взаимодействия с внутренними ресурсами. В этом случае шлюз хранит `client_secret`, но всё равно рекомендуется использовать PKCE как дополнительный уровень защиты от компрометации кода при межслужебных редиректах.
2. **Refresh‑token rotation**: сервер, выдавая `refresh_token`, обязан **ретировать** его после каждой операции, тем самым закрывая окно для повторного использования украденного `refresh_token`.
3. **DPoP (Proof‑of‑Posession) Token**: в комбинации с PKCE DPoP добавляет доказательство владения клиентским ключом, препятствуя replay‑атакам.

### Пример инфраструктурного применения
- **Identity Provider (Keycloak)**
  - Включить *PKCE* в клиентских настройках ("Proof Key for Code Exchange" = *Enabled*).
  - Установить `code_challenge_method` → `S256` как обязательный.
  - Включить *CORS*‑политику, ограничивая `redirect_uri` домены.
- **Auth0**
  - В приложении типа *Native* автоматом включён `PKCE`.
  - При миграции старых клиентов добавить `Allow Legacy` = *Off*.
- **Azure AD B2C**
  - Параметр `client_id` + `code_challenge_method=S256` поддерживается без дополнительных настроек, но рекомендуется включить *Proof Key for Code Exchange (PKCE) required*.

## Ограничения и нюансы (где не работает)
| Сценарий | Ограничения | Почему |
|---|---|---|
| **Клиент без поддержки криптографии** (устройства IoT с ограниченными библиотеками) | Невозможность генерировать безопасный `code_verifier` | Отсутствие RNG → низкая энтропия, уязвимость к перебору |
| **Приложения, использующие `plain`‑method** | Подвержены предсказуемости | Хеш‑функция не применяется, атакующий может подобрать `code_verifier` из известного набора |
| **Публичные API без TLS** | PKCE не спасает от MITM‑атак на канал | Перехват `code_challenge` и `code_verifier` в открытом виде позволяет получить токен |
| **Старые Identity Providers, не поддерживающие PKCE** | Необходимо отключать *Implicit Grant* | Без PKCE авторизационный код может быть украден; без обновления IdP необходимо использовать клиент‑секрет, но это не решает проблему публичных клиентов |
| **Перехват `state` в редиректе** | Атака CSRF может вернуть код злоумышленнику | `state` — дополнительный механизм, но без его проверки PKCE не спасает от полного захвата сессии |

### Практический совет
- **Всегда используйте `S256`.** `plain` допускается только в лабораторных тестах.
- **Проверьте длительность `code_verifier`.** Минимум 43 символа, максимум 128, без пробелов.
- **Сохраняйте `code_verifier` в памяти только до получения токена.** После обмена немедленно удаляйте.
- **Включите строгую проверку `redirect_uri`.** Позвольте только один зарегистрированный URI.
- **Регулярно сканируйте IdP‑платформу на наличие CVE, связанных с PKCE** (CVE‑2023‑48228, GHSA‑fm34‑v8xq‑f2c3).

## Три вопроса с развернутыми ответами (собеседование)
1. **Почему PKCE обязательна для публичных клиентов, а `client_secret` — нет?**
   - Публичные клиенты (мобильные, SPA) не могут надёжно хранить секрет, потому что их бинарники/JS‑код находятся в руках пользователя.  Злоумышленник может декомпилировать приложение, извлечь любой статический секрет и использовать его в собственных запросах к `/token`.  PKCE решает проблему, делая запрос к `/token` зависимым от одноразового `code_verifier`, который генерируется **динамически** и хранится только в памяти.  Даже если атакующий получит `authorization_code`, без `code_verifier` он не сможет обменять его на токен.

2. **Как работает защита от `Authorization Code Interception` в RFC 7636?**
   - При начале авторизации клиент генерирует `code_verifier`.  Затем он вычисляет `code_challenge` (обычно `S256`) и передаёт его серверу в параметре `code_challenge`.  Сервер сохраняет хеш.  После согласия пользователя сервер выдаёт `authorization_code`.  При обмене клиент отправляет оригинальный `code_verifier`.  Сервер повторно вычисляет хеш и сравнивает с ранее сохранённым `code_challenge`.  Если они совпадают, токен выдаётся.  Перехваченный код без `code_verifier` не проходит эту проверку, тем самым защищая от MITM‑атак в фазе редиректа.

3. **Опишите реальный инцидент, связанный с неправильной реализацией PKCE, и какие выводы можно из него сделать.**
   - **CVE‑2023‑48228 (Authentik, 2023‑2024)**: публичный IdP использовал `code_challenge_method=plain` по умолчанию и проверял только наличие `code_challenge`, но не сравнивал его с полученным `code_verifier`.  Злоумышленник смог подменить `code_challenge` в запросе авторизации, получить `authorization_code`, а затем выполнить обмен, передав любой `code_verifier`.  Это привело к получению валидных `access_token` без знания оригинального секрета.  Основные выводы: **нужно явно включать `S256` и строго проверять связь `code_verifier` ↔ `code_challenge`**, а также **проводить регрессионное тестирование всех путей PKCE**.

## Ключевые выводы для собеседования
1. **PKCE = одноразовый доказательство владения**: генерируется `code_verifier`, превращается в `code_challenge`, защищая от перехвата кода.
2. **`S256` — единственный безопасный метод**; `plain` допускается лишь в тестовых средах.
3. **В публичных клиентах PKCE обязательна**; `client_secret` — неприемлем в мобильных/SPA приложениях.
4. **Неправильные реализации (CVE‑2023‑48228 и др.)** показывают, что без строгой проверки `code_verifier` защита падает.
5. **Комбинация PKCE + TLS + проверка `state` + DPoP** формирует современный «defense‑in‑depth» стек для OAuth 2.0.
