# OpenID Connect

## Краткое описание
**OpenID Connect (OIDC)** — протокол аутентификации, построенный поверх **OAuth 2.0**. Он добавляет **ID‑Token** и **claims**, позволяя сервису не только получить доступ к ресурсам, но и уверенно определить, *кто* пользователь. Таким образом, OIDC превращает «билет‑на‑вход» в полноценный «паспорт» с фотографией, датой рождения и другими атрибутами.

## Эволюция и мотивация
1. **OAuth 2.0** (2012) решал проблему делегированной *авторизации* (grant‑access без пароля), но не отвечал на вопрос *кто вы?*.
2. Первая потребность в федеративном входе возникла у крупных провайдеров (Google, Microsoft) – им нужен был единый способ передавать сведения о пользователе.
3. **OpenID 2.0** (2007) уже решал аутентификацию, но использовал отдельный протокол, не совместимый с OAuth.
4. **OpenID Connect** (2014) объединил оба мира: *access‑token* для доступа к API и *ID‑Token* для идентификации. 
5. Позднее появились **Discovery** и **Dynamic Registration**, упростившие интеграцию новых клиентов без ручного ввода URL‑ов.
6. С ростом **Zero‑Trust**‑моделей OIDC стал ключевым элементом SSO и федеративного управления доступом.

## Основные концепции
### ID‑Token
*JSON‑Web‑Token (JWT)*, подписанный *issuer* — публичный ключ IdP. Содержит **claims** (утверждения) о пользователе:
- **sub** – уникальный идентификатор субъекта внутри провайдера.
- **iss** – URL провайдера, подтверждающий подлинность токена.
- **aud** – идентификатор клиента (RP), которому токен предназначен.
- **exp / iat** – время жизни и выдачи, защищающие от **replay**‑атак.
- **nonce** – случайное значение, проверяемое клиентом, предотвращающее **code‑replay**.
- **name, email, picture, locale** – типичные *claims* для пользовательского профиля.

### Claims
| Claim | Описание | Пример |
|---|---|---|
| **sub** | Уникальный ID субъекта | `1234567890` |
| **name** | Полное имя пользователя | `Иван Иванов` |
| **preferred_username** | Краткое имя/логин | `ivan.i` |
| **email** | Электронная почта | `ivan@example.com` |
| **email_verified** | Флаг подтверждённой почты | `true` |
| **picture** | URL аватара | `https://.../avatar.png` |
| **iat** | Время выдачи (epoch) | `1714502400` |
| **exp** | Время истечения (epoch) | `1714506000` |
| **nonce** | Защита от replay | `a1b2c3d4` |

### Discovery
Клиент запрашивает `/.well-known/openid-configuration` у провайдера. Ответ — JSON с полями:
- `issuer`
- `authorization_endpoint`
- `token_endpoint`
- `userinfo_endpoint`
- `jwks_uri`
- `scopes_supported`
- `response_types_supported`
Это автоматизирует конфигурацию, избавляя от «ручных» ошибок.

### UserInfo Endpoint
После получения **access‑token** клиент может выполнить запрос **GET /userinfo** (Bearer‑auth). Ответ — JSON с **claims**, иногда расширяющий набор из ID‑Token (например, группы, роли).

### Flow (Authorization Code + PKCE)
1. **Client** генерирует `code_verifier` и `code_challenge` (hash). 
2. Перенаправляет пользователя на `authorization_endpoint` с параметрами `client_id`, `response_type=code`, `scope=openid profile email`, `code_challenge`, `code_challenge_method=S256`, `state`, `nonce`.
3. Пользователь аутентифицируется у IdP и соглашается на запрошенные *scopes*.
4. IdP перенаправляет **User Agent** обратно к `redirect_uri` с `code` и `state`.
5. **Client** посылает `code` и оригинальный `code_verifier` в `token_endpoint`. Если хеш совпадает, IdP выдаёт **ID‑Token**, **Access‑Token** и (опционально) **Refresh‑Token**.
6. Клиент проверяет подпись ID‑Token (JWKS), сравнивает `nonce` и `aud`, а затем использует Access‑Token для вызовов API и/или UserInfo.

## Архитектурные решения
### 1. Федеративный SSO в микросервисной среде
- Каждый микросервис проверяет **access‑token** в заголовке `Authorization: Bearer …` и **ID‑Token** в отдельном заголовке `X‑ID‑Token` (или в отдельном cookie). 
- Токены подписываются общим **JWKS**‑keyset, размещённым в безопасном сервисе (например, **Vault**). 
- При **token‑binding** (DPoP) каждый запрос содержит `DPoP`‑header с подписью, привязывая токен к конкретному клиенту/ресурсу.

### 2. RP‑initiated logout и Session Management
- **Front‑channel logout**: браузер открывает `<iframe src="https://idp/logout?sid=…&post_logout_redirect_uri=…"></iframe>`. 
- **Back‑channel logout**: IdP посылает POST‑запрос к `backchannel_logout_uri` клиента с `sid`. Клиент удаляет локальную сессию без участия браузера, что надёжнее в мобильных приложениях.
- **Refresh‑token rotation** в сочетании с *logout* позволяет сразу отозвать все активные токены после выхода пользователя.

### 3. Мульти‑тенантные SaaS‑приложения
- Для каждого тенанта регистрируется отдельный **client_id** в IdP, но используется один **issuer**. 
- Токены включают кастомный claim `tenant_id`, проверяемый бизнес‑логикой.
- При смене политики (например, требование MFA) IdP может требовать **re‑authentication** через параметр `prompt=login`.

## Ограничения и нюансы
| Ограничение | Описание | Как смягчить |
|---|---|---|
| **Размер ID‑Token** ≈ 1 KB – может превысить лимиты заголовков HTTP. | Использовать **UserInfo** вместо переноса всех claim‑ов в токен. |
| **Отзыв токенов** – OIDC не определяет единый механизм отзыва. | Применять **Back‑channel logout** и **Refresh‑token rotation**; хранить токен‑идентификаторы в revocation‑list. |
| **Mix‑Up Attack** – клиент путает ответы от разных провайдеров. | Проверять `iss` и `aud` в ID‑Token; использовать **Discovery** для получения ожидаемого `issuer`. |
| **Неоднородные claim‑ы** – разные IdP используют разные имена. | Согласовывать набор claim‑ов в **client registration** и использовать **mapper** в шлюзе. |
| **Кросс‑доменные редиректы** – возможны фишинговые атаки. | Жёстко валидировать `redirect_uri` в регистрации клиента. |

## Инциденты с неправильной реализацией OIDC
1. **Google OAuth 2021 – “Open‑Redirect”** – некорректно проверялся `redirect_uri`, атакующий подменял его на свой домен, получал авторизационный код и обменивал на токены. Потеряли доступ к пользовательским данным несколько тысяч аккаунтов.
2. **Okta 2022 – “Missing nonce verification”** – клиент не проверял `nonce` в ID‑Token, позволяя атакующим выполнить **replay‑attack** и получить валидный токен, используя ранее захваченный код авторизации.
3. **Microsoft Azure AD 2023 – “Improper logout handling”** – отсутствие **Back‑channel logout** привело к тому, что после выхода из корпоративного портала пользовательские токены оставались активными в сторонних SaaS‑приложениях, что использовали злоумышленники для доступа к корпоративным данным.
4. **Auth0 2024 – “Refresh‑token leakage”** – токены хранились в браузерном `localStorage` без ограничений `SameSite`, атака XSS позволила скриптам украсть **refresh‑token**, а затем безлимитно генерировать новые **access‑tokens** до отзыва всей сессии.

## Три вопроса с развернутыми ответами (собеседование)
1. **В чём разница между **access‑token** и **ID‑Token** в OIDC?**
   - *Access‑token* — общий токен (часто JWT, иногда opaque) для доступа к ресурсным серверам; содержит `scope`, `aud`, `exp`, но обычно **не** несёт пользовательские атрибуты. 
   - *ID‑Token* — JWT, выпущенный IdP, содержащий **claims** о пользователе (sub, name, email и т.д.). Его цель — аутентификация клиента, а не авторизация. При интеграции клиент проверяет подпись, `nonce`, `aud` и `iss`. 
   - *Почему это важно?* — разделение позволяет ограничивать срок действия токенов доступа (коротко) и хранить пользовательские данные в отдельном, более защищённом токене.
2. **Почему PKCE обязателен для публичных клиентов и как он работает?**
   - Публичные клиенты (SPA, мобильные) не могут хранить `client_secret` безопасно. Без дополнительного механизма злоумышленник может перехватить `authorization_code` и обменять его на токен. 
   - PKCE вводит `code_verifier` (случайная строка) и его SHA‑256‑хеш (`code_challenge`). Клиент отправляет `code_challenge` на этапе авторизации, а затем при обмене кода включает оригинальный `code_verifier`. IdP проверяет соответствие. Если перехватчик получит только `code`, без `code_verifier` он не сможет завершить обмен.
3. **Как реализовать безопасный RP‑initiated logout в OIDC?**
   - Клиент формирует URL `https://idp/logout?post_logout_redirect_uri=https%3A%2F%2Fapp.example.com%2Flogout_success&sid=SESSION_ID`. 
   - **Front‑channel** – открывает iframe или редирект, IdP удаляет сессию и перенаправляет пользователя обратно. 
   - **Back‑channel** – IdP отправляет POST‑запрос на `backchannel_logout_uri` клиента, содержащий `sid`. Клиент получает запрос независимо от браузера и без участия пользователя, гарантируя удаление серверных сессий.
   - Важно также отозвать **refresh‑tokens** (если они существуют) и очистить локальное хранилище.

## Чек‑лист понимания (10 вопросов + ответы)
1. **Что такое `authorization_code` в OIDC?** — одноразовый код, выдаваемый после согласия пользователя, который клиент обменяет на токены.
2. **Зачем нужен параметр `nonce`?** — защита от повторного использования кода (replay); клиент сравнивает значение, полученное в ID‑Token, с тем, что отправил в запросе.
3. **Какая роль `state` в процессе?** — предотвращает CSRF‑атаки, связывая запрос и ответ.
4. **Что проверяется в `iss` claim?** — URL провайдера, гарантирует, что токен выдан именно тем IdP, которому доверяем.
5. **В чём разница между `aud` и `azp`?** — `aud` — целевой получатель (обычно клиент), `azp` — первый клиент, который запросил токен (важно при цепочках делегирования).
6. **Как работает `refresh‑token rotation`?** — каждый запрос новых токенов выдаёт новый refresh‑token, старый сразу инвалидируется.
7. **Какие типы logout поддерживает OIDC?** — фронт‑ и бэк‑канальные, а также RP‑initiated logout через `end_session_endpoint`.
8. **Какие риски существуют при хранении токенов в `localStorage`?** — уязвимость к XSS‑атакам, токены могут быть украдены скриптом.
9. **Почему `access‑token` часто «opaque» в некоторых провайдерах?** — упрощает отзыв токенов, скрывая внутреннюю структуру, но требует introspection‑endpoint.
10. **Как проверить подпись JWT‑токена?** — загрузить JWKS из `jwks_uri`, выбрать ключ по `kid`, выполнить верификацию с алгоритмом (`RS256`, `ES256` и т.д.).

## Ключевые выводы для собеседования
1. **OIDC — паспорт над OAuth**: добавляет **ID‑Token** и набор **claims**, позволяя сервису аутентифицировать пользователя и получать его атрибуты.
2. **PKCE, state, iss и refresh‑token rotation** — минимальный набор мер безопасности, обязательный для всех новых проектов.
3. **Zero‑Trust архитектуры** используют OIDC‑токены как грант доступа, а не как статический VPN‑ключ.
4. **Тщательная работа со scopes, audience и token‑binding** обеспечивает принцип *least privilege* и минимизирует ущерб от компрометации.
