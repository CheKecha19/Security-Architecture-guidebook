# OAuth 2.0 Basics: Фреймворк делегированного доступа к ресурсам

## Что такое **OAuth 2.0**?
**OAuth 2.0** — протокол делегированной авторизации, позволяющий стороннему приложению получать ограниченный доступ к защищённым ресурсам без передачи пользовательского пароля. 

**Аналогия:** представьте, что вы приходите в отель. У вас есть ключ от своей квартиры (пароль), но вы не хотите отдать его портье. Вместо этого вы выдаёте портье временный **билет‑приглашение** — ключ к вашему номеру только на время вашего пребывания. Портье может открыть дверь, но не знает, где находится ваш дом и не может открыть другие двери. Точно так же OAuth выдаёт **access‑token** — временный билет, ограниченный набором прав (**scopes**) и сроком действия.

### Основные роли
- **Resource Owner** — владелец данных (обычно пользователь). 
- **Client** — приложение, которое запрашивает доступ. 
- **Authorization Server** — служба, выдающая токены. 
- **Resource Server** — API, защищённый токеном.

## Эволюция и мотивация
1. **Basic Auth** (логин + пароль в заголовке) простой, но каждый сервис получал полные учётные данные. 
2. **OAuth 1.0** (2007) ввёл подписи HMAC‑SHA1, но был сложным в реализации. 
3. **OAuth 2.0** (RFC 6749, 2012) упростил модель, избавился от криптографии на уровне протокола, переместив её в TLS. 
4. **RFC 6819 (2013)** добавил Threat Model, указав типичные атаки (code interception, CSRF, phishing). 
5. **PKCE (RFC 7636, 2015)** защитил публичные клиенты от кражи кода. 
6. **RFC 9207 (2022)** ввёл `iss`‑параметр, предотвращающий *Mix‑Up Attack*.
7. **OAuth 2.0 Security BCP (2023)** отклонил Implicit Grant, объявив единственно безопасным поток *Authorization Code + PKCE*.

## Зачем это нужно?
1. **Веб‑логин через Google/Facebook** — пользователь не вводит пароль в стороннем сервисе. 
2. **Мобильные приложения** — используют PKCE и получают токен без хранения client secret. 
3. **API‑доступ к облаку** — интеграции с AWS, GCP, Azure требуют ограниченных токенов. 
4. **Микросервисы** — каждый сервис может получить токен для доступа к другим сервисам по принципу *least privilege*. 
5. **Zero‑Trust сети** — токен заменяет VPN‑политику, позволяя динамически контролировать доступ.

## Основные концепции
| **Компонент** | **Описание** |
|---|---|
| **Authorization Grant** | Временный артефакт (code, token) получаемый после согласия Resource Owner. |
| **Access Token** | Краткоживущий (минуты‑часы) токен, используется для доступа к Resource Server. |
| **Refresh Token** | Долгоживущий (дни‑недели) токен, позволяет получить новый Access Token без повторного ввода пользователя. |
| **Scope** | Набор прав, ограничивающих действия токена (например `email`, `profile`, `openid`). |
| **Audience** | Идентификатор ресурса, к которому токен предназначен (RS). |

### Grant Types (статус)
- **Authorization Code Grant** — рекомендовано для всех веб‑ и мобильных клиентов. 
- **Implicit Grant** — *DEPRECATED* (все новые реализации запрещают). 
- **Resource Owner Password Credentials** — *DEPRECATED*, использовать только для миграции legacy‑систем. 
- **Client Credentials** — для машин‑к‑машине (M2M) сценариев. 
- **Device Code Grant** — для устройств без браузера (смарт‑теле, консоли). 
- **Refresh Token Grant** — обновление Access Token.

### Токены
- **Bearer Token** (RFC 6750) — каждый, кто имеет токен, может его использовать. 
- **Proof‑of‑Possession (PoP) Token** (RFC 9449) — требует доказательства владения клиентским ключом (DPoP, mTLS). 
- **JWT** — самодостаточный токен, содержит claims (sub, aud, exp, scope).

## Сравнение с другими механизмами
| Механизм | Гранулярность | Отзыв | Требования к клиенту | Пример использования |
|---|---|---|---|---|
| **OAuth 2.0** | ✅ Scopes, audience | ✅ Refresh‑token revocation | TLS + client secret (confidential) / PKCE (public) | Google Login, API‑gateway |
| **Basic Auth** | ❌ Полный пароль | ❌ Нет revocation | TLS рекомендуется, но часто без | Legacy REST‑APIs |
| **API‑Key** | ⚪ Один набор прав | ❌ Нет revocation (ключ фиксирован) | Хранится в коде/конфиге | Stripe, Mapbox |
| **SAML 2.0** | ✅ Attributes, Assertions | ✅ Single Logout | XML‑подпись, HTTPS | Enterprise SSO |

## Уроки из реальных инцидентов
1. **SolarWinds 2020** — атакующие получили **application access token** через компрометированный клиент, использовали его для доступа к Microsoft 365. Отсутствие **refresh‑token rotation** позволило длительно использовать токен.
2. **Microsoft Consent Phishing 2021‑2022** — фишинговые письма убеждали пользователя дать согласие OAuth‑приложению с широкими **scopes**. Без **PKCE** и **state**‑проверки атака прошла успешно.
3. **GitHub Token Abuse 2022** — утечки **personal access tokens** (PAT) и **OAuth tokens** привели к массовой репликации кода. В результате GitHub ввёл обязательную **token‑scoping** и **refresh token revocation**.

## Инструменты и средства защиты
| Инструмент | Описание | Применимость к OAuth |
|---|---|---|
| **OAuth2‑Proxy** | Прокси‑сервис, добавляющий OAuth‑авторизацию к веб‑приложениям | Front‑end защита |
| **Keycloak** | Open‑Source Identity‑Provider, поддерживает RFC 6749, PKCE, OIDC | AS/IdP |
| **Auth0** | Managed IdP, автоматическая **refresh token rotation**, **PKCE**, **issuer validation** | AS |
| **Fine‑Grained Scopes Analyzer** | Сканирует токены и проверяет соответствие *least‑privilege* | RS |
| **DPoP‑Enforcer (Node‑JS, Java)** | Библиотеки для проверки PoP‑токенов | RS |
| **mTLS Gateway** | Требует клиент‑сертификат при запросе токенов | AS/RS |

## Архитектурные решения (Zero‑Trust + OAuth)
1. **Пограничные точки** — все запросы проходят через **API‑gateway**, где проверяется **access‑token** и **audience**. 
2. **Dynamic‑scope allocation** — при аутентификации система выдаёт минимальные scopes, затем *progressively* расширяет права после дополнительного MFA. 
3. **Token‑binding (DPoP / mTLS)** — привязывает токен к клиентскому ключу, снижая риск кражи. 
4. **Refresh‑token rotation** — каждый запрос к `/token` заменяет старый refresh‑token новым; компрометированный токен сразу становится недействительным. 
5. **Issuer verification (`iss`)** — клиент проверяет, что авторизационный сервер действительно выдал код, что защищает от *Mix‑Up Attack*.

## Подготовка к собеседованию
| Ошибка | Почему критична |
|---|---|
| **Отсутствие PKCE** в публичных клиентах | Позволяет атакующему перехватить `authorization code`. |
| **Импlicit Grant** в новых проектах | Не обеспечивает confidentiality токена, уязвим к token‑leakage. |
| **Неограниченные scopes** | Нарушает принцип *least privilege*, повышает риск масштабных утечек. |

## Чек‑лист понимания (10 вопросов + ответы)
1. **Что представляет собой `authorization code`?** — Это одноразовый токен, выдаваемый после согласия пользователя, который клиент обменяет на `access token` и `refresh token`.
2. **Почему `client secret` не используется в публичных клиентах?** — Публичные клиенты (SPA, мобильные) не могут хранить секрет безопасно; вместо него применяется **PKCE**.
3. **Какой параметр защищает от `Mix‑Up`‑атаки?** — Параметр `iss` в ответе авторизации, который клиент сравнивает с ожидаемым идентификатором сервера.
4. **Что такое `scope` и зачем их ограничивают?** — Набор прав, ограничивающий действия токена; ограничивая scopes, уменьшаем потенциальный ущерб при компрометации.
5. **Как работает `refresh‑token rotation`?** — При каждом запросе нового `access token` сервер выдает новый `refresh token` и делает предыдущий недействительным.
6. **Каким образом `state`‑параметр борется с CSRF?** — Клиент генерирует случайное значение `state`, сохраняет его и проверяет при возврате коду; атакующий не знает значение и не может подделать запрос.
7. **Что такое `audience`‑claim?** — Идентификатор целевого ресурса; проверка гарантирует, что токен используется только тем сервисом, для которого выдан.
8. **В чём отличие `Bearer` от `PoP`‑токена?** — `Bearer` токен может использовать любой, кто его держит; `PoP` требует доказательства владения клиентским ключом (DPoP, mTLS).
9. **Какие риски несёт `Implicit Grant`?** — Токен передаётся в URL‑фрагменте, может быть записан в истории браузера и перехвачен, не имеет `refresh token`.
10. **Как проверить подпись JWT‑токена?** — С помощью публичного ключа IdP (JWKS) и алгоритма, указанного в `alg` header.

## Ключевые выводы
1. **OAuth 2.0 — фундаментальная модель доступа**; заменяет передачу пароля на ограниченный, управляемый токен. 
2. **PKCE, `state`, `iss` и refresh‑token rotation** — минимальный набор защитных мер, обязательный для всех новых проектов. 
3. **Zero‑Trust** архитектура использует OAuth‑токены как гранты доступа, а не как статические VPN‑ключи. 
4. **Внимание к scopes, audience и токен‑биндингу** обеспечивает принцип *least privilege* и снижает потенциальный ущерб от компрометации. 
5. **Регулярный аудит и ревью** токенов (лог‑анализ, ревокация) — обязательный процесс в любой защищённой организации.
