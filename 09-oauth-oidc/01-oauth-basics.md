# OAuth 2.0: Делегированный доступ

## Что такое OAuth 2.0?

Представь, что ты приезжаешь в отель. У тебя есть багаж. Ты не тащишь его сам — отдаёшь портье. Портье получает ключ от твоего номера, заносит багаж, возвращает ключ. Портье не знает твой пароль от дома, не имеет доступа к твоим деньгам. У него есть только право отнести багаж в номер.

**OAuth 2.0** — это именно такой "ключ от номера" для программ. Когда одно приложение хочет получить доступ к данным в другом приложении, оно не просит твой пароль. Вместо этого оно получает "ключ" — токен. Этот токен даёт ограниченный доступ. Только к тем данным, которые ты разрешил. Только на то время, которое ты разрешил.

OAuth 2.0 — это протокол делегированного доступа. Ты делегируешь права приложению. Не отдаёшь пароль, а даёшь разрешение.

## Зачем нужен OAuth 2.0?

### Ситуация 1: Вход через Google

HR-платформа позволяет войти через Google. Ты нажимаешь "Войти через Google". Ты не вводишь пароль от Google на сайте HR-платформы. Вместо этого Google спрашивает: "Разрешить доступ к email и имени?" Ты соглашаешься. HR-платформа получает токен. Вход выполнен.

Без OAuth: HR-платформа просит пароль от Google. Ты отдаёшь. HR-платформа может сделать что угодно.

### Ситуация 2: Календарь

Приложение хочет добавить событие в твой Google Calendar. Оно просит доступ. Ты разрешаешь: только календарь, только чтение и запись. Приложение не видит твои письма, не видит документы.

### Ситуация 3: Отзыв доступа

Ты дал приложению доступ к данным. Потом передумал. Заходишь в настройки Google. Нажимаешь "Отозвать доступ". Приложение больше не может получить данные. Твой пароль не скомпрометирован.

## Роли в OAuth 2.0

| Роль | Описание | Аналогия |
|------|----------|----------|
| Resource Owner | Владелец данных. Ты. | Гость в отеле |
| Client | Приложение, которое хочет доступ | Портье |
| Authorization Server | Сервер, который выдаёт токены | Ресепшн |
| Resource Server | Сервер, где хранятся данные | Номер |

## Основные потоки

### Authorization Code

| Шаг | Что происходит |
|-----|----------------|
| 1 | Client перенаправляет пользователя на Authorization Server |
| 2 | Пользователь аутентифицируется |
| 3 | Пользователь соглашается дать доступ |
| 4 | Authorization Server перенаправляет на Client с кодом |
| 5 | Client обменивает код на токен |

Это самый безопасный поток. Используется в веб-приложениях.

### PKCE

| Шаг | Что происходит |
|-----|----------------|
| 1 | Client генерирует секрет |
| 2 | Отправляет хеш секрета с запросом |
| 3 | Получает код |
| 4 | Отправляет код + секрет для обмена на токен |

PKCE защищает от перехвата кода. Используется в мобильных приложениях и SPA.

### Client Credentials

| Когда | Описание |
|-------|----------|
| Server-to-server | Нет пользователя |
| API | Доступ к собственным ресурсам |

### Device Code

| Когда | Описание |
|-------|----------|
| ТВ, консоли | Устройство без браузера |
| Пользователь | Подтверждает на другом устройстве |

## Токены

### Access Token

| Параметр | Описание |
|----------|----------|
| Срок жизни | Короткий (15-60 минут) |
| Что даёт | Доступ к ресурсам |
| Формат | JWT или opaque |

### Refresh Token

| Параметр | Описание |
|----------|----------|
| Срок жизни | Длинный (дни, недели) |
| Что даёт | Право получить новый Access Token |
| Хранение | Безопасно |

## OAuth vs OpenID Connect

| Критерий | OAuth 2.0 | OpenID Connect |
|----------|-----------|----------------|
| Цель | Доступ к ресурсам | Аутентификация |
| Токен | Access Token | ID Token + Access Token |
| Информация | Не содержит | Содержит claims о пользователе |
| Слой | Авторизация | Identity над OAuth |

## Архитектурные решения

### Token Validation

| Способ | Описание |
|--------|----------|
| Local | Проверка подписи JWT |
| Introspection | Запрос к Authorization Server |

### Scope

| Scope | Описание |
|-------|----------|
| read | Чтение |
| write | Запись |
| profile | Профиль пользователя |
| email | Email |

## Ограничения

| Проблема | Описание |
|----------|----------|
| Сложность | Много потоков, параметров |
| Implementation | Легко сделать ошибки |
| Token theft | Утечка токена = утечка доступа |
| Phishing | Поддельные страницы согласия |

### Ответы на чек-лист

1. **OAuth 2.0** — протокол делегированного доступа. Приложение получает доступ к данным без пароля пользователя.

2. **Роли**: Resource Owner (владелец), Client (приложение), Authorization Server (выдаёт токены), Resource Server (хранит данные).

3. **Authorization Code**: пользователь авторизуется, получает код, приложение обменивает код на токен. Самый безопасный.

4. **PKCE** — Proof Key for Code Exchange. Защищает от перехвата кода. Используется в мобильных и SPA.

5. **Access Token** — короткий, даёт доступ. **Refresh Token** — длинный, позволяет получить новый Access Token.

6. **Client Credentials** — когда нет пользователя. Server-to-server доступ.

7. **Scope** — область доступа. Что именно разрешено: read, write, profile, email.

8. **OAuth** — авторизация (доступ). **OpenID Connect** — аутентификация (кто ты). OIDC — слой над OAuth.

9. **Валидация**: локальная (проверка подписи JWT) или introspection (запрос к серверу).

10. **Ограничения**: сложность, ошибки реализации, кража токена, phishing.
