# Best Practices OAuth 2.0 / OpenID Connect: Итоги безопасного внедрения

## Что такое OAuth 2.0 / OpenID Connect?
Бытовая аналогия: представьте, что вы отдаёте ключ от своей квартире **доверенному человеку**, чтобы тот смог зайти только в определённые комнаты и только на ограниченный срок. Этот человек получает **разрешение** (authorization), а не полную собственность. Фундаментальный принцип – **делегирование доступа** без передачи учётных данных. Технически **OAuth 2.0** — протокол делегирования, позволяющий клиентскому приложению получать **access‑token** от **authorization server**, а **OpenID Connect (OIDC)** добавляет слой **идентификации**, возвращая **id‑token** с информацией о пользователе. Вместе они формируют современный механизм **single‑sign‑on** и **resource access** в распределённых системах.

## Эволюция и мотивация
- **2012** – появление **OAuth 2.0** как замена OAuth 1.0; цель – упростить интеграцию и поддержать мобильные и веб‑клиенты.
- **Implicit Flow** (token‑in‑fragment) стал популярным в SPA, но со временем показал уязвимости: токен хранится в браузерной истории, уязвим к XSS, облегчает кражу.
- **2016‑2018** – появление **PKCE (Proof Key for Code Exchange)**, изначально для мобильных приложений, чтобы защитить **authorization code** от перехвата.
- **2020** – **FAPI 1.0/2.0** (Financial API) ввёл строгие требования к **mTLS**, **private_key_jwt**, **PAR** и **RAR**, мотивированные регуляторами (PSU, PSD2).
- **2022‑2024** – **BCP (Best Current Practice) OAuth 2.0** обновления включили обязательный **PKCE** даже для серверных приложений, а также рекомендации по **token‑binding** и **token‑introspection**.
- **2025** – рост **Zero‑Trust** архитектур привёл к необходимости комбинировать **OAuth** с **mTLS** и **certificate‑bound access tokens**.

## Зачем это нужно?
### Сценарий 1 – SPA без PKCE
Одностраничное приложение (React) использует **implicit flow**. Злоумышленник встраивает XSS‑скрипт, перехватывает токен из `window.location.hash`, получает полные права доступа к API и совершает финансовые операции от имени пользователя.
### Сценарий 2 – Mobile app с embedded WebView
Приложение открывает **WebView** без проверки `redirect_uri`. Пользователь попадает на фишинговый экран, вводит учётные данные, а злоумышленник получает **authorization code** и, используя фиксированный `client_secret`, получает токен.
### Сценарий 3 – Server‑to‑Server без mTLS
Сервис A напрямую обращается к API B, передавая **client_id** и **client_secret** в запросе. Перехват сетевого трафика (Man‑in‑the‑Middle) раскрывает секреты, позволяя атакующему выдавать токены от имени сервиса.
### Сценарий 4 – Consent phishing
Злоумышленник рассылает фальшивую страницу согласия, имитирующую Google/Microsoft. Пользователь одобряет запрос, а токен попадает в руки атакующего, который может использовать его в течение срока жизни.

## Основные концепции
### Authorization Code Flow + PKCE как единственный допустимый
**Authorization Code Flow** требует двойного обмена: клиент получает **code**, а затем меняет его на **access token**. **PKCE** добавляет случайный `code_verifier` и `code_challenge`, защищая код от перехвата. Именно эта комбинация считается **единственным приемлемым** способом для SPA, native‑mobile и даже серверных приложений.
### mTLS и private_key_jwt для client authentication
Для **confidential clients** (backend‑приложения) рекомендуется использовать **mutual TLS (mTLS)** или **private_key_jwt** вместо простого `client_secret`. mTLS аутентифицирует клиентский сертификат на уровне TLS‑сессии, а `private_key_jwt` подпирает JWT‑запрос клиентским приватным ключом, устраняя риск раскрытия статических секретов.
### PAR (Pushed Authorization Requests) и RAR (Rich Authorization Requests)
**PAR** позволяет клиенту предварительно отправить параметры авторизации на сервер, получив `request_uri`. Затем `request_uri` передаётся в `authorization request`, уменьшая экспозицию чувствительных параметров в URL. **RAR** расширяет стандарт, позволяя передавать политику запросов (scopes, claims) в структуре JSON, повышая гибкость и безопасность.
### FAPI 1.0/2.0 и требования регуляторов
**Financial‑grade API (FAPI)** требует:
- **mTLS** для всех клиент‑серверных соединений.
- **Authorization Code Flow + PKCE** без имплицитных вариантов.
- **Token Binding** (binding токенов к TLS‑каналу или клиентскому сертификату).
- **Strong cryptographic algorithms** (RSA‑2048+, ECDSA‑P‑256).
Эти требования формируют новое «золотое» правило для финансовых и мед‑сервисов.

## Сравнение с другими механизмами
| Механизм | Тип клиента | Требуемый поток | Защита от перехвата кода | Требования к клиенту | Пример использования |
|---|---|---|---|---|---|
| **Implicit Flow** | Public (SPA) | token‑in‑fragment | Нет (token в URL) | Нет | Старые одностраничные приложения (не рекомендуется) |
| **Authorization Code + PKCE** | Public / Confidential | code → token | **PKCE** защищает код | Нет client_secret (public) или mTLS/private_key_jwt (confidential) | Современные SPA, native‑mobile, серверные приложения |
| **Authorization Code + PKCE + mTLS** | Confidential | code → token | **PKCE** + **mTLS** защищает оба этапа | Клиентский сертификат | Финансовые API, корпоративные B2B сервисы |

## Уроки из реальных инцидентов
- **Facebook 2018** – токены доступа, полученные через **implicit flow**, были раскрыты в JavaScript‑консоли, позволив злоумышленникам публиковать контент от имени жертвы.
- **Microsoft/Google Phishing 2020‑2023** – фишинговые сайты копировали страницы согласия OAuth, заставляя пользователей выдавать **authorization code**. Без **PKCE** код был легко обменён на токен.
- **XSS через redirect_uri** – уязвимый параметр `redirect_uri` позволил атакующему указать свой домен, перехватить код и завершить авторизацию без участия пользователя.

## Архитектурные решения
1. **Defense‑in‑Depth для OAuth** – комбинировать несколько слоёв защиты:
   - Переписать все публичные клиенты на **Authorization Code + PKCE**.
   - Для конфиденциальных клиентов использовать **mTLS** или **private_key_jwt**.
   - Включить **PAR** для всех запросов с чувствительными параметрами.
   - Применять **token‑binding** и **short‑lived access tokens** (≤ 5 мин).
2. **Zero‑Trust и токены** – каждый сервис проверяет **access token** через introspection, а не доверяет сетевому периметру.
3. **Roadmap для legacy** – этап 1: внедрить **PKCE** в существующие SPA; этап 2: заменить implicit flow на **Auth Code + PKCE**; этап 3: добавить **mTLS** для сервер‑к‑серверных взаимодействий.

## Подготовка к собеседованию
### Типичные ошибки кандидатов
1. **Смешивание Implicit и Auth Code** – кандидат утверждает, что оба потока одинаково безопасны.
2. **Отсутствие PKCE в мобильных приложениях** – объясняет, зачем нужен `code_challenge`.
3. **Не учитывает revocation** – не знает, как отозвать токен и почему это важно.
### Ситуационные вопросы
1. *«Как бы вы защитили OAuth‑flow в SPA, если клиент не может использовать HTTPS?»* – ответ: **нельзя**, требовать TLS; если невозможно, перейти на серверный прокси, который выполняет код‑обмен.
2. *«Что делать, если в системе обнаружили злоупотребление токеном, полученным через фишинговый consent‑page?»* – ответ: **ревокировать токен**, инжектировать user‑session reset, воспользоваться `revocation_endpoint` и предупредить пользователя.
3. *«Как реализовать динамический consent для разных уровней доступа без изменения клиентского кода?»* – ответ: использовать **Rich Authorization Requests (RAR)**, передавая `claims` в запросе, и настроить IdP на отображение соответствующего UI.

## Чек‑лист понимания
- [ ] 1. Почему **PKCE** обязателен даже для конфиденциальных клиентов?
- [ ] 2. Чем отличается **authorization code** от **access token**?
- [ ] 3. Как **mTLS** усиливает безопасность клиентской аутентификации?
- [ ] 4. Что делает **PAR** и почему он полезен?
- [ ] 5. Как **FAPI 2.0** меняет требования к токен‑binding?
- [ ] 6. Какие риски остаются при использовании **short‑lived access tokens**?
- [ ] 7. Как правильно реализовать **token revocation**?
- [ ] 8. Чем **implicit flow** отличается от **authorization code flow**?
- [ ] 9. Как **Zero‑Trust** модель использует OAuth?
- [ ] 10. Что такое **client assertion** и когда её использовать?

### Ответы на чек‑лист
1. **PKCE** генерирует одноразовый `code_verifier` и привязывает его к `authorization code`, тем самым предотвращая перехват кода в публичных клиентах. Для конфиденциальных клиентов это дополнительный слой, который защищает от компрометации `client_secret`.
2. **Authorization code** – временный одноразовый код, получаемый после согласия пользователя; **access token** – токен доступа, используемый для вызова защищённых ресурсов. Код обменивается на токен через защищённый бекенд.
3. **mTLS** требует клиентского сертификата, проверяемого сервером в TLS‑рукопожатии, тем самым удостоверяя, что именно тот клиент, которому выдан сертификат, делает запрос. Это устраняет необходимость хранить `client_secret`.
4. **PAR** (Pushed Authorization Request) позволяет клиенту отправить параметры авторизации в теле POST‑запроса, получив `request_uri`. Затем `request_uri` передаётся в браузерный запрос, минимизируя экспозицию чувствительных параметров в URL.
5. **FAPI 2.0** требует **token‑binding** к TLS‑каналу или клиентскому сертификату, а также **strict audience validation**. Это повышает защиту от replay‑атак и ограничивает использование токенов в чужих контекстах.
6. **Short‑lived access tokens** снижают окно возможности кражи, но не защищают от утечки `refresh token`. Поэтому необходимо хранить `refresh token` в защищённом хранилище и использовать **rotation**.
7. **Token revocation** реализуется через `revocation_endpoint` IdP, где клиент (или администратор) посылает токен, который должен быть немедленно признан недействительным. Важно также поддерживать **introspection** для проверки статуса токена в реальном времени.
8. **Implicit flow** сразу возвращает `access token` в URL‑фрагменте, без промежуточного кода, что делает его уязвимым к XSS и токен‑leakage. **Authorization code flow** использует отдельный код, который меняется на токен на бекенд‑сервере, что более безопасно.
9. **Zero‑Trust** требует проверки каждой заявки; OAuth обеспечивает токен‑based идентификацию, а **introspection** проверяет актуальность и права токена на каждом микросервисе.
10. **Client assertion** – JWT, подписанный собственным приватным ключом, отправляемый в `client_authentication` вместо `client_secret`. Используется, когда требуется более сильная аутентификация, например в финансовых API.

## Ключевые выводы для собеседования
- **PKCE обязателен** для всех публичных и большинства конфиденциальных клиентов; без него — уязвимо к перехвату кода.
- **mTLS / private_key_jwt** — предпочтительные методы аутентификации сервер‑к‑серверу, устраняющие риск компрометации `client_secret`.
- **PAR** уменьшает экспозицию параметров в URL, а **RAR** повышает гибкость запросов согласия.
- **FAPI** задаёт новый уровень требований для финансовых и регулируемых отраслей: обязательный **token‑binding**, **short‑lived tokens**, **mTLS**.
- **Zero‑Trust** модель полностью полагается на **access token** + **introspection**, а не на сетевой периметр.
