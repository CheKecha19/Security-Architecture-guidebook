# Device Authorization Grant (Device Code Flow)

## Краткое описание
**Device Authorization Grant**, более известный как **Device Code Flow**, представляет собой специальный механизм OAuth 2.0, позволяющий устройствам с ограниченным вводом (смарт‑ТВ, игровые консоли, IoT‑датчики) получить пользовательскую авторизацию без встроенного браузера. Пользователь вводит короткий **user_code** на привычном устройстве (смартфон, ноутбук), а затем «голосующее» устройство получает **access_token** через периодический запрос (polling).

## Эволюция и мотивация
### Истоки проблемы
В оригинальном **Authorization Code Grant** (RFC 6749, 2012) клиент должен открывать браузер, вводить учётные данные и отдавать **client_secret**. Для «пароль‑непригодных» устройств (смарт‑ТВ, принтеры) такой подход невозможен: нет клавиатуры, нет места для ввода пароля, а хранить **client_secret** в прошивке невозможно — любой, кто получит образ прошивки, сможет извлечь секрет.

### Появление Device Flow
В 2015 году IETF принял **RFC 8628** «OAuth 2.0 Device Authorization Grant», позже уточнённый в черновиках *draft‑ietf‑oauth‑device‑flow*. Основная идея — делегировать ввод кода человеку на отдельном устройстве, тем самым «разбивая» процесс на два независимых канала: **устройство‑запроситель** (генерирует **device_code** и **user_code**) и **устройство‑пользователя** (вводит код в браузере). Этот подход устранил необходимость хранить **client_secret** и открыл возможность безопасного MFA.

### Современное использование
Сегодня Device Flow встроен в Azure AD, Google Identity, Auth0, Okta и множество публичных SDK (Azure CLI, AWS CLI, Google Cloud SDK). Он стал почти обязательным для «headless‑клиентов», CI‑pipelines и интерактивных CLI‑утилит, где запуск браузера проблематичен.

## Основные концепции
- **client_id** — публичный идентификатор приложения (не секрет).
- **device_code** — внутренний токен, известный только серверу и клиенту‑запросителю; используется для **polling**.
- **user_code** — короткий (6‑8 символов) код, читаемый пользователем; часто выглядит как `ABCD‑1234`.
- **verification_uri** — URL, куда пользователь перенаправляется (обычно `https://microsoft.com/devicelogin` или `https://login.microsoftonline.com/.../devicecode`).
- **verification_uri_complete** — полная ссылка, уже содержащая `user_code`.
- **expires_in** — время жизни **device_code** (обычно 900 сек.
- **interval** — минимальный интервал запросов к `/token` (рекомендовано 5 сек.).

### Пошаговый сценарий (без кода)
1. **Инициация**: устройство отправляет POST‑запрос к эндпоинту `/devicecode` с `client_id`, `scope` и, опционально, `audience`. В ответ получает JSON‑объект с `device_code`, `user_code`, `verification_uri` и тайм‑параметрами.
2. **Показ пользователю**: на экране устройства показывается короткий `user_code` и URL (или QR‑код) для перехода.
3. **Авторизация пользователя**: пользователь открывает URL на любом браузере, вводит `user_code`, проходит обычный OAuth‑авторизационный процесс (логин + MFA) и подтверждает запрос.
4. **Polling**: пока пользователь не авторизуется, клиент‑запроситель каждые `interval` секунд отправляет POST‑запрос к `/token` с `grant_type=urn:ietf:params:oauth:grant-type:device_code`, `client_id`, `device_code`. Если пользователь ещё не подтвердил, сервер отвечает `authorization_pending` (или `slow_down`).
5. **Получение токена**: после подтверждения сервер выдаёт `access_token` (и, при запросе `offline_access`, `refresh_token`). Токен действует согласно политике провайдера (обычно 1 ч.) и может быть использован для доступа к ресурсам.

### Аналогии
- **«Почтовый ящик на лестнице»**: представьте, что ваш смарт‑телевизор — это ящик, в котором нет замка. Вы кладёте в него письмо‑запрос (`device_code`) и оставляете короткую метку (`user_code`) у двери. Пользователь идёт в кабинет, открывает главный замок (вводит пароль и MFA), ставит печать на письмо, и ящик «получает» готовый ключ (`access_token`).
- **«Союз двух коробок»**: первая коробка генерирует секрет, вторая (пользователь) подтверждает его, а потом обе коробки «соединяются» через токен.

## Архитектурные решения
### Клиентские реализации
- **CLI‑утилиты** (`az login --use-device-code`, `gcloud auth login --no-launch-browser`). Они автоматически выводят `user_code` и `verification_uri`. При успешной авторизации токен сохраняется в локальном профиле (`~/.azure/accessTokens.json`).
- **IoT‑шлюзы**: микроконтроллеры могут использовать лёгкую HTTP‑библиотеку для запроса `device_code`. После авторизации они хранят токен в безопасном NVRAM.
- **Smart TV**: встроенные SDK (e.g., Samsung Tizen, LG webOS) часто используют готовый UI‑модуль, который отображает QR‑код, генерируемый из `verification_uri_complete`.

### Сервер‑сайт (Identity Provider)
- **Точки входа**: `/devicecode` (создание кода) и `/token` (получение токена). Оба эндпоинта защищены TLS.
- **Согласование с MFA**: большинство IdP позволяют настроить **Conditional Access** –‑ блокировать Device Flow без MFA, требовать MFA‑запросов или полностью запрещать поток.
- **Rate‑limiting**: RFC 8628 рекомендует ограничивать запросы `verification_uri` в зависимости от энтропии `user_code`. При 8‑символьном коде (~34 бита) рекомендуется не более 1 запроса в секунду на одно устройство.

## Ограничения и нюансы
1. **Время жизни** – `device_code` обычно истекает через 10‑15 минут. Если пользователь не вводит код в срок, запрос необходимо перезапустить.
2. **Ограниченный набор разрешений** – некоторые IdP запрещают запрос `offline_access` в Device Flow без строгой политики, чтобы не выдавать долгоживущие **refresh_token**.
3. **Brute‑force пользовательского кода** – короткая длина `user_code` (≈34 бита) делает возможным перебор, если сервер не ограничивает запросы к `/devicecode`/`/token`. Лучший способ защиты – rate‑limit по IP и мониторинг `authorization_pending`/`slow_down` ошибок.
4. **MITM‑атаки на verification_uri** – если злоумышленник перехватит HTTPS‑трафик (неправильный сертификат, скомпрометированный Wi‑Fi), он может подменить страницу входа и собрать учётные данные. Поэтому обязательна **HSTS**, **certificate pinning** в клиентском SDK.
5. **Squatting (угон кода)** – атакер может заранее сгенерировать `device_code` и ждать, пока жертва введёт тот же `user_code` в собственный запрос. Если пользователь вводит код без проверки источника (например, в фишинговом письме), атакер получает токены. mitigations: включить **Conditional Access** → *Require approved client apps* и ограничить список `client_id`.
6. **Отсутствие пользовательского UI** – на некоторых головных платформах (например, встраиваемые Linux‑устройства без GUI) нет возможности показать QR‑код. В таком случае рекомендуется воспользоваться **Out‑of‑Band** (SMS‑токен) или fallback‑механизмом **Client Credentials**.
7. **Неявные consent screens** – при использовании публичных клиент‑ID без предварительного согласования пользователь может увидеть только «Разрешить приложению доступ», без подробного списка запрашиваемых прав. Это повышает риск «over‑grant».

## Три вопроса с развернутыми ответами (собеседование)
1. **Почему Device Code Flow считается более безопасным, чем традиционный Implicit Grant, и в каких сценариях его всё‑ещё стоит отключать?**
   - *Ответ*: Device Flow использует **Authorization Code** + **PKCE‑проверку** (если поддерживается) и всегда проходит через HTTPS‑канал, тогда как Implicit Grant выдаёт токен напрямую в URL‑фрагменте, подвержен перехвату в браузере и не поддерживает refresh‑token. Device Flow предпочтителен для устройств без браузера и при необходимости длительного доступа (offline_access). Однако в средах с высоким риском **phishing‑атаки** (например, публичные Wi‑Fi) его рекомендуется отключать через Conditional Access, потому что пользователь может быть обманут вводом `user_code` в фишинговой форме.
2. **Опишите механизм «squatting» в контексте Device Flow и предложите способы защиты от него.**
   - *Ответ*: Squatting – это сценарий, когда атакующий заранее генерирует `device_code` и ждёт, пока жертва введёт тот же `user_code` в фишинговом окне, полагая, что запрос исходит от доверенного приложения. Защита: (a) включить *Client App Restriction* в Conditional Access, позволяя только известные `client_id`; (b) отображать в UI полное имя приложения, из которого пришёл запрос; (c) ограничить частоту запросов к `/devicecode` (rate‑limit) и отслеживать аномальные паттерны (много `device_code` от одного IP). 
3. **Как правильно реализовать «polling back‑off» в клиенте, чтобы избежать блокировки по‑со стороны IdP?**
   - *Ответ*: Клиент должен использовать параметр `interval`, возвращаемый в ответе на `/devicecode`. При получении ошибки `slow_down` необходимо удвоить текущий интервал (или добавить минимум 5 сек). После трёх подряд полученных `slow_down` следует увеличить интервал минимум до 30 сек. При получении `authorization_pending` клиент продолжает полагаться на исходный `interval`. Такой алгоритм соответствует рекомендациям RFC 8628 и предотвращает превышение лимитов `token`‑endpoint.

## Ключевые выводы для собеседования (2‑4 тезиса)
- **Device Code Flow** — безопасный способ аутентификации «headless‑устройств», но его безопасность полностью зависит от правильной настройки Conditional Access (MFA, ограничение `client_id`).
- **Вектор атаки** — phishing и **squatting** — позволяют злоумышленнику получить полноценный `access_token` и `refresh_token` без ввода пароля, используя лишь короткий `user_code`.
- **Защита** включает: (1) строгие лимиты `interval`/`rate‑limit`; (2) отображение полного имени приложения в UI; (3) блокировку Device Flow для неавторизованных клиентских приложений.
- При проектировании систем необходимо оценить необходимость **offline_access** в Device Flow: если долгоживущие токены не требуются, отключите `offline_access` — это уменьшит окно эксплуатации при компрометации кода.
