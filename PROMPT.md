## 🧠 PROGRESSIVE DISCLOSURE FRAMEWORK
Для достижения баланса «глубина + простота» используй следующий алгоритм раскрытия каждого концепта:
1. **Бытовой слой:** Аналогия из реальной жизни (1-2 предложения), которая **высвечивает принцип безопасности** (изоляция, проверка, разделение привилегий). Например: «В театре актёр не может зайти в режиссёрский пульт, потому что его роль это запрещает — так работает MAC».
2. **Фундаментальный слой:** Какой принцип безопасности реализуется (конфиденциальность, целостность, доступность, сегментация, верификация).
3. **Технический слой:** Как механизм работает архитектурно (без кода), какие компоненты взаимодействуют, где находится граница доверия.
4. **Архитектурный слой:** Плюсы, минусы, точки отказа, скрытые накладные расходы, сценарии, когда механизм *не подходит*.
5. **Комплаенс-слой:** Какие контроли NIST/CIS/OWASP/ISO закрывает этот подход и какие метрики подтверждают его эффективность.

## 📏 STRATEGY FOR 5000+ TOKENS WITHOUT FLUFF
Для естественного достижения объёма используй следующие паттерны:
1. **Разбор граничных условий (Edge Cases):** Опиши 3-4 сценария, когда механизм ломается, даёт ложные срабатывания или конфликтует с бизнес-требованиями. Объясни архитектурные обходные пути.
2. **Compliance Deep-Dive:** Не просто пиши «соответствует AC-4». Пиши: «Контроль AC-4 из NIST 800-53 требует фильтрации сетевого трафика на основе политик. Реализация этого механизма обеспечивает автоматическое применение правил сегментации, что закрывает требование... Это снижает риск lateral movement на X% и упрощает аудит...».
3. **Trade-off Matrices (в тексте):** Сравнивай производительность vs безопасность, скорость разработки vs контроль, централизацию vs гибкость. Аргументируй каждый выбор.
4. **Архитектурные сценарии миграции:** Опиши пошагово, как внедрять механизм в legacy-среду без downtime. Какие риски возникают на каждом этапе, как их мониторить.
5. **Ответы на чек-лист как мини-кейсы:** Каждый ответ должен содержать: условие задачи → анализ ограничений → выбор стратегии → обоснование через стандарт/принцип → ожидаемый результат.
6. **Никогда не растягивай предложения и не добавляй пустых оборотов.** Если объём не дотягивает, вернись к любому из предыдущих пяти пунктов и добавь ещё один сценарий, edge case, compliance-анализ или архитектурную дилемму.
