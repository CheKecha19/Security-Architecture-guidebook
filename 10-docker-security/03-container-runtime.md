# Container Runtime Security: Защита работающего контейнера

## Что такое runtime-безопасность контейнеров?

Представьте, что вы построили здание с крепкими стенами (namespaces), установили счётчики на электричество и воду (cgroups), провели проверку всех строительных материалов (сканирование образов). Здание готово к заселению. Но безопасность не заканчивается на этапе строительства: вам нужно следить за тем, что происходит внутри квартир. Кто открывает окна? Кто использует дрель в час ночи? Не пытается ли кто-то разобрать стену? Это и есть runtime-безопасность — защита контейнера на этапе его работы.

Runtime-безопасность — это набор механизмов, которые ограничивают действия контейнера во время его работы. Даже если образ проверен и не содержит известных уязвимостей, злоумышленник может проникнуть в контейнер через уязвимость в приложении (например, SQLi или RCE). Runtime-защита должна ограничить ущерб: если атакующий попал в контейнер, он не должен получить root на хосте, не должен просканировать сеть, не должен внедрить вредоносное ПО.

Docker предоставляет несколько встроенных механизмов runtime-защиты: seccomp фильтрует системные вызовы, AppArmor/SELinux ограничивает возможности процесса, Linux Capabilities дробит root-права на отдельные привилегии. Эти инструменты работают на уровне ядра — они применяются до того, как процесс выполнит опасное действие.

## Зачем это нужно?

**Сценарий 1: Уязвимость в приложении.** В HR-сервисе найдена уязвимость удалённого выполнения кода (RCE). Атакующий через HTTP-запрос может выполнить shell-команды внутри контейнера. Без runtime-защиты он сможет: установить майнер криптовалют, просканировать локальную сеть, подключиться к базе данных, загрузить эксплойты. С runtime-защитой seccomp блокирует системные вызовы `ptrace`, `mount`, `kexec_load` — атакующий не может расширить свои возможности.

**Сценарий 2: Биндинг Docker socket в контейнер.** QA-инженер для удобства смонтировал Docker socket в тестовый контейнер. Контейнер имеет все привилегии для управления Docker демоном. Если злоумышленник проникнет в этот контейнер — он получит полный контроль над всеми контейнерами на хосте. Capabilities и seccomp здесь не помогут, потому что Docker socket даёт прямой доступ к API daemon. Правильное решение: не монтировать Docker socket в контейнеры, использовать remote API с TLS.

**Сценарий 3: Контейнер с сетевыми привилегиями.** Контейнеру дана capability `NET_ADMIN` для настройки сетевых интерфейсов. Злоумышленник, проникший в контейнер, может перехватывать трафик соседних контейнеров (ARP-spoofing), отключать сеть, перенаправлять трафик на свой сервер. Runtime-защита должна ограничивать capabilities до минимально необходимых: если контейнеру не нужно менять сеть — не давайте ему `NET_ADMIN`.

## Основные концепции

### Seccomp: Фильтр системных вызовов

**Seccomp** (Secure Computing Mode) — это механизм ядра Linux, позволяющий ограничить, какие системные вызовы может делать процесс. Системный вызов (syscall) — это способ, которым программа взаимодействует с ядром: открыть файл, создать процесс, выделить память, настроить сеть.

Представьте seccomp как фильтр для телефонных звонков. Телефонная линия (ядро) позволяет позвонить кому угодно (выполнить любой syscall). Seccomp устанавливает чёрный список: «нельзя звонить в полицию, пожарную, скорую» (нельзя выполнять `ptrace`, `mount`, `kexec_load`). Или белый список: «можно звонить только маме и бабушке» (можно выполнять только `read`, `write`, `exit`, `open` — минимальные syscall'ы для работы приложения).

Seccomp работает на уровне ядра: до того, как процесс выполнит syscall, ядро проверяет, разрешён ли вызов. Если запрещён — процесс получает сигнал SIGKILL. Это очень быстро и не обходится без проверки: фильтрация применяется до выполнения вызова, а не после.

Docker использует seccomp по умолчанию. Встроенный профиль блокирует около 40 опасных системных вызовов из 300+ доступных. Заблокированы: `mount` (монтирование файловых систем), `ptrace` (отладка процессов), `kexec_load` (загрузка нового ядра), `swapcontext` (опасные манипуляции с памятью), `userfaultfd` (атаки на память).

| Заблокированный syscall | Опасность |
|-------------------------|-----------|
| `mount` | Контейнер может смонтировать хост внутри себя и получить доступ к файлам |
| `ptrace` | Отладка чужих процессов, инъекция кода |
| `kexec_load` | Загрузка нового ядра — полный захват хоста |
| `bpf` | Установка фильтров, атаки на ядро |
| `perf_event_open` | Мониторинг ядра, извлечение данных |

Seccomp эффективен, но не всеобъемлющ. Он защищает от выполнения отдельных опасных системных вызовов. Но если приложение использует `open` для чтения чужого файла — seccomp это разрешит, потому что `open` нужен для нормальной работы. Для более тонкого контроля нужен AppArmor.

### AppArmor: Профили ограничений

**AppArmor** — это модуль безопасности Linux (LSM — Linux Security Module), позволяющий назначать процессу профиль с ограничениями: какие файлы можно читать/писать, какие сетевые соединения открывать, какие capabilities использовать.

Продолжая аналогию с телефоном: seccomp решает, каким телефонным номерам (системным вызовам) можно звонить. AppArmor решает, кому можно звонить (какие файлы), как долго говорить (ограничения ресурсов), с кем разговаривать (сетевые соединения). Это два дополняющих друг друга механизма.

Профиль AppArmor состоит из правил:
- `/etc/shadow r` — файл можно только читать (read)
- `/etc/passwd rw` — файл можно читать и писать
- `/var/log/** w` — все файлы в папке log можно писать
- `network tcp` — можно открывать TCP соединения
- `capability net_bind_service` — можно биндить порты ниже 1024

Docker автоматически накладывает AppArmor профиль «docker-default». Этот профиль:
- Запрещает монтирование файловых систем
- Запрещает запись в /proc, /sys
- Запрещает открытие raw-сокетов
- Запрещает доступ к блочным устройствам
- Ограничивает capabilities

Можно создать свой профиль AppArmor для конкретного приложения. Например, для контейнера с Nginx можно разрешить только чтение конфигов `/etc/nginx/`, чтение статического контента `/usr/share/nginx/`, запись в `/var/log/nginx/`. Любая попытка прочитать `/etc/shadow` или `/root/` будет заблокирована.

### Linux Capabilities: Дробление root

**Capabilities** — это механизм дробления привилегий root на отдельные «атомарные» разрешения. Вместо того чтобы давать контейнеру либо все права (root), либо никаких (обычный пользователь), можно выдать ровно те capabilities, которые нужны приложению.

Docker по умолчанию запускает контейнер с ограниченным набором capabilities. Полный список включает около 40 capabilities. Docker выдаёт только необходимые для работы контейнера. Например:

- `CHOWN` — можно менять владельца файлов
- `DAC_OVERRIDE` — можно игнорировать права доступа (chmod)
- `SETUID` / `SETGID` — можно менять идентификатор пользователя/группы
- `NET_RAW` — можно создавать raw-сокеты
- `NET_BIND_SERVICE` — можно биндить порты ниже 1024

Опасные capabilities, которые Docker НЕ выдаёт по умолчанию:
- `SYS_ADMIN` — почти root: mount, namespace operations, swapon
- `SYS_MODULE` — загрузка/выгрузка модулей ядра
- `SYS_PTRACE` — отладка чужих процессов
- `NET_ADMIN` — управление сетью: firewall, интерфейсы
- `SYS_BOOT` — перезагрузка сервера

Рекомендуется использовать `--cap-drop=ALL` и добавить только нужные capabilities. Например, для контейнера с веб-сервером нужен только `NET_BIND_SERVICE` (порт 80). Всё остальное должно быть запрещено.

### Read-only Root Filesystem

Эта опция монтирует корневую файловую систему контейнера как read-only. Контейнер может записывать только в смонтированные volumes. Это предотвращает модификацию бинарников, библиотек и конфигов внутри контейнера.

Если злоумышленник проникает в контейнер с root fs read-only — он не может:
- Заменить бинарник приложения на вредоносный
- Добавить свой скрипт в cron или init.d
- Изменить конфигурационные файлы
- Установить вредоносное ПО

Read-only root fs — простой и эффективный механизм, который ломает множество атак. Многие приложения работают без изменений, если правильно настроены volumes для временных файлов и логов.

## Сравнение механизмов runtime-защиты

| Механизм | Уровень | Что ограничивает | Производительность | Сложность |
|----------|---------|------------------|--------------------|-----------|
| Seccomp | Ядро | Системные вызовы | Минимальная | Средняя (профили) |
| AppArmor | LSM модуль | Файлы, сеть, capabilities | Минимальная | Высокая (правила) |
| Capabilities | Ядро | Привилегии root | Нулевая | Низкая (списки) |
| Read-only rootfs | Docker | Запись в файловую систему | Нулевая | Низкая (флаг) |
| User namespace | Ядро | UID внутри контейнера | Нулевая | Средняя |

## Практические применения

**Сценарий 1: Снижение привилегий для Nginx.** Приложение Nginx работает в контейнере. Политика безопасности: container run with `--cap-drop=ALL --cap-add=NET_BIND_SERVICE`. Nginx может слушать порт 80, но не может менять владельца файлов, создавать raw-сокеты, монтировать файловые системы. Root filesystem смонтирована read-only. Nginx пишет только в volume `/var/log/nginx/`. Даже если атакующий найдёт RCE в Nginx — он не сможет установить майнер или просканировать сеть.

**Сценарий 2: Пользовательский seccomp профиль для Java.** Java приложение использует больше системных вызовов, чем веб-сервер. Стандартный seccomp профиль Docker блокирует некоторые вызовы, которые могут понадобиться JVM (например, `futex` с определёнными флагами). Создаётся кастомный seccomp профиль, который разрешает все вызовы, используемые JVM, но блокирует опасные (`ptrace`, `mount`, `kexec_load`). Профиль тестируется на staging и только потом применяется в production.

**Сценарий 3: User namespace remapping.** Компания настраивает Docker daemon на использование user namespace remapping. Root (UID 0) внутри контейнера отображается на UID 50000 на хосте. Даже если злоумышленник вырвется из контейнера — на хосте он будет пользователем с минимальными правами, а не root. User namespace включается глобально в конфиге daemon: `{"userns-remap": "default"}`.

**Сценарий 4: Запуск от non-root.** В Dockerfile создаётся пользователь `appuser`: `RUN adduser -D appuser && USER appuser`. Все процессы внутри контейнера работают от UID 1001, а не от root. Даже если атакующий получит shell внутри контейнера — он не будет root, не сможет выполнять операции, требующие root-прав (установка пакетов, изменение конфигов), и будет ограничен в системных вызовах.

**Сценарий 5: Runtime-мониторинг Falco.** Помимо встроенных механизмов Docker, компания использует Falco — runtime security engine от Sysdig. Falco анализирует системные вызовы в реальном времени и обнаруживает подозрительную активность: запуск shell в контейнере, чтение /etc/shadow, попытку записать в /proc. При обнаружении Falco отправляет alert в SIEM и может автоматически остановить контейнер.

## Ограничения и нюансы

**Seccomp не защищает от всех атак.** Seccomp блокирует конкретные системные вызовы. Но некоторые атаки используют разрешённые вызовы. Например, атака Dirty Pipe (CVE-2022-0847) использовала системный вызов `splice`, который не блокируется seccomp. Seccomp — один слой, не панацея.

**AppArmor может ломать приложения.** Слишком строгий профиль AppArmor может заблокировать нормальную работу приложения. Например, Java JVM использует `mlockall` для управления памятью, что может быть запрещено профилем. Требуется тщательное тестирование профиля на staging перед применением в production.

**Drop всех capabilities — не всегда возможно.** Некоторые приложения требуют определённых capabilities для работы. Например, ping требует `CAP_NET_RAW`. Решение: либо добавлять конкретную capability, либо пересмотреть архитектуру приложения, либо использовать setuid-бинарник для ping.

**Read-only rootfs и временные файлы.** Многие приложения создают временные файлы. С read-only rootfs эти операции упадут. Решение: смонтировать tmpfs для `/tmp` внутри контейнера: `--tmpfs /tmp:size=100M`.

**User namespace не включён по умолчанию.** Docker рекомендует user namespace, но не включает его по умолчанию из-за возможных проблем совместимости. Некоторые драйверы хранения (devicemapper) не работают с user namespace. Требуется тестирование.

## Архитектурные решения

**Least Privilege — базовый принцип.** Контейнер должен иметь ровно те права, которые нужны для работы приложения, и ни на одно больше. `--cap-drop=ALL --cap-add=NEEDED`. Read-only rootfs. Non-root user. Seccomp по умолчанию. Если приложение работает без дополнительных прав — не давайте их.

**Defense in Depth — слои защиты.** Seccomp не заменяет AppArmor, AppArmor не заменяет read-only rootfs. Каждый механизм защищает от разных типов атак. Комбинация всех механизмов даёт синергетический эффект.

**Runtime-мониторинг.** Встроенные механизмы Docker предотвращают атаки, но не обнаруживают их. Для обнаружения нужен Falco, Tracee, или Sysdig. Runtime-мониторинг анализирует системные вызовы в реальном времени и выявляет аномалии.

**Zero Trust для контейнеров.** Даже если контейнер находится внутри корпоративной сети — он не должен доверять другим контейнерам. mTLS между микросервисами, сетевые политики Kubernetes, service mesh. Runtime-защита на каждом контейнере.

### Ответы на чек-лист

1. **Как seccomp фильтрует системные вызовы?** Seccomp устанавливает фильтр (BPF-программу) в ядре. Когда процесс делает системный вызов, ядро сначала проверяет фильтр. Если вызов разрешён — выполняется. Если запрещён — процесс получает SIGKILL или SIGSYS. Фильтрация происходит до выполнения вызова, что предотвращает любые опасные операции. Docker использует профиль seccomp по умолчанию, блокирующий около 40 опасных syscall'ов.

2. **Чем seccomp отличается от AppArmor?** Seccomp фильтрует системные вызовы на уровне ядра — какие syscall'ы можно вызывать. AppArmor работает на уровне LSM и ограничивает, как процесс использует эти вызовы — какие файлы можно читать, к каким сетевым адресам подключаться. Seccomp решает вопрос «можно ли звонить?», AppArmor — «кому и когда можно звонить?». Они дополняют друг друга.

3. **Почему capability CAP_SYS_ADMIN считается опасной?** CAP_SYS_ADMIN — это «ключ от всех дверей» в мире Linux capabilities. Она включает более 30 административных операций: монтирование файловых систем, управление namespace, вызов swapon, управление quota, изменение системных часов. Фактически CAP_SYS_ADMIN даёт процессу почти все привилегии root. В контейнере эта capability должна быть запрещена безусловно.

4. **Что такое read-only root filesystem и как она защищает?** Read-only root fs монтирует корневую файловую систему контейнера в режиме «только чтение». Контейнер не может изменить бинарники, библиотеки, конфигурационные файлы, скрипты. Атакующий не может заменить `nginx` на вредоносную версию, не может добавить свой скрипт, не может установить майнер. Для временных данных используется tmpfs или volumes.

5. **Как user namespace защищает хост от выхода из контейнера?** User namespace отображает UID 0 (root) внутри контейнера на непривилегированного пользователя (например, UID 50000) на хосте. Если атакующий находит способ выйти из контейнера (через уязвимость ядра или misconfiguration), на хосте он будет обычным пользователем с минимальными правами, а не root. Это существенно снижает ущерб от container escape.

6. **Почему не стоит запускать контейнер от root?** Root внутри контейнера имеет максимальные привилегии внутри своей изоляции. Если атакующий найдёт способ выйти из контейнера — он получит root на хосте. Даже без выхода root может устанавливать ПО, менять конфиги, скрывать следы атаки. Рекомендуется создавать в Dockerfile непривилегированного пользователя и запускать приложение от его имени.

7. **Какие системные вызовы Docker блокирует по умолчанию?** Docker блокирует около 40 опасных syscall'ов, включая: mount (монтирование ФС), ptrace (отладка), kexec_load (загрузка ядра), bpf (фильтры пакетов), perf_event_open (мониторинг ядра), swapcontext (манипуляции с памятью), userfaultfd (атаки на память), process_vm_readv/writev (чтение/запись памяти чужих процессов). Полный список доступен в профиле seccomp Docker.

8. **Как создавать кастомные seccomp профили?** Нужно создать JSON-файл, описывающий разрешённые/запрещённые syscall'ы. Docker запускается с флагом `--security-opt seccomp=/path/to/profile.json`. Профиль содержит: defaultAction (что делать по умолчанию: SCMP_ACT_ALLOW или SCMP_ACT_ERRNO), и список исключений с конкретными syscall'ами. Рекомендуется начинать с профиля Docker по умолчанию и добавлять нужные syscall'ы по мере необходимости.

9. **В чём разница между chroot и container runtime security?** chroot меняет корень файловой системы для процесса, но не изолирует его. Процесс с chroot может выйти из «тюрьмы» через `/proc/1/root` или с помощью `openat` (если они разрешены). Container runtime security использует combination: namespaces (изоляция), cgroups (ресурсы), seccomp (syscall'ы), AppArmor (файлы), capabilities (привилегии). chroot — это замóк на двери; container security — это система безопасности аэропорта.

10. **Как runtime-мониторинг (Falco) дополняет seccomp и AppArmor?** Seccomp и AppArmor — превентивные механизмы: они блокируют опасные действия до их выполнения. Falco — детективный механизм: он мониторит системные вызовы в реальном времени и обнаруживает аномалии. Seccomp скажет «нет» на вызов mount. Falco обнаружит, что контейнер запустил shell (хотя это может быть разрешено) и отправит alert. Комбинация: seccomp/AppArmor предотвращает известные атаки, Falco обнаруживает новые.
