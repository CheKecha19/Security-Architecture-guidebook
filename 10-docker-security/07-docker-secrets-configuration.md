# Docker Secrets и управление конфигурацией: защита секретов в контейнеризованных средах

## Что такое Docker Secrets и управление конфигурацией?

Представьте себе роскошный отель с сейфом, защищённым кодовым замком. Ключ от сейфа есть только у управляющего и нескольких доверенных сотрудников. Каждый раз, когда кто-то открывает сейф или меняет его содержимое, система безопасности фиксирует это событие в журнале. Гости отеля никогда не видят внутреннего механизма замка и не знают комбинации — они просто доверяют, что их ценности находятся под надёжной защитой.

Точно так же работает система управления секретами в Docker и контейнерных оркестраторах. **Docker Secrets** — это механизм безопасного хранения и распространения конфиденциальных данных (паролей, API-ключей, TLS-сертификатов, токенов доступа) внутри кластера контейнеров. Секреты создаются централизованно, хранятся в зашифрованном виде, передаются по защищённым каналам и монтируются в контейнеры как файлы в специальном файловой системе tmpfs, которая никогда не записывается на диск.

**Управление конфигурацией** — это более широкая дисциплина, охватывающая не только секреты, но и все параметры, определяющие поведение приложения: адреса сервисов, уровни логирования, размеры пулов соединений, фича-флаги и другие настройки, которые отличают production-окружение от staging и development. В контейнерном мире конфигурация отделена от образа приложения, что позволяет собрать один артефакт и развернуть его в любом окружении, подставив нужные параметры на этапе запуска.

**Контейнерная изоляция** добавляет сложности к управлению секретами. Традиционные серверы живут долго, и администратор может вручную разместить конфиденциальные файлы в защищённых директориях. Контейнеры же эфемерны — они создаются и уничтожаются сотни раз в день, при этом каждый экземпляр нуждается в доступе к секретам. Ручное копирование ключей в каждый контейнер невозможно масштабировать и чревато утечками. Поэтому современные платформы предоставляют автоматизированные механизмы распространения секретов, где оркестратор берёт на себя роль доверенного посредника между хранилищем секретов и потребляющими их сервисами.

## Эволюция и мотивация

История управления конфигурацией в контейнерах началась с простого и очевидного решения — переменных окружения. Когда Docker только появился на сцене в 2013 году, разработчики привыкли передавать настройки приложений через environment variables, и это решение было перенесено в контейнерный мир без изменений. Подход казался элегантным: переменные независимы от языка программирования, легко инспектируются через `docker inspect`, просты в использовании с docker-compose.

Проблемы стали очевидны по мере роста зрелости контейнерных платформ. Переменные окружения видны любому процессу внутри контейнера, попадают в логи при ошибках, оседают в `.bash_history`, отображаются в системах мониторинга и трассировки. Команда `docker inspect` доступна любому пользователю с доступом к Docker daemon, а в swarm-кластере — любому узлу, имеющему управляющую роль. Несколько громких инцидентов, связанных с утечкой API-ключей через environment variables, привели к переосмыслению подхода.

Docker Inc. представила нативный механизм **Docker Secrets** в версии 1.13 (январь 2017 года), но только для Docker Swarm. Секреты хранились в распределённом хранилище Raft, шифровались при передаче по TLS, монтировались в контейнеры через tmpfs и автоматически удалялись при остановке сервиса. Это был значительный шаг вперёд, но ограничение Swarm режимом делало решение малоприменимым в мире, где Kubernetes уже завоёвывал доминирующие позиции.

Kubernetes пошёл похожим путём, но с собственной терминологией и архитектурой. **Kubernetes Secrets** появились в ранних версиях платформы и изначально хранили данные в etcd в кодировке base64 — формате, который не обеспечивает шифрацию, а только предотвращает случайное искажение бинарных данных. Шифрование etcd появилось позже как опциональная функция, требующая явной настройки KMS-провайдера. Постепенно Kubernetes Secrets обросли дополнительными механизмами защиты: RBAC для контроля доступа, admission controllers для валидации, External Secrets Operator для интеграции с внешними хранилищами.

Параллельно развивалась экосистема внешних менеджеров секретов. **HashiCorp Vault** (выпущенный в 2015 году) предложил динамические секреты с автоматическим временем жизни, ротацией и аудитом. **AWS Secrets Manager** (2018), **Azure Key Vault** и **GCP Secret Manager** интегрировались с IAM-политиками облачных провайдеров, позволяя приложениям аутентифицироваться через механизмы облака и получать секреты без долгоживущих credentials. Эволюция привела к архитектуре, где контейнерный оркестратор отвечает за распространение секретов до точки входа в приложение, а менеджер секретов — за жизненный цикл, ротацию и аудит конфиденциальных данных.

## Зачем это нужно?

### Сценарий 1: Защита credentials базы данных в микросервисной архитектуре

Современное приложение состоит из десятков или сотен сервисов, каждый из которых требует доступа к базам данных, кешам, очередям сообщений, внешним API. Если пароль к PostgreSQL вшит в образ контейнера, любой разработчик с доступом к registry может его извлечь. Если пароль передаётся через переменную окружения, он виден в логах оркестратора, системах мониторинга, дампах процессов. Использование Docker Secrets позволяет монтировать пароль как файл в `/run/secrets/db_password`, доступный только читающему процессу, при этом содержимое никогда не попадает в историю команд и не отображается в стандартных инструментах инспекции.

### Сценарий 2: Соответствие требованиям регуляторов и стандартам безопасности

Стандарты PCI DSS требуют шифрования передаваемых данных, защиты ключей шифрования и регулярной смены паролей. NIST SP 800-53 контролирует управление доступом к критическим параметрам конфигурации. ISO 27001 требует политики управления секретами, включающей хранение, распространение и уничтожение. CIS Benchmarks для Docker и Kubernetes содержат конкретные рекомендации по изоляции секретов и запрету хранения чувствительных данных в переменных окружения. Использование централизованного менеджера секретов с аудитом и ротацией позволяет пройти аудит и продемонстрировать контроль над жизненным циклом конфиденциальных данных.

### Сценарий 3: Многоарендная инфраструктура и разграничение доступа

Крупные организации используют общую инфраструктуру Kubernetes для нескольких команд или даже внешних клиентов. Каждый tenant должен иметь доступ только к своим секретам. RBAC в Kubernetes позволяет ограничить доступ к Secret-ресурсам на уровне namespace, но сам по себе Kubernetes Secrets хранит данные в etcd, к которому теоретически имеет доступ администратор кластера. Внешний менеджер секретов с политиками IAM позволяет реализовать более строгое разграничение, где даже cluster-admin не может прочитать чужие credentials, если у него нет соответствующих прав в Vault или AWS Secrets Manager.

### Сценарий 4: Автоматическая ротация секретов без downtime

Утечка секрета — неизбежное событие в долгосрочной перспективе. Разработчик случайно закоммитил API-ключ в публичный репозиторий, логи утекли через уязвимость, бывший сотрудник сохранил копию credentials. Если ротация требует ручного обновления всех сервисов, она выполняется редко или никогда. Автоматизированные системы ротации (например, динамические credentials Vault или automatic rotation в AWS Secrets Manager) позволяют менять пароли каждые несколько часов или даже минут без участия операторов и без перезапуска приложений, поддерживающих горячее перечитывание конфигурации.

### Сценарий 5: Безопасное хранение TLS-сертификатов и ключей

Взаимная TLS-аутентификация между сервисами требует приватных ключей на каждом узле. Хранение ключей в ConfigMap или переменных окружения нарушает принцип минимальных привилегий и делает ключи уязвимыми для любого процесса с доступом к API Kubernetes или Docker. Docker Secrets и Kubernetes Secrets позволяют монтировать сертификаты и ключи как файлы с ограниченными правами доступа (0400), доступные только целевому пользователю внутри контейнера. Интеграция с cert-manager автоматизирует выпуск и обновление сертификатов от Let's Encrypt или внутреннего PKI.

## Основные концепции

### Docker Secrets: нативное управление секретами в Swarm

**Docker Secrets** — это объект Docker Swarm, предназначенный для хранения и распространения конфиденциальных данных внутри кластера. Секрет создаётся централизованно через API менеджера и реплицируется на worker-узлы через защищённый Raft-протокол. При запуске сервиса секрет монтируется в файловую систему контейнера как файл в директории `/run/secrets/`, которая является **tmpfs** (временной файловой системой в памяти). Это гарантирует, что содержимое секрета никогда не записывается на постоянный носитель и исчезает при остановке контейнера.

Важное ограничение: Docker Secrets доступны только в режиме Swarm. Для standalone-контейнеров или docker-compose на одном хосте необходимо использовать альтернативные решения. При обновлении сервиса с новой версией секрета Docker пересоздаёт контейнер, поэтому приложение должно быть готово к кратковременному перезапуску. Docker не предоставляет встроенной ротации — секреты изменяются вручную или через внешние скрипты.

### Kubernetes Secrets: декларативное хранение чувствительных данных

**Kubernetes Secrets** — это ресурс API Kubernetes, предназначенный для хранения небольших объёмов конфиденциальных данных (до 1 МБ). Данные хранятся в кодировке base64, что является транспортной механикой, а не мерой безопасности. Секрет может быть смонтирован в контейнер как файл или передан как переменная окружения, хотя последний способ не рекомендуется из соображений безопасности. По умолчанию Secrets хранятся в etcd без шифрования, что требует явной настройки **EncryptionConfiguration** с KMS-провайдером для защиты данных at rest.

Kubernetes предоставляет несколько типов секретов: **Opaque** (общего назначения), **tls** (для сертификатов и ключей), **docker-registry** (для аутентификации в registry), **basic-auth** и **ssh-auth**. RBAC позволяет контролировать, какие сервисные аккаунты и пользователи могут читать или модифицировать секреты в каждом namespace. Для защиты от случайной утечки рекомендуется использовать **Immutable Secrets** (Kubernetes 1.21+), которые нельзя изменить после создания — при необходимости обновления создаётся новый объект.

### Внешние менеджеры секретов: HashiCorp Vault и облачные решения

**HashiCorp Vault** — это специализированное решение для управления секретами, предлагающее значительно больший набор функций, чем встроенные механизмы Docker и Kubernetes. Vault поддерживает динамические секреты (временные credentials баз данных и облачных сервисов), PKI для выпуска сертификатов, шифрование данных (transit secrets engine), identity-based доступ с интеграцией LDAP, Kubernetes, AWS IAM и других провайдеров аутентификации.

Ключевое преимущество Vault — **lease-based** модель, где каждый секрет имеет ограниченное время жизни и автоматически отзывается по истечении TTL. Это кардинально снижает риск от компрометации: даже если злоумышленник перехватит credentials, они станут недействительными через час, день или неделю в зависимости от политики. Vault также предоставляет подробный аудит всех обращений к секретам, включая идентификатор запросившего, время, путь и результат операции.

**AWS Secrets Manager**, **Azure Key Vault** и **GCP Secret Manager** предлагают управляемые облачные альтернативы с интеграцией в IAM-политики соответствующих платформ. AWS Secrets Manager поддерживает автоматическую ротацию для RDS-credentials через Lambda-функции, что особенно удобно для приложений в AWS. Эти сервисы избавляют от необходимости разворачивать и поддерживать собственный Vault, но создают vendor lock-in и требуют сетевого доступа к облачным API.

### ConfigMaps и разделение конфигурации от секретов

**ConfigMap** — это Kubernetes-ресурс для хранения неконфиденциальной конфигурации: параметров приложения, адресов сервисов, шаблонов файлов. В отличие от Secrets, ConfigMap не предназначен для чувствительных данных и не требует шифрования at rest. Правильное разделение между ConfigMap и Secret критически важно для безопасности: пароли, токены и ключи должны попадать в Secret, а публичные параметры — в ConfigMap.

В Docker Compose аналогичную роль выполняют обычные файлы конфигурации, монтируемые как volumes. Важно избегать смешивания секретов и конфигурации в одном файле — практика, ведущая к утечкам при коммите в репозиторий. Инструменты вроде **git-secrets** и **pre-commit hooks** помогают автоматически обнаруживать и блокировать попытки закоммитить чувствительные данные.

### Шифрование at rest и in transit

**Encryption at rest** означает шифрование данных на постоянном носителе. Для Docker Secrets шифрование at rest обеспечивается автоматически при использовании Docker Swarm — секреты шифруются в Raft-логах. Для Kubernetes требуется явная настройка KMS-провайдера в kube-apiserver. **Encryption in transit** обеспечивается TLS для всех коммуникаций между компонентами Docker Swarm и Kubernetes, а также между приложениями и внешними менеджерами секретов. NIST SP 800-53 требует использования FIPS-валидированных криптографических модулей для шифрования данных в покое и при передаче.

### Минимальные привилегии для секретов

**Principle of Least Privilege** (принцип минимальных привилегий) применяется к секретам в нескольких измерениях. Во-первых, сервис должен иметь доступ только к тем секретам, которые ему необходимы для работы — не более. Во-вторых, доступ к секретам должен быть ограничен по времени: временные credentials предпочтительнее долгоживущих. В-третьих, аудит должен фиксировать каждое обращение к секрету для расследований и compliance. В-четвёртых, секреты должны ротироваться регулярно, чтобы уменьшить окно эксплуатации в случае компрометации.

## Сравнение с другими механизмами

| Механизм | Доступность | Шифрование at rest | Динамические секреты | Ротация | Аудит | Интеграция с Kubernetes |
|----------|-------------|-------------------|----------------------|---------|-------|------------------------|
| Docker Secrets | Только Swarm | Да (Raft) | Нет | Ручная | Ограниченный | Нет |
| Kubernetes Secrets | Нативно | Только с KMS | Нет | Внешние операторы | Через API audit | Нативно |
| HashiCorp Vault | Self-hosted / Managed | Да | Да | Автоматическая | Полный | CSI Driver, Agent Injector |
| AWS Secrets Manager | AWS Cloud | Да | Да (Lambda) | Автоматическая | CloudTrail | External Secrets Operator |
| Azure Key Vault | Azure Cloud | Да | Да | Автоматическая | Azure Monitor | External Secrets Operator |
| GCP Secret Manager | GCP | Да | Да | Автоматическая | Cloud Audit Logs | External Secrets Operator |
| Environment Variables | Везде | Нет | Нет | Ручная | Нет | Через Pod spec |

Анализ компромиссов:

**Docker Secrets** предоставляет простое нативное решение для Swarm-кластеров, но его применимость ограничена — Swarm практически вытеснен Kubernetes. Шифрование at rest работает из коробки, но отсутствие динамических секретов и автоматической ротации делает Docker Secrets непригодным для высокочувствительных данных с жёсткими требованиями compliance.

**Kubernetes Secrets** — универсальный базовый механизм, доступный в любом кластере без дополнительной инфраструктуры. Однако шифрование по умолчанию отсутствует, base64 создаёт ложное ощущение безопасности, а доступ к etcd для cluster-admin раскрывает все секреты. Kubernetes Secrets подходят для умеренно чувствительных данных в доверенных кластерах, но не для критических credentials без дополнительных мер защиты.

**HashiCorp Vault** — наиболее функциональное решение с динамическими секретами, lease-based моделью, полным аудитом и поддержкой множества методов аутентификации. Цена за функциональность — сложность развёртывания, необходимость высокой доступности (Vault не должен быть единой точкой отказа), кривая обучения операторов. Vault также требует своего процесса backup и disaster recovery.

**Облачные менеджеры секретов** (AWS, Azure, GCP) предлагают оптимальный баланс для организаций, уже использующих соответствующие облака. Управляемый сервис снимает операционную нагрузку, автоматическая ротация работает из коробки, интеграция с IAM естественна. Недостатки: vendor lock-in, стоимость использования, необходимость сетевого доступа к облаку (проблематично для on-premise), ограниченная кастомизация по сравнению с Vault.

**Переменные окружения** остаются самым простым и поддерживаемым везде механизмом, но с худшим профилем безопасности. Они видны в `/proc/<pid>/environ`, логах, дампах памяти, истории команд. Использование environment variables для секретов противоречит рекомендациям CIS Docker Benchmark, OWASP Container Security Verification Standard и NIST guidelines.

## Уроки из реальных индидентов

### Инцидент Uber (2016): утечка данных 57 миллионов пользователей через hardcoded credentials

В 2016 году Uber столкнулся с одной из крупнейших утечек данных в истории. Злоумышленники получили доступ к облачной инфраструктуре Uber, использовав credentials, hardcoded в коде приложения и попавшие в систему контроля версий. Несмотря на то, что инцидент произошёл в контексте облачной инфраструктуры, а не Docker, уроки применимы к любой среде: хранение secrets в коде, конфигурационных файлах или переменных окружения, зафиксированных в системе контроля версий, создаёт постоянную угрозу. Если Uber использовал централизованный менеджер секретов с динамическими credentials и автоматической ротацией, компрометация кода не привела бы к длительному доступу к инфраструктуре. Компания заплатила 148 миллионов долларов в рамках урегулирования и понесла огромный репутационный ущерб.

### Инцидент Tesla (2018): криптомайнинг в Kubernetes через утечку dashboard credentials

В 2018 году облачная инфраструктура Tesla была скомпрометирована через открытый в интернет Kubernetes dashboard, защищённый слабым паролем. Злоумышленники получили доступ к кластеру, обнаружили в Kubernetes Secrets credentials для AWS IAM-роли с широкими правами и запустили криптомайнинг на вычислительных мощностях Tesla. Инцидент продолжался месяцами, потому что нагрузка майнеров маскировалась под легитимные процессы. Уроки для управления секретами: нехранение Kubernetes Secrets в etcd без шифрования, использование слабой аутентификации для административных интерфейсов, отсутствие мониторинга на аномальную активность. Если Tesla использовала Vault или AWS Secrets Manager с IAM-ролями, ограниченными по принципу least privilege, ущерб был бы минимизирован. Также критически важна сетевая изоляция: Kubernetes dashboard не должен быть доступен из интернета без VPN.

## Архитектурные решения

### Defense-in-Depth для секретов

**Defense-in-Depth** (защита в глубину) требует нескольких независимых слоёв защиты секретов. Первый слой — разделение конфигурации и секретов: ConfigMap для публичных параметров, Secret или внешний менеджер для чувствительных данных. Второй слой — шифрование at rest: etcd Kubernetes должен быть защищён KMS-провайдером, Vault должен использовать seal/unseal механизм с ключами, разделёнными между доверенными лицами. Третий слой — шифрование in transit: все коммуникации между приложениями и менеджерами секретов проходят по TLS. Четвёртый слой — контроль доступа: RBAC в Kubernetes, IAM-политики в облаке, ACL в Vault ограничивают доступ только необходимым субъектам. Пятый слой — аудит: все операции с секретами логируются для расследований и compliance. Шестой слой — ротация: регулярная смена секретов сокращает окно эксплуатации. Седьмой слой — мониторинг: аномальная активность при доступе к секретам (необычное время, неизвестный IP, резкий рост запросов) триггерит алерты.

### Zero Trust для контейнерных секретов

**Zero Trust** архитектура отказывается от предположения о доверии внутри периметра. Для секретов это означает: никогда не доверять сервису без проверки, даже если он работает внутри кластера. Каждый запрос секрета должен быть аутентифицирован и авторизован. Vault реализует Zero Trust через методы аутентификации (Kubernetes auth, AWS IAM, LDAP), где сервис получает временный токен Vault, обмениваемый на конкретные секреты согласно политикам. В Kubernetes Zero Trust достигается через **Workload Identity**: pod получает идентификатор, аутентифицируется в IAM-провайдере и получает секреты без статических credentials. **SPIFFE/SPIRE** предоставляет стандартизированную систему идентификации рабочих нагрузок с автоматической выдачой SVID (SPIFFE Verifiable Identity Document), которыми сервисы аутентифицируются друг с другом и в менеджерах секретов.

## Подготовка к собеседованию

### Три типичные ошибки кандидатов

**Ошибка 1: Считать Kubernetes Secrets безопасными из коробки**

Многие кандидаты полагают, что Kubernetes Secrets автоматически шифруют данные в etcd. На самом деле, по умолчанию Secrets хранятся в base64 — формате, который легко декодировать. Шифрование at rest требует явной настройки EncryptionConfiguration с KMS-провайдером. Кроме того, любой пользователь или сервисный аккаунт с правами `get secrets` в namespace может прочитать все секреты. Без RBAC и NetworkPolicies Kubernetes Secrets создают ложное чувство безопасности.

**Ошибка 2: Предлагать переменные окружения как основной способ передачи секретов**

Хотя environment variables удобны и поддерживаются везде, их использование для секретов является антипаттерном с точки зрения безопасности. Переменные окружения видны в `/proc/<pid>/environ` любому процессу внутри контейнера, попадают в логи при ошибках, отображаются в инструментах отладки типа `kubectl describe pod`. CIS Docker Benchmark и OWASP рекомендуют избегать environment variables для чувствительных данных в пользу файловых монтирований или интеграции с менеджерами секретов.

**Ошибка 3: Игнорировать проблему секретов при коммите в репозиторий**

Кандидаты часто фокусируются на runtime-защите секретов и забывают о этапе разработки. Hardcoded credentials в коде, конфигурационных файлах или docker-compose.yml попадают в Git-историю навсегда, даже если позже удалены. Инструменты вроде git-secrets, TruffleHog или GitHub secret scanning должны быть интегрированы в CI/CD pipeline для автоматического обнаружения. Идеально — использовать внешний менеджер секретов и шаблоны (например, Vault Agent Injector или External Secrets Operator), чтобы приложение никогда не имело доступа к raw credentials во время разработки.

### Три вопроса с ответами

**Вопрос 1: Как защитить Kubernetes Secrets в etcd от чтения при компрометации узла?**

Для защиты Secrets at rest необходимо настроить EncryptionConfiguration в kube-apiserver с KMS-провайдером. Поддерживаются несколько провайдеров: aescbc, aesgcm, secretbox, а также внешние KMS через KMS Plugin API (например, AWS KMS, Azure Key Vault, HashiCorp Vault Transit). При использовании KMS ключ шифрования etcd хранится в защищённом хранилище, а не в конфигурации Kubernetes. Дополнительно следует ограничить доступ к etcd: запустить etcd на выделенных узлах, защитить peer и client коммуникации TLS, включить firewall-правила, разрешающие подключение к etcd только от API-серверов. Регулярный backup etcd должен храниться в зашифрованном виде. NIST SP 800-53 требует использования FIPS 140-2 валидированных криптографических модулей для шифрования.

**Вопрос 2: Объясните разницу между статическими и динамическими секретами на примере HashiCorp Vault**

**Статические секреты** (static secrets) хранятся в Vault как есть и извлекаются по запросу. Они не меняются, пока оператор не обновит их вручную. Примеры: API-ключи сторонних сервисов, пароли приложений, TLS-сертификаты. Риск статических секретов — долгоживущий характер: если скомпрометированы, злоумышленник имеет доступ до ручной ротации.

**Динамические секреты** (dynamic secrets) генерируются Vault по запросу и имеют ограниченное время жизни (TTL). Например, при обращении к database secrets engine Vault создаёт временного пользователя в PostgreSQL с нужными правами и возвращает его credentials. По истечении TTL или явному отзыву пользователь автоматически удаляется. Динамические секреты гарантируют, что каждый экземпляр приложения получает уникальные credentials, а компрометация одного экземпляра не влияет на другие. Это реализация Zero Trust на уровне credentials.

**Вопрос 3: Как реализовать безопасное хранение секретов в Docker без Docker Swarm?**

Для standalone Docker и docker-compose Docker Secrets недоступны. Альтернативы: (1) Файловые монтирования — создать секреты на хосте с правами 0400, смонтировать в контейнер как read-only volume. Недостаток: секреты хранятся на хосте в открытом виде. (2) Внешние менеджеры секретов — интегрировать приложение с Vault, AWS Secrets Manager или аналогом через SDK или sidecar-контейнер. Приложение аутентифицируется и получает секреты во время старта. (3) Init-контейнеры в Kubernetes (если используется Kubernetes вместо Docker Compose) — init-контейнер с правами на чтение Vault получает секреты, записывает их в shared emptyDir volume, основной контейнер читает из файла. (4) Docker BuildKit secrets — для безопасной передачи секретов на этапе сборки образа без попадания в слои. В любом случае environment variables для секретов следует избегать.

### Ответы на чек-лист

1. **Docker Secrets** монтируются в контейнер как файлы в tmpfs, которая существует только в памяти и исчезает при остановке контейнера. Переменные окружения, напротив, видны любому процессу через `/proc/<pid>/environ`, попадают в логи систем, дампы памяти, историю команд оболочки. Docker Secrets доступны только авторизованным сервисам в Swarm, тогда как переменные окружения читаются любым пользователем с доступом к `docker inspect`. Однако Docker Secrets требуют Swarm-режима и не работают в standalone Docker, что ограничивает их применимость.

2. Kubernetes Secrets по умолчанию хранят данные в **etcd** в кодировке base64, которая является односторонней трансформацией, а не шифрацией — base64 легко декодируется. Это означает, что любой, имеющий доступ к etcd (например, администратор кластера с правами на чтение файловой системы узла etcd), может извлечь все Secrets. Шифрование at rest требует явной настройки EncryptionConfiguration с KMS-провайдером, что часто пропускается при стандартных развёртываниях. Кроме того, по умолчанию Secrets доступны любому сервисному аккаунту в том же namespace без дополнительного RBAC.

3. **Lease-based модель** в HashiCorp Vault предполагает, что каждый выданный секрет имеет ограниченное время жизни (TTL) и должен быть продлён (renewed) или явно отозван (revoked). Для динамических секретов (временных пользователей баз данных, облачных credentials) Vault автоматически удаляет объект по истечении TTL. Это обеспечивает временную изоляцию: даже если credentials перехвачены, они становятся бесполезными через заданный интервал. Lease-based модель требует от приложений способности обновлять секреты без перезапуска или перезапуска по расписанию, что добавляет сложности архитектуре.

4. Хранение секретов в Git-репозитории создаёт **постоянную неизменяемую запись** — даже при удалении секрета из текущей версии он остаётся в истории коммитов. Любой, имеющий доступ к репозиторию, может извлечь секрет из старых коммитов. Кроме того, если репозиторий публичный или станет публичным (случайная настройка, приобретение компании), секреты становятся доступны всему интернету. Инструменты сканирования (TruffleHog, git-secrets) должны блокировать коммиты с секретами, но лучшая практика — никогда не коммитить raw credentials, используя внешние менеджеры секретов.

5. **Динамические секреты** генерируются по запросу с уникальными credentials для каждого потребителя и автоматически истекают через заданное время. Это обеспечивает несколько преимуществ: (а) каждый сервис имеет уникальные credentials, что упрощает аудит и расследования; (б) компрометация одного сервиса не влияет на другие; (в) отсутствие долгоживущих credentials снижает риск; (г) автоматическое истечение исключает забытые credentials при удалении сервиса. Статические секреты проще в реализации, но требуют ручной ротации и создают более широкую поверхность атаки.

6. Шифрование etcd в Kubernetes настраивается через файл **EncryptionConfiguration**, указываемый в параметре `--encryption-provider-config` kube-apiserver. Файл определяет список провайдеров шифрования в порядке приоритета: kms (для внешних KMS), aescbc, aesgcm, secretbox. При использовании KMS-провайдера ключ шифрования хранится вне кластера (AWS KMS, Azure Key Vault, Vault Transit), что обеспечивает дополнительную изоляцию. После изменения конфигурации необходимо выполнить `kubectl rewrite` для перешифрования существующих Secrets. NIST SP 800-53 требует регулярной ротации ключей шифрования.

7. **ConfigMap** предназначен для хранения неконфиденциальной конфигурации: параметров приложения, адресов сервисов, шаблонов файлов. Данные в ConfigMap не шифруются и видны всем, кто имеет доступ к namespace. **Secret** предназначен для чувствительных данных: паролей, токенов, ключей. Хотя Secret технически хранит данные в base64, Kubernetes и инструменты оркестрации обрабатывают его с большей осторожностью: ограниченный доступ в kubectl get, отдельные политики RBAC, возможность монтирования как файлы с правами 0400. Смешивание секретов и конфигурации в ConfigMap является антипаттерном безопасности.

8. **Principle of Least Privilege** для секретов означает предоставление доступа только к необходимым секретам, только необходимым субъектам, только на необходимое время. В Kubernetes это реализуется через RBAC: сервисный аккаунт приложения получает права на чтение только конкретных Secrets в своём namespace. В Vault — через политики ACL, определяющие пути и операции, доступные каждому аутентифицированному субъекту. В облаках — через IAM-роли с условиями (IP-диапазоны, время, MFA). Временные credentials предпочтительнее долгоживущих, а динамические секреты — статических.

9. **External Secrets Operator (ESO)** — это Kubernetes-оператор, синхронизирующий секреты из внешних хранилищ (Vault, AWS Secrets Manager, Azure Key Vault, GCP Secret Manager) в Kubernetes Secrets. ESO позволяет централизованно управлять секретами во внешнем менеджере (с его аудитом, ротацией, политиками), при этом приложения в Kubernetes продолжают использовать привычный механизм Secrets и монтирования файлов. ESO решает проблему vendor lock-in на уровне приложения: при смене менеджера секретов достаточно обновить конфигурацию ESO, а не модифицировать код приложений. ESO поддерживает templating, позволяя преобразовывать внешние секреты в нужный формат Kubernetes Secret.

10. Инцидент Tesla продемонстрировал, что **Kubernetes Secrets в сочетании с слабой аутентификацией административных интерфейсов создают критическую уязвимость**. Злоумышленники получили доступ к открытому Kubernetes dashboard, обнаружили в Secrets credentials для AWS IAM и запустили криптомайнинг. Уроки: (а) Kubernetes dashboard не должен быть публично доступен; (б) Secrets должны храниться с шифрованием etcd; (в) IAM-роли должны следовать least privilege; (г) необходим мониторинг аномальной активности. Если Tesla использовала Vault с динамическими AWS credentials вместо статических ключей в Secrets, ущерб был бы ограничен временем жизни lease.

## Ключевые выводы для собеседования

1. **Docker Secrets и Kubernetes Secrets — это базовые механизмы, но недостаточные для высокочувствительных данных без дополнительных мер.** Docker Secrets ограничены Swarm, а Kubernetes Secrets требуют явного шифрования etcd и строгого RBAC. Для production-grade защиты необходима интеграция с внешними менеджерами (Vault, облачные KMS) или External Secrets Operator.

2. **Переменные окружения для секретов — антипаттерн безопасности, запрещённый CIS Benchmarks и OWASP.** Файловые монтирования через tmpfs (Docker Secrets) или Kubernetes Secrets с readOnly volumeMounts предпочтительнее. Environment variables видны в `/proc`, логах, истории команд и создают ненужную поверхность для утечек.

3. **Динамические секреты с автоматической ротацией — золотой стандарт для критических систем.** HashiCorp Vault и облачные менеджеры секретов предоставляют lease-based модель, временные credentials и аудит, которые кардинально снижают риск от компрометации. Статические долгоживущие credentials должны быть исключены где возможно.

4. **Defense-in-Depth и Zero Trust применимы к секретам через многослойную защиту: шифрование at rest/in transit, строгий RBAC, аудит, ротация, мониторинг аномалий, workload identity.** Ни один слой не является достаточным сам по себе; их комбинация создаёт устойчивую архитектуру безопасности для контейнерных сред.
