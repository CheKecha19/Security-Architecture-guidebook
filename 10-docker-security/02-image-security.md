# Docker Image Security: Безопасность образов контейнеров

## Что такое Docker-образ?

Представьте, что вы собираетесь в поездку. Вместо того чтобы каждый раз собирать чемодан с нуля, вы создаёте «шаблон чемодана»: в него уже положены все необходимые вещи — зубная щётка, запасные носки, зарядка для телефона. Каждая поездка начинается с этого шаблона, и вы добавляете только специфичные вещи: купальник для пляжа или термобельё для гор. Docker-образ — это как раз такой «шаблон чемодана» для приложения.

Docker-образ — это неизменяемый файл, содержащий всё необходимое для запуска приложения: код, библиотеки, зависимости, системные утилиты, конфигурационные файлы. Образ состоит из слоёв, где каждый слой — это набор изменений файловой системы. Базовый слой может содержать минимальную операционную систему, следующий — Java Runtime, следующий — код приложения. Все слои складываются в единую файловую систему через UnionFS.

Ключевая особенность образа — его неизменяемость. Образ нельзя изменить после создания. Если нужно что-то поменять — создаётся новый образ. Это как печатная книга: если автор хочет исправить опечатку, он выпускает новое издание, а не исправляет каждый проданный экземпляр. Эта неизменяемость обеспечивает воспроизводимость: один и тот же образ всегда ведёт себя одинаково, где бы он ни запустился.

## Зачем это нужно?

**Сценарий 1: Компрометация базового образа.** В 2024 году злоумышленники взломали учётную запись на Docker Hub и загрузили вредоносную версию популярного образа `python:3.9`. Тысячи компаний автоматически подтянули этот образ при сборке своих приложений. Вредоносный код оказался внутри контейнеров банков, госучреждений, стартапов. Если бы компании сканировали образы перед использованием — атака была бы обнаружена за минуты.

**Сценарий 2: Уязвимости в зависимостях.** HR-платформа Сбера использует базовый образ `openjdk:11-jre`, который содержит уязвимую версию библиотеки log4j (CVE-2021-44228). Все микросервисы, собранные на этом базовом образе, наследуют уязвимость. Пока команда безопасности проверяет production-серверы, злоумышленник уже эксплуатирует log4shell в контейнерах. Регулярное сканирование образов на уязвимости позволяет находить такие проблемы до развёртывания.

**Сценарий 3: Секреты в образах.** Разработчик для удобства добавил в Dockerfile команду `ENV DATABASE_PASSWORD=supersecret123`. Этот секрет оказался внутри образа, доступным любому, кто может его скачать. Даже если позже разработчик удалил переменную из Dockerfile — секрет остался в истории слоёв и виден через `docker history`. Сканирование образов на наличие секретов — обязательная практика.

## Основные концепции

### Многослойная архитектура образов

Docker-образ строится по принципу слоёв. Каждая инструкция в Dockerfile создаёт новый слой. Базовый слой содержит операционную систему. Следующий слой добавляет зависимости. Следующий — код приложения. Каждый слой кэшируется: если базовый слой не изменился — Docker использует кэш и не пересобирает его.

Эта архитектура имеет важное следствие для безопасности: слои наследуются. Если базовый слой содержит уязвимость — все производные образы её наследуют. Если базовый слой скомпрометирован — все производные образы скомпрометированы. Это как если бы в фабрике, выпускающей кирпичи, один из компонентов бетона был ядовит — все дома, построенные из этих кирпичей, будут опасны.

С другой стороны, многослойность позволяет быстро обновлять образы. Если вышла новая версия базового образа с исправлением уязвимости — достаточно пересобрать производные образы, и они автоматически получат исправление. Docker использует кэш для неизменных слоёв, поэтому пересборка идёт быстро.

### Базовые образы: Откуда начинается всё

Базовый образ — это фундамент, на котором строится всё остальное. Популярные базовые образы:

**Alpine Linux** — минималистичный дистрибутив, всего 5 МБ. Содержит минимум пакетов, что снижает поверхность атаки. Но использует musl libc вместо glibc, что может вызывать проблемы совместимости.

**Debian Slim** — облегчённая версия Debian, около 30 МБ. Хороший баланс между размером и совместимостью. Использует glibc, работает с большинством приложений.

**Distroless** — образы от Google, содержащие только приложение и его зависимости, без shell, без пакетного менеджера, без curl. Поверхность атаки минимальна, но отладка невозможна без дополнительных инструментов.

**Scratch** — пустой образ. Используется для статически скомпилированных приложений (Go, Rust). Нулевая поверхность атаки, но требует статической линковки всех зависимостей.

Выбор базового образа — архитектурное решение. Alpine экономит место и снижает атакуемую поверхность, но может ломать приложения. Distroless максимально безопасен, но сложен в отладке. Debian — универсальный выбор для enterprise.

### Сканирование образов на уязвимости

Сканирование образов — это процесс автоматического анализа содержимого образа на наличие известных уязвимостей. Сканер разбирает слои образа, извлекает установленные пакеты и библиотеки, сравнивает их версии с базой данных уязвимостей (CVE — Common Vulnerabilities and Exposures).

**Clair** — open-source сканер от CoreOS (теперь Red Hat). Работает статически: анализирует слои образа без запуска контейнера. Интегрируется с Harbor, Quay, GitLab CI.

**Trivy** — лёгкий сканер от Aqua Security. Поддерживает OS-пакеты, библиотеки приложений, секреты, misconfigurations. Работает быстро, имеет интеграцию с CI/CD.

**Snyk** — коммерческий сканер с бесплатным тарифом для open-source. Предоставляет детальные рекомендации по исправлению, отслеживает новые уязвимости в реальном времени.

**Grype** — сканер от Anchore. Open-source, поддерживает SBOM (Software Bill of Materials) форматы.

Сканирование должно происходить на нескольких этапах: при сборке образа в CI/CD, перед размещением в registry, регулярно в production (новые уязвимости появляются каждый день). Обнаружение критической уязвимости должно блокировать развёртывание.

### Минимизация образов

Чем меньше образ — тем меньше поверхность атаки. Минимизация достигается несколькими техниками:

**Многоэтапная сборка (Multi-stage builds).** Dockerfile содержит несколько этапов. Первый этап собирает приложение с полным набором инструментов (компилятор, тестовые библиотеки). Второй этап копирует только собранный артефакт в минимальный образ. В финальном образе нет компилятора, исходников, тестовых зависимостей — только то, что нужно для работы.

**Удаление ненужных пакетов.** После установки зависимостей удаляйте кэш пакетного менеджера. Не устанавливайте curl, wget, ssh, vim в production-образ — они нужны только для отладки.

**Использование Distroless.** Образы без shell и пакетного менеджера. Даже если злоумышленник проникнет в контейнер — он не сможет запустить shell, установить инструменты, просканировать сеть.

**Non-root пользователь.** По умолчанию контейнер запускается от root. Создайте в Dockerfile обычного пользователя и запускайте приложение от его имени. Даже если злоумышленник получит доступ — он не будет root.

### Подпись и верификация образов

Docker Content Trust (DCT) — механизм подписи образов с помощью цифровых подписей. Когда разработчик публикует образ, он подписывает его своим приватным ключом. Когда пользователь скачивает образ, Docker проверяет подпись публичным ключом. Если подпись не совпадает — образ не запускается.

Это как печать на документе: государственная печать гарантирует подлинность документа. Цифровая подпись гарантирует, что образ создан доверенным автором и не был изменён в пути.

Notary — open-source система доверия для Docker. Хранит подписи, управляет ключами, поддерживает ротацию ключей. Интегрируется с Docker Content Trust.

Cosign — современный инструмент от Sigstore для подписи и верификации контейнеров. Проще Notary, не требует инфраструктуры ключей, использует прозрачные логи.

В enterprise-среде подпись образов обязательна: только образы, подписанные CI/CD системой, могут разворачиваться в production. Это предотвращает развёртывание образов, собранных на ноутбуке разработчика или скачанных из недоверенного источника.

## Практические применения

**Сценарий 1: Политика «только проверенные образы».** В Сбере установлена политика: все Docker-образы должны быть собраны из корпоративного GitLab CI и подписаны корпоративным ключом. Kubernetes отказывается запускать образ без подписи. Разработчик не может запустить образ с Docker Hub — только образы из корпоративного Harbor с валидной подписью. Это предотвращает случайное использование скомпрометированных образов.

**Сценарий 2: Автоматическое сканирование в CI/CD.** При каждом коммите GitLab CI собирает Docker-образ и запускает Trivy. Если Trivy находит критическую уязвимость — pipeline падает, разработчик получает отчёт. Если уязвимость средняя — pipeline проходит, но создаётся тикет на исправление. Раз в неделю запускается полное сканирование всех образов в registry на предмет новых CVE.

**Сценарий 3: Многоэтапная сборка для безопасности.** Dockerfile для HR-сервиса имеет три этапа. Этап «builder» содержит JDK, Maven, исходники — собирает jar-файл. Этап «test» запускает интеграционные тесты. Этап «production» копирует только jar-файл в distroless образ с JRE. Финальный образ весит 50 МБ вместо 500 МБ, не содержит shell, curl, ssh — поверхность атаки минимальна.

**Сценарий 4: Управление секретами в образах.** Компания использует Aqua Security для сканирования образов на наличие секретов (паролей, токенов, ключей API). Во время аудита Aqua находит в образе закоммиченный `.env` файл с паролем от production-базы. Политика безопасности блокирует развёртывание. Разработчик переносит секреты в HashiCorp Vault, образ пересобирается без секретов. Секреты передаются контейнеру через переменные окружения во время запуска.

**Сценарий 5: Регулярный пересмотр базовых образов.** Команда безопасности раз в квартал анализирует используемые базовые образы. Устаревший `openjdk:8-jre` заменяется на `eclipse-temurin:17-jre-alpine`. Образы на CentOS 7 (End-of-Life) мигрируют на Rocky Linux. Это предотвращает использование образов с неподдерживаемой ОС, в которых не будут исправляться уязвимости.

## Ограничения и нюансы

**Сканирование не находит неизвестные уязвимости.** Сканеры сравнивают версии пакетов с базой CVE. Если уязвимость ещё не зарегистрирована (zero-day) — сканер её не найдёт. Сканирование — не панацея, а один из слоёв защиты.

**False positives.** Сканер может сообщить об уязвимости, которая на самом деле не эксплуатируема в данном контексте. Например, уязвимость в OpenSSL, связанная с TLS 1.0, но приложение использует только TLS 1.3. Нужна экспертиза для приоритизации находок.

**Секреты в истории слоёв.** Даже если удалить секрет из текущего Dockerfile — он остаётся в истории слоёв и виден через `docker history`. Для полного удаления нужно пересобрать образ с изменёнными инструкциями. Лучшая практика — не класть секреты в образ вообще.

**Базовые образы — точка доверия.** Мы доверяем Docker Hub, Alpine Linux, Debian — но эти проекты тоже могут быть скомпрометированы. В 2024 году произошла атака на XZ Utils (CVE-2024-3094), когда вредоносный код был внедрён в популярную библиотеку сжатия. Никакой сканер не нашёл бы эту уязвимость, потому что она была backdoor, а не известный CVE.

**Distroless усложняет отладку.** Если приложение падает в distroless-контейнере — нельзя зайти внутрь через `docker exec` и посмотреть логи, потому что нет shell. Нужны внешние системы сбора логов (ELK, Splunk) и инструменты отладки (delve для Go, jcmd для Java).

## Архитектурные решения

**Immutable Infrastructure.** Образы никогда не изменяются после создания. Обновление = новый образ + rolling deployment. Это позволяет легко откатиться к предыдущей версии и гарантирует, что production идентичен staging.

**Shift Left Security.** Сканирование образов происходит на самых ранних этапах: при коммите кода, при сборке, перед публикацией. Чем раньше найдена уязвимость — тем дешевле её исправить. Уязвимость, найденная в production, стоит в 100 раз дороже, чем уязвимость, найденная в CI.

**SBOM для прозрачности.** Software Bill of Materials — это «ингредиенты» образа: список всех пакетов, библиотек, их версий, лицензий. SBOM позволяет быстро ответить на вопрос: «Используем ли мы уязвимую версию log4j?» Генерация SBOM при сборке — обязательная практика для enterprise.

**Defense in Depth.** Сканирование образов — один слой. Другие слои: runtime-защита (Falco), сетевые политики, secrets management, least privilege, regular patching. Ни один слой не заменяет остальные.

### Ответы на чек-лист

1. **Чем Docker-образ отличается от Docker-контейнера?** Образ — это неизменяемый шаблон, содержащий всё необходимое для запуска приложения. Контейнер — это запущенный экземпляр образа, работающий процесс с собственными namespaces и cgroups. Образ нельзя изменить; контейнер можно изменить во время работы, но изменения не сохраняются в образе. Аналогия: образ — это чертёж дома, контейнер — это построенный дом.

2. **Почему базовый образ критичен для безопасности?** Все производные образы наследуют слои базового образа. Если базовый образ содержит уязвимую версию библиотеки — все дочерние образы наследуют эту уязвимость. Если базовый образ скомпрометирован (вредоносный код внедрён на этапе сборки) — все производные образы скомпрометированы. Базовый образ — это фундамент: если фундамент треснул — весь дом под угрозой.

3. **Как многоэтапная сборка повышает безопасность?** Многоэтапная сборка разделяет процесс на этапы: сборка (builder), тестирование, production. В production-этап копируются только собранные артефакты, а инструменты сборки (компилятор, исходники, тестовые библиотеки) не попадают в финальный образ. Это уменьшает размер образа и снижает поверхность атаки: в production-контейнере нет gcc, make, curl, ssh — инструментов, которыми мог бы воспользоваться злоумышленник.

4. **Что такое CVE и как сканеры их находят?** CVE (Common Vulnerabilities and Exposures) — это система идентификации публично известных уязвимостей. Каждой уязвимости присваивается идентификатор (например, CVE-2021-44228 для Log4Shell). Сканер разбирает слои Docker-образа, извлекает список установленных пакетов и библиотек с их версиями, сравнивает этот список с базой данных CVE. Если версия пакета совпадает с уязвимой — сканер сообщает о находке.

5. **Почему секреты в Dockerfile опасны?** Dockerfile создаёт слои, и каждая инструкция сохраняется в истории образа. Команда `ENV PASSWORD=secret123` создаёт слой, содержащий секрет. Любой, у кого есть доступ к образу, может просмотреть историю слоёв (`docker history`) и увидеть секрет. Даже если позже удалить переменную из Dockerfile — старый слой с секретом остаётся в репозитории. Правильный подход: передавать секреты через переменные окружения во время запуска, не включая их в образ.

6. **Как Docker Content Trust защищает от подделки образов?** Docker Content Trust использует цифровые подписи для верификации образов. Разработчик подписывает образ приватным ключом перед публикацией. Пользователь настраивает Docker на проверку подписей: при скачивании образа Docker проверяет подпись публичным ключом. Если подпись невалидна (образ изменён или создан третьей стороной) — Docker отказывается скачивать образ. Это защищает от man-in-the-middle атак и подделки образов в registry.

7. **Чем distroless отличается от Alpine?** Alpine — минимальная ОС с shell, пакетным менеджером (apk), базовыми утилитами. Distroless — не ОС, а минимальное окружение, содержащее только runtime для приложения (например, JRE) без shell, без пакетного менеджера, без curl. Alpine весит 5 МБ и позволяет отлаживать контейнер. Distroless ещё меньше (или сопоставим) и не позволяет отлаживать через shell, но максимально снижает поверхность атаки — злоумышленник не сможет запустить shell или установить инструменты.

8. **Что такое SBOM и зачем он нужен?** SBOM (Software Bill of Materials) — это «список ингредиентов» программного обеспечения. Для Docker-образа SBOM содержит список всех пакетов, библиотек, их версий, лицензий, хешей. SBOM нужен для: быстрого ответа на вопрос «Используем ли мы уязвимую версию X?»; проверки лицензий (не нарушаем ли GPL?); аудита поставщиков; соответствия регуляторным требованиям (EO 14028 в США требует SBOM для критической инфраструктуры).

9. **Какие типы уязвимостей сканеры НЕ находят?** Сканеры находят только известные уязвимости с зарегистрированными CVE. Они не находят: zero-day уязвимости (ещё не зарегистрированные); логические ошибки в приложении (SQLi, XSS, бизнес-логика); misconfigurations в runtime (слишком широкие права, открытые порты); backdoors, внедрённые на этапе разработки (как XZ backdoor); уязвимости в собственном коде приложения (для этого нужен SAST). Сканирование образов — это не полная защита, а один из слоёв.

10. **Почему образы должны быть неизменяемыми?** Неизменяемость образов гарантирует воспроизводимость: один и тот же образ ведёт себя одинаково в development, staging и production. Если бы образы можно было менять — разработчик мог бы вручную поправить конфиг на staging, и production оказался бы другим. Неизменяемость также предотвращает «drift»: злоумышленник, проникнув в контейнер, не может изменить образ и распространить вредоносный код. Обновление происходит через создание нового образа и rolling deployment.
