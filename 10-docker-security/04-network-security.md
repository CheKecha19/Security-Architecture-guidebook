# Docker Network Security: Безопасность сетевых взаимодействий

## Что такое Docker-сеть?

Представьте многоквартирный дом. В каждой квартире живут люди, у каждой есть свой почтовый ящик. Некоторые квартиры хотят общаться между собой, другие — только через администратора, третьи — только с внешним миром. Docker-сеть — это система коммуникаций этого дома: какие квартиры могут общаться напрямую, какие только через центральный узел, какие вообще отключены от связи.

Docker-сеть — это виртуальная сетевая инфраструктура, которая соединяет контейнеры между собой, с хостом и с внешним миром. Каждый контейнер получает собственный сетевой стек: IP-адрес, сетевые интерфейсы, таблицу маршрутизации, правила iptables. Это изоляция через network namespace: контейнер живёт в своей собственной сети.

Docker предоставляет несколько встроенных сетевых драйверов: bridge, host, overlay, macvlan, none. Каждый драйвер имеет свои особенности безопасности. Неправильный выбор драйвера или конфигурации может открыть контейнер для атак из других контейнеров, с хоста или из внешней сети.

## Зачем это нужно?

**Сценарий 1: Изоляция микросервисов.** В HR-платформе есть микросервисы: auth-service (аутентификация), api-gateway (входная точка), user-service (данные пользователей), payment-service (платежи). Auth-service должен общаться с user-service, но не должен с payment-service (зачем сервису аутентификации доступ к платёжным данным?). Сетевые политики Docker/kubernetes позволяют описать такие правила: разрешить трафик только между определёнными сервисами.

**Сценарий 2: Защита базы данных от внешнего мира.** База данных PostgreSQL работает в контейнере. К ней должны подключаться только микросервисы приложения. Никто из внешней сети не должен обращаться к базе напрямую. Docker bridge network решает эту задачу: база данных не имеет публичного IP, только внутренний IP в bridge-сети, доступный только другим контейнерам на этом хосте.

**Сценарий 3: Атака из соседнего контейнера.** Злоумышленник проникает в контейнер с веб-сервером. Он начинает сканировать локальную сеть Docker в поисках других контейнеров. Если контейнеры находятся в одной bridge-сети — атакующий может найти базу данных, кэш Redis, админ-панель и попытаться атаковать их. Сетевые политики должны изолировать контейнеры так, чтобы компрометация одного не вела к компрометации всех.

## Основные концепции

### Bridge Network: Изолированный мост

**Bridge network** — это частная внутренняя сеть, создаваемая Docker на хосте. Контейнеры, подключённые к одному bridge, могут общаться между собой по внутренним IP-адресам. Контейнеры из разных bridge не видят друг друга.

Docker по умолчанию создаёт bridge `docker0`. Каждый новый контейнер подключается к `docker0`, если не указано иное. Контейнеры в `docker0` могут общаться между собой по IP. С хоста к контейнерам можно обратиться через опубликованные порты (`-p 80:80`).

С точки зрения безопасности:
- Контейнеры в одном bridge видят друг друга — это удобно, но опасно
- Опубликованные порты доступны с хоста и, если хост открыт в сеть — из внешней сети
- Все правила iptables для NAT и фильтрации управляются Docker автоматически

Рекомендуется создавать отдельный bridge для каждого приложения. Например, для HR-платформы: `hr-network` для всех микросервисов. Другое приложение (например, мониторинг) работает в отдельном bridge и не видит контейнеры HR-платформы.

### Host Network: Полная интеграция

**Host network** — контейнер использует сетевой стек хоста напрямую. У контейнера нет собственного IP — он использует IP хоста. Все порты, которые контейнер слушает, становятся доступными на хосте.

Host network — это как жить в квартире с выбитыми стенами: вы в одной комнате с хозяином, видите все его вещи, слышите все его разговоры. Для безопасности host network — плохой выбор: контейнер может конфликтовать с процессами хоста по портам, может прослушивать сетевой трафик хоста.

Используйте host network только если понимаете последствия. Например, для сетевых утилит (tcpdump, iperf), которые требуют доступа к сетевым интерфейсам хоста. Для production — избегайте.

### Overlay Network: Сеть между хостами

**Overlay network** — это виртуальная сеть, работающая поверх физической сети нескольких хостов Docker. Используется в Docker Swarm и Kubernetes. Создаёт туннель между хостами, позволяя контейнерам на разных серверах общаться так, как будто они на одном хосте.

Overlay network использует шифрование трафика по умолчанию (IPSec или VXLAN). Трафик между контейнерами на разных хостах зашифрован, даже если он проходит через общую сеть. Это как если бы две квартиры в разных домах соединялись бронированной трубой — никто не может подслушать разговор.

С точки зрения безопасности overlay network:
- Трафик между хостами шифруется
- Каждый хост получает токен для присоединения к overlay
- Токен должен быть защищён (хранится в Docker config)

### Macvlan: Реальный сетевой интерфейс

**Macvlan** — контейнер получает реальный MAC-адрес и IP-адрес из физической сети. Контейнер выглядит как отдельный сервер в локальной сети. Он может иметь свой IP из DHCP, свой DNS-имя, свой доступ к сетевым ресурсам.

Macvlan — это как дать каждой квартире не внутренний почтовый ящик, а отдельный адрес на городской почте. Контейнер напрямую доступен по сети, как обычный сервер. С одной стороны — это удобно для интеграции с существующей сетевой инфраструктурой. С другой — контейнер становится напрямую атакуемым из сети.

Macvlan не рекомендуется использовать без дополнительной защиты: контейнер должен быть защищён файрволом, обновлениями, WAF. Macvlan не имеет встроенной изоляции Docker — контейнер может ARP-spoofить соседей.

### None Network: Полная изоляция

**None network** — контейнер не имеет сетевых интерфейсов, кроме loopback. Контейнер полностью изолирован от сети. Он не может общаться с другими контейнерами, с хостом, с внешним миром.

None network — это карцер для контейнера. Используется для офлайн-обработки данных, для batch-задач, для контейнеров с конфиденциальными данными, которые не должны передаваться по сети. Или для первоначальной настройки: контейнер настраивается, потом подключается к сети.

## Сравнение сетевых драйверов Docker

| Критерий | Bridge | Host | Overlay | Macvlan | None |
|----------|--------|------|---------|---------|------|
| Изоляция от хоста | Да | Нет | Да | Частичная | Да |
| Межконтейнерная изоляция | Внутри bridge | Нет | В overlay | Да | Нет сети |
| Доступ извне | Через publish | Прямой | Через ingress | Прямой | Нет |
| Шифрование трафика | Нет | Нет | Да (IPSec) | Нет | Н/Д |
| Производительность | Средняя | Максимальная | Ниже средней | Высокая | Н/Д |
| Мульти-хост | Нет | Нет | Да | Нет | Да |
| Сложность | Низкая | Низкая | Средняя | Средняя | Низкая |

## Практические применения

**Сценарий 1: Микросервисы в изолированной сети.** HR-платформа использует custom bridge `hr-net`. В этой сети работают: api-gateway, auth-service, user-service, payment-service. Только api-gateway публикует порт 443 на хосте. Все остальные сервисы не имеют опубликованных портов и доступны только изнутри `hr-net`. Атакующий, проникший в auth-service, может атаковать payment-service — но не может атаковать хост или другие приложения на хосте.

**Сценарий 2: Сетевые политики Kubernetes.** В Kubernetes NetworkPolicy описывает правила межсервисного трафика. Создаётся политика: «auth-service может общаться только с user-service на порту 5432». Payment-service недоступен для auth-service. Если атакующий проникает в auth-service — он не может подключиться к базе данных payment-service, даже зная её адрес.

**Сценарий 3: Overlay network для банка.** HR-данные обрабатываются на кластере из трёх серверов. Контейнеры на разных серверах общаются через overlay network с шифрованием. Трафик между контейнерами шифруется автоматически, даже если данные проходят через общую сеть дата-центра. Compliance-отчёт подтверждает, что трафик между микросервисами зашифрован (соответствие PCI DSS, GDPR).

**Сценарий 4: База данных без сети.** Задача batch-обработки: контейнер загружает данные, обрабатывает их локально, сохраняет результат в volume. Контейнер не должен иметь сети — ни для входящих, ни для исходящих соединений. Используется `--network none`. Контейнер гарантированно не может отправить данные наружу — идеально для обработки PII (персональных данных).

**Сценарий 5: Legacy-интеграция через macvlan.** Старая ERP-система в дата-центре требует, чтобы приложение было отдельным устройством в сети с собственным IP. Новый микросервис на Docker использует macvlan, получает IP из пула DHCP дата-центра. ERP-система общается с микросервисом как с физическим сервером. Но микросервис дополнительно защищён файрволом хоста и WAF.

## Ограничения и нюансы

**Bridge network не изолирует контейнеры от внешней сети.** Если контейнер публикует порт (`-p 80:80`), этот порт доступен с хоста и, если хост не защищён файрволом — из любой сети. Публикуйте только те порты, которые действительно нужны.

**Host network на Windows не поддерживается.** Docker Desktop на Windows использует виртуальную машину Linux, и host network ведёт себя по-другому. Для production на Windows Server используйте Windows-контейнеры с транспарентной сетью.

**Overlay network требует хранение токена.** Ключ шифрования overlay хранится в Docker config. Если злоумышленник получает доступ к одному хосту в кластере — он получает все ключи и может расшифровать весь трафик. Ключи должны ротироваться.

**Macvlan может исчерпать MAC-адреса.** Каждый контейнер с macvlan получает уникальный MAC-адрес. В виртуализированных средах (AWS, Azure) количество доступных MAC-адресов может быть ограничено. Планируйте сетевые ресурсы заранее.

**Docker автоматически управляет iptables.** Docker добавляет правила iptables для NAT и фильтрации. Если у вас есть свои правила iptables — Docker может их перезаписать. Порядок правил может сломать вашу сетевую политику. Используйте `iptables-persistent` или `ufw` с осторожностью.

## Архитектурные решения

**Микросегментация.** Разделите сеть на мелкие сегменты. Каждое приложение — отдельный bridge. Каждый контейнер — отдельный участок сети. Сетевые политики должны описывать, кто с кем может общаться. Микросегментация ограничивает lateral movement: если атакующий проник в один контейнер — он не может свободно перемещаться по сети.

**Шифрование трафика между сервисами.** Даже внутри одного хоста шифруйте трафик между микросервисами (mTLS). Да, это дополнительные 10-15% нагрузки на CPU. Но это единственный способ защитить трафик от чтения на уровне ядра (если злоумышленник получил доступ к хосту). Service mesh (Istio, Linkerd) реализует mTLS прозрачно.

**Default deny для входящего трафика.** По умолчанию Docker bridge network разрешает весь входящий трафик от других контейнеров в той же сети. Настройте `--icc=false` (inter-container communication) в Docker daemon, чтобы запретить межконтейнерное взаимодействие. Разрешайте только явно указанные соединения.

**Публикация портов — осознанно.** Каждый опубликованный порт — это дверь в контейнер. Публикуйте только то, что нужно внешнему миру: 80/443 для веб-сервера, 22 для SSH (и лучше не надо). Внутренние сервисы (базы данных, кэши, очереди) не должны иметь опубликованных портов.

### Ответы на чек-лист

1. **В чём разница между bridge и host сетью?** Bridge создаёт изолированную виртуальную сеть. Контейнер имеет свой IP, свои интерфейсы, свою таблицу маршрутизации — полностью изолирован от хоста. Host использует сетевой стек хоста напрямую: контейнер имеет те же IP и порты, что и хост. Bridge безопаснее, host — производительнее (нет NAT). Для production используйте bridge или overlay.

2. **Как overlay network шифрует трафик между хостами?** Overlay network создаёт туннель VXLAN или IPSec между Docker-хостами. При инициализации overlay генерируется ключ шифрования. Этот ключ распространяется на все хосты в кластере. Каждый пакет, отправленный с одного хоста другому, шифруется на уровне ядра. Получатель расшифровывает пакет. Трафик внутри одного хоста не шифруется (контейнеры общаются напрямую).

3. **Почему macvlan может быть опасен?** Macvlan даёт контейнеру реальный IP в локальной сети. Контейнер становится напрямую доступным для всех устройств в сети. Docker не добавляет свои сетевые фильтры для macvlan — контейнер защищён только приложением и OS-файрволом внутри контейнера. Кроме того, macvlan может использоваться для ARP-spoofing: контейнер может отвечать на ARP-запросы за IP других устройств, перехватывая их трафик.

4. **Как микросегментация защищает от lateral movement?** Микросегментация делит сеть на мелкие изолированные сегменты. Каждый сегмент содержит только один контейнер или микросервис. Сетевые политики описывают точные правила: кто с кем может общаться. Если атакующий проникает в один контейнер — он не может автоматически подключиться к другим. Для перемещения по сети ему нужно найти новую уязвимость в каждом сегменте.

5. **Как межконтейнерное взаимодействие работает в bridge сети?** В bridge сети все контейнеры подключены к одному виртуальному коммутатору. Docker добавляет правила iptables, разрешающие трафик между контейнерами. Каждый контейнер может обратиться к другому по IP (или по имени контейнера, если используется `--link` или user-defined bridge). Отключить межконтейнерное взаимодействие можно через `--icc=false` в Docker daemon.

6. **Что такое network namespace в контексте Docker?** Network namespace — это механизм ядра Linux, создающий изолированный сетевой стек. Каждый контейнер имеет свой network namespace: свои сетевые интерфейсы, свои IP-адреса, свои таблицы маршрутизации, свои правила iptables, свой DNS. Контейнер не видит сетевые интерфейсы и трафик других контейнеров или хоста, если они не в одном namespace.

7. **Как NetworkPolicy работает в Kubernetes?** NetworkPolicy — это ресурс Kubernetes, описывающий правила сетевого трафика. Политика использует селекторы pod, namespace, IP-блоков. Правила бывают ingress (входящий трафик к pod) и egress (исходящий трафик от pod). Если NetworkPolicy не создана — весь трафик разрешён (default allow). Если создана хотя бы одна — всё, что не разрешено, запрещено (default deny). Для работы NetworkPolicy нужен сетевой плагин (Calico, Cilium, Weave).

8. **Как Docker управляет iptables?** Docker добавляет правила iptables в цепочки FORWARD и NAT. FORWARD нужен, чтобы разрешить трафик между контейнерами и от контейнеров к внешнему миру. NAT нужен для маскарадинга (SNAT) при выходе контейнера в интернет и для DNAT при публикации портов. Docker удаляет свои правила при остановке. Если вы добавляете свои правила iptables — убедитесь, что они не конфликтуют с правилами Docker.

9. **Какие риски host-сети?** Контейнер делит сетевой стек с хостом: может конфликтовать по портам (два контейнера не могут слушать один порт), может прослушивать сетевой трафик хоста, может модифицировать правила iptables хоста. Host network полностью снимает сетевую изоляцию. Используйте только для утилит, требующих доступа к сетевым интерфейсам (мониторинг, отладка). Для production — избегайте.

10. **Когда использовать none network?** Когда контейнер не должен иметь сетевого доступа вообще. Примеры: batch-обработка конфиденциальных данных (файлы загружаются через volume, обрабатываются, результат сохраняется в volume); изолированные тесты; контейнеры с секретами, которые не должны отправляться по сети (ключи шифрования, сертификаты). None network — максимальный уровень сетевой изоляции.
