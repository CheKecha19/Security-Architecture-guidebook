# Docker Host Security: Фундамент, без которого падает всё

## Что такое Docker Host Security?

Представьте себе современный бизнес-центр, в котором размещается сотня небольших офисов-гостиниц. Каждый номер оборудован по последнему слову техники: сейфы, биометрические замки, автономные системы пожаротушения. Но фундамент здания сделан из неармированного бетона, входная дверь не запирается, камеры слежения отсутствуют, а лифтовая шахта открыта для всех желающих. Никакая защита отдельных номеров не спасёт арендаторов, если злоумышленник свободно заходит в подвал и отключает электричество всему зданию. Docker Host Security — это ровно тот самый фундамент: безопасность операционной системы, на которой работает демон Docker и все его контейнеры.

В этой аналогии контейнеры — это отдельные номера гостиницы, каждый со своей дверью, изоляцией и гостевым доступом. Но если хост-система скомпрометирована, злоумышленник получает «ключ-мастер» ко всем номерам сразу. Он может остановить любые контейнеры, подменить образы, перехватить сетевой трафик между ними, украсть данные из томов или просто зашифровать всё и потребовать выкуп. Именно поэтому безопасность хоста — это не дополнительная опция, а самый нижний и самый важный слой в стеке контейнерной безопасности.

Технически Docker Host Security охватывает комплекс мер по защите операционной системы, на которой работает Docker Engine: от **харденинга ядра** (kernel hardening) и **защиты сокета Docker-демона** (Docker daemon socket protection) до систем обнаружения вторжений на уровне хоста и правил аудита. Речь идёт о том, чтобы предотвратить ситуацию, когда компрометация одного контейнера через уязвимость приложения превращается в полный захват хоста и, как следствие, всех остальных контейнеров — это называется **контейнерным прорывом** (container breakout).

Почему это критически важно именно сейчас? По данным многочисленных отчётов по безопасности облачных сред, более 90% уязвимостей, ведущих к компрометации контейнерных сред, эксплуатируют не уязвимости самих контейнеров, а неправильную конфигурацию хост-системы. Открытый TCP-сокет Docker API без аутентификации, запуск контейнеров с флагом --privileged, монтирование Docker-сокета внутрь контейнера, отсутствие обновлений безопасности на хосте — каждый из этих факторов по отдельности способен превратить самую защищённую контейнерную среду в проходной двор.

## Эволюция и мотивация

История безопасности Docker-хостов — это история о том, как удобство разработки постепенно столкнулось с суровой реальностью продакшена. Когда Docker только появился в 2013 году, его позиционировали как инструмент для разработчиков: «упакуйте приложение и его зависимости в контейнер и запускайте где угодно». Вопросы безопасности хоста тогда мало кого волновали — контейнеры запускались на локальных машинах разработчиков, и риски казались минимальными.

Переломный момент наступил в 2014-2015 годах, когда Docker начал проникать в продакшен-среды и облачные платформы. Оказалось, что демон Docker по умолчанию слушает Unix-сокет с правами root, любой пользователь в группе docker имеет эффективный root-доступ к системе, а механизмы изоляции Linux (namespaces, cgroups), на которых держится контейнеризация, не являются полноценными песочницами — они проектировались для изоляции процессов, а не для противодействия злонамеренным атакам.

В 2016 году Центр интернет-безопасности (CIS) выпустил первый **CIS Docker Benchmark** — эталонный набор рекомендаций по безопасной конфигурации Docker-хостов и контейнеров. Этот документ стал поворотным: индустрия наконец признала, что безопасность контейнерной среды начинается не с Dockerfile и не с сетевых политик Kubernetes, а с операционной системы, на которой всё это работает. Параллельно сообщество Linux активно развивало механизмы ядра: seccomp, AppArmor, SELinux, user namespaces — все они стали стандартными инструментами защиты Docker-хостов.

Мотивация к усилению безопасности хостов росла вместе с громкими инцидентами. В 2018 году исследователи обнаружили уязвимость CVE-2019-5736 в runc — компоненте, отвечающем за запуск контейнеров, — которая позволяла контейнеру перезаписать бинарный файл runc на хосте и получить выполнение кода с правами root. Этот инцидент наглядно продемонстрировал: граница между контейнером и хостом тоньше, чем кажется, и её необходимо укреплять с обеих сторон. Сегодня, в эпоху Kubernetes-кластеров и GitOps, безопасность Docker-хоста стала неотъемлемой частью DevSecOps-практик, а такие стандарты как NIST SP 800-190 и обновлённый CIS Benchmark v8 прямо предписывают харденинг хостов наряду с защитой самих контейнеров.

## Зачем это нужно?

Сценарий первый: открытый Docker API. Команда DevOps для удобства CI/CD-пайплайна оставила Docker API доступным по TCP-порту 2375 без TLS и аутентификации. Злоумышленник, просканировавший публичные диапазоны IP-адресов, обнаруживает этот порт и получает полный удалённый root-доступ к хосту через вызов docker run с монтированием корневой файловой системы. Результат: все контейнеры скомпрометированы, данные украдены, хост превращён в майнинг-ферму за считанные минуты.

Сценарий второй: побег из контейнера через привилегированный режим. Разработчик микросервиса для отладки сети в контейнере добавил флаг --privileged, «чтобы не разбираться с capabilities отдельно». Через несколько месяцев в библиотеке, используемой контейнером, обнаружена уязвимость. Эксплуатация уязвимости внутри контейнера, запущенного в привилегированном режиме с доступом ко всем устройствам хоста, позволяет мгновенно получить контроль над всей хост-системой через монтирование дисковых устройств или манипуляцию модулями ядра. Атака, которая должна была ограничиться одним сервисом, привела к компрометации всего кластера.

Сценарий третий: общий Docker-сокет. Следуя распространённому анти-паттерну, сокет Docker-демона (/var/run/docker.sock) проброшен внутрь контейнера, чтобы контейнер мог управлять другими контейнерами (например, для CI/CD или мониторинга). Злоумышленник, получивший доступ к этому контейнеру, немедленно получает и root-доступ к хосту — достаточно выполнить docker run -v /:/host снаружи контейнера. Это превращает уязвимость одного сервиса в компрометацию всей инфраструктуры, без каких-либо дополнительных шагов.

Сценарий четвёртый: необновлённое ядро хоста. Хост-система работает на устаревшей версии ядра Linux с известными уязвимостями локального повышения привилегий (LPE). Злоумышленник, получивший доступ к контейнеру с правами обычного пользователя (например, через SQL-инъекцию в веб-приложение), использует Dirty COW или аналогичную уязвимость ядра для повышения привилегий внутри контейнера, а затем эксплуатирует отсутствие user namespace remapping для получения root на хосте.

## Основные концепции

### Харденинг ядра и операционной системы хоста

Харденинг ОС хоста — это процесс минимизации поверхности атаки путём отключения неиспользуемых служб, удаления лишних пакетов, применения принципа минимальных привилегий и настройки обязательного контроля доступа. Docker-хост не должен выполнять никаких функций, кроме запуска и управления контейнерами — каждый дополнительный сервис (SSH на стандартном порту, веб-сервер, NFS) увеличивает поверхность атаки. **CIS Docker Benchmark** прямо требует использовать специализированные минималистичные дистрибутивы — такие как Container Linux (бывший CoreOS), Flatcar Linux или Bottlerocket от AWS, — которые спроектированы именно для роли Docker-хостов и не содержат компонентов, ненужных для этой задачи.

Харденинг ядра включает настройку **seccomp**-профилей (Secure Computing Mode), которые ограничивают системные вызовы, доступные контейнерам; активацию и настройку **LSM**-модулей (Linux Security Modules) — AppArmor на Ubuntu или SELinux на RHEL/CentOS; и применение системных параметров sysctl для защиты сети и памяти (например, отключение IP-форвардинга, защита от IP-спуфинга, ограничение доступа к kernel logs). Важно понимать, что контейнер разделяет ядро с хостом — если ядро уязвимо, никакая контейнерная изоляция не спасёт.

### Защита сокета Docker-демона

Сокет Docker-демона — это ключ-мастер ко всей Docker-инфраструктуре на хосте. Через него можно создавать, запускать, останавливать и удалять любые контейнеры, образы, тома и сети. **Защита Docker-сокета** (Docker daemon socket protection) включает несколько уровней: во-первых, никогда не открывать TCP-сокет без TLS-аутентификации (флаг -H tcp:// без --tlsverify — это мгновенный приговор); во-вторых, всегда использовать аутентификацию на основе взаимного TLS с клиентскими и серверными сертификатами; в-третьих, ни при каких обстоятельствах не монтировать /var/run/docker.sock внутрь контейнеров.

Альтернативой пробросу сокета является использование Docker API через TLS-прокси или переход на rootless-режим Docker, при котором демон работает от имени непривилегированного пользователя и не предоставляет root-доступа к хосту даже через сокет. Важно также настроить аудит доступа к сокету — каждый вызов API должен логироваться для последующего анализа инцидентов.

### Привилегированные контейнеры и механизмы предотвращения прорыва

Привилегированный контейнер (запущенный с --privileged) не имеет практически никакой изоляции от хоста: он видит все устройства в /dev, может загружать модули ядра, изменять сетевые настройки и параметры sysctl. Это грубый и опасный инструмент, который следует заменять на гранулярную настройку **Linux Capabilities** — механизма, позволяющего выдавать контейнеру только те привилегии, которые действительно нужны (например, NET_ADMIN для настройки сети, SYS_TIME для изменения времени).

Ключевой механизм предотвращения контейнерного прорыва — **User Namespace Remapping** (userns-remap). По умолчанию root внутри контейнера соответствует root на хосте, что означает: любой побег из контейнера даёт атакующему полный контроль. При включённом userns-remap пользователь root внутри контейнера проецируется на непривилегированного пользователя на хосте (обычно с UID ≥ 65536), который не имеет доступа к реальным файлам хоста, не может влиять на другие процессы и не представляет угрозы даже в случае эксплуатации уязвимости ядра из контейнера.

### Аудит, логирование и Host-based IDS/IPS

Безопасность Docker-хоста невозможна без видимости происходящего. **Host-based Intrusion Detection System** (HIDS) — например, OSSEC, Wazuh или Falco — мониторит системные вызовы, обращения к файлам, сетевую активность и аномалии поведения на уровне хоста. Falco, созданный компанией Sysdig специально для контейнерных сред, способен детектировать попытки запуска привилегированных контейнеров, монтирования чувствительных директорий хоста, выполнения команд внутри контейнеров и другие подозрительные действия в реальном времени.

**Аудит логов** (audit logging) с использованием Linux Audit Daemon (auditd) — это обязательное требование стандарта CIS Docker Benchmark и NIST SP 800-53 (контрольная группа AU — Audit and Accountability). Конфигурация auditd должна логировать все вызовы к демону Docker, изменения конфигурационных файлов Docker, доступ к sensitive-файлам хоста из контейнеров, использование команд docker exec и docker cp. Эти логи должны централизованно собираться в SIEM-систему для корреляции событий и своевременного обнаружения атак.

## Сравнение с другими механизмами

Защита Docker-хоста не существует в вакууме — она является частью многослойной стратегии безопасности контейнеризованных сред. В таблице ниже представлено сравнение безопасности хоста с другими уровнями защиты:

| Уровень защиты | Что защищает | Типичные инструменты | От чего НЕ защищает | Релевантные стандарты |
|---|---|---|---|---|
| **Docker Host Security** | ОС хоста, ядро, Docker-демон, сокет API | CIS Benchmark, auditd, AppArmor/SELinux, seccomp, userns-remap, Falco | Уязвимости внутри контейнера, атаки на приложения | CIS Docker Benchmark v1.x, NIST SP 800-53 (CM, SI, AU), ISO 27001 A.12.6.1 |
| **Container Image Security** | Содержимое образов: пакеты, библиотеки, конфигурации | Trivy, Clair, Grype, Docker Scout, Sigstore Cosign | Компрометация хоста, атаки на оркестратор | OWASP Docker Top 10, NIST SP 800-190, CIS v8 IG1 |
| **Container Runtime Security** | Поведение контейнера во время выполнения: системные вызовы, сеть, процессы | Falco, Tracee, Aqua, Prisma Cloud | Уязвимости нулевого дня на хосте, атаки на API Kubernetes | NIST SP 800-190, CIS Kubernetes Benchmark |
| **Orchestrator Security (Kubernetes)** | API-сервер, etcd, RBAC, сетевые политики, Pod Security Standards | kube-bench, OPA/Gatekeeper, Kyverno, Network Policies | Компрометация Docker-хоста (если на нём работает оркестратор), побег из контейнера на уровень хоста | CIS Kubernetes Benchmark, NIST SP 800-53 (AC, SC), ISO 27001 A.9.4 |
| **Network/Cloud Perimeter Security** | Граница сети, межсетевые экраны, Cloud Security Groups | WAF, NGFW, Cloud Firewall, VPC Security Groups | Атаки, исходящие изнутри скомпрометированного хоста, боковое перемещение | NIST SP 800-53 (SC-7), CIS v8 Control 4 |

Ключевой вывод из сравнения: безопасность Docker-хоста — это нижний слой, на котором надстраиваются все остальные уровни. Компрометация хоста делает бессмысленной защиту образов, рантайма и оркестратора, потому что злоумышленник получает доступ ко всем слоям снизу в обход их защитных механизмов. Именно поэтому принцип **Defense-in-Depth** требует, чтобы защита хоста была приоритетом номер один — без неё остальные инвестиции в безопасность контейнерной среды теряют смысл.

Компромиссный выбор состоит в следующем: жёсткая защита хоста (AppArmor/SELinux в enforcing-режиме, userns-remap, seccomp) увеличивает эксплуатационные накладные расходы — некоторые легитимные приложения могут не запускаться без дополнительной настройки политик, а отладка усложняется. Умеренный подход (AppArmor в complain-режиме, seccomp-профили по умолчанию, без userns-remap) проще в эксплуатации, но не обеспечивает защиты от целенаправленных атак. Выбор всегда остаётся между удобством эксплуатации и реальной безопасностью — и этот компромисс должен приниматься осознанно на основе модели угроз конкретной организации.

## Уроки из реальных инцидентов

Первый урок — инцидент с компанией Tesla в 2018 году, произошедший из-за открытого консольного доступа к Kubernetes-панели, но прекрасно иллюстрирующий проблему открытого доступа к контейнерному API. Неавторизованный доступ к Kubernetes dashboard, работавшему в подах, позволил злоумышленникам получить креденшлы AWS из переменных окружения и запустить криптомайнер в облачной среде Tesla. Если бы на уровне хостов были настроены HIDS и мониторинг сетевой активности, аномальная активность была бы обнаружена до того, как злоумышленники получили доступ к облачным ресурсам. Главный вывод: открытый неаутентифицированный доступ к контейнерному API (будь то Kubernetes или Docker) — это всегда вопрос «когда», а не «если» произойдёт компрометация.

Второй урок, более технический — уязвимость runc CVE-2019-5736, обнаруженная исследователями Адамом Иваньски и Борисом Любшински. Суть уязвимости: злонамеренный контейнер мог перезаписать бинарный файл runc на хосте через race condition при обращении к /proc/self/exe внутри контейнера. После этого любой новый контейнер, запущенный на хосте, выполнял код злоумышленника с правами root. Уязвимость затронула практически все версии Docker, Kubernetes и runc до патча. Этот инцидент стал катализатором для индустрии: он доказал, что граница между контейнером и хостом — это не магический барьер, а инженерная конструкция с неизбежными дефектами. Именно после этого инцидента userns-remap из «рекомендации» стал «требованием» во многих compliance-фреймворках. Вывод: даже изолированный контейнер с минимальными привилегиями может прорваться наружу через уязвимость в системном ПО хоста — поэтому хост должен быть защищён со всех сторон, включая патч-менеджмент, мониторинг целостности бинарных файлов и обязательное использование user namespace remapping.

| **User Namespace Remapping** | Проецирование root из контейнера в непривилегированного пользователя хоста | userns-remap (Docker native), Podman rootless | На уровне Docker Engine и ядра; до начала исполнения процессов контейнера | CIS Docker Benchmark 2.x, NIST SP 800-190 |
| **File Integrity Monitoring (FIM)** | Мониторинг изменений критичных бинарных файлов Docker и хоста | AIDE, Tripwire, Wazuh FIM | Агент на хосте, работает с inotify или периодическим сканированием | NIST SP 800-53 (SI-7), PCI DSS 11.5 |
| **Audit & Logging** | Регистрация административных действий, доступов к API, изменений конфигурации | auditd, Fluentd, Filebeat, Logstash | Агент на хосте, централизованный сбор в ELK/Splunk/SIEM | NIST SP 800-53 (AU-2, AU-3, AU-12), ISO 27001 A.12.4 |
| **Vulnerability Scanners (Host)** | Выявление уязвимостей в пакетах и ядре ОС хоста | Trivy (host mode), Tenable Nessus, Qualys, OpenVAS | Подключается к хосту удалённо или запускается локально | CIS v8 Control 7, NIST SP 800-53 (RA-5) |

Важное замечание: таблица инструментов не должна создавать ложное ощущение, что можно «купить безопасность». Ни один инструмент не заменяет продуманную архитектуру и осознанное проектирование. Инструменты автоматизируют проверки, обеспечивают видимость и ускоряют обнаружение — но решение о том, какие риски допустимы, а какие нет, всегда остаётся за архитектором безопасности.

## Архитектурные решения

### Defense-in-Depth на уровне Docker-хоста

Принцип **эшелонированной защиты** (Defense-in-Depth) применительно к Docker-хосту означает, что компрометация одного защитного слоя не должна приводить к компрометации всей системы. Рассмотрим эту архитектуру снизу вверх, от физического уровня к прикладному.

Первый слой — это физическая и сетевая изоляция самого хоста. Docker-хост должен находиться в изолированном сетевом сегменте, отделённом межсетевым экраном от публичного интернета и даже от внутренней сети. Доступ к хосту по SSH должен осуществляться только через бастион-хост или VPN с двухфакторной аутентификацией. Порты Docker API (2375/TCP и 2376/TLS) должны быть закрыты на межсетевом экране на вход — если API нужен удалённо, он должен быть доступен только через зашифрованный VPN-туннель.

Второй слой — харденинг операционной системы. Использование минималистичного дистрибутива (Flatcar Linux, Bottlerocket), отключение всех неиспользуемых сервисов, удаление компиляторов и отладочных инструментов (gcc, strace, tcpdump), применение автоматических обновлений безопасности с тестированием на staging-среде, настройка SELinux или AppArmor в enforcing-режиме для ограничения демона Docker и всех запускаемых им процессов.

Третий слой — безопасная конфигурация Docker Engine. Включение userns-remap, использование кастомного seccomp-профиля, настройка no-new-privileges, ограничение на переименование процессов и доступ к дополнительным группам, запрет на межконтейнерный сетевой обмен по умолчанию (icc: false), использование опции live-restore для предотвращения остановки контейнеров при перезапуске демона, аудит конфигурации демона через docker info.

Четвёртый слой — runtime-защита и мониторинг. Развёртывание Falco с правилами, настроенными на специфику организации, интеграция с SIEM через webhook или syslog, настройка auditd для записи всех административных действий, мониторинг целостности файлов Docker (демон, бинарники runc и containerd), настройка Prometheus и Grafana с алертингом на аномальное потребление ресурсов — это может быть признаком криптомайнинга.

Ключевой принцип: каждый слой проектируется с допущением, что уровень выше уже скомпрометирован. Если злоумышленник получил доступ к контейнеру (слой пять), он не должен иметь возможность прорваться на хост (слой три), а если и прорвался — не должен иметь возможности выйти в корпоративную сеть (слой один). Именно такое мышление создаёт по-настоящему устойчивую архитектуру.

### Zero Trust для контейнерных хостов

Архитектура **нулевого доверия** (Zero Trust) радикально меняет подход к защите Docker-хостов. В традиционной модели хост, находящийся внутри периметра корпоративной сети, считается «доверенным» — ему разрешено многое просто потому, что он находится «внутри». Zero Trust исходит из противоположного: ни один хост, ни один процесс, ни один контейнер не является доверенным по умолчанию, независимо от его сетевого положения.

Применительно к Docker-хосту Zero Trust означает: каждый контейнер получает ровно те привилегии, которые ему нужны для работы, и не более — это реализуется через гранулярные Linux Capabilities, кастомные seccomp-профили и AppArmor/SELinux-политики. Сетевые взаимодействия между контейнерами запрещены по умолчанию и разрешаются только явно сконфигурированными правилами. Docker API доступен только при наличии валидного TLS-сертификата клиента, и сертификаты регулярно ротируются. Каждое обращение к API логируется и анализируется на предмет аномалий.

Доступ администраторов к хосту осуществляется по принципу Just-In-Time (JIT): права предоставляются на ограниченное время, под конкретную задачу, с обязательным аудитом всех действий. Это может быть реализовано через системы Privileged Access Management (CyberArk, HashiCorp Vault с SSH-секретами) и интеграцию с тикет-системой. Ни у кого нет постоянного root-доступа к Docker-хосту.

Наконец, сам Docker-хост непрерывно верифицирует свою целостность: remote attestation с использованием TPM-чипа, измерение целостности загрузки, верификация бинарных файлов Docker по контрольным суммам. Если хост не может подтвердить свою целостность — оркестратор автоматически исключает его из кластера и переназначает контейнеры на другие узлы.

## Подготовка к собеседованию

### Три распространённые ошибки при обсуждении Docker Host Security на собеседовании

**Ошибка первая: фокус только на контейнерах.** Кандидат подробно рассказывает про сканирование образов, минимальные базовые образы, multi-stage builds, но не упоминает хост вообще. Это сигнал, что кандидат мыслит в рамках разработки и не понимает, что контейнерная безопасность начинается с операционной системы. Правильный ответ всегда начинается с хоста: харденинг ОС, защита сокета демона, userns-remap, а уже потом переходит к образам и рантайму.

**Ошибка вторая: предложение монтировать docker.sock внутрь контейнера.** Кандидат предлагает прокинуть /var/run/docker.sock в контейнер для решения задачи мониторинга или управления соседними контейнерами, не упоминая связанных с этим рисков. Это одна из самых опасных (и распространённых) практик в отрасли, которая по сути даёт контейнеру root-доступ к хосту. Правильный подход — использование Docker API через TLS, rootless Docker или специализированных инструментов, не требующих доступа к сокету.

**Ошибка третья: игнорирование user namespace remapping.** Кандидат рассказывает про capabilities, seccomp, AppArmor, но не упоминает userns-remap — самый мощный механизм предотвращения прорыва на хост. Это указывает на недостаток практического опыта: userns-remap действительно может конфликтовать с некоторыми сценариями (например, с volume mounts, где владелец файлов не соответствует пользователю контейнера), и обсуждение этих компромиссов — признак зрелого специалиста.

### Вопрос 1: В чём разница между защитой контейнера и защитой Docker-хоста, и почему вторая важнее?

Защита контейнера фокусируется на содержимом контейнера — образах, библиотеках, конфигурациях приложения, сетевых политиках. Защита хоста — это безопасность операционной системы, ядра и демона Docker, на которых запускаются все контейнеры. Хост важнее потому, что он является общим фундаментом: компрометация хоста означает компрометацию всех контейнеров одновременно, независимо от того, насколько хорошо они защищены по отдельности. Это иерархическая зависимость: безопасность верхних слоёв не имеет значения, если нижний слой скомпрометирован. Аналогия: сколько бы замков ни было на дверях отдельных квартир, если здание не имеет охраняемого входа и подвала, грабитель обойдёт все замки через фундамент.

### Вопрос 2: Почему монтирование docker.sock в контейнер — плохая практика, и какие есть альтернативы?

Сокет Docker-демона предоставляет полный программный контроль над Docker Engine, включая запуск привилегированных контейнеров с монтированием корневой ФС хоста. Монтирование этого сокета в контейнер означает, что любой процесс внутри контейнера с доступом к сокету может получить root на хосте — это превращает даже незначительную уязвимость в контейнере в полную компрометацию инфраструктуры. Альтернативы включают: использование Docker API по TLS-порту с аутентификацией и авторизацией, применение Docker-in-Docker (dind) с собственным демоном внутри контейнера, переход на rootless Docker, или использование специализированных инструментов вроде Kaniko для сборки образов без доступа к демону.

### Вопрос 3: Как работает user namespace remapping и какие компромиссы он создаёт?

User namespace remapping переназначает UID/GID внутри контейнера на другие UID/GID на хосте. Например, root (UID 0) внутри контейнера отображается на UID 100000 на хосте — непривилегированного пользователя без доступа к файлам хоста. Главное преимущество: даже если злоумышленник вырывается из контейнера, он не получает root на хосте. Компромиссы: проблемы с правами на файлы в volume mounts и bind mounts — файлы, созданные из контейнера, принадлежат remapped-пользователю на хосте и могут быть недоступны другим контейнерам; невозможность использования некоторых функций, требующих настоящих root-привилегий; увеличение сложности отладки, когда нужно сопоставлять внутренние и внешние UID. Эти компромиссы требуют продуманной стратегии управления правами на shared volumes.

### Ответы на чек-лист

**1. Почему безопасность Docker-хоста критичнее безопасности отдельного контейнера?**

Docker-хост — это общая среда исполнения для всех контейнеров. Если хост скомпрометирован, злоумышленник получает доступ ко всем контейнерам, их данным, сети и томам единовременно, независимо от защитных механизмов внутри каждого контейнера. Компрометация отдельного контейнера при правильно защищённом хосте — это локальный инцидент, ограниченный одним сервисом. Компрометация хоста — это тотальный инцидент, затрагивающий всю инфраструктуру. Здесь работает иерархический принцип: уровень ниже всегда важнее уровня выше, потому что поддерживает его. Компромисс: полная изоляция хоста требует ресурсов и усложняет оркестрацию, особенно в динамических облачных средах с автоскейлингом.

**2. Какие механизмы ядра Linux обеспечивают изоляцию контейнеров от хоста, и в чём их ограничения?**

Основные механизмы: namespaces (PID, net, mount, UTS, IPC, user, cgroup) для изоляции представления ресурсов; control groups (cgroups) для ограничения потребления ресурсов; capabilities для разделения привилегий root; seccomp для фильтрации системных вызовов. Ограничения: namespaces не являются песочницами безопасности и не проектировались для защиты от целенаправленных атак — они изолируют представления, но разделяют общее ядро. Уязвимость в ядре открывает путь к обходу всех этих механизмов. Cgroups ограничивают ресурсы, но не предотвращают их исчерпание через fork-бомбы без дополнительных настроек. Именно поэтому требуется эшелонированная защита: ограничения контейнера (seccomp, capabilities) плюс защита хоста (харденинг, обновления ядра) плюс мониторинг (HIDS).

**3. Чем опасен привилегированный контейнер и как заменить --privileged гранулярными настройками?**

Привилегированный контейнер отключает практически все механизмы изоляции: включает все capabilities, отключает seccomp-профиль, предоставляет доступ ко всем устройствам хоста в /dev, снимает ограничения AppArmor/SELinux. Это позволяет злоумышленнику, попавшему в такой контейнер, мгновенно получить root на хосте через монтирование дисков или загрузку модулей ядра. Замена --privileged гранулярными настройками: определить точный набор capabilities, необходимый приложению (например, NET_ADMIN для VPN, SYS_TIME для NTP), и добавить их через --cap-add; смонтировать только нужные устройства через --device; использовать стандартный или кастомный seccomp-профиль; настроить AppArmor-профиль для дополнительной защиты. Компромисс: гранулярная настройка требует понимания потребностей приложения и итеративного тестирования, но риски --privileged слишком велики, чтобы этим пренебрегать.

**4. Что такое user namespace remapping и почему его рекомендуют CIS Docker Benchmark и NIST SP 800-190?**

User namespace remapping (userns-remap) — это механизм ядра Linux, при котором пользователи и группы внутри контейнера отображаются на другие, специально выделенные UID/GID на хосте. Главный эффект: root (UID 0) в контейнере становится непривилегированным пользователем на хосте (UID 100000+), лишённым доступа к системным файлам, процессам и устройствам. CIS Docker Benchmark рекомендует userns-remap как фундаментальную защиту от контейнерного прорыва, а NIST SP 800-190 перечисляет его среди контрмер для рисков, связанных с разделяемым ядром. Компромисс: userns-remap создаёт сложности с правами на общих томах — файлы, записанные из одного контейнера, могут быть недоступны другому из-за несовпадения UID; некоторые унаследованные приложения, жёстко завязанные на конкретные UID, требуют дополнительной конфигурации.

**5. Какие сценарии атак реализуются через незащищённый Docker API и как им противодействовать?**

Сценарий первый: Docker API открыт по TCP без TLS — злоумышленник сканирует интернет, находит открытый порт 2375 и выполняет docker run -v /:/host для монтирования всей ФС хоста с последующим хищением данных, установкой бэкдоров или развёртыванием майнера. Сценарий второй: даже при TLS-шифровании, но без авторизации, любой, кто получил сертификат (или украл его при утечке), получает полный контроль. Противодействие: всегда использовать TLS с взаимной аутентификацией (--tlsverify), ограничивать сетевой доступ на уровне межсетевого экрана, перейти на Unix-сокет с контролем доступа через группу docker с ограниченным членством, использовать прокси с дополнительной аутентификацией перед Docker API, настроить аудит каждого вызова API для обнаружения аномалий.

**6. Как работает Falco и какие типы угроз для Docker-хоста он способен обнаружить?**

Falco — host-based система обнаружения вторжений, работающая путём перехвата системных вызовов ядра через kernel module или eBPF probes. Она применяет набор правил (rules) к потоку системных вызовов от всех процессов на хосте, включая процессы внутри контейнеров, и генерирует алерты при совпадении. Типы обнаруживаемых угроз: запуск привилегированного контейнера, монтирование чувствительных директорий хоста в контейнер, выполнение shell-команд внутри контейнера (exec), запись в системные директории (например, /etc), попытки чтения файлов паролей или SSH-ключей, аномальные сетевые подключения, изменение системных параметров (sysctl). Falco умеет обогащать алерты контекстом контейнера (ID, образ, Kubernetes pod) и направлять их в SIEM через webhook. Компромисс: eBPF-решения менее инвазивны, но предоставляют чуть меньший охват, чем kernel module.

**7. В чём разница между AppArmor и SELinux применительно к защите контейнеров и Docker-демона?**

AppArmor работает на основе путей к файлам: политики задаются для конкретных бинарных файлов и директорий, что делает их проще в написании и администрировании. SELinux работает на основе меток безопасности (labels) на файлах и процессах, обеспечивая более гранулярный и строгий контроль, но со значительно более высокой сложностью настройки. В контексте Docker: AppArmor широко применяется с Ubuntu-хостами, SELinux — с RHEL/CentOS, но оба способны ограничивать контейнеры и демон Docker. AppArmor проще интегрируется в пайплайны DevSecOps, SELinux даёт более сильные гарантии в средах с высокими требованиями (госсектор, финтех). Компромисс: выбор между удобством и строгостью — AppArmor для большинства коммерческих сред, SELinux для regulated environments.

**8. Как принцип Defense-in-Depth применяется к архитектуре Docker-хоста: перечислите слои?**

Снизу вверх: (1) физическая безопасность и доступ к оборудованию, ограничение физического доступа к серверу; (2) сетевая изоляция — выделенный VLAN, межсетевой экран, запрет входящего трафика на порты Docker и SSH из недоверенных сетей; (3) харденинг ОС — минималистичный дистрибутив, отключение лишних сервисов, патч-менеджмент; (4) защита Docker Engine — userns-remap, seccomp, AppArmor/SELinux, TLS на API, отказ от --privileged; (5) мониторинг и детектирование — Falco, auditd, FIM, сбор логов в SIEM; (6) процессы и люди — JIT-доступ, аудит админских действий, обучение команд. Каждый слой проектируется с допущением, что слои выше уже скомпрометированы — это ключевой принцип эшелонирования.

**9. Что означает Zero Trust в контексте контейнерных хостов и как его реализовать на практике?**

Zero Trust означает, что ни один хост или контейнер не получает доверия по умолчанию только на основании своего сетевого положения. На практике: все межконтейнерные сетевые соединения запрещены по умолчанию и требуют явного разрешения; доступ к Docker API аутентифицируется и авторизуется при каждом запросе; административный доступ предоставляется по принципу Just-in-Time с коротким временем жизни и обязательным аудитом; каждый хост непрерывно проверяется на целостность (attestation); микросервисы общаются с взаимным TLS и SPIFFE/SPIRE-based идентификацией. Реализация требует зрелой инфраструктуры: сервис-меш (Istio/Linkerd), системы управления секретами (Vault), автоматизированного аудита. Компромисс: высокая зрелость процессов и операционные расходы — Zero Trust сложнее классической периметровой модели.

**10. Какие уроки из инцидента CVE-2019-5736 (уязвимость runc) применимы к архитектуре безопасности Docker-хостов сегодня?**

Главный урок: граница между контейнером и хостом — это программный код, содержащий ошибки, а не фундаментальный закон природы. Уязвимость runc позволила контейнеру перезаписать бинарный файл на хосте через /proc/self/exe — то есть контейнер смог изменить критичный системный компонент, не имея привилегий. Применённые уроки: обязательное использование userns-remap, чтобы даже изменённый бинарник не дал root-доступа; мониторинг целостности runc, containerd и других компонентов Docker через FIM; немедленное применение патчей безопасности на всех хостах с автоматизацией процесса; использование неизменяемой инфраструктуры (immutable infrastructure), при которой хосты не патчат, а пересоздают из обновлённых образов. Компромисс: immutable infrastructure требует зрелых CI/CD-процессов и может конфликтовать с stateful-нагрузками на хостах.

## Ключевые выводы для собеседования

**Тезис первый:** Безопасность Docker-хоста — это не опция, а фундамент всей контейнерной безопасности. Ссылайтесь на модель угроз: ядро Linux разделяется между всеми контейнерами, поэтому уязвимость ядра или демона Docker делает бесполезными любые меры защиты внутри контейнеров. Без харденинга хоста контейнерная безопасность — это фасад, за которым ничего не стоит.

**Тезис второй:** Не существует одного «серебряного решения» — безопасность Docker-хоста строится на эшелонировании защиты (Defense-in-Depth) и архитектуре нулевого доверия. На собеседовании важно показать системное мышление: перечислить слои защиты от физического доступа до мониторинга аномалий, объяснить, как каждый следующий слой страхует предыдущий, и продемонстрировать понимание того, что архитектура всегда важнее конкретных инструментов.

**Тезис третий:** Компетентный архитектор безопасности всегда обсуждает компромиссы. Защита Docker-хоста неизбежно конфликтует с удобством разработки и эксплуатации: userns-remap усложняет работу с томами, SELinux в enforcing-режиме ломает легитимные сценарии, сегрегация сети замедляет коммуникацию. Умение найти баланс между риском и удобством для конкретного контекста (стартап versus банк versus госсектор) — это то, что отличает зрелого специалиста от энтузиаста, просто заучившего чек-лист.

**Тезис четвёртый:** Реальные инциденты — лучший инструмент аргументации. Ссылайтесь на CVE-2019-5736 (runc breakout) и инцидент с Tesla/Kubernetes, чтобы показать, что ваши рекомендации основаны не на абстрактных «best practices», а на конкретных уроках из истории отрасли. Это демонстрирует как техническую, так и практическую глубину понимания проблемы.
