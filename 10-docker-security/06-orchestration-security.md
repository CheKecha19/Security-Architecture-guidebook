# Docker Compose and Orchestration Security: Безопасность оркестрации

## Что такое оркестрация контейнеров?

Представьте, что у вас одна квартира. Управлять ею просто: один счётчик, один почтовый ящик, одна входная дверь. Но когда у вас сотни квартир в десятке домов в разных районах — нужен управляющий, который координирует всё: обслуживание, ремонты, безопасность. В мире контейнеров этот управляющий называется оркестратором.

Оркестрация контейнеров — это автоматическое управление жизненным циклом контейнеров: запуск, остановка, масштабирование, сетевое взаимодействие, обновление. Docker Compose — оркестратор для одного хоста (разработка/тестирование). Docker Swarm — оркестратор для кластера хостов. Kubernetes — промышленный оркестратор, стандарт для production.

Безопасность оркестрации — это защита не одного контейнера, а системы управления контейнерами. Если скомпрометирован оркестратор — злоумышленник контролирует все контейнеры. Как если бы злоумышленник получил ключи от кабинета управляющего домом — он может открыть любую квартиру, пустить газ, отключить воду, вызвать хаос.

## Зачем это нужно?

**Сценарий 1: Распространение вредоносной конфигурации через Compose.** Разработчик использует Docker Compose для локальной разработки. В `docker-compose.yml` он монтирует Docker socket в контейнер для удобства (`/var/run/docker.sock:/var/run/docker.sock`). Другой разработчик копирует эту конфигурацию на сервер CI, не понимая риска. CI получает root-доступ к хосту. Если CI скомпрометирован — злоумышленник может управлять Docker daemon, читать секреты, запускать произвольные контейнеры.

**Сценарий 2: Swarm с незащищённым join-токеном.** Docker Swarm использует join-токены для присоединения новых узлов к кластеру. Токен хранится в Docker config. Если токен скомпрометирован — злоумышленник может присоединить свой сервер к кластеру и получить доступ к сервисам и данным. Токен — это ключ от парадной двери: потерял ключ — злоумышленник может войти.

**Сценарий 3: Утечка секретов через переменные окружения.** В Compose-файле определены переменные окружения: `DATABASE_PASSWORD: "supersecret"`. Этот файл хранится в Git. Все разработчики имеют доступ к репозиторию. Пароль от production-базы данных известен всей команде, включая стажёров и подрядчиков. Docker Compose поддерживает механизм secrets, который предотвращает такую утечку.

## Основные концепции

### Docker Compose Security

Docker Compose — инструмент для определения и запуска многоконтейнерных приложений на одном хосте. Файл `docker-compose.yml` описывает сервисы, сети, volumes, порты.

Безопасность Compose связана с тем, что разработчики часто копируют Compose-файлы из разработки в staging и production, не адаптируя их для более строгой среды. Compose-файл для разработки может содержать удобства (монтирование Docker socket, широкие права, открытые порты), которые опасны в production.

**Deploy section** в Compose позволяет определить конфигурацию для production: ограничения ресурсов (resources limits), политики перезапуска, healthchecks, конфигурацию при развёртывании. В разработке deploy-секция игнорируется, в Swarm/Kubernetes — применяется.

Секреты в Compose: вместо переменных окружения используйте `secrets`:

```yaml
secrets:
  db_password:
    external: true  # секрет существует в Swarm/K8s

services:
  app:
    secrets:
      - db_password
```

Механизм secrets гарантирует, что секрет не попадёт в Compose-файл, не будет виден в Docker inspect, не будет сохранён в истории CI. Секрет монтируется как файл внутрь контейнера во время запуска.

### Docker Swarm Security

Docker Swarm — встроенный оркестратор Docker для кластера хостов. Swarm — это дом, где несколько управляющих (manager nodes) координируют работу.

**Manager nodes** управляют кластером: принимают решения, хранят состояние (Raft-консенсус), обрабатывают API-запросы. Если злоумышленник получает контроль над manager node — он может запускать произвольные контейнеры на всех узлах.

**Worker nodes** выполняют контейнеры. Worker node не принимает решений, не хранит состояние кластера. Компрометация worker node опасна, но локальна — злоумышленник контролирует контейнеры только на этом узле.

Безопасность Swarm:
- **mTLS между узлами**: каждый узел имеет сертификат, подписанный внутренним CA. Взаимодействие между узлами шифруется.
- **Raft-консенсус**: состояние кластера хранится в распределённом хранилище. Для изменения состояния нужно большинство manager nodes.
- **Доступ к API**: API Swarm защищён аутентификацией (сертификаты). Только manager nodes обрабатывают API-запросы.
- **Join-токены**: временные токены для присоединения к кластеру. Автоматически ротируются при компрометации.

**Locked Swarm**: Swarm может быть заблокирован. При перезапуске manager node требует ключ разблокировки (хранится отдельно). Это предотвращает автоматическое восстановление кластера злоумышленником, который получил физический доступ к серверу.

### Secrets Management

Docker Secrets — встроенный механизм для безопасного хранения и передачи секретов (паролей, токенов, ключей) контейнерам.

Как работают Docker Secrets:
1. DevOps создаёт секрет: `echo "mypassword" | docker secret create db_password -`
2. Секрет шифруется и хранится в Raft-логе manager nodes
3. Только сервисы, которым явно разрешён доступ, получают секрет
4. Секрет передаётся контейнеру через RAM-диск, а не через disk
5. Когда контейнер останавливается — секрет уничтожается

Docker Secrets решает проблему «секреты в репозиториях». Секреты не хранятся в Dockerfile, не видны в `docker inspect`, не попадают в логи. Но у механизма есть ограничения: секреты статичны (требуют пересоздания сервиса для обновления), не поддерживают автоматическую ротацию.

Для более сложных сценариев (ротация, аудит, версионирование) используйте HashiCorp Vault или CyberArk.

### Service Updates and Rollbacks

Обновление сервиса — один из самых рискованных моментов. Неудачное обновление может привести к простою production. Docker Swarm поддерживает стратегии обновления:

**Rolling update** — обновление контейнеров по одному, с проверкой healthcheck. Если новый контейнер не проходит healthcheck — обновление останавливается.

**Automatic rollback** — если обновление не удалось, Swarm автоматически откатывается к предыдущей версии. Время отката — секунды.

С точки зрения безопасности: обновления должны быть подписаны, образы — просканированы, политики — проверены. Автоматический откат — защита от неудачного деплоя, но не от вредоносного (если вредоносный образ прошёл через CI — откат к предыдущей вредоносной версии не поможет).

### Resource Limits and Quotas

Оркестратор должен ограничивать ресурсы каждого сервиса: CPU, память, диск. Это не только экономика, но и безопасность. Если один сервис может использовать все ресурсы хоста — DoS-атака на один сервис выводит все сервисы на этом хосте.

Docker Compose/Swarm поддерживает:
- `deploy.resources.limits` — максимальные ресурсы, которые может использовать контейнер
- `deploy.resources.reservations` — гарантированные ресурсы (минимум)

Ограничения памяти (memory limits) критичны: если контейнер имеет утечку памяти (memory leak), cgroup убивает только этот контейнер, а не все контейнеры на хосте. Ограничение CPU предотвращает ситуацию, когда один контейнер занимает 100% CPU и замедляет все остальные.

## Сравнение Compose, Swarm и Kubernetes

| Критерий | Docker Compose | Docker Swarm | Kubernetes |
|----------|----------------|--------------|------------|
| Область | Один хост | Кластер (multi-host) | Кластер (enterprise) |
| Сложность | Низкая | Средняя | Высокая |
| Встроенная безопасность | Минимальная | mTLS, Secrets, Raft | RBAC, Pod Security, NetworkPolicy |
| Secrets | Внешние | Встроенные (Docker Secrets) | Vault, External Secrets |
| Обновления | Ручные | Rolling + Rollback | Rolling, Blue/Green, Canary |
| Аудит | Нет | Docker events | Audit logging |
| Сетевые политики | Нет | Overlay + encryption | NetworkPolicy (Calico, Cilium) |
| Production-ready | Нет | Да (limited) | Да |
| Scalability | Один хост | До 1000 узлов | До 5000+ узлов |

## Практические применения

**Сценарий 1: Compose для разработки, Swarm для staging, K8s для production.** В Сбере трехуровневая архитектура: разработчики используют Docker Compose локально с монтированием кода; staging работает на Docker Swarm с полными ограничениями ресурсов; production — на Kubernetes (Managed K8s в облаке Сбера). Каждый уровень имеет свои политики безопасности. CI/CD проверяет Compose-файлы перед мержем в master.

**Сценарий 2: Locked Swarm для защиты от физического доступа.** Swarm заблокирован ключом. При перезапуске manager node DevOps должен ввести ключ разблокировки. Ключ хранится в HashiCorp Vault с MFA. Это защищает от атаки, когда злоумышленник получает физический доступ к серверу, перезагружает его и перехватывает управление кластером (который без блокировки автоматически восстанавливается).

**Сценарий 3: Docker Secrets для платёжных данных.** Сервис приёма платежей использует Docker Secrets для хранения ключа API платёжного шлюза. Секрет создаётся через CI только во время деплоя. Разработчики не имеют доступа к production-секретам. Даже если злоумышленник проникнет в контейнер — секрет находится в RAM, а не на диске, и уничтожается при остановке контейнера.

**Сценарий 4: Healthcheck и automatic rollback.** Сервис hr-api обновляется. Новый образ содержит ошибку: не реагирует на HTTP-запросы. Docker Swarm после rolling update проверяет healthcheck. Healthcheck не проходит (контейнер не отвечает на /health). Swarm автоматически откатывается к предыдущей версии за 5 секунд. Простой — 10 секунд (время обнаружения + откат). Без rollback простой мог бы составить часы.

**Сценарий 5: Resource limits против DoS.** Сервис поиска сотрудников имеет memory limit = 512 MB. Если злоумышленник отправляет миллион поисковых запросов — приложение может начать потреблять 2 GB памяти. Cgroup killer убивает контейнер. Другие сервисы (auth, payments) продолжают работать. Kubernetes перезапускает убитый контейнер через restart policy. DoS на один сервис не выводит всё приложение.

## Ограничения и нюансы

**Compose — не для production.** Docker Compose не поддерживает multi-host, не имеет high availability, не обеспечивает автоматическое восстановление. Использование Compose в production — архитектурная ошибка. Для production используйте Swarm или Kubernetes.

**Swarm — legacy.** Docker Swarm существует, но Docker официально фокусируется на Kubernetes. Новые функции появляются в Kubernetes, а не в Swarm. Для новых проектов выбирайте Kubernetes. Swarm — для существующей инфраструктуры, где миграция сложна.

**Docker Secrets не поддерживает ротацию.** Чтобы обновить секрет в Docker Swarm, нужно удалить и пересоздать сервис. Это требует простоя или blue/green деплоя. Для ротации секретов в реальном времени нужен Vault.

**Обновление сервиса — точка уязвимости.** Rolling update может выявить race condition: старый контейнер обрабатывает запрос, пока новый ещё не готов. Нужны: graceful shutdown (SIGTERM, ожидание завершения запросов), readiness probe (проверка, что контейнер готов к работе), lifetime timeout (максимальное время жизни контейнера для плавного завершения).

**Resource limits не защищают от всех DoS.** Memory cgroup убивает контейнер при превышении лимита, но не защищает от медленной атаки (slow loris, slow read). Для защиты нужны application-level средства: rate limiting, connection pooling, timeout'ы.

## Архитектурные решения

**Least Privilege для Compose-файлов.** Compose-файлы и Helm-чарты — это код. Они должны проходить code review как и любой другой код. Вредоносный Compose-файл может: смонтировать Docker socket, пробросить host-каталоги, открыть порты на хост. Политика: «ни один Compose-файл не содержит mount /var/run/docker.sock».

**Разделение окружений.** Разработка, staging, production — разные кластеры с разными политиками. Разработчик не может деплоить в production. CI может деплоить только в staging. Только DevOps с MFA — в production. Разделение предотвращает случайный или вредоносный деплой не проверенного кода.

**Immutable Infrastructure через оркестрацию.** Контейнеры не обновляются — они заменяются. Новый образ = новый контейнер. Оркестратор выполняет rolling update. Если что-то пошло не так — rollback к предыдущей версии. Никто не заходит в контейнер и не меняет конфиг вручную — это создаёт drift (отклонение от конфигурации) и открывает дверь для атак.

**Audit trail.** Все изменения оркестратора логируются: кто изменил сервис, кто создал секрет, кто присоединил новый узел. Логи отправляются в SIEM. Это позволяет расследовать инциденты и доказывать compliance.

### Ответы на чек-лист

1. **Чем отличается оркестрация от простого запуска контейнеров?** Простой запуск (docker run) запускает один контейнер на одном хосте. Оркестрация (Swarm, Kubernetes) управляет сотнями контейнеров на кластере хостов: автоматически восстанавливает упавшие контейнеры, масштабирует сервисы, распределяет нагрузку, обновляет версии без простоя. Оркестрация — это не запуск, а жизненный цикл.

2. **Почему Docker Compose не подходит для production?** Compose работает на одном хосте: если хост падает — падает всё приложение. Compose не обеспечивает high availability, автоматическое масштабирование, распределение между хостами. Compose-файл может содержать настройки, которые работают в разработке (монтирование кода, дебаг-порты), но опасны в production. Compose — инструмент для разработки и тестирования, не для production.

3. **Как mTLS защищает узлы Docker Swarm?** Каждый узел Swarm имеет сертификат, подписанный внутренним CA. При соединении узлы проверяют сертификаты друг друга (взаимная аутентификация). Все данные между узлами шифруются. Если злоумышленник пытается присоединить к кластеру свой сервер — он не может предоставить валидный сертификат. mTLS защищает от man-in-the-middle и несанкционированного присоединения.

4. **Что такое locked Swarm и зачем он нужен?** Locked Swarm — режим, при котором manager node шифрует Raft-лог ключом. При перезапуске manager node (после отключения питания, краша, перезагрузки) она не может восстановить состояние кластера без ключа разблокировки. Ключ хранится отдельно (Vault, бумажный сейф). Это защищает от атак, когда злоумышленник получает доступ к выключенному серверу и пытается прочитать Raft-лог.

5. **Как Docker Secrets защищают пароли и ключи?** Secrets шифруются при передаче и при хранении в Raft-логе. Они монтируются в контейнер как файл на ramfs (в памяти, а не на диске). Secrets не видны в Docker CLI (кроме создания), не попадают в логи, не сохраняются в Dockerfile. После остановки контейнера secrets уничтожаются. Это защищает секреты от перехвата, случайной утечки и восстановления из диска.

6. **Что такое rolling update и automatic rollback?** Rolling update — последовательное обновление контейнеров один за другим. Swarm заменяет старые контейнеры новыми с проверкой healthcheck. Если новый контейнер работает — Swarm продолжает. Если нет — останавливает обновление. Automatic rollback — при неудаче Swarm автоматически возвращается к предыдущей версии сервиса за секунды. Это минимизирует простой в случае ошибки деплоя.

7. **Почему resource limits — это не только экономия?** Resource limits — это защита от DoS. Если один сервис имеет утечку памяти — cgroup убивает только его контейнер, а не все контейнеры на хосте. Если один сервис использует 100% CPU — другие сервисы продолжают работать (limited by CPU shares). Если злоумышленник атакует один сервис — другие остаются доступны. Resource limits — часть defense in depth.

8. **Какие риски связаны с Compose-файлами в репозитории?** Compose-файлы могут содержать: секреты (пароли, токены), монтирование опасных путей (/var/run/docker.sock, /root/.ssh), открытые порты на хост, wide-open network policies. Разработчики копируют Compose-файлы между проектами, рапространяя опасные конфигурации. Решение: code review Compose-файлов, исключение секретов из репозитория, scanning Compose-файлов в CI.

9. **Чем Swarm отличается от Kubernetes?** Swarm проще (десятки параметров), меньше (один бинарник Docker), быстрее в настройке. Kubernetes сложнее (сотни параметров), мощнее (RBAC, Network Policy, Service Mesh, Custom Resources), имеет огромное сообщество. Swarm подходит для небольших кластеров (до 100 узлов) и простых приложений. Kubernetes — для enterprise (5000+ узлов) со сложными требованиями безопасности и compliance.

10. **Как audit logging оркестратора помогает безопасности?** Audit logging записывает все API-запросы к оркестратору: кто, когда, что изменил. Логи позволяют: обнаружить несанкционированный доступ (сторонний IP изменяет сервис), расследовать инцидент (кто подменил образ на вредоносный?), доказать compliance (мы контролируем изменения в production). Логи аудита должны быть защищены от изменения и храниться отдельно от кластера (SIEM).
