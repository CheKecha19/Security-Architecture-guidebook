# Docker Registry Security: Безопасность хранения и распространения образов

## Что такое Docker registry?

Docker registry — это хранилище Docker-образов, аналог библиотеки для книг. Разработчик пишет код, собирает образ и «сдаёт его в библиотеку» (registry). Другие разработчики или автоматические системы забирают образ из библиотеки для запуска контейнеров. Registry — центральное звено в цепочке поставки ПО.

Docker Hub — это публичная библиотека, доступная всем. Любой может скачать образ, любой может опубликовать свой. Но если ваша компания — Сбер, вы не будете хранить образы HR-платформы в публичной библиотеке. Вам нужна частная библиотека: корпоративный registry (Harbor, GitLab Container Registry, Amazon ECR, Azure Container Registry).

Registry — это не просто хранилище. Это точка контроля: кто может публиковать образы, кто может их скачивать, проверены ли образы на уязвимости, подписаны ли они. Если registry скомпрометирован — злоумышленник может подменить образ на вредоносный, и все, кто скачает этот образ, будут заражены.

## Зачем это нужно?

**Сценарий 1: Атака на цепочку поставки.** Злоумышленник получает доступ к незащищённому registry и загружает вредоносную версию образа HR-сервиса под тем же тегом (например, `hr-service:latest`). CI/CD система автоматически скачивает обновление и развёртывает вредоносный контейнер. Данные 250 000 сотрудников Сбера скомпрометированы. Защита registry (аутентификация, авторизация, сканирование, подпись) предотвращает такую атаку.

**Сценарий 2: Утечка проприетарного образа.** Стартап хранит образы своего флагманского продукта в публичном Docker Hub. Конкуренты скачивают образы, разбирают слои и воруют код, архитектуру, секреты. Использование частного registry с контролем доступа предотвращает утечку интеллектуальной собственности.

**Сценарий 3: Нарушение лицензий.** В образе используется библиотека с лицензией GPL, которая требует открытия исходного кода. Компания использует образ на production-серверах, не предоставляя исходный код. Это нарушение лицензии, ведущее к судебным искам. Registry со встроенным сканированием лицензий (Harbor) обнаруживает проблему до развёртывания.

## Основные концепции

### Аутентификация и авторизация

**Аутентификация** в registry — это проверка, кто пытается получить доступ. Docker registry поддерживает несколько способов: базовую аутентификацию (логин/пароль), токены (JWT), сертификаты (mTLS), интеграцию с внешними провайдерами (LDAP, OIDC, Active Directory).

**Авторизация** — это проверка, что пользователь (или система) имеет право на конкретное действие: публикацию образа, его скачивание, удаление. Правила авторизации описывают, кто какие образы может изменять:

- Разработчик может публиковать образы только в проект `hr-app/dev`
- CI/CD система может публиковать в `hr-app/staging` и `hr-app/prod`
- QA может только скачивать образы для тестирования
- Внешние подрядчики не имеют доступа

Harbor поддерживает RBAC (Role-Based Access Control) с ролями: Guest, Developer, Maintainer, Project Admin, System Admin. Каждая роль имеет набор разрешений, которые можно гибко настраивать.

### Сканирование уязвимостей

Registry должен сканировать образы не только при загрузке, но и регулярно для уже хранящихся образов. Новые уязвимости появляются каждый день. Образ, который вчера был безопасен, сегодня может быть скомпрометирован.

Harbor включает встроенный сканер (Clair или Trivy). При загрузке образа:
1. Образ помещается в очередь сканирования
2. Сканер разбирает слои и определяет ОС и установленные пакеты
3. Результаты сравниваются с базой CVE
4. Образ получает оценку безопасности: Pass / Warning / Danger
5. Политика registry может блокировать загрузку образов с критическими уязвимостями

Регулярное сканирование (раз в день или неделю) пересканирует все образы. Если для уже хранящегося образа обнаружена новая CVE — администратор получает уведомление.

### Подпись образов (Notary, Cosign)

Подпись образов гарантирует подлинность: образ опубликован доверенным автором и не был изменён. Docker Content Trust (DCT) использует Notary для хранения подписей и управления ключами.

Как работает подпись:
1. Разработчик собирает образ и подписывает его приватным ключом
2. Подпись загружается в Notary-сервер
3. При скачивании образа Docker проверяет подпись через Notary
4. Если подпись недействительна — образ не скачивается

Более современный подход — Sigstore Cosign. Cosign проще: не требует отдельного сервера (использует облачное хранилище подписей — Rekor), поддерживает краткосрочные ключи, интегрируется с GitHub/GitLab.

В enterprise-среде рекомендуется политика «только подписанные образы»: Kubernetes отказывается запускать контейнер из неподписанного образа. Это предотвращает развёртывание образов, собранных на ноутбуке разработчика или скачанных из недоверенных источников.

### Репликация и гео-распределение

Крупные компании имеют дата-центры в разных регионах. Репликация образов между registry в разных регионах позволяет:

- Ускорить скачивание образов (образ локально, не через WAN)
- Обеспечить отказоустойчивость (при отказе одного дата-центра образ доступен в другом)
- Соблюдать регуляторные требования (данные не покидают регион)

Harbor поддерживает push- и pull-репликацию: образ автоматически копируется из основного registry в реплики. Репликация может быть однонаправленной (основной → реплика) или двунаправленной (актив-актив).

С точки зрения безопасности трафик репликации должен шифроваться (HTTPS). Доступ к реплике должен быть ограничен: реплика используется только для чтения, запись — только в основной registry.

### Garbage Collection и Retention

В registry накапливаются старые образы и неиспользуемые слои. Они занимают место и создают риск: старый образ с непропатченной уязвимостью может быть случайно развёрнут.

Retention policy описывает правила удаления образов:
- Хранить последние 5 версий каждого тега
- Хранить образы не старше 30 дней
- Не удалять образы, используемые в production

Garbage Collection (GC) — процесс удаления неиспользуемых слоёв. В Docker Registry 2.7+ GC автоматический. В более старых версиях — ручной (`registry garbage-collect`).

Политика retention снижает риски: старые уязвимые образы автоматически удаляются, поверхность атаки уменьшается, экономится дисковое пространство.

## Сравнение registry-решений

| Критерий | Docker Hub | Harbor | GitLab Registry | Amazon ECR |
|----------|------------|--------|----------------|------------|
| Лицензия | Freemium | Open-source | Встроена в GitLab | Проприетарная |
| Сканирование | Да (платный) | Да (Trivy) | Да | Да (Inspector) |
| Подпись образов | DCT | DCT + Cosign | Через CI | Через KMS |
| RBAC | Нет | Да | Да (GitLab RBAC) | Да (IAM) |
| Репликация | Нет | Да | Нет | Да (кросс-регион) |
| LDAP/OIDC | Нет | Да | Да (GitLab) | Да (IAM) |
| Webhooks | Нет | Да | Да | Да |
| Образов | Безлимит (публ.) | Безлимит | По тарифу | По тарифу |
| Self-hosted | Нет | Да | Да (EE) | Нет |

## Практические применения

**Сценарий 1: Корпоративный registry для Сбера.** Сбер разворачивает Harbor в production. Настройки: строгая аутентификация через LDAP (пользователи Active Directory), RBAC: DevOps — Maintainer для всех проектов, разработчики — Developer только для своих проектов, QA — Guest (только чтение). Сканирование при загрузке с блокировкой критических уязвимостей. Подпись образов через Cosign: CI/CD подписывает каждый образ. Репликация между тремя дата-центрами.

**Сценарий 2: GitLab CI интеграция.** В GitLab CI определён этап сборки Docker-образа. После сборки запускается сканирование Trivy. Если CVE нет — образ публикуется в GitLab Registry. Если найдена критическая уязвимость — pipeline падает. После успешной публикации образ подписывается через Cosign. В production деплоятся только образы, прошедшие сканирование и имеющие подпись.

**Сценарий 3: Политика retention для экономии места.** В registry установлена политика: хранить последние 10 версий образа, старше 90 дней — удалять. Еженедельно запускается garbage collection. Это экономит 60% дискового пространства и гарантирует, что в registry нет образов с уязвимостями, обнаруженными год назад.

**Сценарий 4: Air-gapped registry.** Компания не имеет доступа к интернету из дата-центра. Все образы загружаются через промежуточный сервер (bastion host). Образы проверяются на вредоносное ПО, подписываются, затем загружаются в локальный Harbor. CI/CD не может использовать образы из Docker Hub — только из локального registry. Это предотвращает поставку через неконтролируемые каналы.

**Сценарий 5: Audit logging в registry.** Harbor включил аудит всех действий: кто, когда, какой образ загрузил, скачал, удалил. Логи отправляются в ELK. SIEM анализирует логи: не было ли несанкционированного доступа? Не скачивает ли сотрудник образы, к которым у него нет прав? Audit logging обязателен для compliance (SOC 2, PCI DSS).

## Ограничения и нюансы

**Сканирование только после загрузки.** Сканер проверяет образ после того, как он загружен. Если злоумышленник загрузит образ с backdoor'ом, который не является известным CVE — сканер его не обнаружит. Подпись образов и контроль доступа — дополнительные слои защиты.

**Notary требует инфраструктуры.** Для Docker Content Trust нужен отдельный Notary-сервер, управление ключами. Если Notary упадёт — пользователи не смогут скачивать подписанные образы (или скачают неподписанные, если отключат проверку). Cosign решает эту проблему через прозрачные логи.

**Репликация — не backup.** Репликация копирует образы, но не настройки registry, не пользователей, не политики. При полном отказе основного registry реплика будет доступна для чтения, но не для записи. Для полной отказоустойчивости нужна актив-актив конфигурация.

**Docker Hub rate limits.** Бесплатные аккаунты Docker Hub имеют ограничения: 100 pull-запросов в 6 часов для анонимных, 200 — для аутентифицированных. Корпоративный registry (Harbor, ECR) не имеет rate limits и кэширует образы в вашей сети.

**Безопасность самого registry.** Registry — это веб-приложение со своей поверхностью атаки. Harbour — это набор контейнеров (core, portal, notary, clair, registry). Каждый из них нужно обновлять, патчить, сканировать. Registry тоже должен быть защищён.

## Архитектурные решения

**Единый registry для всей компании.** Harbour или Artifactory как единое хранилище всех образов. Проекты для каждого отдела: `sber-hr`, `sber-psb` (Промсвязьбанк), `sber-as` (Автоматизированные Системы). RBAC на уровне проектов. Репликация между ЦОД.

**Shift Left в supply chain.** Сканирование образов начинается на этапе CI, а не в registry. Если образ уже содержит уязвимость — registry отклоняет загрузку, а не просто помечает «опасно». Политика: «ни один образ с критической уязвимостью не попадает в production».

**Supply chain attestation.** Каждый образ содержит attestation — набор утверждений о том, как он был собран: из какого коммита, каким CI-пайплайном, какие сканирования пройдены, кто подписал. Attestation в формате SLSA позволяют проверить цепочку поставки от коммита до production.

### Ответы на чек-лист

1. **В чём разница между Docker Hub и корпоративным registry?** Docker Hub — публичный сервис, доступный всем. Любой может скачать образ, любой может опубликовать свой (если пройдёт модерацию). Корпоративный registry (Harbor, ECR) — частное хранилище, контролируемое компанией. Он поддерживает RBAC, интеграцию с корпоративной аутентификацией, сканирование образов, подпись, репликацию. Для enterprise обязателен корпоративный registry.

2. **Как RBAC защищает registry?** RBAC (Role-Based Access Control) назначает каждому пользователю роль с набором разрешений. Гость может только скачивать образы. Разработчик — публиковать в dev-проекты. DevOps — публиковать в production и управлять конфигурацией. Администратор — управлять registry. Это предотвращает несанкционированную публикацию (разработчик не может залить образ в production) и несанкционированное чтение (подрядчик не может скачать образы, не относящиеся к его проекту).

3. **Как подпись образов защищает от подмены?** Подпись использует асимметричную криптографию. Разработчик подписывает образ приватным ключом. Пользователь проверяет подпись публичным ключом. Если злоумышленник подменяет образ (в registry, в сети, на диске) — подпись становится недействительной. Docker отказывается запускать образ с недействительной подписью. Это защищает от supply chain атак, когда образ изменяется после публикации.

4. **Что такое retention policy и зачем она нужна?** Retention policy — правила удаления старых образов и слоёв. Например: хранить последние 5 версий, удалять образы старше 90 дней, не удалять образы с тегом production. Это нужно для: экономии дискового пространства, снижения риска развёртывания старых (уязвимых) образов, соблюдения compliance (не хранить данные дольше необходимого).

5. **Как репликация образов повышает доступность?** Репликация автоматически копирует образы из основного registry в реплики в других регионах. При отказе основного registry контейнеры продолжают запускаться из реплики. При высокой нагрузке образы скачиваются из ближайшей реплики, снижая задержку. Репликация также защищает от потери данных при катастрофе дата-центра.

6. **Почему сканирование после загрузки недостаточно?** Сканирование при загрузке проверяет образ на известные CVE на момент загрузки. Но новые уязвимости появляются каждый день. CVE-2021-44228 (Log4Shell) была опубликована 9 декабря 2021, а уязвимость существовала с 2013. Образ, загруженный в 2020, не был проверен на эту уязвимость. Поэтому нужно регулярное (ежедневное/еженедельное) пересканирование всех образов в registry.

7. **Как Cosign отличается от Notary?** Notary — архитектура с отдельным сервером и репозиторием подписей. Требует управления ключами и сервером. Cosign от Sigstore — более современный подход: подписи хранятся прямо в registry (object storage), используется прозрачный лог (Rekor), ключи могут быть краткосрочными (короткоживущие сертификаты, действительные только при сборке). Cosign проще, не требует отдельной инфраструктуры, интегрируется с GitHub Actions и GitLab CI.

8. **Что такое supply chain attestation (SLSA)?** SLSA (Supply-chain Levels for Software Artifacts) — фреймворк для определения уровня безопасности цепочки поставки. Attestation — криптографически заверенные утверждения о процессе сборки образа: «Этот образ собран из коммита abc123 в репозитории hr-app, CI-пайплайном №456, проверен SAST и SCA, подписан DevOps-командой». Эти утверждения позволяют верифицировать, что образ прошёл все стадии контроля и может быть развёрнут.

9. **Какие риски незащищённого registry?** Незащищённый registry (без аутентификации, без HTTPS, без сканирования) может быть скомпрометирован. Злоумышленник может: загрузить вредоносный образ, который будет развёрнут в production; скачать все образы (утечка IP и секретов); удалить образы (отказ в обслуживании); изменить теги (подмена «latest» на старый уязвимый образ). Защита registry — критична для безопасности цепочки поставки.

10. **Как аудит событий в registry помогает безопасности?** Аудит записывает все действия: кто, когда, какой образ загрузил/скачал/удалил/изменил. Эти логи позволяют: обнаружить несанкционированный доступ (скачивание образов сотрудником, у которого нет прав); расследовать инцидент (кто загрузил вредоносный образ?); доказать compliance (мы контролируем, кто имеет доступ к образам). Логи аудита должны быть защищены от изменения (WORM-хранилище).
