# Docker Daemon Security: Безопасность Докер-демона

## Что такое Docker daemon?

Docker daemon (`dockerd`) — это центральный процесс, управляющий всеми контейнерами на хосте. Это как администратор здания: он знает все квартиры, открывает и закрывает двери, управляет доступом, обрабатывает запросы жильцов. Если администратор скомпрометирован — весь дом под угрозой.

Docker daemon работает как фоновый процесс с root-правами. Он слушает API-запросы (через UNIX-сокет или TCP), управляет образами, контейнерами, сетями, volumes. Любой, кто может отправлять запросы Docker daemon, может выполнить практически любую операцию от root: создать привилегированный контейнер, смонтировать диск хоста внутрь контейнера, прочитать любой файл на хосте.

Безопасность Docker daemon — это защита доступа к самому daemon. Если злоумышленник получит доступ к Docker socket, он получает контроль над всеми контейнерами и, фактически, над хостом. Поэтому daemon — одна из самых критичных точек в Docker-инфраструктуре.

## Зачем это нужно?

**Сценарий 1: Docker socket в CI/CD.** CI-сервер (Jenkins, GitLab Runner) работает внутри Docker-контейнера с монтированным Docker socket. Это удобно: CI может собирать и запускать Docker-образы изнутри контейнера. Если CI-контейнер скомпрометирован (через уязвимость в CI-скриптах) — злоумышленник получает полный контроль над Docker host. Он может запустить контейнер с майнером, смонтировать файловую систему хоста, украсть все секреты.

**Сценарий 2: Docker API открыт в сеть.** DevOps для удобства открыл Docker API по TCP без TLS. Любой в локальной сети может отправить запрос к Docker daemon: запустить контейнер, удалить образы, остановить production-сервисы. Это как оставить ключи от управления домом в подъезде — любой прохожий может войти.

**Сценарий 3: Docker daemon с уязвимостью.** В Docker Engine обнаружена уязвимость (например, CVE-2019-5736 — runc container breakout). Уязвимый daemon позволяет контейнеру перезаписать runc-бинарник хоста и получить root. Если daemon не обновлён — любой контейнер может скомпрометировать хост. Регулярное обновление Docker Engine — критично.

## Основные концепции

### Docker Socket: Ключ от всех дверей

Docker daemon слушает запросы через UNIX-сокет `/var/run/docker.sock`. Любой процесс, который может писать в этот сокет, может управлять Docker. Допуск к сокету эквивалентен root-доступу.

Почему Docker socket так опасен:
- Можно создать контейнер с монтированием `/` хоста: `docker run -v /:/host ...` и прочитать любые файлы
- Можно создать привилегированный контейнер: `docker run --privileged ...` с полным доступом к устройствам хоста
- Можно запускать произвольные образы (например, майнер криптовалют)
- Можно удалять образы и контейнеры (DoS)
- Можно читать логи и конфигурацию Docker daemon

Рекомендации:
- Никогда не монтируйте Docker socket в контейнер без крайней необходимости
- Если нужно управлять Docker изнутри контейнера — используйте remote API с TLS
- Ограничьте доступ к сокету через UNIX-права (только пользователи из группы `docker`)

### TLS для Docker API

Docker daemon может слушать TCP-порт (по умолчанию 2375 — без TLS, 2376 — с TLS). Настройка TCP без TLS — грубейшее нарушение безопасности.

TLS-сертификаты для Docker:
- **CA сертификат** — удостоверяющий центр, подписывающий остальные сертификаты
- **Server сертификат** — сертификат daemon, удостоверяющий его подлинность
- **Client сертификат** — сертификат клиента (docker CLI), удостоверяющий подлинность клиента

Настройка TLS включает:
1. Генерация CA и сертификатов (можно через OpenSSL или Docker certs)
2. Конфигурация daemon: `dockerd --tlsverify --tlscacert=ca.pem --tlscert=server-cert.pem --tlskey=server-key.pem -H=0.0.0.0:2376`
3. Конфигурация клиента: `docker --tlsverify --tlscacert=ca.pem --tlscert=cert.pem --tlskey=key.pem -H=tcp://server:2376`

mTLS защищает: аутентификацию (только authorised клиенты), шифрование (трафик зашифрован), целостность (данные не изменены).

**Крайне не рекомендуется**: использовать `-H tcp://0.0.0.0:2375` без TLS. Это открывает Docker API всему миру. Такая конфигурация — находка для пентестеров и злоумышленников, сканирующих интернет на открытые Docker API.

### Rootless Mode

Rootless mode — Docker, работающий без root-прав. Daemon запускается от обычного пользователя, использует user namespace и rootlesskit для изоляции.

Rootless mode изменяет архитектуру:
- Daemon работает от непривилегированного пользователя
- Контейнеры используют пользовательский namespace
- Сеть использует slirp4netns (виртуальный стек, не требующий прав)
- Файловая система использует fuse-overlayfs вместо overlay2

Преимущества rootless:
- Компрометация daemon не даёт root на хосте
- Компрометация контейнера — только внутри user namespace
- Меньше поверхность атаки (ограничение привилегий)

Недостатки rootless:
- Некоторые функции недоступны (нельзя слушать порты ниже 1024 без дополнительных настроек, cgroups v2 обязателен)
- Производительность сети ниже (slirp4netns медленнее bridge)
- Не все storage драйверы работают

Rootless — рекомендованный режим для Docker, когда это возможно. Docker Desktop на macOS и Windows использует rootless mode по умолчанию (через виртуальную машину).

### Конфигурация daemon

Daemon конфигурируется через файл `/etc/docker/daemon.json`. Этот файл определяет поведение Docker и его безопасность.

Критические параметры безопасности:

```json
{
  "icc": false,
  "live-restore": true,
  "userland-proxy": false,
  "userns-remap": "default",
  "log-driver": "json-file",
  "log-opts": { "max-size": "10m", "max-file": "3" },
  "selinux-enabled": true,
  "storage-driver": "overlay2"
}
```

- `icc: false` — отключает межконтейнерное взаимодействие по умолчанию
- `live-restore: true` — контейнеры продолжают работу, даже если daemon падает (для отказоустойчивости)
- `userland-proxy: false` — не использовать userland proxy для портов (снижает поверхность атаки)
- `userns-remap: "default"` — включает user namespace remapping
- `selinux-enabled: true` — включает SELinux (или AppArmor) для контейнеров

### Обновление Docker Engine

Docker Engine — сложное программное обеспечение, в котором регулярно находят уязвимости. CVE-2019-5736 (runc escape), CVE-2024-21626 (runc container breakout) — критические уязвимости, позволяющие выйти из контейнера.

Обновление Docker Engine должно происходить регулярно:
- Security патчи — в течение 24-48 часов
- Minor версии — в течение недели
- Major версии — после тестирования (2-4 недели)

Процесс обновления: staging → тестирование совместимости → production (rolling update с остановкой не более 10% кластера за раз).

## Сравнение режимов работы daemon

| Критерий | Rootful (по умолчанию) | Rootless | Rootful + TLS |
|----------|------------------------|----------|---------------|
| Привилегии daemon | root | Обычный пользователь | root |
| Привилегии контейнера | root (внутри ns) | User namespace | root (внутри ns) |
| Container escape risk | Высокий | Низкий | Высокий |
| Сеть | Bridge (nftables) | slirp4netns | Bridge |
| Порты < 1024 | Да | Нет (без настройки) | Да |
| Storage драйверы | Все | overlay2 (ограниченно) | Все |
| Производительность | Максимальная | Ниже (сеть) | Максимальная |
| Сложность | Низкая | Средняя | Средняя |
| Поддержка enterprise | Да | Ограниченно | Да |

## Практические применения

**Сценарий 1: TLS для remote Docker API.** DevOps настраивает удалённое управление Docker через TLS. Генерируются сертификаты: CA на защищённом сервере, server-сертификат для Docker daemon, client-сертификаты для CI-системы и администраторов. Daemon запускается с `--tlsverify -H=0.0.0.0:2376`. CI-система использует client-сертификат для подключения. Администраторы — свои сертификаты. Доступ только с авторизованных клиентов, трафик шифруется.

**Сценарий 2: Rootless Docker для multi-tenant окружения.** В облачном провайдере каждый клиент получает rootless Docker на своей виртуальной машине. Даже если клиент скомпрометирует свой Docker daemon — он не получит root на гипервизоре. Rootless mode — обязательное требование для публичных PaaS-сервисов.

**Сценарий 3: Audit logging для daemon.** Включён аудит Docker daemon через auditd (система аудита Linux). Все операции Docker: запуск контейнера, создание сети, изменение образа — записываются в лог. Логи отправляются в SIEM. Аналитик безопасности может увидеть: «в 3:15 ночи из контейнера CI запущен привилегированный контейнер с образом майнера». Это позволяет быстро обнаружить компрометацию.

**Сценарий 4: Ограничение доступа к Docker socket.** На production-хосте доступ к `/var/run/docker.sock` имеет только пользователь `root` (права 700). Группа `docker` не существует — ни один разработчик не может запускать контейнеры напрямую. Все манипуляции с контейнерами — только через Swarm/Kubernetes API, который имеет собственный аудит и RBAC. Это предотвращает случайные или вредоносные действия через direct Docker CLI.

**Сценарий 5: Docker daemon с live-restore.** На сервере с production-контейнерами настраивается `live-restore: true`. При обновлении Docker Engine daemon перезапускается. Контейнеры продолжают работу — они не зависят от daemon для выполнения. Это позволяет обновлять Docker без простоя приложения. После перезапуска daemon подхватывает управление контейнерами.

## Ограничения и нюансы

**Docker socket — не единственный вектор.** Docker daemon можно атаковать и другими способами: через API (если открыт без TLS), через вредоносные образы (если не проверять), через уязвимости в runc и containerd. Защита socket-а — необходимый, но не достаточный уровень.

**Rootless mode не для всех.** Rootless Docker требует cgroups v2 (Linux 5.2+, дистрибутив с systemd 243+). На старых дистрибутивах (CentOS 7, Ubuntu 18.04) rootless не работает. Для enterprise со старыми системами rootless недоступен — нужна миграция на более свежие ОС.

**TLS — не панацея.** Если злоумышленник украдёт client-сертификат (например, с ноутбука разработчика), он сможет подключаться от его имени. Нужна ротация сертификатов (30-90 дней), защита приватных ключей (не хранить в репозитории), отзыв компрометированных сертификатов.

**Userland proxy — legacy.** Docker по умолчанию использует userland proxy (`docker-proxy`) для проброса портов. Это процесс, работающий в userspace. Если userland proxy падает — порты становятся недоступны. Рекомендуется отключить userland proxy (`userland-proxy: false`) и полагаться на iptables-правила Docker.

**Обновление Docker может сломать кластер.** Docker Swarm или Kubernetes на старых версиях могут быть несовместимы с новой версией Docker. Перед обновлением проверьте matrix совместимости. Используйте rolling update daemon на кластере: обновляйте по одному узлу и проверяйте его работу, прежде чем переходить к следующему.

## Архитектурные решения

**Минимизация доступа к daemon.** Чем меньше процессов имеют доступ к Docker daemon, тем безопаснее. В production доступ к daemon должен быть только у Docker Swarm/Kubernetes. Никаких разработчиков с `docker run` на production-хостах. CI/CD управляет контейнерами через Swarm/Kubernetes API, а не через Docker socket.

**Defense in Depth.** Защита daemon — один слой. Другие слои: ограничение доступа к хосту (firewall, SSH с ключами), мониторинг действий daemon (auditd), обновление Docker Engine (patcher), безопасность образов (сканирование), runtime-защита (seccomp, AppArmor). Ни один слой не защищает полностью.

**Immutable daemon config.** Конфигурация daemon (`daemon.json`) должна быть неизменяемой и управляться через Infrastructure as Code (Ansible, Terraform). Любое изменение конфигурации — через Git. Это предотвращает несанкционированные изменения daemon (например, отключение TLS).

**Ротация сертификатов.** Сертификаты для Docker API должны ротироваться автоматически. Vault или cert-manager могут автоматически выпускать и обновлять сертификаты. Срок действия сертификата — 30-90 дней. Ротация без простоя — обязательна.

### Ответы на чек-лист

1. **Почему доступ к Docker socket эквивалентен root-правам?** Docker socket позволяет выполнять любые команды Docker API от root. Можно создать привилегированный контейнер с монтированием корневой ФС хоста (`-v /:/host`). Внутри такого контейнера root имеет полный доступ к хосту. Можно запускать произвольные образы (майнеры), удалять контейнеры/образы (DoS), читать/писать любые файлы. Любой процесс с доступом к socket — de facto root на хосте.

2. **Как TLS защищает Docker API?** TLS защищает Docker API трёхсторонне: аутентификация (клиент и сервер проверяют сертификаты друг друга), шифрование (трафик зашифрован, перехват невозможен), целостность (данные не могут быть изменены в пути). Без TLS любой в сети может отправить запрос к Docker API. С TLS только authorised клиенты с валидными сертификатами могут управлять Docker.

3. **Чем rootless mode отличается от rootful?** Rootful — стандартный режим: daemon работает от root, контейнеры от root (внутри namespaces). Rootless — daemon работает от обычного пользователя, контейнеры в user namespace. При компрометации rootful daemon злоумышленник получает root на хосте. При компрометации rootless — только права обычного пользователя. Rootless использует slirp4netns (медленнее) и не поддерживает порты < 1024 без настройки.

4. **Какие параметры daemon.json критичны для безопасности?** icc: false (запрет межконтейнерного взаимодействия по умолчанию), userns-remap (user namespace), selinux-enabled (SELinux/AppArmor), live-restore (контейнеры продолжают работу при падении daemon), userland-proxy: false (отключение лишнего процесса), log-driver и log-opts (ограничение размера логов), storage-driver: overlay2 (рекомендуемый драйвер).

5. **Что такое live-restore и зачем он нужен?** live-restore — опция daemon, которая позволяет контейнерам продолжать работу, даже если daemon временно недоступен (обновление, краш, перезапуск). Контейнеры — изолированные процессы, которые не зависят от daemon для выполнения. Live-restore позволяет обновлять Docker Engine без остановки production-контейнеров. Когда daemon восстанавливается, он «подхватывает» управление и продолжает мониторинг.

6. **Как auditd помогает в мониторинге daemon?** auditd — система аудита Linux. Она настраивается на мониторинг системных вызовов, связанных с Docker: вызовы к daemon, открытие socket, запуск контейнеров. Каждая операция записывается в лог с информацией: кто (UID), что (syscall), когда (timestamp), откуда (PID). Логи отправляются в SIEM для анализа и обнаружения аномалий. auditd — стандарт для аудита в enterprise.

7. **Почему userland proxy — legacy?** Docker использует отдельный процесс `docker-proxy` для проброса портов из контейнера на хост. Это legacy-механизм. В современных Docker версиях рекомендуется `userland-proxy: false`. Docker использует iptables/nftables для прямого проброса, без отдельного процесса. Это быстрее, надёжнее и снижает поверхность атаки (меньше процессов).

8. **Как часто нужно обновлять Docker Engine?** Security-патчи Docker Engine должны применяться в течение 24-48 часов после релиза. Minor-обновления — в течение недели. Major-обновления — в течение 2-4 недель после тестирования на staging. Обновление кластера — rolling: по одному узлу, с проверкой healthcheck. Если Docker Engine обновляется, а containerd/runc — нет, поверхность атаки сохраняется. Обновляйте всё сразу.

9. **Какие риски связаны с открытым Docker API без TLS?** Открытый Docker API (tcp://0.0.0.0:2375) — это дверь для злоумышленника. Сканеры интернета (Shodan, Censys) постоянно ищут открытые Docker API. Злоумышленник может: запустить майнер криптовалют (дорогой счёт за электричество), прочитать все файлы на хосте (утечка данных), удалить production-контейнеры (простой бизнеса), присоединить хост к ботнету (DDoS из вашей инфраструктуры). Никогда не открывайте Docker API без TLS.

10. **Как управлять сертификатами Docker API?** Сертификаты должны выпускаться через корпоративный CA (или Vault). Каждый администратор и CI-система получают свой client-сертификат. Срок действия — 90 дней, с автоматической ротацией. При увольнении сотрудника сертификат отзывается (CRL или OCSP). Приватные ключи не хранятся в репозиториях. Сертификаты не копируются между системами — каждый клиент получает свой.
