# Host Security: Защита хоста от выхода из контейнера

## Что такое host security для Docker?

Представьте, что вы живёте в квартире с очень тонкими стенами. Ваша безопасность зависит не только от замков на вашей двери (контейнер), но и от того, защищён ли подъезд, есть ли консьерж, установлены ли камеры. Хост — это весь дом. Если дом не защищён, даже самый крепкий замок на квартире не спасёт.

Host security — это совокупность мер по защите хоста, на котором работают контейнеры Docker. Контейнеры разделяют ядро с хостом. Если злоумышленник находит уязвимость в ядре — он может «вырваться» из контейнера и получить доступ к хосту. Это называется container escape.

Container escape — самый страшный сценарий для Docker. Разработчик запускает контейнер, думая, что он изолирован. Но изоляция контейнеров — это изоляция видимости и ресурсов, а не аппаратная изоляция. При правильном подходе контейнер не может выйти наружу. Но ошибки в конфигурации, устаревшее ядро, опасные capabilities — открывают дверь для escape.

Host security включает: защиту ядра, CIS Docker Benchmark, правильную конфигурацию хоста, мониторинг, ограничение доступа к хосту. Это не опционально для enterprise — это обязательно.

## Зачем это нужно?

**Сценарий 1: Атака Dirty Pipe (CVE-2022-0847).** Уязвимость в ядре Linux (5.8 — 5.16) позволяет процессу перезаписывать файлы, к которым у него нет прав на запись. Контейнер с непривилегированным пользователем эксплуатирует эту уязвимость и перезаписывает файл `/etc/passwd` на хосте, добавляя себе root-доступ. Если бы хост был обновлён — уязвимости бы не было. Обновление ядра — первая линия защиты.

**Сценарий 2: Контейнер с --privileged.** Разработчик запускает контейнер с `--privileged`. Этот флаг даёт контейнеру все capabilities и доступ ко всем устройствам хоста. Злоумышленник, проникший в контейнер, может монтировать диски хоста, загружать модули ядра, читать /dev/mem (память). Это не эксплуатация уязвимости — это конфигурационная ошибка. CIS Docker Benchmark проверяет конфигурацию на такие ошибки.

**Сценарий 3: Атака через runc (CVE-2024-21626).** Уязвимость в runc (runtime контейнера) позволяет файловому дескриптору «вытечь» за пределы контейнера. Контейнер может получить доступ к файловой системе хоста, используя пути `/proc/.../fd/...`. Уязвимость была исправлена в runc 1.1.12. Обновление runc — критично для всех, кто использует Docker.

## Основные концепции

### CIS Docker Benchmark

**CIS (Center for Internet Security) Docker Benchmark** — это набор рекомендаций по безопасной конфигурации Docker. Бенчмарк охватывает: конфигурацию хоста, конфигурацию Docker daemon, конфигурацию контейнера, конфигурацию образов, runtime-безопасность.

CIS Docker Benchmark содержит около 100 проверок, разделённых на секции:

**Section 1: Host Configuration**
- Установка последней версии Docker
- Настройка разделов (partitions) для Docker-данных (отдельный раздел для `/var/lib/docker`)
- Использование LVM или encryption для Docker-данных
- Аудит Docker daemon

**Section 2: Docker Daemon Configuration**
- Отключение межконтейнерного взаимодействия по умолчанию
- Настройка logging
- Использование user namespace remapping
- Отключение userland proxy
- Ограничение registry (только корпоративный registry)

**Section 3: Docker Daemon Configuration Files**
- Права на `/etc/docker/daemon.json` (600, owned by root)
- Права на Docker TLS-сертификаты (400, owned by root)
- Права на `/var/run/docker.sock` (700, owned by root)

**Section 4: Container Images and Build**
- Сканирование образов на уязвимости
- Использование trusted базовых образов
- Запуск контейнеров от non-root пользователя
- Read-only root filesystem

**Section 5: Container Runtime**
- Ограничение capabilities (`--cap-drop=ALL`)
- Использование seccomp и AppArmor
- Запрет привилегированных контейнеров
- Healthcheck для контейнеров

**Section 6: Docker Swarm**
- Использование locked swarm
- Настройка autolock
- Ротация сертификатов

CIS Docker Benchmark — стандарт для аудита безопасности Docker. Инструменты автоматической проверки: `docker-bench-security` (open-source), `kube-bench` (для Kubernetes). В enterprise политика: «0 critical failures в CIS Benchmark для production-хостов».

### Container Escape: Механизмы выхода

Container escape — это способ выйти за пределы изоляции контейнера и получить доступ к хосту. Основные механизмы:

**Escape через уязвимость ядра.** Уязвимость в ядре Linux (Dirty Pipe, Dirty Cow, OverlayFS) позволяет процессу в контейнере выполнить операцию, которую namespaces должны блокировать. Защита: обновление ядра, live patching (Kpatch, Ksplice), использование ядра с grsecurity.

**Escape через Docker socket.** Если Docker socket смонтирован в контейнер — escape не нужен, контейнер уже имеет полный доступ к хосту. Защита: не монтировать socket.

**Escape через mount.**
Контейнер с capability `SYS_ADMIN` может смонтировать устройства хоста (например, `/dev/sda`). После монтирования диска он может читать любые файлы на хосте. Защита: не давать `CAP_SYS_ADMIN`.

**Escape через /proc/sysrq-trigger.**
Контейнер может отправлять команды ядру через `/proc/sysrq-trigger` (если он доступен для записи). Команда `echo b > /proc/sysrq-trigger` перезагружает хост. Защита: read-only /proc, seccomp блокирует write в /proc.

**Escape через небезопасные mount paths.**
Если хост монтирует каталог `/dev` или `/proc` внутрь контейнера — контейнер получает доступ к устройствам и процессам хоста. Защита: не монтировать системные каталоги без крайней необходимости.

### Ядро и live patching

Ядро Linux — самый критичный компонент для безопасности контейнеров. Уязвимость в ядре — это потенциальный container escape. Регулярное обновление ядра — базовая защита.

Но обновление ядра требует перезагрузки. В production с сотнями контейнеров перезагрузка — сложная операция. Live patching (Kpatch, Ksplice, Livepatch) позволяет применять патчи безопасности к ядру без перезагрузки.

Live patching работает как «заплатка на лету»: новая функция заменяет уязвимую в памяти, не останавливая ядро. Это позволяет закрыть уязвимость (Dirty Pipe, Dirty Cow) за минуты, а не ждать планового окна перезагрузки.

В enterprise-среде live patching — обязательный инструмент. Canonical Livepatch (Ubuntu), Ksplice (Oracle), Kpatch (Red Hat) — решения для разных дистрибутивов.

### Docker Data Security

Данные Docker хранятся в `/var/lib/docker`: образы, контейнеры, volumes, overlay2 данные. Если злоумышленник получает доступ к этому каталогу — он получает:
- Данные volumes (базы данных, файлы приложений)
- Слои образов (проприетарный код)
- Конфигурацию контейнеров (секреты?)
- Overlay2 данные (файлы контейнеров)

Рекомендации:
- Отдельный раздел (LVM, XFS) для `/var/lib/docker` — предотвращает заполнение корневого раздела
- Шифрование раздела (LUKS) — защищает данные при физическом доступе к диску
- Регулярное резервное копирование volumes
- Мониторинг размера `/var/lib/docker`

## Сравнение механизмов защиты от escape

| Механизм | Тип | Что защищает | Эффективность | Сложность |
|----------|-----|--------------|---------------|-----------|
| Capabilities | Конфигурация | Привилегированные операции | Высокая | Низкая |
| Seccomp | Ядро | Системные вызовы | Высокая | Средняя |
| AppArmor/SELinux | LSM | Файлы, сеть, устройствa | Высокая | Высокая |
| User namespace | Ядро | UID отображение | Высокая | Низкая |
| Read-only rootfs | Docker | Запись в ФС контейнера | Средняя | Низкая |
| No privileged | Docker | Все привилегии | Высокая | Нулевая |
| CIS Benchmark | Аудит | Конфигурация | Средняя | Средняя |
| Live patching | Ядро | Уязвимости ядра | Высокая | Средняя |

## Практические применения

**Сценарий 1: Еженедельный CIS Benchmark audit.** В Сбере каждую неделю запускается `docker-bench-security` на всех production-хостах. Результаты отправляются в SIEM. Если найдены критические failures — создаётся тикет с приоритетом P0 (необходимо исправить в 24 часа). Пример: если хост запускает контейнер с `--privileged` — это P0, требующий немедленного вмешательства.

**Сценарий 2: Защита от Dirty Pipe через live patching.** При публикации CVE-2022-0847 (Dirty Pipe) команда безопасности Сбера в течение часа применяет live patch к ядру на всех production-хостах. Ни один сервер не перезагружается. Через неделю, во время планового окна, применяется полноценное обновление ядра. Live patch предотвращает эксплуатацию уязвимости, не нарушая SLA.

**Сценарий 3: Аудит Docker socket permissions.** На production-хосте проверяются права на `/var/run/docker.sock`. Если права не 600 (только root) — создаётся alert. Если Docker socket смонтирован в любой контейнер — автоматически запускается script, который останавливает контейнер. Политика: «ни один production-контейнер не имеет монтирования Docker socket».

**Сценарий 4: Изоляция /var/lib/docker.** Для `/var/lib/docker` создан отдельный LVM-раздел. Раздел зашифрован (LUKS), ключ хранится в TPM (Trusted Platform Module) сервера. При физической краже диска данные недоступны злоумышленнику. Размер раздела мониторится — при заполнении >80% создаётся alert.

**Сценарий 5: Политика «нет privileged контейнеров».**
В Docker Swarm/Kubernetes настроен admission controller, который запрещает создание контейнеров с `--privileged`. Любая попытка создать privileged контейнер отклоняется с сообщением: «Privileged containers запрещены политикой безопасности». Это предотвращает случайный или умышленный запуск небезопасных контейнеров.

## Ограничения и нюансы

**CIS Benchmark — руководство, не закон.** Следование всем рекомендациям CIS может сломать некоторые приложения. Например, отключение межконтейнерного взаимодействия (icc: false) может потребовать явного разрешения трафика между микросервисами. Каждую рекомендацию нужно тестировать на staging перед применением в production.

**Live patching — не панацея.** Live patch закрывает только конкретную уязвимость, для которой он выпущен. Другие уязвимости в той же функции могут остаться неприкрытыми. Live patching — временная мера до полноценного обновления ядра.

**User namespace — не для всех приложений.** Некоторые приложения (системные утилиты, Samba, NFS) не работают с user namespace. Перед включением userns-remap проверьте совместимость всех приложений.

**Container escape — постоянно эволюционирующая угроза.** Новые уязвимости в ядре, в runc, в containerd появляются регулярно. Снижение риска — через defense in depth: обновление + конфигурация + мониторинг. Ни одна мера не защищает от всех escape.

**Docker vs VM isolation.** Контейнеры никогда не будут так же изолированы, как виртуальные машины. Если вам нужна максимальная изоляция (multi-tenant, недоверенные образы) — используйте виртуальные машины внутри контейнеров (Kata Containers, gVisor) или VM-based решения (Firecracker).

## Архитектурные решения

**Defense in Depth против escape.** Ни один механизм не защитит от всех escape. Комбинация: обновлённое ядро + user namespace + drop all capabilities + seccomp + AppArmor/SELinux + read-only rootfs + нет privileged контейнеров + нет mount socket. Чем больше слоёв, тем сложнее злоумышленнику.

**Immutable host.** Хосты с Docker должны быть immutable: конфигурация управляется через Git, обновления — через образы, никаких ручных изменений. При компрометации хоста он просто пересоздаётся из образа. Immutable host снижает window of exposure: скомпрометированный хост быстро заменяется чистым.

**Sandboxing для недоверенных образов.** Если в кластере запускаются образы из недоверенных источников — используйте sandboxing: Kata Containers (виртуальная машина для каждого контейнера) или gVisor (userspace ядро). Это даёт аппаратную изоляцию, даже если контейнер скомпрометирован.

**Мониторинг escape попыток.** Falco, Tracee, Sysdig мониторят системные вызовы и обнаруживают escape-попытки. Например, попытка монтирования диска из контейнера — немедленный alert. Мониторинг — детективный контроль, который срабатывает, когда превентивные меры не сработали.

### Ответы на чек-лист

1. **Что такое container escape и как он работает?** Container escape — это выход за пределы изоляции контейнера. Злоумышленник, находясь внутри контейнера, получает доступ к хосту. Механизмы: уязвимость ядра (Dirty Pipe), misconfiguration (privileged контейнер, монтирование socket), эксплуатация уязвимостей runtime (runc CVE-2024-21626). Выход из контейнера эквивалентен получению root на хосте.

2. **Как CIS Docker Benchmark помогает защитить хост?** CIS Docker Benchmark — это чек-лист из 100+ рекомендаций по безопасной конфигурации Docker. Он покрывает конфигурацию хоста, daemon, контейнеров, образов. Инструменты автоматической проверки (docker-bench-security) сканируют хост и сообщают о нарушениях. Следование бенчмарку снижает риск misconfiguration, которая является основной причиной успешных escape.

3. **Почему привилегированный контейнер — это опасно?** `--privileged` даёт контейнеру ВСЕ capabilities, включая CAP_SYS_ADMIN (монтирование дисков, управление kernel), CAP_NET_ADMIN (управление сетью), доступ ко всем устройствам (/dev/*). Привилегированный контейнер может монтировать диски хоста, читать /dev/mem (память), загружать модули ядра, выполнять любые операции. Это не container escape — это открытая дверь наружу.

4. **Как live patching защищает от уязвимостей ядра?** Live patching (Kpatch, Ksplice) позволяет применить патч к работающему ядру без перезагрузки. При публикации CVE, затрагивающей ядро, live patch применяется в течение часов. Функция ядра с уязвимостью заменяется в памяти на исправленную. Это закрывает уязвимость (Dirty Pipe, Dirty Cow) без простоя production. Live patching — временная мера до полноценного обновления ядра при плановой перезагрузке.

5. **Какие механизмы prevent container escape?** Основные: user namespace (UID remapping), drop всех dangerous capabilities (CAP_SYS_ADMIN, SYS_PTRACE, SYS_MODULE), seccomp (блокировка опасных syscall'ов), AppArmor/SELinux (ограничение файлов и сети), read-only rootfs (защита от модификации), запрет privileged контейнеров, обновлённое ядро (патченые CVE). Комбинация всех механизмов — defense in depth против escape.

6. **Почему Docker socket — самый опасный mount?** Docker socket даёт полный доступ к Docker daemon. С доступом к socket можно: создать контейнер с монтированием всей ФС хоста, запустить привилегированный контейнер, читать все образы, удалять контейнеры. Docker socket — это root-доступ, обёрнутый в удобный API. Если socket смонтирован в контейнер — изоляция контейнера не имеет значения, потому что злоумышленник может управлять Docker изнутри.

7. **Как user namespace защищает хост?** User namespace отображает UID внутри контейнера на непривилегированного пользователя на хосте. Root (UID 0) внутри контейнера — это UID 50000 на хосте. Если контейнер вырывается наружу (через уязвимость ядра) — он получает права обычного пользователя (UID 50000), а не root. Он не может читать /etc/shadow, не может менять системные файлы, не может устанавливать ПО. User namespace снижает ущерб от escape.

8. **Что такое immutable host в контексте Docker?** Immutable host — сервер, чья конфигурация никогда не изменяется вручную. Все изменения — через Infrastructure as Code (Ansible, Terraform, образы ОС). При обновлении хост не патчится — он пересоздаётся из нового образа. Для Docker это означает: конфигурация daemon в Git, daemon.json не меняется вручную, обновление Docker — пересоздание хоста. Immutable host предотвращает drift и упрощает восстановление после компрометации.

9. **Как Kata Containers отличаются от обычных контейнеров?** Kata Containers — это контейнеры, которые запускаются внутри лёгкой виртуальной машины. Каждый контейнер получает собственное ядро (через VM), а не разделяет ядро хоста. Это даёт аппаратную изоляцию: даже при полной компрометации ядра контейнера хост остаётся в безопасности. Минусы: больше накладные расходы (дополнительное ядро), медленнее запуск, не все Docker-функции поддерживаются.

10. **Как мониторинг помогает обнаружить escape?** Инструменты (Falco, Tracee) мониторят системные вызовы контейнеров и детектят аномалии. Примеры правил Falco: контейнер вызывает mount (попытка монтирования диска), контейнер пишет в /proc (попытка escape через /proc), контейнер вызывает ptrace (попытка инъекции в процесс хоста), контейнер открывает /dev/mem (прямое чтение памяти). Мониторинг — детективный контроль, срабатывающий, когда превентивные меры не сработали.
