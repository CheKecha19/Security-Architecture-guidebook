# Docker Monitoring and Logging: Наблюдаемость и аудит контейнеров

## Что такое мониторинг контейнеров?

Представьте, что в большом доме с сотней квартир вы — управляющий. Вы не можете заходить в каждую квартиру каждую минуту, чтобы проверить, всё ли в порядке. Вам нужны датчики: температуры, дыма, протечки воды, открытия дверей. Если датчик срабатывает — вы получаете сигнал и идёте проверять. Мониторинг контейнеров — это такие же датчики для вашего приложения.

Мониторинг Docker — это сбор, анализ и визуализация метрик и логов контейнеров: загрузка CPU, потребление памяти, дисковые операции, сетевой трафик, статус процессов. Без мониторинга вы слепы: вы не знаете, не падает ли контейнер, не перегревается ли CPU, не заполняется ли диск.

Логирование — это запись событий, происходящих в контейнере: HTTP-запросы, ошибки, предупреждения, информационные сообщения. Логи — это «чёрный ящик» приложения: если что-то пошло не так, логи покажут, что именно и в какой последовательности.

С точки зрения безопасности мониторинг и логирование — это детективные контроли. Они не предотвращают атаки, но позволяют обнаружить их на ранней стадии, расследовать инцидент и восстановиться после него.

## Зачем это нужно?

**Сценарий 1: DoS-атака на HR-платформу.** Злоумышленник запускает DDoS-атаку на HR-сервис. Без мониторинга инцидент обнаруживается, когда пользователи жалуются («сайт не работает»). С мониторингом — alert срабатывает через 30 секунд после начала атаки: резкий рост CPU и сетевого трафика. DevOps автоматически запускает масштабирование и включает WAF. Простой — минуты, а не часы.

**Сценарий 2: Обнаружение аномалий через Falco.** Falco (runtime security engine) обнаруживает, что контейнер с веб-сервером запустил shell. Это аномалия: веб-сервер не должен запускать shell. Falco отправляет alert в SIEM. Аналитик проверяет — да, это RCE-атака (Remote Code Execution). Контейнер изолируется, атака блокируется до того, как злоумышленник получил доступ к данным.

**Сценарий 3: Утечка логов с PII.** Логи приложения содержат PII (персональные данные): имена, номера паспортов, адреса. Если logs хранятся без ограничения доступа — это нарушение 152-ФЗ (закон о персональных данных). Нужно: маскировать PII в логах, ограничить доступ к логам, хранить логи не дольше необходимого.

## Основные концепции

### Docker Logging Drivers

Docker logging driver определяет, куда и как отправляются логи контейнера. Docker поддерживает несколько драйверов:

**json-file** — драйвер по умолчанию. Логи сохраняются в файл JSON на хосте (`/var/lib/docker/containers/<id>/<id>-json.log`). Просто, но без ротации логов файл может вырасти до гигабайтов.

**syslog** — отправляет логи в syslog (системный логер Linux). Позволяет использовать существующую инфраструктуру логирования.

**journald** — отправляет логи в systemd journal. Интегрируется с systemd, поддерживает структурированные логи.

**fluentd** — отправляет логи в Fluentd (лог-агрегатор). Fluentd может фильтровать, трансформировать и отправлять логи в разные системы (Elasticsearch, S3, Kafka).

**gelf** — отправляет логи в формат GELF (Graylog Extended Log Format). Используется с Graylog (лог-менеджмент система).

**splunk** — отправляет логи напрямую в Splunk.

**awslogs** — отправляет логи в Amazon CloudWatch Logs.

**gcplogs** — отправляет логи в Google Cloud Logging.

Для enterprise рекомендуется fluentd или gelf: они позволяют централизованно собирать логи со всех хостов, применять политики хранения и доступа. json-file — для разработки, не для production.

### Метрики контейнеров: cAdvisor, Prometheus, Grafana

**cAdvisor** (Container Advisor) — инструмент от Google для сбора метрик контейнеров: CPU, память, диск, сеть, файловые системы. cAdvisor работает как отдельный контейнер, который «видит» все контейнеры на хосте через Docker API.

**Prometheus** — система мониторинга и алертинга. Собирает метрики из cAdvisor (и других источников) и сохраняет их в time-series базе данных. Поддерживает мощный язык запросов (PromQL).

**Grafana** — система визуализации. Отображает метрики из Prometheus (и других источников) в виде дашбордов. Позволяет настраивать оповещения.

Стандартный стек мониторинга для Docker: cAdvisor (сбор метрик) → Prometheus (хранение + alerting) → Grafana (дашборды+визуализация). Этот стек открыт, бесплатен, поддерживается огромным сообществом.

### Runtime Security: Falco

**Falco** — runtime security engine от Sysdig. Анализирует системные вызовы (syscalls) контейнеров в реальном времени и обнаруживает подозрительную активность.

Falco — это как охранник, который смотрит в камеры наблюдения (системные вызовы) и знает, что должно происходить нормально. Если охранник видит аномалию (кто-то идёт не в свою квартиру, пытается открыть чужую дверь) — он сигнализирует.

Правила Falco описывают, что считается подозрительным:

| Правило | Описание | Severity |
|---------|----------|----------|
| Terminal shell in container | Запуск shell внутри контейнера | Critical |
| Read of sensitive file | Чтение /etc/shadow, /root/.ssh | High |
| Outbound network connection | Неожиданное сетевое соединение | Medium |
| Write to binary directory | Запись в /usr/bin, /sbin | High |
| User mgmt command | Создание нового пользователя | High |
| Packet socket creation | Создание raw-сокета (сниффинг) | Medium |

Falco также поддерживает обнаружение аномалий на уровне приложения с помощью Falco plugins (например, плагин для PostgreSQL, для Kubernetes).

Falco — один из ключевых инструментов для runtime-безопасности контейнеров. Используется в enterprise вместе с SIEM (Splunk, ELK) для централизованного анализа и алертинга.

### ELK Stack: Централизация логов

ELK Stack (Elasticsearch, Logstash, Kibana) — стандарт для централизованного сбора, хранения и анализа логов.

**Elasticsearch** — поисковая и аналитическая БД. Хранит логи и позволяет быстро искать по ним (полнотекстовый поиск, фильтры, агрегации).

**Logstash** — лог-агрегатор. Собирает логи из разных источников (Filebeat, Docker logging driver), фильтрует, трансформирует (парсинг JSON, маскирование PII) и отправляет в Elasticsearch.

**Kibana** — визуализация и дашборды. Позволяет искать по логам, строить графики, настраивать алерты.

Для Docker-инфраструктуры используется Filebeat (лёгкий log shipper от Elastic), который собирает логи из Docker json-file драйвера и отправляет в Logstash. Filebeat поддерживает автоматическое обнаружение контейнеров: при запуске нового контейнера Filebeat начинает собирать его логи без дополнительной конфигурации.

### Audit Logging Docker Daemon

Docker daemon генерирует события (events): запуск/остановка контейнера, создание образа, монтирование volume. Эти события можно собирать через Docker Events API и отправлять в систему мониторинга.

Docker Events — это история всех действий с Docker. Кто (не для всех событий), когда, что сделал. Events помогают ответить на вопросы: «Когда был запущен этот контейнер?», «Кто создал этот образ?», «Почему упал контейнер?».

Для аудита Docker daemon также используется auditd (Linux audit framework). Auditd логирует все системные вызовы, связанные с Docker. Это более детальный, но и более ресурсоёмкий подход. Auditd рекомендуется для SOC 2 / PCI DSS, где требуется детальный аудит всех действий с контейнерами.

## Сравнение инструментов мониторинга

| Инструмент | Назначение | Тип | Источники | Формат данных | Интеграция |
|------------|------------|-----|-----------|---------------|------------|
| cAdvisor | Метрики контейнеров | Сборщик | Docker API | Prometheus | Prometheus |
| Prometheus | Хранение + алерты | TSDB | cAdvisor, node_exporter | Time-series | Grafana |
| Grafana | Дашборды + визуализация | Визуализация | Prometheus, Loki | PromQL | Сотни плагинов |
| Loki | Логи | Хранилище | Promtail, Docker | Logs | Grafana |
| Falco | Runtime security | Анализ syscalls | Ядро (eBPF) | JSON alerts | SIEM, Slack, PagerDuty |
| ELK Stack | Логи + анализ | Хранилище + UI | Filebeat, Logstash | JSON | Сотни |
| Docker Events | Аудит Docker | API | Docker daemon | JSON | ELK, Prometheus |

## Практические применения

**Сценарий 1: Prometheus + Grafana для HR-платформы.** На каждом хосте Docker запущен cAdvisor. Он собирает метрики всех контейнеров: CPU, память, сеть. Prometheus забирает метрики из cAdvisor. Grafana показывает дашборды: загрузка каждого микросервиса, процент используемой памяти, сетевой трафик. Настроены алерты: CPU >80% — P3 (уведомление), контейнер не отвечает >30 сек — P2 (проверка), контейнер крашится >3 раз за час — P1 (инцидент).

**Сценарий 2: Falco для обнаружения атак.** На всех production-хостах установлен Falco. Правила: «Terminal shell in container» — критический алерт. Когда разработчик случайно заходит в production-контейнер через `docker exec -it`, Falco отправляет alert в Slack и SIEM. Если это злоумышленник — alert отправлен за секунду после выполнения shell. Команда безопасности может отреагировать до того, как нанесён ущерб.

**Сценарий 3: ELK для централизованных логов.** Все контейнеры HR-платформы отправляют логи в Elasticsearch через Filebeat. Каждое приложение пишет в stdout/stderr. Filebeat собирает логи, добавляет метаданные (имя сервиса, хост, ID контейнера). Logstash фильтрует PII (маскирует номера паспортов). Elasticsearch хранит логи 30 дней (текущие) + 1 год в cold storage. Kibana — дашборды для разработчиков, аналитиков, безопасности.

**Сценарий 4: Docker Events для аудита.** Docker Events отправляются в ELK. Создан дашборд «Docker Activity»: события за последние 24 часа, топ создателей контейнеров, топ используемых образов. Если кто-то запускает контейнер из образа, не прошедшего security review — создаётся alert. Если контейнер запущен не через Swarm/K8s, а напрямую — alert. Docker Events — основа для audit logging Docker-инфраструктуры.

**Сценарий 5: OOM и restart monitoring.** Memory cgroup убила контейнер из-за превышения лимита. cAdvisor фиксирует событие: память выросла до лимита, контейнер убит. Prometheus alert срабатывает. DevOps получает уведомление: «Контейнер hr-api убит OOM. Текущая память: 512 MB, лимит: 512 MB». DevOps увеличивает лимит или находит утечку памяти. Без мониторинга OOM был бы обнаружен, когда пользователи сообщили бы об ошибке.

## Ограничения и нюансы

**Мониторинг — это накладные расходы.** cAdvisor потребляет ~1-3% CPU на хосте. Prometheus может потреблять много памяти при большом количестве метрик. В enterprise с тысячами контейнеров мониторинг — отдельная инфраструктура, требующая ресурсов.

**Логи могут содержать PII.** Логи приложения могут содержать персональные данные. ELK должен маскировать PII, иметь RBAC, хранить логи ограниченное время. Нарушение — штрафы за GDPR/152-ФЗ.

**Falco генерирует false positives.** Shell в контейнере может быть легитимным (разработчик отлаживает). Но для Falco это критический алерт. Нужно настраивать exceptions и rulesets, чтобы отсекать легитимную активность. Иначе команда перестанет реагировать на alerts (alert fatigue).

**Docker Events не содержат аутентификацию.** Docker Events не говорят, кто выполнил операцию — только что и когда. Если несколько администраторов имеют доступ к Docker — невозможно определить, кто запустил подозрительный контейнер. Для аудита «кто» нужен дополнительный logging (Linux auditd, SSH logs).

**Хранение логов стоит денег.** В enterprise с сотнями контейнеров логи генерируют терабайты в день. Хранение всех логов дорого. Политика: логи для отладки — 7 дней, ошибки (ERROR, FATAL) — 90 дней, аудит безопасности — 1 год.

## Архитектурные решения

**Централизованное логирование.** Все контейнеры отправляют логи в центральное хранилище (ELK, Loki, Splunk). Логи не хранятся локально — при падении хоста логи не теряются. Доступ к логам — через RBAC: разработчик видит логи своего сервиса, аналитик — ошибки, безопасность — все логи.

**Мониторинг как часть CI/CD.** В CI/CD pipeline включается проверка: «Не сломает ли этот деплой мониторинг?» Новый микросервис должен быть зарегистрирован в Prometheus (ServiceMonitor), отправлять логи в ELK, иметь healthcheck endpoint. Если мониторинг не настроен — развёртывание блокируется.

**Alert on security events.** Security alerts (Falco) имеют самый высокий приоритет. Они отправляются не только в SIEM, но и в PagerDuty/Opsgenie для немедленной реакции. Rule: critical security alert → дежурный проверяет в течение 5 минут. Если false positive — rule уточняется. Если реальная атака — инцидент эскалируется.

**Log rotation и retention.** Настраивается ротация логов Docker: максимальный размер файла лога, максимальное количество файлов. В daemon.json: `log-opts: { "max-size": "10m", "max-file": "3" }`. Это предотвращает заполнение диска логами. В ELK retention policy: hot (7 дней) → warm (30 дней) → cold (1 год) → delete.

### Ответы на чек-лист

1. **Чем мониторинг отличается от логирования?** Мониторинг — сбор числовых метрик (CPU, память, скорость запросов) для наблюдения за состоянием системы в реальном времени. Логирование — запись событий (строковые сообщения) для отладки и расследования. Мониторинг отвечает на вопрос «Что происходит?», логирование — «Что именно произошло и в какой последовательности?». Мониторинг — датчики на приборной панели, логирование — чёрный ящик.

2. **Как cAdvisor собирает метрики контейнеров?** cAdvisor использует Docker API для получения списка контейнеров и cgroups-интерфейс ядра для чтения метрик. Он читает файлы в /sys/fs/cgroup/memory/, /sys/fs/cgroup/cpu/, /sys/fs/cgroup/blkio/ для каждого контейнера. cAdvisor экспортирует метрики в формате Prometheus по HTTP-эндпоинту /metrics. Prometheus периодически забирает эти метрики для хранения и анализа.

3. **Как Falco обнаруживает аномалии в runtime?** Falco использует eBPF (Extended Berkeley Packet Filter) для мониторинга всех системных вызовов контейнера. Он перехватывает syscall'ы (open, execve, mount, ptrace) и проверяет их против набора правил. Правила описывают, что считается нормальным, а что — подозрительным. Если syscall соответствует правилу с тегом MALWARE, FALCO немедленно отправляет alert. Falco работает на уровне ядра, не требует изменения кода приложения.

4. **Что такое ELK Stack и как он используется?** ELK — Elasticsearch (поисковая БД), Logstash (агрегатор/трансформер), Kibana (визуализация). Используется для централизованного логирования: Filebeat собирает логи с хостов и отправляет в Logstash, Logstash парсит и обогащает логи (добавляет метаданные, маскирует PII), Elasticsearch индексирует логи для быстрого поиска, Kibana позволяет искать и визуализировать логи через веб-интерфейс.

5. **Как Docker Events помогают в аудите?** Docker Events — поток событий от Docker daemon: запуск/остановка контейнера, создание/удаление образа, монтирование volume, создание сети. Events собираются через команду `docker events` или API. Они отправляются в ELK для анализа. Events помогают ответить: «Когда был запущен контейнер?», «Какой образ использовался?», «Кто создал этот контейнер?» (если настроена аутентификация). Events — основа для аудита Docker-инфраструктуры.

6. **Почему логи могут содержать PII?** Приложение может логировать запросы пользователей, включая персональные данные: имена, телефоны, email, паспортные данные, адреса. Если эти логи хранятся без маскирования и доступны широкому кругу сотрудников — это нарушение GDPR, 152-ФЗ, CCPA. Нужно: настроить маскирование PII в Logstash (фильтр mutate/grok), ограничить доступ к логам (RBAC в Kibana), установить политику хранения (удалять логи с PII после обработки).

7. **Как настроить ротацию логов Docker?** В daemon.json: `"log-driver": "json-file", "log-opts": {"max-size": "10m", "max-file": "3"}`. Docker будет хранить максимум 3 файла логов по 10 MB каждый. Когда файл достигает 10 MB — Docker создаёт новый файл. После 3 файлов — самый старый удаляется. Ротация предотвращает заполнение диска логами (размер логов на контейнер ограничен 30 MB). Для production используйте внешний logging driver (fluentd, gelf) и отключите локальное хранение логов.

8. **Что такое OOM и как его мониторить?** OOM (Out Of Memory) — ситуация, когда контейнер превышает лимит памяти. Ядро Linux вызывает OOM killer, который убивает процесс в контейнере. cAdvisor фиксирует: memory.usage_in_bytes достигла memory.limit_in_bytes. Prometheus alert: `container_memory_working_set_bytes{container="hr-api"} > container_spec_memory_limit_bytes * 0.9`. Если OOM произошёл — DevOps увеличивает memory limit или находит утечку памяти. Мониторинг OOM предотвращает падение контейнера.

9. **Как Prometheus alerting помогает безопасности?** Prometheus Alertmanager отправляет уведомления при выполнении условий алерта. Примеры security-алертов: контейнер запущен не из корпоративного registry (container_image != "harbor.internal/*"); контейнер имеет привилегированный режим (container_privileged == true); неожиданные сетевые соединения (container_network_receive_bytes_total spike). Alerts отправляются в SIEM, Slack, PagerDuty. Alerting превращает мониторинг из пассивного наблюдения в активную защиту.

10. **Как централизовать логи со всех хостов?** Шаги: (1) настроить Docker logging driver на всех хостах на syslog или fluentd; (2) развернуть Logstash или Fluentd централизованно; (3) Filebeat на каждом хосте собирает логи из Docker и отправляет в Logstash; (4) Logstash фильтрует, маскирует PII, добавляет метаданные; (5) логи индексируются в Elasticsearch; (6) Kibana для поиска и визуализации.
