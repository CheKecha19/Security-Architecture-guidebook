# Docker Network Security: Моделирование и защита сетевого периметра контейнеров через аналогию с дорожной системой города

## Что такое Docker Network Security?

Представьте себе современный мегаполис с развитой дорожной инфраструктурой. В нём есть скоростные магистрали (магистральные overlay-сети, соединяющие дата-центры), квартальные улицы с односторонним движением (bridge-сети на отдельном хосте), шлагбаумы на въезде в жилые комплексы (ingress-контроль на границе кластера), подземные паркинги с пропускной системой (service mesh с взаимной TLS-аутентификацией) и пешеходные зоны, куда автомобилям въезд запрещён (изолированные сегменты для sensitive workloads). Docker Network Security — это комплекс правил дорожного движения, дорожных знаков, шлагбаумов, камер наблюдения и патрульных экипажей, которые в совокупности гарантируют, что каждый участник движения (контейнер) попадёт ровно туда, куда ему разрешено, и не окажется там, где его присутствие нежелательно или опасно.

В классической монолитной архитектуре сеть была простой: приложение слушает порт на сервере, файрвол решает, кого пускать. В мире контейнеров эта простота испаряется мгновенно. На одном физическом хосте могут работать десятки контейнеров, каждый со своим сетевым неймспейсом, виртуальным интерфейсом и IP-адресом. Они общаются друг с другом через виртуальные коммутаторы (bridge), маршрутизируются через overlay-сети, пробрасывают порты на хост и наружу. Без продуманной сетевой политики контейнер с уязвимым WordPress может внезапно получить прямой доступ к внутренней базе данных PostgreSQL, которая по замыслу архитектора должна быть доступна только бэкенд-сервису, или — того хуже — просканировать всю внутреннюю сеть кластера и найти незащищённый etcd.

Ключевая проблема контейнерной сети в том, что изоляция по умолчанию недостаточна. Docker создаёт bridge-сеть, где все контейнеры видят друг друга, если явно не указано иное. Kubernetes идёт чуть дальше: Pod'ы в разных неймспейсах изолированы на уровне DNS, но сетевой доступ между ними открыт, пока не применены **Network Policies**. Это всё равно что построить жилой комплекс, где в каждой квартире свой ключ, но двери между квартирами отсутствуют — соседи могут свободно ходить друг к другу через общие коридоры без каких-либо ограничений. Именно сетевая безопасность Docker превращает эти «открытые коридоры» в контролируемые проходы с турникетами, проверяющими удостоверение каждого, кто пытается пройти.

Сетевая безопасность контейнеров охватывает несколько измерений одновременно. Во-первых, это контроль трафика между контейнерами на одном хосте (восток-запад, east-west). Во-вторых, это контроль трафика между контейнерами и внешним миром — как входящего (ingress), так и исходящего (egress). В-третьих, это шифрование трафика на проводе между узлами кластера (север-юг, north-south, если считать дата-центры «верхом» и «низом»). И наконец, это обеспечение того, что сам механизм виртуализации сети не создаёт новых уязвимостей — например, контейнер в режиме host-сети не получает неограниченный доступ к сетевым интерфейсам хоста.

## Эволюция и мотивация

Истоки контейнерной сетевой безопасности лежат задолго до Docker — в изоляции процессов через chroot, FreeBSD jails и Solaris Zones начала 2000-х. Но настоящий катализатор появился в 2013 году с запуском Docker, когда модель «один процесс — один контейнер» столкнулась с реальностью распределённых микросервисов. Первые версии Docker предлагали простейший сетевой драйвер: bridge с NAT, проброс портов через docker-proxy и никаких средств фильтрации трафика между контейнерами. Разработчики были в восторге от скорости развёртывания, безопасники — в ужасе от сетевого хаоса.

Перелом наступил в 2015 году, когда Docker представил концепцию **libnetwork** и подключаемых сетевых драйверов (bridge, overlay, macvlan, host, none). Появилась возможность создавать изолированные bridge-сети, где контейнеры из разных сетей не видели друг друга на уровне L2/L3. Однако этого всё ещё было недостаточно для сложных политик — требовалась фильтрация на уровне сетевых протоколов, портов и конкретных IP-адресов. Kubernetes в версии 1.3 (2016 год) представил **NetworkPolicy API** — декларативный способ описания правил ingress/egress для Pod'ов, который реализуется через CNI-плагины (**Container Network Interface**). Это был момент, когда контейнерная сетевая безопасность перестала быть «чёрным ящиком» и стала программируемой.

Параллельно развивались service mesh-технологии. **Linkerd** (первый релиз в 2016 году) и **Istio** (анонсирован в 2017 году компаниями Google, IBM и Lyft) предложили радикально иной подход: вместо того чтобы полагаться на сетевые правила на уровне L3/L4, они внедрили sidecar-прокси (**Envoy**) в каждый Pod, который перехватывает весь трафик и применяет политики на уровне L7 (HTTP, gRPC). Service mesh стал аналогом «умного светофора» на каждом перекрёстке, который не просто пропускает или блокирует машины, но проверяет их документы (mTLS-сертификаты), маршрут (путь URL или gRPC-метод) и даже содержимое (JWT-токены в заголовках).

Ещё один важный виток — появление **encrypted overlay**. Если на заре Docker overlay-сети передавали трафик между узлами в открытом виде (полагаясь на безопасность физической сети дата-центра), то после эпидемии атак man-in-the-middle и требований compliance (PCI DSS, HIPAA) индустрия двинулась к шифрованию всего межхостового трафика. Docker Swarm с флагом --opt encrypted, Weave Net с встроенной NaCl-криптографией, Calico с WireGuard, Cilium с IPsec и WireGuard — каждое решение предлагало свой механизм создания зашифрованной магистрали между «районами города».

Современный ландшафт (2024–2026) характеризуется конвергенцией сетевой безопасности и runtime-безопасности. **Cilium**, построенный на eBPF, способен не только применять сетевые политики, но и наблюдать за системными вызовами, обеспечивая сквозную видимость: какой процесс в каком контейнере инициировал сетевое соединение и что именно он передал. Проект **Tetragon** от тех же авторов демонстрирует, как eBPF стирает границу между сетевой и хостовой безопасностью, позволяя детектировать аномалии на стыке сетевого трафика и поведения процессов.

## Зачем это нужно?

Первый сценарий — **предотвращение латерального перемещения**. Злоумышленник скомпрометировал фронтенд-контейнер (например, через XSS, приведшую к RCE на Node.js). Без сетевых политик он может немедленно начать сканировать внутреннюю сеть кластера: nmap по диапазонам Pod-сети, обнаружение Redis без пароля, MongoDB с дефолтными креденшелами, внутренних API микросервисов без аутентификации. Каждый такой скачок (lateral hop) приближает его к критичным данным. С правильно настроенными Network Policies фронтенд-контейнер физически не может доставить пакет до базы данных или внутреннего сервиса — его трафик отбрасывается на уровне iptables/eBPF ещё до того, как покинет сетевой интерфейс Pod'а. Бизнес-риск сценария: потенциальная утечка всей клиентской базы данных, простой сервиса, штраф GDPR.

Второй сценарий — **предотвращение эксфильтрации данных**. Предположим, злоумышленник всё-таки добрался до базы данных (через скомпрометированный внутренний сервис, у которого доступ был) и хочет выгрузить её содержимое на внешний сервер. Без egress-контроля (исходящих сетевых политик) он свободно устанавливает TCP-соединение с любым хостом в интернете — загружает данные через curl на свой VPS, отправляет через DNS-туннель или использует ICMP-эксфильтрацию. Egress Network Policies и egress-гейтвеи (как в Istio) решают эту проблему: контейнер с базой данных может вообще не иметь доступа в интернет, а контейнер, которому интернет нужен, выходит через контролируемый прокси с DLP-инспекцией. Бизнес-риск: потеря интеллектуальной собственности, нарушение data residency, регуляторные санкции.

Третий сценарий — **защита от компрометации самого сетевого слоя**. Атакующий может целиться не в приложение, а в инфраструктуру: ARP-спуфинг в bridge-сети, эксплуатация уязвимости в CNI-плагине, подмена DNS-записей, перехват незашифрованного трафика между нодами. Шифрованный overlay (IPsec/WireGuard) делает перехват трафика между нодами бессмысленным. Kubernetes гарантирует, что каждый Pod получает уникальный IP и не может выдать себя за другой Pod. Включение **hairpin NAT** и правильная настройка **iptables** в bridge-драйвере предотвращают атаки на уровне L2 внутри хоста. Бизнес-риск: полная потеря конфиденциальности коммуникаций всех микросервисов, возможность инжекции вредоносных данных в трафик.

Четвёртый сценарий — **изоляция мультитенантных сред**. В крупных организациях один кластер Kubernetes часто разделяется между множеством команд (tenant'ов). Без сетевой сегментации команда аналитики может случайно (или злонамеренно) подключиться к Pod'у с production-данными финансового отдела. Network Policies, привязанные к namespace labels, обеспечивают изоляцию tenant'ов на сетевом уровне, а Cilium ClusterwideNetworkPolicy — на уровне всего кластера. Бизнес-риск: нарушение разделения обязанностей (segregation of duties), кросс-тенантная утечка данных, аудиторские замечания по SOX/PCI DSS.

Пятый сценарий — **обеспечение Zero Trust сетевой архитектуры**. Zero Trust отказывается от концепции «внутренний значит доверенный» и требует аутентификации, авторизации и шифрования каждого сетевого соединения. Docker Network Security в сочетании с service mesh реализует это на практике: каждый запрос от сервиса A к сервису B предъявляет mTLS-сертификат, проверяется на соответствие политике авторизации (Istio AuthorizationPolicy) и логируется. Без правильно выстроенной сетевой безопасности Zero Trust остаётся красивой презентацией без практического воплощения.

## Основные концепции

### Bridge Networks: квартальные дороги одного хоста

**Bridge-сеть** — это частная виртуальная сеть уровня L2, изолированная в пределах одного Docker-хоста. Когда вы запускаете контейнер без указания сети, он подключается к дефолтной bridge-сети docker0, в которой все контейнеры видят друг друга по IP и могут свободно обмениваться трафиком. Это похоже на закрытый жилой квартал, где все дома соединены общими дорожками без заборов: сосед может дойти до вашей двери беспрепятственно. С точки зрения безопасности дефолтная bridge-сеть — анти-паттерн.

Пользовательские bridge-сети (созданные через docker network create --driver bridge) обеспечивают лучшую изоляцию: контейнеры в разных bridge-сетях не имеют L2/L3-связности друг с другом. Кроме того, пользовательские bridge-сети предоставляют автоматическое DNS-разрешение имён контейнеров — это не только удобство, но и защита от атак DNS-спуфинга, поскольку резолвинг происходит через встроенный DNS-сервер Docker (127.0.0.11), а не через внешний, потенциально скомпрометированный резолвер. Правило хорошего тона: каждое логическое приложение получает собственную bridge-сеть, и контейнеры разных приложений друг с другом не общаются.

На уровне реализации bridge-сеть — это виртуальный Linux bridge плюс цепочки iptables, обеспечивающие NAT для исходящего трафика и проброс портов для входящего. Контейнер видит виртуальный Ethernet-интерфейс (veth pair), второй конец которого подключён к bridge. Весь входящий и исходящий трафик проходит через netfilter — это одновременно и механизм обеспечения связности, и потенциальная поверхность эскалации, поскольку ошибки в правилах iptables могут открыть доступ к хостовым интерфейсам. Встроенный межсетевой экран Docker (**docker-proxy** и правила межконтейнерной коммуникации с флагом --icc) обеспечивают базовую фильтрацию, но не заменяют Network Policies в production-среде.

### Overlay Networks: магистрали между городами

**Overlay-сеть** — это виртуальная сеть, построенная поверх физической (андерлейной) сети, объединяющая хосты в распределённый L2-сегмент. В контексте Docker Swarm и Kubernetes overlay-сети позволяют контейнерам на разных физических узлах общаться так, как будто они находятся в одной локальной сети. Это аналогично системе скоростных автомагистралей, связывающих разные города: автомобиль (пакет) въезжает на магистраль в одном городе, движется по выделенной полосе (VXLAN/Geneve-туннель) и съезжает в нужном районе другого города, не взаимодействуя с местным трафиком промежуточных населённых пунктов.

Технически overlay-сеть инкапсулирует Ethernet-фреймы контейнера в UDP-пакеты (VXLAN) или использует другие протоколы туннелирования и отправляет их через физическую сеть на целевой хост, где они декапсулируются и доставляются контейнеру-получателю. Docker Swarm overlay использует VXLAN с шифрованием IPSec (при включении флага --opt encrypted), Kubernetes CNI-плагины — VXLAN (Flannel), Geneve (Calico в режиме overlay), VXLAN/Geneve (Cilium) или собственные протоколы (Weave Net с sleeve/fastdp).

Главный риск overlay-сети без шифрования — пассивный перехват трафика. Физическая сеть дата-центра может считаться доверенной, но после Snowden-откровений и массовых атак на цепочки поставок сетевого оборудования полагаться на это — опасная иллюзия. Злоумышленник, получивший доступ к физическому коммутатору или гипервизору, может зеркалировать VXLAN-трафик и читать межсервисные коммуникации в открытом виде. Encrypted overlay решает эту проблему ценой увеличения задержки на 5–15% и дополнительной нагрузки на CPU.

### Network Policies: правила дорожного движения

**Network Policy** в Kubernetes — это ресурс API, описывающий, какой трафик разрешён между Pod'ами (и от/к внешним эндпоинтам). Это полный аналог правил дорожного движения в городе: знак «кирпич» на въезде в служебную зону (запрет ingress на Pod базы данных откуда-либо, кроме бэкенда), «только прямо» на перекрёстке (разрешение egress от Pod'а фронтенда только к бэкенду на порту 8080) или «жилая зона — сквозной проезд запрещён» (полная изоляция Pod'а платёжного шлюза от всех, кроме процессингового сервиса).

Network Policy оперирует селекторами Pod'ов и неймспейсов: policy применяется к Pod'ам, соответствующим podSelector, и определяет ingress/egress-правила через комбинацию namespaceSelector и podSelector. Правило описывает «от кого» (from) или «кому» (to), а также порты и протоколы (TCP/UDP/SCTP). По умолчанию в Kubernetes все Pod'ы могут общаться со всеми — Network Policy не «разрешает» трафик, а наоборот, «запрещает» его для тех Pod'ов, к которым применён policy. Как только Pod селектится хотя бы одной Network Policy с ingress-правилом, весь входящий трафик, не соответствующий ни одному разрешающему правилу, блокируется.

Важно понимать, что Network Policy — это декларация намерения, а не механизм. Её реализует CNI-плагин, и возможности политик зависят от того, какой плагин выбран. Базовые CNI (Flannel) не поддерживают Network Policies вовсе. Calico, Cilium и Weave Net транслируют их в правила iptables или eBPF-программы. Cilium идёт дальше и реализует **CiliumNetworkPolicy** — расширенную версию, поддерживающую правила уровня L7 (HTTP-пути, gRPC-методы, Kafka-топики) и правила на основе DNS-имён (FQDN), что невозможно в стандартном Kubernetes NetworkPolicy.

### Ingress/Egress Control: таможня на границах

**Ingress-контроль** — это фильтрация входящего трафика: что из внешнего мира может попасть в кластер и к каким Pod'ам. На уровне Docker это проброс портов (docker run -p 8080:80), который открывает порт на всех сетевых интерфейсах хоста и направляет трафик в контейнер. На уровне Kubernetes это Ingress Controller (nginx, Traefik, HAProxy, Ambassador), который терминует TLS, маршрутизирует HTTP(S)-запросы и может выполнять WAF-фильтрацию, rate-limiting и аутентификацию. Ingress — это пограничный контрольно-пропускной пункт на въезде в город: весь транспорт проходит досмотр, прежде чем попасть на городские улицы.

**Egress-контроль** — обратная сторона: фильтрация исходящего трафика из Pod'ов наружу. В классическом Docker ограничения egress реализуются через правила iptables на хосте (с помощью DOCKER-USER цепочки) или через egress-прокси. Kubernetes расширяет это: Egress Network Policy может ограничить, к каким внешним IP-сетям и портам Pod может устанавливать исходящие соединения. Service mesh-решения (Istio, Linkerd) добавляют Egress Gateway — выделенный Pod или набор Pod'ов, через которые проходит весь исходящий трафик кластера, с возможностью инспекции на уровне L7, логирования и блокировки по доменным именам.

Особая тема — **DNS egress**. Даже если Pod ограничен сетевыми политиками, злоумышленник может использовать DNS-запросы для эксфильтрации данных: закодировать содержимое базы в поддомены и отправить их на свой DNS-сервер (тип атаки — DNS exfiltration). Базовые L3/L4 Network Policies не защищают от этого, потому что разрешают UDP на порт 53 для DNS-резолвинга. Cilium с правилами на основе FQDN может ограничить, к каким доменным именам разрешены запросы, что делает DNS-туннелирование значительно сложнее.

### Service Mesh: умные светофоры с проверкой документов

**Service Mesh** — это выделенный инфраструктурный слой для управления сервис-сервисными коммуникациями. Он состоит из двух компонентов: плоскости данных (sidecar-прокси в каждом Pod'е, перехватывающие весь входящий и исходящий трафик) и плоскости управления (конфигурирующей эти прокси). В терминах дорожной аналогии service mesh — это система «умных» светофоров на каждом перекрёстке, которые не просто переключают цвета по таймеру, но проверяют удостоверение личности водителя (mTLS-сертификат), сверяют маршрут с разрешённым списком (AuthorizationPolicy) и записывают в журнал факт проезда каждой машины (distributed tracing, access logging).

Главный вклад service mesh в сетевую безопасность — **mTLS** (mutual TLS), или взаимная аутентификация транспорта. В обычной сети HTTP-трафик между сервисами не шифруется (они «внутри периметра»), и любой, кто прослушивает сеть, может читать чувствительные данные. Service mesh автоматически выпускает X.509-сертификаты для каждого Pod'а (через свой CA — Citadel в Istio, identity controller в Linkerd), sidecar-прокси при установлении соединения предъявляют сертификаты друг другу и проверяют их. Только после успешной взаимной аутентификации трафик начинает передаваться в зашифрованном виде. Это превращает «город» из места, где любой может подслушать разговор на улице, в пространство, где каждый разговор происходит в звукоизолированной кабине.

Второй компонент — **Authorization Policy** (в терминологии Istio) или **Server Authorization Policy** (Linkerd). Это правила, определяющие, какие сервисы могут обращаться к данному сервису, по каким HTTP-путям или gRPC-методам и с какими JWT-токенами. Например: GET /api/users разрешён только сервисам с меткой app: auth-service и требует валидный JWT с claim audience=user-api. Это уровни L7-безопасности, недоступные чисто сетевым механизмам — аналог проверки не просто прав на проезд по улице, но и цели визита в конкретный дом.

Третий компонент — observability. Service mesh генерирует метрики (latency, error rate, request rate), распределённые трейсы и access-логи для каждого запроса. С точки зрения безопасности это означает полную видимость: кто, когда, к кому и с каким результатом обращался. В случае инцидента SOC-аналитик может восстановить всю цепочку вызовов от ingress-запроса пользователя до SQL-запроса в базе данных секунда за секундой.

### CNI (Container Network Interface): подрядчики дорожного строительства

**CNI** (Container Network Interface) — это стандарт и экосистема плагинов, отвечающих за подключение контейнеров (Pod'ов в Kubernetes) к сети. CNI-плагин решает три ключевые задачи: выделение IP-адреса Pod'у, создание сетевого интерфейса внутри сетевого namespace Pod'а и настройку маршрутизации/туннелирования для обеспечения связности. Разные CNI-плагины — это как разные подрядчики, строящие дороги в одном городе: кто-то кладёт асфальт быстро и дёшево, но без разметки (Flannel), кто-то устанавливает турникеты на каждом перекрёстке (Calico с Network Policies), кто-то встраивает камеры прямо в асфальт (Cilium с eBPF).

Выбор CNI — это фундаментальное архитектурное решение с прямыми последствиями для безопасности. Flannel обеспечивает базовую связность (VXLAN или host-gw), но не поддерживает Network Policies — подходит только для dev-сред. Calico поддерживает стандартные Network Policies, транслирует их в iptables, обеспечивает IP-в IP или VXLAN/VXLAN+BPF туннелирование и может шифровать трафик WireGuard. Cilium — самый продвинутый: eBPF-реализация Network Policies, L7-правила, прозрачное шифрование (IPsec или WireGuard), интеграция с service mesh через Envoy, observability на уровне отдельных сетевых потоков. Обратная сторона продвинутости — сложность отладки: когда пакет дропается где-то в eBPF-программе, классические средства диагностики (iptables -L, traceroute) бесполезны.

Отдельного упоминания заслуживает проблема безопасности самого CNI-плагина. Поскольку CNI-компоненты работают как DaemonSet с привилегированным доступом (для создания сетевых интерфейсов в namespace Pod'ов), уязвимость в CNI — это прямая дорога к компрометации узла. Исторические CVE, такие как CVE-2022-23648 в containerd (уязвимость в обработке CNI-конфигурации) и CVE-2020-10749 в Kubernetes (возможность отправки поддельных IPv6 router advertisements через CNI), напоминают: безопасность сетевой инфраструктуры — это не только политики, но и безопасность самих сетевых компонентов.

### Host Networking: опасный режим прямого подключения

Режим **host networking** (docker run --network host, Kubernetes spec.hostNetwork: true) означает, что контейнер разделяет сетевой namespace с хостом. Он видит те же сетевые интерфейсы, слушает те же порты и имеет тот же IP-адрес, что и хост. В дорожной аналогии это не отдельный автомобиль на дороге, а мотоциклист, который едет прямо по тротуару среди пешеходов — никакого разделения. Для приложений, которым нужна максимальная производительность (обход NAT и виртуальных интерфейсов), host networking соблазнителен, но с точки зрения безопасности это катастрофа: любой процесс в контейнере может прослушивать порты хостовых сервисов, перехватывать трафик через raw sockets и взаимодействовать с другими контейнерами без каких-либо ограничений.

В production-среде host networking должен быть запрещён политикой Kubernetes (Pod Security Standard Restricted, OPA Gatekeeper). Исключение — системные компоненты (CNI-агенты, kube-proxy, мониторинг-агенты), которые по архитектуре требуют доступа к хостовой сети. Даже тогда предпочтительнее минимизировать их привилегии через capabilities вместо host networking, где это возможно. CIS Kubernetes Benchmark v1.7.0 содержит конкретные контрольные точки: 5.2.4 «Minimize the admission of containers wishing to share the host network namespace» и 5.2.5 «Minimize the admission of containers with allowPrivilegeEscalation».

### Port Exposure: открытые окна в городской крепости

Проброс портов — самый простой и самый опасный механизм Docker-сети. Команда docker run -p 8080:80 делает порт 80 контейнера доступным на порту 8080 на всех сетевых интерфейсах хоста (0.0.0.0). Это всё равно что прорубить окно в крепостной стене и повесить табличку «добро пожаловать». По умолчанию Docker добавляет правила в цепочку DOCKER таблицы filter iptables, которая обрабатывается до правил, заданных обычными файрволами (UFW, firewalld). Результат: даже если администратор настроил UFW на блокировку порта 8080, Docker-правила будут иметь приоритет, и порт останется открытым — классическая и опасная ловушка, задокументированная в issue docker/docker#22054.

Правильный подход к пробросу портов: всегда указывать конкретный IP-адрес интерфейса (docker run -p 127.0.0.1:8080:80), а не слушать на 0.0.0.0; использовать Docker Compose с явным expose (только между сервисами в одной сети) вместо ports (проброс наружу); на уровне Kubernetes не использовать NodePort и LoadBalancer без крайней необходимости, предпочитая ClusterIP с Ingress Controller; и всегда помещать Docker за reverse proxy (nginx, Traefik, HAProxy), который выполняет терминацию TLS, rate limiting и базовую WAF-фильтрацию до того, как трафик достигнет контейнера.

## Сравнение с другими механизмами

### Сравнение решений контейнерной сетевой безопасности

| Решение | Уровень изоляции | Модель политик | Шифрование | Плюсы | Минусы |
|---|---|---|---|---|---|
| Docker Bridge + ICC=false | L3 между контейнерами одного хоста | Бинарный (вкл/выкл межконтейнерная связь) | Нет | Простота, нулевая конфигурация | Никакой гранулярности: либо все контейнеры видят друг друга, либо никто |
| Kubernetes NetworkPolicy (Calico) | L3/L4 между Pod'ами кластера | Селекторы Pod/Namespace + порты | Опционально (WireGuard) | Стандарт K8s, гранулярность, интеграция с CI/CD | Только L3/L4, сложность отладки iptables |
| Cilium NetworkPolicy | L3/L4/L7 между Pod'ами + FQDN | eBPF, DNS-имена, HTTP/gRPC/Kafka | WireGuard/IPsec | Максимальная гранулярность, производительность, observability | Сложность диагностики eBPF, требователен к ядру |
| Istio Service Mesh (mTLS + AuthorizationPolicy) | L7 между сервисами | mTLS-идентичность + JWT-токены | Автоматический mTLS | Zero Trust из коробки, L7-правила, полная видимость | Высокий overhead (sidecar CPU/память), сложность эксплуатации |
| Docker Swarm Overlay + encrypted | L3 между узлами Swarm | Только сетевой уровень | IPSec (AES-256-GCM) | Простота для небольших кластеров | Проект в стагнации, нет аналога Network Policy |
| Host Firewall (iptables/nftables) | Доступ к Docker-портам снаружи | Ручные правила | Неприменимо | Тонкий контроль над проброшенными портами | Docker обходит UFW/firewalld, сложность поддержки |

Компромисс между Network Policy и Service Mesh отражает фундаментальный выбор между простотой и мощью. Network Policies работают на уровне сети: они не требуют модификации приложения, не добавляют sidecar-контейнеры, не потребляют дополнительную память и CPU. Но они слепы к содержимому трафика: нельзя запретить POST /admin без токена, но разрешить GET /health. Service Mesh добавляет L7-интеллект ценой увеличения latency (1–5 ms на хоп, в зависимости от нагрузки), расхода ресурсов (sidecar потребляет 50–200 MB RAM и 0.1–0.5 vCPU) и эксплуатационной сложности (сертификаты, обновления sidecar, дебаггинг проблем с connectivity). Современная best practice — комбинация: Network Policies для базовой L3/L4-сегментации (кто с кем вообще может говорить) и service mesh для L7-авторизации (что именно разрешено).

Выбор между Calico и Cilium — компромисс стабильности и инноваций. Calico проверен годами в enterprise, его модель iptables понятна любому сетевику Linux, и проблемы диагностируются привычными инструментами. Cilium на eBPF быстрее (особенно при большом количестве правил — iptables деградирует экспоненциально с ростом числа цепочек), предоставляет L7 и FQDN-правила и встроенную observability через Hubble. Но eBPF требует современного ядра (≥ 5.4, а для полной функциональности ≥ 5.10), а ошибки в eBPF-программах сложнее диагностировать — классические утилиты вроде conntrack и iptables -L бесполезны.

Ещё одна ось сравнения: overlay vs native routing. Overlay-сети (Flannel VXLAN, Calico IP-in-IP) добавляют инкапсуляцию и работают в любых сетевых топологиях, но создают overhead и усложняют диагностику. Native routing (Calico с BGP, Cilium в native routing mode) требует, чтобы физическая сеть знала маршруты Pod-сети, но не инкапсулирует трафик — меньше overhead, проще мониторинг. Компромисс: overlay универсален, native routing эффективен, но требует контроля над физической сетью (облачные провайдеры вроде AWS и GCP не дают BGP-пиринг, но поддерживают native routing через свои route tables).

## Уроки из реальных инцидентов

### Инцидент Tesla Kubernetes Cluster Cryptojacking (2018)

В феврале 2018 года исследователи RedLock (ныне часть Palo Alto Networks) обнаружили, что Kubernetes-кластер компании Tesla в AWS был скомпрометирован злоумышленниками, которые использовали его для криптомайнинга. Механизм компрометации был удручающе прост: административная консоль Kubernetes (dashboard) не была защищена паролем и оказалась доступна из интернета. Более того, внутри кластера отсутствовала сетевая сегментация — злоумышленник, получивший доступ к одному Pod'у через dashboard, мог свободно сканировать и эксплуатировать остальные. Криптомайнер был развёрнут в отдельном Pod'е и маскировался под нестандартный порт, используя скрытый mining pool через CloudFlare CDN — классический пример отсутствия egress-контроля.

Главный урок этого инцидента: сетевая безопасность Kubernetes не может полагаться на «защищённый периметр», особенно в облаке. Dashboard должен быть за аутентификацией и не экспонироваться вовне. Network Policies должны предотвращать латеральное перемещение — даже если один Pod скомпрометирован, он не должен иметь сетевого доступа к остальным. Egress-контроль должен блокировать соединения с неизвестными mining pool'ами и нестандартными портами. Tesla впоследствии внедрила Network Policies, изолировала production- и non-production-неймспейсы и добавила мониторинг исходящего трафика. Инцидент также стимулировал развитие Kubernetes Pod Security Standards и RBAC-политик в community.

### Уязвимость CVE-2019-5736: побег через runc и роль сетевой изоляции

CVE-2019-5736 (обнаружена Адамом Иванюком и опубликована в феврале 2019) — это одна из самых громких уязвимостей контейнерной экосистемы, позволявшая злоумышленнику вырваться из контейнера на хост через манипуляцию с файловым дескриптором процесса runc (контейнерного рантайма). Уязвимость эксплуатировалась через перезапись бинарника runc внутри контейнера — с помощью /proc/self/exe — так, что при следующем запуске любого контейнера на хосте выполнялся код злоумышленника с привилегиями root на хосте.

Сетевой аспект этого инцидента часто упускается из виду, но он фундаментален. После компрометации хоста злоумышленник получает контроль над всеми сетевыми интерфейсами узла — он может перехватывать трафик всех контейнеров, работающих на этом хосте, независимо от того, насколько тщательно настроены Network Policies внутри Kubernetes. Все overlay-сети, все bridge-сети, все CNI-плагины становятся прозрачными для атакующего, потому что он находится на уровне, где трафик ещё не инкапсулирован. Хуже того: если overlay-сеть не зашифрована, злоумышленник может читать трафик, уходящий с этого хоста на другие узлы — он сидит на «магистрали» в точке въезда.

Главный сетевой урок: defense-in-depth требует шифрования трафика даже внутри «доверенного» дата-центра. Encrypted overlay (IPsec/WireGuard) или mTLS на уровне service mesh гарантирует, что даже при полной компрометации одного узла злоумышленник не сможет читать трафик Pod'ов, работающих на этом узле и общающихся с Pod'ами на других узлах — ключи шифрования находятся в памяти Pod'а и недоступны хост-процессу. Кроме того, микросегментация через Network Policies снижает радиус поражения: даже если злоумышленник контролирует узел и может инициировать произвольный трафик, он не пройдёт аутентификацию mTLS для доступа к сервисам на других узлах.

| Класс средства | Назначение | Позиция в стеке | Релевантные стандарты |
|---|---|---|---|

Ключевая рекомендация при выборе инструментов: сетевая безопасность контейнеров не внедряется одним продуктом. Это всегда комбинация CNI (Calico/Cilium), Ingress (nginx/Traefik), mesh (Istio/Linkerd — опционально) и мониторинга (Hubble/Falco). Все компоненты должны интегрироваться между собой: Network Policy от Calico блокирует трафик на L3/L4, Istio AuthorizationPolicy добавляет L7-контроль, Hubble визуализирует потоки и показывает, какие соединения были разрешены, а какие заблокированы. Только имея все три слоя — превентивный, детективный и визуализацию — команда безопасности получает полный контроль над контейнерной сетью.

## Архитектурные решения

### Defense-in-Depth: многослойная дорожная безопасность

Defense-in-Depth в контексте Docker-сетей — это не просто лозунг, а инженерная необходимость. Один уровень защиты всегда может быть обойдён — через уязвимость в CNI, ошибку конфигурации, человеческий фактор. Городская дорожная система, построенная по принципу Defense-in-Depth, не полагается на единственный шлагбаум на въезде: у неё есть камеры на перекрёстках (мониторинг), патрули (detection), правила дорожного движения (политики), физические барьеры на опасных участках (сегментация) и подушки безопасности в каждом автомобиле (шифрование). Все эти слои должны работать одновременно.

Первый слой — изоляция на уровне Docker-демона. Отключение межконтейнерной коммуникации в дефолтной bridge-сети (--icc=false), использование пользовательских bridge-сетей для логической группировки контейнеров, явное указание IP-адресов при пробросе портов (-p 127.0.0.1:8080:80) и запрет режима host networking через конфигурацию демона. Этот слой защищает от случайного раскрытия и простейших атак на уровне одного хоста.

Второй слой — сегментация на уровне Kubernetes. Создание Network Policies «deny-all по умолчанию» для каждого неймспейса, явное разрешение только необходимых путей трафика, изоляция production/staging/development неймспейсов друг от друга и использование Pod Security Standards Restricted (запрет hostNetwork, hostPort, привилегированных Pod'ов). CIS Kubernetes Benchmark рекомендует: «Apply a default 'deny all' Network Policy to every namespace to ensure pods are isolated by default».

Третий слой — шифрование трафика. Encrypted overlay (IPsec/WireGuard) для защиты межхостового трафика от пассивного перехвата, mTLS через service mesh для аутентификации и шифрования каждого межсервисного соединения, TLS 1.3 на Ingress с HSTS и валидацией сертификатов. Шифрование на проводе — страховка на случай, если первые два слоя откажут и злоумышленник получит доступ к физической сети или скомпрометированному узлу.

Четвёртый слой — мониторинг и детекция. Falco для детекции подозрительных сетевых событий (nmap-сканирование, неожиданные исходящие соединения, эксплуатация CVE), Hubble/Cilium для визуализации потоков и поиска аномалий, egress-мониторинг DNS-запросов для обнаружения туннелирования. Этот слой не предотвращает, но обнаруживает атаку на ранней стадии — критически важно, потому что prevention может быть обойдён через 0-day.

Пятый слой — аудит и compliance. Docker Bench Security, kube-bench, CIS-аудиты на регулярной основе, автоматическая валидация Network Policies в CI/CD (проверка, что каждый новый сервис имеет Network Policy, ограничивающую его входящий и исходящий трафик). Аудит проверяет, что первые четыре слоя не деградировали со временем.

### Zero Trust: никому не верь, каждый должен предъявить пропуск

Zero Trust Architecture (NIST SP 800-207) переворачивает традиционную модель «внутри периметра — безопасно». В дорожной аналогии это город, где на каждом перекрёстке стоит патруль и проверяет документы у всех, включая местных жителей — неважно, что ты живёшь в этом квартале 20 лет, предъяви пропуск. Применительно к Docker Network Security это означает три ключевых правила: ни один Pod не доверяет другому Pod'у просто потому, что они в одном неймспейсе; каждое соединение аутентифицируется и авторизуется; весь трафик шифруется независимо от того, идёт ли он внутри одного узла или между дата-центрами.

Практическая реализация Zero Trust для контейнерных сетей требует трёх компонентов. Первый — **identity-based segmentation**: вместо правил на основе IP-адресов («разрешить трафик с IP 10.244.1.5 на порт 5432»), используются правила на основе идентичности сервиса («разрешить трафик от сервиса с identity spiffe://cluster.local/ns/backend/sa/api-service на порт 5432»). IP-адреса нестабильны (Pod пересоздаётся — получает новый IP), а identity в service mesh привязана к ServiceAccount и не меняется. Это основа Zero Trust — аутентификация на основе того, кем сервис является, а не где он находится в сети.

Второй компонент — **continuous verification**: проверка авторизации происходит на каждый запрос, а не только при установлении TCP-соединения. Istio AuthorizationPolicy применяется к каждому HTTP/2-стриму или gRPC-вызову, а не один раз на TCP-рукопожатие. Если сертификат был отозван между запросами, следующий запрос будет заблокирован. Это отличает Zero Trust от классических файрволов, которые принимают решение «разрешить/запретить» при установлении соединения и больше не проверяют.

Третий компонент — **micro-segmentation** (микросегментация): гранулярность политик доводится до уровня отдельных Pod'ов или даже контейнеров внутри Pod'а. Если классическая сегментация VLAN — это разделение на районы («трафик из района A в район B запрещён»), микросегментация — это запрет проезда между конкретными домами на конкретной улице. В Kubernetes это достигается через Network Policies с podSelector и через service mesh AuthorizationPolicy. Микросегментация минимизирует радиус поражения: даже если один сервис скомпрометирован, злоумышленник может достичь только тех Pod'ов, с которыми этому сервису явно разрешено общаться.

Важно не путать Zero Trust с «максимально строгими политиками». Zero Trust не означает «заблокировать всё, кроме двух портов» — это архитектурный принцип. Главный инженерный компромисс Zero Trust: стоимость внедрения и эксплуатационная сложность против повышения защищённости. Внедрение mTLS в кластер с 1000 Pod'ов требует управления тысячами сертификатов и их ротацией (срок жизни сертификата Istio по умолчанию — 24 часа). Написание AuthorizationPolicy для каждого эндпоинта каждого сервиса — это постоянная работа команды платформы, которая должна поспевать за скоростью изменений в коде. Zero Trust оправдан для regulated environments (финтех, healthcare, defence), где цена инцидента безопасности перевешивает стоимость внедрения. Для стартапов и внутренних инструментов разумным может быть более простой подход: Network Policies + Ingress-защита без полного service mesh.

## Подготовка к собеседованию

### Три типичные ошибки

Первая ошибка — **полагаться на ICC (Inter-Container Communication) как на Network Policy**. Администраторы часто отключают межконтейнерную коммуникацию в Docker (--icc=false) и считают, что сетевая безопасность достигнута. Это грубое упрощение: ICC=false предотвращает прямую коммуникацию между контейнерами в одной bridge-сети, но контейнеры всё ещё могут общаться через проброшенные порты друг друга (используя docker-proxy), через общие volumes с Unix-сокетами или через манипуляции с DNS. ICC — не Network Policy, это бинарный выключатель без гранулярности, и он не применяется в Kubernetes вовсе. Правильный ответ: всегда используйте Network Policies.

Вторая ошибка — **забывать, что Docker обходит host-файрвол**. Инженер настраивает UFW: ufw deny 8080, затем запускает контейнер с -p 8080:80 и удивляется, что порт открыт. Docker модифицирует iptables напрямую, и его правила обрабатываются до UFW-правил в цепочке FORWARD. Решение: использовать цепочку DOCKER-USER (которая вызывается до DOCKER в таблице filter), либо — что лучше — не пробрасывать порты на 0.0.0.0 и всегда ставить reverse proxy перед Docker. Это фундаментальная дихотомия Docker и host-файрвола, которую нужно понимать на уровне iptables-цепочек.

Третья ошибка — **игнорировать egress**. Команды тратят недели на настройку ingress-контроля и Network Policies для входящего трафика, но оставляют исходящий трафик неограниченным. Между тем, egress — это вектор эксфильтрации данных, C2-команд (Command & Control), сканирования внешних сетей и DNS-туннелирования. CIS v8 Control 4 требует ограничения сетевых портов, протоколов и сервисов, включая исходящие. Хорошая архитектура: каждый неймспейс получает egress Network Policy, разрешающую только DNS (UDP 53 на корпоративный DNS-сервер) и HTTPS на известные эндпоинты; всё остальное блокируется по умолчанию.

### Три вопроса с ответами

**Вопрос 1: Чем отличается безопасность Docker bridge-сети от Kubernetes Network Policy?**

Docker bridge-сеть обеспечивает изоляцию на L2/L3 в пределах одного хоста через виртуальный bridge и iptables. Она бинарна: контейнеры в одной bridge-сети видят друг друга (если не отключён ICC), в разных — не видят. Kubernetes Network Policy — это декларативный API, реализуемый CNI-плагином, работающий на всём кластере и позволяющий тонко настраивать разрешённый трафик: от каких Pod'ов, к каким Pod'ам, на каких портах и протоколах. Главное отличие — гранулярность и масштаб: bridge управляет связностью на одном хосте грубо (вкл/выкл), Network Policy управляет доступом на кластере тонко (allow/deny по селекторам). На практике Docker bridge используется для локальной разработки, Network Policy — для production-кластеров.

**Вопрос 2: В чём компромисс между Network Policy и Service Mesh для Zero Trust?**

Network Policy (Calico, Cilium) обеспечивает L3/L4-сегментацию с минимальным overhead — правила транслируются в iptables или eBPF и применяются на уровне ядра без дополнительных прокси. Но они не дают аутентификации (кроме IP-адреса, который может быть подделан) и не работают на L7. Service Mesh (Istio, Linkerd) даёт mTLS-аутентификацию на основе identity, L7-авторизацию (HTTP-пути, JWT), observability и шифрование — ценой дополнительных ресурсов (sidecar-контейнер), latency (+1-5 мс) и эксплуатационной сложности (управление сертификатами, обновления sidecar). Компромисс: для сред с высокими требованиями compliance необходим service mesh; для менее критичных сред достаточно Network Policies. Оптимальная архитектура использует оба: Network Policies как базовую L3/L4-защиту (кто с кем может говорить) и service mesh как L7-контроль (что именно разрешено и с какой аутентификацией).

**Вопрос 3: Как защититься от DNS-эксфильтрации данных из контейнера?**

DNS-эксфильтрация эксплуатирует разрешённый UDP-трафик на порт 53 — злоумышленник кодирует данные в поддомены и отправляет их на контролируемый DNS-сервер. Защита требует комбинации мер. Во-первых, egress Network Policy, ограничивающая исходящий DNS-трафик только корпоративным DNS-сервером, а не «любым IP на порту 53». Во-вторых, Cilium FQDN-правила, разрешающие DNS-запросы только к известным, белолистированным доменам. В-третьих, мониторинг DNS-трафика — Falco и Hubble могут детектировать аномально длинные поддомены (> 52 символов) или аномально частые DNS-запросы. В-четвёртых, на уровне корпоративного DNS-сервера — DLP-инспекция и блокировка запросов с высокой энтропией в поддоменах. Полная защита требует комбинации превентивных (Network Policy, FQDN-правила) и детективных (мониторинг DNS-аномалий) мер.

### Ответы на чек-лист

**1. Почему дефолтная Docker bridge-сеть (docker0) небезопасна и какой альтернативой следует пользоваться?**

Дефолтная bridge-сеть docker0 связывает все контейнеры, запущенные без явного указания сети, в один L2-сегмент, где они видят друг друга и могут свободно обмениваться трафиком. Это эквивалентно отсутствию дверей между квартирами: любой скомпрометированный контейнер немедленно получает доступ ко всем остальным контейнерам на хосте. Кроме того, дефолтная сеть не предоставляет автоматического DNS-разрешения имён контейнеров — разработчики вынуждены использовать устаревший флаг --link. Альтернатива — пользовательские bridge-сети: каждая логическая группа контейнеров получает свою сеть, контейнеры в разных сетях изолированы на L2/L3, а встроенный DNS-сервер обеспечивает безопасное разрешение имён без риска спуфинга.

**2. Как злоумышленник может эксплуатировать отсутствие egress-контроля для эксфильтрации данных из скомпрометированного контейнера?**

При отсутствии egress-контроля злоумышленник может свободно инициировать исходящие TCP/UDP/ICMP-соединения на любые IP-адреса в интернете. Самый простой путь — curl/wget на свой VPS с данными в теле POST-запроса. Более изощрённый — DNS-туннелирование: данные кодируются в поддомены и уходят через легитимные DNS-запросы на контролируемый злоумышленником авторитативный сервер. Третий путь — ICMP-туннелирование: данные в payload ICMP Echo Request. Четвёртый и самый труднообнаружимый — стеганография в исходящем HTTPS-трафике к легитимным сервисам (GitHub, Pastebin, S3), где данные маскируются под обычные запросы. Egress Network Policy решает все эти векторы, ограничивая IP-сети и порты, на которые Pod может отправлять трафик.

**3. Чем Network Policy в Kubernetes принципиально отличается от правил iptables на хосте?**

Network Policy — это декларативная абстракция уровня оркестратора, оперирующая Kubernetes-примитивами: Pod, Namespace, Label Selector. Правила iptables на хосте — это императивная конфигурация сетевого стека Linux, оперирующая IP-адресами, портами и интерфейсами. Принципиальная разница в уровне абстракции и времени жизни: Network Policy переживает пересоздание Pod'ов (IP меняется, но лейбл приложения app=backend остаётся), iptables-правило с хардкодом IP 10.244.1.5 ломается при первом же рестарте Pod'а. Кроме того, Network Policy — это API, проверяемый RBAC и audit-log'ом Kubernetes, в то время как iptables-правила живут вне контроля оркестратора и могут быть изменены любым процессом с CAP_NET_ADMIN. Компромисс: iptables дают более тонкий контроль на уровне хоста (можно защитить саму Docker-инфраструктуру), но Network Policies — единственный масштабируемый способ управления трафиком в динамическом окружении.

**4. Как работает mTLS в service mesh и почему это критично для Zero Trust архитектуры?**

mTLS (mutual TLS) в service mesh означает, что при установлении TCP-соединения между сервисом A и сервисом B оба sidecar-прокси предъявляют друг другу X.509-сертификаты, подписанные внутренним CA (например, Istio Citadel или cert-manager). Каждый сертификат содержит SPIFFE identity (spiffe://cluster.local/ns/namespace/sa/serviceaccount), что привязывает сетевую идентичность к Kubernetes ServiceAccount. Обе стороны проверяют сертификат партнёра: срок действия, подпись CA, соответствие identity ожидаемой. Только после успешной взаимной аутентификации начинается обмен данными, зашифрованными TLS 1.3. Для Zero Trust это критично, потому что снимает зависимость от сетевого местоположения — даже если злоумышленник находится на том же физическом сегменте и имеет IP-связность с сервисом B, он не обладает корректным сертификатом и не сможет ни аутентифицироваться, ни расшифровать трафик.

**5. В чём отличие host networking от bridge networking с точки зрения Security Context Pod'а?**

В режиме host networking Pod разделяет сетевой namespace с хостом — он видит хостовые интерфейсы (eth0, lo), использует хостовый IP-адрес и имеет прямой доступ к сетевым портам хоста без NAT. Pod Security Standard Restricted запрещает hostNetwork, потому что это разрушает изоляцию. В режиме bridge networking Pod получает собственный сетевой namespace с виртуальным Ethernet-интерфейсом (veth), уникальным IP-адресом из Pod-сети и общается с внешним миром через NAT/MASQUERADE. С точки зрения SecurityContext Pod'а ключевая разница в поле spec.hostNetwork (булево) и spec.dnsPolicy (с hostNetwork DNS-политика ClusterFirstWithHostNet для разрешения конфликтов с хостовым /etc/resolv.conf). Компромисс: host networking даёт максимальную производительность сети (нет NAT, нет инкапсуляции), но для безопасности это неприемлемо за исключением инфраструктурных DaemonSet'ов, которым архитектурно необходим доступ к хостовой сети.

**6. Как Cilium Network Policy на основе FQDN предотвращает DNS-туннелирование?**

Cilium FQDN-правила действуют на уровне DNS-запросов: политика декларирует список разрешённых доменных имён, и Cilium через eBPF-программы перехватывает DNS-запросы, проверяет запрашиваемое FQDN и разрешает или блокирует ответ. Если Pod пытается резолвить exfiltrated-data.attacker-controlled-dns-server.com, Cilium видит FQDN и, поскольку оно не соответствует белым спискам, блокирует DNS-ответ — IP-адрес не возвращается, TCP-соединение не устанавливается. Это работает на уровне L7 (DNS-протокол), а не L3/L4, и предотвращает саму возможность отправки данных через DNS, в отличие от классических Network Policies, которые разрешают UDP:53 и не анализируют содержание запроса. Компромисс: DNS-правила полагаются на правильность DNS-резолвинга (если белый лист включает *.internal.corp.com, злоумышленник может использовать поддомен этого домена, если контролирует DNS-зону), поэтому они комбинируются с DNS-мониторингом для детекции аномалий.

**7. Почему Docker проброс портов обходит UFW/firewalld и как это исправить?**

Docker добавляет правила iptables непосредственно в цепочку FORWARD и DOCKER таблицы filter, а также в цепочку PREROUTING таблицы nat. UFW и firewalld — это фронтенды к iptables, которые управляют цепочкой INPUT (трафик, адресованный хосту) и ufw-user-input, обрабатываемой после Docker-правил в FORWARD. Трафик на проброшенный порт классифицируется как FORWARD (он адресован контейнеру, а не хосту), и Docker-правила в FORWARD разрешают его до того, как хост-файрвол успевает проверить. Исправление: добавить правила блокировки в цепочку DOCKER-USER таблицы filter — она специально предназначена для пользовательских правил и вызывается до цепочки DOCKER. Также можно слушать порты только на loopback-интерфейсе (-p 127.0.0.1:8080:80) и ставить reverse proxy перед Docker для контроля входящего трафика.

**8. Какие риски создаёт отсутствие шифрования в overlay-сети между узлами кластера?**

Незашифрованная overlay-сеть передаёт трафик между узлами в открытом виде внутри VXLAN/Geneve-туннелей. Злоумышленник, получивший доступ к физической сети (скомпрометированный коммутатор, гипервизор, облачный провайдер, малварь на промежуточном хосте), может пассивно перехватывать и читать все межсервисные коммуникации — запросы к базам данных, JWT-токены, чувствительные данные клиентов, секреты, передаваемые между сервисами. Второй риск — активный man-in-the-middle: модификация трафика (инжекция вредоносных ответов от якобы легитимного сервиса) или replay-атаки (повтор ранее перехваченного запроса). Третий риск — нарушение compliance: PCI DSS 4.0 Req 4.2.1, HIPAA Security Rule §164.312(e)(1) и ISO 27001 A.13.1.1 требуют защиты передачи данных, и open traffic на проводе — прямое несоответствие аудиторским требованиям.

**9. Как комбинировать Network Policies и service mesh для достижения Defense-in-Depth?**

Network Policies реализуют первый слой — L3/L4-сегментацию: они определяют, какие Pod'ы могут вообще обмениваться IP-пакетами, по принципу «deny all, explicitly allow». Service mesh добавляет второй слой поверх — L7-авторизацию: проверяет HTTP-путь, метод, JWT-токен, mTLS-сертификат. Комбинация работает так: Network Policy блокирует TCP SYN на порт 5432 от Pod'а фронтенда к Pod'у базы данных, пакет даже не достигает sidecar-прокси. Если Network Policy разрешает трафик, он доходит до sidecar, где Istio AuthorizationPolicy проверяет, имеет ли вызывающий сервис корректный mTLS-сертификат и разрешён ли ему конкретный gRPC-метод. Эта многослойность минимизирует риск: ошибка в Network Policy компенсируется AuthorizationPolicy, ошибка в AuthorizationPolicy — Network Policy. Компромисс — сложность отладки: при connectivity-проблеме нужно проверять оба слоя.

**10. Какие стандарты и фреймворки релевантны для аудита контейнерной сетевой безопасности?**

CIS Docker Benchmark v1.6.0 (контрольные точки 2.x — настройка Docker-демона, сети, межконтейнерной коммуникации), CIS Kubernetes Benchmark v1.7.0 (секции 5.2.x — Pod Security Policies/Standards, 5.3.x — Network Policies и CNI), NIST SP 800-190 (Application Container Security Guide — разделы о network segmentation, encrypted overlay, ограничении привилегий), NIST SP 800-207 (Zero Trust Architecture — принципы identity-based segmentation и mTLS), OWASP Docker Security Cheat Sheet, CSA Container Security Matrix, PCI DSS 4.0 (Requirement 4.2 о шифровании передачи данных и сетевой сегментации), ISO 27001 Annex A controls A.13.1.1 (Network controls), A.13.1.2 (Security of network services), A.13.1.3 (Segregation in networks). Аудит должен проверять соответствие этим стандартам с использованием инструментов вроде Docker Bench Security, kube-bench и коммерческих CSPM-решений.

## Ключевые выводы для собеседования

1. **Сетевая безопасность контейнеров требует перехода от модели «периметр — наше всё» к Zero Trust микросегментации.** Docker по умолчанию создаёт «плоскую» сеть, где контейнеры свободно общаются друг с другом. Kubernetes не сильно лучше — трафик между Pod'ами открыт до первой применённой Network Policy. Единственный способ обеспечить реальную изоляцию в production — комбинация Network Policies (L3/L4 deny-all + allow-specific) и mTLS (L7 аутентификация и шифрование каждого соединения). Периметр в виде файрвола на границе кластера не защищает от латерального перемещения внутри.

2. **Egress-контроль так же важен, как и ingress, но его игнорируют в разы чаще.** Большинство инцидентов с утечкой данных происходят через исходящий трафик: злоумышленник выгружает данные на внешний сервер после успешной компрометации. Network Policy для egress-трафика — это не «дополнительная фича», а базовое требование безопасности. По стандартам CIS v8 Control 4 и NIST SP 800-53 SC-7(5), deny-by-default для исходящего трафика — обязательная практика.

3. **Выбор CNI-плагина — это архитектурное решение, определяющее границы возможного в сетевой безопасности на годы вперёд.** Flannel не поддерживает Network Policies; Calico даёт L3/L4-гранулярность; Cilium добавляет L7, FQDN-правила и eBPF-observability. Мигрировать с одного CNI на другой в production-кластере — сложная и рискованная операция, поэтому выбор должен делаться осознанно на старте проекта. Для enterprise-сред с высокими требованиями безопасности Cilium — это не хайп, а обоснованный выбор.

4. **Ни один инструмент не решает проблему контейнерной сети в одиночку — успех в комбинации слоёв.** Docker-демон с правильными настройками сети + CNI с Network Policies + Ingress Controller с WAF + опционально Service Mesh для L7-авторизации + Falco/Hubble для мониторинга. Каждый слой защищает от определённого класса атак и служит страховкой при отказе других слоёв. Сетевая безопасность контейнеров, выстроенная по принципу Defense-in-Depth, превращает «плоскую деревню» Docker-сетей в укреплённый город с многоуровневой системой защиты.
