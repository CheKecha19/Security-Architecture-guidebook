# Безопасность стека TCP/IP

## Часть 1. Знакомство с проблемой: Почему TCP/IP — это дом без замков?

Представьте, что вы строите город, где каждый дом связан дорогами с каждым другим домом. Эти дороги — единственный способ доставить почту, посылки, сообщения. Но при строительстве вы думали только о том, чтобы дороги были быстрыми и надёжными. Замки на дверях? Охрана? Пропускная система? Об этом никто не позаботился. Так был создан TCP/IP в 1970-х — сеть, где главное было соединить компьютеры, а не защитить их.

TCP/IP — это фундамент интернета, его архитектурная основа. Но этот фундамент заложен в эпоху, когда "безопасность" означала физическую защиту серверной комнаты, а не защиту от хакеров на другом континенте. Каждый уровень стека TCP/IP несёт в себе архитектурные компромиссы, которые сегодня превращаются в векторы атак. **IP** позволяет подделывать адреса отправителей. **TCP** можно перегрузить ложными соединениями. **ICMP** используется для разведки и атак. **UDP** не проверяет, кто на другом конце.

В современной инфраструктуре эти уязвимости не теоретические. **DDoS-атаки** уровня 1 Тбит/с, **IP-спуфинг** для обхода аутентификации, **TCP RST-атаки** для разрыва соединений, **ICMP-редиректы** для перехвата трафика — всё это реальные инструменты в арсенале злоумышленников.

| Уровень TCP/IP | Протоколы | Главная угроза | Почему возникает |
|---------------|-----------|---------------|-----------------|
| 4. Транспортный | TCP, UDP, SCTP | Перегрузка, спуфинг, перехват | Нет встроенной аутентификации конечных точек |
| 3. Сетевой | IP, ICMP, IGMP | Подделка адресов, реконnaissance, DoS | IP не проверяет подлинность источника |
| 2. Канальный | Ethernet, Wi-Fi, ARP | ARP-spoofing, перехват кадров | Локальная сеть доверяет всем участникам |
| 1. Физический | Кабель, радио | Перехват сигнала, DoS физического уровня | Доступ к инфраструктуре = доступ к данным |

**Эволюция:** В 1980-х взлом сетей был редкостью, потому что интернет был закрытым клубом университетов. В 1990-х появились первые черви (**Morris Worm**, 1988 — первый массовый взлом через TCP/IP-уязвимости). В 2000-х **DDoS** стал оружием конкурентной разведки. В 2010-х **Nation-state actors** начали систематическую эксплуатацию сетевых протоколов. В 2026 году **IoT-устройства** создали миллиарды новых точек входа, каждая со своим незащищённым TCP/IP-стеком.

## Часть 2. Три реальных сценария, почему это важно

### Сценарий 1: Синтетический банк и DDoS через TCP-флуд
Представьте крупный банк с онлайн-платёжной системой. Злоумышленник отправляет миллионы **SYN-пакетов** на порт 443 банковского шлюза. Каждый SYN заставляет сервер выделить память под полуоткрытое соединение (SYN-RECEIVED state). Через 30 секунд таблица соединений переполнена. Легитимные клиенты не могут подключиться. Банк теряет $2M в час простоя. Атака стоит злоумышленнику $50 в аренде ботнета.

### Сценарий 2: Фармацевтическая компания и ARP-spoofing
Сеть производственного комплекса использует **VLAN** для разделения сегментов: IT, OT (operational technology), лаборатории. Сотрудник фармацевтической компании подключает заражённый ноутбук к гостевому Wi-Fi. Зловред отправляет поддельные **ARP-ответы**, заявляя: "Я — шлюз для VLAN лабораторий". Весь трафик лабораторных приборов, включая данные о формулах лекарств, проходит через ноутбук злоумышленника. **Man-in-the-Middle** в локальной сети за 3 секунды.

### Сценарий 3: Облачный провайдер и IP-спуфинг
SaaS-платформа развёрнута в AWS. Разработчик случайно открыл **Security Group** для доступа из "0.0.0.0/0" на порт 22 (SSH). Злоумышленник подделывает IP-адрес из разрешённого диапазона AWS (IP-спуфинг внутри облака), обходит Security Group, брутфорсит SSH-ключ. Получает доступ к инстансу, находит IAM-роль с правами на S3. Сливает базу клиентов — 50M записей. Всё началось с одной ошибки в IP-уровне.

## Часть 3. Техническая глубина: Как устроены атаки на TCP/IP

### Уровень 1: Физический — Перехват и DoS
**Аналогия:** Это как если бы злоумышленник просверлил дыру в стене вашего банка и подслушивал разговоры через трубу.

На физическом уровне атаки включают **пассивный перехват** (tap'ы на кабелях, анализ электромагнитного излучения), **активное вмешательство** ( cutting fibers, jamming радиосигнала), и **side-channel** анализ (измерение потребления энергии для восстановления ключей). **TEMPEST** — классификация атак по побочным электромагнитным излучениям. Защита: экранирование кабелей, физическая охрана, оптоволокно вместо меди.

### Уровень 2: Канальный — ARP-spoofing и перехват кадров
**ARP** (Address Resolution Protocol) переводит IP-адреса в MAC-адреса. Он работает на доверии: "Если кто-то сказал, что IP 192.168.1.1 имеет MAC AA:BB:CC:DD:EE:FF — значит, так и есть". Злоумышленник отправляет поддельный ARP-ответ, и весь трафик жертвы идёт через его интерфейс.

| Атака канального уровня | Механизм | Детектирование | Предотвращение |
|------------------------|----------|---------------|---------------|
| ARP-spoofing | Поддельные ARP-ответы | ARPwatch, динамическая ARP-инспекция | Static ARP entries, DAI (Dynamic ARP Inspection) |
| MAC-flooding | Заполнение CAM-таблицы свитча | Мониторинг CAM-таблицы | Port security, лимит MAC-адресов на порт |
| VLAN-hopping | Двойное тегирование 802.1Q | Мониторинг необычных VLAN-тегов | Native VLAN отличная от user VLAN, pruning |
| STP-manipulation | Подделка BPDU для изменения топологии | Spanning-tree guard | Root guard, BPDU guard |

### Уровень 3: Сетевой — IP-спуфинг, ICMP-атаки, маршрутизация
**IP** не проверяет подлинность отправителя. Заголовок IP-пакета содержит поле Source Address, которое может быть любым. Это позволяет **IP-спуфинг**, **смурф-атаки** (ICMP echo request с поддельным source на broadcast — все отвечают жертве), **ICMP-редиректы** (перенаправление трафика через хост злоумышленника).

**BGP** — протокол маршрутизации между автономными системами. Уязвим к **route hijacking**: злоумышленник анонсирует префиксы, которые ему не принадлежат. В 2008 **Pakistan Telecom** случайно анонсировал YouTube (/24), и YouTube был недоступен 2 часа. В 2018 **Route53** был атакован через BGP hijacking для перенаправления DNS-запросов на серверы злоумышленников.

### Уровень 4: Транспортный — TCP-атаки, UDP-амплификация
**TCP SYN-flood** — классика. Злоумышленник отправляет SYN, сервер отвечает SYN-ACK, ждёт ACK. Если ACK не приходит, соединение висит в полуоткрытом состоянии. Полуоткрытых соединений может быть ограниченное количество. Решение: **SYN-cookies** (нет сохранения состояния до ACK), **SYN-proxy** (проверка валидности перед форвардом).

**UDP-амплификация** — отправка UDP-запроса с поддельным source на сервис, который отвечает большим пакетом. **DNS-амплификация**: запрос на 60 байт, ответ на 4000 байт. **NTP monlist**: запрос на 1 байт, ответ на 446 байт (список 600 клиентов). **Memcached**: амплификация 51,000x, атаки >1 Тбит/с.

| Тип TCP-атаки | Механизм | Цель | Защита |
|--------------|----------|------|--------|
| SYN-flood | Полуоткрытые соединения | Исчерпание backlog | SYN-cookies, SYN-proxy, rate limiting |
| RST-attack | Поддельные RST-пакеты | Разрыв существующих соединений | TCP MD5 signature (RFC 2385), ACL |
| Session hijacking | Предсказание sequence numbers | Перехват сессии | Random ISN, TCP MD5, IPSec |
| ACK-flood | Лавина ACK-пакетов | Перегрузка обработки | Stateful firewall, rate limiting |

## Часть 4. Архитектура защиты: Defense in Depth для TCP/IP

### Zero Trust в сетевом уровне
**Принцип:** Не доверяй никому. Каждый пакет проверяется. Каждое соединение аутентифицируется. Микросегментация — не VLAN'ы, а **identity-based segmentation**: не "разреши трафик из сети A в сеть B", а "разреши трафик от сервиса X (с сертификатом Y) к сервису Z (с сертификатом W)".

**Многоуровневая защита TCP/IP:**

| Уровень | Технология | Стандарт | Реализация |
|---------|-----------|----------|------------|
| 4. Транспорт | TLS 1.3 | RFC 8446 | OpenSSL, BoringSSL |
| 4. Транспорт | IPSec | RFC 4301 | StrongSwan, libreswan |
| 3. Сетевой | BGPsec | RFC 8205 | RPKI, ROA |
| 3. Сетевой | DNSSEC | RFC 4033 | BIND, Unbound |
| 2. Канальный | MACsec | IEEE 802.1AE | Cisco TrustSec |
| 2. Канальный | 802.1X | IEEE 802.1X | RADIUS, TACACS+ |
| 1. Физический | TEMPEST shielding | NATO SDIP-27 | Фарадеевы клетки |

### Мониторинг и метрики
**Ключевые метрики безопасности сети:**
- **SYN-flood ratio:** SYN / (SYN+ACK) — если >2, возможная атака
- **ARP-cache churn rate:** Количество изменений ARP-таблицы в минуту
- **BGP route changes:** Внезапные изменения префиксов
- **Fragmentation rate:** Подозрительно высокая фрагментация (evasion technique)
- **Protocol distribution:** Неожиданный рост ICMP или UDP

### Реальные инциденты

**Инцидент 1: DYN DDoS (2016)**
Провайдер DNS **DYN** подвергся атаке в 1.2 Тбит/с через **Mirai botnet**. Ботнет сканировал интернет на предмет IoT-устройств с дефолтными паролями (камеры, роутеры), заражал их, использовал для **UDP-амплификации** через DNS и NTP. Twitter, GitHub, Reddit, Netflix были недоступны. **Урок:** IoT + дефолтные пароли + UDP-амплификация = массовый DDoS. Защита: сегментация IoT, rate limiting на DNS/NTP, BGP Flowspec для фильтрации на уровне провайдера.

**Инцидент 2: BGP Hijacking Amazon (2018)**
Злоумышленники анонсировали префиксы **Amazon Route53** через украинского провайдера. DNS-запросы к myetherwallet.com перенаправлялись на фишинговый сайт. Пользователи отправляли ETH на кошельки злоумышленников. Потеря: ~$150K за 2 часа. **Урок:** BGP hijacking + DNS = идеальная комбинация для крипто-кражи. Защита: RPKI + ROA для валидации BGP-анонсов, DNSSEC для аутентичности ответов.

## Часть 5. Практика: Подготовка к собеседованию и чек-лист

### Три распространённых заблуждения
1. **"NAT — это безопасность"** — NAT скрывает внутреннюю топологию, но не защищает от атак изнутри, не фильтрует payload, не предотвращает спуфинг. NAT ≠ firewall.
2. **"Внутренняя сеть безопасна"** — 70% атак начинаются изнутри (инсайдеры, скомпрометированные endpoint'ы). Zero Trust применяется ко всем сегментам.
3. **"TLS решает все проблемы"** — TLS защищает payload, но не защищает от DDoS, не предотвращает IP-спуфинг, не решает проблемы маршрутизации. Безопасность TCP/IP многоуровневая.

### Три ситуационных вопроса для Senior+

**Q1:** Как защититься от SYN-flood на уровне application, если firewall уже пропускает трафик?
**A:** На application уровне — rate limiting per IP, proof-of-work для новых клиентов (hashcash), connection pooling с ограничением, circuit breaker для backend. Но лучшее решение — SYN-cookies на kernel level или cloud-based DDoS mitigation (AWS Shield, CloudFlare Magic Transit).

**Q2:** Как обнаружить ARP-spoofing в сети без специализированных инструментов?
**A:** Анализ ARP-таблицы: `arp -a` покажет, если MAC шлюза изменился. Ping gateway + tcpdump: два разных MAC отвечают на ARP-запрос. Лучше: DAI (Dynamic ARP Inspection) на свитчах — проверяет ARP-пакеты через DHCP snooping binding table.

**Q3:** Почему BGPsec не решает проблему BGP hijacking полностью?
**A:** BGPsec (RFC 8205) криптографически подписывает BGP-анонсы, но требует 100% deployment — если один AS не поддерживает, цепочка ломается. Кроме того, BGPsec не защищает от **route leak** (ошибочная анонсация валидного маршрута). RPKI + ROA проще внедрять постепенно.

### Чек-лист защиты TCP/IP (10 пунктов)

| # | Проверка | Ответ | Критичность |
|---|----------|-------|-------------|
| 1 | SYN-cookies включены на всех Linux-серверах? | `sysctl net.ipv4.tcp_syncookies = 1` | Обязательно |
| 2 | RPKI + ROA настроены для собственных префиксов? | RIR (RIPE, ARIN) выдали ROA | Обязательно |
| 3 | DHCP snooping + DAI включены на access-свитчах? | `show ip dhcp snooping`, `show ip arp inspection` | Обязательно |
| 4 | BPDU guard на всех портах с endpoint'ами? | `spanning-tree portfast bpduguard default` | Обязательно |
| 5 | Rate limiting на ICMP, DNS, NTP на границе? | ACL или Flowspec правила | Высокая |
| 6 | IPSec или WireGuard для межсегментного трафика? | Сертификаты, не pre-shared keys | Высокая |
| 7 | Мониторинг BGP route changes (префикс hijacking)? | BGPmon, EXA, или自建 система | Высокая |
| 8 | TLS 1.3 enforced, отключены старые cipher suites? | `openssl s_client -connect` проверка | Средняя |
| 9 | Физическая безопасность: оптоволокно, экранирование? | Audit кабельной инфраструктуры | Средняя |
| 10 | TEMPEST audit для critical infrastructure? | Сертификация, если требуется | Низкая |

### Key Takeaways
1. **TCP/IP создан без учёта безопасности** — каждый уровень имеет архитектурные уязвимости, которые нельзя "запатчить", только компенсировать.
2. **Defense in depth** — защита на всех 4 уровнях одновременно, не надеясь на один механизм.
3. **Zero Trust networking** — микросегментация по identity, не по IP/VLAN.
4. **Мониторинг критичен** — BGP hijacking, ARP-spoofing, SYN-flood обнаруживаются по аномалиям, не по сигнатурам.
5. **Современные атаки комбинированные** — BGP hijacking + DNS + TLS-сертификат фейк = полная компрометация цепочки доверия.
