# chroot: Изоляция файловой системы

## Что такое chroot?

Представь квартиру в многоквартирном доме. У квартиры есть дверь, за которой начинается её пространство. Комнаты, коридор, кухня, ванная — всё внутри. За дверью — общий подъезд, лифт, соседи.

Теперь представь, что ты можешь поставить ещё одну дверь прямо внутри квартиры. За эту дверь ты водишь гостя, и он видит только ту комнату, в которую попал. Он не видит остальные комнаты, не слышит, что происходит за дверью. Для него его комната — это весь мир.

**chroot** — это системный вызов Linux, который меняет корневую директорию для процесса. После chroot процесс считает, что указанная директория — это корень файловой системы. Он не видит ничего выше этой директории.

Если процесс попытается обратиться к /etc/passwd после chroot в /home/jail — он увидит /home/jail/etc/passwd, а не системный /etc/passwd. Для процесса мир начинается и заканчивается в его тюрьме.

## Зачем нужен chroot?

### Ситуация 1: FTP-сервер

FTP-сервер позволяет пользователям загружать и скачивать файлы. Каждый пользователь должен видеть только свою домашнюю директорию.

chroot позволяет ограничить пользователя его домашней папкой. Он видит / как свою домашнюю директорию и не может выйти за её пределы.

### Ситуация 2: Сборка ПО

Разработчик собирает программу из исходников. Сборочное окружение должно быть изолировано от системы.

chroot в директорию с минимальным набором инструментов позволяет собирать в чистом окружении. Системные библиотеки и утилиты не мешают.

### Ситуация 3: Тестирование

Тестировщик проверяет программу в окружении, отличном от основной системы. chroot позволяет создать такое окружение без виртуальной машины.

## Как работает chroot?

### Смена корня

Когда процесс вызывает chroot:
1. Ядро запоминает новую корневую директорию для процесса
2. Все пути, начинающиеся с /, теперь отсчитываются от новой корневой директории
3. Процесс не может выйти за пределы новой корневой директории через абсолютные пути

### Ограничения chroot

chroot имеет важные ограничения:
- Процесс должен иметь root для вызова chroot
- chroot не изолирует процессы, сеть, устройства
- Если внутри chroot есть файлы с setuid — процесс может повысить права
- chroot можно обойти через открытые файловые дескрипторы

### Что chroot изолирует

| Что | Изоляция | Пояснение |
|-----|----------|-----------|
| Файловая система | Частичная | Только пути через / |
| Процессы | Нет | Видит все процессы |
| Сеть | Нет | Использует сеть хоста |
| Память | Нет | Делит память с хостом |
| Устройства | Нет | /dev может содержать реальные устройства |

## chroot и безопасность

### Почему chroot не является полноценной песочницей

chroot был создан не для безопасности, а для удобства. Он позволяет тестировать системы в изолированном окружении. Но он не защищает от злонамеренных действий.

**Проблема 1: Обход через дескрипторы**

Если процесс открыл файл или директорию до вызова chroot — он сохраняет дескриптор. Через этот дескриптор процесс может получить доступ к файлам за пределами chroot.

Это как если бы гость, которого посадили в одну комнату, всё ещё имел ключ от входной двери квартиры.

**Проблема 2: setuid программы**

Если внутри chroot есть программа с setuid root — процесс может выполнить её и получить root. С root-правами внутри chroot он может сделать второй chroot и выйти за пределы.

**Проблема 3: /proc и /sys**

Если в chroot смонтированы /proc или /sys — процесс может видеть и модифицировать системную информацию. Это открывает путь к выходу из chroot.

**Проблема 4: ptrace**

Процесс за пределами chroot может использовать ptrace для отладки процесса внутри chroot. Это позволяет влиять на изолированный процесс.

### Когда chroot безопасен

chroot можно использовать безопасно, если:
- Процесс работает от обычного пользователя
- Нет setuid программ внутри chroot
- Нет смонтированных /proc и /sys
- Все дескрипторы закрыты перед chroot
- Используются дополнительные механизмы: namespaces, seccomp

### chroot в современных системах

Современные системы используют chroot как часть более сложной изоляции. Один chroot — недостаточно. Но chroot + namespaces + seccomp + capabilities создаёт серьёзную защиту.

## chroot vs другие механизмы

### chroot vs Namespaces

| Критерий | chroot | Namespaces |
|----------|--------|--------------|
| Изоляция | Только файловая система | Процессы, сеть, файловая система |
| Нужен root | Да | Нет (для user namespace) |
| Сложность | Просто | Сложнее |
| Безопасность | Низкая | Высокая |
| Применение | Тестирование, сборка | Контейнеры, песочницы |

### chroot vs Docker

Docker использует namespaces, cgroups, seccomp, capabilities и overlayfs. chroot — это всего лишь малая часть того, что делает Docker.

chroot меняет корневую директорию. Docker создаёт полноценное изолированное окружение.

### chroot vs Landlock

chroot меняет видимую корневую директорию для процесса. Landlock создаёт набор правил доступа к файлам.

chroot говорит: "Вот твой новый корень". Landlock говорит: "Вот файлы, к которым ты можешь обращаться".

## Архитектурные решения

### Defense in Depth

chroot — это один из слоёв защиты, но не единственный. Правильный подход:
- chroot для изоляции файловой системы
- namespaces для изоляции процессов и сети
- seccomp для фильтрации системных вызовов
- capabilities для снижения привилегий
- AppArmor/SELinux для контроля доступа

### Минимальное окружение

Для максимальной безопасности chroot должен содержать минимум необходимого:
- Только нужные библиотеки
- Только нужные программы
- Нет оболочки
- Нет компилятора
- Нет setuid программ

### Тестирование и разработка

chroot остаётся полезным для тестирования и разработки. Он позволяет быстро создать изолированное окружение без накладных расходов виртуальных машин и контейнеров.

## Ограничения

### Только файловая система

chroot изолирует только видимость файловой системы. Он не изолирует процессы, сеть, память, устройства.

### Требует root

Для вызова chroot нужны привилегии root. Обычный пользователь не может вызвать chroot.

### Не защищает от root

Если процесс получает root внутри chroot — он может обойти chroot. Защита работает только для непривилегированных процессов.

### Сложность настройки

Создание рабочего chroot окружения требует копирования библиотек, утилит, конфигурационных файлов. Легко забыть что-то важное или скопировать лишнее.

### Ответы на чек-лист

1. **chroot** меняет корневую директорию для процесса. После chroot процесс видит указанную директорию как корень файловой системы и не может выйти за её пределы через абсолютные пути.

2. **chroot не является полноценной песочницей**, потому что он не изолирует процессы, сеть, память, устройства. Он только меняет видимую корневую директорию. Процесс всё ещё работает на том же ядре, видит другие процессы, использует сеть хоста.

3. **Обход через дескрипторы**: если процесс открыл файл или директорию до вызова chroot — он сохраняет дескриптор. Через этот дескриптор процесс может получить доступ к файлам за пределами chroot, используя относительные пути.

4. **setuid программы внутри chroot опасны**, потому что они позволяют процессу получить root-права. С root-правами внутри chroot процесс может выполнить второй chroot и выйти за пределы изоляции.

5. **chroot** изолирует видимость файловой системы. **Namespaces** добавляют изоляцию процессов, сети, пользователей. Вместе они создают более полную изоляцию, чем chroot один.

6. **chroot** — это всего лишь смена корневой директории. **Docker** использует namespaces, cgroups, seccomp, capabilities, overlayfs и другие механизмы для создания полноценного изолированного окружения.

7. **Более безопасный chroot**: работа от обычного пользователя, удаление setuid программ, отсутствие /proc и /sys, закрытие всех дескрипторов, использование namespaces и seccomp в дополнение.

8. **chroot требует root**, потому что смена корневой директории — это привилегированная операция. Она позволяет процессу скрыть часть файловой системы, что может быть опасно в руках обычного пользователя.

9. **chroot не изолирует**: процессы, сеть, оперативную память, устройства, IPC, сигналы. Он только меняет видимую файловую систему.

10. **chroot** использовать для простой изоляции файловой системы, тестирования, сборки ПО. **Namespaces** использовать для полноценной изоляции контейнеров, песочниц, многопользовательских систем.
