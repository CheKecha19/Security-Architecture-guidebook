# Linux Capabilities: Мелкозернистые привилегии

## Что такое Capabilities?

Представь огромную связку ключей от целого здания. Классический root — это как быть директором со связкой, которая открывает абсолютно все двери: вход, серверная, архив, лаборатория, директорский кабинет, сейфовая.

Но директору не нужен доступ в серверную — там работает системный администратор. А системному администратору не нужен доступ к финансовым документам.

**Capabilities** в Linux — это разбиение прав суперпользователя на отдельные, независимые возможности. Вместо одной связки ключей "всё или ничего" — множество отдельных ключей, каждый для своей двери.

Каждая capability отвечает за конкретное действие. Например, одна capability позволяет менять владельца файла, другая — настраивать сеть, третья — загружать модули ядра, четвёртая — монтировать файловые системы.

Когда программа запускается от root, у неё есть все capabilities. Но можно дать программе только те capabilities, которые ей реально нужны. Остальные — убрать. Теперь, даже если программа скомпрометирована, злоумышленник не сможет сделать то, что не разрешено конкретными capabilities.

## Зачем нужны Capabilities?

### Ситуация 1: Веб-сервер

Веб-серверу нужно слушать порт 80. Порты ниже 1024 считаются привилегированными — открыть их может только root.

Без capabilities: веб-сервер запускается от root, открывает порт, потом понижает права до обычного пользователя. Но в промежутке между запуском и понижением прав — уязвимость. Если злоумышленник атакует в этот момент — сервер работает от root.

С capabilities: веб-серверу дают capability CAP_NET_BIND_SERVICE. Теперь он может открыть порт 80, работая от обычного пользователя. Root ему не нужен вообще.

### Ситуация 2: Утилита резервного копирования

Программа резервного копирования должна читать файлы всех пользователей. Для этого ей нужно обходить проверки прав доступа.

Без capabilities: программа запускается от root и имеет доступ ко всему. Если её взломают — злоумышленник получает полный контроль.

С capabilities: программе дают CAP_DAC_READ_SEARCH — возможность обходить проверки прав при чтении. Она может читать все файлы, но не может их изменять, не может менять владельцев, не может загружать модули ядра.

### Ситуация 3: Контейнеры

Контейнеры изолируют процессы. Но иногда контейнеру нужна какая-то привилегия. Например, контейнер с фаерволом должен менять правила iptables.

Вместо запуска контейнера от root — даём ему только CAP_NET_ADMIN. Контейнер может настраивать сеть, но не может делать всё остальное.

## Основные capabilities

### Сетевые возможности

| Capability | Что позволяет | Пример использования |
|------------|---------------|---------------------|
| CAP_NET_ADMIN | Настройка сети | iptables, ifconfig |
| CAP_NET_BIND_SERVICE | Привязка к портам ниже 1024 | Веб-сервер на порту 80 |
| CAP_NET_RAW | Сырые сокеты | Пинг, tcpdump |
| CAP_NET_BROADCAST | Широковещание | DHCP сервер |

### Файловые возможности

| Capability | Что позволяет | Пример использования |
|------------|---------------|---------------------|
| CAP_CHOWN | Менять владельца файла | chown |
| CAP_DAC_OVERRIDE | Обходить проверки прав | Удаление чужих файлов |
| CAP_DAC_READ_SEARCH | Обходить проверки при чтении | Резервное копирование |
| CAP_FOWNER | Игнорировать проверки владельца | Удаление открытых файлов |

### Системные возможности

| Capability | Что позволяет | Пример использования |
|------------|---------------|---------------------|
| CAP_SYS_ADMIN | Администрирование системы | mount, swapon, sethostname |
| CAP_SYS_MODULE | Загрузка модулей ядра | insmod, modprobe |
| CAP_SYS_TIME | Изменение системного времени | ntpd |
| CAP_SYS_PTRACE | Отладка других процессов | strace, gdb |
| CAP_KILL | Посылать сигналы любым процессам | kill -9 |

### Безопасность

| Capability | Что позволяет | Пример использования |
|------------|---------------|---------------------|
| CAP_SETUID | Менять UID | su, sudo |
| CAP_SETGID | Менять GID | newgrp |
| CAP_SETPCAP | Менять capabilities | Изменение своих привилегий |

## Наборы capabilities

### Effective

Effective capabilities — это те, которые процесс реально имеет в данный момент. Они определяют, что процесс может делать.

### Permitted

Permitted capabilities — это максимальный набор capabilities, который процесс может получить. Процесс может отказаться от effective capabilities, но не может получить больше, чем в permitted.

### Inheritable

Inheritable capabilities — это те, которые процесс передаёт дочерним процессам при fork и exec. Не все capabilities наследуются.

### Bounding set

Bounding set — это глобальный лимит для всей системы. Никакой процесс не может получить capabilities, которые не входят в bounding set. Это верхняя граница.

### Ambient

Ambient capabilities — это новый тип, который появился для решения проблемы наследования. Ambient capabilities наследуются при exec без необходимости явно указывать их в файле.

Это как если бы у сотрудника был бейдж с определёнными правами доступа, который он всегда носит с собой, и который автоматически передаётся его заместителю.

## Как capabilities работают?

### Процесс и его capabilities

Каждый процесс имеет наборы capabilities. Ядро проверяет эти наборы при выполнении привилегированных операций.

Когда процесс пытается сделать что-то, требующее привилегий:
1. Ядро проверяет, есть ли нужная capability в effective наборе
2. Если есть — операция разрешена
3. Если нет — операция запрещена

### Файлы и capabilities

Можно назначить capabilities исполняемому файлу. Когда пользователь запускает такой файл, процесс получает эти capabilities.

Это позволяет обычному пользователю выполнять привилегированные операции без root. Например, программа ping может иметь CAP_NET_RAW — и обычный пользователь сможет пинговать хосты.

### Наследование

При fork дочерний процесс наследует capabilities родителя.

При exec процесс получает capabilities из файла (если они назначены файлу) или теряет все capabilities (если файл не имеет capabilities).

## Capabilities и безопасность

### Drop all capabilities

Контейнеры часто запускаются с флагом "drop all capabilities" — убрать все capabilities. Процесс не может делать ничего привилегированного.

Это безопасно, но ограничивает функциональность. Некоторые capabilities нужно добавлять обратно.

### Add specific capabilities

Можно добавить конкретные capabilities:
- CAP_NET_BIND_SERVICE — для веб-серверов
- CAP_SYS_TIME — для серверов времени
- CAP_NET_ADMIN — для фаерволов

### Безопасный минимум

Принцип: дать минимум необходимых capabilities. Чем меньше capabilities у процесса — тем меньше ущерма от взлома.

| Уровень безопасности | Capabilities |
|---------------------|-------------|
| Максимальный | Drop all |
| Высокий | Только необходимые |
| Средний | Стандартный набор |
| Низкий | Все capabilities |

## Сравнение с другими механизмами

### Capabilities vs root

| Критерий | Root | Capabilities |
|----------|------|--------------|
| Гранулярность | Всё или ничего | Мелкозернистый контроль |
| Безопасность | Низкая | Высокая |
| Сложность | Просто | Требует настройки |
| Отладка | Легко | Сложнее |

### Capabilities vs sudo

Sudo даёт полный root на время. Capabilities дают конкретные права постоянно.

Sudo: "Дай мне права администратора на 5 минут".
Capabilities: "У этого процесса есть право открывать порты ниже 1024".

### Capabilities vs AppArmor/SELinux

Capabilities говорят: "Что процесс может делать".
AppArmor/SELinux говорят: "К чему процесс может обращаться".

Это ортогональные механизмы. Они дополняют друг друга.

## Архитектурные решения

### Минимальные привилегии

Каждый процесс должен иметь минимум необходимых capabilities. Веб-серверу нужно CAP_NET_BIND_SERVICE. Не нужно CAP_SYS_ADMIN, CAP_SYS_MODULE, CAP_KILL.

### Контейнерная безопасность

Docker и Kubernetes позволяют управлять capabilities контейнеров:
- Drop all — убрать всё
- Add — добавить необходимые
- Контейнер работает без root, но имеет нужные привилегии

### Defense in Depth

Capabilities — это один из слоёв защиты. Они работают вместе с:
- Namespaces — изоляция видимости
- Seccomp — фильтрация системных вызовов
- AppArmor/SELinux — контроль доступа к файлам
- Cgroups — ограничение ресурсов

## Ограничения

### Не все привилегии разделены

Не всё можно сделать через capabilities. Некоторые операции всё ещё требуют полного root.

### Сложность отладки

Когда процесс не может сделать что-то из-за отсутствия capabilities — ошибка не всегда очевидна. Нужно понимать, какая capability требуется.

### Наследование

Некоторые capabilities сложно передавать дочерним процессам. Ambient capabilities решают часть проблем, но добавляют сложности.

### Совместимость

Старые программы могут не поддерживать capabilities. Они ожидают root и могут работать некорректно без полных привилегий.

### Ответы на чек-лист

1. **Capabilities** разбивают права root на отдельные возможности. Root — это "всё или ничего". Capabilities — мелкозернистый контроль: процесс может иметь только нужные ему привилегии.

2. **Четыре набора**: Effective — реально имеющиеся; Permitted — максимально возможные; Inheritable — передаваемые дочерним процессам; Bounding set — глобальный лимит системы.

3. **Bounding set** — это глобальный лимит capabilities для всей системы. Никакой процесс не может получить capabilities вне bounding set. Это верхняя граница безопасности.

4. **Capabilities исполняемому файлу** назначаются через специальные атрибуты файловой системы. Когда пользователь запускает файл, процесс получает эти capabilities. Это позволяет обычным пользователям выполнять привилегированные операции.

5. **Sudo** даёт полный root на время — все capabilities сразу. Если во время sudo процесс взломан — злоумышленник получает всё. **Capabilities** дают только конкретные права — поверхность атаки меньше.

6. **Ambient capabilities** — новый тип, который наследуется при exec без необходимости явно указывать их в файле. Решает проблему, когда capabilities терялись при запуске новых программ.

7. В контейнерах обычно используется **drop all capabilities** — убираются все привилегии. Затем **добавляются** только необходимые: CAP_NET_BIND_SERVICE для веб-серверов, CAP_NET_ADMIN для сетевых сервисов.

8. Веб-серверу обычно нужна только **CAP_NET_BIND_SERVICE** — возможность привязки к портам ниже 1024. Всё остальное (чтение файлов, запись логов) не требует capabilities — работает от обычного пользователя.

9. **Namespaces** изолируют видимость ресурсов. **Seccomp** фильтрует системные вызовы. **Capabilities** определяют, какие привилегированные операции процесс может выполнять. Вместе они создают многоуровневую защиту.

10. **Drop all capabilities** убирает все привилегии по умолчанию. Затем добавляются только необходимые. Это реализация принципа минимальных привилегий — чем меньше capabilities, тем меньше ущерма при взломе.
