# POSIX ACL: Расширенные права доступа

## Что такое POSIX ACL?

Представь офисное здание. В обычной системе безопасности у каждой комнаты есть владелец, группа сотрудников и все остальные. Владелец может всё, группа может почти всё, остальные — ничего.

Но реальная жизнь сложнее. Бухгалтерия должна иметь доступ к финансовым документам. HR — к кадровым. Руководители проектов — к документам своих проектов. Некоторые сотрудники работают в нескольких проектах одновременно.

**POSIX ACL** — это расширение стандартной модели прав Linux. Оно позволяет назначать права доступа конкретным пользователям и группам, независимо от владельца файла.

Вместо трёх классических категорий — владелец, группа, остальные — ACL позволяет создать список из множества записей. Каждая запись говорит: "Пользователь Иван имеет право читать и писать этот файл". Или: "Группа разработчиков может читать, но не писать".

## Зачем нужен ACL?

### Ситуация 1: Совместная работа над документами

HR-платформа генерирует отчёты. Несколько отделов должны иметь доступ к разным частям отчёта. Бухгалтерия — к финансовым данным. HR — к кадровым. Руководство — ко всему.

С классическими правами можно дать доступ всей группе. Но что, если бухгалтер Вася временно помогает HR-отделу? ACL позволяет дать Васе доступ к конкретным HR-файлам, не добавляя его в группу HR.

### Ситуация 2: Временный доступ

Аудитор приехал проверять документацию. Ему нужен доступ к файлам на две недели. С ACL можно дать доступ конкретному пользователю на конкретные файлы. Потом просто удалить запись.

### Ситуация 3: Сложные группы

В компании есть кросс-функциональные команды. Разработчик Иван работает в проекте А и проекте Б. В классической модели его нужно добавить в обе группы — но Linux позволяет только одну основную группу. ACL решает эту проблему.

## Стандартные права vs ACL

### Классическая модель

В Linux каждый файл имеет три набора прав:

| Категория | Обозначение | Права |
|-----------|-------------|-------|
| Владелец | u | rwx |
| Группа | g | rwx |
| Остальные | o | rwx |

Это просто, но ограничено. Можно дать доступ только одной группе. Если нужно дать доступ нескольким группам — проблема.

### ACL модель

ACL добавляет две категории записей:

| Тип записи | Описание |
|------------|----------|
| Named user | Конкретный пользователь |
| Named group | Конкретная группа |

Теперь можно создать такой список:
- Владелец: полный доступ
- Группа разработчиков: чтение и запись
- Группа тестировщиков: только чтение
- Пользователь Вася (аудитор): чтение на две недели
- Все остальные: нет доступа

### Маска

Маска ACL ограничивает максимальные права для named users и named groups. Даже если записи дают полный доступ — маска может его ограничить.

Это как если бы у двери был общий замок. У каждого сотрудника есть свой ключ, но замок может быть настроен так, что ключи работают только в определённое время.

## Типы записей ACL

### Access ACL

Access ACL — это основной список, который определяет, кто и как может обращаться к файлу или директории.

Состоит из записей:
- Владелец файла
- Named users — конкретные пользователи
- Владеющая группа
- Named groups — конкретные группы
- Маска
- Остальные

### Default ACL

Default ACL применяется к директориям. Он определяет, какие ACL получат файлы и поддиректории, созданные внутри этой директории.

Это как если бы в конференц-зале было правило: "Все документы, созданные в этом зале, доступны всем участникам встречи".

Default ACL наследуется. Если в директории /projects есть default ACL, то файл /projects/report.txt получит ACL от /projects.

### Наследование

Default ACL работает так:
- Поддиректории наследуют default ACL как свой default ACL
- Файлы получают access ACL из default ACL директории

Но есть нюанс: если файл создаётся в поддиректории, которая уже имеет свой default ACL — используется default ACL поддиректории, а не родительской.

## Права доступа

### Базовые права

| Право | Файл | Директория |
|-------|------|------------|
| Чтение (r) | Читать содержимое | Список файлов |
| Запись (w) | Изменять содержимое | Создавать/удалять файлы |
| Исполнение (x) | Запускать как программу | Заходить в директорию |

### Дополнительные права

| Право | Описание |
|-------|----------|
| Setuid | Запускать файл с правами владельца |
| Setgid | Запускать файл с правами группы / наследовать группу |
| Sticky bit | В директории — удалять только свои файлы |

Setgid на директории особенно полезен с ACL. Новые файлы в директории наследуют группу директории, а не группу создателя.

## ACL и безопасность

### Преимущества

**Гибкость**

ACL позволяет создавать сложные схемы доступа. Можно дать доступ нескольким группам, конкретным пользователям, ограничить по времени.

**Совместимость**

ACL работает поверх стандартных прав. Если программа не поддерживает ACL — она видит только стандартные права. ACL не ломает старое ПО.

**Аудит**

Можно посмотреть, кто имеет доступ к файлу. Полный список записей ACL показывает всех, кто может обращаться к файлу.

### Риски

**Сложность**

Чем больше записей ACL, тем сложнее понять, кто реально имеет доступ. Можно случайно дать доступ не тому пользователю.

**Backup и restore**

При копировании файлов ACL может теряться или изменяться. Не все утилиты копирования сохраняют ACL по умолчанию.

**Производительность**

Проверка ACL требует больше операций, чем проверка стандартных прав. При большом количестве записей это может замедлить доступ.

## ACL и другие механизмы

### ACL + SELinux

ACL контролирует дискреционный доступ — кто может что делать. SELinux контролирует мандатный доступ — типы и домены.

Они работают последовательно. Сначала проверяется SELinux, потом ACL, потом стандартные права.

### ACL + AppArmor

AppArmor работает на уровне путей и профилей. ACL работает на уровне пользователей и групп. Они решают разные задачи.

AppArmor говорит: "Веб-сервер может читать файлы в /var/www". ACL говорит: "Пользователь Вася может читать этот конкретный файл".

### ACL + файловые системы

ACL поддерживается большинством современных файловых систем: ext4, XFS, Btrfs, ZFS. Но нужно включить поддержку при монтировании.

Для резервного копирования важно использовать утилиты, которые сохраняют ACL. Иначе при восстановлении права будут потеряны.

## Архитектурные решения

### Модель доступа к документам

Для HR-платформы можно построить модель доступа:

**Уровень 1: Роль**
HR-менеджер, бухгалтер, руководитель, аудитор.

**Уровень 2: Проект**
Проект А, проект Б, общие документы.

**Уровень 3: Временной доступ**
Временный сотрудник, консультант, аудитор.

ACL позволяет реализовать все три уровня. Группы ролей дают базовый доступ. Named users дают доступ к конкретным проектам. Временные записи дают ограниченный доступ.

### Defense in Depth

ACL — это один из слоёв защиты. Он работает на уровне файловой системы, контролируя, кто может обращаться к файлам.

Другие слои:
- Аутентификация — проверяет, кто пользователь
- Авторизация — проверяет права на уровне приложения
- ACL — проверяет права на уровне файловой системы
- Шифрование — защищает данные при хранении

### Наследование прав

Default ACL позволяет создать шаблоны прав для директорий. Например:
- /projects/ — все новые проекты наследуют доступ для руководителей
- /shared/ — все новые файлы доступны всем сотрудникам
- /confidential/ — все новые файлы имеют ограниченный доступ

## Ограничения

### Не замена RBAC

ACL работает на уровне файлов. Для управления доступом в приложениях лучше использовать RBAC — ролевое управление доступом.

### Нет временных ограничений

ACL не поддерживает временные ограничения natively. Нельзя сказать: "дать доступ на две недели". Нужно внешними средствами удалять записи.

### Нет аудита изменений

ACL не логирует, кто и когда изменил права. Для аудита нужны дополнительные механизмы.

### Переносимость

При копировании файлов между системами ACL может теряться. Не все файловые системы поддерживают ACL одинаково.

## Чек-лист понимания

- [ ] Чем ACL отличается от стандартных прав Linux?
- [ ] Что такое named user и named group?
- [ ] Как работает default ACL?
- [ ] Что такое маска ACL?
- [ ] Как наследуются ACL при создании файлов?
- [ ] Какие права действуют на директории?
- [ ] Чем ACL отличается от SELinux?
- [ ] Какие риски связаны с ACL?
- [ ] Как ACL работает с резервным копированием?
- [ ] Когда использовать ACL, а когда RBAC?

### Ответы на чек-лист

1. **ACL** позволяет назначать права множеству пользователей и групп, а не только владельцу, одной группе и остальным. Это гибкое расширение классической модели.

2. **Named user** — конкретный пользователь с конкретными правами. **Named group** — конкретная группа с конкретными правами. Они добавляются к базовым записям владельца и группы.

3. **Default ACL** применяется к директориям и определяет, какие ACL получат файлы и поддиректории, созданные внутри. Поддиректории наследуют default ACL как свой default ACL.

4. **Маска ACL** ограничивает максимальные права для named users и named groups. Даже если записи дают полный доступ — маска может его ограничить.

5. **Файлы** получают access ACL из default ACL родительской директории. **Поддиректории** наследуют default ACL родительской директории как свой default ACL. Но если поддиректория уже имеет свой default ACL — используется он.

6. На директориях: **чтение** — список файлов, **запись** — создание/удаление файлов, **исполнение** — заход в директорию.

7. **ACL** — дискреционный контроль, решает кто может что делать с файлом. **SELinux** — мандатный контроль, решает на основе типов и доменов. Они работают последовательно.

8. **Риски ACL**: сложность при большом количестве записей, потеря при backup/restore, возможные ошибки доступа, производительность при множестве записей.

9. **Не все утилиты копирования сохраняют ACL**. Важно использовать специальные опции или утилиты, поддерживающие ACL. При восстановлении из backup ACL может быть потерян.

10. **ACL** — для файловой системы, гибкого управления правами на файлы. **RBAC** — для приложений, управления доступом на уровне бизнес-логики. ACL не заменяет RBAC в приложениях.
