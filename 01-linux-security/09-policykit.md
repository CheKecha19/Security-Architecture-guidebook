# PolicyKit: Авторизация действий

## Что такое PolicyKit?

Представь большой офисный центр с множеством комнат. Есть общие зоны — кухня, переговорная, лаундж. Есть специальные зоны — серверная, финансовый отдел, архив. И есть зоны, куда пускают только по записи — комната управления зданием, сейфовая, комната видеонаблюдения.

В каждой зоне свои правила. На кухню может зайти любой сотрудник. В серверную — только администраторы. В сейфовую — только финансовый директор и главный бухгалтер.

Но иногда нужно дать временный доступ. Например, уборщице нужно зайти в серверную, чтобы убрать. Или аудитору — в финансовый отдел. Или специалисту по пожарной безопасности — в комнату управления зданием.

**PolicyKit** — это система, которая решает, кто может выполнить конкретное действие. Не просто войти в систему — а выполнить действие. Перезагрузить сервер? Изменить сетевые настройки? Установить обновления? Каждое действие требует своего уровня доступа.

PolicyKit работает по принципу: "У тебя есть права на это действие?" Вместо того чтобы давать пользователю root на всё — PolicyKit даёт права на конкретные действия.

## Зачем нужен PolicyKit?

### Ситуация 1: Графический интерфейс

Пользователь хочет подключиться к Wi-Fi. Для этого нужно изменить сетевые настройки — операция, требующая привилегий.

PolicyKit спрашивает: "Вы уверены, что хотите подключиться к сети?" Если пользователь подтверждает — действие выполняется с временным повышением прав.

Пользователь не получает root. Он получает право выполнить одно конкретное действие.

### Ситуация 2: Системные обновления

Обычный пользователь хочет обновить систему. PolicyKit проверяет: есть ли у пользователя право устанавливать обновления? Если да — запрашивает подтверждение и выполняет.

### Ситуация 3: Управление пользователями

HR-администратор создаёт учётную запись нового сотрудника. PolicyKit проверяет: есть ли у администратора право создавать пользователей? Если да — операция выполняется.

## Как работает PolicyKit?

### Действия

PolicyKit управляет не пользователями, а действиями. Каждое действие имеет уникальный идентификатор.

Примеры действий:
- Подключение к сети
- Изменение системного времени
- Установка обновлений
- Перезагрузка системы
- Управление пользователями
- Настройка принтера

### Агенты

Агент PolicyKit — это программа, которая спрашивает пользователя. Когда действие требует подтверждения — агент показывает диалог.

Агенты бывают разные:
- Графический — показывает окно с запросом пароля
- Текстовый — запрашивает в консоли
- Автоматический — не спрашивает, если правила разрешают

### Правила

Правила PolicyKit определяют, кто может выполнять действия. Правила проверяют:
- Является ли пользователь администратором?
- Находится ли пользователь в нужной группе?
- Активна ли сессия пользователя?
- Прошло ли достаточно времени с последнего подтверждения?

### Результат проверки

PolicyKit может вернуть один из результатов:

| Результат | Описание |
|-----------|----------|
| yes | Действие разрешено |
| no | Действие запрещено |
| auth_self | Нужно подтвердить свою личность |
| auth_admin | Нужны права администратора |
| auth_self_keep | Подтверждение запоминается на время |
| auth_admin_keep | Права администратора запоминаются |

## Сравнение с sudo

| Критерий | PolicyKit | sudo |
|----------|-----------|------|
| Что контролирует | Конкретные действия | Выполнение команд от root |
| Гранулярность | Высокая | Низкая |
| Интеграция с GUI | Отличная | Ограниченная |
| Временные права | Да | Да, но на всё |
| Запоминание | Да, для действия | Да, для времени |
| Логирование | Да | Да |

**sudo** говорит: "Дай мне права суперпользователя на 5 минут".
**PolicyKit** говорит: "Разреши мне перезагрузить систему".

## PolicyKit и безопасность

### Преимущества

**Минимальные привилегии**

PolicyKit даёт права только на конкретное действие. Пользователь не получает root. Он получает возможность выполнить одну операцию.

**Гибкость**

Разные действия — разные требования. Подключение к Wi-Fi требует простого подтверждения. Установка обновлений — прав администратора.

**Интеграция**

PolicyKit интегрирован в графические окружения. Пользователь видит понятные диалоги, а не консольные запросы.

### Риски

**Сложность конфигурации**

Правила PolicyKit могут быть сложными. Ошибка в правиле может разрешить слишком много.

**Зависимость от сессии**

PolicyKit привязан к пользовательской сессии. Вне сессии — другие правила.

**Обход**

Если PolicyKit не настроен — программа может просто запросить sudo. Это обходит защиту PolicyKit.

## Архитектурные решения

### Разделение обязанностей

PolicyKit позволяет разделить обязанности:
- Обычные пользователи — повседневные действия
- Опытные пользователи — настройка системы
- Администраторы — критичные операции

Каждый уровень имеет свои действия, не требующие полного root.

### Защита от случайных ошибок

Когда пользователь пытается сделать что-то важное — PolicyKit просит подтвердить. Это защищает от случайных нажатий.

### Аудит

Все действия через PolicyKit логируются. Можно посмотреть, кто и когда выполнял какие действия.

## Ограничения

### Только авторизация

PolicyKit управляет тем, КТО может выполнить действие. Он не управляет тем, КАК действие выполняется. Это задача программы, выполняющей действие.

### Зависимость от окружения

PolicyKit работает в графических окружениях. В чистой консоли его нет. Там используется sudo.

### Не замена RBAC

PolicyKit — это не полноценная ролевая модель. Он управляет отдельными действиями, а не ролями пользователей.

## Чек-лист понимания

- [ ] Чем PolicyKit отличается от sudo?
- [ ] Что такое действие в PolicyKit?
- [ ] Какие результаты может вернуть PolicyKit?
- [ ] Что делает агент PolicyKit?
- [ ] Как PolicyKit запрашивает подтверждение?
- [ ] Почему PolicyKit безопаснее sudo для GUI?
- [ ] Как PolicyKit интегрируется с системой?
- [ ] Какие риски связаны с PolicyKit?
- [ ] Как PolicyKit дополняет RBAC?
- [ ] В чём разница между auth_self и auth_admin?

### Ответы на чек-лист

1. **PolicyKit** управляет конкретными действиями — пользователь получает право на одну операцию. **sudo** даёт полный root на время — все операции. PolicyKit гранулярнее и безопаснее для GUI.

2. **Действие** — это операция, которую пользователь хочет выполнить. Каждое действие имеет уникальный идентификатор. Например, подключение к сети, перезагрузка системы, установка обновлений.

3. **Результаты PolicyKit**: yes — действие разрешено; no — запрещено; auth_self — нужно подтвердить свою личность; auth_admin — нужны права администратора; auth_self_keep и auth_admin_keep — подтверждение запоминается.

4. **Агент PolicyKit** — это программа, которая взаимодействует с пользователем. Показывает диалог с запросом пароля, текста подтверждения, или автоматически разрешает действие.

5. **PolicyKit запрашивает подтверждение** через агента. Агент показывает диалог, пользователь вводит пароль или нажимает подтвердить. PolicyKit проверяет права и решает, выполнять ли действие.

6. **PolicyKit безопаснее sudo для GUI**, потому что он даёт права только на конкретное действие. Пользователь не получает root на всё. Он подтверждает одну операцию. sudo же даёт полный root на время.

7. **PolicyKit интегрирован** в графические окружения Linux. Он работает с сессией пользователя, знает, кто вошёл в систему, и применяет правила на основе сессии.

8. **Риски PolicyKit**: сложность конфигурации, привязка к сессии, возможность обхода через sudo, ошибки в правилах. Нужно тщательно настраивать правила.

9. **PolicyKit** управляет отдельными действиями. **RBAC** управляет ролями и их правами. PolicyKit может работать в рамках RBAC: роль определяет, какие действия разрешены, а PolicyKit — запрашивает подтверждение.

10. **auth_self** требует подтверждения от текущего пользователя — просто ввести свой пароль. **auth_admin** требует прав администратора — ввести пароль администратора или быть в группе администраторов.
