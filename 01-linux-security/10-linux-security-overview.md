# Обзор механизмов безопасности Linux

## Введение

Linux — это операционная система, на которой работает большинство серверов мира. От поисковых систем до банковских платформ, от облачных сервисов до HR-систем. Безопасность Linux — это фундамент безопасности всей инфраструктуры.

Но Linux не защищён по умолчанию. Стандартная установка даёт широкие возможности, которые можно злоупотребить. Поэтому в Linux существует множество механизмов безопасности, каждый из которых решает свою задачу.

Этот обзор объединяет все механизмы, которые мы рассмотрели, и показывает, как они работают вместе.

## Архитектура защиты Linux

### Многоуровневая защита

Безопасность Linux строится по принципу многоуровневой защиты. Если один уровень пробит — есть следующий. Это как замок, шпингалет, дверной цепочка и домофон — каждый уровень добавляет защиту.

| Уровень | Механизм | Что защищает |
|---------|----------|--------------|
| **Пользователь** | Аутентификация | Кто входит в систему |
| **Сессия** | PAM | Как проверяется пользователь |
| **Процесс** | Capabilities | Что процесс может делать |
| **Изоляция** | Namespaces | Что процесс видит |
| **Файлы** | DAC, ACL, SELinux/AppArmor | Кто имеет доступ к файлам |
| **Системные вызовы** | Seccomp | Какие вызовы разрешены |
| **Ресурсы** | cgroups | Сколько ресурсов можно использовать |
| **Сеть** | Netfilter, SELinux | Какие сетевые операции разрешены |

### Каждый уровень в деталях

**Уровень пользователя**

Прежде чем что-то делать в системе, нужно войти. Аутентификация проверяет, кто ты. Это могут быть пароли, ключи, биометрия, одноразовые коды.

PAM объединяет все способы аутентификации в единую систему. Он позволяет комбинировать факторы: пароль и код с телефона, ключ и биометрия.

**Уровень процесса**

После входа пользователь запускает программы. Каждая программа — это процесс. У процесса есть права.

Capabilities разбивают права root на отдельные возможности. Процесс может иметь только то, что ему нужно. Веб-серверу нужно открыть порт 80 — ему дают CAP_NET_BIND_SERVICE. Не нужен полный root.

**Уровень изоляции**

Namespaces создают иллюзию отдельной системы. Процесс внутри namespace видит только своё. Он не видит процессы других, не видит сетевые интерфейсы хоста, не видит файлы за пределами своего пространства.

User namespace особенно важен. Он позволяет процессу быть root внутри контейнера, но иметь обычные права на хосте. Даже если контейнер взломан — злоумышленник не выходит за пределы.

**Уровень файлов**

Файлы — это данные. Данные нужно защищать.

Стандартные права Linux (DAC) — первый уровень. Владелец, группа, остальные. Но этого мало.

POSIX ACL добавляет гибкости. Можно дать доступ конкретным пользователям и группам, независимо от владельца.

AppArmor и SELinux поднимают защиту на новый уровень. Они решают, к каким файлам может обращаться процесс, независимо от прав пользователя.

AppArmor использует пути. SELinux использует метки. Оба механизма не дают взломанному процессу выйти за пределы разрешённых файлов.

**Уровень системных вызовов**

Процессы обращаются к ядру через системные вызовы. Это окно в мир привилегий.

Seccomp фильтрует системные вызовы. Он говорит: "Эти вызовы разрешены, остальные — нет". Docker по умолчанию блокирует около сорока опасных вызовов.

Landlock добавляет ещё один фильтр. Он контролирует доступ к файлам на уровне путей. Процесс сам устанавливает себе границы.

**Уровень ресурсов**

Даже изолированный процесс может исчерпать ресурсы. Забить всю память. Занять все процессоры. Заполнить диск.

cgroups ограничивают использование ресурсов. Можно установить лимит на память, процессор, диск, сеть. Если процесс превышает лимит — ядро его останавливает.

## Взаимодействие механизмов

### Сценарий: Веб-сервер в контейнере

Рассмотрим, как работают механизмы вместе на примере веб-сервера в контейнере.

**Аутентификация**

Администратор входит в систему через SSH. PAM проверяет его ключ и пароль. Двухфакторная аутентификация подтверждает личность.

**Запуск контейнера**

Администратор запускает Docker-контейнер с веб-сервером. Docker создаёт namespaces: PID, NET, MNT, USER. Контейнер видит только свои процессы, свою сеть, свою файловую систему.

**Права процесса**

Веб-сервер внутри контейнера запускается от пользователя nginx. Его UID на хосте — обычный пользователь, но внутри контейнера он может иметь UID 0 (root) благодаря user namespace.

У процесса ограниченные capabilities. Docker убрал большинство capabilities, оставив только необходимые: CAP_NET_BIND_SERVICE для порта 80.

**Доступ к файлам**

Веб-сервер читает файлы из /var/www. Файлы имеют тип httpd_sys_content_t в SELinux. Политика SELinux разрешает процессам в домене httpd_t читать файлы этого типа.

Если злоумышленник взламывает веб-сервер и пытается прочитать /etc/shadow — SELinux блокирует. Тип файла /etc/shadow — shadow_t, а политика не разрешает httpd_t доступ к shadow_t.

**Системные вызовы**

Seccomp профиль Docker блокирует опасные вызовы. Злоумышленник не может загрузить модуль ядра, изменить системное время, смонтировать файловую систему.

**Ресурсы**

cgroups ограничивают контейнер: 512 МБ памяти, 1 CPU core, 10 МБ/с сети. Даже если веб-сервер попытается занять больше — ядро ограничит его.

### Сценарий: Уязвимость в приложении

Представь, что в веб-приложении нашли уязвимость, позволяющую выполнить произвольный код.

**Без защиты**

Злоумышленник получает контроль над веб-сервером. Он может читать файлы системы, модифицировать базу данных, устанавливать бэкдоры. Он пытается получить root.

**С защитой**

Namespaces изолируют процесс. Он не видит процессы хоста и других контейнеров. User namespace означает, что root внутри контейнера — обычный пользователь на хосте.

Capabilities ограничивают привилегии. У процесса нет CAP_SYS_ADMIN, CAP_SYS_MODULE, CAP_SETUID. Он не может получить root на хосте.

Seccomp блокирует опасные системные вызовы. Злоумышленник не может загрузить модуль ядра или изменить системное время.

SELinux/AppArmor блокируют доступ к файлам. Процесс не может читать /etc/shadow, /root/.ssh/id_rsa, или модифицировать системные бинарники.

cgroups ограничивают ресурсы. Процесс не может исчерпать память или процессор.

Результат: злоумышленник заперт в контейнере. Ущерб минимален.

## Принципы построения безопасной системы

### Принцип минимальных привилегий

Каждый процесс должен иметь минимум необходимых прав. Не нужно давать root, если достаточно одной capability. Не нужно давать доступ к всей файловой системе, если достаточно одной папки.

### Defense in Depth

Безопасность не должна зависеть от одного механизма. Если SELinux отключён — есть AppArmor. Если AppArmor не настроен — есть DAC и ACL. Если всё это не работает — есть seccomp и namespaces.

### Fail Secure

При ошибке система должна переходить в безопасное состояние. Если SELinux не может применить политику — он запрещает доступ, а не разрешает. Если seccomp не может загрузить фильтр — он блокирует вызовы.

### Изоляция

Каждый компонент должен быть изолирован. Веб-сервер от базы данных. Почтовый сервер от DNS. Пользовательские процессы от системных.

Namespaces, cgroups, SELinux/AppArmor создают слои изоляции. Чем больше слоёв — тем лучше.

## Типичные ошибки конфигурации

### Отключение SELinux

Вместо настройки политик под приложение администраторы часто отключают SELinux. Это устраняет проблему, но снижает безопасность. Лучше настроить политику или перевести в permissive для отладки.

### Запуск от root

Многие сервисы запускаются от root без необходимости. Веб-сервер, почтовый сервер, DNS — им нужны ограниченные capabilities, а не полный root.

### Отсутствие namespaces

Контейнеры без user namespace опасны. Если процесс в контейнере получает root — он имеет root на хосте. User namespace решает эту проблему.

### Пропуск seccomp

Запуск контейнеров без seccomp профиля увеличивает поверхность атаки. Стандартный Docker профиль блокирует множество опасных вызовов.

### Широкие права на файлы

Файлы с правами 777, директории с открытым доступом для всех — типичная ошибка. Минимальные права, ACL для точного контроля, SELinux/AppArmor для мандатного контроля.

## Выбор механизмов

### Для сервера

| Задача | Рекомендуемые механизмы |
|--------|------------------------|
| Аутентификация | PAM с многофакторной аутентификацией |
| Изоляция сервисов | Namespaces + cgroups |
| Контроль доступа | SELinux (RHEL) или AppArmor (Ubuntu) |
| Фильтрация вызовов | Seccomp |
| Ограничение прав | Capabilities |
| Управление файлами | DAC + ACL |

### Для контейнеров

| Задача | Рекомендуемые механизмы |
|--------|------------------------|
| Изоляция | Namespaces (все типы) |
| Безопасность ядра | Seccomp + Capabilities drop all |
| Файлы | Read-only rootfs + tmpfs |
| Сеть | Network policies |
| Ресурсы | cgroups limits |

### Для рабочей станции

| Задача | Рекомендуемые механизмы |
|--------|------------------------|
| Аутентификация | PAM + биометрия + ключи |
| GUI действия | PolicyKit |
| Приложения | AppArmor profiles |
| Браузер | Sandboxing (namespaces, seccomp) |

## Будущее безопасности Linux

### Landlock

Landlock — относительно новый механизм. Он позволяет приложениям самим устанавливать границы. Не нужен root, не нужна глобальная политика. Приложение говорит: "Я работаю только с этими файлами".

Ожидается расширение Landlock на сеть и другие ресурсы. Это сделает песочницы ещё более доступными.

### eBPF

eBPF — технология для запуска программ в ядре. Она используется для мониторинга, сетевой фильтрации, и безопасности.

eBPF может заменить или дополнить seccomp, предоставляя более гибкую фильтрацию системных вызовов и сетевого трафика.

### Confidential Computing

Новые процессоры поддерживают защищённые анклавы — изолированные области памяти, которые даже ядро не может прочитать. Это добавляет ещё один уровень защиты для чувствительных данных.

### Ответы на чек-лист

1. **Уровни защиты Linux**: аутентификация (PAM), процесс (capabilities), изоляция (namespaces), файлы (DAC, ACL, SELinux/AppArmor), системные вызовы (seccomp, Landlock), ресурсы (cgroups), сеть (netfilter, SELinux).

2. **PAM** объединяет способы аутентификации в единую систему. Программы делегируют проверку пользователя PAM, который использует настраиваемые модули. Это позволяет комбинировать пароли, ключи, биометрию, одноразовые коды.

3. **Capabilities** разбивают права root на отдельные возможности. Вместо "всё или ничего" процесс получает только нужные ему привилегии. Это снижает ущерб при компрометации процесса.

4. **Namespaces** создают изолированные представления системных ресурсов. PID namespace изолирует процессы, NET — сеть, MNT — файловую систему, USER — пользователей. Контейнеры строятся на namespaces.

5. **DAC** — стандартные права Linux (владелец, группа, остальные). **ACL** — расширение DAC с произвольными пользователями и группами. **SELinux** — мандатный контроль через метки (label-based). **AppArmor** — мандатный контроль через пути (path-based).

6. **Seccomp** фильтрует системные вызовы, которые процесс может делать к ядру. Он блокирует опасные вызовы, даже если процесс имеет необходимые привилегии. Docker по умолчанию использует seccomp профиль.

7. **cgroups** ограничивают использование физических ресурсов: CPU, память, диск, сеть. Они предотвращают исчерпание ресурсов одним процессом и обеспечивают справедливое распределение.

8. **Защита контейнера** использует namespaces для изоляции, user namespace для снижения привилегий, seccomp для фильтрации вызовов, capabilities для ограничения прав, AppArmor/SELinux для контроля файлов, cgroups для лимитов ресурсов.

9. **Принцип минимальных привилегий** требует давать процессу ровно столько прав, сколько ему нужно для работы. Не root, а нужные capabilities. Не полный доступ, а конкретные файлы.

10. **Выбор механизмов**: для серверов — SELinux/AppArmor + namespaces + seccomp + cgroups. Для контейнеров — все namespaces + seccomp + drop capabilities. Для рабочих станций — AppArmor + PAM с MFA + PolicyKit.
