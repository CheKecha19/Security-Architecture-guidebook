# PAM: Модульная аутентификация

## Что такое PAM?

Представь здание с множеством входов. Главный вход — пропускной пункт с охраной. Технический вход — замок с ключом. Грузовой — домофон. Подземная парковка — шлагбаум с пультом.

Каждый вход имеет свой способ проверки: кто может войти. Но за всеми этими способами стоит одна и та же система — база данных сотрудников. Охрана сверяет пропуск, замок проверяет ключ, домофон — код, шлагбаум — частоту пульта.

**PAM** — это система, которая позволяет приложениям выбирать способ аутентификации. Вместо того чтобы каждая программа сама проверяла логин и пароль — PAM берёт это на себя. Программа говорит PAM: "Проверь этого пользователя". PAM смотрит в конфигурацию и решает, как проверять.

Конфигурация PAM говорит: "Для входа через SSH сначала проверь пароль, потом одноразовый код, потом биометрию". Или: "Для входа на сервер используй Kerberos". Или: "Для входа в систему достаточно смарт-карты".

## Зачем нужен PAM?

### Ситуация 1: Многофакторная аутентификация

HR-платформа требует надёжной защиты учётных записей. Пароль — это один фактор. Но пароли можно украсть, подобрать, подсмотреть.

PAM позволяет настроить многофакторную аутентификацию. При входе пользователь должен предоставить:
- Что-то, что он знает (пароль)
- Что-то, что у него есть (телефон с кодом)
- Что-то, что он есть (отпечаток пальца)

PAM объединяет все эти проверки в одну цепочку.

### Ситуация 2: Разные способы для разных сервисов

Система имеет несколько точек входа: SSH для администраторов, веб-интерфейс для HR, VPN для удалённых сотрудников.

PAM позволяет настроить разные способы аутентификации для разных сервисов. Для SSH — ключи и пароль. Для веба — LDAP. Для VPN — сертификаты.

### Ситуация 3: Централизованное управление

Компания имеет центральный сервер аутентификации. Все системы должны обращаться к нему.

PAM позволяет настроить единый способ проверки для всех сервисов. Изменение политики в одном месте применяется ко всем.

## Как работает PAM?

### Типы модулей

PAM разделяет аутентификацию на четыре типа операций:

| Тип | Описание | Когда используется |
|-----|----------|-------------------|
| **auth** | Проверка личности | Вход в систему, sudo |
| **account** | Проверка доступа | Проверка срока действия аккаунта |
| **password** | Изменение пароля | Смена пароля пользователем |
| **session** | Управление сессией | Открытие/закрытие сессии |

### Типы управления

Каждая запись в конфигурации PAM имеет тип управления, который определяет, что делать при успехе или неудаче:

| Тип | Описание | Поведение |
|-----|----------|-----------|
| **required** | Обязательно | При неудаче продолжает проверку, но в итоге откажет |
| **requisite** | Критично | При неудаче немедленно отказывает |
| **sufficient** | Достаточно | При успехе сразу разрешает (если ранее не было отказа) |
| **optional** | Опционально | Не влияет на результат, если единственный модуль |
| **include** | Включение | Включает другой файл конфигурации |

### Цепочка модулей

Конфигурация PAM — это список модулей, которые вызываются последовательно. Каждый модуль проверяет что-то своё.

Например, цепочка для входа:
1. Модуль pam_securetty.so — проверяет, можно ли входить с этого терминала
2. Модуль pam_nologin.so — проверяет файл /etc/nologin
3. Модуль pam_unix.so — проверяет локальный пароль
4. Модуль pam_google_authenticator.so — проверяет одноразовый код
5. Модуль pam_limits.so — применяет ограничения ресурсов

Модули вызываются по порядку. Результат зависит от типов управления.

### Модули аутентификации

| Модуль | Описание |
|--------|----------|
| pam_unix | Локальные пароли из /etc/shadow |
| pam_ldap | Аутентификация через LDAP |
| pam_krb5 | Аутентификация через Kerberos |
| pam_google_authenticator | Одноразовые коды TOTP |
| pam_fprintd | Биометрия (отпечатки пальцев) |
| pam_ssh | Аутентификация по SSH-ключам |
| pam_yubico | Аппаратные ключи YubiKey |
| pam_sss | SSSD (System Security Services Daemon) |

### Модули авторизации

| Модуль | Описание |
|--------|----------|
| pam_time | Ограничение по времени |
| pam_access | Ограничение по хосту/терминалу |
| pam_nologin | Блокировка входа |
| pam_succeed_if | Условная проверка |
| pam_listfile | Проверка списка пользователей |

### Модули сессии

| Модуль | Описание |
|--------|----------|
| pam_limits | Ограничение ресурсов (ulimit) |
| pam_env | Установка переменных окружения |
| pam_mail | Уведомление о почте |
| pam_lastlog | Запись времени входа |
| pam_motd | Показ сообщения дня |

## Многофакторная аутентификация

### Первый фактор: знание

Пароль — классический фактор. Пользователь должен знать секретную строку.

Проблемы: пароли можно подобрать, украсть, забыть.

### Второй фактор: владение

Одноразовые коды на телефоне, аппаратные ключи, смарт-карты. Пользователь должен иметь физический объект.

Реализация через PAM: модуль проверяет код с устройства или вызывает внешний сервис.

### Третий фактор: биометрия

Отпечатки пальцев, распознавание лица, сканирование радужки. Пользователь — это сам фактор.

Реализация через PAM: модуль обращается к датчику и сравнивает биометрические данные.

### Комбинирование факторов

PAM позволяет комбинировать факторы гибко:

**Вариант 1: Обязательные последовательно**

Пароль И одноразовый код И биометрия. Все три должны пройти успешно.

**Вариант 2: Альтернативы**

Пароль ИЛИ ключевая карта. Достаточно одного.

**Вариант 3: Комбинированный**

Паробязательно. Плюс — либо код с телефона, либо биометрия.

## PAM и безопасность

### Преимущества

**Модульность**

Можно добавлять и удалять способы аутентификации без изменения приложений. Нужно добавить биометрию — просто добавляем модуль в конфигурацию.

**Гибкость**

Разные сервисы могут иметь разные требования. SSH требует ключ и пароль. Веб — только пароль. Админка — пароль и код.

**Централизация**

Вся политика аутентификации в одном месте. Не нужно менять каждое приложение отдельно.

### Риски

**Сложность конфигурации**

Ошибка в конфигурации PAM может заблокировать вход в систему. Неправильный порядок модулей или неправильные типы управления — система становится недоступна.

**Зависимость от внешних сервисов**

Если PAM использует LDAP или Kerberos — отказ этих сервисов блокирует вход. Нужны запасные способы.

**Локальная уязвимость**

Модули PAM работают с высокими привилегиями. Уязвимость в модуле — это уязвимость в системе аутентификации.

## Архитектурные решения

### Защита сервера

Для критичных серверов:
- Обязательная многофакторная аутентификация
- Ограничение по времени входа
- Ограничение по IP-адресам
- Уведомление о входе

### Резервные способы

Всегда иметь запасной способ входа. Если основной сервер аутентификации недоступен — должна быть локальная учётная запись.

### Аудит

Логировать все попытки аутентификации. Анализировать неудачные попытки. Обнаруживать атаки brute force.

## Ограничения

### Только аутентификация

PAM управляет тем, КТО может войти. Он не управляет тем, ЧТО пользователь может делать после входа. Это задача авторизации.

### Не шифрование

PAM не шифрует данные. Он только проверяет личность. Шифрование — задача других механизмов.

### Зависимость от реализации

Разные дистрибутивы имеют разные версии PAM и модулей. Конфигурация может не переноситься между системами.

### Ответы на чек-лист

1. **PAM** — это система модульной аутентификации. Она позволяет приложениям делегировать проверку личности PAM, а PAM использует настраиваемые модули для проверки. Программе не нужно знать, как именно проверять — PAM берёт это на себя.

2. **Четыре типа модулей**: auth — проверка личности; account — проверка доступа (срок действия, время); password — изменение пароля; session — управление сессией (ресурсы, окружение).

3. **Required** при неудаче продолжает проверку остальных модулей, но в итоге откажет. **Requisite** при неудаче немедленно останавливает проверку и отказывает. Requisite строже.

4. **Многофакторная аутентификация** реализуется цепочкой модулей auth. Сначала проверяется пароль (pam_unix), потом одноразовый код (pam_google_authenticator), потом биометрия (pam_fprintd). Тип управления required требует успеха всех модулей.

5. **pam_unix** — стандартный модуль для проверки локальных паролей. Он смотрит в /etc/shadow и проверяет хеш пароля.

6. **pam_limits** применяет ограничения ресурсов при открытии сессии. Устанавливает лимиты на память, процессор, количество процессов через файл конфигурации limits.conf.

7. **Ограничение по времени** делается модулем pam_time. Он проверяет, разрешён ли вход в текущее время суток и день недели. Настраивается через файл time.conf.

8. **Ошибка в конфигурации PAM** может сделать систему полностью недоступной. Если ни один модуль не может пройти — никто не сможет войти. Поэтому важно иметь физический доступ или аварийный режим.

9. **PAM** — это интерфейс, который может использовать разные бэкенды. **LDAP** и **Kerberos** — это бэкенды аутентификации. PAM модули pam_ldap и pam_krb5 обращаются к этим серверам для проверки личности.

10. **Риски PAM**: сложность конфигурации, зависимость от внешних сервисов, локальные уязвимости модулей, возможность блокировки системы. Нужно тестировать изменения и иметь запасные способы входа.
