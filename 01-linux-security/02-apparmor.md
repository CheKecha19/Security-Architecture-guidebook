# AppArmor: Защита через профили

## Что такое AppArmor?

Представь, что ты — директор театра. У тебя есть актёры, и каждый актёр должен играть только свою роль. Главный герой не должен лезть в гримёрку антагониста, статисты не должны трогать декорации, а костюмеры — не должны управлять светом.

**AppArmor** — это система, которая создаёт для каждой программы "роль" — профиль безопасности. Профиль говорит: "Эта программа может работать только с этими файлами и делать только эти действия". Всё остальное — запрещено.

В отличие от обычной модели безопасности Linux, где пользователь сам решает, кто имеет доступ к его файлам, AppArmor говорит от имени системы: "Я решаю, что эта программа может делать, независимо от прав пользователя".

## Зачем нужен AppArmor?

### Ситуация 1: Веб-сервер

Веб-сервер обрабатывает запросы из интернета. Если в нём есть уязвимость, злоумышленник может попытаться прочитать файлы, к которым сервер не должен иметь доступ.

AppArmor создаёт профиль: "Веб-сервер может читать файлы из /var/www, писать логи в /var/log/nginx, и больше ничего".

### Ситуация 2: Почтовый сервер

Почтовый сервер получает письма отовсюду. Вредоносное письмо может попытаться выполнить код. AppArmor не даст этому коду выйти за пределы разрешённых папок.

### Ситуация 3: Утилита резервного копирования

Программа бэкапа должна читать файлы для архивирования. Но она не должна иметь доступ к приватным ключам или паролям. AppArmor разделяет эти зоны.

## Основные концепции

### Профиль — это роль

Профиль AppArmor — это текстовый файл, описывающий, что программа может делать. Профиль содержит:

- Путь к программе, которую он защищает
- Список файлов и папок с правами доступа
- Сетевые возможности
- Способности (capabilities)
- Правила для дочерних процессов

Представь профиль как контракт с актёром: "Ты играешь гамлета, твои реплики — вот эти, твои выходы — вот тут, всё остальное — не твоё".

### Режимы работы

AppArmor поддерживает три режима для каждого профиля:

**Enforce — Принудительный**

Профиль активен и блокирует всё запрещённое. Если программа пытается сделать что-то вне профиля — действие блокируется, в журнал пишется запись.

Это как директор театра, который останавливает спектакль, если актёр выходит за рамки роли.

**Complain — Жалобный**

Профиль только логирует нарушения, но не блокирует. Полезен для разработки и тестирования профилей. Можно увидеть, что программа реально делает, и настроить профиль соответственно.

Это как репетиция, где директор делает заметки, но не останавливает актёра.

**Disabled — Отключён**

Профиль не применяется. Программа работает без ограничений.

Это как если бы актёр мог делать что угодно — играть все роли, лезть на сцену в любое время.

### Path-based подход

AppArmor определяет доступ по путям файловой системы. Он не использует метки безопасности, как SELinux. Вместо этого он говорит: "Программа может читать /etc/nginx/nginx.conf, но не может читать /etc/shadow".

Преимущество такого подхода — простота. Не нужно разбираться в метках, контекстах, политиках. Достаточно знать пути к файлам.

Недостаток — зависимость от путей. Если программа перемещается или использует символические ссылки, профиль может работать некорректно.

### Права доступа к файлам

AppArmor использует сокращённые обозначения прав:

| Право | Описание |
|-------|----------|
| r | Чтение |
| w | Запись |
| a | Дозапись (append) |
| x | Выполнение |
| k | Блокировка |
| l | Ссылка |
| m | Память |
| px | Выполнение с профилем |
| cx | Выполнение с переходом |
| Cx | Выполнение с переходом и очисткой |

Права можно комбинировать: "rw" — чтение и запись, "rwx" — полный доступ.

### Специальные правила

**Выполнение с профилем (px)**

Когда программа запускает другую программу, AppArmor может переключиться на профиль этой другой программы. Это важно для сервисов, запускающих дочерние процессы.

**Выполнение с переходом (cx, Cx)**

Переход на профиль нижнего уровня. Разница между cx и Cx — в том, передаётся ли окружение. Cx очищает окружение, делая переход более безопасным.

## Как создать профиль?

### Автоматическое создание

Существуют инструменты для автоматического создания шаблонов профилей. Они анализируют программу и создают базовый профиль на основе наблюдаемого поведения.

Это как если бы режиссёр записывал на видео репетицию актёра, а потом составлял для него роль на основе того, что он реально делал.

### Интерактивное профилирование

Инструмент интерактивного профилирования запускает программу в режиме complain и предлагает администратору принять или отклонить каждое наблюдаемое действие.

Администратор видит: "Программа пытается прочитать /etc/passwd. Разрешить?" — и принимает решение.

Это долгий процесс, но он даёт точный профиль, соответствующий реальному поведению программы.

### Ручное создание

Опытный администратор может написать профиль вручную, основываясь на знании архитектуры программы. Это быстрее, но требует глубокого понимания.

## AppArmor vs SELinux

Это два самых популярных механизма мандатного контроля доступа в Linux. Они решают одну задачу, но разными способами.

### Различия подходов

**AppArmor** использует пути:
- "Программа /usr/sbin/nginx может читать /etc/nginx/*"
- Просто и понятно
- Привязка к расположению файлов

**SELinux** использует метки:
- "Процессы с типом httpd_t могут читать файлы с типом httpd_sys_content_t"
- Мощно и гибко
- Независимость от расположения

### Когда что выбирать?

| Критерий | AppArmor | SELinux |
|----------|----------|---------|
| Сложность настройки | Низкая | Высокая |
| Гибкость политик | Средняя | Высокая |
| Защита от symlink attacks | Ниже | Выше |
| Поддержка дистрибутивов | Ubuntu, SUSE | RHEL, CentOS, Fedora |
| Время внедрения | Короткое | Долгое |

### Почему коммерческое ПО часто требует отключения SELinux?

Коммерческое ПО часто не рассчитано на SELinux. Оно пытается делать вещи, которые SELinux блокирует. Вместо настройки политик под SELinux, вендоры предлагают отключить SELinux.

AppArmor реже вызывает такие проблемы, потому что он менее строг и проще в настройке.

## Применение профилей

### Загрузка профилей

Профили хранятся в специальной директории. При загрузке системы AppArmor читает все профили и применяет их к соответствующим программам.

### Переключение режимов

Администратор может переключить профиль из enforce в complain и обратно без перезагрузки. Это полезно для отладки.

### Отключение профилей

Если профиль мешает работе программы, его можно отключить. Но это снижает безопасность.

### Статус системы

Команда для просмотра статуса AppArmor показывает:
- Сколько профилей загружено
- Сколько в режиме enforce
- Сколько в режиме complain
- Какие процессы ограничены

## Логирование и аудит

### Журнал нарушений

Когда программа пытается сделать что-то запрещённое профилем, AppArmor записывает это в журнал. Запись содержит:
- Время события
- Имя программы
- Тип нарушения
- Путь к файлу
- Запрошенное действие

### Анализ журналов

Журналы помогают:
- Отлаживать профили
- Обнаруживать попытки атак
- Аудировать действия программ
- Улучшать политики безопасности

### Интеграция с SIEM

Журналы AppArmor можно отправлять в системы мониторинга безопасности. Это позволяет коррелировать события с другими источниками.

## Практические примеры профилей

### Профиль для веб-сервера

Веб-серверу нужно:
- Читать конфигурацию из /etc/nginx
- Читать статические файлы из /var/www
- Писать логи в /var/log/nginx
- Создавать временные файлы в /tmp/nginx
- Слушать сетевые порты

Профиль разрешает всё это и запрещает всё остальное. Он не даёт серверу читать файлы пользователей, системные конфигурации, или писать в системные папки.

### Профиль для DNS-сервера

DNS-серверу нужно:
- Читать зоны из /etc/bind
- Писать журналы в /var/log/bind
- Слушать порты DNS
- Обновлять зоны динамически

Профиль ограничивает сервер этими действиями. Он не может читать домашние папки пользователей или модифицировать системные файлы.

### Профиль для базы данных

Базе данных нужно:
- Читать и писать файлы данных в /var/lib/postgresql
- Писать логи в /var/log/postgresql
- Слушать сетевой порт
- Создавать временные файлы

Профиль защищает от нежелательного доступа к другим частям системы.

## Ограничения AppArmor

### Path-based ограничения

Поскольку AppArmor опирается на пути, он уязвим к:
- Символическим ссылкам
- Hard links
- Mount namespaces
- Перемещённым файлам

Если программа создаёт символическую ссылку на файл за пределами разрешённого пути, AppArmor может не блокировать доступ.

### Нет MLS/MCS

AppArmor не поддерживает Multi-Level Security и Multi-Category Security. Он не может разделять данные по уровням секретности.

### Ограниченная гранулярность

AppArmor контролирует доступ к файлам и сети, но не может контролировать:
- Системные вызовы (это делает seccomp)
- Память процесса
- IPC (межпроцессное взаимодействие)
- Сигналы между процессами

## Интеграция с другими механизмами

### AppArmor + Seccomp

AppArmor контролирует доступ к файлам. Seccomp контролирует системные вызовы. Вместе они создают мощную защиту.

### AppArmor + Namespaces

Namespaces изолируют процессы друг от друга. AppArmor добавляет гранулярный контроль доступа внутри namespace.

### AppArmor + Capabilities

AppArmor может ограничивать, какие capabilities имеет программа. Это уменьшает поверхность атаки.
