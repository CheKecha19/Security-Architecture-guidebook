# Аутентификация в Kafka: идентификация клиентов в распределённой системе

## Что такое аутентификация в Kafka?

Представь ночной клуб. У входа стоит охранник. Каждый, кто входит, показывает документ. Охранник проверяет: это действительно тот человек, за кого он себя выдаёт? Без этой проверки — в клуб войдёт кто угодно. Владельцы не будут знать, кто у них внутри. Проблемы начнутся при первом инциденте.

**Аутентификация в Kafka** — это именно такой «охранник у входа» в брокер сообщений. Каждый клиент, который подключается к кластеру Kafka — будь то **продюсер**, **консьюмер**, **административный инструмент** или **другой брокер** — должен доказать свою личность, прежде чем получит доступ к ресурсам.

Почему это критично? Kafka — центральный узел потоков данных. Если в него попадёт нелегитимный клиент, он сможет читать конфиденциальные сообщения, писать мусор или вредоносные данные, удалять топики, перенастраивать репликацию. Аутентификация — первая линия обороны, которая отделяет «свои» от «чужих».

## Эволюция и мотивация

Ранние версии Kafka (0.8-0.9) не имели встроенного механизма аутентификации. Вся безопасность сводилась к network-level firewall: если клиент находился в доверенной сети — он считался легитимным. Это был мир «мягкого периметра», где внутренние хосты доверяли друг другу.

К 2015 году три тренда уничтожили эту модель. Во-первых, **миграция в облака**: Kafka-кластер может быть в AWS, а продюсер в Azure — между ними интернет, а не корпоративная сеть. Во-вторых, **Zero Trust Architecture**: концепция «никогда не доверяй, всегда проверяй» требует аутентификации каждого подключения, даже изнутри. В-третьих, **комплаенс**: PCI DSS требует идентификации всех субъектов, обрабатывающих данные карт; HIPAA — для PHI; SOX — для финансовых систем.

Версия Kafka 0.9 (2015) ввела поддержку **SASL** (Simple Authentication and Security Layer) — фреймворк, позволяющий подключать разные механизмы аутентификации. Версия 0.10 добавила **SSL-аутентификацию клиентов** (mTLS). Версия 2.0 упростила настройку SASL/SCRAM. Современная Kafka поддерживает пять механизмов: GSSAPI (Kerberos), PLAIN, SCRAM-SHA-256/512, OAUTHBEARER, Delegation Token.

Эта эволюция отражает зрелость индустрии: от «сеть защищает» до «каждый клиент идентифицирован и проверен».

## Зачем это нужно?

### Сценарий 1: Поддельный продюсер

Компания использует Kafka для потока заказов. Брокер доступен из корпоративной VPN. Злоумышленник получил доступ к VPN (через скомпрометированный ноутбук сотрудника) и настраивает продюсер на адрес кластера. Без аутентификации — брокер примет любые данные. Злоумышленник начинает писать фейковые заказы: «покупатель: Иванов, сумма: 1000000, статус: оплачен». Система учёта разносит их, склад готовит отгрузку, но денег нет. Через три дня — разбирательство, убытки, репутационный ущерб.

С аутентификацией SCRAM-SHA-256: злоумышленник подключается, брокер требует username/password. У злоумышленника нет учётных данных приложения. Соединение отклонено. Алерт в SIEM: «Failed authentication from IP 10.0.0.55». SOC реагирует за минуты.

### Сценарий 2: Компрометация консьюмера

Консьюмер `analytics-app` читает топик `payments` для построения отчётов. Без аутентификации — любой другой процесс в сети может присоединиться к той же консьюмер-группе или создать новую, и читать все платежные данные. Сотрудник отдела QA запускает тестовый скрипт, который «случайно» подключается к продакшен-топику. Данные утекают.

С аутентификацией: каждый консьюмер идентифицирован как `User:analytics-app`. ACL разрешает чтение только этому пользователю. Тестовый скрипт отклонён. Попытка записывается в аудит-лог.

### Сценарий 3: Административный доступ

Команда `kafka-topics.sh --delete --topic payments` удаляет топик с финансовыми данными. Без аутентификации — любой, кто имеет сетевой доступ к брокеру, может выполнить эту команду. Внутренний злоумышленник, уволенный сотрудник, скомпрометированный аккаунт — все имеют одинаковые возможности.

С аутентификацией: только `User:admin` с правами **ClusterAction** может управлять топиками. Команда от имени другого пользователя отклонена. Изменение логируется: «User:admin deleted topic payments at 2024-01-15 14:32».

## Основные концепции

### PLAINTEXT и SSL без SASL

| Механизм | Аутентификация | Шифрование | Когда использовать |
|----------|---------------|------------|------------------|
| **PLAINTEXT** | Нет | Нет | Только localhost dev |
| **SSL без client auth** | Нет | TLS | Внутри доверенной сети (устаревает) |
| **SSL + client auth (mTLS)** | Сертификат X.509 | TLS | Высокая безопасность, Kubernetes |

**PLAINTEXT** — зло. Никогда в production. Это как ночной клуб без охраны: кто угодно входит, никаких вопросов.

**SSL без client auth** — лучше, но не решает проблему идентификации клиента. Шифрует трафик, но любой, кто может подключиться, считается легитимным. Это подходит для закрытых сетей, где доступ контролируется firewall, но не соответствует Zero Trust.

### SASL — фреймворк аутентификации

**SASL** — это не конкретный механизм, а фреймворк. Kafka поддерживает несколько механизмов SASL:

| Механизм | Протокол | Безопасность | Когда использовать |
|----------|----------|--------------|-------------------|
| **GSSAPI** | Kerberos | Высокая | Enterprise с Active Directory |
| **PLAIN** | Логин/пароль в открытом виде | ❌ Низкая | Никогда в production |
| **SCRAM-SHA-256** | Challenge-response | ✅ Высокая | Рекомендуется по умолчанию |
| **SCRAM-SHA-512** | Challenge-response | ✅ Высокая | Ещё выше безопасность |
| **OAUTHBEARER** | OAuth 2.0 Access Token | ✅ Высокая | Облачные среды, delegating auth |
| **Delegation Token** | Временные токены | ✅ Высокая | Длительные задачи, batch jobs |

### SCRAM-SHA-256 — рекомендуемый метод

**SCRAM** (Salted Challenge Response Authentication Mechanism) — современный метод аутентификации, введённый в Kafka для решения проблемы передачи паролей по сети.

**Как работает:**
1. Клиент отправляет логин (username)
2. Сервер отправляет salt (случайная строка) и nonce (случайное число, используемое один раз)
3. Клиент вычисляет proof — криптографическое доказательство, что он знает пароль, без передачи самого пароля
4. Сервер проверяет proof, используя сохранённый хеш пароля. Пароль никогда не передаётся по сети

**Преимущества SCRAM:**
- **Пароль не передаётся** — защита от перехвата
- **Защита от replay** — nonce гарантирует уникальность каждой сессии
- **Salt** — уникален для каждого пользователя, защита от rainbow tables
- **Итерации** — по умолчанию 4096, замедляют brute-force

**Настройка в Kafka:** сервер хранит хеши паролей в ZooKeeper/KRaft. Клиент настраивается через JAAS (Java Authentication and Authorization Service). Не требует внешней инфраструктуры — работает из коробки.

### Kerberos (GSSAPI) — enterprise стандарт

**GSSAPI** (Generic Security Services Application Program Interface) через **Kerberos** — стандарт аутентификации в корпоративных Windows/Linux средах.

**Как работает:**
1. Пользователь получает **TGT** (Ticket Granting Ticket) от KDC (Key Distribution Center) — обычно Active Directory
2. TGT используется для получения **service ticket** для конкретного сервиса (Kafka)
3. Клиент предъявляет service ticket брокеру
4. Брокер проверяет ticket через KDC

**Преимущества:**
- **Single Sign-On** — пользователь аутентифицируется один раз
- **Централизованное управление** — AD управляет учётными записями
- **Не требует хранения паролей в Kafka** — только keytab файлы

**Недостатки:**
- Сложность настройки — требует Kerberos инфраструктуры
- Зависимость от KDC — если KDC недоступен, аутентификация невозможна
- Время — Kerberos чувствителен к рассинхронизации часов

### OAuth 2.0 (OAUTHBEARER) — облачная аутентификация

**OAUTHBEARER** позволяет использовать OAuth 2.0 Access Token для аутентификации в Kafka.

**Как работает:**
1. Клиент получает Access Token от Authorization Server (например, Keycloak, Okta, Auth0)
2. Token предъявляется брокеру при подключении
3. Брокер валидирует token (подпись, срок действия, scopes)
4. Если валиден — аутентификация успешна

**Преимущества:**
- **Delegation** — токены выдаются централизованно
- **Масштабируемость** — не нужно хранить пользователей в Kafka
- **Integration** — работает с существующими IAM системами

**Недостатки:**
- Требует OAuth сервера — дополнительная инфраструктура
- Token expiration — нужна логика обновления
- Настройка сложнее SCRAM

### mTLS — взаимная аутентификация

**mTLS** (mutual TLS) — клиент и сервер аутентифицируют друг друга сертификатами X.509.

**Как работает:**
1. Сервер предъявляет свой сертификат клиенту
2. Клиент проверяет сертификат (CA, hostname, срок действия)
3. Клиент предъявляет свой сертификат серверу
4. Сервер проверяет клиентский сертификат
5. Оба уверены в идентичности друг друга

**Identity в Kafka:** при mTLS идентификатор пользователя берётся из **Common Name (CN)** сертификата. Сертификат с CN=alice → Principal = User:alice.

**Преимущества:**
- **Нет паролей** — сертификаты вместо username/password
- **Автоматическая ротация** — можно обновлять сертификаты без downtime
- **Подходит для Kubernetes** — Service Mesh (Istio, Linkerd) управляет mTLS автоматически

**Недостатки:**
- **PKI** — требует инфраструктуры управления сертификатами
- **Сложность** — настройка и отладка сложнее SCRAM
- **Revocation** — нужен механизм отзыва сертификатов (CRL, OCSP)

## Сравнение механизмов аутентификации

| Критерий | SCRAM-SHA-256 | Kerberos (GSSAPI) | OAuth 2.0 | mTLS |
|----------|---------------|-------------------|-----------|------|
| **Сложность настройки** | Низкая | Высокая | Средняя | Средняя |
| **Требует внешней инфраструктуры** | Нет | Да (KDC/AD) | Да (OAuth Server) | Да (PKI) |
| **Подходит для Kubernetes** | Да | Нет | Да | Да (лучший) |
| **Подходит для облака** | Да | Нет | Да | Да |
| **Enterprise SSO** | Нет | Да | Да | Нет |
| **Нет паролей по сети** | Да | Да | Да | Да |
| **Автоматическая ротация** | Вручную | Через AD | Через token refresh | Через PKI |
| **Стандарт** | RFC 7677 | RFC 4120 | RFC 6749 | RFC 5246 |

## Уроки из реальных инцидентов

### Инцидент 1: Утечка через SASL/PLAIN (2018)

Компания настроила Kafka с SASL/PLAIN для простоты. Пароли передавались по сети в открытом виде (base64, но легко декодировать). Злоумышленник перехватил трафик между продюсером и брокером, извлёк пароль, подключился от имени легитимного приложения.

**Ущерб:** Чтение всех сообщений из топика `customer-pii` за 6 месяцев. Утечка данных 500,000 клиентов.

**Как SCRAM изменил бы исход:** SCRAM не передаёт пароль по сети. Даже при перехвате трафика злоумышленник получает только proof, который бесполезен для повторного использования.

### Инцидент 2: Компрометация keytab (2020)

Компания использовала Kerberos для аутентификации. Keytab файл (содержащий хеш пароля сервисной учётной записи) был сохранён в NFS с правами 644 (чтение для всех). Злоумышленник прочитал keytab, создал собственный Kerberos ticket, аутентифицировался в Kafka как сервис.

**Ущерб:** 3 месяца незамеченного доступа. Модификация конфигурации репликации, потеря данных.

**Как защита изменила бы исход:** Keytab должен храниться с правами 600, принадлежать пользователю kafka. Regular audit прав доступа к keytab файлам. Использование keytab только на хостах, где необходим.

## Архитектурные решения

### Выбор механизма аутентификации

| Сценарий | Рекомендация | Почему |
|----------|-------------|--------|
| Новый проект, нет IAM инфраструктуры | **SCRAM-SHA-256** | Просто, безопасно, работает из коробки |
| Enterprise с Active Directory | **Kerberos (GSSAPI)** | SSO, централизованное управление |
| Облако (AWS/Azure/GCP) | **OAuth 2.0** | Интеграция с облачным IAM |
| Kubernetes, Service Mesh | **mTLS** | Istio/Linkerd управляют автоматически |
| Высочайшая безопасность | **mTLS + SCRAM** | Defense-in-depth, два фактора |

### Настройка по умолчанию

```
Правило: никогда не использовать PLAINTEXT или SASL/PLAIN в production.
Правило: всегда использовать SSL/TLS для шифрования.
Правило: SCRAM-SHA-256 — default choice, если нет специфических требований.
Правило: separate credentials per application (не shared accounts).
Правило: regular rotation of passwords and certificates.
```

### Метрики безопасности аутентификации

| Метрика | Целевое значение | Действие при превышении |
|---------|-----------------|------------------------|
| Failed authentication rate | < 5 за 5 минут | Блокировать IP, алерт SOC |
| Successful auth from new IP | Алерт | Проверить легитимность |
| Certificate expiration | > 30 дней | Автоматическая ротация |
| Password age | < 90 дней | Принудительная смена |

## Подготовка к собеседованию

### Три типичные ошибки кандидатов

**Ошибка 1:** «SSL достаточно для аутентификации».

SSL обеспечивает шифрование и аутентификацию сервера. Но без client auth (mTLS) — клиент не идентифицирован. Нужна комбинация: SSL + SASL или mTLS.

**Ошибка 2:** «SCRAM и PLAIN — одно и то же».

PLAIN передаёт пароль в открытом виде (base64). SCRAM использует challenge-response, пароль никогда не передаётся. PLAIN — security risk, SCRAM — secure.

**Ошибка 3:** «Один пользователь на все приложения».

Shared account невозможно аудировать. Если увольняется разработчик — нужно менять пароль для всех приложений. Нужен отдельный пользователь на каждое приложение.

### Три ситуационных вопроса

**Вопрос 1:** «Как защитить Kafka в мультитенантной среде?»

Ответ: **SASL_SSL с SCRAM** для аутентификации каждого тенанта. **ACL** для изоляции топиков (tenant-a не видит tenant-b). **Quotas** для ограничения ресурсов на тенанта. **Encryption at-rest** с разными ключами (или разные tablespaces). **Audit** логирование для отслеживания跨tenant access.

**Вопрос 2:** «Что делать, если Kerberos KDC недоступен?»

Ответ: Kerberos требует доступности KDC. Решения: **репликация KDC** (несколько контроллеров домена). **Fallback на SCRAM** — настроить SASL_SSL с несколькими механизмами. **Local cache** — Kerberos ticket valid for hours, cache на клиенте. **Monitoring** — алерт при недоступности KDC.

**Вопрос 3:** «Как ротировать сертификаты mTLS без downtime?»

Ответ: **Dual certificate** — брокер принимает и старый, и новый сертификат одновременно. **Rolling update** — обновлять брокеры по очереди. **CA rotation** — добавить новый CA в truststore до удаления старого. **Automation** — Cert-manager в Kubernetes автоматически ротирует сертификаты.

## Чек-лист понимания

- [ ] Чем **аутентификация** отличается от **авторизации**?
- [ ] Какие механизмы **SASL** поддерживает Kafka?
- [ ] Почему **SCRAM** безопаснее **PLAIN**?
- [ ] Как работает **Kerberos** с Kafka?
- [ ] Что такое **mTLS** и когда его использовать?
- [ ] Как **OAuth 2.0** интегрируется с Kafka?
- [ ] Что такое **Delegation Token**?
- [ ] Как выбрать механизм аутентификации?
- [ ] Как защитить **keytab** файлы?
- [ ] Как мониторить failed authentication?

### Ответы на чек-лист

1. **Аутентификация** — проверка личности (кто ты?). **Авторизация** — проверка прав (что тебе можно?). Сначала аутентификация, потом авторизация.

2. **SASL механизмы:** GSSAPI (Kerberos), PLAIN (опасен), SCRAM-SHA-256/512 (рекомендуется), OAUTHBEARER (OAuth 2.0), Delegation Token (временные).

3. **SCRAM** использует challenge-response: пароль никогда не передаётся по сети. **PLAIN** передаёт логин/пароль в открытом виде (base64), легко декодируется.

4. **Kerberos** через GSSAPI: клиент получает TGT от KDC, затем service ticket для Kafka. Брокер проверяет ticket. Требует AD/KDC инфраструктуры.

5. **mTLS** — взаимная аутентификация сертификатами. Клиент и сервер проверяют друг друга. Использовать в Kubernetes, high-security средах.

6. **OAuth 2.0** через OAUTHBEARER: клиент предъявляет Access Token. Брокер валидирует подпись и срок. Требует OAuth сервера (Keycloak, Okta).

7. **Delegation Token** — временный токен, выданный брокером. Использовать для длительных задач вместо постоянных учётных данных. Время жизни ограничено.

8. Выбор: **SCRAM-SHA-256** — default; **Kerberos** — enterprise с AD; **OAuth** — облако; **mTLS** — Kubernetes / high security.

9. **Keytab** должен иметь права 600, принадлежать пользователю kafka. Хранить только на хостах, где необходим. Никогда в Git или Docker image.

10. Мониторить: количество failed auth (алерт при > 5/5min), новые IP адреса (при аутентификации), частота успешных входов (аномалии), срок сертификатов и паролей.

## Ключевые выводы для собеседования

- **Kafka по умолчанию небезопасна** — аутентификация требует явной настройки
- **SCRAM-SHA-256 — default choice** — просто, безопасно, стандартизировано
- **Никогда PLAIN в production** — передаёт пароль в открытом виде
- **Defense-in-depth** — комбинация SSL + SASL + ACL
- **Separate credentials per app** — не shared accounts, для аудита и ротации
