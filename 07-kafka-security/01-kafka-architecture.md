# Архитектура Kafka и безопасность распределённого журнала

## Что такое архитектура Kafka?

Представь центральную диспетчерскую аэропорта. Каждый рейс приземляется — диспетчер делает запись в журнал: рейс SU-1402, полоса 2, 14:35. Служба багажа читает этот журнал и отправляет тележки. Служба питания читает ту же запись — готовит еду. Пограничный контроль сверяется — выставляет наряды. Все читают одну и ту же запись, каждый делает свою работу. Никто не может подменить уже сделанную запись.

**Apache Kafka** — это именно такой «диспетчерский журнал» для корпоративных данных. Распределённая система потоковой передачи событий, которая принимает потоки сообщений, сохраняет их на диске, и позволяет множеству приложений читать эти сообщения независимо друг от друга. Ключевое отличие от обычной очереди сообщений: Kafka **сохраняет данные на диске** даже после прочтения, и любой потребитель может «перемотать» назад и прочитать с начала.

С точки зрения безопасности Kafka решает фундаментальную проблему **разделения зон ответственности** в потоке данных. В традиционных системах продюсер и консьюмер жёстко связаны: если консьюмер падает — продюсер должен ждать. Kafka вводит буферный слой, который физически разъединяет отправителя и получателя, позволяя применять политики безопасности на границе этого слоя — аутентифицировать каждого подключившегося, авторизовать каждую операцию, зашифровать каждый байт и запротоколировать каждое действие для аудита.

## Эволюция и мотивация

До появления Kafka типичная интеграция выглядела как паутина прямых HTTP-соединений между сервисами. Отчётный сервис стучался в базу биллинга, сервис уведомлений дёргал API заказов, аналитическая система слала SQL-запросы в оперативную БД. С точки зрения безопасности это был кошмар: каждая пара сервисов требовала уникальных учётных данных, сетевые файрволы пропускали десятки нестандартных портов, аудит был размазан по логам двух десятков систем, утечка одного пароля компрометировала конкретную бизнес-функцию.

**LinkedIn создал Kafka в 2010 году** для решения именно этой проблемы. Изначально это был внутренний проект для обработки логов активности пользователей — миллионы событий в секунду, сотни потребителей. В 2011 году Kafka стал open-source проектом Apache. С тех пор он прошёл путь от простой очереди сообщений до полноценной **event-streaming platform**, поддерживающей транзакции, потоковую обработку (Kafka Streams), SQL-подобные запросы (ksqlDB) и интеграцию с внешними системами (Kafka Connect).

Три технологических сдвига ускорили принятие Kafka. Во-первых, **миграция в облака** и эфемерная природа контейнеров: Pod в Kubernetes живёт минуты, прямое соединение с ним ненадёжно; нужен персистентный посредник. Во-вторых, **взрывной рост микросервисов**: в Netflix более тысячи сервисов, соединять каждый с каждым — квадратичная сложность, что не масштабируется ни с операционной, ни с безопасной точки зрения. В-третьих, **регуляторное давление**: GDPR требует знать, кто обрабатывал персональные данные и куда они пошли; Kafka как центральный хаб позволяет ответить на этот вопрос аудитом одного кластера.

## Зачем это нужно?

### Сценарий 1: Финансовый учёт в банке

Транзакционная система публикует событие «перевод выполнен». Подписываются три сервиса: anti-fraud (проверяет на мошенничество), accounting (разносит по счетам), notification (шлёт SMS клиенту). Без Kafka каждый из трёх стучался бы напрямую в core banking с разными учётными данными. Компрометация любого сервиса дала бы доступ к конфиденциальным транзакциям. С Kafka каждый сервис имеет доступ только к своей консьюмер-группе, а брокер авторизует каждое подключение индивидуально — это реализация **принципа наименьших привилегий** (least privilege), требуемого контролем AC-6 NIST SP 800-53.

### Сценарий 2: E-commerce пик нагрузки

В Чёрную пятницу поток заказов возрастает в сто раз. Без Kafka бэкенд-сервисы не справляются с прямым потоком запросов, валятся с таймаутами, заказы теряются — нарушение доступности и прямой финансовый ущерб. Kafka как буфер принимает всплеск на себя: продюсеры пишут с максимальной скоростью, консьюмеры читают в своём темпе. Контроль CP-9 (System Backup) из NIST 800-53 требует механизмов восстановления — реплицированный и ретенционный лог Kafka напрямую закрывает это требование.

### Сценарий 3: SOC-аналитика и SIEM

Каждый микросервис пишет security-события в Kafka-топик «security-events». SIEM-система читает этот топик как консьюмер. SOC-аналитик видит единую временную линию: запрос пришёл на API Gateway, затем был аутентифицирован в IAM, затем обработан бизнес-логикой, затем записан в БД. Kafka обеспечивает строгий порядок записей в рамках одной партиции — это даёт **каузальность** (causal ordering), критичную для расследования инцидентов. ISO 27001 Annex A.12.4.1 требует сбора и анализа событий безопасности; Kafka-топик как шина аудита закрывает этот контроль централизованно.

## Основные концепции

### Компоненты Kafka

**Брокер** — сервер Kafka. Хранит данные, обрабатывает запросы. В production — кластер из 3-9 брокеров. Каждый брокер имеет уникальный **broker.id**, слушает на определённом порту, хранит данные в **log.dirs**. Безопасность требует: брокер должен слушать только внутренние сети, данные должны храниться на зашифрованных дисках.

**Топик** — категория сообщений. Например, «payments», «orders», «user-events». Топик разбивается на **партиции** — подмножества сообщений. Каждая партиция — это упорядоченный лог: offset 0, offset 1, offset 2... Сообщения внутри партиции упорядочены, но порядок между партициями не гарантирован. Для sensitive данных может потребоваться одна партиция — потеря параллелизма, но строгий порядок.

**Продюсер** — приложение, пишущее в Kafka. Отправляет сообщения в определённый топик и партицию. Producer должен быть **аутентифицирован** и **авторизован** для записи в конкретный топик.

**Консьюмер** — приложение, читающее из Kafka. Объединяется в **консьюмер-группы** — каждый partition читается одним консьюмером в группе. Нужно контролировать: кто может присоединяться к группе, что можно читать.

**ZooKeeper / KRaft** — управление метаданными кластера. ZooKeeper (устаревает) — внешняя зависимость, требует отдельной аутентификации. KRaft (Kafka Raft, новое) — встроено в Kafka, единая система аутентификации, проще и безопаснее. Рекомендация для новых установок — KRaft.

### Модели безопасности

| Модель | Аутентификация | Шифрование | Когда использовать | Риски |
|--------|---------------|------------|-------------------|-------|
| **PLAINTEXT** | Нет | Нет | Только localhost dev | Полный доступ |
| **SSL** | Нет | TLS | Внутри доверенной сети | Нет контроля подключений |
| **SASL_SSL** | SCRAM/Kerberos/OAuth | TLS | Рекомендуется для большинства | Сложность настройки |
| **mTLS** | Сертификаты | TLS | Высокая безопасность | Управление сертификатами |

### Репликация и отказоустойчивость

Каждая партиция имеет **лидера** и **реплики**. Лидер обрабатывает чтение и запись, реплики следуют за лидером, применяя WAL. Если лидер падает — один из инсинк-реплик становится новым лидером. Это **Leader Election**, управляется контроллером кластера.

С точки зрения безопасности репликация создаёт копии данных на нескольких брокерах. Если один брокер скомпрометирован — данные всё ещё доступны на других. Но это также означает: если злоумышленник получит доступ к одной реплике — он получит все данные партиции. Шифрование at-rest (LUKS, облачное шифрование) решает эту проблему.

## Сравнение с другими системами

| Критерий | Apache Kafka | RabbitMQ | Apache Pulsar | Amazon Kinesis |
|----------|--------------|----------|---------------|----------------|
| **Пропускная способность** | Миллионы msg/s | Сотни тысяч msg/s | Миллионы msg/s | Масштабируется |
| **Сохранение сообщений** | Длительное (retention) | Короткое (подтверждение) | Длительное | 1-365 дней |
| **Повторное чтение** | Да (offset) | Нет (ack удаляет) | Да | Да |
| **Репликация** | Leader-follower | Mirror Queue | Geo-replication | Managed |
| **Безопасность** | SASL, SSL, ACL | SASL, TLS, RBAC | TLS, JWT, RBAC | IAM, SSL |
| **Комплаенс** | NIST, ISO, CIS | PCI DSS ready | NIST, ISO | SOC2, PCI |

Kafka выигрывает в пропускной способности и гибкости хранения. RabbitMQ проще в настройке, но ограничен в масштабировании. Pulsar обеспечивает geo-replication из коробки. Kinesis удобен для облачных сред, но vendor lock-in.

## Уроки из реальных инцидентов

### Инцидент 1: Утечка данных через незащищённый Kafka (2019)

Компания хранила персональные данные клиентов в Kafka-топике. Брокеры были доступны из корпоративной сети без аутентификации (PLAINTEXT). Сотрудник департамента аналитики случайно подключился к продакшен-кластеру вместо тестового — и получил доступ ко всем данным клиентов.

**Хронология:** Аналитик запустил консьюмер без указания группы, Kafka назначил новую группу, консьюмер начал читать с самого начала (default offset = earliest), и аналитик получил все данные за год. Никаких препятствий — ни аутентификации, ни авторизации.

**Ущерб:** Утечка данных 2 миллионов клиентов. Штраф по GDPR — €50,000. Репутационные потери.

**Как Kafka с правильной архитектурой изменила бы исход:** Если бы был настроен SASL_SSL + ACL — аналитик не смог бы подключиться без учётных данных. Если бы был ACL с правилом «только приложения из белого списка» — подключение от неизвестного клиента было бы отклонено. Если бы было включено аудитирование — инцидент обнаружили бы за минуты, а не за недели.

### Инцидент 2: Ransomware через Kafka Connect (2021)

Компания использовала Kafka Connect для интеграции с внешними системами. Connector загружал JAR-файлы из публичного Maven-репозитория. Злоумышленник подменил артефакт в репозитории — внедрил вредоносный код.

**Хронология:** Connector скачал и выполнил вредоносный JAR. Код получил доступ к файловой системе брокера, нашёл незашифрованные WAL-файлы, зашифровал их. Kafka-кластер перестал работать. Злоумышленник потребовал выкуп.

**Ущерб:** Остановка обработки заказов на 8 часов. Потеря $500,000 выручки. Платёж выкупа — $50,000.

**Как защита изменила бы исход:** Если бы plugin.path был ограничен доверенными директориями — вредоносный JAR не загрузился бы. Если бы использовался verify для JAR-подписей — подмена обнаружилась бы. Если бы WAL был зашифрован at-rest — шифрование злоумышленника не имело бы эффекта.

## Архитектурные решения

### Defense-in-Depth для Kafka

| Слой | Защита | Контроль |
|------|--------|----------|
| Периметр | Firewall, Security Groups | Только white-listed IP |
| Транспорт | TLS 1.3 | Все соединения шифрованы |
| Аутентификация | SASL SCRAM-SHA-256 | Каждый клиент идентифицирован |
| Авторизация | ACL / RBAC | Только необходимые права |
| Данные | Шифрование at-rest | LUKS / облачное KMS |
| Аудит | Логирование всех операций | SIEM, алерты |

### Zero Trust для Kafka

Принцип: «Никогда не доверяй, всегда проверяй».

- Каждый producer аутентифицируется и авторизуется
- Каждый consumer аутентифицируется и авторизуется
- Каждый broker аутентифицирует другие брокеры (inter-broker)
- Внутренние топики (offsets, configs) защищены от внешнего доступа
- Все данные шифруются в пути и в покое
- Все действия логируются и анализируются

### Метрики безопасности

| Метрика | Целевое значение | Что показывает |
|---------|-----------------|----------------|
| SSL coverage | 100% | Все соединения шифрованы |
| Auth failures | < 5 за час | Brute-force или компрометация |
| Denied operations | < 10 за час | Попытки несанкционированного доступа |
| Unencrypted topics | 0 | Нет открытых данных |
| Audit log lag | < 1 мин | Централизация работает |

## Подготовка к собеседованию

### Три типичные ошибки кандидатов

**Ошибка 1:** «Kafka сама по себе безопасна, потому что это Apache проект».

Kafka по умолчанию настроена на максимальную производительность и минимальное трение при развёртывании. Безопасные настройки (SSL, SASL, ACL) требуют явной конфигурации. По умолчанию listeners=PLAINTEXT, аутентификация отключена, ACL не включены.

**Ошибка 2:** «SSL достаточно для безопасности Kafka».

SSL обеспечивает только конфиденциальность и целостность в пути. Он не аутентифицирует клиентов (если не mTLS) и не авторизует операции. Нужна комбинация: SSL + SASL + ACL.

**Ошибка 3:** «Одного суперпользователя достаточно для администрирования».

Разделение обязанностей требует разных ролей: один администратор управляет топиками, другой — ACL, третий — мониторит. CIS Control 6.1 требует «Use of Dedicated Administrative Accounts».

### Три ситуационных вопроса

**Вопрос 1:** «Как бы вы защитили Kafka в Kubernetes, где Pod-ы эфемерны?»

Ответ: Использовать **SASL_SSL** для аутентификации и шифрования. **Service Account** для идентификации Pod-ов. **Network Policies** для ограничения трафика. **mTLS между брокерами** для идентификации узлов кластера. **Pod Security Policies** или **OPA Gatekeeper** для запрета привилегированных контейнеров. **Falco** для обнаружения аномалий в runtime.

**Вопрос 2:** «Как обеспечить compliance PCI DSS для данных карт в Kafka?»

Ответ: **Шифрование at-rest** (LUKS или облачное KMS). **TLS 1.2+** для всех соединений. **SASL_SSL с SCRAM** для аутентификации. **ACL** — только процессинговый сервис может читать топик card-transactions. **pgaudit** или эквивалент для аудита. **Retention** не более необходимого времени. **Удаление** после обработки (не хранить дольше необходимого).

**Вопрос 3:** «В чём разница между физической и логической репликацией с точки зрения безопасности?»

Ответ: **Физическая репликация** копирует байт-в-байт, включая все данные и структуру. Безопасность: если primary скомпрометирован — standby тоже (данные идентичны). **Логическая репликация** копирует только данные определённых таблиц. Безопасность: можно выбирать, что реплицировать; можно фильтровать строки (publish WHERE). Но DDL не реплицируется — структуру нужно синхронизировать отдельно.

### Ответы на чек-лист

1. **Брокер** — сервер Kafka. **Топик** — категория сообщений. **Партиция** — подмножество топика, упорядоченный лог. Несколько партиций обеспечивают параллелизм, одна — строгий порядок.

2. **Leader Election** — процесс выбора нового лидера при падении текущего. Контроллер кластера назначает нового лидера из инсинк-реплик. Важно: минимум 3 брокера, replication.factor >= 3, min.insync.replicas >= 2.

3. **Физическая репликация** — байт-в-байт копия WAL. **Логическая** — только изменения определённых таблиц. Физическая проще, но реплицирует всё. Логическая гибче, но требует ручной синхронизации DDL.

4. Модели: **PLAINTEXT** (нет защиты), **SSL** (шифрование, но нет аутентификации), **SASL_SSL** (аутентификация + шифрование), **mTLS** (взаимная аутентификация + шифрование).

5. **PLAINTEXT** — без шифрования и аутентификации. **SASL_SSL** — комбинация SASL (SCRAM, Kerberos, OAuth) для аутентификации и SSL для шифрования. **mTLS** — клиент и сервер аутентифицируют друг друга сертификатами.

6. **ACL** (Access Control List) — список правил: кто может что делать с каким ресурсом (топик, группа, кластер). Пример: User:app может Write в Topic:orders.

7. Риски: перехват данных в сети, несанкционированное чтение, подделка сообщений, утечка через незащищённые консьюмеры, отсутствие аудита.

8. **KRaft** — встроенный механизм консенсуса Kafka (Kafka Raft). Убирает зависимость от ZooKeeper. Проще в настройке, единая система аутентификации, меньше поверхность атаки. **ZooKeeper** — внешняя система, требует отдельной защиты.

9. Метрики: SSL coverage (100%), auth failures (< 5/час), denied operations (< 10/час), unencrypted topics (0), replication lag (< 1 сек).

10. В Kubernetes: SASL_SSL для клиентов, Service Accounts для Pod-ов, Network Policies для ограничения трафика, mTLS между брокерами, Pod Security Policies, Falco для runtime security.

## Ключевые выводы для собеседования

- **Kafka по умолчанию небезопасна** — требует явной конфигурации SSL + SASL + ACL
- **Defense-in-Depth** — периметр, транспорт, аутентификация, авторизация, данные, аудит
- **Zero Trust** — никогда не доверяй, всегда проверяй, даже внутри сети
- **KRaft > ZooKeeper** — проще, безопаснее, современнее
- **Мониторинг критичен** — метрики безопасности должны быть частью operational dashboard
