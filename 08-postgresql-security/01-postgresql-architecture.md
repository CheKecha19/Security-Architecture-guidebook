# Архитектура PostgreSQL: безопасный фундамент

## Что такое PostgreSQL?
PostgreSQL — это **объектно‑реляционная СУБД** с открытым исходным кодом, известная своей **расширяемостью**, **соответствием стандарту SQL** и богатым набором **фич безопасности**. По своей природе она представляет собой набор процессов, файловой структуры, системных каталогов и механизмов репликации, каждый из которых имеет свои **угрозо‑поведенческие границы**. Понимание этой архитектуры критично для построения надёжного периметра и минимизации **атак поверхностных** векторных поверхностей.

## Эволюция и мотивация
С момента первого релиза в 1996 году PostgreSQL прошла путь от простого **проектного репозитория** к **корпоративной платформе**. В каждой версии добавлялись новые механизмы защиты: от **модульной аутентификации** (pg_hba.conf) до **логирования аудита** (pgAudit) и **контроля доступа на уровне строк** (RLS). Эволюция мотивирована ростом требований регуляторов (**GDPR**, **PCI‑DSS**, **HIPAA**) и ростом количества публичных атак, таких как CVE‑2018‑16850 и криптомайнинг‑кампании 2025‑2026 гг. 

## Зачем это нужно?
Без‑адресное представление о «базе данных» скрывает многочисленные **атакующие поверхности**: неправильные права файловой системы, незащищённые сетевые порты, скриптовые функции с привилегиями **SUPERUSER**. Архитектурный обзор позволяет **разделить зоны ответственности**, **внедрить принцип наименьших привилегий** и подготовить **план реагирования на инциденты**.

## Основные концепции

### Процессная модель (postmaster, backends, workers)
* **postmaster** – главный процесс, прослушивающий порт(ы) и отвечающий за **fork‑ing** новых **backend‑процессов** для каждой клиентской сессии. Он также запускает **walwriter**, **bgwriter**, **checkpointer**, **archiver**, **stats collector**, **autovacuum launcher** и **logical replication launcher**. Каждый из этих воркеров работает в отдельном **POSIX‑процессе**, что изолирует сбои и облегчает **контроль ресурсов**.
* **Backend‑процессы** работают под учётной записью **postgres** (или под другим системным пользователем) и наследуют **права ОС**. Поэтому важно **ограничить системные привилегии** (например, использовать `systemd`‑cgroup’ы).
* **Worker‑процессы** (логическая репликация, background workers) могут быть расширены пользователями через `CREATE EXTENSION`. Для безопасного расширения следует проверять **подписи** и **контрольные суммы**.

### Система каталогов и pg_catalog
* **pg_catalog** – системный **скелет**: таблицы, представления и функции, описывающие **метаданные** (схемы, типы, роли, ACL). Доступ к `pg_catalog` обычно разрешён **PUBLIC**, но **чтение** чувствительных атрибутов (например, пароли в `pg_authid`) ограничивается только **SUPERUSER**.
* **Файловая иерархия** (`base/`, `pg_wal/`, `pg_tblspc/`, `pg_logical/`, `pg_stat/`, `pg_replslot/`, `pg_snapshots/`). Защита достигается через **chmod 700** и **ownership postgres**. Неправильные права могут позволить локальному атакующему **прочитать WAL** и восстановить данные.

### Роли и привилегии
* **Роли** (users, groups) управляются через `CREATE ROLE`. Атрибуты **SUPERUSER**, **CREATEROLE**, **CREATEDB**, **INHERIT**, **LOGIN**, **REPLICATION**. **Лучшее практическое правило** – предоставлять **только необходимые** атрибуты.
* **GRANT/REVOKE** применимы к **схемам**, **таблицам**, **колонкам**, **функциям**. По‑умолчанию **ACL** наследуются, но их можно **запретить** через `ALTER DEFAULT PRIVILEGES`.

### Схемы и изоляция
* **Схемы** позволяют логически организовать объекты в отдельные **namespaces**. При включённом **search_path** пользователь может попасть в **нежелательную схему**. Лучшее решение – явно указывать `search_path` в `postgresql.conf` или `ALTER ROLE ... SET search_path`.
* **Row‑Level Security (RLS)** (позволяет ограничить доступ к строкам). Включается через `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`. **Политики** (`CREATE POLICY`) могут задавать **контроль доступа на основе атрибутов** (например, `department_id`). Эта модель часто используется в **мульти‑тенантных** приложениях.

### WAL и точки восстановления
* **Write‑Ahead Log (WAL)** – последовательность записей, гарантирующая **атомарность** и **надёжность**. Каждый **segement** хранится в `pg_wal/`. При включённом **archive_mode** WAL копируется в **архив** (`archive_command`). Это фундамент для **Point‑In‑Time Recovery (PITR)**.
* **Replication slots** позволяют **стримить** WAL на реплики без потери данных. **Logical replication** (pub/sub) использует **wal2json** или **pgoutput**‑плагин. Для **security‑focused** репликации рекомендуется **шифровать канал** (TLS) и ограничить **пользователей репликации** отдельными ролями с атрибутом **REPLICATION**.

## Сравнение с другими механизмами
| Параметр | PostgreSQL | MySQL (InnoDB) | Oracle DB |
|----------|------------|----------------|-----------|
| **Модель процессов** | fork‑based, каждый клиент → отдельный процесс | thread‑based (one thread per connection) | multi‑process (SMON, PMON) + shared memory |
| **WAL** | отдельный каталог `pg_wal`, целостность через контрольные суммы | binlog, менее детализированный | redo logs, сильно интегрированы с RAC |
| **RLS** | встроенный, гибко настраиваемый | отсутствует (в 8.0+ можно использовать Views) | Fine‑grained Access Control (FGAC) |
| **Аудит** | pgAudit, log_line_prefix, event triggers | audit_log plugin, MySQL Enterprise Audit | Fine‑grained Auditing, Unified Auditing |
| **Шифрование** | pgcrypto, TLS для соединений, Transparent Data Encryption (TDE) в 15+ | TLS, Per‑Table Encryption (Enterprise) | TDE, Transparent Data Encryption |
| **Публичный источник** | ✅ Open‑source | ✅ Open‑source | ❌ Proprietary |
| **Расширяемость** | C extensions, FDWs, PL/pgSQL, PL/Python | Plugins, UDFs | Java, PL/SQL, C extensions |

## Уроки из реальных инцидентов
### CVE‑2018‑16850 (PostgreSQL RCE)
* **Вектор**: SQL‑инъекция в `pg_upgrade`/`pg_dump` через `CREATE TRIGGER … REFERENCING`. 
* **Последствия**: удалённый код‑выполнение на уровне **postgres**.
* **Вывод**: всегда **обновляйте** версии, ограничивайте **SUPERUSER** доступ, отключайте **pg_dump**/`pg_upgrade` для небезопасных ролей.

### Криптомайнинг через weak‑passwords (Wiz 2025‑2026)
* **Вектор**: публичные инстансы с **по умолчанию** `postgres`/`admin` паролем, последующее использование `COPY FROM PROGRAM` для загрузки и выполнения майнеров.
* **Меры**:
  * Жёсткая политика паролей (`password_encryption = scram-sha-256`).
  * Отключить `COPY FROM PROGRAM` (`postgresql.conf -> trusted_extensions = ''`).
  * Мониторинг `pg_hba.conf` и **audit**‑логов `pgaudit`.
* **Вывод**: **сетевой** контроль доступа (`pg_hba.conf` `host all all 0.0.0.0/0 reject`) и **парольная** политика – первая линия обороны.

## Инструменты и средства защиты
| Инструмент | Задача | Примечание |
|------------|--------|------------|
| **pgAudit** | детальное журналирование запросов (SESSIONS, OBJECTS) | требует `shared_preload_libraries` |
| **pgaudit-analyze** | парсинг и визуализация audit‑логов | использует Perl/Regex |
| **pgaudit‑extension** | расширение для **row‑level** аудита | совместимо с RLS |
| **pgcrypto** | функции шифрования/хеширования в‑базе | поддерживает AES, SHA‑2 |
| **ssh‑tunnel + stunnel** | шифрование репликационного канала | альтернатива встроенному TLS |
| **fail2ban + PostgreSQL‑log** | блокировка IP после неудачных логинов | настраивается под `pg_hba.conf` |
| **Patroni / Stolon** | автоматическое управление кластером и **failover** | повышает **availability**, но добавляет **операционную поверхность** |

## Архитектурные решения
1. **Разделение ролей**: отдельные роли для **операций** (backup), **репликации**, **приложения**. При необходимости использовать **role inheritance** для простого управления.
2. **Изоляция файловой системы**: монтировать `pg_wal`, `pg_data` на **separate partitions** с `noexec,nosuid,nodev`.
3. **TLS everywhere**: включить `ssl = on`, задать `ssl_cert_file`, `ssl_key_file`. Для **replication** добавить `hostssl replication ...` в `pg_hba.conf`.
4. **Контроль доступа к сети**: использовать **firewall** (iptables / Windows Firewall) чтобы разрешать только **trusted CIDR** к **5432** (или кастомному порту).
5. **Автоматическое ревью конфигураций**: `pg_validate_extschema` + CI‑pipeline проверяют `postgresql.conf`, `pg_hba.conf` на **лишние разрешения**.

| WAL | *Что такое checkpoint и как он влияет на производительность?* |
| Аудит | *Какие возможности даёт pgAudit и как его включить?* |
| Репликация | *Опишите разницу между физической и логической репликацией.* |
| Безопасность | *Какие шаги предпринимать после обнаружения открытого `pg_hba.conf`?* |

### Ответы на чек‑лист (коротко)
1. **postmaster** — главный процесс, отвечает за **socket‑listen** и **fork‑ing** backends.
2. **RLS** включается `ALTER TABLE … ENABLE ROW LEVEL SECURITY;` и добавляется `CREATE POLICY … USING (condition);`.
3. **Checkpoint** — запись всех dirty‑page‑ов из shared buffers в диск, вызывается автоматически (по времени/лог‑размеру) или вручную `CHECKPOINT;`.
4. **pgAudit** требует `shared_preload_libraries = 'pgaudit'` и `pgaudit.log = 'all'` в `postgresql.conf`.
5. **Физическая репликация** копирует файловый уровень (WAL), **логическая** — отдельные изменения (INSERT/UPDATE) через публикации/подписки.
6. Закрыть открытый доступ: изменить `pg_hba.conf` на `host all all 127.0.0.1/32 trust` + **restart**; включить **fail2ban** для brute‑force.

## Ключевые выводы для собеседования
* **Процессная изоляция** (postmaster → backends) обеспечивает **надежность** и **контроль ресурсов**.
* **pg_catalog** хранит метаданные; доступ к `pg_authid` – только **SUPERUSER**.
* **RLS** + **Row‑level policies** – лучшая практика для **многоарендных** приложений.
* **WAL** = фундамент **PITR**; архивация и **replication slots** усиливают **доступность**.
* **pgAudit**, **pgaudit‑analyze**, **pgcrypto** – базовые инструменты **комплаенса**.
* **Конфигурация сети** (`pg_hba.conf`, firewall) – первая линия обороны; **TLS** обязателен для репликации и клиентских соединений.
* **Обновления** (CVE‑2018‑16850, etc.) и **жёсткие пароли** – критически важны после реальных инцидентов.
* При собеседовании следует подчеркнуть **политику наименьших привилегий**, **разделение ролей**, **чтение/понимание** `postgresql.conf` и `pg_hba.conf`.
