# Архитектура PostgreSQL и модель безопасности

## Что такое архитектура PostgreSQL?

Представь огромную библиотеку. Там есть каталог (где описано, что есть), стеллажи (где хранятся книги), читатели (кто берёт книги), библиотекари (кто выдаёт). Всё организовано: каждая книга на своём месте, каждый читатель зарегистрирован, каждая выдача записана.

**PostgreSQL** — это именно такая «библиотека» для данных. Открытая объектно-реляционная база данных. Хранит таблицы, индексы, функции, процедуры. Обеспечивает целостность, отказоустойчивость, расширяемость. И главное — контролирует: кто и к чему имеет доступ.

Почему архитектура важна для безопасности? Потому что безопасность не наложена сверху — она встроена в дизайн. PostgreSQL создавалась с учётом принципа **least privilege**: каждый процесс, каждый пользователь, каждый объект — имеет ровно те права, которые необходимы.

## Эволюция и мотивация

PostgreSQL начинался как проект Майкла Стоунбрейкера в UC Berkeley в 1986 году. Изначально цель была академическая — продемонстрировать, что объектно-реляционная модель работает. Но к 1996 году проект стал open-source, и к 2000-м — стандартом для enterprise-приложений.

Три тренда заставили обратить внимание на безопасность архитектуры. Во-первых, **взрывной рост интернет-приложений**: PostgreSQL вышла за пределы корпоративных сетей, стала доступна из интернета. Во-вторых, **регуляторное давление**: HIPAA (1996), PCI DSS (2004), GDPR (2016) — все требуют контроля доступа к данным. В-третьих, **рост утечек данных**: каждый год десятки миллионов записей утекают из незащищённых баз.

PostgreSQL ответила расширением модели безопасности. Версия 8.1 (2005) ввела **SECURITY DEFINER** функции. Версия 9.5 (2016) добавила **Row-Level Security** (RLS). Версия 10 (2017) ввела **SCRAM-SHA-256** аутентификацию. Версия 11 улучшила шифрование. Современная PostgreSQL — это зрелая система с defense-in-depth архитектурой.

## Зачем это нужно?

### Сценарий 1: Утечка персональных данных

HR-платформа хранит резюме, зарплаты, адреса, паспортные данные. База доступна из приложения, приложение доступно из интернета. Если архитектура не обеспечивает **layered security** — одна SQL Injection даёт доступ ко всему.

С правильной архитектурой: приложение подключается как `User:app` с минимальными правами (SELECT, INSERT на рабочие таблицы). Нет прав на DDL. Нет прав на системные таблицы. RLS ограничивает видимость строк. Даже при компрометации приложения — злоумышленник видит только то, что видит приложение.

### Сценарий 2: SQL Injection

Приложение неправильно обрабатывает входные данные. Злоумышленник вставляет SQL-код в поисковую строку: `'; DROP TABLE users; --`. Без защиты — таблица удалена. С защитой: приложение использует **Prepared Statements**, `User:app` не имеет права **DROP**, **audit log** записывает попытку.

### Сценарий 3: Privilege Escalation

Пользователь `app_user` имеет права на `SELECT` из таблицы `orders`. Он создаёт функцию с `SECURITY DEFINER` от имени владельца таблицы — и получает полный доступ. Без защиты — незамеченно. С защитой: `User:app` не имеет права **CREATE FUNCTION**. Если имеет — функция создаётся с `SECURITY INVOKER` по умолчанию. Администратор регулярно аудит функции.

## Основные концепции

### Процессы PostgreSQL

| Процесс | Роль | Безопасность |
|---------|------|------------|
| **postmaster** | Главный процесс, принимает подключения | Слушает на порту, управляет правами |
| **backend** | Обрабатывает запросы клиентов | Выполняет с правами подключившегося пользователя |
| **wal writer** | Пишет WAL (Write-Ahead Log) | Журнал транзакций, критичен для recovery |
| **background writer** | Сбрасывает dirty pages | Фоновая запись на диск |
| **autovacuum** | Очищает мёртвые tuple | Необходим для performance |
| **checkpointer** | Создаёт контрольные точки | Обеспечивает консистентность |
| **stats collector** | Собирает статистику | pg_stat_* представления для мониторинга |

**Ключевой принцип:** каждый backend — отдельный процесс ОС. Если backend падает — не влияет на остальные. Изоляция на уровне процессов = изоляция на уровне безопасности.

### Хранение данных

```
$PGDATA/
├── base/              # Базы данных (по OID)
│   ├── 1/             # template1
│   ├── 13456/         # конкретная база
│   │   ├── 1234       # таблица (filenode)
│   │   ├── 1234_fsm   # Free Space Map
│   │   └── 1234_vm    # Visibility Map
├── global/            # Глобальные таблицы (pg_database, pg_authid)
├── pg_wal/            # WAL (ранее pg_xlog)
├── pg_hba.conf        # Аутентификация
└── postgresql.conf    # Конфигурация
```

**Безопасность на уровне файлов:** PostgreSQL хранит данные в обычных файлах ОС. Права доступа к файлам определяют, кто может читать данные напрямую (минуя PostgreSQL). Правильная настройка: `PGDATA` принадлежит пользователю `postgres`, права `700`.

**Tablespace:** можно создать tablespace на зашифрованном диске (LUKS). Тогда конкретные таблицы хранятся в зашифрованном виде.

### Системный каталог

Все метаданные — в системных таблицах `pg_catalog`:

| Таблица | Описание | Безопасность |
|---------|----------|--------------|
| `pg_class` | Таблицы, индексы, последовательности | Видно всем, но не все колонки |
| `pg_user` | Пользователи | Пароли хешированы |
| `pg_roles` | Роли (расширенная информация) | Зависит от прав |
| `pg_authid` | Аутентификация (включая хеши паролей) | Только суперпользователи |
| `pg_database` | Базы данных | Права на подключение |
| `pg_tablespace` | Tablespaces | Права на создание |

**pg_authid** содержит хеши паролей (SCRAM-SHA-256). Только суперпользователь может читать эту таблицу. Обычный пользователь видит `pg_user`, но без колонки с хешем.

### Расширения и безопасность

| Расширение | Описание | Безопасность |
|------------|----------|--------------|
| **pgcrypto** | Криптография | Шифрование, хеширование, генерация случайных чисел |
| **pgaudit** | Аудит | Логирование SQL-запросов |
| **pg_stat_statements** | Статистика | Performance, обнаружение аномалий |
| **postgis** | Геоданные | Дополнительная поверхность атаки |
| **uuid-ossp** | UUID | Уникальные идентификаторы |

**Принцип:** каждое расширение — это код, выполняющийся внутри процесса PostgreSQL. Расширение может делать всё, что может процесс. Рекомендация: минимизировать установленные расширения, проверять происхождение.

### Репликация

**Физическая репликация:** primary → standby, байт-в-байт копия WAL. Безопасность: standby имеет идентичные данные. Если primary скомпрометирован — standby тоже. Но: standby может быть read-only, с ограниченным доступом.

**Логическая репликация:** primary → subscriber, только изменения определённых таблиц. Безопасность: можно выбирать, что реплицировать. Можно фильтровать строки (publish WHERE). Но DDL не реплицируется — структуру нужно синхронизировать отдельно.

## Сравнение с другими СУБД

| Критерий | PostgreSQL | MySQL | Oracle | SQL Server |
|----------|-----------|-------|--------|------------|
| **Row-Level Security** | ✅ Нative | ✅ (Enterprise) | ✅ | ✅ |
| **Column-Level Encryption** | ✅ (pgcrypto) | ✅ (TDE) | ✅ (TDE) | ✅ |
| **Audit** | ✅ (pgaudit) | ✅ (Enterprise) | ✅ (Unified Audit) | ✅ |
| **SSL/TLS** | ✅ | ✅ | ✅ | ✅ |
| **Kerberos** | ✅ | ✅ | ✅ | ✅ |
| **LDAP** | ✅ | ✅ | ✅ | ✅ |
| **SCRAM** | ✅ (10+) | ⚠️ (8+) | ❌ | ❌ |
| **Open Source** | ✅ | ✅ (Community) | ❌ | ❌ |
| **Комплаенс** | HIPAA, PCI, SOC2 | HIPAA, PCI | Все | Все |

PostgreSQL выигрывает в гибкости и open-source nature. Oracle и SQL Server — в enterprise features «из коробки». MySQL — в простоте, но enterprise features требуют коммерческой версии.

## Уроки из реальных инцидентов

### Инцидент 1: Утечка через pg_authid (2019)

Администратор настроил репликацию pg_catalog на standby. `pg_authid` — системная таблица с хешами паролей. Standby был доступен для чтения аналитикам. Аналитик сделал `SELECT * FROM pg_authid` — получил SCRAM-хеши всех пользователей.

**Хронология:** Хеши переданы злоумышленнику. Брутфорс SCRAM (4096 итераций, salt — медленно, но возможно для слабых паролей). Несколько паролей взломаны. Несанкционированный доступ.

**Ущерб:** Утечка данных. Смена всех паролей. Простой.

**Как защита изменила бы исход:** pg_authid — только для суперпользователей. Репликация pg_catalog — опасна. Аудит: кто читал pg_authid? Мониторинг: аномалия — аналитик читает системную таблицу.

### Инцидент 2: RCE через расширение (2020)

Компания установила расширение из непроверенного источника. Расширение содержало `C` функцию, вызывающую `system()`. Любой пользователь с правом `EXECUTE` на эту функцию — получал shell на сервере.

**Хронология:** Разработчик нашёл расширение на GitHub. Установил без code review. Функция использовалась в приложении. Злоумышленник обнаружил функцию через `pg_proc`. Вызвал с параметром `rm -rf /`.

**Ущерб:** Удаление данных. Восстановление из backup. 6 часов простоя.

**Как защита изменила бы исход:** `shared_preload_libraries` — только доверенные расширения. Code review для любого расширения. `TRUSTED` vs `UNTRUSTED` — только superuser может создавать untrusted. Аудит: кто установил расширение?

## Архитектурные решения

### Defense-in-Depth для PostgreSQL

| Слой | Защита | Контроль |
|------|--------|----------|
| Perimeter | Firewall, Security Groups | Только white-listed IP |
| Transport | SSL/TLS | Все соединения шифрованы |
| Authentication | SCRAM-SHA-256 | Каждый идентифицирован |
| Authorization | ACL + RLS | Только необходимые права |
| Application | Prepared Statements | Защита от SQL Injection |
| Data | pgcrypto, TDE | Шифрование sensitive |
| Audit | pgaudit | Все действия записаны |

### Zero Trust для PostgreSQL

- **Не доверяй приложению** — `User:app` имеет минимум прав
- **Не доверяй сети** — SSL для всех соединений
- **Проверяй каждый запрос** — RLS проверяет строки
- **Аудируй всё** — pgaudit логирует все операции
- **Шифруй данные** — pgcrypto для sensitive колонок

### Метрики безопасности

| Метрика | Целевое значение | Что показывает |
|---------|-----------------|----------------|
| SSL coverage | 100% | Все соединения шифрованы |
| Auth failures | < 5 за час | Brute-force или компрометация |
| Superuser queries | Алерт | Неожиданное использование привилегий |
| DDL changes | Алерт | Изменение структуры |
| RLS violations | 0 | Попытки обхода |

## Подготовка к собеседованию

### Три типичные ошибки кандидатов

**Ошибка 1:** «PostgreSQL безопасна по умолчанию».

После `initdb` PostgreSQL слушает на `localhost`, но `listen_addresses` легко изменить на `*`. По умолчанию `trust` для локальных подключений. `superuser` — `postgres` без пароля (зависит от ОС). Безопасность требует явной конфигурации.

**Ошибка 2:** «RLS заменяет авторизацию».

RLS — дополнительный слой, не замена. RLS проверяет строки, но не заменяет проверку таблиц. Если пользователь имеет `DELETE` на таблицу — RLS не защитит от `DELETE` (если нет соответствующей политики). Нужна комбинация: ACL + RLS.

**Ошибка 3:** «Расширения безопасны, потому что они в официальном репозитории».

Расширения из `contrib` — проверенные. Но любое расширение — это код, выполняющийся внутри процесса. `UNTRUSTED` расширения требуют superuser для создания. Непроверенные расширения из интернета — потенциальный RCE.

### Три ситуационных вопроса

**Вопрос 1:** «Как защитить PostgreSQL в Kubernetes, где Pod-ы эфемерны?»

Ответ: **SSL/TLS** для всех соединений. **SCRAM-SHA-256** для аутентификации. **Network Policies** для ограничения трафика. **Secrets** (Kubernetes Secrets или Vault) для хранения паролей, не в env vars. **Readiness/Liveness probes** — но не раскрывать sensitive info. **Pod Security Policies** — запрет привилегированных контейнеров. **Init containers** для миграций — с ограниченными правами.

**Вопрос 2:** «Как обеспечить compliance PCI DSS для данных карт в PostgreSQL?»

Ответ: **TLS 1.2+** для всех соединений. **SCRAM-SHA-256** для аутентификации. **RLS** — только процессинговый сервис видит полные номера карт. **pgcrypto** — шифрование PAN (Primary Account Number). **pgaudit** — аудит всех обращений к таблице карт. **TDE** (LUKS или EBS encryption) для log.dirs. **Retention** — удаление после обработки. **Regular audits** — проверка кто имеет доступ.

**Вопрос 3:** «В чём разница между физической и логической репликацией с точки зрения безопасности?»

Ответ: **Физическая репликация** — байт-в-байт копия WAL. Включает всё: данные, структура, пользователи, пароли. Если primary скомпрометирован — standby идентичен. Но standby может быть read-only с ограниченным доступом. **Логическая репликация** — только изменения определённых таблиц. Можно фильтровать строки (publish WHERE), выбирать колонки. Но DDL не реплицируется — структуру синхронизировать отдельно. Для безопасности: логическая репликация для partial data sharing, физическая для DR.

- [ ] Как защитить **pg_hba.conf**?
- [ ] Что такое **tablespace** и как использовать для безопасности?
- [ ] Как мониторить необычную активность?

### Ответы на чек-лист

1. **Backend** — процесс ОС, обрабатывающий запросы конкретного клиента. Каждый клиент = отдельный backend. Изоляция на уровне процессов.

2. Данные хранятся в файлах в `PGDATA/base/<OID>/`. Каждая таблица — файл (filenode). WAL — в `PGDATA/pg_wal/`. Права файлов определяют доступ на уровне ОС.

3. **WAL** (Write-Ahead Log) — журнал всех изменений. Записывается перед применением к данным. Обеспечивает durability и используется для репликации.

4. **Физическая** — байт-в-байт копия WAL. **Логическая** — только изменения определённых таблиц. Физическая проще, но реплицирует всё. Логическая гибче, но DDL не реплицируется.

5. **pg_authid** содержит хеши паролей. **pg_shadow** — то же, но только для superuser. **pg_stat_activity** — текущие подключения. **pg_settings** — конфигурация.

6. **pg_authid** — только superuser. Обычный пользователь видит `pg_user`, но без колонки с хешем.

7. Любое **C-based** расширение может выполнять произвольный код. **UNTRUSTED** языки (plpythonu, plperlu) опасны. Рекомендация: минимизировать расширения, использовать `TRUSTED` версии.

8. **pg_hba.conf** должен иметь права 600, принадлежать `postgres`. Никаких `trust` для удалённых подключений. Только `hostssl` + `scram-sha-256`.

9. **Tablespace** — отдельная директория для хранения объектов. Можно создать на зашифрованном диске (LUKS) — тогда объекты в этом tablespace зашифрованы.

10. Мониторить: `pg_stat_activity` (необычные запросы), `pg_stat_statements` (медленные запросы), `pgaudit` (все операции), `pg_authid` доступ (кто читал).

## Ключевые выводы для собеседования

- **PostgreSQL безопасна, но требует настройки** — не из коробки
- **Defense-in-depth** — процессы, файлы, сеть, аутентификация, авторизация, аудит
- **Least privilege** — каждый процесс, каждый пользователь — минимум прав
- **Расширения = код** — проверять, минимизировать, аудит
- **Репликация копирует всё** — включая уязвимости
