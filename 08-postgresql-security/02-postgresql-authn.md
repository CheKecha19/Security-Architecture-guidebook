# Аутентификация в PostgreSQL

## Часть 1: Что такое аутентификация в PostgreSQL?

Представь, что ты подходишь к входу в защищённое здание. У двери стоит охранник. Он спрашивает: «Пропуск?» Ты показываешь документ. Охранник проверяет: действителен ли пропуск, не просрочен ли, твоя ли фотография. Если всё в порядке — пропускает.

**Аутентификация в PostgreSQL** — это именно такой «охранник» для базы данных. Когда приложение или пользователь пытается подключиться, PostgreSQL проверяет: кто это и имеет ли право входа.

## Зачем нужна аутентификация?

### Ситуация 1: Несанкционированный доступ

Разработчик случайно оставил пароль `postgres`/`postgres` на тестовом сервере. Злоумышленник нашёл сервер в интернете и подключился к базе данных production.

### Ситуация 2: Перехват пароля

Приложение подключается к базе по незашифрованному каналу. Злоумышленник перехватывает трафик и видит пароль в открытом виде.

### Ситуация 3: Компрометация учётных данных

Бывший сотрудник знает пароль к базе данных. После увольнения пароль не изменили, и он продолжает иметь доступ.

## Часть 2: Архитектура аутентификации

### Файл pg_hba.conf

**pg_hba.conf** (Host-Based Authentication configuration) — это файл, который определяет: кто может подключаться к каким базам данных, с каких адресов и каким методом.

| Колонка | Описание | Пример |
|---------|----------|--------|
| TYPE | Тип подключения | local (Unix-сокет), host (TCP/IP), hostssl (TLS) |
| DATABASE | Имя базы данных | all, mydb, replication |
| USER | Имя пользователя | all, alice, @admins (группа) |
| ADDRESS | IP-адрес или сеть | 127.0.0.1/32, 10.0.0.0/8, ::1/128 |
| METHOD | Метод аутентификации | scram-sha-256, md5, ldap, cert |

**Пример конфигурации:**

    # Локальные подключения только через Unix-сокет
    local   all             all                                     scram-sha-256
    
    # Удалённые подключения только через SSL
    hostssl all             all             10.0.0.0/8              scram-sha-256
    
    # Запретить всё остальное
    host    all             all             0.0.0.0/0               reject

**Порядок важен.** PostgreSQL проверяет правила сверху вниз. Первое подходящее правило применяется. Поэтому `reject` для `0.0.0.0/0` должен быть в конце.

### Методы аутентификации

| Метод | Описание | Когда использовать | Безопасность |
|-------|----------|-------------------|--------------|
| trust | Без пароля, любой может войти | Только для локальной разработки | ❌ Нет |
| reject | Явный запрет | Блокировка подсетей | ✅ Да |
| md5 | MD5-хеш пароля | Устаревший, не рекомендуется | ⚠️ Средняя |
| password | Пароль в открытом виде | Никогда в production | ❌ Нет |
| scram-sha-256 | SCRAM-SHA-256 (RFC 7677) | Рекомендуемый метод | ✅ Высокая |
| ldap | Аутентификация через LDAP | Enterprise с Active Directory | ✅ Высокая |
| gss | Kerberos (GSSAPI) | Enterprise с Kerberos | ✅ Высокая |
| sspi | Windows SSPI | Windows-домены | ✅ Высокая |
| ident | Имя пользователя ОС | Локальные подключения | ⚠️ Низкая |
| peer | Имя пользователя ОС через Unix-сокет | Локальные подключения | ✅ Средняя |
| radius | RADIUS-сервер | Enterprise с централизованной аутентификацией | ✅ Средняя |
| cert | Клиентский SSL-сертификат | Высокая безопасность, mTLS | ✅ Высокая |

### SCRAM-SHA-256

**SCRAM** (Salted Challenge Response Authentication Mechanism) — современный метод аутентификации, введённый в PostgreSQL 10.

**Как работает:**
1. Клиент отправляет имя пользователя
2. Сервер генерирует случайный **nonce** и отправляет клиенту вместе с **salt**
3. Клиент вычисляет **proof** (доказательство) на основе пароля, salt и nonce
4. Сервер проверяет proof, не получая пароль в открытом виде

**Преимущества:**
- Пароль никогда не передаётся по сети
- Защита от replay-атак (благодаря nonce)
- Защита от man-in-the-middle (mutual authentication)
- Salt уникален для каждого пользователя

**Настройка:**

    # postgresql.conf
    password_encryption = scram-sha-256
    
    # pg_hba.conf
    hostssl all all 0.0.0.0/0 scram-sha-256

## Часть 3: SSL/TLS для PostgreSQL

### Зачем нужен SSL?

Даже с SCRAM-SHA-256 трафик между клиентом и сервером может быть перехвачен. SSL обеспечивает:
- **Конфиденциальность** — данные нельзя прочитать при перехвате
- **Целостность** — данные нельзя изменить без обнаружения
- **Аутентификация сервера** — клиент уверен, что подключается к правильному серверу

### Настройка SSL

    # postgresql.conf
    ssl = on
    ssl_cert_file = 'server.crt'
    ssl_key_file = 'server.key'
    ssl_ca_file = 'root.crt'
    ssl_crl_file = 'root.crl'
    
    # Требовать SSL для всех подключений
    ssl_min_protocol_version = 'TLSv1.2'

### Клиентские настройки SSL

| Параметр connection string | Описание |
|---------------------------|----------|
| sslmode=disable | SSL не используется (❌) |
| sslmode=allow | Предпочитать SSL, но не требовать |
| sslmode=prefer | Предпочитать SSL (по умолчанию) |
| sslmode=require | Требовать SSL, но не проверять сертификат |
| sslmode=verify-ca | Требовать SSL и проверять CA |
| sslmode=verify-full | Требовать SSL, проверять CA и hostname |

**Рекомендация:** `sslmode=verify-full` для production.

### Клиентские сертификаты (mTLS)

Для максимальной безопасности можно требовать клиентский сертификат:

    # pg_hba.conf
    hostssl all all 0.0.0.0/0 cert

Клиент предоставляет сертификат, PostgreSQL проверяет его через CA. Никаких паролей — только криптография.

## Часть 4: LDAP и Active Directory

### Интеграция с LDAP

Для enterprise-сред PostgreSQL может аутентифицировать пользователей через LDAP/AD:

    # pg_hba.conf
    hostssl all all 10.0.0.0/8 ldap ldapserver=ad.company.com ldapprefix="cn=" ldapsuffix=",dc=company,dc=com"

**Варианты:**
- **LDAP** — открытый текст (не рекомендуется без TLS)
- **LDAPS** — LDAP over SSL (порт 636)
- **LDAP + StartTLS** — шифрование после установки соединения

### Kerberos (GSSAPI)

Для Windows/Linux enterprise-сред:

    # postgresql.conf
    krb_server_keyfile = 'FILE:/etc/postgresql/krb5.keytab'
    
    # pg_hba.conf
    hostssl all all 0.0.0.0/0 gss include_realm=0

**Преимущества:**
- Single Sign-On (SSO)
- Нет паролей в конфигурациях
- Централизованное управление через Active Directory

## Часть 5: Лучшие практики и аудит

### Чек-лист безопасности аутентификации

| Проверка | Статус |
|----------|--------|
| Используется scram-sha-256 | ⬜ |
| Запрещён trust/password | ⬜ |
| Только hostssl для удалённых подключений | ⬜ |
| Ограничение IP-адресов (не 0.0.0.0/0) | ⬜ |
| SSL включён и настроен | ⬜ |
| Регулярная ротация паролей | ⬜ |
| Удалённые учётные записи отключены | ⬜ |
| Проверка pg_hba.conf перед применением | ⬜ |

### Типичные ошибки

| Ошибка | Последствие | Исправление |
|--------|-------------|-------------|
| `host all all 0.0.0.0/0 trust` | Полный доступ без пароля | Заменить на scram-sha-256 |
| `local all all trust` | Любой локальный пользователь может войти | Заменить на peer или scram-sha-256 |
| Отсутствие SSL | Перехват паролей | Включить ssl=on |
| Слабые пароли | Brute-force | Политика сложности паролей |

### Команды для проверки

    -- Список пользователей и их методов аутентификации
    SELECT rolname, rolsuper, rolcreaterole, rolcreatedb 
    FROM pg_roles;
    
    -- Проверка текущих подключений
    SELECT usename, client_addr, ssl, ssl_version 
    FROM pg_stat_ssl 
    JOIN pg_stat_activity USING (pid);
    
    -- Проверка настроек SSL
    SHOW ssl;
    SHOW ssl_cert_file;
    SHOW ssl_key_file;

## Вывод

Аутентификация в PostgreSQL — многослойная система. От выбора метода (scram-sha-256) до настройки SSL и интеграции с enterprise-источниками (LDAP, Kerberos). Ключевой принцип: **минимум доверия, максимум проверок.**
