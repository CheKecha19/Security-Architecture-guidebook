# Аутентификация в PostgreSQL: от паролей до Kerberos

## Что такое аутентификация в PostgreSQL?

Представь банк. У входа стоит охранник. Каждый клиент показывает паспорт. Охранник проверяет: это действительно тот человек? Без этой проверки — в банк войдёт кто угодно. Вор. Конкурент. Журналист, ищущий сенсацию.

**Аутентификация в PostgreSQL** — это именно такой «охранник у входа» в базу данных. Каждое подключение — от приложения, от администратора, от реплики — проходит проверку: кто это и имеет ли право входа.

Почему это критично? PostgreSQL — хранилище данных. Если в него попадёт нелегитимный пользователь, он сможет читать, изменять, удалять данные. Аутентификация — первая линия обороны.

## Эволюция и мотивация

Ранние версии PostgreSQL (до 8.4) поддерживали ограниченный набор методов: trust, password, md5, ident. В корпоративных сетях этого хватало: «если ты в сети — ты свой».

Но три тренда изменили требования. Во-первых, **взломы баз данных через слабые пароли**: атаки brute-force, словари, rainbow tables. MD5 — устарел, легко взломать. Во-вторых, **PCI DSS и HIPAA**: требуют strong authentication, MFA, шифрование паролей. В-третьих, **Zero Trust**: аутентификация каждого подключения, даже изнутри сети.

PostgreSQL 8.4 (2009) ввела **SCRAM-SHA-256** через расширение. Версия 10 (2017) сделала SCRAM-SHA-256 встроенным методом. Версия 13 улучшила поддержку **GSSAPI** (Kerberos). Современная PostgreSQL поддерживает 11 методов аутентификации: от простого trust до сложного Kerberos и LDAP.

Эта эволюция отражает рост угроз: от «все в сети доверенные» до «каждый подключающийся должен доказать свою личность».

## Зачем это нужно?

### Сценарий 1: Стандартный пароль

После установки PostgreSQL создаёт пользователя `postgres`. В некоторых ОС (Debian, Ubuntu) этот пользователь имеет peer-аутентификацию через Unix-сокет. Но если администратор включает TCP/IP и не меняет конфигурацию — `postgres` может подключиться без пароля через `trust`.

Злоумышленник сканирует сеть, находит PostgreSQL на порту 5432, подключается как `postgres` — и имеет полный доступ. Создаёт новых пользователей, читает все данные, удаляет таблицы.

С SCRAM-SHA-256: подключение требует пароля. Пароль хранится как SCRAM-хеш — нельзя восстановить. Brute-force замедлен salt и итерациями.

### Сценарий 2: Сетевой доступ

Администратор изменил `listen_addresses` с `localhost` на `*` — «чтобы приложения могли подключаться». Но `pg_hba.conf` остался с `trust` для локальной сети. Теперь любой хост в сети может подключиться без аутентификации.

С правильной настройкой: `hostssl all all 10.0.0.0/24 scram-sha-256` — только SSL, только SCRAM, только из конкретной подсети.

### Сценарий 3: Приложение с паролем в коде

Разработчик написал в коде: `connection_string = "postgresql://app:password123@db/myapp"`. Код в GitHub (публичный репозиторий). Злоумышленник находит строку подключения — и имеет доступ к базе.

С правильной практикой: пароль в Vault/Secrets Manager. Приложение получает пароль runtime. Регулярная ротация. Отдельный пользователь для каждого приложения.

## Основные концепции

### pg_hba.conf — Host-Based Authentication

**pg_hba.conf** (Host-Based Authentication) — файл, определяющий: кто может подключаться, к какой базе, с какого адреса, каким методом.

**Формат:**
```
# TYPE    DATABASE    USER    ADDRESS         METHOD    [OPTIONS]
local     all         all                     trust
hostssl   all         all     10.0.0.0/24     scram-sha-256
host      all         all     0.0.0.0/0       reject
```

**Типы подключений:**

| Тип | Описание | Безопасность |
|-----|----------|--------------|
| `local` | Unix-сокет | Безопаснее TCP (нет сетевого перехвата) |
| `host` | TCP/IP | Требует SSL |
| `hostssl` | TCP/IP + SSL | Рекомендуется для production |
| `hostnossl` | TCP/IP без SSL | ❌ Не рекомендуется |

**Порядок записей критичен!** PostgreScript применяет первую подходящую запись. `reject` для `0.0.0.0/0` должен быть в конце.

### Методы аутентификации

| Метод | Описание | Безопасность | Когда использовать |
|-------|----------|--------------|-------------------|
| **trust** | Без пароля | ❌ Нет | Только localhost dev |
| **reject** | Запретить | ✅ Да | Блокировка |
| **md5** | MD5-хеш пароля | ⚠️ Устарел | Не использовать |
| **password** | Пароль открытым текстом | ❌ Нет | Никогда |
| **scram-sha-256** | SCRAM (RFC 7677) | ✅ Высокая | Рекомендуется |
| **ident** | Имя ОС-пользователя | ⚠️ Низкая | Локально |
| **peer** | Имя ОС через Unix-сокет | ✅ Средняя | Локально |
| **ldap** | LDAP/Active Directory | ✅ Высокая | Enterprise |
| **gss** | Kerberos (GSSAPI) | ✅ Высокая | Enterprise |
| **sspi** | Windows SSPI | ✅ Высокая | Windows-домены |
| **radius** | RADIUS | ✅ Средняя | Enterprise |
| **cert** | Клиентский сертификат | ✅ Высокая | mTLS |

### SCRAM-SHA-256 — рекомендуемый метод

**SCRAM** (Salted Challenge Response Authentication Mechanism) — современный метод, введённый в PostgreSQL 10.

**Как работает:**
1. Клиент отправляет username
2. Сервер отправляет **salt** (случайная строка, 16 байт) и **iteration count** (по умолчанию 4096)
3. Клиент вычисляет **ClientProof** — криптографическое доказательство знания пароля
4. Сервер проверяет ClientProof, используя сохранённый **ServerKey** и **StoredKey**

**Преимущества SCRAM:**
- **Пароль никогда не передаётся** — ни по сети, ни в хешированном виде
- **Защита от replay** — каждая аутентификация уникальна (nonce)
- **Защита от server compromise** — сервер хранит только StoredKey, не пароль
- **Brute-force resistant** — 4096 итераций HMAC-SHA-256

**Настройка:**
```sql
-- Установить метод шифрования
ALTER SYSTEM SET password_encryption = 'scram-sha-256';

-- Создать пользователя
CREATE USER alice WITH PASSWORD 'strong-password';
-- Пароль автоматически хешируется SCRAM

-- Проверить
SELECT rolname, rolpassword FROM pg_authid WHERE rolname = 'alice';
-- Результат: SCRAM-SHA-256$4096:...
```

### SSL/TLS для соединений

SSL шифрует весь трафик между клиентом и сервером. PostgreSQL поддерживает SSL since версии 7.0.

**Настройка сервера:**
```
# postgresql.conf
ssl = on
ssl_cert_file = 'server.crt'
ssl_key_file = 'server.key'
ssl_ca_file = 'ca.crt'
ssl_crl_file = 'crl.pem'
ssl_min_protocol_version = 'TLSv1.2'
```

**Клиентские настройки (sslmode):**

| Режим | Описание | Безопасность |
|-------|----------|--------------|
| `disable` | Без SSL | ❌ Нет |
| `allow` | SSL, если доступен | ⚠️ |
| `prefer` | Предпочитать SSL | ⚠️ |
| `require` | Требовать SSL | ✅ |
| `verify-ca` | Проверить CA | ✅ |
| `verify-full` | Проверить CA и hostname | ✅✅ |

**Рекомендация для production:** `sslmode=verify-full`.

### LDAP и Active Directory

**LDAP** — интеграция с корпоративными directory-сервисами.

```
# pg_hba.conf
hostssl all all 10.0.0.0/24 ldap ldapserver=ad.company.com \
  ldapprefix="cn=" ldapsuffix=",dc=company,dc=com"
```

**LDAPS** (LDAP over SSL) — порт 636. **StartTLS** — переключение на TLS после установки соединения.

**Kerberos (GSSAPI):**
```
# postgresql.conf
krb_server_keyfile = 'FILE:/etc/postgresql/krb5.keytab'

# pg_hba.conf
hostssl all all 0.0.0.0/0 gss include_realm=0
```

## Сравнение методов аутентификации

| Критерий | SCRAM-SHA-256 | Kerberos | LDAP | mTLS |
|----------|---------------|----------|------|------|
| **Сложность** | Низкая | Высокая | Средняя | Средняя |
| **Внешняя инфраструктура** | Нет | Да (KDC) | Да (LDAP Server) | Да (PKI) |
| **Single Sign-On** | Нет | Да | Да | Нет |
| **Пароль по сети** | Нет | Нет | Иногда (StartTLS) | Нет |
| **Brute-force resistant** | Да | Да | Зависит от LDAP | N/A |
| **Certificate management** | Нет | Нет | Нет | Да |
| **PostgreSQL версия** | 10+ | 8.0+ | 8.0+ | 9.2+ |

## Уроки из реальных инцидентов

### Инцидент 1: Утечка через MD5 (2017)

Компания использовала `md5` аутентификацию. Злоумышленник получил доступ к `pg_authid` (через SQL Injection в приложении). Извлёк MD5-хеши паролей.

**Хронология:** MD5 — устарел, уязвим к collision attacks и rainbow tables. Злоумышленник использовал предвычисленные rainbow tables для MD5. За неделю взломаны 60% паролей. Несанкционированный доступ к базе.

**Ущерб:** Утечка данных клиентов. Регуляторный штраф. Смена всех паролей.

**Как SCRAM изменил бы исход:** SCRAM использует salt (уникальный для каждого пользователя) и итерации (4096). Rainbow tables бесполезны — нужен per-user brute-force. 4096 итераций HMAC-SHA-256 — слишком медленно для практического brute-force.

### Инцидент 2: Компрометация через trust (2020)

Администратор настроил `local all all trust` в pg_hba.conf для «удобства разработки». Но PostgreSQL был доступен через SSH tunnel. Злоумышленник получил SSH-доступ (через скомпрометированный ключ), создал SSH tunnel к PostgreSQL, подключился через Unix-сокет — и аутентифицировался без пароля.

**Хронология:** SSH-ключ разработчика был в `.ssh/id_rsa` с правами 644. Злоумышленник прочитал, подключился к серверу, создал tunnel, подключился к PostgreSQL как `postgres`.

**Ущерб:** Полный доступ к базе. Эксфильтрация данных. Удаление audit logs.

**Как защита изменила бы исход:** `local all all peer` вместо `trust` — аутентификация через Unix-пользователя. SSH-ключи с правами 600. Отключение SSH-туннелирования, если не нужно. Мониторинг подключений.

## Архитектурные решения

### Выбор метода аутентификации

| Сценарий | Рекомендация | Почему |
|----------|-------------|--------|
| Новый проект, нет IAM | **SCRAM-SHA-256** | Просто, безопасно, из коробки |
| Enterprise с AD | **Kerberos (GSSAPI)** | SSO, централизованное управление |
| Облако | **SCRAM-SHA-256 + IAM** | Комбинация |
| Kubernetes | **mTLS** | Service Mesh управляет автоматически |
| Высочайшая безопасность | **SCRAM + mTLS** | Defense-in-depth |

### Защита pg_hba.conf

```
# Права файла
chmod 600 pg_hba.conf
chown postgres:postgres pg_hba.conf

# Структура записей:
# 1. Локальные подключения (peer или scram)
# 2. Удалённые подключения (hostssl + scram)
# 3. Репликация (отдельная подсеть)
# 4. Запрет всего остального (reject)
```

### Мониторинг аутентификации

| Метрика | Целевое значение | Действие |
|---------|-----------------|----------|
| Failed login attempts | < 5 за 5 минут | Блокировка IP (fail2ban) |
| Successful logins from new IP | Алерт | Проверить легитимность |
| Superuser connections | Логировать все | Ревью |
| SCRAM usage | 100% | Убрать MD5, password |

## Подготовка к собеседованию

### Три типичные ошибки кандидатов

**Ошибка 1:** «SCRAM и MD5 — по сути одно и то же, оба хешируют пароль».

MD5 хеширует пароль один раз, без salt. SCRAM использует salt, итерации (4096), и challenge-response. MD5 — устарел, SCRAM — современный стандарт. MD5-хеш можно взломать rainbow tables. SCRAM — практически невозможно.

**Ошибка 2:** «SSL достаточно, не нужна аутентификация».

SSL шифрует трафик, но не проверяет, кто подключается (без mTLS). Любой, кто может подключиться по SSL, считается легитимным. Нужна комбинация: SSL + SCRAM или mTLS.

**Ошибка 3:** «Один пользователь на все приложения».

Shared account невозможно аудировать. Невозможно понять, какое приложение выполнило запрос. Невозможно отозвать доступ одного приложения без влияния на другие. Нужен отдельный пользователь на каждое приложение.

### Три ситуационных вопроса

**Вопрос 1:** «Как защититься от brute-force атаки на PostgreSQL?»

Ответ: **SCRAM-SHA-256** (медленный brute-force из-за итераций). **Fail2ban** — блокировка IP после 5 неудачных попыток. **Connection throttling** — `max_connections`, `superuser_reserved_connections`. **Network segmentation** — PostgreSQL доступен только из application tier. **Monitoring** — алерты на failed logins. **No default passwords** — смена всех дефолтных паролей при установке.

**Вопрос 2:** «Как мигрировать с MD5 на SCRAM-SHA-256 без downtime?»

Ответ: **Постепенная миграция:**
1. `ALTER SYSTEM SET password_encryption = 'scram-sha-256'`
2. `ALTER USER ... PASSWORD 'same-password'` — обновляет хеш до SCRAM
3. Обновить pg_hba.conf: `md5` → `scram-sha-256`
4. Перезагрузить PostgreSQL (SIGHUP или restart)
5. Удалить MD5-хеши: `UPDATE pg_authid SET rolpassword = NULL WHERE rolpassword LIKE 'md5%'`
6. Заставить пользователей сменить пароль

**Вопрос 3:** «Как обеспечить аутентификацию для PostgreSQL в Docker/Kubernetes?»

Ответ: **SCRAM-SHA-256** для приложений. **Init container** для инициализации пользователей. **Kubernetes Secrets** (или Vault sidecar) для паролей. **Network Policy** — доступ только из pod-ов приложения. **Peer аутентификация** для init/backup контейнеров через Unix-сокет. **mTLS** для высокой безопасности.

### Ответы на чек-лист

1. **Аутентификация** — проверка личности (кто ты?). **Авторизация** — проверка прав (что тебе можно?). Сначала аутентификация, потом авторизация.

2. **SCRAM-SHA-256:** клиент отправляет username → сервер отправляет salt + iterations → клиент вычисляет proof → сервер проверяет. Пароль никогда не передаётся.

3. **MD5** устарел: collision attacks, rainbow tables, быстрый brute-force. Не использовать в production.

4. **Методы:** trust, reject, md5, password, scram-sha-256, ident, peer, ldap, gss, sspi, radius, cert.

5. **pg_hba.conf** — файл Host-Based Authentication. Определяет: тип подключения, базу, пользователя, адрес, метод аутентификации. Применяет первую подходящую запись.

6. **Kerberos:** настроить `krb_server_keyfile` в postgresql.conf. Использовать `gss` в pg_hba.conf. Требует KDC (Active Directory).

7. **LDAP:** указать `ldapserver`, `ldapprefix`, `ldapsuffix` в pg_hba.conf. Использовать LDAPS (порт 636) или StartTLS.

8. **mTLS:** настроить `ssl_ca_file`, требовать `clientcert=1` в pg_hba.conf. Клиент предъявляет сертификат. Идентификация по CN.

9. **Мониторинг:** `log_connections = on`, `log_disconnections = on`. Анализ `pg_stat_activity`. Fail2ban. SIEM алерты.

10. **Миграция:** `ALTER SYSTEM SET password_encryption = 'scram-sha-256'`. `ALTER USER ... PASSWORD`. Обновить pg_hba.conf. Перезагрузить.

## Ключевые выводы для собеседования

- **SCRAM-SHA-256 — default choice** — просто, безопасно, стандартизировано
- **Никогда MD5 в production** — устарел, уязвим
- **SSL + SCRAM — минимум** — не SSL вместо аутентификации
- **Separate users per app** — для аудита и ротации
- **pg_hba.conf — critical file** — права 600, правильный порядок записей
