# OWASP Threat Modeling: Моделирование угроз от OWASP

## Что такое OWASP Threat Modeling?

Представь строительство моста. Инженер не просто рисует красивые арки. Он считает: какой ветер, какой вес, какие материалы, какая нагрузка, какие течения, какие землетрясения возможны. Для каждого фактора — запас прочности.

**OWASP Threat Modeling** — это систематический подход к поиску угроз в приложениях. Он помогает ответить на вопросы: что может пойти не так? Где у нас слабые места? Как защититься?

OWASP (Open Web Application Security Project) — некоммерческая организация, которая занимается безопасностью приложений. Их методология — практичная, понятная и ориентирована на разработчиков.

## Зачем нужен OWASP Threat Modeling?

### Ситуация 1: Новое веб-приложение

Команда начинает разработку HR-платформы. Архитектор спроектировал систему. Но он думал о функциональности, не о безопасности. Где уязвимости?

OWASP Threat Modeling проверяет архитектуру. Находит: незащищённые API, отсутствие авторизации, утечки данных. До того, как написана первая строка кода.

### Ситуация 2: Legacy-система

Приложение работает пять лет. Никто не помнит, почему так спроектировано. Нужно понять, какие угрозы существуют.

OWASP Threat Modeling помогает разобраться в архитектуре и найти угрозы.

### Ситуация 3: Проверка перед аудитом

Компания готовится к SOC 2 аудиту. Нужно показать, что угрозы были проанализированы.

OWASP Threat Modeling — документированный процесс. Результат — отчёт для аудиторов.

## Четыре шага OWASP Threat Modeling

### Шаг 1: Decompose the Application (Декомпозиция)

**Что делаем:** Создаём модель приложения. Показываем компоненты, потоки данных, trust boundaries.

**Инструмент:** Data Flow Diagram (DFD).

**Элементы DFD:**
| Элемент | Описание | Пример |
|---------|----------|--------|
| External Entity | Внешний пользователь или система | Кандидат, HR-менеджер |
| Process | Процесс в приложении | API, веб-сервер |
| Data Store | Хранилище данных | База данных, S3 |
| Data Flow | Поток данных | HTTP-запрос, SQL-запрос |
| Trust Boundary | Граница доверия | Интернет → DMZ → Internal |

### Шаг 2: Determine and Rank Threats (Определение угроз)

**Что делаем:** Применяем STRIDE к каждому элементу DFD и потоку.

**Таблица угроз:**

| Компонент | S | T | R | I | D | E |
|-----------|---|---|---|---|---|---|
| Веб-приложение | Подмена кандидата | Изменение запроса | Отрицание загрузки | Утечка резюме | DDoS | Получение shell |
| База данных | Подмена админа | Изменение данных | Отрицание запроса | Утечка всех данных | Блокировка | Повышение прав |
| S3 | Подмена ключа | Изменение файла | Отрицание доступа | Публичный bucket | — | — |

### Шаг 3: Determine Countermeasures and Mitigation (Контрмеры)

**Что делаем:** Для каждой угрозы определяем контрмеру.

| Угроза | Контрмера | Приоритет |
|--------|-----------|-----------|
| SQL-инъекция | Параметризованные запросы | Критический |
| XSS | Output encoding | Высокий |
| IDOR | Авторизация на уровне объекта | Высокий |
| CSRF | CSRF-токены | Средний |
| DDoS | Rate limiting | Средний |

### Шаг 4: Rank Mitigations (Приоритизация)

**Что делаем:** Ранжируем угрозы и митигации. Исправляем критичные первыми.

| Угроза | Вероятность | Влияние | Риск | Приоритет |
|--------|-------------|---------|------|-----------|
| SQL-инъекция | Высокая | Критическое | 9 | 1 |
| XSS | Высокая | Высокое | 7 | 2 |
| IDOR | Средняя | Высокое | 6 | 3 |
| CSRF | Низкая | Среднее | 3 | 4 |

## OWASP Threat Modeling Cheat Sheet

OWASP выпустил cheat sheet — краткое руководство:

### Вопросы для каждого компонента

| Вопрос | Зачем |
|--------|-------|
| Кто взаимодействует с компонентом? | Понять поверхность атаки |
| Какие данные обрабатываются? | Определить чувствительность |
| Где границы доверия? | Найти точки перехода |
| Какие протоколы используются? | Оценить защищённость |
| Какие механизмы аутентификации? | Проверить надёжность |
| Какие механизмы авторизации? | Проверить права |
| Какие данные выходят из системы? | Найти утечки |

### Шаблон отчёта

1. Описание приложения
2. DFD
3. Список угроз (STRIDE)
4. Оценка рисков
5. Контрмеры
6. Приоритеты
7. План реализации

## OWASP vs другие методологии

| Критерий | OWASP | STRIDE | PASTA |
|----------|-------|--------|-------|
| Фокус | Веб-приложения | Любые системы | Бизнес-риск |
| Сложность | Низкая | Низкая | Высокая |
| Время | Часы | Часы | Недели |
| Требует DFD | Да | Да | Да |
| Результат | Cheat sheet + отчёт | Список угроз | Risk score |

## Ограничения OWASP Threat Modeling

### Фокус на веб

Оптимизирован для веб-приложений. Для IoT, embedded, industrial — другие подходы.

### Зависимость от качества DFD

Плохая схема → плохие угрозы. Нужно время на хорошую декомпозицию.

### Не покрывает бизнес-риск

Показывает технические угрозы. Не считает стоимость риска. Для этого — PASTA.

### Ответы на чек-лист

1. **OWASP Threat Modeling** — систематический подход к поиску угроз в веб-приложениях от OWASP. Помогает найти уязвимости до написания кода.

2. **Четыре шага**: Decompose (создать DFD), Determine Threats (применить STRIDE), Determine Countermeasures (найти защиту), Rank Mitigations (приоритизировать).

3. **DFD** (Data Flow Diagram) — схема потоков данных. Показывает компоненты, потоки, хранилища, внешние сущности и границы доверия.

4. **Элементы DFD**: External Entity (внешний пользователь), Process (процесс), Data Store (хранилище), Data Flow (поток), Trust Boundary (граница доверия).

5. **Применение STRIDE к DFD**: для каждого компонента и потока задаём шесть вопросов. Для процессов — все шесть. Для потоков — Spoofing, Tampering, Info Disclosure. Для хранилищ — Tampering, Info Disclosure, DoS.

6. **Приоритизация**: оцениваем вероятность и влияние. Риск = вероятность × влияние. Сортируем по убыванию риска.

7. **Trust Boundary** — граница, где уровень доверия меняется. Например, переход из интернета в DMZ, из DMZ во внутреннюю сеть. Там, где уровень доверия падает — повышенный риск.

8. **OWASP** фокусируется на веб-приложениях и технических угрозах. **PASTA** — на бизнес-рисках и стоимости. OWASP быстрее, PASTA глубже.

9. **OWASP Cheat Sheet** — краткое руководство по threat modeling. Содержит вопросы для каждого компонента, шаблон отчёта, примеры.

10. **Ограничения**: фокус на веб-приложениях, зависимость от качества DFD, не покрывает бизнес-риск, требует опыта в security.
