# CVSS: Оценка серьёзности уязвимостей

## Что такое CVSS?

Представь врача, который осматривает пациента. У пациента температура, боль в груди, сыпь. Врач не говорит "всё плохо". Он измеряет: температура 39, давление повышенное, пульс учащённый. На основе цифр ставит диагноз: лёгкая, средняя или тяжёлая степень.

**CVSS** — это система оценки серьёзности уязвимостей. Она переводит сложную техническую проблему в простое число от 0 до 10. Чем выше число — тем опаснее уязвимость.

CVSS разработан организацией FIRST. Используется повсеместно: в NVD, в сканерах уязвимостей, в бюллетенях безопасности.

## Зачем нужен CVSS?

### Ситуация 1: Приоритизация патчей

Система имеет сто уязвимостей. На исправление всех — недели. Нужно выбрать, с чего начать.

CVSS даёт число каждой уязвимости. Сортируем по убыванию. Исправляем девятки и десятки первыми. Тройки — потом.

### Ситуация 2: SLA для исправления

Компания устанавливает правило: критичные уязвимости (9.0+) — за 24 часа, высокие (7.0-8.9) — за неделю, средние — за месяц.

CVSS определяет категорию. SLA работает автоматически.

### Ситуация 3: Отчётность

Руководство спрашивает: "Насколько мы уязвимы?" Ответ: "У нас 3 критичные уязвимости, 12 высоких, 45 средних". Конкретно и понятно.

## Версии CVSS

### CVSS v3.1

Текущая версия. Три группы метрик:

**Base Metrics** — основные характеристики уязвимости:
| Метрика | Варианты | Описание |
|---------|----------|----------|
| Attack Vector (AV) | Network, Adjacent, Local, Physical | Откуда может быть проведена атака |
| Attack Complexity (AC) | Low, High | Сложность атаки |
| Privileges Required (PR) | None, Low, High | Какие права нужны |
| User Interaction (UI) | None, Required | Нужно ли действие пользователя |
| Scope (S) | Unchanged, Changed | Влияет ли атака за пределы уязвимого компонента |
| Confidentiality (C) | None, Low, High | Влияние на конфиденциальность |
| Integrity (I) | None, Low, High | Влияние на целостность |
| Availability (A) | None, Low, High | Влияние на доступность |

**Temporal Metrics** — временные характеристики:
| Метрика | Варианты | Описание |
|---------|----------|----------|
| Exploit Code Maturity (E) | Not Defined, Unproven, Proof of Concept, Functional, High, Not Defined | Наличие эксплойта |
| Remediation Level (RL) | Not Defined, Official Fix, Temporary Fix, Workaround, Unavailable, Not Defined | Доступность исправления |
| Report Confidence (RC) | Not Defined, Unknown, Reasonable, Confirmed, Not Defined | Уверенность в информации |

**Environmental Metrics** — контекст окружения:
| Метрика | Варианты | Описание |
|---------|----------|----------|
| Confidentiality Requirement (CR) | Not Defined, Low, Medium, High | Важность конфиденциальности |
| Integrity Requirement (IR) | Not Defined, Low, Medium, High | Важность целостности |
| Availability Requirement (AR) | Not Defined, Low, Medium, High | Важность доступности |

### Расчёт базового балла

Базовый балл считается по формуле из векторов. Не вручную, а через калькулятор.

Пример: уязвимость с AV:N, AC:L, PR:N, UI:N, C:H, I:H, A:H → Base Score: 9.8 (Critical).

### Шкала оценки

| Балл | Степень | Цвет | Действие |
|------|---------|------|----------|
| 0.0 | None | — | — |
| 0.1-3.9 | Low | 🟢 | Плановое исправление |
| 4.0-6.9 | Medium | 🟡 | Исправление в ближайшем релизе |
| 7.0-8.9 | High | 🟠 | Исправление в течение недели |
| 9.0-10.0 | Critical | 🔴 | Немедленное исправление |

## CVSS в практике

### Пример: Log4Shell (CVE-2021-44228)

| Метрика | Значение | Описание |
|---------|----------|----------|
| AV | Network | Атака через сеть |
| AC | Low | Простая атака |
| PR | None | Не требует прав |
| UI | None | Не требует пользователя |
| C | High | Полная утечка данных |
| I | High | Полная потеря целостности |
| A | High | Полная потеря доступности |

**Base Score: 10.0 (Critical)**

Это максимум. Уязвимость критична по всем параметрам.

### Пример: Утечка информации через логи

| Метрика | Значение | Описание |
|---------|----------|----------|
| AV | Local | Требует локального доступа |
| AC | High | Сложная атака |
| PR | High | Требует админских прав |
| C | Low | Частичная утечка |
| I | None | Нет влияния на целостность |
| A | None | Нет влияния на доступность |

**Base Score: 2.3 (Low)**

### Использование в организации

| Процесс | Как используется |
|---------|-----------------|
| Сканирование | CVSS сортирует уязвимости |
| Patch management | SLA по CVSS |
| Отчётность | Метрики: средний CVSS, количество критичных |
| Risk acceptance | Уязвимости с CVSS < 4 могут быть приняты |
| Compliance | Требование: нет критичных уязвимостей |

## Критика CVSS

### Не учитывает контекст

Уязвимость с CVSS 9.8 на внутреннем тестовом сервере — не страшно. Та же уязвимость на публичном production — катастрофа. CVSS одинаков для обеих.

Решение: использовать Environmental Metrics. Они учитывают контекст окружения.

### Exploitability vs Impact

CVSS v3.1 увеличил вес Exploitability. Уязвимости, которые легко эксплуатировать, получают высокий балл, даже если impact небольшой.

Решение: смотреть не только на общий балл, но и на вектор.

### Не отражает реальную опасность

Некоторые уязвимости с высоким CVSS редко эксплуатируются. Некоторые с низким — активно используются.

Решение: комбинировать CVSS с threat intelligence. Смотреть: эксплуатируется ли уязвимость в реальности?

### CVSS v2 vs v3

CVSS v2 давал более низкие баллы. Переход на v3 поднял баллы многих уязвимостей. Это создало путаницу.

## CVSS и другие системы

### CVSS vs EPSS

| Система | Что оценивает |
|---------|--------------|
| CVSS | СЕРЬЁЗНОСТЬ уязвимости (теоретическая) |
| EPSS | ВЕРОЯТНОСТЬ эксплуатации (реальная) |

EPSS (Exploit Prediction Scoring System) предсказывает вероятность эксплуатации в ближайшие 30 дней. Основан на машинном обучении.

Комбинация: CVSS × EPSS = приоритет. Высокий CVSS + высокий EPSS = исправлять немедленно.

### CVSS vs SSVC

SSVC (Stakeholder-Specific Vulnerability Categorization) — альтернатива CVSS. Вместо числа — категории: Track, Track*, Attend, Act.

SSVC учитывает контекст: техническое влияние, автоматизацию эксплуатации, важность системы.

### Ответы на чек-лист

1. **CVSS** — Common Vulnerability Scoring System. Стандарт оценки серьёзности уязвимостей. Переводит техническую проблему в число 0-10.

2. **Три группы метрик**: Base Metrics (основные характеристики), Temporal Metrics (временные), Environmental Metrics (контекст окружения).

3. **Максимальный балл** — 10.0. Минимальный — 0.0.

4. **CVSS v2** давал более низкие баллы, v3 увеличил вес Exploitability. v3.1 — текущая версия с уточнениями.

5. **Base Metrics**: Attack Vector, Attack Complexity, Privileges Required, User Interaction, Scope, Confidentiality, Integrity, Availability. Описывают фундаментальные свойства уязвимости.

6. **Использование в организации**: сортировка уязвимостей по приоритету, SLA на исправление (критичные — 24ч), отчётность руководству, risk acceptance (низкие можно принять).

7. **CVSS не учитывает контекст**, потому что оценивает уязвимость абстрактно. На тестовом и production сервере балл одинаков. Решение: Environmental Metrics для учёта контекста.

8. **EPSS** — Exploit Prediction Scoring System. Предсказывает вероятность эксплуатации уязвимости в ближайшие 30 дней. Основан на машинном обучении.

9. **Комбинация CVSS + threat intelligence**: высокий CVSS + активная эксплуатация в реальности = максимальный приоритет. Высокий CVSS + нет эксплуатации = пониженный приоритет.

10. **Альтернативы**: EPSS (вероятность эксплуатации), SSVC (категории вместо чисел), VPR (Tenable), Risk-based scoring (комбинация многих факторов).
