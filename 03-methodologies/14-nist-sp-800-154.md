# NIST SP 800-154: Руководство по моделированию угроз

## Что такое NIST SP 800-154?

Представь строительные нормы. Не просто советы, а официальный документ, которому следуют все инженеры. В нем написано: как строить фундамент, какие материалы использовать, какие проверки проводить.

**NIST SP 800-154** — это руководство от Национального института стандартов и технологий США по моделированию угроз. Оно не изобретает новую методологию. Оно собирает лучшие практики и описывает, как применять их правильно.

SP 800-154 — это фреймворк, который объединяет разные подходы: STRIDE, PASTA, attack trees, и другие. Он помогает выбрать правильный метод для конкретной ситуации.

## Зачем нужен NIST SP 800-154?

### Ситуация 1: Выбор методологии

Компания хочет начать моделировать угрозы. Но какую методологию выбрать? STRIDE? PASTA? Attack trees?

SP 800-154 помогает выбрать. В зависимости от целей, ресурсов, контекста — разные методологии.

### Ситуация 2: Стандартизация

Компания хочет, чтобы все команды моделировали угрозы одинаково. Нужен общий фреймворк.

SP 800-154 — стандарт. Описывает: как начать, какие шаги, какие артефакты, как оценивать.

### Ситуация 3: Compliance

Аудитор спрашивает: "Как вы моделируете угрозы?" Нужно показать процесс.

SP 800-154 — авторитетный источник. "Мы следуем рекомендациям NIST".

## Структура NIST SP 800-154

### Раздел 1: Introduction (Введение)

Почему моделирование угроз важно. Что дает организации.

### Раздел 2: Fundamentals (Основы)

Базовые концепции:
- Что такое threat modeling
- Какие существуют методологии
- Как threat modeling связано с risk management
- Жизненный цикл моделирования угроз

### Раздел 3: Methodologies (Методологии)

Обзор методологий:

| Методология | Когда использовать |
|-------------|-------------------|
| STRIDE | Быстрая проверка приложений |
| PASTA | Глубокий анализ с бизнес-риском |
| Attack Trees | Понимание путей атаки |
| Kill Chain | Анализ этапов атаки |
| CVSS | Оценка уязвимостей |

### Раздел 4: Process (Процесс)

Общий процесс моделирования угроз:

**Шаг 1: Scope definition**
- Что моделируем?
- Какие границы?
- Какие активы?

**Шаг 2: Model creation**
- Создаем модель системы (DFD, architecture diagram)
- Определяем trust boundaries
- Описываем потоки данных

**Шаг 3: Threat identification**
- Применяем выбранную методологию
- Находим угрозы
- Документируем

**Шаг 4: Risk assessment**
- Оцениваем каждую угрозу
- Определяем приоритеты

**Шаг 5: Mitigation**
- Проектируем контрмеры
- Планируем реализацию

**Шаг 6: Validation**
- Проверяем, что угрозы закрыты
- Обновляем модель

### Раздел 5: Integration (Интеграция)

Как встроить threat modeling в SDLC:

| Этап SDLC | Как применять |
|-----------|---------------|
| Требования | Определяем security requirements |
| Проектирование | Моделируем угрозы архитектуры |
| Разработка | Проверяем код на найденные угрозы |
| Тестирование | Проверяем митигации |
| Эксплуатация | Мониторинг, обновление модели |

### Раздел 6: Metrics (Метрики)

Как измерять эффективность:
- Количество найденных угроз
- Количество закрытых угроз
- Время от обнаружения до митигации
- Покрытие системы моделированием

## NIST SP 800-154 и другие стандарты

### Связь с NIST Cybersecurity Framework

| CSF Function | Как threat modeling помогает |
|--------------|------------------------------|
| Identify | Находит активы и угрозы |
| Protect | Определяет контрмеры |
| Detect | Определяет индикаторы |
| Respond | Помогает готовить playbooks |
| Recover | Оценивает ущерб |

### Связь с ISO 27001

Threat modeling — часть risk assessment по ISO 27001. Результаты — input для risk register.

## Ограничения NIST SP 800-154

### Обзорный характер

SP 800-154 — обзорный документ. Не заменяет глубокое изучение конкретных методологий.

### Фокус на федеральных системах

Разработан для правительственных систем США. Нужна адаптация для коммерческих.

### Не конкретная методология

Не говорит "используйте STRIDE всегда". Помогает выбрать. Выбор остается за экспертом.

### Ответы на чек-лист

1. **NIST SP 800-154** — руководство от NIST по моделированию угроз. Собирает лучшие практики и помогает выбрать методологию.

2. **Разделы**: Introduction (зачем), Fundamentals (основы), Methodologies (обзор методов), Process (процесс), Integration (в SDLC), Metrics (метрики).

3. **Выбор методологии**: STRIDE — быстрая проверка приложений. PASTA — глубокий анализ с бизнес-риском. Attack Trees — пути атаки. Kill Chain — этапы атаки. CVSS — оценка уязвимостей.

4. **Шесть шагов**: Scope definition (границы), Model creation (модель), Threat identification (угрозы), Risk assessment (оценка), Mitigation (контрмеры), Validation (проверка).

5. **Интеграция в SDLC**: Требования — security requirements. Проектирование — моделирование архитектуры. Разработка — проверка кода. Тестирование — проверка митигаций. Эксплуатация — мониторинг.

6. **Метрики**: количество найденных/закрытых угроз, время до митигации, покрытие системы, средний риск.

7. **Связь с NIST CSF**: Identify — находит активы. Protect — определяет контрмеры. Detect — определяет индикаторы. Respond — помогает playbooks. Recover — оценивает ущерб.

8. **Связь с ISO 27001**: threat modeling — часть risk assessment. Результаты — input для risk register.

9. **Использовать NIST SP 800-154** при: выборе методологии, стандартизации процесса, compliance, обучении команд, интеграции в SDLC.

10. **Ограничения**: обзорный характер (не заменяет глубокое изучение), фокус на федеральных системах, не конкретная методология (требует выбора).
