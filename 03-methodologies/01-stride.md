# STRIDE: Шесть категорий угроз

## Что такое STRIDE?

Представь, что ты проверяешь безопасность своего дома. Ты смотришь на каждую дверь, окно, замок, сигнализацию и задаёшь вопросы. Может ли кто-то притвориться твоим гостем? Может ли кто-то подменить замок? Может ли кто-то войти незамеченным?

**STRIDE** — это методология, которая помогает задавать правильные вопросы. Она разбивает все возможные угрозы на шесть категорий. Каждая буква — это одна категория. Если ты проверил все шесть — значит, посмотрел на систему со всех сторон.

STRIDE придумали в Microsoft. Она простая, понятная и работает почти для любой системы. Не нужно быть экспертом по безопасности, чтобы начать её использовать.

## Зачем нужна STRIDE?

### Ситуация 1: Новый сервис HR-платформы

Команда разрабатывает сервис для загрузки резюме. Разработчики думают: "Файлы загружаются, сохраняются, всё работает". Но безопасность?

STRIDE заставляет задать вопросы:
- Может ли кто-то притвориться HR-менеджером и скачать чужие резюме?
- Может ли кто-то изменить резюме после загрузки?
- Может ли кто-то отрицать, что загружал файл?
- Может ли кто-то увидеть резюме, к которому не имеет доступа?
- Может ли кто-то забить сервис фейковыми загрузками?
- Может ли загрузивший файл получить права администратора?

### Ситуация 2: API для партнёров

HR-платформа открывает API для интеграции с job boards. STRIDE помогает найти угрозы до того, как API станет публичным.

### Ситуация 3: Ревью архитектуры

Архитектор спроектировал новую систему. Перед согласованием нужно проверить, не упущены ли угрозы. STRIDE — быстрый способ провести такую проверку.

## Шесть категорий STRIDE

### S — Spoofing (Подмена личности)

**Что это:** Злоумышленник притворяется кем-то другим. Он не является тем, за кого себя выдаёт.

**Примеры:**
- Фишинговое письмо от имени CEO
- Поддельный JWT-токен
- Использование украденных учётных данных
- Подмена IP-адреса
- Подделка цифровой подписи

**Что защищает:**
- Многофакторная аутентификация
- Цифровые подписи
- Сертификаты
- Биометрия
- Anti-spoofing механизмы

**Вопрос для проверки:** Может ли кто-то притвориться другим пользователем или системой?

### T — Tampering (Изменение данных)

**Что это:** Злоумышленник изменяет данные без авторизации. Он модифицирует то, что не должен трогать.

**Примеры:**
- Изменение резюме в базе данных
- Модификация HTTP-запроса в пути
- Изменение логов
- Подмена конфигурационного файла
- Манипуляция с checksum

**Что защищает:**
- Хеширование
- Цифровые подписи
- Целостность данных
- Версионирование
- Неизменяемые логи (WORM)

**Вопрос для проверки:** Может ли кто-то изменить данные без обнаружения?

### R — Repudiation (Отрицание действий)

**Что это:** Пользователь совершает действие, а потом отрицает это. Система не может доказать, что действие было совершено.

**Примеры:**
- Пользователь утверждает, что не переводил деньги
- Администратор отрицает, что удалял файлы
- Злоумышленник стирает следы из логов
- Отсутствие аудита критичных операций

**Что защищает:**
- Аудит всех действий
- Неизменяемые логи
- Цифровые подписи транзакций
- Timestamp с доверенным источником времени
- Non-repudiation механизмы

**Вопрос для проверки:** Может ли кто-то отрицать совершённые действия?

### I — Information Disclosure (Утечка информации)

**Что это:** Данные становятся доступны тем, кому не должны. Это самая распространённая угроза.

**Примеры:**
- SQL-инъекция, возвращающая данные других пользователей
- Отладочная информация в production
- Утечка через ошибки (stack traces)
- Неправильные права доступа к файлам
- Перехват трафика без шифрования

**Что защищает:**
- Шифрование (at rest и in transit)
- RBAC, ABAC
- Маскирование данных
- Input validation
- Output encoding

**Вопрос для проверки:** Может ли кто-то получить доступ к данным без разрешения?

### D — Denial of Service (Отказ в обслуживании)

**Что это:** Система становится недоступной для легитимных пользователей. Злоумышленник "забивает" ресурсы.

**Примеры:**
- DDoS-атака на веб-сервер
- Флуд запросами к API
- Исчерпание дискового пространства
- Бесконечный цикл, потребляющий CPU
- Заполнение очереди сообщений

**Что защищает:**
- Rate limiting
- DDoS protection
- Load balancing
- Auto-scaling
- Quotas и throttling

**Вопрос для проверки:** Может ли кто-то сделать систему недоступной?

### E — Elevation of Privilege (Повышение привилегий)

**Что это:** Злоумышленник получает больше прав, чем ему положено. Обычный пользователь становится администратором.

**Примеры:**
- Эксплуатация уязвимости для получения root
- Обход проверок авторизации
- IDOR (Insecure Direct Object Reference)
- Path traversal для доступа к системным файлам
- Container escape

**Что защищает:**
- Принцип минимальных привилегий
- RBAC с granулярными правами
- Sandbox'ы
- ASLR, DEP
- Регулярные патчи

**Вопрос для проверки:** Может ли кто-то получить больше прав, чем ему разрешено?

## Как использовать STRIDE

### Шаг 1: Создать DFD

DFD (Data Flow Diagram) — схема потоков данных. Показывает, какие компоненты есть в системе, как они общаются, где хранятся данные.

Компоненты DFD:
- **Процессы** — что делает система
- **Хранилища** — где хранятся данные
- **Потоки** — как данные движутся
- **Внешние сущности** — кто взаимодействует с системой

### Шаг 2: Применить STRIDE к каждому элементу

Для каждого процесса, хранилища, потока и внешней сущности задать шесть вопросов:
- Можно ли подменить?
- Можно ли изменить?
- Можно ли отрицать действия?
- Можно ли утечь данные?
- Можно ли заблокировать?
- Можно ли повысить права?

### Шаг 3: Найти угрозы

Для каждого вопроса, на который ответ "да" — описать конкретную угрозу. Не абстрактно, а конкретно: "При загрузке резюме через API злоумышленник может подменить JWT-токен и загрузить файл от имени другого пользователя".

### Шаг 4: Оценить и приоритизировать

Найденные угрозы оцениваются по критичности. Критические — исправляются первыми. Некритичные — планируются.

### Шаг 5: Спроектировать митигацию

Для каждой угрозы — контрмера. Угроза подмены → MFA. Угроза утечки → шифрование.

## STRIDE в применении к HR-платформе

### Сервис: загрузка резюме

**Компоненты:**
- Пользователь (кандидат)
- Веб-приложение (frontend)
- API (backend)
- База данных (хранилище резюме)
- S3 (хранилище файлов)
- HR-менеджер (просматривает резюме)

**Угрозы по STRIDE:**

| Категория | Угроза | Митигация |
|-----------|--------|-----------|
| Spoofing | Подмена кандидата | MFA, подпись JWT |
| Tampering | Изменение резюме после загрузки | Контроль целостности, версионирование |
| Repudiation | Кандидат отрицает загрузку | Аудит, подпись транзакций |
| Info Disclosure | Утечка резюме другим кандидатам | RBAC, шифрование |
| DoS | Забивание диска фейковыми резюме | Rate limiting, quotas |
| Elevation | Получение доступа к чужим резюме | IDOR защита, авторизация |

## Ограничения STRIDE

### Не оценивает риски

STRIDE находит угрозы, но не говорит, какие из них важнее. Для приоритизации нужна другая методология — например, DREAD.

### Не покрывает всё

STRIDE не включает:
- Privacy-угрозы (для этого LINDDUN)
- Физические угрозы
- Социальную инженерию
- Supply chain атаки

### Требует DFD

Без хорошей схемы системы STRIDE работает плохо. Нужно понимать архитектуру.

### Качество зависит от эксперта

Один эксперт найдёт больше угроз, другой — меньше. Опыт решает.

## Сравнение с другими методологиями

| Критерий | STRIDE | PASTA | Kill Chain |
|----------|--------|-------|------------|
| Сложность | Простая | Сложная | Средняя |
| Время | 1-2 часа | Недели | 2-4 часа |
| Фокус | Категории угроз | Риски | Этапы атаки |
| Результат | Список угроз | Risk score | Attack path |
| Требует DFD | Да | Да | Нет |

## Архитектурные решения

### STRIDE + DREAD

STRIDE находит угрозы. DREAD оценивает их. Вместе — полноценный анализ.

### STRIDE + Attack Trees

STRIDE находит угрозы. Attack Trees показывает пути их реализации.

### STRIDE в CI/CD

STRIDE запускается на этапе проектирования. Найденные угрозы добавляются в backlog. Митигации — в требования.

### Ответы на чек-лист

1. **S**poofing — подмена личности. **T**ampering — изменение данных. **R**epudiation — отрицание действий. **I**nformation Disclosure — утечка информации. **D**enial of Service — отказ в обслуживании. **E**levation of Privilege — повышение привилегий.

2. **Spoofing** — притворяться кем-то другим. **Tampering** — изменить данные. Разница: spoofing — "я не тот, за кого себя выдаю", tampering — "я изменил то, что мне не принадлежит".

3. **Repudiation** — возможность отрицать совершённое действие. Борьба: аудит всех действий, неизменяемые логи, цифровые подписи транзакций, timestamp от доверенного источника.

4. **Контрмеры против Information Disclosure**: шифрование данных при хранении и передаче, role-based access control, маскирование чувствительных данных, input validation, output encoding, минимизация данных в логах.

5. **Защита от DoS**: rate limiting на API, DDoS-защита (CloudFlare, AWS Shield), load balancing, auto-scaling, ограничение ресурсов (cgroups, quotas), кэширование, circuit breakers.

6. **Elevation of Privilege** — получение бóльших прав, чем положено. Защита: принцип минимальных привилегий, granулярный RBAC, sandbox'ы, ASLR/DEP, регулярные обновления, защита от IDOR и path traversal.

7. **DFD** (Data Flow Diagram) — схема потоков данных. Содержит: процессы (что делает система), хранилища (где данные), потоки (как движутся данные), внешние сущности (кто взаимодействует).

8. Каждому элементу DFD (процесс, хранилище, поток, сущность) задаются шесть вопросов STRIDE. Для потоков — spoofing, tampering, info disclosure. Для хранилищ — tampering, info disclosure, DoS. Для процессов — все шесть.

9. **Ограничения STRIDE**: не оценивает риски (нужен DREAD), не покрывает privacy (нужен LINDDUN), требует DFD, качество зависит от эксперта, не включает supply chain и социальную инженерию.

10. **STRIDE** лучше комбинировать с: DREAD (оценка рисков), Attack Trees (пути атак), Kill Chain (этапы атаки). STRIDE находит угрозы, другие методологии помогают их приоритизировать и детализировать.
