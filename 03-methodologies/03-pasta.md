# PASTA: Процесс-ориентированное моделирование угроз

## Что такое PASTA?

Представь, что ты строишь мост через реку. Перед тем как начать стройку, ты не просто смотришь на чертежи. Ты изучаешь, зачем мост нужен, сколько людей будет по нему ходить, какие машины будут ездить, какая глубина реки, какие течения, какой грунт, какие материалы доступны, сколько стоит стройка.

Потом ты моделируешь: что если пойдёт дождь? Что если грунт окажется не таким прочным? Что если ветер будет сильнее расчётного? Для каждого риска — стоимость митигации.

**PASTA** — это методология, которая строит моделирование угроз вокруг бизнес-рисков. Она начинает не с уязвимостей, а с вопроса: "Что нам грозит потерять?"

PASTA состоит из семи этапов. Каждый этап — это шаг от бизнеса к техническим деталям и обратно к бизнесу.

## Зачем нужна PASTA?

### Ситуация 1: Руководство не понимает угроз

Security-команда говорит: "У нас XSS и SQL-инъекция". Руководство спрашивает: "И сколько это будет стоить? Сколько мы потеряем, если не исправим?"

PASTA связывает технические угрозы с бизнес-потерями. "SQL-инъекция в HR-платформе может привести к утечке данных 100,000 сотрудников. Штраф по 152-ФЗ — до 500 млн рублей. Плюс репутационные потери. Плюс отток клиентов. Итого — риск 750 млн рублей. Стоимость исправления — 2 млн".

Теперь руководство понимает.

### Ситуация 2: Ограниченный бюджет

У команды есть бюджет на безопасность на год. Нужно решить, куда его потратить. Исправить всё невозможно.

PASTA считает: стоимость риска vs стоимость митигации. Если исправление стоит 10 млн, а риск — 1 млн — принимаем риск. Если исправление стоит 1 млн, а риск — 100 млн — исправляем.

### Ситуация 3: Новый продукт

Компания запускает новый сервис. Нужно понять, какие угрозы реальны, а какие — теоретические. Куда вкладывать в безопасность сразу, а что можно отложить.

PASTA помогает расставить приоритеты на основе реальных бизнес-рисков, а не страхов.

## Семь этапов PASTA

### Этап 1: Define Objectives (Определить цели)

**Вопрос:** Зачем существует система? Какой бизнес-процесс она поддерживает? Что будет, если она перестанет работать?

**Что делаем:**
- Описываем бизнес-процессы
- Определяем критичные активы
- Устанавливаем приемлемый уровень риска
- Определяем compliance-требования

**Пример для HR-платформы:**
- Бизнес-процесс: найм сотрудников
- Критичные активы: база резюме, персональные данные, процесс найма
- Приемлемый риск: утечка данных менее 100 записей
- Compliance: 152-ФЗ (ПДн), трудовое законодательство

### Этап 2: Define Technical Scope (Определить технический scope)

**Вопрос:** Какие компоненты входят в систему? Где границы?

**Что делаем:**
- Создаём DFD (Data Flow Diagram)
- Определяем trust boundaries
- Перечисляем компоненты, сервисы, интеграции
- Определяем архитектурные решения

**Пример:**
- Компоненты: веб-приложение, API, база данных, S3, email-сервис
- Trust boundaries: интернет → DMZ → internal network
- Интеграции: job boards, email, SMS gateway

### Этап 3: Application Decomposition (Декомпозиция приложения)

**Вопрос:** Как работает приложение внутри? Какие данные движутся?

**Что делаем:**
- Детальное DFD
- Use cases и abuse cases
- Анализ потоков данных
- Определение точек входа и выхода

**Пример:**
- Кандидат загружает резюме → API валидирует → сохраняет в S3 → записывает метаданные в БД → отправляет email HR
- Abuse case: кандидат загружает вредоносный файл → получает shell → читает чужие резюме

### Этап 4: Threat Analysis (Анализ угроз)

**Вопрос:** Какие угрозы существуют? Какие реальны для нашей системы?

**Что делаем:**
- Применяем STRIDE, Kill Chain, Attack Trees
- Анализируем реальные угрозы для отрасли
- Изучаем threat intelligence
- Определяем threat actors

**Пример:**
- Конкурент хочет украсть базу кандидатов
- Хактивист хочет опубликовать данные
- Внутренний сотрудник хочет продать данные
- Случайная утечка через уязвимость

### Этап 5: Vulnerability Analysis (Анализ уязвимостей)

**Вопрос:** Какие уязвимости позволяют реализовать угрозы?

**Что делаем:**
- Сканирование уязвимостей
- Анализ кода
- Проверка конфигурации
- Изучение CVE для используемых технологий

**Пример:**
- SQL-инъекция в API позволяет прочитать всю базу
- XSS позволяет украсть сессию HR-менеджера
- S3 bucket misconfig позволяет скачать все файлы

### Этап 6: Attack Modeling (Моделирование атак)

**Вопрос:** Как именно будет происходить атака? Сколько это стоит атакующему?

**Что делаем:**
- Строим attack trees
- Оцениваем вероятность атаки
- Оцениваем стоимость атаки для злоумышленника
- Оцениваем сложность защиты

**Пример:**
- Атака: SQL-инъекция → дамп базы → продажа данных
- Стоимость для атакующего: $500 (инструменты) + 2 дня
- Вероятность: высокая (есть готовый эксплойт)
- Выгода для атакующего: $50,000 на чёрном рынке

### Этап 7: Risk Analysis (Анализ рисков)

**Вопрос:** Сколько стоит риск? Сколько стоит защита? Окупается ли?

**Что делаем:**
- Считаем стоимость риска: Impact × Probability
- Считаем стоимость митигации
- Сравниваем
- Принимаем решение: исправлять, принять, передать

**Пример:**
| Угроза | Стоимость риска | Стоимость митигации | Решение |
|--------|----------------|---------------------|---------|
| SQL-инъекция | 750 млн ₽ | 2 млн ₽ | Исправить |
| XSS | 50 млн ₽ | 1 млн ₽ | Исправить |
| DoS | 10 млн ₽ | 5 млн ₽ | Принять риск |

## PASTA vs другие методологии

| Критерий | PASTA | STRIDE | Kill Chain |
|----------|-------|--------|------------|
| Фокус | Бизнес-риск | Технические угрозы | Этапы атаки |
| Результат | Стоимость риска | Список угроз | Путь атаки |
| Время | Недели | Часы | Часы |
| Сложность | Высокая | Низкая | Средняя |
| Требует | Бизнес-анализ | DFD | Нет |

## Архитектурные решения

### Risk-based security

PASTA реализует risk-based подход. Не всё защищаем одинаково. Защищаем то, что дорого стоит.

### Cost-benefit анализ

Каждое решение обосновано числами. "Защита стоит X, риск стоит Y. Y > X — защищаем".

### Business alignment

Безопасность выровнена с бизнесом. Руководство понимает, зачем тратить деньги.

## Ограничения PASTA

### Сложность

PASTA требует времени и ресурсов. Недели работы. Не для быстрых проверок.

### Требует бизнес-экспертизы

Нужно понимать бизнес-процессы. Технический эксперт не справится один.

### Не для runtime

PASTA — проектирование. Для runtime — другие методологии.

### Ответы на чек-лист

1. **Семь этапов**: Define Objectives, Define Technical Scope, Application Decomposition, Threat Analysis, Vulnerability Analysis, Attack Modeling, Risk Analysis.

2. **PASTA начинается** с бизнес-целей. Не с технических деталей, а с вопроса: "Зачем существует система и что мы можем потерять?"

3. **Abuse case** — негативный use case. Сценарий, в котором злоумышленник использует систему не по назначению. Например: кандидат загружает вредоносный файл вместо резюме.

4. **PASTA связывает угрозы с бизнес-рисками** через стоимость. Техническая уязвимость → потенциальная атака → ущерб для бизнеса → стоимость риска. Руководство видит риски в деньгах.

5. **На этапе Risk Analysis** считается: стоимость риска (impact × probability), стоимость митигации, сравнение, принятие решения (исправить, принять, передать).

6. **PASTA** фокусируется на бизнес-рисках и стоимости. **STRIDE** — на технических категориях угроз. PASTA отвечает "сколько стоит", STRIDE — "что может случиться".

7. **Использовать PASTA** при: проектировании новой системы, оценке критичных систем, подготовке к аудиту, обосновании бюджета на безопасность.

8. **Threat actor** — субъект угрозы. Кто может атаковать: конкурент, хактивист, киберпреступник, внутренний сотрудник, государство (APT).

9. **Стоимость атаки** для злоумышленника: время, инструменты, знания, ресурсы. Чем дешевле атака — тем выше вероятность.

10. **Ограничения PASTA**: сложность (недели работы), требует бизнес-экспертизы, не для runtime, ресурсоёмкость, требует точных данных для расчётов.
