# Безопасность контейнеров в облаке

## Что такое контейнер?

Представь, что ты переезжаешь. У тебя много вещей: одежда, посуда, книги. Ты кладешь всё в коробки. Каждая коробка — это контейнер. В коробке есть всё необходимое для жизни: одежда, посуда. Коробка изолирована. Вещи из одной коробки не смешиваются с другой.

Теперь представь, что ты открываешь коробку в новом доме. Всё работает. Те же книги, та же посуда. Не важно, в каком доме — всё одинаково.

**Контейнер** — это именно такая "коробка" для приложений. Внутри: приложение + всё, что ему нужно (библиотеки, настройки). Контейнер изолирован. Работает одинаково везде: на твоём ноутбуке, на сервере, в облаке.

## Зачем нужна безопасность контейнеров?

### Ситуация 1: Уязвимость в образе

Контейнер собран на основе образа Ubuntu. В Ubuntu есть уязвимость CVE-2021-44228 (Log4Shell). Все контейнеры на этом образе — уязвимы.

### Ситуация 2: Container Escape

Злоумышленник взломал приложение в контейнере. Вышел за пределы контейнера. Получил доступ к хосту.

### Ситуация 3: Секреты в образе

Разработчик добавил API-ключ в Dockerfile. Ключ попал в образ. Образ в репозитории. Кто угодно может прочитать.

## Архитектура контейнерной безопасности

### Уровни защиты

| Уровень | Что защищаем | Как |
|---------|--------------|-----|
| Образ | Базовый контейнер | Сканирование, минимизация |
| Реестр | Хранилище образов | Доступ, подпись |
| Оркестратор | Kubernetes, ECS | RBAC, сетевые политики |
| Контейнер | Работающий процесс | Seccomp, AppArmor |
| Хост | Сервер | Hardening, мониторинг |

| Практика | Описание |
|----------|----------|

## Kubernetes Security

### RBAC

Кто может что делать в кластере.

| Роль | Права |
|------|-------|
| cluster-admin | Всё |
| admin | Внутри namespace |
| edit | Чтение + запись |
| view | Только чтение |

### Network Policies

Файрвол внутри кластера.

```
Разрешить: frontend → backend:80
Запретить: всё остальное
```

### Pod Security

| Уровень | Описание |
|---------|----------|
| Privileged | Полный доступ (опасно) |
| Baseline | Ограниченный |
| Restricted | Максимально безопасный |

### Secrets

| Способ | Описание |
|--------|----------|
| Kubernetes Secrets | Базовый, не шифрует по умолчанию |
| Sealed Secrets | Шифрованные |
| Vault | HashiCorp, мощный |
| Cloud KMS | AWS/Azure/GCP KMS |

## Container Runtime Security

### Seccomp

Фильтр системных вызовов. Контейнер может использовать только разрешённые вызовы.

### AppArmor / SELinux

Мандатный контроль доступа. Что контейнер может делать с файлами.

### Capabilities

Урезание прав Linux-процесса. Не root, но с нужными правами.

### Ответы на чек-лист

1. **Контейнер** — изолированное окружение с приложением и зависимостями. Работает одинаково везде.

2. **Уровни**: образ, реестр, оркестратор, контейнер, хост.

3. **Container escape** — выход из контейнера на хост-систему. Критичная угроза.

4. **Инструменты сканирования**: Trivy, Clair, Snyk, Anchore.

5. **Kubernetes RBAC** — управление доступом в кластере. Роли: cluster-admin, admin, edit, view.

6. **Network Policy** — файрвол внутри Kubernetes. Определяет, кто может общаться с кем.

7. **Seccomp** — фильтр системных вызовов. Ограничивает, что контейнер может делать.

8. **Секреты**: Kubernetes Secrets (базовый), Sealed Secrets (шифрованные), Vault, Cloud KMS.

9. **Distroless** — минимальный образ без shell, package manager. Меньше поверхности атаки.

10. **Лучшие практики**: сканирование образов, runtime monitoring, least privilege, network segmentation, secrets management, hardening.
