# IAM в облаке: Архитектура управления идентификацией, аутентификацией и авторизацией в распределённых средах

## Что такое облачный IAM?

Представьте театр с многоуровневой системой доступа. Кассир продаёт билеты, но не может выйти на сцену. Актёр находится на сцене, но не может войти в кассу. Режиссёр может входить в любые помещения, но не имеет доступа к банковскому счёту театра. Гардеробщик принимает пальто, но не может включить софиты. У каждого — бейдж с чипом. Бейдж определяет: в какие двери можно войти, в какое время, и оставляет запись в журнале охраны. Если актёр попытается пройти в кассу в 3 часа ночи — бейдж не сработает, и охрана получит alert. Более того, есть гости (зрители), которые могут находиться только в фойе и зале, иметь временный билет на конкретный спектакль и не более. Это и есть **IAM (Identity and Access Management)** — не просто «кто куда может», а полноценная архитектура управления идентификациями, временем, контекстом и подотчётностью каждого действия в облачной среде.

Фундаментальная проблема безопасности, которую решает IAM: в облаке нет физического периметра, офиса, VPN, забора. Каждый API-вызов к облачному ресурсу — это запрос, приходящий из интернета, и единственный механизм отличить легитимного администратора от злоумышленника — проверить: кто делает запрос, имеет ли он право, является ли это действие нормальным для его роли, не скомпрометированы ли его учётные данные. IAM превращает облако из анонимного пространства в среду, где каждое действие атрибутировано конкретной идентичности с конкретным набором разрешений.

## Эволюция и мотивация

Двадцать лет назад IAM в корпоративных сетях сводился к Active Directory и спискам групп. Пользователь логинился в домен, получал членство в группе (Domain Admins, HR, Engineering), и периметр сети гарантировал, что злоумышленник не может даже достучаться до сервера. Это работало, пока сеть была физически изолирована. С появлением SaaS (Salesforce в 1999 году, Google Workspace в 2006) периметр начал размываться: корпоративные данные оказались за пределами корпоративной сети, и единственным барьером остался логин-пароль.

Три технологических сдвига сделали облачный IAM критической дисциплиной. Во-первых, переход инфраструктуры на API-управление: в облаке создание сервера, удаление базы данных, изменение сетевой политики — всё это REST API-вызовы, которые не проходят через Active Directory, а проверяются облачным IAM-движком. Во-вторых, микросервисная архитектура: сотни сервисов, общающихся друг с другом, требуют не «человеческих» учётных записей, а машинных идентификаций (service accounts, workload identities) с автоматической ротацией credentials. В-третьих, экспоненциальный рост числа участников: к людям добавились CI/CD-пайплайны, Kubernetes-поды, serverless-функции, внешние партнёры, IoT-устройства.

Индустрия прошла через три парадигмы. Сетевая парадигма: «раз ты внутри сети, доступ разрешён» — убита облаком. Ролевая парадигма: RBAC как универсальное решение — уступила место Attribute-Based Access Control (ABAC) и Policy-Based Access Control, потому что в облаке контекст запроса (время, геолокация, тип устройства, MFA-статус) так же важен, как роль. Текущая парадигма — Identity-Centric Security: идентичность становится новым периметром, а IAM — центральной плоскостью управления безопасностью, вокруг которой строятся все остальные контроли. NIST SP 800-207 (Zero Trust Architecture) формализует это: каждый запрос должен быть аутентифицирован, авторизован и зашифрован независимо.

## Зачем нужен облачный IAM?

**Сценарий 1: Жизненный цикл сотрудника.** Разработчик Иван проработал в компании три года, имел доступ к production AWS-аккаунту, писал Lambda-функции. При увольнении его корпоративный аккаунт деактивирован, но команда использовала shared IAM-пользователя с долгоживущими access keys, которые Иван сохранил локально. Через месяц после увольнения он использует ключи для доступа к S3-бакетам с исходным кодом — без IAM, автоматически синхронизирующего жизненный цикл с HR-системой, это классический «offboarding gap». Контроль CIS Controls v8, п. 5.3 (Disable Dormant Accounts), требует автоматического отключения неактивных учётных записей, что в облаке решается через федеративную IAM-архитектуру без долгоживущих credentials.

**Сценарий 2: Избыточные права как норма.** Команда data engineering развернула Airflow в EKS. Для простоты airflow-сервис получил IAM-роль `AdministratorAccess` — «потом разберёмся». Через полгода Kubernetes-под скомпрометирован через уязвимость в Airflow-плагине — злоумышленник получает administrative-доступ ко всему AWS-аккаунту и разворачивает криптомайнинг на 50 EC2-инстансах. Счёт за месяц: $120,000. Принцип минимальных привилегий (NIST SP 800-53 AC-6) требует, чтобы каждая рабочая нагрузка имела ровно ту IAM-роль, которая необходима для её функции — и ни одним правом больше.

**Сценарий 3: Мультиаккаунт и федеративный доступ.** Организация из 2000 разработчиков использует AWS Organizations с 50 аккаунтами (по одному на команду). Без федеративного IAM каждому разработчику нужно создать IAM-пользователя в каждом аккаунте, управлять 100,000 учётных записей и ротировать 200,000 access keys. С федеративным IAM (AWS IAM Identity Center + Okta/Azure AD как IdP) разработчик аутентифицируется один раз, временные credentials генерируются on-the-fly через SAML/OIDC, и количество долгоживущих ключей равно нулю. Контроль NIST SP 800-53 IA-5 (Authenticator Management) требует минимизации использования статических аутентификаторов.

**Сценарий 4: Комплаенс аудит доступа.** Аудитор PCI DSS запрашивает: «Покажите всех пользователей, имеющих доступ к S3-бакету с платёжными данными, и подтвердите, что каждый имеет business justification». Без IAM Access Analyzer, без CloudTrail audit logs организация потратит две недели на ручной сбор evidence, и с вероятностью 80% аудитор найдёт хотя бы один «призрачный» доступ (уволенный сотрудник, legacy role). IAM обеспечивает: каждая политика имеет трассируемый lifecycle, каждая роль — documented justification, каждый доступ — continuous validation через CIEM-инструменты.

**Сценарий 5: Атака на цепочку поставок через compromised CI/CD.** Злоумышленник получает доступ к GitHub Actions-секретам, которые содержат IAM-access keys с правами на деплой в production. Ключи не имеют ограничения по IP, не требуют MFA, не ограничены конкретными ресурсами. Атакующий использует их для внедрения backdoor в production-приложение. Эффективный IAM-ответ: использование OIDC federation между GitHub Actions и AWS вместо статических ключей; ограничение IAM-роли конкретным репозиторием, веткой и environment; mandatory MFA для всех человеческих учётных записей с production-доступом.

## Основные концепции

### Четыре столпа IAM: Identity, Authentication, Authorization, Accountability

IAM — не один механизм, а интегрированная система из четырёх компонентов, каждый из которых должен работать безупречно, потому что провал любого делает бесполезными остальные три.

**Identity (Идентичность)** отвечает на вопрос «кто это?». В облаке идентичность — не обязательно человек. Это может быть пользователь (User), группа пользователей (Group), роль с временными разрешениями (Role), сервисный аккаунт для приложений (Service Account) или даже внешний федеративный субъект, аутентифицированный через доверенный IdP. Каждая идентичность имеет уникальный идентификатор (ARN в AWS, Object ID в Azure, email в GCP), набор атрибутов (department, clearance level, cost center) и жизненный цикл (created → provisioned → active → suspended → deleted). NIST SP 800-53 IA-4 (Identifier Management) требует, чтобы организация управляла идентификаторами на всём жизненном цикле с предотвращением повторного использования в течение определённого периода.

**Authentication (Аутентификация)** доказывает, что субъект действительно является тем, за кого себя выдаёт. Три фактора: то, что ты знаешь (пароль), то, что ты имеешь (токен, TOTP), то, чем ты являешься (биометрия). В облаке аутентификация работает на двух уровнях: человеческий (интерактивный вход через консоль, CLI с MFA) и машинный (API-вызов с подписанными запросами, OIDC-токены для CI/CD, X.509-сертификаты для mTLS). Ключевое архитектурное решение: человеческие учётные записи никогда не должны использовать долгоживущие статические credentials. Только временные токены через SSO — это исключает класс атак на украденные access keys. Контроль CIS v8, п. 5.6 (Use of Privileged Accounts), требует использования отдельных аккаунтов для административных действий с обязательной MFA.

**Authorization (Авторизация)** определяет, что именно разрешено данной идентичности в данный момент. В облаке это реализуется через многоуровневую систему политик: явные разрешения (Allow policies), явные запреты (Deny policies, имеющие приоритет), ограничения по условиям (Condition: время, IP, MFA-статус, теги ресурса), границы разрешений (Permission Boundaries, ограничивающие максимальный набор прав, даже если другие политики пытаются расширить). Модели авторизации: RBAC (роль предопределяет набор прав), ABAC (атрибуты идентичности и ресурса определяют доступ динамически), PBAC (Policy-Based, кастомные правила на языке политик провайдера). Контроль OWASP ASVS V4.1 (Access Control Verification Requirements) требует, чтобы авторизация происходила на стороне сервера и проверялась для каждого запроса без предположения, что предыдущий запрос был авторизован.

**Accountability (Подотчётность)** гарантирует, что каждое действие в облаке имеет неотказуемую запись: кто сделал, что сделал, когда, с какого IP, с каким результатом. Реализуется через сервисы аудита (AWS CloudTrail, Azure Activity Log, GCP Cloud Audit Logs), которые пишут неизменяемые логи каждого API-вызова вплоть до параметров запроса. Без accountability IAM не имеет смысла: можно настроить минимальное право доступа, но если нет логов, вы не узнаете, что право было использовано злонамеренно. NIST SP 800-53 AU-3 (Content of Audit Records) требует детализации записей аудита, достаточной для реконструкции событий.

### Федеративная архитектура IAM

В масштабе организации невозможно управлять учётными записями в каждом облачном аккаунте отдельно. Федеративный IAM (Federated Identity Management) решает эту проблему через архитектуру «Hub and Spoke»: центральный Identity Provider (IdP) — Okta, Azure AD, Google Workspace, Ping Identity — является единственным источником truth об идентификациях, а облачные провайдеры выступают как Service Providers (SP), принимающие аутентификационные утверждения через стандартные протоколы (SAML 2.0, OpenID Connect).

Поток федеративной аутентификации: пользователь открывает AWS-консоль; AWS перенаправляет на корпоративный IdP; IdP аутентифицирует пользователя (пароль + MFA + device posture check через Conditional Access); IdP выпускает SAML-assertion с набором атрибутов (роль, отдел, группа); AWS валидирует assertion, извлекает атрибуты, сопоставляет их с IAM-ролью и генерирует временные STS-credentials на 1 час. Весь процесс занимает менее двух секунд, не создаёт долгоживущих access keys, и отключение пользователя в IdP мгновенно лишает его облачного доступа, потому что без свежего SAML-assertion временные credentials не генерируются.

Архитектурный компромисс федерации: единая точка отказа — если IdP недоступен, никто не может залогиниться в облако. Это требует high-availability конфигурации IdP (multi-region, disaster recovery), что увеличивает инфраструктурные затраты, но критически необходимо для организаций с zero-trust архитектурой, где логин — единственный путь к ресурсам (нет fallback-паролей, нет emergency-access без MFA).

### Машинные идентичности и Workload Identity

В современном облаке большинство API-вызовов исходит не от людей, а от приложений, CI/CD-пайплайнов, Kubernetes-подов, serverless-функций. Традиционный подход — создать IAM-пользователя, сгенерировать access key и сохранить его в CI/CD-секретах или переменных окружения — создаёт два класса проблем: долгоживущие статические ключи, которые никогда не ротируются, и secrets sprawl (размножение секретов по десяткам систем).

Workload Identity — современный подход, при котором рабочая нагрузка получает временные credentials через доверенные отношения, а не через секреты. Реализация: в Kubernetes, через OIDC federation между Service Account и облачной IAM-ролью (IRSA в AWS EKS, Workload Identity в GKE, Azure AD Workload Identity в AKS). Сервисный аккаунт в Kubernetes аннотирован ARN's облачной IAM-роли. При старте пода, мутирующий webhook инжектит OIDC-токен, подписанный Kubernetes API server, в файловую систему пода. SDK облачного провайдера обменивает этот OIDC-токен на временные облачные credentials через STS. Ключевое преимущество: ни один секрет не хранится, токены живут час и автоматически ротируются, компрометация пода даёт доступ только на время жизни одного токена.

Контроль NIST SP 800-53 IA-5 (Authenticator Management) и CIS v8, п. 16.11 (Secrets Management for Applications), прямо предписывают использование временных credentials для машинных идентичностей и запрещают хранение статических ключей в коде, конфигурациях, образах контейнеров.

## Сравнение моделей авторизации

| Критерий | RBAC (Role-Based Access Control) | ABAC (Attribute-Based Access Control) | PBAC/Policy-as-Code |
|----------|----------------------------------|---------------------------------------|---------------------|
| Принцип принятия решения | Назначенная роль определяет фиксированный набор разрешений | Атрибуты субъекта, ресурса и окружения определяют доступ динамически | Внешний движок (OPA, Cedar) оценивает политику на языке правил против контекста запроса |
| Гибкость | Низкая: изменение разрешений требует изменения роли | Высокая: комбинация атрибутов позволяет гранулярные правила без модификации ролей | Максимальная: произвольная логика на Turing-полном языке |
| Производительность | Высокая: проверка O(1), membership lookup | Средняя: оценка условий (IP, время, теги) добавляет latency | Может быть значительной: внешний вызов к Policy Engine добавляет 20-50ms на запрос |
| Типичный сценарий | Статические должности: Admin, Developer, ReadOnly | Динамические среды: доступ разрешён только из корпоративного IP и только в рабочие часы | Многоуровневые организации: разные правила для разных OU/аккаунтов с делегированием |
| Связь со стандартами | NIST 800-53 AC-2 (Account Management), AC-6 (Least Privilege) | NIST 800-53 AC-3 (Access Enforcement), Zero Trust 800-207 | OWASP ASVS V4.1, CIS v8 Control 5 |
| Сложность внедрения | Низкая: предопределённые роли провайдера | Средняя: требует таксономии атрибутов, тегирования ресурсов, condition-политик | Высокая: требует написания и тестирования политик, CI/CD-интеграции |

RBAC — отправная точка, но недостаточная для зрелых организаций. Типичная эволюция: стартап использует три роли (Admin/Dev/ReadOnly) — это RBAC. С ростом появляются условия: «ReadOnly, но только для ресурсов с тегом Environment=Production, и только из корпоративного VPN» — это ABAC. Наконец, в multi-account среде с сотнями команд появляется требование «Каждая команда может создавать IAM-роли, но не может превысить permission boundary, определённую центральной security-командой» — это Policy-as-Code через SCP (Service Control Policies) и Permission Boundaries.

Ни одна модель не является «лучшей». ABAC даёт гибкость, но условия нужно поддерживать: если атрибут (тег, IP-диапазон) меняется, доступ неожиданно пропадает, вызывая инциденты. PBAC даёт максимальный контроль, но требует компетенции написания и отладки политик, которая не является core skill для разработчиков. Архитектурный компромисс: RBAC для грубой нарезки (профессиональные роли) + ABAC для контекстных ограничений (гео, время, MFA) + PBAC для организационных границ (SCP на уровне AWS Organization).

## Уроки из реальных инцидентов

### Инцидент Uber (2016, раскрыт в 2017): Утечка credentials в коде

Хронология: в 2016 году злоумышленники обнаружили IAM access keys, захардкоженные в приватном GitHub-репозитории инженера Uber. Ключи давали полный административный доступ к production AWS-аккаунту, где хранились персональные данные 57 миллионов пользователей и 600,000 водителей. Uber попытался скрыть инцидент, заплатив злоумышленникам $100,000 через bug bounty программу под видом легальной выплаты, и не уведомил регуляторов. В 2017 году факт сокрытия стал публичным, что привело к увольнению CISO, расследованию FTC и многомиллионным штрафам.

Корневая причина с точки зрения IAM: access keys, сгенерированные для IAM-пользователя, были сохранены в коде, закоммичены в git и оставались валидными более года. Ни автоматического сканирования репозиториев на секреты (git-secrets, truffleHog), ни mandatory ротации ключей, ни IAM-политик с условием SourceIP/PrincipalTag не было реализовано.

Как зрелый IAM изменил бы исход: федеративная аутентификация без статических ключей — разработчики используют временные STS-credentials, которые невозможно закоммитить, потому что они живут час. Автоматическое сканирование CI/CD на patterns секретов (AWS Access Key ID начинается с `AKIA`) — блокировка push. Mandatory IAM Access Analyzer с алертингом на external access findings. Контроль NIST SP 800-53 AC-6 (Least Privilege) — производственная роль ограничена конкретными сервисами, а не `*:*`.

Урок: IAM credentials — не просто «логин и пароль», а ключ к королевству. Каждый access key в коде — это будущий инцидент с вероятностью, стремящейся к 100% в масштабе организации. Единственное надёжное решение — миграция на архитектуру без статических ключей (federation + temporary credentials + workload identity). OWASP ASVS V2.10 (Service Authentication) требует, чтобы сервисы аутентифицировались без долгоживущих секретов.

### Типичный сценарий инцидента: Privilege Escalation через IAM Role Chaining

Хронология: DevOps-инженер создал IAM-роль `JenkinsRole` с разрешением `sts:AssumeRole` на любую роль в аккаунте, чтобы CI/CD-пайплайн мог ассумить production-роли. Позднее Jenkins-сервер был скомпрометирован через уязвимость плагина. Злоумышленник, получив Jenkins-токен, выполнил `sts:AssumeRole` на `AdminRole` (которая не имела condition, ограничивающего, кто может её ассумить), и получил полный административный доступ.

Корневая причина: role chaining без ограничений. `JenkinsRole` имела слишком широкую trust policy (разрешён ассумпт любых ролей). `AdminRole` не имела условия, что ассумить её могут только роли с конкретным PrincipalTag.

IAM-решение: принцип максимального ограничения trust relationships. Jenkins-роль должна иметь permission ассумить только одну конкретную целевую роль (не wildcard-ARN). Целевая роль должна включать условие: ассумить можно только если source identity прошла MFA. Контроль NIST SP 800-53 AC-6 (Least Privilege) и CIS v8, п. 5.4, требуют ограничения привилегированных ролей списком доверенных субъектов с явными условиями передачи привилегий.

Последствия: cleanup (удаление несанкционированных ресурсов), forensic audit всех API-вызовов за период компрометации, mandatory rotation всех credentials в аккаунте, обновление trust policy для всех production-ролей.

Урок: каждая `sts:AssumeRole` — это потенциальный вектор privilege escalation. Trust relationships должны быть предметом mandatory review при каждом изменении IAM. Инструменты типа IAM Access Analyzer могут автоматически находить роли, которые могут быть ассумлены внешними субъектами или через transitive trust chain, которую команда не осознаёт.

## Инструменты и средства защиты облачного IAM

| Класс инструмента | Назначение | Позиция в жизненном цикле | Связь со стандартами |
|-------------------|------------|---------------------------|----------------------|
| Cloud Infrastructure Entitlement Management (CIEM) | Анализ фактического использования IAM-прав, поиск unused permissions, рекомендация сокращения привилегий | Continuous monitoring (daily/weekly scan) | NIST 800-53 AC-6 (Least Privilege), CIS v8 Control 5.4 |
| Identity Governance and Administration (IGA) | Автоматизация жизненного цикла идентичностей: provisioning, deprovisioning, access reviews, role mining | Onboarding/offboarding, quarterly access reviews | ISO 27001 A.9.1 (Access Control), NIST 800-53 AC-2 |
| IAM Policy Analysis & Simulation | Симуляция IAM-политик, определение effective permissions для конкретной идентичности, поиск путей privilege escalation | Pre-deployment (policy authoring) + continuous | OWASP ASVS V4.1.1, CIS v8 Control 5.1 |
| Privileged Access Management (PAM) | Управление привилегированным доступом: JIT elevation, session recording, approval workflows | Runtime: каждая elevation-сессия | NIST 800-53 AC-6(5) (Privileged Accounts), PCI DSS Req 7.1 |
| Secrets Detection & Scanning | Сканирование репозиториев, образов, CI/CD-логов на patterns access keys, паролей, API-токенов | CI/CD pipeline (pre-commit, pre-push, pre-deploy) | CIS v8 Control 16.11, NIST 800-53 IA-5 |

CIEM — самый востребованный класс в 2024-2025 годах, потому что организации обнаружили систематическую проблему избыточных прав. Исследования показывают, что в среднем 95% предоставленных IAM-прав не используются в течение 90-дневного периода. CIEM (NetSPI, Ermetic, Solvo) парсит CloudTrail-логи, строит матрицу «granted vs used permissions» и предлагает сократить политику до фактически используемых разрешений. Метрика эффективности: Permission Usage Ratio. При значении ниже 20% организация имеет неоправданно высокий blast radius при компрометации любой рабочей нагрузки.

PAM для облака отличается от традиционного PAM: нет VPN, нет jump-host, нет Bastion server. PAM в облаке работает на уровне IAM: вместо постоянной `AdminRole` — Just-in-Time elevation через отдельный PAM-инструмент, который на 1 час добавляет пользователя в административную группу, логирует все действия и автоматически отзывает доступ. Azure PIM (Privileged Identity Management) — эталонная реализация этого паттерна.

Secrets Detection — первая линия защиты от утечек credentials. Инструменты (git-secrets, truffleHog, Spectral, GitGuardian) сканируют каждый коммит на предмет patterns (AKIA-префикс, Azure Storage Key pattern, GCP Service Account JSON) и блокируют push при совпадении. Однако это защита от симптомов, а не от причины. Архитектурная цель — устранить долгоживущие секреты как класс через миграцию на OIDC federation и workload identity, после чего secrets detection становится secondary (catching legacy cases), а не primary контролем.

## Архитектурные решения

Внедрение зрелого облачного IAM требует многоуровневой архитектуры, где каждый уровень независимо реализует свою часть модели безопасности:

- **Организационный уровень (AWS Organizations SCP, Azure Management Groups, GCP Organization Policy):** ограничения сверху, которые не может обойти ни один аккаунт. SCP запрещает создание IAM-пользователей (требует federation), запрещает публичные S3-бакеты, ограничивает регионы, запрещает `iam:PassRole` для `*`-ресурсов.
- **Аккаунт-уровень (Permission Boundaries, IAM Roles):** гранулярные политики для приложений. Permission Boundary определяет максимальный набор разрешений для ролей, создаваемых командами разработки: они могут самовыражаться в этих границах, но не могут превысить потолок.
- **Ресурс-уровень (Resource-based Policies, Bucket Policies):** политики, привязанные к конкретному ресурсу, определяющие, кто может к нему обращаться. Bucket Policy на S3 с условием: доступ разрешён только через VPC Endpoint конкретного VPC.
- **Сессионный уровень (Session Policies, STS):** ограничения, применяемые на время сессии. Даже если роль имеет широкие права, session policy может сузить их для конкретной сессии: разрешить `s3:GetObject` только для префикса `logs/`, запретить `s3:DeleteObject`.

Метрики эффективности: время от увольнения сотрудника до отзыва всех облачных доступов (цель: менее 1 часа через JIT revoking); доля рабочих нагрузок, использующих временные credentials вместо статических ключей (цель: 100% для новых, migration target 90%+ для legacy); frequency of access reviews для привилегированных ролей (цель: monthly automated review с human approval для ролей с `iam:*` или `*:*`).

Красные флаги деградации: появление IAM-пользователей в production-аккаунтах (должны быть запрещены SCP); CloudTrail event `CreateAccessKey` в production; более трёх IAM-ролей с wildcard `Resource: "*"`; CIEM report показывает рост числа неиспользуемых разрешений.

Сценарий миграции legacy IAM на современную архитектуру: на первом этапе — полный аудит существующих IAM-пользователей и их ключей (возраст, дата последнего использования, ротация). На втором этапе — подключение IdP и развёртывание SSO для человеческих пользователей, параллельный run: сотрудник может логиниться и старым способом (ключи), и новым (SSO). На третьем этапе — заморозка создания новых IAM-пользователей (SCP deny `iam:CreateUser`), принудительная миграция каждой legacy-учётной записи на федеративный доступ, деактивация всех access keys старше 90 дней без ротации. На четвёртом этапе — внедрение CIEM и автоматическое удаление неиспользуемых разрешений.

## Подготовка к собеседованию: Common pitfalls & key questions

**Типичная ошибка 1: «IAM roles — это то же самое, что IAM users, только временные».**
Нет. IAM user — долгоживущая статическая идентичность с permanent credentials (access keys), которую нужно вручную ротировать. IAM role — контейнер для временных разрешений, у которого нет постоянных credentials, только trust policy, определяющая, кто может его ассумить. Архитектурно это противоположные паттерны: user — для legacy/exception, role — для всего остального. Интервьюер ожидает понимания, что организация должна стремиться к нулю IAM-пользователей.

**Типичная ошибка 2: «Пароль + MFA достаточно для аутентификации администратора».**
В Zero Trust — нет. Аутентификация администратора должна учитывать device posture (управляемое устройство? соответствует security baseline?) и риск-сигналы (невозможная travel — вход из Москвы и через 5 минут из Токио? atypical behavior — логин в 3 часа ночи?). Azure Conditional Access и AWS Verified Access реализуют эту логику. Интервьюер проверяет, понимает ли кандидат, что аутентификация — не бинарное решение, а continuous risk assessment.

**Типичная ошибка 3: «Если доступ настроен через IAM, он автоматически защищён».**
IAM политика — это разрешение, а не защита. Если роль имеет `s3:*` для `*`, это IAM-конфигурация, но не безопасность. Безопасность требует непрерывной валидации: используется ли это право? не является ли роль избыточно привилегированной? не может ли она быть ассумлена через неожиданную trust chain? IAM — механизм, безопасность — процесс, использующий этот механизм.

**Сложный вопрос 1: «Как предотвратить ситуацию, при которой разработчик создаёт IAM-роль с AdministratorAccess для своего сервиса? Какие превентивные и детективные меры вы примените?»**
Проверяется: архитектурное мышление, многоуровневость защиты. Ответ: превентивно — SCP на уровне Organization запрещает `iam:CreateRole` с `AdministratorAccess`, Permission Boundary на уровне аккаунта ограничивает максимальные права создаваемых ролей списком разрешённых сервисов. Детективно — AWS Config rule `iam-role-managed-policy-check` с запрещённым списком managed policies; CloudTrail alert на событие `AttachRolePolicy` с `AdministratorAccess`; автоматическая Lambda-ремедиация, открепляющая запрещённую политику в течение 2 минут. Компромисс: разработчики могут найти обходной путь (создать кастомную политику с такими же правами), поэтому defence-in-depth требует также runtime detection — GuardDuty anomaly detection на резкое расширение действий роли.

**Сложный вопрос 2: «Вы — security architect в мульти-аккаунтной организации (100+ AWS-аккаунтов, 3000 разработчиков). Каждая команда хочет управлять своими IAM-ролями самостоятельно. Как обеспечить governance, не став bottleneck?»**
Проверяется: баланс между безопасностью и автономией, масштабирование. Ответ: модель delegated administration. Центральная security-команда определяет Permission Boundaries для каждого OU (Development/Staging/Production) и привязывает их к аккаунтам через SCP и CloudFormation StackSets. Команды могут создавать любые роли в пределах границы. Ежемесячный CIEM-скан всех аккаунтов с automatic notification командам о unused permissions; KPI команды — Permission Usage Ratio, который влияет на approval для расширения границы. Access Review — quarterly certification менеджерами команд с mandatory re-justification всех ролей, имеющих доступ к production-данным. Компромисс: delegation увеличивает поверхность misconfigurations; компенсируется mandatory pre-deployment проверкой через OPA-правила в CI/CD пайплайне.

**Сложный вопрос 3: «Как вы реализуете emergency access ("break glass") для администраторов, если штатная система федеративной аутентификации недоступна? Какие риски это создаёт?»**
Проверяется: практический опыт, понимание компромисса availability vs security. Ответ: emergency access — заранее созданные IAM-пользователи (не роли, потому что для ролей нужен работающий IdP) с extremely restricted trust: пароль из 40 символов, разделённый на две части и хранящийся у двух разных человек (CISO и VP Engineering), MFA-токен в физическом сейфе, автоматический alert на любую активацию с немедленным уведомлением всей security-команды, автоматическая force password change после каждого использования, CloudTrail monitoring с real-time streaming в SIEM. Процедура break glass должна быть задокументирована как часть Incident Response Plan и тестироваться минимум раз в год через tabletop exercise. Контроль NIST SP 800-53 AC-2(2) (Automated Temporary Account Management) описывает этот кейс. Ключевой риск: emergency credentials — permanent static keys, что противоречит zero-trust архитектуре; их существование оправдано только как компенсирующий контроль против риска потери доступа.

- [ ] В каком случае ABAC-политика, ограничивающая доступ по IP-адресу, неожиданно заблокирует легитимного администратора и вызовет инцидент?
- [ ] Почему Permission Boundary не заменяет IAM-политику, а дополняет её, и в каком порядке они оцениваются?
- [ ] Как OIDC federation между Kubernetes Service Account и облачной IAM-ролью устраняет проблему secrets sprawl?
- [ ] В каком случае SCP на уровне AWS Organization не сможет предотвратить создание избыточно привилегированной роли?
- [ ] Почему CIEM-анализ без автоматической ремедиации — только половина решения?
- [ ] Как incident response меняется, если скомпрометирована IAM-роль с долгоживущими access keys vs роль с временными STS-credentials?
- [ ] В каком случае MFA не защищает от компрометации облачной учётной записи?
- [ ] Почему каждая `iam:PassRole` в политике должна проходить mandatory review на этапе pull request?

### Ответы на чек-лист

1. **Ответ:** Статические access keys — долгоживущие (valid until manually revoked), могут быть скомпрометированы через утечку в код, фишинг, вредоносное ПО — и после компрометации дают неограниченный по времени доступ. Временные STS-credentials живут 15 минут — 12 часов, автоматически expire, и их компрометация ограничена окном жизни токена. Федеративный подход также централизует управление идентификацией: отключение пользователя в IdP мгновенно лишает его облачного доступа, без необходимости искать и удалять ключи в десятках аккаунтов. Контроль NIST SP 800-53 IA-5 (Authenticator Management) и CIS v8 п. 5.6 требуют минимизации статических аутентификаторов.

2. **Ответ:** IAM-роль имеет trust policy (определяет, кто может ассумить роль) и permissions policy (что можно делать). Если trust policy разрешает ассумить роль субъекту с широкими границами (другой роли, федеративному пользователю из внешнего аккаунта, any authenticated AWS principal), и роль имеет привилегированные permissions — любой субъект, удовлетворяющий trust policy, может получить привилегии роли. Особенно опасен transitive trust: роль A доверяет роли B, роль B скомпрометирована — злоумышленник через B ассумит A. IAM Access Analyzer находит такие цепочки, и OWASP ASVS V4.1.2 требует валидации доверенных субъектов.

3. **Ответ:** Когда администратор работает из другого места (командировка, миграция офиса, смена VPN-провайдера), его IP-адрес меняется и попадает под deny condition. Без fallback-процедуры (temporary exception request) или без дополнительного фактора (например, device certificate, не привязанного к IP), администратор оказывается locked out. ABAC требует поддержки exception flow: временное расширение IP-диапазона через change request с автоматическим откатом через заданное время — иначе команда начнёт обходить IAM через теневые IT-каналы, что ещё хуже.

4. **Ответ:** Permission Boundary — это потолок; IAM-политика — наполнение под потолком. Оценка effective permissions: сначала оцениваются identity-based политики на Allow, затем Permission Boundary пересекает (intersects) разрешённый набор — права, не входящие в Permission Boundary, отбрасываются, даже если identity-based политика их разрешает. Затем Deny-политики вычитаются. Permission Boundary не заменяет identity policy, потому что он сам по себе не даёт доступа — нужна идентичность с explicit Allow. Контроль NIST SP 800-53 AC-6 (Least Privilege) требует ограничения максимального набора прав, что прямо реализуется через Permission Boundary.

5. **Ответ:** Без OIDC federation каждый под нуждается в статических credentials (access key + secret), которые кто-то должен сгенерировать, сохранить в Kubernetes Secret, управлять их ротацией, синхронизировать с облачной IAM. Это создаёт secrets sprawl: секреты в кластере, в CI/CD, в конфигурационных файлах. OIDC federation заменяет статические ключи доверенным отношением: Kubernetes Service Account токен, подписанный OIDC-эндпоинтом кластера, автоматически обменивается на временные облачные credentials. Секреты не хранятся нигде: токен генерируется при старте пода, облачные credentials — на время выполнения запроса. CIS v8 п. 16.11 требует именно такого подхода.

6. **Ответ:** SCP ограничивает API actions, но не может ограничить параметры разрешённых actions. Если SCP не запрещает `iam:CreateRole` или `iam:PutRolePolicy`, разработчик может создать роль с кастомной политикой, текстуально дублирующей `AdministratorAccess`, но не являющейся managed policy `AdministratorAccess`. SCP работает на уровне API names, а не на уровне семантики политик. Поэтому SCP — первая линия, но не достаточная; нужна дополнительная детекция через AWS Config rule, анализирующую содержимое инлайн-политик.

7. **Ответ:** CIEM находит неиспользуемые разрешения, но не удаляет их. Если команда не действует по рекомендациям (удаление неиспользуемого права требует тестирования, может сломать редко используемый функционал), отчёт CIEM остаётся «инвентаризацией рисков», а не их устранением. Полноценное решение: CIEM + автоматизированный workflow — неиспользуемые права автоматически удаляются после 90 дней non-usage, с grace period в 7 дней и возможностью exception request. Без автоматической ремедиации CIEM-результаты деградируют в backlog, который никто не разбирает.

8. **Ответ:** При компрометации access keys — необходимо немедленно deactivate ключи (но злоумышленник мог создать дополнительные ключи, если имел `iam:CreateAccessKey`), проверить CloudTrail на всё, что ключ сделал (может быть дни или недели), ротировать все credentials в аккаунте. При компрометации STS-credentials — токен автоматически истекает (максимум 12 часов, обычно 1 час), ограничивая blast radius временным окном, но злоумышленник мог успеть ассумить дополнительные роли. Ключевая разница: access keys дают persistent backdoor; STS даёт временный доступ, но sophisticated attacker может использовать STS-сессию для создания persistent access через другие средства (создание нового IAM-пользователя, если политика позволяла).

9. **Ответ:** Если злоумышленник использует session hijacking (кража сессионного токена после успешной MFA-аутентификации) или MFA bombing fatigue (многократные push-уведомления, пока пользователь не подтвердит от усталости) — MFA пройдена легитимным пользователем, но сессия захвачена атакующим. Также MFA не защищает от компрометации машинных credentials (service account keys, у которых нет MFA). OWASP ASVS V2.8 (Authenticator Strength) требует consideration этих векторов.

10. **Ответ:** `iam:PassRole` позволяет субъекту передать IAM-роль сервису (например, Lambda function, EC2 instance). Если разработчик имеет `iam:PassRole` для роли с высокими привилегиями (S3 `*`, IAM `*`), он может создать Lambda-функцию с этой ролью и обойти собственные ограничения, потому что Lambda выполнится с привилегиями переданной роли. Это классический вектор privilege escalation. Mandatory review каждого `iam:PassRole` в политике на PR-этапе должен подтверждать, что роль, которую можно передать, имеет права не выше, чем у самого разработчика, и что передача ограничена конкретными сервисами (condition `iam:PassedToService`). Контроль NIST SP 800-53 AC-6(2) (Non-Privileged Access) требует ограничения передачи привилегий.

## Ключевые выводы для собеседования

- **Самый критичный принцип:** Идентичность — это новый периметр. В облаке нет сети, которую можно огородить файрволом, поэтому IAM — единственный универсальный механизм разделения «свой-чужой». Каждый инцидент в облаке начинается с компрометации IAM credentials или с избыточных прав.
- **Главный компромисс:** Чем гранулярнее IAM-политики (ABAC, множество условий), тем точнее контроль доступа, но тем выше операционная нагрузка и риск ложных срабатываний, которые парализуют работу разработчиков. Баланс достигается через RBAC для статики + ABAC для контекста + автоматизированный exception-флоу.
- **Связь с ключевым стандартом:** NIST SP 800-53 AC-6 (Least Privilege) и NIST SP 800-207 (Zero Trust Architecture) — два столпа, на которых строится облачный IAM. Без минимальных прав IAM ничего не защищает. Без Zero Trust даже минимальные права могут быть использованы неожиданным субъектом.
- **Практическое правило:** Если у вас в production-аккаунте есть хотя бы одна рабочая нагрузка, использующая статические access keys вместо временных credentials — это не вопрос «произойдёт ли инцидент?», а вопрос «когда?». Миграция на временные credentials — highest priority IAM initiative.
