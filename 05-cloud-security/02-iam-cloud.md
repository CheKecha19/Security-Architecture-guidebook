# IAM в облаке: Управление доступом

## Что такое IAM?

Представь офисное здание. В здании есть комнаты: бухгалтерия, разработка, директорский кабинет, серверная. У каждого сотрудника — пропуск. На пропуске написано: куда можно, куда нельзя.

У бухгалтера — доступ в бухгалтерию. У разработчика — в разработку. У директора — везде. У курьера — только в приёмную.

**IAM** (Identity and Access Management) — это именно такая система пропусков. Но не для офиса, а для облака. Кто имеет доступ к каким ресурсам? Что можно делать? Как отслеживаются действия?

В облаке всё — ресурсы: серверы, базы данных, файлы, сети. IAM определяет: кто и что может с ними делать.

## Зачем нужен IAM?

### Ситуация 1: Уволенный сотрудник

Разработчик уволился. Его учётная запись осталась активной. Через месяц он зашёл в облако и удалил базу данных.

IAM: своевременное отключение доступа. Автоматизация. Ревизия прав.

### Ситуация 2: Права доступа

Все разработчики — администраторы в облаке. Один случайно удалил production-сервер.

IAM: принцип минимальных привилегий. Каждый получает только необходимые права.

### Ситуация 3: Кто что сделал?

Данные исчезли. Никто не признаётся. Кто виноват?

IAM: аудит действий. Логи. Можно посмотреть: кто, когда, что сделал.

## Основные концепции IAM

### Identity (Идентичность)

Кто это? Пользователь, группа, роль, сервис.

| Тип | Описание | Пример |
|-----|----------|--------|
| User | Человек | ivan.petrov@company.com |
| Group | Набор пользователей | Developers, Admins |
| Role | Временные права | Для сервиса, для сессии |
| Service Account | Для приложений | backup-service |

### Authentication (Аутентификация)

Доказать, что ты — это ты.

| Метод | Описание | Безопасность |
|-------|----------|--------------|
| Пароль | Знание секрета | Низкая |
| MFA | + телефон, токен | Высокая |
| SSO | Однократный вход | Средняя |
| API Key | Для сервисов | Средняя |
| Certificate | TLS-клиентский | Высокая |

### Authorization (Авторизация)

Что тебе разрешено?

| Модель | Описание | Пример |
|--------|----------|--------|
| RBAC | Роль-Based | Админ, Читатель, Редактор |
| ABAC | Атрибут-Based | Если отдел = бухгалтерия → доступ |
| ACL | Список доступа | Пользователь X → ресурс Y |
| Policy | Политика | JSON с правилами |

### Accountability (Подотчётность)

Кто что сделал? Логи, аудит.

## IAM в AWS

### Пользователи и группы

- **IAM Users** — люди или сервисы
- **IAM Groups** — коллекция пользователей (Developer, Admin)
- **IAM Roles** — временные права

### Политики

JSON-документ: кто может что делать.

```
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": "arn:aws:s3:::bucket-name/*",
      "Condition": {
        "IpAddress": {
          "aws:SourceIp": "10.0.0.0/8"
        }
      }
    }
  ]
}
```

### Лучшие практики AWS IAM

| Практика | Описание |
|----------|----------|
| Root account | Не используйте для повседневных задач |
| MFA | На root, на админов |
| Роли, не ключи | Используйте роли, не access keys |
| Минимальные права | Только необходимые права |
| Ротация ключей | Меняйте регулярно |
| Аудит | CloudTrail для всех действий |

## IAM в Azure

### Azure AD

- **Users** — люди
- **Groups** — коллекции
- **Service Principals** — для приложений
- **Managed Identities** — автоматическое управление

### RBAC

| Роль | Описание |
|------|----------|
| Owner | Полный доступ |
| Contributor | Чтение + запись |
| Reader | Только чтение |
| User Access Admin | Управление доступом |

### Лучшие практики Azure

| Практика | Описание |
|----------|----------|
| PIM | Privileged Identity Management — временный доступ |
| Conditional Access | MFA, геолокация, устройство |
| Managed Identities | Не хранить секреты в коде |
| RBAC | Минимальные роли |

## IAM в Google Cloud

### IAM

- **Members** — users, groups, service accounts
- **Roles** — предопределённые или кастомные
- **Policies** — привязка ролей к ресурсам

### Роли

| Тип | Описание |
|-----|----------|
| Basic | Owner, Editor, Viewer |
| Predefined | Для конкретных сервисов |
| Custom | Создаёте сами |

### Лучшие практики GCP

| Практика | Описание |
|----------|----------|
| Service accounts | Для приложений, не для людей |
| Workload Identity | Для Kubernetes |
| Policy Intelligence | Анализ избыточных прав |
| Audit Logs | Для всех действий |

## Zero Trust в облаке

### Принципы

| Принцип | Описание |
|---------|----------|
| Не доверяй, проверяй | Каждый запрос — проверка |
| Минимальный доступ | Только необходимое |
| Предполагай нарушение | Защита от insider threat |

### Реализация

- Identity-Aware Proxy (IAP)
- BeyondCorp (Google)
- Azure AD Application Proxy

### Ответы на чек-лист

1. **IAM** — Identity and Access Management. Управление идентичностями и доступом к ресурсам облака.

2. **Три концепции**: Identity (кто), Authentication (доказательство личности), Authorization (что разрешено), Accountability (логи, аудит).

3. **RBAC** — Role-Based Access Control. Доступ на основе ролей: Админ, Читатель, Редактор.

4. **ABAC** — Attribute-Based Access Control. Доступ на основе атрибутов: отдел, время, локация.

5. **Методы аутентификации**: пароль, MFA, SSO, API Key, сертификат.

6. **Аудит важен**, потому что позволяет отследить: кто, когда, что сделал. Без аудита — невозможно расследовать инциденты.

7. **Zero Trust** — модель, где ничему не доверяют. Каждый запрос проверяется. Минимальный доступ. Предполагается нарушение.

8. **AWS IAM**: не использовать root, MFA на всё, роли вместо ключей, минимальные права, ротация ключей, CloudTrail.

9. **PIM** — Privileged Identity Management (Azure). Временный доступ для привилегированных ролей.

10. **Ограничения IAM**: сложность управления в больших организациях, конфликт политик, затруднённый аудит в мультиоблаке.
