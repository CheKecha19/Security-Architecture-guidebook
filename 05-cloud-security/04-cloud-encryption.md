# Шифрование в облаке

## Что такое шифрование в облаке?

Представь, что ты отправляешь письмо. Не электронное, а настоящее, бумажное. Ты кладешь его в конверт. Но конверт — прозрачный. Все видят, что внутри.

Теперь представь, что ты кладешь письмо в сейф. Сейф запираешь. Отправляешь. Получатель имеет ключ. Открывает. Читает. Никто по пути не видел содержимого.

**Шифрование** — это именно такой "сейф" для данных. Превращает понятную информацию в непонятный набор символов. Только у кого есть ключ — может прочитать.

В облаке шифрование — критично. Данные лежат на чужих серверах. Едут по чужим сетям. Шифрование защищает: даже если кто-то перехватит — не прочитает.

## Зачем нужно шифрование?

### Ситуация 1: Утечка диска

Жёсткий диск облачного сервера вышел из строя. Облачный провайдер заменил его. Старый диск — куда-то делся. Если данные были зашифрованы — никто не прочитает.

### Ситуация 2: Перехват трафика

Данные едут от сервера к клиенту. Хакер перехватывает пакеты. Если трафик не шифрован — он видит всё. Если шифрован — видит мусор.

### Ситуация 3: Compliance

GDPR, PCI DSS, HIPAA — все требуют шифрования. Без шифрования — нет compliance.

## Типы шифрования

### At Rest (в покое)

Данные хранятся на диске. Не используются.

| Где | Как |
|-----|-----|
| Диски (EBS) | Шифрование на уровне хранилища |
| S3 / Blob / GCS | Server-side encryption |
| Базы данных | Transparent Data Encryption (TDE) |
| Backup | Шифрование backup |

### In Transit (в движении)

Данные передаются по сети.

| Где | Как |
|-----|-----|
| HTTPS | TLS 1.2+ |
| VPN | IPSec, SSL |
| Базы данных | SSL/TLS для подключения |
| API | mTLS (mutual TLS) |

### In Use (в использовании)

Данные обрабатываются в памяти. Самое сложное.

| Технология | Описание |
|------------|----------|
| Confidential Computing | Зашифрованная память |
| Intel SGX | Software Guard Extensions |
| AMD SEV | Secure Encrypted Virtualization |

## Управление ключами (KMS)

### Что такое KMS?

**Key Management Service** — сервис для управления ключами шифрования.

| Возможность | Описание |
|-------------|----------|
| Генерация | Создание ключей |
| Хранение | Защищённое хранение |
| Ротация | Автоматическая смена |
| Аудит | Кто использовал ключ |
| Удаление | Безопасное уничтожение |

### Типы ключей

| Тип | Описание | Когда использовать |
|-----|----------|-------------------|
| AWS managed | Управляет облако | Простые случаи |
| Customer managed | Управляете вы | Контроль, политики |
| Customer provided | Ваш ключ | Максимальный контроль |

### Bring Your Own Key (BYOK)

Вы генерируете ключ. Храните у себя. Используете в облаке через HSM.

### Hold Your Own Key (HYOK)

Вы генерируете и храните ключ. Облако не имеет доступа.

## Шифрование в разных облаках

### AWS

|--------|------------|

### GCP

| Сервис | Шифрование |
|--------|------------|

## Архитектурные решения

### End-to-end шифрование

Клиент шифрует → передаёт → облако хранит → передаёт → клиент расшифровывает.

Облако никогда не видит открытые данные.

### Шифрование на уровне приложения

Приложение шифрует данные перед отправкой в базу. База знает только зашифрованные данные.

### Поле-уровневое шифрование

Шифруются отдельные поля: номер карты — да, имя — нет.

## Лучшие практики

| Практика | Описание |
|----------|----------|
| Шифруйте всё | По умолчанию, всегда |
| Используйте KMS | Не храните ключи в коде |
| Ротация | Меняйте ключи регулярно |
| Минимум прав | Кто может расшифровать |
| Аудит | Кто использовал ключ |
| Backup ключей | HSM, escrow |

### Ответы на чек-лист

1. **Шифрование** — преобразование данных в нечитаемый формат. Только с ключом можно прочитать.

2. **Три типа**: At Rest (на диске), In Transit (в сети), In Use (в памяти).

3. **KMS** — Key Management Service. Управление ключами: генерация, хранение, ротация, аудит.

4. **Типы ключей**: AWS managed (облако), Customer managed (вы), Customer provided (ваш ключ).

5. **BYOK** — Bring Your Own Key. Вы генерируете ключ, используете в облаке.

6. **S3 шифруется**: SSE-S3 (ключ AWS), SSE-KMS (ключ KMS), SSE-C (ваш ключ).

7. **End-to-end** — клиент шифрует, облако не расшифровывает, клиент расшифровывает.

8. **TDE** — Transparent Data Encryption. База шифрует данные автоматически.

9. **Лучшие практики**: шифруйте всё, используйте KMS, ротация, минимум прав, аудит, backup ключей.

10. **Compliance**: GDPR, PCI DSS, HIPAA требуют шифрования. Без него — штрафы.
