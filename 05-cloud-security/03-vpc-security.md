# VPC и сетевая безопасность в облаке: Изоляция, сегментация и микросегментация в виртуальных приватных сетях

## Что такое VPC?

Представьте закрытый жилой комплекс. У комплекса есть общий периметр с забором и КПП, через который проходят все — жильцы, гости, курьеры. Но внутри комплекс разделён на корпуса, у каждого свой подъезд с отдельным домофоном. Более того, внутри корпуса квартиры изолированы друг от друга капитальными стенами — если в одной квартире пожар, соседняя не пострадает. Есть общие зоны (парковка, детская площадка, подземный паркинг), куда доступ есть у всех жильцов, и технические помещения (серверная, электрощитовая), куда может войти только управляющая компания. Есть чёрный ход для вывоза мусора, через который нельзя войти, только выйти. Это и есть **VPC (Virtual Private Cloud)** в чистом виде — не просто «виртуальная сеть в облаке», а многоуровневая архитектура изоляции, маршрутизации и сегментации, где каждый уровень имеет собственный набор правил доступа.

Фундаментальная проблема безопасности, которую решает VPC: в облаке ваши серверы физически соседствуют с серверами тысяч других организаций на одних и тех же физических машинах. Без сетевой изоляции ваша база данных потенциально доступна любому, кто угадает IP-адрес. VPC создаёт «частную вселенную» с вашим собственным IP-пространством, вашими правилами маршрутизации и вашими межсетевыми экранами, которую не видят другие пользователи облака, если вы явно не открыли доступ. Это эквивалент того, что в огромном офисе Open Space каждому сотруднику выдали изолированный звуконепроницаемый кабинет со своим замком и своими правилами входа.

## Эволюция и мотивация

Классическая сетевая безопасность строилась на трёх аксиомах: физический периметр (файрвол на границе дата-центра), жёсткая внутренняя сегментация (VLAN, ACL на коммутаторах) и доверие ко всему, что внутри периметра. Администратор, подключённый к внутренней сети, мог обращаться к любому серверу, потому что «он уже внутри, значит проверен». Эта модель, известная как perimeter-based security, работала до тех пор, пока периметр был чётко очерчен одним зданием и одним коммутатором.

Облачные вычисления уничтожили периметр. Серверы распределены по зонам доступности. Одна подсеть может быть в регионе А, другая — в регионе Б. Трафик между ними идёт через магистральную сеть провайдера, а не через ваш коммутатор. Второй тектонический сдвиг — микросервисная архитектура: вместо трёх серверов (веб, апп, БД) появились сотни подов, каждый из которых является отдельным сетевым субъектом, и традиционная VLAN-сегментация с сотней правил на границе не масштабируется.

Третий фактор: горизонтальное масштабирование атак. Раньше злоумышленник, взломавший веб-сервер, видел максимум десяток соседних хостов в той же VLAN, которые нужно разведывать. В облаке lateral movement через Security Group misconfigurations, VPC Peering и открытые endpoint'ы позволяет прыгать между зонами доступности, регионами и даже разными облаками через транзитивные шлюзы.

Индустрия прошла эволюцию: от trust-based network (доверяем всем внутри) к zero-trust network (проверяем каждый пакет между любыми субъектами, даже внутри VPC). VPC стал не просто изолятором от внешнего мира, а фреймворком для реализации микросегментации — правил доступа на уровне отдельных ресурсов, а не целых подсетей. NIST SP 800-207 (Zero Trust Architecture) формализует это требование, а AWS VPC, Azure VNet, GCP VPC предоставляют примитивы для его реализации.

## Зачем нужна VPC?

**Сценарий 1: Утечка через публичную базу данных.** Стартап разворачивает PostgreSQL на EC2-инстансе, назначает публичный IP для удобства удалённого подключения разработчиков и «временно» открывает порт 5432 в Security Group с `0.0.0.0/0`. Через неделю Shodan индексирует базу, ботнет начинает брутфорс-атаку на PostgreSQL, успешно подбирает пароль и эксфильтрует данные. С VPC: база данных размещается в приватной подсети без публичного IP и без Internet Gateway; разработчики подключаются через VPN в VPC или Session Manager; порт 5432 открыт только для ограниченного диапазона адресов.

**Сценарий 2: Lateral movement в production-окружении.** Злоумышленник компрометирует веб-сервер в публичной подсети. Если Security Group веб-сервера разрешает исходящий трафик на все порты во всех направлениях (дефолтное поведение Security Group в некоторых облаках), он начинает сканирование внутренних подсетей и находит незакрытый Elasticsearch на порту 9200 с данными клиентов. С VPC + микросегментацией: каждый сервис имеет Security Group, разрешающую только порты, необходимые для его функции — веб-сервер может обращаться только к app-серверу на порту 8080 и ни к чему больше. Lateral movement физически невозможен, потому что сетевой пакет до Elasticsearch будет отброшен.

**Сценарий 3: Гибридная инфраструктура банка.** Банк имеет legacy mainframe в собственном дата-центре и новые микросервисы в облаке. Mainframe не может быть перенесён в облако, но должен обмениваться данными с облачными сервисами. VPC решает это через VPN (IPsec) или выделенный канал (AWS Direct Connect, Azure ExpressRoute, GCP Cloud Interconnect). Трафик идёт в обход интернета, зашифрован, с гарантированной пропускной способностью, а доступ регулируется как облачными (Security Groups), так и on-premise (физический файрвол) правилами.

**Сценарий 4: Мультиаккаунтная изоляция.** Организация использует AWS Organizations с отдельными аккаунтами для Development, Staging, Production. Без VPC-изоляции сервер в Development может случайно обратиться к production-базе, если разработчик скопипастил production-конфигурацию в development-код. С VPC: каждый аккаунт имеет свой VPC с непересекающимся CIDR-диапазоном; связность между аккаунтами — через явные VPC Peering или Transit Gateway с одобрением обеих сторон; трафик между Development и Production по умолчанию невозможен, пока security-команда не создаст routing table entry.

**Сценарий 5: DDoS-защита и ingress-контроль.** Интернет-магазин подвергается volumetric DDoS-атаке. VPC с правильно спроектированным ingress path (CloudFront/Cloud CDN + AWS Shield/Azure DDoS Protection + WAF) принимает трафик на границе облака, фильтрует на уровне CDN/WAF, и только очищенный трафик доходит до VPC через Application Load Balancer. Без VPC-архитектуры весь сырой трафик приходил бы напрямую на инстанс в публичной подсети, и никакой upscaling не спас бы от 500 Gbps flood.

## Основные концепции

### Подсети: градуированная модель видимости

Подсеть — не просто «диапазон IP-адресов». Это фундаментальная единица изоляции, определяющая политику маршрутизации для всех ресурсов, находящихся внутри неё. Через Route Table, привязанную к подсети, вы управляете тем, может ли ресурс отправлять и получать трафик из интернета, других VPC, on-premise сети или вообще куда-либо.

**Публичная подсеть** имеет маршрут `0.0.0.0/0 → Internet Gateway`. Любой ресурс в этой подсети с публичным IP может принимать трафик из интернета и инициировать исходящие соединения. Применение: Application Load Balancer, Bastion Host (хотя в современной архитектуре Bastion заменяют Session Manager/Identity-Aware Proxy), публичные API-эндпоинты. Ошибка: размещение базы данных в публичной подсети «потому что так проще подключиться» — классический anti-pattern, открытый в 40% аудитов.

**Приватная подсеть** не имеет прямого маршрута в интернет (`0.0.0.0/0 → NAT Gateway` для исходящего, но нет Internet Gateway для входящего). Внешний субъект не может инициировать соединение с ресурсом в приватной подсети, но ресурс может скачивать обновления, отправлять метрики и писать в внешние API. Применение: базы данных, внутренние сервисы, серверы приложений. Контроль NIST SP 800-53 AC-4 (Information Flow Enforcement) требует, чтобы критичные сервисы не имели прямого доступа из внешних сетей.

**Изолированная подсеть** — максимальный уровень изоляции: в Route Table нет маршрута ни к Internet Gateway, ни к NAT Gateway, ни к Virtual Private Gateway. Единственная связность — с другими подсетями того же VPC и, возможно, с VPC Endpoints для доступа к облачным сервисам без интернета. Применение: системы хранения ключей (HSM), чувствительные базы данных, compliance-изолированные окружения (PCI DSS CDE), где стандарт PCI DSS Req 1.3.3 требует изоляции cardholder data environment от всех остальных сетей.

### Межсетевые экраны облака: Security Groups vs NACL

Это два независимых слоя фильтрации, работающие на разных уровнях и с разной логикой. Их необходимо использовать вместе — каждый закрывает пробелы другого.

**Security Group** — stateful-файрвол на уровне сетевого интерфейса (ENI), привязанный к конкретному ресурсу (EC2, RDS, Lambda в VPC). Ключевые свойства: stateful (если разрешён исходящий запрос, ответный трафик пропускается автоматически, без явного правила), только Allow-правила (нет Deny — подход whitelist), может ссылаться на другие Security Groups как на источник (Source: sg-123456 — разрешить доступ с любого ресурса, имеющего эту SG), а не только IP-адреса.

**Network ACL (NACL)** — stateless-файрвол на уровне подсети. Ключевое отличие: stateless (для каждого направления нужны отдельные правила и для запроса, и для ответа), поддерживает Allow и Deny (можно явно запретить трафик из конкретного IP), применяется ко всем ресурсам в подсети, независимо от их Security Groups.

Архитектурный компромисс: Security Groups дают гранулярный контроль на уровне инстанса, но не могут запретить трафик (только разрешить). NACL дают запрещающие правила и охватывают всю подсеть, но stateless-модель создаёт операционную сложность. Паттерн defence-in-depth: Security Groups — первая линия (разрешить только нужное), NACL — вторая (запретить конкретные IP-диапазоны, известные плохие акторы, или ограничить исходящий трафик на уровне подсети для compliance).

### VPC Peering и Transit Gateway: масштабирование связности

В облаке быстро возникает десяток VPC, которым нужно общаться: shared services VPC (мониторинг, логирование, CI/CD), team-specific VPC, production/staging VPC, VPC для внешних партнёров. Наивное решение — попарный VPC Peering — создаёт full-mesh из `n*(n-1)/2` соединений, каждое со своей route table, каждое нужно настраивать вручную.

**VPC Peering** — прямое соединение двух VPC через магистральную сеть провайдера, без интернета, с высокой пропускной способностью и низкой задержкой. Ограничения: нетранзитивно (A—B и B—C не означает A—C), требует непересекающихся CIDR-диапазонов, не поддерживает Security Group cross-referencing между регионами (в некоторых облаках).

**Transit Gateway** (AWS Transit Gateway, Azure Virtual WAN Hub, GCP Cloud Router) — центральный маршрутизатор, к которому подключаются все VPC (hub-and-spoke модель). Каждый VPC имеет одно соединение с Transit Gateway, который маршрутизирует трафик между spoke-сетями согласно routing tables. Архитектурное преимущество: 50 VPC требуют 50 аттачментов (а не 1225 пирингов), централизованный контроль маршрутизации, поддержка транзитивной связности (VPC A → TGW → VPC C), attachment-level изоляция (через TGW route tables можно запретить spoke-to-spoke и разрешить только spoke-to-hub).

### VPC Endpoints и PrivateLink: приватный доступ к сервисам

Традиционный доступ к облачным сервисам (S3, DynamoDB, Kinesis) из приватной подсети требует выхода через NAT Gateway в интернет, что добавляет latency, стоимость (трафик через NAT) и поверхность атаки (пакеты идут через публичную сеть).

**VPC Endpoint (Gateway тип)** — виртуальное устройство в VPC, которое маршрутизирует трафик к S3/DynamoDB напрямую через магистральную сеть провайдера, не выходя в интернет. Трафик не проходит через NAT Gateway, не имеет стоимости NAT-трафика и дополнительно защищён Endpoint-политикой, ограничивающей, к каким бакетам можно обращаться через этот endpoint.

**VPC Endpoint (Interface тип / PrivateLink)** — ENI в вашей подсети, через которую трафик идёт к любому облачному сервису (не только S3/DynamoDB) или даже к сервису в другом VPC через PrivateLink. Применение: доступ к вашему SaaS-продукту из VPC клиента без публикации в интернет — клиент создаёт Interface Endpoint к вашему NLB, и трафик идёт через приватную сеть.

Контроль NIST SP 800-53 AC-4 (Information Flow Enforcement) требует, чтобы критичные информационные потоки проходили через контролируемые каналы. VPC Endpoints реализуют это требование, позволяя ограничить доступ к облачным сервисам конкретными endpoint-политиками.

## Сравнение подходов к сетевой изоляции

| Критерий | Security Groups (уровень ресурса) | NACL (уровень подсети) | Network Firewall (уровень VPC) |
|----------|-----------------------------------|------------------------|--------------------------------|
| Модель фаервола | Stateful | Stateless | Stateful |
| Гранулярность | ENI (сетевой интерфейс конкретного ресурса) | Вся подсеть: все ресурсы в ней | Весь VPC: меж-VPC, ingress/egress на периметре |
| Типы правил | Allow only (whitelist) | Allow + Deny | Allow + Deny, IDS/IPS сигнатуры, domain-based фильтрация |
| Сложность управления | Низкая (allow list + auto-response) | Средняя (нужны правила для request+response) | Высокая (централизованный policy engine) |
| Производительность | Без overhead: правила — часть hypervisor networking | Минимальный overhead | Заметный overhead при глубокой инспекции пакетов (TLS decryption) |
| Типичный сценарий | Разрешить web-инстансам общаться с app-инстансами на порту 8080 | Запретить исходящий трафик в страны из санкционного списка для compliance | Централизованный контроль egress трафика всего VPC, блокировка C&C-серверов по доменным именам |
| Связь со стандартами | NIST 800-53 AC-3 (Access Enforcement), CIS v8 Control 12.4 | NIST 800-53 AC-4 (Information Flow Enforcement) | NIST 800-53 SI-4 (System Monitoring), PCI DSS Req 1.1 |

Сетевой файрвол (AWS Network Firewall, Azure Firewall, GCP Cloud Firewall) добавляет уровень, которого нет у Security Groups и NACL: stateful-инспекция трафика на периметре VPC с IDS/IPS-сигнатурами, доменной фильтрацией (разрешить доступ только к `*.internal-api.example.com`, запретить всё остальное) и возможностью TLS-decryption на intermediate прокси.

Выбор стратегии: Security Group — mandatory baseline для каждого ресурса, определяет микросегментацию. NACL — stateless protection на уровне подсети, в основном для Deny-правил (geo-blocking, blocking known-bad IPs). Network Firewall — central egress control для compliance, где нужно гарантировать, что ни один ресурс VPC не может обращаться к неавторизованным эндпоинтам.

## Уроки из реальных инцидентов

### Инцидент Imperva (2019): Утечка через незащищённый инстанс

Хронология: в августе 2019 года Imperva, поставщик облачных WAF и DDoS-защиты, раскрыла, что внутренняя база данных Cloud WAF была скомпрометирована. Злоумышленник получил доступ к AWS API-ключам и к базе данных, содержащей email-адреса, хэши паролей и API-ключи клиентов. Расследование показало, что в августе 2017 года команда создала EC2-инстанс для тестирования, который имел избыточные права и отсутствие сетевых ограничений, и забыла о нём. Спустя два года инстанс был взломан и использован как точка входа.

VPC-анализ: скомпрометированный инстанс находился в подсети с чрезмерно широкой связностью — Security Groups разрешали доступ ко внутренним сервисам, а NACL не ограничивали lateral movement. Кроме того, отсутствовала VPC Flow Log сегрегация, не было мониторинга межподсетного трафика, и никто не заметил, что инстанс, неиспользуемый два года, вдруг начал генерировать трафик.

Как VPC-архитектура изменила бы исход: каждый инстанс в Security Group с минимально необходимыми правилами (разрешить только то, что нужно для тестирования), автоматический expiry и уничтожение тестовых ресурсов через 48 часов (Tag-based lifecycle policies), VPC Flow Logs с аномалия-детекшн: любой инстанс, не имевший трафика более 30 дней, при возобновлении трафика генерирует critical alert. Контроль CIS v8, п. 2.1 (Inventory of Software Assets), требует учёта всех активов, включая неиспользуемые.

Урок: в VPC каждый ресурс должен иметь documented network connectivity justification. Неиспользуемые ресурсы должны автоматически уничтожаться, а не ждать, пока их найдут злоумышленники. Мониторинг VPC Flow Logs — не опциональная функция, а mandatory baseline.

### Инцидент Adobe (2013, уроки для облачной эпохи): Утечка через отсутствие сегментации

Хотя инцидент произошёл до массового облачного adoption, его архитектурный урок прямо применим. Злоумышленники взломали веб-сервер, обслуживающий Adobe Connect, и обнаружили, что исходный код других продуктов Adobe (Photoshop, Acrobat) лежит на том же сетевом сегменте без сегментации. Масштаб: исходный код флагманских продуктов, 38 миллионов записей пользователей с зашифрованными паролями и банковскими данными.

В облачном эквиваленте: отсутствие VPC-микросегментации. Сервис Adobe Connect должен был находиться в отдельной подсети с Security Group, исключающей доступ к любым ресурсам, не являющимся его зависимостями. VPC Flow Logs показали бы аномальный объём трафика между веб-сервером и файловым хранилищем с исходным кодом, но для этого нужна была baseline network behavior analysis. Контроль NIST SP 800-53 AC-4 (Information Flow Enforcement) требует явного разрешения и мониторинга всех информационных потоков между системами разного уровня чувствительности.

Урок: микросегментация — не про удобство, а про ограничение blast radius. Компрометация одного сервиса не должна давать доступ к данным другого, даже если оба находятся в том же VPC. Принцип least-privilege network access: между двумя сервисами существует connectivity только если она явно задекларирована и одобрена.

| Класс инструмента | Назначение | Позиция в жизненном цикле | Связь со стандартами |
|-------------------|------------|---------------------------|----------------------|
| Cloud Network Firewall | Stateful фильтрация на уровне VPC с IDS/IPS-сигнатурами, доменной фильтрацией и централизованным policy management | Runtime (периметр VPC, ingress/egress весь трафик) | NIST 800-53 SI-4 (System Monitoring), PCI DSS Req 1.1-1.4 |
| VPC Flow Logs Analysis | Запись метаданных каждого сетевого потока (source, dest, port, bytes, accept/reject), анализ аномалий и визуализация | Continuous runtime monitoring + forensic investigation | NIST 800-53 AC-4, AU-3 (Audit Records), CIS v8 Control 12.5 |
| Network Policy as Code | Декларативное описание разрешённых сетевых путей между сервисами (Kubernetes Network Policies, OPA для облачных эндпоинтов) | Pre-deployment (CI/CD) + runtime enforcement | OWASP ASVS V4.1, NIST 800-207 Zero Trust |
| Cloud Access Security Broker (CASB) | Контроль доступа к облачным сервисам, DLP для сетевого трафика к SaaS, shadow IT detection | Runtime: проксирование или API-интеграция с облачными сервисами | CIS v8 Control 3.3 (Data Protection), NIST 800-53 AC-4 |
| Service Mesh (Istio, Linkerd, AWS App Mesh) | Транспортная безопасность и микросегментация на уровне L7 (HTTP/gRPC): mTLS, RBAC для сетевых вызовов между сервисами | Runtime: sidecar proxy для каждого пода/контейнера | NIST 800-207 Zero Trust, OWASP ASVS V10 (Service Security) |

VPC Flow Logs — критичный, но часто игнорируемый инструмент. Каждый отклонённый пакет в VPC Flow Logs — это потенциальная попытка несанкционированного доступа. Инструменты анализа (Amazon Detective, Azure Network Watcher, GCP VPC Flow Logs в BigQuery) позволяют агрегировать отклонённые потоки, находить сканирование портов, попытки подключения к запрещённым эндпоинтам и неожиданные cross-VPC соединения. Метрика эффективности: 100% VPC и подсетей имеют включённые Flow Logs с доставкой в SIEM в реальном времени (latency менее 5 минут).

Service Mesh представляет следующий эволюционный шаг микросегментации: вместо фильтрации на уровне IP и портов (L3/L4), Service Mesh оперирует на уровне сервисных идентичностей (L7). Правило: «Checkout Service может вызывать Payment Service на POST /charge» — не привязано к IP-адресам, которые меняются при каждом деплое. mTLS между sidecar-прокси гарантирует шифрование и взаимную аутентификацию каждого межсервисного вызова, реализуя Zero Trust на практике. Архитектурный overhead: sidecar-прокси добавляют latency (2-5ms на хоп) и потребление ресурсов (50-100 MB RAM на sidecar), что для high-throughput систем может быть неприемлемо; альтернатива — eBPF-based решения (Cilium), работающие на уровне ядра без sidecar.

## Архитектурные решения

Построение защищённой облачной сетевой архитектуры требует интеграции на четырёх слоях:

**Периметр облака:** WAF, DDoS Protection, CDN — фильтрация трафика до попадания в VPC. На этом слое отбрасываются volumetric-атаки, SQL injection/XSS через WAF-правила, и трафик из географических регионов, где нет бизнеса (geo-blocking).

**Периметр VPC:** Network Firewall + Internet Gateway/NAT Gateway. Egress filtering — гарантия, что скомпрометированный ресурс не сможет обращаться к C&C-серверам, эксфильтровать данные на внешние серверы или майнить криптовалюту. Ingress filtering — ограничение входящего трафика разрешёнными протоколами и портами.

**Подсети и микросегментация:** NACL + Security Groups. Каждая подсеть имеет свою политику, каждая группа Security Group привязана к логической функции (web-sg, app-sg, db-sg), и разрешены только явно задекларированные пути.

**Межсервисный уровень:** Service Mesh с mTLS. Даже внутри подсети, где все поды могут обращаться друг к другу по IP, Service Mesh проверяет, разрешён ли вызов от сервиса A к сервису B на основе сервисной идентификации, а не IP. Это финальная страховка: если Security Group и Network Policy пропустили трафик, Service Mesh проверит его на уровне приложения.

Метрики эффективности: доля ресурсов, размещённых в приватных подсетях (цель: > 95%, публичные только для ALB/ingress); процент VPC Flow Logs coverage (цель: 100%); число сетевых инцидентов с lateral movement успешным (цель: 0); MTTD для аномального сетевого трафика (цель: менее 5 минут).

Красные флаги деградации: ресурсы в публичных подсетях без WAF перед ними; Security Groups с портом 22 открытым для `0.0.0.0/0`; отсутствие VPC Flow Logs на production-подсетях; маршрут `0.0.0.0/0 → IGW` присутствует в Route Table приватной подсети (прямое нарушение дизайна).

Сценарий миграции legacy плоской сети на VPC-микросегментацию: этап 1 — полный inventory всех сетевых потоков через VPC Flow Logs за 90 дней, построение baseline: кто с кем общается, какие порты. Этап 2 — группировка ресурсов по логическим сервис-доменам (web, app, data, monitoring, admin). Этап 3 — создание Security Groups для каждого домена с разрешёнными путями, применение в monitor-only режиме (без блокировки) через дополнительные Security Group. Этап 4 — валидация: отсутствие отклонённых пакетов на легитимном трафике в течение недели. Этап 5 — включение enforcing mode с exception window для экстренных случаев.

## Подготовка к собеседованию: Common pitfalls & key questions

**Типичная ошибка 1: «NAT Gateway — это просто шлюз для выхода приватных подсетей в интернет».**
Упрощение. NAT Gateway — не только шлюз, но и точка наблюдения за egress-трафиком. Все исходящие потоки из приватных подсетей проходят через NAT Gateway — это идеальное место для VPC Flow Logs, egress-фильтрации и обнаружения data exfiltration. Если организация использует NAT Gateway, но не мониторит его Flow Logs, она слепа к исходящим атакам.

**Типичная ошибка 2: «Security Group с разрешением "All traffic" для Source: sg-same — это нормальная практика для микросервисов».**
Нет. Хотя это удобно, это эквивалент отсутствия микросегментации внутри группы. Если любой ресурс с этой SG может открыть любое TCP-соединение к любому другому ресурсу с той же SG на любом порту — lateral movement после компрометации одного ресурса неограничен. Минимальный стандарт: каждая SG разрешает конкретные порты (8080, 5432, 6379), а не весь диапазон.

**Типичная ошибка 3: «Если у меня VPC, я защищён».**
VPC — изолированная all-inclusive песочница, но её защищённость зависит от конфигурации. Открытый Internet Gateway, Security Group с `0.0.0.0/0`, публичный IP на базе данных — и VPC ничего не защищает. VPC предоставляет примитивы; безопасность — в том, как они сконфигурированы.

**Сложный вопрос 1: «Как вы спроектируете VPC для multi-tenant SaaS-платформы, где tenant'ы (B2B-клиенты) не должны видеть данные друг друга, но приложение должно обслуживать их из одной кодовой базы?»**
Проверяется: способность применить микросегментацию к multi-tenancy. Ответ: на сетевом уровне — каждый tenant получает отдельную подсеть со своей NACL, запрещающей cross-tenant трафик. На уровне приложения — tenant isolation через базу данных (row-level security + tenant-id во всех запросах). Для compliance-чувствительных tenant'ов, требующих dedicated инфраструктуру (банки, healthcare) — отдельный VPC с PrivateLink к shared application layer. Компромисс: выделенные VPC на каждого tenant'а дают максимальную изоляцию, но не масштабируются после 20-30 tenant'ов; общая инфраструктура с tenant-тегированием и row-level security масштабируется, но сложнее аудируется. Контроль OWASP ASVS V4.1.1 требует, чтобы multi-tenant приложение обеспечивало изоляцию на уровне данных.

**Сложный вопрос 2: «Ваша команда запускает тысячи Lambda-функций, которым нужен доступ к RDS в VPC. Каждая функция имеет VPC-аттачмент, что вызывает cold start latency 5-7 секунд. Какие альтернативные архитектуры вы рассмотрите?»**
Проверяется: знание компромисса security vs performance в serverless networking. Ответ: первая альтернатива — RDS Proxy, который поддерживает IAM-аутентификацию через функцию без VPC-аттачмента (функция вне VPC, RDS Proxy с IAM auth endpoint, доступным публично с TLS client certificate). Минус: трафик идёт через интернет. Вторая альтернатива: поднять API-шлюз в VPC перед базой, функция вызывает шлюз через приватный API Gateway endpoint. Третья альтернатива (наилучшая для high-security): Hyperplane ENI trunking, уменьшающее число ENI с N (на функцию) до 1 на Security Group — сохраняет VPC-аттачмент, но редуцирует cold start overhead. Компромисс: первая альтернатива жертвует сетевой изоляцией ради скорости; третья — сохраняет изоляцию, но сложнее в настройке. Контроль NIST SP 800-53 AC-4 требует enforcement информационных потоков между функциями и базой.

**Сложный вопрос 3: «Как VPC Flow Logs и Network Firewall взаимодействуют для обнаружения data exfiltration? Опишите детекционный пайплайн для аномального исходящего трафика.»**
Проверяется: практическое понимание сетевого мониторинга. Ответ: VPC Flow Logs пишут метаданные каждого потока в S3/CloudWatch/BigQuery. Инструмент анализа агрегирует объём исходящего трафика по source resource, destination IP, порту. Baseline: для каждого ресурса вычисляется средний egress-объём за 30 дней с deviation bands. Детекция: любой ресурс, чей egress-объём превышает baseline на 3 sigma (или абсолютный порог — 1GB за час для типичного веб-сервера), генерирует alert. Network Firewall добавляет domain-based filtering: если ресурс пытается обратиться к `.xyz`, `.tk` или known malicious domain — alert с немедленной блокировкой. Вместе они покрывают и volume-based, и signature-based ексфильтрацию. Компромисс: VPC Flow Logs не видят содержимое пакетов (только метаданные), поэтому шифрованный ексфильтрационный канал через HTTPS не обнаруживается — для этого нужен TLS-decryption прокси, который добавляет latency и privacy concerns.

## Чек-лист понимания

- [ ] Как stateful природа Security Group упрощает конфигурацию, но создаёт риск, если ресурс скомпрометирован?
- [ ] В каком случае NACL с Deny-правилами — единственный способ реализовать compliance-требование (например, блокировка конкретной страны)?
- [ ] Почему VPC Peering не масштабируется за пределы 5-10 VPC и какой архитектурный предел это нарушает?
- [ ] Как VPC Endpoint (Gateway тип) для S3 предотвращает data exfiltration в дополнение к приватному доступу?
- [ ] В каком случае Service Mesh с mTLS избыточен по сравнению с Security Group микросегментацией?
- [ ] Почему мониторинг VPC Flow Logs — обязательное, а не опциональное требование для production-окружения?
- [ ] Как Network Firewall с доменной фильтрацией блокирует C&C-канал, который проходит через HTTPS?
- [ ] В каком случае PrivateLink между VPC двух разных AWS-аккаунтов безопаснее, чем VPC Peering?
- [ ] Почему микросегментация на основе IP-адресов не работает в Kubernetes, и как Service Mesh решает эту проблему?

### Ответы на чек-лист

1. **Ответ:** Security Group защищает только от несанкционированного сетевого соединения, но не от ошибок конфигурации на других слоях. Если Security Group открыта для `0.0.0.0/0:5432`, любой IP в мире может попытаться подключиться; единственная защита — пароль базы данных, которая может быть слабой. Публичная подсеть также делает ресурс видимым для сканирования через Shodan и Censys. Compliance-стандарты (PCI DSS Req 1.3.1) требуют размещения баз данных с sensitive данными в изолированных сетях, что физически невозможно в публичной подсети.

2. **Ответ:** Stateful означает: если SG разрешает исходящий HTTPS-запрос, входящий ответный трафик пропускается без явного правила. Это упрощает конфигурацию (не нужно правило для ephemeral портов). Но если ресурс скомпрометирован, stateful логика не видит разницы между «ответным трафиком от легитимного API» и «ответным трафиком от C&C-сервера, к которому malware инициировал соединение» — оба пройдут. Решение: egress-фильтрация через Network Firewall, который анализирует destination домены/репутацию, а не только state таблицы соединений.

3. **Ответ:** Security Group поддерживает только Allow-правила. Если compliance требует явного запрета трафика из конкретной страны (санкционные требования, экспортный контроль), сделать это через Security Group невозможно — нужен NACL с Deny-правилом для IP-диапазонов этой страны. Geo-IP Deny на NACL — это stateless-запрет, поэтому нужно добавить и Deny для ответного трафика на ephemeral порты. Это классический пример, где NACL необходим как комплементарный механизм. ISO 27001 A.13.1 (Network Security Management) требует возможности запрета определённых сетевых соединений.

4. **Ответ:** VPC Peering создаёт pairwise-соединения. 50 VPC требуют 1225 пирингов, каждый с route table entry в обеих сторонах, каждый требует одобрения, и нетранзитивность означает, что Shared Services VPC должен peerиться со всеми остальными. Это нарушает принцип scalability: количество конфигурационных операций растёт квадратично, а не линейно. Transit Gateway решает это через hub-and-spoke: 50 аттачментов + центральные routing tables. NIST SP 800-53 AC-4 требует масштабируемого enforcement информационных потоков, что квадратичный Peering не обеспечивает.

5. **Ответ:** VPC Endpoint для S3 имеет Endpoint Policy — IAM-политику, прикреплённую к самому endpoint, которая ограничивает, к каким S3-бакетам можно обращаться через этот endpoint. Можно сказать: «Этот VPC Endpoint может обращаться только к бакету my-logs и ни к какому другому». Если скомпрометированный ресурс пытается использовать этот endpoint для доступа к чужому бакету (data exfiltration через другой S3-аккаунт), Endpoint Policy отклоняет запрос. Без endpoint все ресурсы VPC могли бы обращаться к любым бакетам, на которые у них есть IAM-права.

6. **Ответ:** Service Mesh оправдан, когда: микросервисов больше 20-30, они часто передеплоиваются (IP меняются), нужна observability на уровне L7 (latency per service, retry rates, circuit breaking), и нужен uniform policy enforcement для всех сервисов (mTLS everywhere). Для монолитного приложения с тремя компонентами (web, app, db) на статических IP — Security Groups достаточно, и Service Mesh добавит ненужный latency overhead. Компромисс: Service Mesh — дополнительный слой, который нужно поддерживать (версионирование sidecar, конфигурация, дебаггинг), и он оправдан при определённом пороге сложности.

7. **Ответ:** VPC Flow Logs — единственный источник truth о том, какой сетевой трафик реально происходит в VPC. Без них организация не знает: открыл ли разработчик порт для отладки и забыл закрыть; общается ли скомпрометированный сервер с внешним IP; есть ли сканирование внутри VPC. В расследовании инцидента Flow Logs отвечают на вопрос «какой сетевой след оставил злоумышленник?». CIS v8 Control 12.5 (Network Monitoring) и NIST SP 800-53 SI-4 (System Monitoring) требуют сбора и анализа сетевых данных, включая Flow Logs.

8. **Ответ:** HTTPS скрывает содержимое трафика от глубокой инспекции, но не скрывает SNI (Server Name Indication) в TLS handshake — доменное имя целевого сервера видно в plaintext. Network Firewall извлекает SNI из Client Hello в TLS-рукопожатии и сравнивает с доменным списком (allowed/blocked). Если C&C-сервер использует домен `xyz.malware.net`, даже при HTTPS-трафике файрвол видит SNI и блокирует соединение до установления TLS-сессии. Эффективность ограничена: если C&C использует IP-адрес без домена, SNI отсутствует, и детекция требует более глубокого анализа (JA3 fingerprinting, который некоторые файрволы поддерживают).

9. **Ответ:** PrivateLink — unidirectional connection: только consumer VPC может инициировать соединения в provider VPC (где находится NLB), но не наоборот. VPC Peering — bidirectional: обе стороны могут инициировать соединения, если route tables и Security Groups разрешают. Если два аккаунта принадлежат разным организациям (поставщик SaaS и клиент), bidirectional peering создаёт риск: клиент может попытаться сканировать VPC поставщика. PrivateLink ограничивает направление инициации — SaaS-провайдер выставляет endpoint, клиент потребляет его, обратного пути нет. NIST SP 800-53 AC-4 требует enforcement направления информационного потока, что PrivateLink выполняет из коробки.

10. **Ответ:** В Kubernetes IP-адреса подов динамические (пересоздаются при перезапуске), и полагаться на IP-адрес в сетевых правилах ненадёжно — после перезапуска сервиса его IP изменится, и правило Security Group устареет. Service Mesh решает это через идентификацию на основе SPIFFE-идентификаторов (X.509-сертификат, привязанный к сервисному аккаунту), а не IP-адресов. Правило: «Сервис X может вызывать сервис Y» — реализуется через mTLS и авторизационную политику Istio/Linkerd без единого IP. Это decoupling сетевой адресации от политик безопасности, необходимый для масштабирования микросервисной архитектуры.

## Ключевые выводы для собеседования

- **Самый критичный принцип:** VPC — не защита сама по себе, а набор примитивов (подсети, SG, NACL, Flow Logs, Endpoints), которые должны быть правильно сконфигурированы. Каждый инцидент в облаке, связанный с сетью — это ошибка использования примитивов, а не их отсутствие.
- **Главный компромисс:** Чем гранулярнее микросегментация (per-resource SG, Service Mesh, Network Policies), тем меньше blast radius инцидента, но тем выше операционная нагрузка на поддержку правил и риск false-positive блокировок, которые парализуют разработку. Баланс: автоматизированное создание правил на основе declarative service definitions + self-service exception portal.
- **Связь с ключевым стандартом:** NIST SP 800-207 (Zero Trust Architecture) и AC-4 (Information Flow Enforcement) — две оси: первая требует микросегментации и проверки каждого потока, вторая — мониторинга и enforcement всех разрешённых потоков.
- **Практический императив:** VPC Flow Logs на всех production-подсетях, доставляемые в SIEM с задержкой менее 5 минут, и автоматический anomaly detection — не опция, а mandatory baseline. Без этого организация слепа к сетевой активности в своём облаке на 100%.
