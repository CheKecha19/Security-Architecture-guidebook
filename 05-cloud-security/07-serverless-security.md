# Безопасность Serverless

## Что такое Serverless?

Представь, что ты хочешь приготовить ужин. Обычно ты идёшь на кухню, включаешь плиту, готовишь, моешь посуду, убираешь. Это как "серверный" подход — у тебя есть кухня, ты ей управляешь.

А теперь представь, что ты просто заказываешь еду. Ты не знаешь, где кухня, кто готовит, какая плита. Ты просто заказал — через 30 минут еда у двери. Ты платишь только за еду, не за кухню, не за повара, не за аренду.

**Serverless** — это именно такой "заказ еды" для программ. Ты пишешь код. Загружаешь в облако. Облако само запускает, когда нужно. Само масштабирует. Само выключает. Ты платишь только за время выполнения.

Примеры: AWS Lambda, Azure Functions, Google Cloud Functions.

## Зачем нужна безопасность Serverless?

### Ситуация 1: Функция с правами админа

Функция Lambda написана для простой задачи: преобразовать файл. Но ей дали права администратора "на всякий случай". Взломали функцию — получили полный доступ.

### Ситуация 2: Cold start — утечка данных

Функция запускается в общей среде. Прошлый запуск оставил данные в памяти. Новый запуск видит их.

### Ситуация 3: Event injection

Функция обрабатывает событие от API Gateway. Злоумышленник отправил crafted event. Функция выполнила вредоносный код.

## Архитектура Serverless

### Компоненты

| Компонент | Описание | Риск |
|-----------|----------|------|
| Function | Код | Уязвимости, права |
| Trigger | Событие, запускающее функцию | Injection |
| Event Source | Источник события | Подделка |
| IAM Role | Права функции | Избыточные права |
| VPC | Сеть | Доступ |
| Storage | S3, DynamoDB | Данные |

### Жизненный цикл

| Этап | Что происходит | Риск |
|------|----------------|------|
| Deploy | Код загружается | Секреты в коде |
| Cold start | Среда создаётся | Утечка данных |
| Execution | Код работает | Injection |
| Warm | Среда переиспользуется | Остатки данных |
| Idle | Среда уничтожается | Данные в логах |

## Основные угрозы

### 1. Overprivileged Functions

Функция имеет больше прав, чем нужно.

| Риск | Пример |
|------|--------|
| Чтение чужих данных | S3:GetObject для всех бакетов |
| Модификация | DynamoDB:PutItem |
| Удаление | EC2:TerminateInstances |
| Создание ресурсов | IAM:CreateUser |

**Защита:** Минимальные права. Роль только для нужных действий.

### 2. Event Injection

Входные данные содержат вредоносный код.

| Источник | Атака |
|----------|-------|
| API Gateway | SQL injection, XSS |
| S3 trigger | Вредоносный файл |
| SNS/SQS | Поддельное сообщение |
| CloudWatch | Log injection |

**Защита:** Input validation, sanitization.

### 3. Dependency Vulnerabilities

Функция использует библиотеки с уязвимостями.

| Проблема | Пример |
|----------|--------|
| Устаревшие библиотеки | Log4j, old npm packages |
| Transitive dependencies | Зависимости зависимостей |
| Typosquatting | Поддельные пакеты |

**Защита:** SCA, регулярное обновление.

### 4. Secrets in Code

API-ключи, пароли в коде функции.

| Где прячут | Почему плохо |
|------------|--------------|
| Environment variables | Видны в консоли |
| Code | В git |
| Config files | В образе |

**Защита:** Secrets manager, KMS, Parameter Store.

### 5. Cold Start Data Leakage

Данные из прошлого запуска доступны новому.

| Причина | Пример |
|---------|--------|
| Глобальные переменные | Остались в памяти |
| /tmp | Файлы на диске |
| Кэш | Результаты прошлых вызовов |

**Защита:** Не хранить sensitive в глобальных переменных. Очищать /tmp.

## Архитектурные решения

### IAM Least Privilege

```
Разрешено: s3:GetObject для bucket "my-bucket"
Запрещено: всё остальное
```

### Input Validation

| Проверка | Описание |
|----------|----------|
| Тип | String, number, JSON |
| Длина | Не больше N |
| Формат | Regex |
| Диапазон | Мин, макс |

### API Gateway Security

| Мера | Описание |
|------|----------|
| Authentication | Cognito, Lambda authorizer |
| Throttling | Rate limiting |
| WAF | Защита от injection |
| Logging | CloudWatch Logs |

### VPC for Functions

Запуск функций внутри VPC для доступа к приватным ресурсам.

| Плюс | Минус |
|------|-------|
| Приватность | Cold start дольше |
| Сетевая изоляция | Сложнее настроить |

## Мониторинг

| Метрика | Описание |
|---------|----------|
| Invocations | Количество вызовов |
| Errors | Ошибки |
| Duration | Время выполнения |
| Throttles | Превышение лимитов |
| Concurrency | Одновременные вызовы |

## Лучшие практики

| Практика | Описание |
|----------|----------|
| Минимальные права | IAM least privilege |
| Input validation | Проверка всех входных данных |
| Dependencies | SCA, обновление |
| Secrets | Хранилище секретов |
| Logging | Не логировать sensitive |
| Timeout | Минимальный таймаут |
| Memory | Минимально необходимая |
| Concurrency limits | Защита от DoS |

### Ответы на чек-лист

1. **Serverless** — модель, где облако управляет инфраструктурой. Ты пишешь код, облако запускает.

2. **Основные угрозы**: overprivileged functions, event injection, dependency vulnerabilities, secrets in code, cold start data leakage.

3. **Overprivileged function** — функция с избыточными правами. Взлом = полный доступ.

4. **Event injection** — вредоносные данные во входном событии. SQL injection, XSS.

5. **Секреты в коде** — API-ключи, пароли видны в git, консоли, логах.

6. **Cold start** — первый запуск функции. Создаётся среда. Может оставить данные в памяти.

7. **API Gateway**: authentication (Cognito, Lambda authorizer), throttling, WAF, logging.

8. **VPC для функций** — запуск внутри приватной сети. Плюс: приватность. Минус: дольше cold start.

9. **Метрики**: invocations, errors, duration, throttles, concurrency.

10. **Лучшие практики**: минимальные права, input validation, SCA, secrets manager, не логировать sensitive, минимальный timeout, concurrency limits.
